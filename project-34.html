🔴 HYPOTHETICAL PROBLEM

You run a financial services platform managing 500,000 bank accounts.

CURRENT ARCHITECTURE (Traditional CRUD):
  → PostgreSQL table: accounts (account_id, balance, last_updated)
  → When customer deposits $100:
      UPDATE accounts SET balance = balance + 100 WHERE account_id = 'ACC-001';
  → Current balance: $1,500
  → That's ALL you know

THE CRISES:
───────────

CRISIS 1: AUDIT FAILURE
  → Regulator asks: "Show me every transaction for ACC-001 in last 90 days"
  → Database has: balance = $1,500
  → HOW did it get to $1,500? 
  → Was it $1,000 + $500 deposit?
  → Or $10,000 - $8,500 in fraudulent withdrawals?
  → CRUD destroyed the history — you can only see CURRENT state
  → AUDIT IMPOSSIBLE

CRISIS 2: BUG RECOVERY
  → Bug in fee calculation charged customers 10x fees for 3 days
  → 50,000 accounts affected
  → Current balances are ALL WRONG
  → Cannot "undo" because original balances were OVERWRITTEN
  → Must manually calculate correct balances from bank statements
  → Takes 2 weeks and costs $500,000 in engineering time

CRISIS 3: TEMPORAL QUERY
  → Legal team needs: "What was ACC-001's balance at 2024-01-15 14:30:00?"
  → Database only has CURRENT balance
  → No time-travel capability
  → Cannot answer historical questions

CRISIS 4: NEW FEATURE BACKFILL
  → New requirement: "Show average daily balance for interest calculation"
  → Need historical balance at end of each day for past year
  → Data doesn't exist — was overwritten with each transaction
  → Cannot build new features that need historical state

CRISIS 5: SYSTEM MIGRATION
  → Moving from monolith to microservices
  → New account service needs complete transaction history
  → Source system only has current state
  → Migration is lossy — historical context lost forever

THE ROOT CAUSE:
───────────────
  CRUD stores STATE → destroys HISTORY
  Every UPDATE overwrites previous value
  DELETE removes existence entirely
  
  What if instead of storing STATE,
  we stored every EVENT that led to that state?

  DEPOSIT $500 at T1 → stored forever
  WITHDRAWAL $200 at T2 → stored forever
  FEE_CHARGED $10 at T3 → stored forever
  INTEREST_ADDED $5 at T4 → stored forever

  Current balance = replay(all events) = $500 - $200 - $10 + $5 = $295

  THIS IS EVENT SOURCING.
  And Kafka is the PERFECT event store.

🔵 HOW TO APPROACH IT

Step 1: UNDERSTAND EVENT SOURCING vs CRUD
  ┌─────────────────────────┬──────────────────────────────┐
  │ CRUD (Traditional)      │ EVENT SOURCING               │
  ├─────────────────────────┼──────────────────────────────┤
  │ Stores current state    │ Stores all events            │
  │ UPDATE overwrites       │ APPEND only — never modify   │
  │ History lost            │ History preserved forever     │
  │ One table, one row/key  │ Event log + materialized view│
  │ Query current state     │ Replay events → derive state │
  │ Simple reads            │ Read from projection/snapshot│
  │ Cannot audit            │ Complete audit trail          │
  │ Cannot time-travel      │ Replay to any point in time  │
  │ Bug = permanent damage  │ Bug = replay with fix         │
  └─────────────────────────┴──────────────────────────────┘

Step 2: MAP KAFKA CONCEPTS TO EVENT SOURCING
  ┌────────────────────────┬────────────────────────────────┐
  │ Event Sourcing Concept │ Kafka Implementation           │
  ├────────────────────────┼────────────────────────────────┤
  │ Event Store            │ Kafka Topic (append-only log)  │
  │ Event                  │ Kafka Record (key + value)     │
  │ Aggregate ID           │ Kafka Key (e.g., account_id)  │
  │ Event Stream           │ All records with same key      │
  │ Aggregate State        │ KTable / State Store (Proj 21) │
  │ Projection             │ Consumer → materialized view   │
  │ Snapshot               │ Compacted topic (Project 14)   │
  │ Replay                 │ Consumer from offset 0         │
  │ Temporal Query         │ Replay events up to timestamp  │
  │ Command                │ Producer record (intent)       │
  │ Event                  │ Confirmed record (fact)        │
  └────────────────────────┴────────────────────────────────┘

Step 3: DESIGN THE EVENT STORE
  Topic: "account-events"
  Key: account_id
  Value: {event_type, payload, timestamp, sequence_number, metadata}
  
  Retention: INFINITE (cleanup.policy=delete, retention.ms=-1)
  → Never delete events — they ARE the source of truth
  
  OR: cleanup.policy=compact+delete
  → Keep events for N days, then compact to latest per key
  → Snapshots preserve state even after event deletion

Step 4: UNDERSTAND THE READ/WRITE SPLIT
  WRITE SIDE (Command):
    → Validate command (sufficient funds?)
    → If valid → append event to Kafka topic
    → Event is IMMUTABLE — once written, never changed

  READ SIDE (Query):
    → Consumer reads events → builds materialized view
    → View stored in state store (Project 21/33)
    → Queries hit the materialized view (not event log)
    → This IS the CQRS pattern (Project 35 will expand)

🟡 WHAT COULD BE DONE

Option A: Database Event Table
  → Store events in PostgreSQL events table
  → Simple, familiar, ACID transactions
  → Problem: Not distributed, limited throughput
  → Problem: Retention management complex
  → Problem: No built-in streaming/replay

Option B: Kafka as Event Store (✓ CORRECT APPROACH)
  → Kafka IS an append-only log — natural event store
  → Distributed, fault-tolerant, horizontally scalable
  → Built-in retention, compaction, replay
  → Consumers build projections (state stores)
  → Multiple projections from same event stream
  → Problem: No built-in aggregate validation (must implement)
  → Problem: Ordering guaranteed only within partition

Option C: Dedicated Event Store (EventStoreDB, Axon)
  → Purpose-built for event sourcing
  → Built-in aggregate streams, projections, subscriptions
  → Optimistic concurrency control
  → Problem: Additional infrastructure
  → Problem: Doesn't integrate as naturally with Kafka ecosystem

Option D: Hybrid — Kafka Events + Database State
  → Events in Kafka (source of truth)
  → Current state in PostgreSQL (query optimization)
  → Kafka Connect syncs events → database
  → Best of both worlds but more complexity
  → This IS what most production systems use

✅ WHAT SHOULD DO

1. MAKE EVENTS IMMUTABLE AND SELF-DESCRIBING
   → Event once written MUST NEVER be modified
   → Each event contains ALL information needed to understand it
   → Include: event_type, timestamp, aggregate_id, payload, metadata
   → Include: schema_version for evolution (Project 38)
   → Include: causation_id linking to triggering command

2. USE INFINITE RETENTION FOR EVENT STORE TOPIC
   → retention.ms=-1 OR retention.bytes=-1
   → Events ARE your source of truth — deleting = data loss
   → If storage is concern → use tiered storage or archive

3. IMPLEMENT SNAPSHOTS FOR PERFORMANCE
   → Replaying 10 million events to get current balance = slow
   → Periodically snapshot: {account_id, balance, event_count, snapshot_at}
   → Replay only events AFTER snapshot
   → Store snapshots in compacted topic (Project 14)

4. DESIGN EVENTS AS PAST-TENSE FACTS
   → NOT: "DepositCommand" (intent — might fail)
   → YES: "FundsDeposited" (fact — already happened)
   → Events represent things that DID happen
   → Commands represent things we WANT to happen

5. INCLUDE SEQUENCE NUMBERS FOR ORDERING
   → Per-aggregate sequence: ACC-001 event #1, #2, #3...
   → Detect gaps: if #42 follows #40, event #41 was lost
   → Enable optimistic concurrency: "only write if last event is #41"

6. SEPARATE COMMAND VALIDATION FROM EVENT STORAGE
   → Load current state → validate command → if valid → store event
   → Never store events for invalid commands
   → Command handler is the ONLY writer to event store

7. BUILD MULTIPLE PROJECTIONS FROM SAME EVENTS
   → Projection 1: Current balance per account (for queries)
   → Projection 2: Daily transaction summary (for reports)
   → Projection 3: Fraud detection signals (for monitoring)
   → Same event stream, different consumers, different views

❌ WHAT SHOULD NOT DO

1. DO NOT MODIFY OR DELETE EVENTS
   → Events are IMMUTABLE FACTS — they happened
   → Correcting a mistake = append a COMPENSATING event
   → Wrong charge of $100? → append "CHARGE_REVERSED $100"
   → NEVER update the original event

2. DO NOT STORE DERIVED STATE AS EVENTS
   → Event: "FUNDS_DEPOSITED amount=500" ✓
   → NOT event: "BALANCE_UPDATED new_balance=1500" ✗
   → Derived state (balance) is computed, not stored as event
   → If you store derived state, you lose the ability to recompute

3. DO NOT USE LOG COMPACTION ON EVENT STORE TOPIC
   → Compaction keeps only LATEST value per key
   → Would delete ALL events except the last one per account
   → Entire history destroyed
   → Use compaction ONLY for snapshot topics, NOT event store
   → Event store: cleanup.policy=delete with retention.ms=-1

4. DO NOT REPLAY ALL EVENTS FOR EVERY QUERY
   → 10M events × 0.1ms each = 1000 seconds to answer one query
   → Use SNAPSHOTS + incremental replay
   → Or: materialized view (state store from Project 33)
   → Query the view, not the event log

5. DO NOT SKIP SEQUENCE NUMBER VALIDATION
   → If events arrive out of order → state corruption
   → Always validate: new_event.sequence == last_event.sequence + 1
   → If gap detected → halt processing, alert, investigate

6. DO NOT COUPLE EVENT SCHEMA TO CONSUMER LOGIC
   → Events should be self-describing
   → Consumer should handle unknown event types gracefully
   → Schema evolution (Project 38) applies to events too
   → New event types should not break existing consumers

7. DO NOT USE EVENT SOURCING FOR EVERYTHING
   → User preferences → CRUD is fine
   → Shopping cart → CRUD is fine (temporary state)
   → Financial transactions → EVENT SOURCING ✓
   → Audit-required domains → EVENT SOURCING ✓
   → Overengineering simple CRUD with event sourcing = pain

🚫 WHEN NOT TO USE

1. WHEN DOMAIN HAS NO AUDIT REQUIREMENTS
   → User preferences, UI settings, feature flags
   → Nobody asks "what was the setting at 3pm last Tuesday?"
   → CRUD is simpler, faster, cheaper

2. WHEN EVENT VOLUME IS EXTREMELY HIGH WITH NO REPLAY NEED
   → IoT sensor readings: 1 million events/sec per device
   → Storing ALL events = petabytes of storage
   → If only latest reading matters → use KTable (compaction)
   → Exception: if regulatory requires storing all readings

3. WHEN STRONG CONSISTENCY IS REQUIRED BETWEEN WRITE AND READ
   → Event sourcing is EVENTUALLY CONSISTENT
   → Write event → consumer processes → projection updated
   → Milliseconds to seconds delay between write and read
   → If "read-your-writes" is mandatory → use database

4. WHEN TEAM LACKS EVENT SOURCING EXPERIENCE
   → Significant learning curve
   → Event schema design is hard to change later
   → Debugging event replay issues is non-trivial
   → Start with one bounded context, not entire system

5. WHEN AGGREGATES HAVE VERY LONG EVENT HISTORIES
   → 100 million events per aggregate → replay takes minutes
   → Even with snapshots, managing this is complex
   → Consider: archive old events, keep recent + snapshots

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: USING LOG COMPACTION ON EVENT STORE
  → cleanup.policy=compact on "account-events" topic
  → Compaction keeps only LATEST event per account_id
  → 49,999 events deleted, only last one remains
  → ENTIRE HISTORY DESTROYED
  → FIX: Event store topic MUST use retention.ms=-1 (or large value)
  → Use compaction ONLY on snapshot topics

PITFALL 2: STORING DERIVED STATE AS EVENTS
  → Event: {type: "BALANCE_UPDATED", new_balance: 1500}
  → This stores DERIVED STATE, not the CAUSE
  → Cannot answer: "How did balance get to 1500?"
  → Cannot recompute if calculation was wrong
  → FIX: Store CAUSE events: "DEPOSITED 500", "WITHDRAWN 200"

    owner_name: str = ""

    currency: str = "USD"

    last_sequence: int = 0
    created_at: int = 0
    last_updated: int = 0
    total_deposits: float = 0.0
    total_withdrawals: float = 0.0
    total_fees: float = 0.0
    total_interest: float = 0.0

    def to_dict(self) -> dict:
        return asdict(self)

    def apply_event(self, event: DomainEvent) -> 'AccountState':
        """
        Apply a single event to derive new state.
        This is the core of event sourcing — state is a LEFT FOLD over events.
        state = events.reduce(initial_state, apply_event)
        """
        event_type = event.event_type
        payload = event.payload
        timestamp = event.metadata.timestamp

        # Update common fields
        self.event_count += 1
        self.last_sequence = event.sequence_number
        self.last_updated = timestamp

        if event_type == "ACCOUNT_OPENED":
            self.account_id = event.aggregate_id
            self.owner_name = payload.get("owner_name", "")
            self.account_type = payload.get("account_type", "")
            self.balance = payload.get("initial_deposit", 0.0)
            self.currency = payload.get("currency", "USD")
            self.status = "ACTIVE"
            self.created_at = timestamp
            if self.balance > 0:
                self.total_deposits += self.balance

        elif event_type == "FUNDS_DEPOSITED":
            amount = payload.get("amount", 0)
            self.balance += amount
            self.total_deposits += amount

        elif event_type == "FUNDS_WITHDRAWN":
            amount = payload.get("amount", 0)
            self.balance -= amount
            self.total_withdrawals += amount

        elif event_type == "FEE_CHARGED":
            amount = payload.get("amount", 0)
            self.balance -= amount
            self.total_fees += amount

        elif event_type == "INTEREST_APPLIED":
            amount = payload.get("amount", 0)
            self.balance += amount
            self.total_interest += amount

        elif event_type == "ACCOUNT_FROZEN":
            self.status = "FROZEN"

        elif event_type == "ACCOUNT_UNFROZEN":
            self.status = "ACTIVE"

        elif event_type == "ACCOUNT_CLOSED":
            self.status = "CLOSED"
            self.balance = 0.0

        elif event_type == "CHARGE_REVERSED":
            amount = payload.get("amount", 0)
            self.balance += amount  # Reverse = add back
            self.total_fees -= amount

        self.balance = round(self.balance, 2)
        return self

"""
PROJECT 34: Event Store — Core Implementation
================================================
The event store is the heart of event sourcing.
It stores, retrieves, and replays events for aggregates.
"""

import json
import time
import threading
from collections import defaultdict
from confluent_kafka import Producer, Consumer, TopicPartition
from confluent_kafka.admin import AdminClient
from event_models import DomainEvent, AccountState


class EventStore:
    """
    Kafka-backed Event Store.
    
    Responsibilities:
    1. Append events to the event log (Kafka topic)
    2. Read events for an aggregate (by key)
    3. Read events up to a timestamp (temporal queries)
    4. Manage snapshots for performance
    5. Reconstruct aggregate state from events
    """

    def __init__(self, bootstrap_servers: str = "localhost:9092"):
        self.bootstrap = bootstrap_servers
        self.event_topic = "account-events"
        self.snapshot_topic = "account-snapshots"

        # ── Producer for writing events ──
        self.producer = Producer({
            "bootstrap.servers": bootstrap_servers,
            "acks": "all",          # Durability (Project 11)
            "enable.idempotence": True,  # Exactly-once (Project 17)
            "max.in.flight.requests.per.connection": 5,
            "retries": 10,
        })

        # ── In-memory event cache (for fast replay) ──
        # In production: use RocksDB or query Kafka directly
        self._event_cache = defaultdict(list)
        self._snapshot_cache = {}
        self._cache_lock = threading.RLock()

        # ── Sequence tracking (for optimistic concurrency) ──
        self._sequence_tracker = {}  # {aggregate_id: last_sequence}

        # ── Metrics ──
        self.events_appended = 0
        self.events_read = 0
        self.snapshots_created = 0
        self.replays_performed = 0

    # ═══════════════════════════════════════════════════════════
    #  APPEND EVENT — Write to event log
    # ═══════════════════════════════════════════════════════════

    def append(self, event: DomainEvent, expected_sequence: int = None) -> bool:
        """
        Append an event to the event store.

        Optimistic Concurrency:
        If expected_sequence is provided, only append if the
        current last sequence matches. Prevents concurrent
        modifications from corrupting state.

        Returns True if event was appended, False if concurrency conflict.
        """
        aggregate_id = event.aggregate_id

        # ── Optimistic Concurrency Check ──
        if expected_sequence is not None:
            with self._cache_lock:
                current_seq = self._sequence_tracker.get(aggregate_id, 0)
                if current_seq != expected_sequence:
                    print(f"    ⚠️ Concurrency conflict for {aggregate_id}: "
                          f"expected seq {expected_sequence}, "
                          f"current is {current_seq}")
                    return False

        # ── Serialize and produce event ──
        event_json = event.to_json()

        self.producer.produce(
            topic=self.event_topic,
            key=aggregate_id.encode("utf-8"),
            value=event_json.encode("utf-8"),
            headers=[
                ("event_type", event.event_type.encode("utf-8")),
                ("sequence", str(event.sequence_number).encode("utf-8")),
                ("aggregate_type", event.aggregate_type.encode("utf-8")),
            ],
            callback=self._delivery_callback,
        )
        self.producer.flush()

        # ── Update cache ──
        with self._cache_lock:
            self._event_cache[aggregate_id].append(event)
            self._sequence_tracker[aggregate_id] = event.sequence_number

        self.events_appended += 1
        return True

    def _delivery_callback(self, err, msg):
        if err:
            print(f"    ❌ Event delivery failed: {err}")

    # ═══════════════════════════════════════════════════════════
    #  READ EVENTS — Retrieve event history for an aggregate
    # ═══════════════════════════════════════════════════════════

    def get_events(self, aggregate_id: str,
                   from_sequence: int = 0,
                   to_timestamp: int = None) -> list:
        """
        Read events for an aggregate.
        
        from_sequence: Start from this sequence (for snapshot + replay)
        to_timestamp: Stop at this timestamp (for temporal queries)
        
        Returns list of DomainEvent objects in order.
        """
        with self._cache_lock:
            all_events = self._event_cache.get(aggregate_id, [])

        # Filter by sequence
        filtered = [e for e in all_events if e.sequence_number > from_sequence]

        # Filter by timestamp (for temporal queries)
        if to_timestamp is not None:
            filtered = [e for e in filtered if e.metadata.timestamp <= to_timestamp]

        self.events_read += len(filtered)
        return filtered

    def get_all_events_from_topic(self, aggregate_id: str) -> list:
        """
        Read events directly from Kafka topic (bypasses cache).
        Used for cache rebuild or verification.
        """
        consumer = Consumer({
            "bootstrap.servers": self.bootstrap,
            "group.id": f"event-reader-{aggregate_id}-{int(time.time())}",
            "auto.offset.reset": "earliest",
            "enable.auto.commit": False,
        })
        consumer.subscribe([self.event_topic])

        events = []
        idle_count = 0

        while idle_count < 3:
            msg = consumer.poll(timeout=1.0)
            if msg is None:
                idle_count += 1
                continue
            if msg.error():
                continue
            idle_count = 0

            if msg.key() and msg.key().decode("utf-8") == aggregate_id:
                try:
                    event_data = json.loads(msg.value().decode("utf-8"))
                    event = DomainEvent.from_dict(event_data)
                    events.append(event)
                except (json.JSONDecodeError, UnicodeDecodeError):
                    continue

        consumer.close()
        events.sort(key=lambda e: e.sequence_number)
        return events

    # ═══════════════════════════════════════════════════════════
    #  RECONSTRUCT STATE — Replay events to build current state
    # ═══════════════════════════════════════════════════════════

    def reconstruct_state(self, aggregate_id: str,
                          at_timestamp: int = None) -> AccountState:
        """
        Reconstruct aggregate state by replaying events.
        
        This is THE CORE OPERATION of event sourcing:
        state = events.reduce(initial_state, apply_event)
        
        Strategy:
        1. Check for snapshot → start from snapshot state
        2. Read events after snapshot
        3. Apply each event to state
        4. Return final state
        
        at_timestamp: If provided, replay only up to this time (temporal query)
        """
        self.replays_performed += 1
        start_time = time.time()

        # ── Step 1: Load snapshot (if exists) ──
        snapshot = self._load_snapshot(aggregate_id)

        if snapshot and at_timestamp is None:
            # Start from snapshot
            state = AccountState(**snapshot["state"])
            from_sequence = snapshot["last_sequence"]
        elif snapshot and at_timestamp and snapshot.get("snapshot_at", 0) <= at_timestamp:
            # Snapshot is before target time — use it
            state = AccountState(**snapshot["state"])
            from_sequence = snapshot["last_sequence"]
        else:
            # No usable snapshot — replay from beginning
            state = AccountState()
            from_sequence = 0

        # ── Step 2: Read events after snapshot ──
        events = self.get_events(
            aggregate_id,
            from_sequence=from_sequence,
            to_timestamp=at_timestamp,
        )

        # ── Step 3: Apply each event (LEFT FOLD) ──
        for event in events:
            state = state.apply_event(event)

        elapsed_ms = (time.time() - start_time) * 1000

        return state

    # ═══════════════════════════════════════════════════════════
    #  SNAPSHOTS — Periodic state checkpoints
    # ═══════════════════════════════════════════════════════════

    def create_snapshot(self, aggregate_id: str) -> dict:
        """
        Create a snapshot of current aggregate state.
        
        Stored in compacted topic (Project 14):
        Key = aggregate_id → only latest snapshot retained
        
        Snapshot includes:
        - Full current state
        - Sequence number of last event included
        - Timestamp of snapshot creation
        """
        # Reconstruct current state
        state = self.reconstruct_state(aggregate_id)

        snapshot = {
            "aggregate_id": aggregate_id,
            "state": state.to_dict(),
            "last_sequence": state.last_sequence,
            "event_count": state.event_count,
            "snapshot_at": int(time.time() * 1000),
            "snapshot_version": state.event_count // 100 + 1,
        }

        # Store in compacted snapshot topic
        self.producer.produce(
            topic=self.snapshot_topic,
            key=aggregate_id.encode("utf-8"),
            value=json.dumps(snapshot).encode("utf-8"),
        )
        self.producer.flush()

        # Update cache
        with self._cache_lock:
            self._snapshot_cache[aggregate_id] = snapshot

        self.snapshots_created += 1
        return snapshot

    def _load_snapshot(self, aggregate_id: str) -> dict:
        """Load latest snapshot from cache"""
        with self._cache_lock:
            return self._snapshot_cache.get(aggregate_id)

    def should_snapshot(self, aggregate_id: str,

class TransactionSummaryProjection:

        summary["transaction_count"] += 1

            deposit = event.payload.get("initial_deposit", 0)
            self.running_balances[account_id] = deposit
            summary["deposits"] += deposit

            amount = event.payload.get("amount", 0)
            summary["deposits"] += amount
            self.running_balances[account_id] += amount

        elif event.event_type == "FUNDS_WITHDRAWN":
            amount = event.payload.get("amount", 0)
            summary["withdrawals"] += amount
            self.running_balances[account_id] -= amount

        elif event.event_type == "FEE_CHARGED":
            amount = event.payload.get("amount", 0)
            summary["fees"] += amount
            self.running_balances[account_id] -= amount

        elif event.event_type == "INTEREST_APPLIED":
            amount = event.payload.get("amount", 0)
            summary["interest"] += amount
            self.running_balances[account_id] += amount

        elif event.event_type == "CHARGE_REVERSED":
            amount = event.payload.get("amount", 0)
            summary["fees"] -= amount
            self.running_balances[account_id] += amount

        summary["end_of_day_balance"] = round(self.running_balances[account_id], 2)

        if date_str:
