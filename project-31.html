🔴 HYPOTHETICAL PROBLEM

You run a global e-commerce platform
  → Orders arrive from 6 geographic regions:
      NA (North America) — 40% of all traffic
      EU (Europe) — 25% of all traffic
      APAC (Asia Pacific) — 20% of all traffic
      LATAM (Latin America) — 8% of all traffic
      MEA (Middle East & Africa) — 5% of all traffic
      OCE (Oceania) — 2% of all traffic

You have a Kafka topic with 12 partitions
  → Default key-based hashing (Project 4) assigns:
      Key "NA" → hashes to Partition 7
      Key "EU" → hashes to Partition 3
      Key "APAC" → hashes to Partition 11
      Key "LATAM" → hashes to Partition 1
      Key "MEA" → hashes to Partition 9
      Key "OCE" → hashes to Partition 5

THE CRISIS:
  → Partition 7 (NA) receives 40% of ALL messages
  → Partition 3 (EU) receives 25%
  → Partition 11 (APAC) receives 20%
  → 6 partitions sit completely EMPTY (0 messages)
  → 3 partitions are overwhelmed → consumer lag explodes
  → Consumer assigned to Partition 7 is 10x busier than others
  → Disk usage on broker hosting Partition 7 is skewed
  → Tail latency for NA orders is 15 seconds (unacceptable)

DEFAULT HASHING CANNOT SOLVE THIS
  → Because 6 unique keys can only map to 6 of 12 partitions
  → High-volume keys create "HOT PARTITIONS"
  → No built-in mechanism for weighted distribution

🔵 HOW TO APPROACH IT

Step 1: UNDERSTAND WHY DEFAULT FAILS
  → Project 4 showed: key.hashCode() % numPartitions = partition
  → When you have FEW unique keys but MANY partitions
    → Most partitions receive ZERO messages
  → When one key dominates traffic
    → That single partition becomes a bottleneck

Step 2: RECOGNIZE THE PATTERN
  → This is NOT a "random distribution" problem
  → This is a "WEIGHTED SPREAD" problem
  → NA (40% traffic) should spread across MULTIPLE partitions
  → OCE (2% traffic) needs only ONE partition

Step 3: IDENTIFY THE SOLUTION LAYERS
  Layer 1: Custom Partitioner (replaces default hash)
  Layer 2: Weighted partition mapping (proportional allocation)
  Layer 3: Sub-key randomization (spread within allocated range)
  Layer 4: Partition-aware consumers (aligned processing)

🟢 HOW CAN WE SOLVE IT

SOLUTION: BUILD A GEOGRAPHIC WEIGHTED PARTITIONER

Partition allocation strategy (12 partitions total):
  → NA   (40%) → Partitions 0, 1, 2, 3, 4     (5 partitions)
  → EU   (25%) → Partitions 5, 6, 7            (3 partitions)
  → APAC (20%) → Partitions 8, 9               (2 partitions)
  → LATAM (8%) → Partition 10                   (1 partition)
  → MEA   (5%) → Partition 11                   (1 partition)
  → OCE   (2%) → Partition 11 (shared with MEA) (shared)

WITHIN each region's allocated range:
  → Messages distribute ROUND-ROBIN or RANDOM
  → This prevents hot-spotting within the range

RESULT:
  → NA's 40% traffic spreads across 5 partitions = ~8% each
  → EU's 25% traffic spreads across 3 partitions = ~8.3% each
  → Every partition receives roughly ~8% of total traffic
  → BALANCED ✓

🟡 WHAT COULD BE DONE

Option A: Sub-key Approach (Modify the Key)
  → Instead of key="NA", use key="NA-{random_uuid}"
  → Default partitioner will hash different values → spread across partitions
  → Problem: Loses ordering guarantee per region
  → Problem: Consumers cannot identify region from partition assignment

Option B: Round-Robin (No Key)
  → Send all messages with null key
  → Kafka uses sticky partitioning (round-robin batch)
  → Problem: Completely loses geographic affinity
  → Problem: Cannot route region-specific consumers

Option C: Custom Partitioner (✓ CORRECT APPROACH)
  → Implement org.apache.kafka.clients.producer.Partitioner
  → Define region-to-partition-range mapping
  → Distribute within range using sub-key or counter
  → Preserves geographic affinity
  → Enables partition-aware consumers per region
  → Allows weighted distribution proportional to traffic

Option D: Increase Partition Count
  → Use 100+ partitions, hope hashing spreads better
  → Problem: 6 unique keys still map to only 6 partitions
  → DOES NOT SOLVE THE FUNDAMENTAL PROBLEM

✅ WHAT SHOULD DO

1. DESIGN PARTITION MAP BEFORE CODING
   → Calculate traffic percentages per region
   → Allocate partition ranges proportionally
   → Document the mapping explicitly

2. IMPLEMENT Partitioner INTERFACE
   → Override partition() method with custom logic
   → Override configure() method for dynamic config loading
   → Override close() for resource cleanup

3. USE THREAD-SAFE COUNTERS FOR ROUND-ROBIN
   → AtomicInteger or ThreadLocalRandom for within-range distribution
   → Multiple producer threads share the same Partitioner instance

4. MAKE MAPPING CONFIGURABLE
   → Pass region-partition mapping via producer config
   → Avoid hardcoding — allow runtime changes

5. BUILD FALLBACK LOGIC
   → Unknown region → default to hash-based assignment
   → Never throw exceptions from partition() method
   → Log warnings for unmapped regions

6. ALIGN CONSUMERS WITH PARTITION RANGES
   → NA processing team consumes partitions 0–4
   → EU processing team consumes partitions 5–7
   → Use manual partition assignment (assign()) not subscribe()

7. MONITOR PARTITION BALANCE
   → Track messages-per-partition metrics (Project 28)
   → Alert if any partition deviates >20% from expected load

❌ WHAT SHOULD NOT DO

1. DO NOT THROW EXCEPTIONS IN partition() METHOD
   → Producer will fail ENTIRELY if partition() throws
   → Always return a valid partition number
   → Fallback to default hashing for unknown inputs

2. DO NOT HARDCODE PARTITION NUMBERS
   → Topic partition count can change (though rarely)
   → Always use the partitions parameter passed to partition()
   → Calculate ranges dynamically based on available partitions

3. DO NOT IGNORE THREAD SAFETY
   → One Partitioner instance shared across all producer threads
   → Mutable state (counters, maps) must be thread-safe
   → AtomicInteger, ConcurrentHashMap — not HashMap, int

4. DO NOT ALLOCATE TOO FEW PARTITIONS TO HIGH-TRAFFIC REGIONS
   → Underprovision → right back to hot partition problem
   → Always leave room for traffic growth

5. DO NOT MIX subscribe() AND assign() IN CONSUMERS
   → Region-aware consumers use assign() for specific partitions
   → Cannot combine with group-managed subscription
   → Choose one approach per consumer instance

6. DO NOT FORGET TO REGISTER THE PARTITIONER
   → Must set 'partitioner.class' in producer config
   → Missing config → silently uses DefaultPartitioner
   → No error thrown — just wrong behavior

7. DO NOT CREATE PARTITION MAPPING THAT DOESN'T SUM TO TOTAL
   → 12 partitions allocated must exactly equal topic partition count
   → Gaps → orphan partitions that never receive data
   → Overlaps → unexpected double-routing

3. WHEN PARTITION COUNT CHANGES FREQUENTLY
   → Custom mapping must be recalculated on every change
   → Rebalance invalidates hardcoded ranges
   → Prefer default hashing if topology is dynamic

4. WHEN CONSUMERS DON'T NEED PARTITION AFFINITY
   → If all consumers process all regions identically
   → Custom partitioning adds no value
   → Just use round-robin for maximum balance

5. WHEN USING KAFKA STREAMS
   → Kafka Streams manages its own partitioning via repartition topics
   → Custom partitioner on input topic may conflict with
     co-partitioning requirements (Project 32 will cover this)

═══════════════════════════════════════════════════════════════
         WEIGHTED DISTRIBUTION INTERNAL LOGIC
═══════════════════════════════════════════════════════════════

Given: region = "NA", allocated partitions = [0, 1, 2, 3, 4]

Method A: AtomicInteger Round-Robin
  counter.getAndIncrement() % allocatedPartitions.length
  Call 1: 0 % 5 = 0 → Partition 0
  Call 2: 1 % 5 = 1 → Partition 1
  Call 3: 2 % 5 = 2 → Partition 2
  Call 4: 3 % 5 = 3 → Partition 3
  Call 5: 4 % 5 = 4 → Partition 4
  Call 6: 5 % 5 = 0 → Partition 0  (wraps around)
  → PERFECTLY EVEN within range
  → Slight ordering concern: messages from same sub-entity
    may land on different partitions

Method B: Sub-Key Hashing
  hash(orderId) % allocatedPartitions.length
  → Same order_id always → same partition within range
  → Maintains per-order ordering
  → Distribution depends on hash quality

Method C: ThreadLocalRandom
  ThreadLocalRandom.current().nextInt(allocatedPartitions.length)
  → Statistically even over large volume
  → No counter overflow concerns
  → Slightly less deterministic

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: INTEGER OVERFLOW ON ROUND-ROBIN COUNTER
  → AtomicInteger wraps to negative at Integer.MAX_VALUE
  → negative % 5 = NEGATIVE NUMBER → ArrayIndexOutOfBounds
  → FIX: Use Math.abs() or bitwise AND: (counter & 0x7fffffff)
  → BETTER FIX: Use toUnsignedInt or modular arithmetic

PITFALL 2: PARTITION COUNT MISMATCH
  → Partitioner coded for 12 partitions
  → Someone adds partitions → topic now has 15
  → Partitions 12, 13, 14 never receive messages
  → FIX: Read partition count dynamically from Cluster metadata
  → Recalculate ranges based on actual count

PITFALL 3: NULL KEY HANDLING
  → partition() receives null key for some messages
  → Your region-extraction logic throws NullPointerException
  → Producer DIES for entire batch
  → FIX: Always check for null key first → fallback to default

PITFALL 4: SLOW PARTITION LOGIC
  → Making HTTP calls or DB lookups inside partition()
  → Called for EVERY message → latency multiplied by millions
  → Producer throughput drops to near-zero
  → FIX: Pre-compute everything in configure(), cache in memory

PITFALL 5: FORGETTING close() CLEANUP
  → Partitioner opens monitoring connections in configure()
  → close() never implemented → resource leak on producer shutdown
  → FIX: Always implement close() even if empty

PITFALL 6: CONSUMER assign() WITHOUT OFFSET MANAGEMENT
  → assign() does not commit offsets to consumer group
  → Consumer restart → starts from unknown position
  → FIX: Manually commit offsets or use seek() on startup

PITFALL 7: TESTING WITH SINGLE PARTITION
  → Dev environment has 1 partition
  → Custom partitioner always returns 0
  → "Works fine" → deploy to prod with 12 partitions → breaks
  → FIX: Integration test with production-like partition count

"""
PROJECT 31: Geographic Weighted Partitioner
============================================
Builds on: Project 4 (key-based routing), Project 9 (multi-partition producing)
Solves: Hot partition problem when few keys dominate traffic

Region-to-Partition Allocation (12 partitions):
  NA    → Partitions [0, 1, 2, 3, 4]  — 5 partitions (40% traffic)
  EU    → Partitions [5, 6, 7]         — 3 partitions (25% traffic)
  APAC  → Partitions [8, 9]            — 2 partitions (20% traffic)
  LATAM → Partition  [10]              — 1 partition  (8% traffic)
  MEA   → Partition  [11]              — 1 partition  (5% traffic)
  OCE   → Partition  [11]              — shared with MEA (2% traffic)
"""

import hashlib
import json
import threading
import time
from dataclasses import dataclass, field


# ═══════════════════════════════════════════════════════════════
#  PARTITION MAP CONFIGURATION
# ═══════════════════════════════════════════════════════════════

@dataclass
class RegionPartitionConfig:
    """
    Defines the partition range for a geographic region.
    - partitions: list of partition numbers allocated to this region
    - counter: thread-safe round-robin counter within the range
    """
    partitions: list
    counter: int = 0
    lock: threading.Lock = field(default_factory=threading.Lock)

    def next_partition_round_robin(self) -> int:
        """Thread-safe round-robin within allocated partitions"""
        with self.lock:
            partition = self.partitions[self.counter % len(self.partitions)]
            self.counter += 1
            # Prevent integer overflow (Python handles big ints,
            # but reset for cleanliness)
            if self.counter >= 1_000_000_000:
                self.counter = 0
            return partition

    def partition_by_subkey(self, sub_key: str) -> int:
        """Hash-based distribution within range using sub-key (e.g., order_id)"""
        hash_val = int(hashlib.md5(sub_key.encode()).hexdigest(), 16)
        index = hash_val % len(self.partitions)
        return self.partitions[index]


class GeographicPartitioner:
    """
    Custom partitioner that distributes messages across partitions
    based on geographic region with weighted allocation.

    This replaces the default murmur2 hash partitioner.
    """

    def __init__(self, total_partitions: int = 12, config: dict = None):
        """
        Initialize with partition mapping.

        In Java Kafka, this would be the configure() method.
        In Python confluent-kafka, we handle this in __init__.
        """
        self.total_partitions = total_partitions

        # ── Default weighted allocation ──
        # Can be overridden via config parameter
        if config is None:
            self.region_map = self._build_default_map(total_partitions)
        else:
            self.region_map = self._build_map_from_config(config, total_partitions)

        # ── Metrics tracking ──
        self.partition_counts = {i: 0 for i in range(total_partitions)}
        self.unknown_region_count = 0
        self.metrics_lock = threading.Lock()

        # ── Log the mapping at startup ──
        self._log_mapping()

    def _build_default_map(self, total: int) -> dict:
        """
        Build default region-to-partition mapping for 12 partitions.
        Proportional to expected traffic:
          NA=40%, EU=25%, APAC=20%, LATAM=8%, MEA=5%, OCE=2%
        """
        if total < 6:
            raise ValueError(
                f"Need at least 6 partitions for 6 regions, got {total}"
            )

        # ── Calculate proportional allocation ──
        # Minimum 1 partition per region, remaining distributed by weight
        weights = {
            "NA": 0.40,
            "EU": 0.25,
            "APAC": 0.20,
            "LATAM": 0.08,
            "MEA": 0.05,
            "OCE": 0.02,
        }

        allocation = self._allocate_partitions(weights, total)
        return self._build_region_configs(allocation)

    def _allocate_partitions(self, weights: dict, total: int) -> dict:
        """
        Proportional partition allocation with minimum guarantee.
        Every region gets at least 1 partition.
        Remaining partitions distributed by weight.
        """
        regions = list(weights.keys())
        num_regions = len(regions)

        # Step 1: Everyone gets 1 partition minimum
        alloc = {r: 1 for r in regions}
        remaining = total - num_regions

        # Step 2: Distribute remaining by weight
        if remaining > 0:
            # Calculate weighted shares of remaining
            raw_shares = {r: weights[r] * remaining for r in regions}

            # Floor allocation
            floor_shares = {r: int(raw_shares[r]) for r in regions}
            for r in regions:
                alloc[r] += floor_shares[r]

            # Distribute leftover (due to rounding) to highest remainders
            distributed = sum(floor_shares.values())
            leftover = remaining - distributed
            remainders = {
                r: raw_shares[r] - floor_shares[r] for r in regions
            }
            sorted_by_remainder = sorted(
                remainders.keys(), key=lambda r: remainders[r], reverse=True
            )
            for i in range(leftover):
                alloc[sorted_by_remainder[i]] += 1

        return alloc

    def _build_region_configs(self, allocation: dict) -> dict:
        """Convert allocation counts to partition range assignments"""
        region_map = {}
        current_partition = 0

        for region, count in allocation.items():
            partitions = list(range(current_partition, current_partition + count))
            region_map[region] = RegionPartitionConfig(partitions=partitions)
            current_partition += count

        return region_map

    def _build_map_from_config(self, config: dict, total: int) -> dict:
        """
        Build from explicit configuration.
        config format: {"NA": [0,1,2,3,4], "EU": [5,6,7], ...}
        """
        region_map = {}
        all_partitions = set()

        for region, partitions in config.items():
            # Validate partition numbers
            for p in partitions:
                if p < 0 or p >= total:
                    raise ValueError(
                        f"Partition {p} for region {region} "
                        f"is out of range [0, {total})"
                    )
                if p in all_partitions:
                    raise ValueError(
                        f"Partition {p} assigned to multiple regions"
                    )
                all_partitions.add(p)

            region_map[region] = RegionPartitionConfig(partitions=list(partitions))

        return region_map

    def _log_mapping(self):
        """Log the complete partition mapping for observability"""
        print("=" * 60)
        print("GEOGRAPHIC PARTITIONER — PARTITION MAPPING")
        print("=" * 60)
        for region, config in self.region_map.items():
            print(f"  {region:>8s} → Partitions {config.partitions}")
        print("=" * 60)

    # ═══════════════════════════════════════════════════════════════
    #  CORE PARTITION METHOD — Called for EVERY message
    # ═══════════════════════════════════════════════════════════════

    def partition(self, region: str, sub_key: str = None) -> int:
        """
        Determine the partition for a message.

        This is the equivalent of Java's Partitioner.partition() method.

        Args:
            region: Geographic region code (NA, EU, APAC, etc.)
            sub_key: Optional sub-key for deterministic within-range routing
                     (e.g., order_id — same order always goes to same partition)

        Returns:
            int: Partition number in range [0, total_partitions)
        """
        # ── Guard: null/empty region ──
        if not region:
            # Fallback: distribute across all partitions
            fallback = hash(sub_key or str(time.time_ns())) % self.total_partitions
            self._track_metric(fallback, unknown=True)
            return fallback

        # ── Normalize region ──
        region_upper = region.upper().strip()

        # ── Lookup region in map ──
        region_config = self.region_map.get(region_upper)

        if region_config is None:
            # Unknown region — fallback to default hashing
            print(f"  [WARN] Unknown region '{region_upper}' — using fallback hash")
            fallback = hash(region_upper) % self.total_partitions
            self._track_metric(fallback, unknown=True)
            return fallback

        # ── Determine partition within region's range ──
        if sub_key:
            # Deterministic: same sub_key → same partition (within range)
            # Use case: all orders for same customer go to same partition
            partition = region_config.partition_by_subkey(sub_key)
        else:
            # Round-robin: even spread across region's partitions
            partition = region_config.next_partition_round_robin()

        self._track_metric(partition)
        return partition

    def _track_metric(self, partition: int, unknown: bool = False):
        """Thread-safe metrics tracking"""
        with self.metrics_lock:
            self.partition_counts[partition] = (
                self.partition_counts.get(partition, 0) + 1
            )
            if unknown:
                self.unknown_region_count += 1

    #  MONITORING & DIAGNOSTICS
    # ═══════════════════════════════════════════════════════════════

    def get_distribution_report(self) -> dict:
        """Get current partition distribution statistics"""
        with self.metrics_lock:
            total = sum(self.partition_counts.values())
            if total == 0:
                return {"total": 0, "partitions": {}, "balance_score": 0.0}

            report = {
                "total_messages": total,
                "unknown_region_count": self.unknown_region_count,
                "partitions": {},
                "regions": {},
            }

            # Per-partition stats
            for p, count in sorted(self.partition_counts.items()):
                pct = (count / total) * 100 if total > 0 else 0
                report["partitions"][p] = {
                    "count": count,
                    "percentage": round(pct, 2)
                }

            # Per-region stats
            for region, config in self.region_map.items():
                region_total = sum(
                    self.partition_counts.get(p, 0)
                    for p in config.partitions
                )
                pct = (region_total / total) * 100 if total > 0 else 0
                report["regions"][region] = {
                    "partitions": config.partitions,
                    "total_messages": region_total,
                    "percentage": round(pct, 2),
                }

            # Balance score: 1.0 = perfectly balanced, 0.0 = completely skewed
            counts = [c for c in self.partition_counts.values() if c > 0]
            if counts:
                ideal = total / len(counts)
                max_deviation = max(abs(c - ideal) for c in counts)
                report["balance_score"] = round(
                    1.0 - (max_deviation / total), 4
                )
            else:
                report["balance_score"] = 0.0

            return report

    def close(self):
        """Cleanup resources (mirrors Java Partitioner.close())"""
        print("\n[PARTITIONER] Closing — Final distribution report:")
        report = self.get_distribution_report()
        for region, stats in report.get("regions", {}).items():
            print(
                f"  {region}: {stats['total_messages']} messages "
                f"({stats['percentage']}%) across partitions {stats['partitions']}"
            )
        print(f"  Balance Score: {report.get('balance_score', 'N/A')}")

                        → Consumer group rebalancing applies
                        → No control over WHICH partitions

                             → NO consumer group rebalancing
                             → Full control over partition assignment
                             → Must handle failover manually
    ─────────────────────────────────────────────────────────────
    """

        self,
        region: str,

        start_from: str = "latest"

        self.running = True

            raise ValueError(
                f"Unknown region '{self.region}'. "
                f"Valid: {list(REGION_PARTITION_MAP.keys())}"

        # ── Consumer Configuration ──
        # Note: group.id is still useful for offset tracking
        # even with manual partition assignment

            "enable.auto.commit": True,
            "auto.commit.interval.ms": 5000,

        print(f"  Assigned Partitions: {self.assigned_partitions}")

        self.messages_consumed = 0
        self.region_match_count = 0
        self.region_mismatch_count = 0

"""

=========================================================
Monitors partition distribution and recommends rebalancing
when hot partitions are detected.

Builds on: Project 28 (monitoring concepts)
"""

import json
import time
from confluent_kafka.admin import AdminClient
from confluent_kafka import Consumer, TopicPartition


class HotPartitionDetector:
    """
    Detects partition imbalance by sampling message rates
    and recommends partition reallocation.
    """

    def __init__(self, bootstrap_servers: str = "localhost:9092"):
        self.admin = AdminClient({
            "bootstrap.servers": bootstrap_servers
        })

        # Temporary consumer just for offset queries
        self.consumer = Consumer({
            "bootstrap.servers": bootstrap_servers,
            "group.id": "hot-partition-detector-temp",
            "enable.auto.commit": False,
        })

    def get_partition_offsets(self, topic: str) -> dict:
        """Get latest offset for each partition (indicates message count)"""
        metadata = self.admin.list_topics(topic=topic, timeout=10)
        topic_metadata = metadata.topics[topic]
        num_partitions = len(topic_metadata.partitions)

        offsets = {}
        for p in range(num_partitions):
            tp = TopicPartition(topic, p)
            low, high = self.consumer.get_watermark_offsets(tp, timeout=10)
            offsets[p] = {
                "low_offset": low,
                "high_offset": high,
                "message_count": high - low,
            }

        return offsets

    def detect_hot_partitions(
        self,
        topic: str,
        threshold_multiplier: float = 2.0
    ) -> dict:
        """
        Detect partitions receiving disproportionate traffic.

        A partition is "hot" if its message count exceeds
        the average by more than threshold_multiplier times.
        """
        offsets = self.get_partition_offsets(topic)

        counts = [v["message_count"] for v in offsets.values()]
        total = sum(counts)
        num_partitions = len(counts)

        if total == 0:
            return {"status": "NO_DATA", "partitions": {}}

        avg = total / num_partitions

        result = {
            "topic": topic,
            "total_messages": total,
            "num_partitions": num_partitions,
            "average_per_partition": round(avg, 2),
            "threshold_multiplier": threshold_multiplier,
            "hot_partitions": [],
            "cold_partitions": [],
            "balanced_partitions": [],
            "partitions": {},
        }

        for p, info in offsets.items():
            count = info["message_count"]
            ratio = count / avg if avg > 0 else 0
            status = "BALANCED"

            if ratio > threshold_multiplier:
                status = "HOT"
                result["hot_partitions"].append(p)
            elif ratio < (1.0 / threshold_multiplier):
                status = "COLD"
                result["cold_partitions"].append(p)
            else:
                result["balanced_partitions"].append(p)

            result["partitions"][p] = {
                "message_count": count,
                "ratio_to_avg": round(ratio, 3),
                "status": status,
            }

        # Overall health assessment
        hot_count = len(result["hot_partitions"])
        cold_count = len(result["cold_partitions"])

        if hot_count == 0 and cold_count == 0:
            result["health"] = "HEALTHY"
            result["recommendation"] = "No action needed"
        elif hot_count > 0:
            result["health"] = "UNHEALTHY"
            result["recommendation"] = (
                f"Partitions {result['hot_partitions']} are HOT. "
                f"Consider implementing a custom partitioner to spread "
                f"traffic from high-volume keys across multiple partitions."
            )
        else:
            result["health"] = "SUBOPTIMAL"
            result["recommendation"] = (
                f"Partitions {result['cold_partitions']} are COLD (unused). "
                f"Consider reducing partition count or adjusting key strategy."
            )

        return result

    def recommend_reallocation(
        self,
        topic: str,
        region_traffic: dict
    ) -> dict:
        """
        Given observed traffic percentages per region,
        recommend optimal partition allocation.
        """
        offsets = self.get_partition_offsets(topic)
        num_partitions = len(offsets)

        # Sort regions by traffic (descending)
        sorted_regions = sorted(
            region_traffic.items(), key=lambda x: x[1], reverse=True
        )

        # Allocate proportionally with minimum 1 per region
        allocation = {}
        remaining = num_partitions
        total_weight = sum(region_traffic.values())

        # First pass: minimum 1 each
        for region, _ in sorted_regions:
            allocation[region] = 1
            remaining -= 1

        # Second pass: distribute remaining by weight
        for region, weight in sorted_regions:
            extra = int((weight / total_weight) * remaining)
            allocation[region] += extra

        # Handle any unallocated due to rounding
        allocated = sum(allocation.values())
        if allocated < num_partitions:
            allocation[sorted_regions[0][0]] += (num_partitions - allocated)

        # Build partition ranges
        partition_map = {}
        current = 0
        for region, count in allocation.items():
            partition_map[region] = list(range(current, current + count))
            current += count

        return {
            "total_partitions": num_partitions,
            "region_traffic_input": region_traffic,
            "recommended_allocation": allocation,
            "recommended_partition_map": partition_map,
            "expected_per_partition_balance": {
                region: round(
                    (region_traffic[region] / allocation[region]) * 100, 2
                )
                for region in allocation
                if allocation[region] > 0
            },
        }

    def print_report(self, topic: str):
        """Print a comprehensive partition health report"""
        print(f"\n{'═' * 70}")
        print(f"  HOT PARTITION DETECTION REPORT — Topic: {topic}")

            )

            for topic, future in futures.items():
                try:
                    future.result()

                        print(f"  [CONFIG] Config topic already exists: {topic}")
                    else:
                        raise
        except Exception as e:

    def publish_new_mapping(self, mapping: dict, version: int):

        only the latest mapping is retained.

class PartitionConfigListener:
    """
    Listens for partition mapping updates on the config topic.
    Updates the GeographicPartitioner in real-time.
    """

    def __init__(
        self,
        partitioner: GeographicPartitioner,
        bootstrap_servers: str = "localhost:9092"
    ):
        self.partitioner = partitioner
        self.running = True
        self.current_version = 0

        self.consumer = Consumer({
            "bootstrap.servers": bootstrap_servers,
            "group.id": "partitioner-config-listener",
            "auto.offset.reset": "earliest",
            "enable.auto.commit": True,
        })
        self.consumer.subscribe([CONFIG_TOPIC])

        # Start listener in background thread
        self.thread = threading.Thread(
            target=self._listen_loop,
            daemon=True,
            name="config-listener"
        )
        self.thread.start()
        print("  [CONFIG LISTENER] Started background config listener")

    def _listen_loop(self):
        """Background loop that watches for config updates"""
        while self.running:
            msg = self.consumer.poll(timeout=1.0)

            if msg is None:
                continue
            if msg.error():
                print(f"  [CONFIG LISTENER] Error: {msg.error()}")
                continue

            try:
                config = json.loads(msg.value().decode("utf-8"))
                version = config.get("version", 0)

                # Only apply if newer version
                if version > self.current_version:
                    self._apply_new_mapping(config)
                    self.current_version = version

            except json.JSONDecodeError:
                print("  [CONFIG LISTENER] Malformed config message")

    def _apply_new_mapping(self, config: dict):
        """
        Hot-swap the partition mapping in the running partitioner.
        Thread-safe: new map is built separately, then swapped atomically.
        """
        mapping = config["mapping"]
        version = config["version"]

        print(f"\n  [CONFIG LISTENER] Applying new mapping v{version}...")

        # Build new region map
        new_region_map = {}
        for region, partitions in mapping.items():
            new_region_map[region] = RegionPartitionConfig(
                partitions=partitions
            )

        # Atomic swap (Python dict assignment is atomic due to GIL)
        self.partitioner.region_map = new_region_map

        print(f"  [CONFIG LISTENER] Mapping v{version} applied successfully")
        for region, config_obj in new_region_map.items():
            print(f"    {region:<8} → Partitions {config_obj.partitions}")

    def stop(self):
        """Stop the listener"""
        self.running = False
        self.thread.join(timeout=5)
        self.consumer.close()
        print("  [CONFIG LISTENER] Stopped")
