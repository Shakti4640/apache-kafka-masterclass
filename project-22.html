PROJECT 22: THE TRAFFIC SURGE MONITOR — WINDOWED OPERATIONS

🚦 HYPOTHETICAL PROBLEM

A ride-sharing platform processes 10 million trip requests per day

SCENARIO: Real-Time Traffic Surge Detection & Dynamic Pricing
→ Every trip request generates an event to "trip-requests" topic
→ Business needs FOUR time-based metrics:

  METRIC 1: "Trip requests per zone in each 5-minute window"
  → Zone-A [10:00-10:05]: 342 requests
  → Zone-A [10:05-10:10]: 891 requests ← SURGE DETECTED
  → Used for: Dynamic pricing ("surge pricing" activation)

  METRIC 2: "Rolling 15-minute request count, updated every 5 minutes"
  → Zone-A [09:50-10:05]: 800 requests
  → Zone-A [09:55-10:10]: 1,200 requests
  → Zone-A [10:00-10:15]: 1,450 requests
  → Used for: Trend analysis (is demand increasing or decreasing?)

  METRIC 3: "Average trip fare per zone in sliding 10-minute window"
  → Zone-B [any 10-min span containing event]: $24.50 avg
  → Used for: Revenue monitoring

  METRIC 4: "User activity sessions with 30-minute inactivity gap"
  → User-123 session: 10:00 → 10:45 (3 requests, 45 min active)
  → User-123 new session: 11:30 → 11:35 (1 request after 45 min gap)
  → Used for: User engagement analytics

THE CRISIS WITH PROJECT 21 (UNBOUNDED) APPROACH:

  Developer uses count() from Project 21:

  requests.groupBy((k,v) -> v.getZone())
          .count()
          .toStream()
          .to("zone-counts");

  Output: "Zone-A: 45,230"

  Problems:
  → 45,230 total since when? Since application started? Since January?
  → Business asks: "How many requests in Zone-A in the LAST 5 MINUTES?"
  → Unbounded count CANNOT answer this — it only goes up
  → Cannot detect SURGE: "Zone-A jumped from 200 to 800 in 5 minutes"
  → Cannot compute TRENDS: "Is Zone-A demand increasing or decreasing?"
  → State grows FOREVER: every unique zone accumulates indefinitely
  → After 1 year: state store contains ALL historical data

  Developer tries manual windowing:

  requests.mapValues(v -> {
      long windowStart = v.getTimestamp() / 300000 * 300000; // 5-min bucket
      return windowStart + ":" + v.getZone();
  }).groupByKey().count();

  Problems:
  → Manual window calculation is error-prone
  → No handling for LATE arrivals (event arrives after window closes)
  → No automatic window EXPIRATION (old windows accumulate forever)
  → No grace period management
  → Session windows are IMPOSSIBLE to implement manually
  → Kafka Streams provides ALL of this natively

THE ANSWER: WINDOWED AGGREGATIONS
  → TimeWindows (tumbling) — fixed non-overlapping buckets
  → TimeWindows (hopping) — overlapping sliding buckets
  → SlidingWindows — event-triggered differential windows
  → SessionWindows — activity-based variable-length windows
  → Built-in retention, grace periods, late event handling

🔍 HOW TO APPROACH IT

Step 1: Understand the FOUR window types

  TUMBLING WINDOW (fixed, non-overlapping):
  ┌──────────┐┌──────────┐┌──────────┐┌──────────┐
  │ 10:00    ││ 10:05    ││ 10:10    ││ 10:15    │
  │   -      ││   -      ││   -      ││   -      │
  │ 10:05    ││ 10:10    ││ 10:15    ││ 10:20    │
  └──────────┘└──────────┘└──────────┘└──────────┘
  → Each event belongs to EXACTLY ONE window
  → Windows don't overlap
  → Simplest windowing model

  HOPPING WINDOW (fixed, overlapping):
  ┌────────────────┐
  │ 10:00 - 10:15  │ (window 1)
  └────────────────┘
       ┌────────────────┐
       │ 10:05 - 10:20  │ (window 2)
       └────────────────┘
            ┌────────────────┐
            │ 10:10 - 10:25  │ (window 3)
            └────────────────┘
  → Window size: 15 min, Advance: 5 min
  → Each event belongs to MULTIPLE windows (size/advance = 3 windows)
  → Provides "rolling" view of data

  SLIDING WINDOW (event-triggered):
  Event A at 10:02, Event B at 10:07, Event C at 10:11
  
  Window for pair (A,B): [10:02, 10:07] — within 10-min difference
  Window for pair (B,C): [10:07, 10:11] — within 10-min difference
  Window for pair (A,C): [10:02, 10:11] — EXCEEDS 10 min → different windows
  
  → Windows are created based on EVENT TIMESTAMPS
  → Not based on fixed time boundaries
  → Two events in same window if their timestamps differ by ≤ window size
  → Most granular windowing — fewest events missed

  SESSION WINDOW (activity-based, variable length):
  User-123: request at 10:00, 10:05, 10:20, 10:45
            ────────────────────────────────────────
            [10:00 ──────────────────────── 10:45] ONE session (gaps < 30 min)

  User-123: request at 10:00, 10:05, then 11:00
            [10:00 ── 10:05] session 1 (gap to 11:00 > 30 min)
            [11:00] session 2
  
  → No fixed window size — sessions GROW with activity
  → Inactivity gap defines session boundary
  → Sessions can MERGE when new event bridges two sessions

Step 2: Understand window KEY structure
  → Unbounded: KTable<String, Long>     key = "Zone-A"
  → Windowed:  KTable<Windowed<String>, Long>
               key = Windowed("Zone-A", [10:00-10:05])
  → The key INCLUDES the window time range
  → Same zone in different windows = different keys

Step 3: Understand RETENTION and GRACE PERIOD
  → Retention: How long to KEEP window results in state store
    → After retention expires: window data deleted from RocksDB
    → Default: window size + 24 hours
  → Grace period: How LATE an event can arrive and still be counted
    → Event for window [10:00-10:05] arrives at 10:12
    → If grace = 10 min → accepted (10:12 - 10:05 = 7 min < 10 min)
    → If grace = 5 min → REJECTED (7 min > 5 min)
    → Rejected events are silently dropped (or sent to DLQ)

Step 4: Map each metric to a window type
  Metric 1 (requests per zone per 5-min): TUMBLING WINDOW
  Metric 2 (rolling 15-min count):        HOPPING WINDOW
  Metric 3 (avg fare in sliding 10-min):  SLIDING WINDOW
  Metric 4 (user activity sessions):      SESSION WINDOW

✅ WHAT COULD BE DONE

Option A: Manual Timestamp Bucketing (Pre-Kafka Streams)
  → Round timestamps to nearest 5 minutes manually
  → Use as part of grouping key: "Zone-A:10:00"
  → PROBLEM: No late event handling
  → PROBLEM: No automatic window expiration
  → PROBLEM: Session windows impossible
  → PROBLEM: Hopping windows require complex multi-key logic

Option B: External Time-Series Database
  → Write raw events to InfluxDB/TimescaleDB
  → Query with time-based aggregations
  → PROBLEM: Not real-time (query latency)
  → PROBLEM: Additional infrastructure
  → PROBLEM: No stream processing integration

Option C: Apache Flink Windowing
  → More advanced windowing than Kafka Streams
  → Better event-time watermark handling
  → PROBLEM: Separate cluster infrastructure
  → PROBLEM: Overkill for most Kafka-centric use cases
  → NOTE: Appropriate for very complex CEP (Project 44 bridge)

Option D: Kafka Streams Windowed Aggregations ← CORRECT
  → Native windowing in the DSL
  → Four window types cover all use cases
  → Automatic state expiration via retention
  → Grace period for late events
  → Built on top of state stores (Project 21)
  → No external infrastructure

✅ WHAT SHOULD DO

1. CHOOSE THE RIGHT WINDOW TYPE FOR THE BUSINESS QUESTION
   → "How many per fixed period?" → TUMBLING
   → "What's the rolling trend?" → HOPPING
   → "Events within N time of each other?" → SLIDING
   → "Group by activity with inactivity gaps?" → SESSION

2. SET GRACE PERIOD BASED ON EXPECTED LATENESS
   → Events may arrive out of order (network delays, retries)
   → Grace period = maximum expected lateness
   → Too small: late events silently dropped → incorrect counts
   → Too large: windows stay open longer → more memory → delayed output
   → Typical: 1-5 minutes for most streaming applications
   → For exactly zero tolerance: .ofSizeWithNoGrace()

3. SET RETENTION BASED ON QUERY NEEDS
   → Retention = how long to keep closed window results
   → Must be ≥ window size + grace period
   → If downstream queries historical windows: increase retention
   → If only current windows matter: keep retention minimal
   → Default: window size + 24 hours

4. USE SUPPRESS() FOR FINAL RESULTS ONLY
   → Without suppress: KTable emits on EVERY update within window
   → 100 events in 5-min window = 100 intermediate emissions
   → suppress(untilWindowCloses()) waits until window closes
   → Emits ONCE with final count
   → Trade-off: higher latency but cleaner output

5. USE EVENT-TIME, NOT PROCESSING-TIME
   → Kafka Streams uses RECORD TIMESTAMP by default
   → TimestampExtractor controls which timestamp to use
   → Event-time: timestamp from the event payload (business time)
   → Processing-time: time when record is processed (wall clock)
   → Event-time gives correct results even with delayed processing
   → Set via StreamsConfig.DEFAULT_TIMESTAMP_EXTRACTOR_CLASS_CONFIG

6. MONITOR WINDOW STATE SIZE
   → Each open window = state store entries
   → Hopping windows: each event exists in (windowSize/advance) windows
   → 15-min window / 1-min advance = 15 copies of each event
   → Monitor RocksDB size and changelog topic size

❌ WHAT SHOULD NOT DO

1. DO NOT USE TUMBLING WINDOWS WHEN YOU NEED OVERLAP
   → Tumbling: event at 10:04 is in [10:00-10:05] ONLY
   → If window boundary misses a surge: "10:03-10:08 had 500 requests"
   → But tumbling windows [10:00-10:05] and [10:05-10:10] split it
   → Use HOPPING windows for rolling visibility

2. DO NOT SET GRACE PERIOD TO ZERO IN PRODUCTION
   → .ofSizeWithNoGrace() drops ANY late event
   → In distributed systems, events ALWAYS arrive late sometimes
   → Even 30 seconds of grace prevents most data loss
   → Zero grace is only for testing or when lateness is truly impossible

3. DO NOT USE SESSION WINDOWS FOR FIXED-TIME ANALYSIS
   → Session windows have VARIABLE length
   → Cannot compare "session count for hour X vs hour Y"
   → Sessions may span multiple hours
   → Use tumbling or hopping for fixed-time analysis

4. DO NOT IGNORE HOPPING WINDOW STATE MULTIPLICATION
   → Window size 1 hour, advance 1 minute = 60 windows per event
   → State size = 60x compared to tumbling
   → For 1 million events: 60 million state entries
   → Choose advance interval carefully
   → Larger advance = fewer windows = less state

5. DO NOT SUPPRESS() WITHOUT UNDERSTANDING THE LATENCY IMPACT
   → suppress(untilWindowCloses()) delays output by window size + grace
   → 5-min window + 2-min grace = 7 minutes before ANY output
   → For real-time dashboards: emit intermediates (no suppress)
   → For batch downstream: suppress for clean final results

6. DO NOT FORGET WINDOWED KEY SERIALIZATION
   → Windowed<String> key needs WindowedSerdes
   → Cannot use Serdes.String() for windowed keys
   → TimeWindowedDeserializer / SessionWindowedDeserializer required
   → Common ClassCastException source

STATE SIZE COMPARISON (for same data volume):

  Tumbling 5-min:       1x (baseline)
  Hopping 15-min/5-min: 3x (15/5 = 3 windows per event)
  Hopping 60-min/1-min: 60x (60/1 = 60 windows per event!)
  Sliding 10-min:       ~2x (variable, depends on event distribution)
  Session 30-min gap:   variable (depends on user activity patterns)

🚫 WHEN NOT TO USE

1. WHEN ALL-TIME AGGREGATION IS SUFFICIENT
   → "Total orders per category" doesn't need windowing
   → Use unbounded count() from Project 21
   → Windowing adds state overhead for time boundaries

2. WHEN BATCH PROCESSING IS ACCEPTABLE
   → "Daily report of orders per category"
   → Can be done with batch query on stored data
   → Windowed streaming is overkill for daily batch reports
   → Unless real-time daily running total is needed

3. WHEN EVENT ORDER IS GUARANTEED AND LATENESS IS ZERO
   → Grace period and retention add complexity
   → If events always arrive in order with no delay
   → Simpler custom logic might suffice
   → BUT: this scenario is rare in distributed systems

4. WHEN WINDOW STATE WOULD BE TOO LARGE
   → High-cardinality key + hopping windows with small advance
   → 10M unique keys × 60 windows = 600M state entries
   → Consider pre-aggregation or external time-series DB

MECHANISM 1: TUMBLING WINDOW — FIXED BOUNDARY CALCULATION
═══════════════════════════════════════════════════════════════════

  TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(5))

  For event with timestamp 10:07:23:
  
  Window calculation:
    windowSize = 5 * 60 * 1000 = 300,000 ms
    eventTime = timestamp in ms
    windowStart = (eventTime / windowSize) * windowSize
    windowEnd = windowStart + windowSize

    10:07:23 = epoch + some ms
    windowStart = floor(eventTime / 300000) * 300000
    → Window: [10:05:00, 10:10:00)
    
  ┌──────────┐┌──────────┐┌──────────┐
  │10:00-10:05││10:05-10:10││10:10-10:15│
  │           ││  ↑ HERE  ││           │
  └──────────┘└──────────┘└──────────┘

  State store key: Windowed("Zone-A", TimeWindow(10:05, 10:10))
  State store value: 43 (count so far in this window)

MECHANISM 2: HOPPING WINDOW — MULTIPLE WINDOW MEMBERSHIP
═══════════════════════════════════════════════════════════════════

  TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(15))
             .advanceBy(Duration.ofMinutes(5))

  For event at 10:07:23:
  
  Which windows contain this event?
    windowSize = 15 min
    advance = 5 min
    numWindows = windowSize / advance = 3

    Window 1: [09:55:00, 10:10:00) ← contains 10:07:23 ✓
    Window 2: [10:00:00, 10:15:00) ← contains 10:07:23 ✓
    Window 3: [10:05:00, 10:20:00) ← contains 10:07:23 ✓

  ┌────────────────┐
  │ 09:55 - 10:10  │ ← event counted here
  └────────────────┘
       ┌────────────────┐
       │ 10:00 - 10:15  │ ← AND here
       └────────────────┘
            ┌────────────────┐
            │ 10:05 - 10:20  │ ← AND here
            └────────────────┘

  State store: THREE entries updated for ONE event:
    Windowed("Zone-A", [09:55, 10:10]) → count + 1
    Windowed("Zone-A", [10:00, 10:15]) → count + 1
    Windowed("Zone-A", [10:05, 10:20]) → count + 1

  STATE MULTIPLICATION:
  → Each event updates (windowSize / advance) windows
  → 15min / 5min = 3 windows per event
  → 60min / 1min = 60 windows per event ← DANGEROUS

═══════════════════════════════════════════════════════════════════
MECHANISM 3: SLIDING WINDOW — EVENT-DRIVEN BOUNDARIES
═══════════════════════════════════════════════════════════════════

  SlidingWindows.ofTimeDifferenceWithNoGrace(Duration.ofMinutes(10))

  Events for "Zone-A": A@10:02, B@10:07, C@10:14

  Windows created:
    For event A@10:02:
      Left window:  [09:52:01, 10:02:00]  ← events within 10 min BEFORE A
      Right window: [10:02:01, 10:12:00]  ← events within 10 min AFTER A

    For event B@10:07:
      Left window:  [09:57:01, 10:07:00]  ← contains A ✓
      Right window: [10:07:01, 10:17:00]  ← contains C ✓

    For event C@10:14:
      Left window:  [10:04:01, 10:14:00]  ← contains B ✓
      Right window: [10:14:01, 10:24:00]  ← empty so far

  KEY DIFFERENCE FROM HOPPING:
  → Hopping: fixed boundaries (10:00, 10:05, 10:10...)
  → Sliding: boundaries SHIFT based on event timestamps
  → Sliding is most PRECISE — never splits events at arbitrary boundaries
  → But creates MORE windows (two per event)

MECHANISM 4: SESSION WINDOW — MERGING BEHAVIOR
═══════════════════════════════════════════════════════════════════

  SessionWindows.ofInactivityGapWithNoGrace(Duration.ofMinutes(30))

  User-123 events: 10:00, 10:15, 10:25, 11:30

  Processing step by step:

  Event 10:00:
    → No existing session → Create new: [10:00, 10:00]
    State: {User-123: [10:00, 10:00] → count=1}

  Event 10:15:
    → Check: any session within 30 min of 10:15?
    → [10:00, 10:00] + 30min gap → extends to 10:30 → 10:15 is within range
    → MERGE: [10:00, 10:15] → count=2
    State: {User-123: [10:00, 10:15] → count=2}

  Event 10:25:
    → Check: [10:00, 10:15] + 30min gap → extends to 10:45 → 10:25 is within
    → MERGE: [10:00, 10:25] → count=3
    State: {User-123: [10:00, 10:25] → count=3}

  Event 11:30:
    → Check: [10:00, 10:25] + 30min gap → extends to 10:55 → 11:30 is OUTSIDE
    → CREATE NEW session: [11:30, 11:30] → count=1
    State: {
      User-123: [10:00, 10:25] → count=3,  (closed session)
      User-123: [11:30, 11:30] → count=1   (new session)
    }

  SESSION MERGE COMPLEXITY:
  → Out-of-order event can bridge two separate sessions
  → Sessions A=[10:00,10:05] and B=[10:40,10:50]
  → Late event at 10:20 arrives
  → 10:20 within 30min of A end (10:05+30=10:35) → merge with A
  → New A=[10:00,10:20], end+30=10:50 → overlaps with B start!
  → MERGE ALL: [10:00, 10:50] → single session
  → This cascading merge is handled automatically by Kafka Streams

MECHANISM 5: GRACE PERIOD — LATE EVENT HANDLING
═══════════════════════════════════════════════════════════════════

  TimeWindows.ofSizeAndGrace(Duration.ofMinutes(5), Duration.ofMinutes(2))

  Window: [10:00, 10:05)
  Grace: 2 minutes → window accepts events until 10:07

  Timeline:
  10:03 → Event A → Window [10:00,10:05) → ACCEPTED (within window)
  10:06 → Event B for 10:04 → ACCEPTED (10:06 < 10:05+2min grace)
  10:08 → Event C for 10:03 → REJECTED (10:08 > 10:05+2min grace)

  How Kafka Streams tracks "current time":
  → Stream time = maximum event timestamp seen so far
  → NOT wall clock time
  → If no new events arrive, stream time doesn't advance
  → Window closes when: stream time > window end + grace period

  Rejected events:
  → Silently dropped (default)
  → No error, no exception
  → Lost forever unless captured by custom handler
  → In production: monitor dropped-late-records metric

MECHANISM 6: WINDOW RETENTION — STATE CLEANUP
═══════════════════════════════════════════════════════════════════

  Materialized.as("store").withRetention(Duration.ofHours(2))

  → Windows older than 2 hours are DELETED from state store
  → Changelog topic also cleaned (compaction removes expired keys)
  → Prevents unbounded state growth

  Default retention = max(window size + grace period, 24 hours)

  Retention timeline:
  ┌──────────┐                    ┌─ RETAINED ─┐
  │ 08:00    │ ← EXPIRED          │ 10:00      │
  │ window   │   (deleted from    │ window     │
  │          │    RocksDB)        │            │
  └──────────┘                    └────────────┘

  With retention = 2 hours, at 12:00:
  → Windows before 10:00 are deleted
  → Windows 10:00-12:00 are retained
  → Queries for windows before 10:00 return null

MECHANISM 7: SUPPRESS — FINAL RESULTS ONLY
═══════════════════════════════════════════════════════════════════

  WITHOUT suppress():
    Window [10:00, 10:05):
    Event 1 → emit ("Zone-A", 1)
    Event 2 → emit ("Zone-A", 2)
    Event 3 → emit ("Zone-A", 3)
    ...
    Event 100 → emit ("Zone-A", 100)
    → 100 emissions for one window!

  WITH suppress(untilWindowCloses()):
    Window [10:00, 10:05):
    Event 1 → buffer internally
    Event 2 → buffer internally
    ...
    Event 100 → buffer internally
    Window closes (stream time > 10:05 + grace) →
    emit ("Zone-A", 100)
    → 1 emission for one window!

  suppress() uses memory to buffer:
  → Suppressed.BufferConfig.maxRecords(1000)
  → Suppressed.BufferConfig.maxBytes(1024 * 1024)
  → If buffer full: spill to disk or emit early (configurable)

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: HOPPING WINDOW STATE EXPLOSION
─────────────────────────────────────────
  Symptom: Disk fills up, OOM, slow recovery
  Root Cause: Small advance relative to window size
  Example:
    TimeWindows.ofSizeWithNoGrace(Duration.ofHours(1))
               .advanceBy(Duration.ofSeconds(10))
    → 3600/10 = 360 windows per event per key!
    → 1M events × 360 = 360M state entries
  Fix: Use larger advance interval
       Rule of thumb: advance ≥ windowSize / 10

PITFALL 2: LATE EVENTS SILENTLY DROPPED
─────────────────────────────────────────
  Symptom: Aggregation counts lower than expected
  Root Cause: Grace period too small for actual event lateness
  Detection: Monitor "dropped-late-records" metric
  Fix: Increase grace period
       Or use .ofSizeWithNoGrace() only when lateness is truly zero

PITFALL 3: SUPPRESS() CAUSING MEMORY PRESSURE
─────────────────────────────────────────
  Symptom: OOM errors during high-volume windows
  Root Cause: suppress() buffers ALL records until window closes
  Example: 5-minute window with 100K events → 100K buffered
  Fix: Set buffer limits:
       .suppress(untilWindowCloses()
         .withBufferConfig(BufferConfig.maxBytes(50_000_000)))
       Or: don't suppress — let downstream handle intermediates

PITFALL 4: WINDOWED KEY SERIALIZATION ERRORS
─────────────────────────────────────────
  Symptom: ClassCastException or SerializationException
  Root Cause: Using Serdes.String() for Windowed<String> keys
  Fix: Use WindowedSerdes.timeWindowedSerdeFrom(String.class, windowSizeMs)
       Or TimeWindowedSerializer / TimeWindowedDeserializer

PITFALL 5: SESSION WINDOWS GROWING UNBOUNDED
─────────────────────────────────────────
  Symptom: Single session spanning hours or days
  Root Cause: Continuous activity without gaps exceeding inactivity threshold
  Example: Automated system sends events every 10 seconds
           Session gap = 30 minutes → session NEVER closes
  Fix: Combine session windows with max session duration check
       Or use tumbling windows for always-active sources

─────────────────────────────────────────
  Symptom: Consumer can't parse keys from windowed output topic
  Root Cause: Windowed keys are serialized as: key + windowStart + windowEnd
  Fix: Use .toStream((windowedKey, value) -> windowedKey.key())
       to extract just the key portion
       Or use proper WindowedDeserializer on consumer side

     * Sliding window average fare per zone.
     * Needs aggregate() because we track sum + count for average.
     * Same pattern as RegionStats in Project 21 but windowed.
     */
    public static class ZoneFareStats {

        @JsonProperty("zone")
        private String zone;

        @JsonProperty("total_fare")
        private double totalFare;

        @JsonProperty("request_count")
        private long requestCount;

        @JsonProperty("average_fare")
        private double averageFare;

        @JsonProperty("min_fare")
        private double minFare;

        @JsonProperty("max_fare")
        private double maxFare;

        public ZoneFareStats() {
            this.totalFare = 0.0;
            this.requestCount = 0;
            this.averageFare = 0.0;
            this.minFare = Double.MAX_VALUE;
            this.maxFare = 0.0;
        }

        public ZoneFareStats update(String zone, TripRequest request) {
            ZoneFareStats updated = new ZoneFareStats();
            updated.zone = zone;
            updated.totalFare = this.totalFare + request.getFareEstimate();
            updated.requestCount = this.requestCount + 1;
            updated.averageFare = updated.totalFare / updated.requestCount;
            updated.minFare = Math.min(this.minFare, request.getFareEstimate());
            updated.maxFare = Math.max(this.maxFare, request.getFareEstimate());
            return updated;
        }

        public String getZone()         { return zone; }
        public double getTotalFare()    { return totalFare; }
        public long getRequestCount()   { return requestCount; }
        public double getAverageFare()  { return averageFare; }
        public double getMinFare()      { return minFare == Double.MAX_VALUE ? 0 : minFare; }
        public double getMaxFare()      { return maxFare; }

        @Override
        public String toString() {
            return String.format(
                "ZoneFare{zone=%s, count=%d, avg=$%.2f, min=$%.2f, max=$%.2f}",
                zone, requestCount, averageFare, getMinFare(), maxFare);

        // TimeWindows.ofSizeAndGrace(5 min, 1 min)
        // → Fixed 5-minute non-overlapping windows
        // → Each event belongs to exactly ONE window

def parse_windowed_key(key_str):
    """Parse windowed key format: entity|windowStart|windowEnd"""
    if key_str and '|' in key_str:
        parts = key_str.split('|')
        if len(parts) >= 3:
            entity = parts[0]
            try:
                start_ms = int(parts[1])
                end_ms = int(parts[2])
                start_str = datetime.fromtimestamp(
                    start_ms / 1000, tz=timezone.utc
                ).strftime('%H:%M:%S')
                end_str = datetime.fromtimestamp(
                    end_ms / 1000, tz=timezone.utc
                ).strftime('%H:%M:%S')
                duration_sec = (end_ms - start_ms) / 1000
                return {
                    'entity': entity,
                    'start': start_str,
                    'end': end_str,
                    'start_ms': start_ms,
                    'end_ms': end_ms,
                    'duration_sec': duration_sec,
                }
            except (ValueError, OSError):
                pass
    return {'entity': key_str, 'start': '?', 'end': '?', 'duration_sec': 0}


def deserialize_value(raw_value):
    """Attempt multiple deserialization strategies"""
    if raw_value is None:
        return None
    try:
        # Try as 8-byte Long (count output)
        if len(raw_value) == 8:
            return struct.unpack('>q', raw_value)[0]
    except struct.error:
        pass
    try:
        # Try as String (toString() output)
        return raw_value.decode('utf-8')
    except UnicodeDecodeError:
        return f"(binary: {len(raw_value)} bytes)"
