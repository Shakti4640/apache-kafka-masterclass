  → CEO: "How can adding ONE field bring down the entire system?"

ROOT CAUSE ANALYSIS:

  Problem 1: NO COMPATIBILITY RULES
    → Schema Registry accepted ANY schema change
    → No validation that new schema works with existing consumers

  Problem 2: NO EVOLUTION STRATEGY
    → Team didn't understand backward vs forward compatibility
    → Didn't know HOW to add/remove fields safely

  Problem 3: MIXED SCHEMA VERSIONS IN TOPIC
    → Old producers still sending v1 messages
    → New producers sending v2 messages
    → Consumers must handle BOTH versions simultaneously

  Problem 4: NO TESTING OF SCHEMA CHANGES
    → Schema deployed directly to production
    → No compatibility check before deployment

THE QUESTION:
  How do we evolve schemas safely in a system where:
  → Producers and consumers are deployed independently
  → Old and new messages coexist in the same topic
  → Dozens of consumers parse the same topic
  → Zero downtime is required
  → Rollback must be possible at any time

🟡 HOW TO APPROACH IT

CORE INSIGHT:
→ In distributed systems, you CANNOT update all producers and consumers simultaneously
→ There is ALWAYS a window where old and new versions coexist
→ Schema evolution rules define WHAT changes are safe during this window
→ Schema Registry ENFORCES these rules before any schema reaches Kafka

THE FOUR COMPATIBILITY TYPES:

  ┌─────────────────────────────────────────────────────────────────┐
  │                    COMPATIBILITY MATRIX                         │
  ├───────────────────┬────────────────────────────────────────────┤
  │                   │ Can NEW consumers   Can OLD consumers      │
  │ Compatibility     │ read OLD messages?  read NEW messages?     │
  ├───────────────────┼────────────────────────────────────────────┤
  │ BACKWARD          │ ✅ YES               ❌ NOT GUARANTEED       │
  │                   │                                            │
  │ FORWARD           │ ❌ NOT GUARANTEED     ✅ YES                 │
  │                   │                                            │
  │ FULL              │ ✅ YES               ✅ YES                  │
  │                   │                                            │
  │ NONE              │ ❌ NO GUARANTEES     ❌ NO GUARANTEES        │
  └───────────────────┴────────────────────────────────────────────┘

  TRANSITIVE VARIANTS:
  → BACKWARD_TRANSITIVE:  New consumer reads ALL previous versions
  → FORWARD_TRANSITIVE:   ALL previous consumers read new version
  → FULL_TRANSITIVE:      ALL versions interoperable with ALL others


WHICH ONE SOLVES OUR PROBLEM?

  Deployment reality at FinanceFlow:
  → Consumers deployed BEFORE producers (safety first)
  → Consumers must handle old messages during upgrade
  → After producer upgrade, old consumers still running for hours
  → Need: new consumers read old + old consumers read new
  → Answer: FULL COMPATIBILITY (or at minimum BACKWARD)

  Recommended default: BACKWARD
  → Most common in industry
  → New consumers can always read old messages
  → Allows: add optional fields with defaults
  → Forbids: remove fields, rename fields, change types

AVRO SCHEMA EVOLUTION — HOW IT WORKS:

  Avro uses READER and WRITER schemas:
  → WRITER schema: schema used when message was PRODUCED
  → READER schema: schema used when message is CONSUMED
  → Avro resolution rules handle differences between the two

  Example:
    Writer (v1): {name, amount}
    Reader (v2): {name, amount, currency}

    Avro resolution:
    → 'name': present in both → use writer value
    → 'amount': present in both → use writer value
    → 'currency': in reader but not writer → use DEFAULT value
    → Result: reader gets {name: "Alice", amount: 150, currency: "USD"}

    This works ONLY if 'currency' has a default value in v2 schema.
    Without default → Avro cannot resolve → deserialization FAILS.

SCHEMA REGISTRY WIRE FORMAT (from Project 16):
  ┌──────────────────────────────────────────────────┐
  │ Byte 0    │ Bytes 1-4     │ Bytes 5+             │
  │ Magic (0) │ Schema ID (4B)│ Avro-encoded payload │
  └──────────────────────────────────────────────────┘

  → Every Kafka message carries its schema ID
  → Consumer fetches schema by ID from Schema Registry
  → Uses that schema as WRITER schema
  → Uses its own compiled schema as READER schema
  → Avro resolves differences using evolution rules

✅ WHAT SHOULD DO

1. SET COMPATIBILITY LEVEL BEFORE FIRST SCHEMA REGISTRATION
   → Default is BACKWARD — good starting point
   → Set at global level AND per-subject level
   → Change only with careful analysis

2. ALWAYS ADD DEFAULT VALUES TO NEW FIELDS
   → New field without default = breaking change in BACKWARD mode
   → Default values allow old messages to be read by new consumers
   → Use meaningful defaults: currency default "USD", not empty string

3. USE ALIASES FOR FIELD RENAMES
   → Instead of renaming 'amount' to 'total_amount':
   → Add 'total_amount' with default, keep 'amount' deprecated
   → Or use Avro aliases: {"name": "total_amount", "aliases": ["amount"]}

4. TEST COMPATIBILITY BEFORE DEPLOYING
   → Schema Registry has a compatibility check API
   → POST /compatibility/subjects/{subject}/versions/latest
   → Returns "is_compatible": true/false
   → Integrate into CI/CD pipeline — fail build on incompatible schema

5. VERSION YOUR SCHEMAS IN SOURCE CONTROL
   → Store .avsc files in Git alongside application code
   → Code review for schema changes
   → Schema changes are as important as API changes

6. USE TRANSITIVE COMPATIBILITY FOR LONG-LIVED TOPICS
   → BACKWARD checks only against latest version
   → BACKWARD_TRANSITIVE checks against ALL previous versions
   → For topics with months of retained data → TRANSITIVE is safer
   → Prevents: v3 compatible with v2, but NOT with v1 (still in topic)

❌ WHAT SHOULD NOT DO

1. NEVER SET COMPATIBILITY TO NONE IN PRODUCTION
   → NONE = no checks = any change accepted
   → This is exactly what caused FinanceFlow's outage
   → Only use NONE for development/testing environments

2. NEVER REMOVE FIELDS WITHOUT A MIGRATION STRATEGY
   → Removing a field breaks BACKWARD compatibility
   → Old messages still in topic have that field
   → New consumers compiled without it → crash on old messages
   → Strategy: deprecate → add replacement → wait → remove after retention expires

3. NEVER CHANGE FIELD TYPES DIRECTLY
   → int → string is ALWAYS a breaking change
   → int → long is safe (type promotion in Avro)
   → Strategy: add new field with new type, deprecate old field

4. NEVER DEPLOY PRODUCER BEFORE CONSUMERS (in BACKWARD mode)
   → BACKWARD: new consumer reads old messages ✅
   → But old consumer reading new messages ❌
   → Deploy consumers first → then producers
   → In FULL compatibility: order doesn't matter (both directions safe)

5. NEVER SKIP COMPATIBILITY CHECKS IN CI/CD
   → "It's just adding one field, it'll be fine" → famous last words
   → Automate: every schema change → compatibility API → pass/fail

6. NEVER USE SCHEMA REGISTRY AS RUNTIME DEPENDENCY WITHOUT CACHING
   → Schema Registry down → producers/consumers fail
   → Avro serializer caches schemas locally
   → Configure schema.registry.url with multiple instances
   → Use client-side caching: auto.register.schemas=false in production

DECISION TREE:

  Q: Can you deploy consumers before producers?
  ├─ YES → BACKWARD (most flexible for schema additions)
  ├─ NO, producers deploy first → FORWARD
  └─ UNCERTAIN / mixed teams → FULL (safest)

  Q: Do consumers replay old data (weeks/months)?
  ├─ YES → Use TRANSITIVE variant
  └─ NO, only recent data → Non-transitive is fine

  Q: Is this a financial/regulatory topic?
  ├─ YES → FULL_TRANSITIVE (maximum safety)
  └─ NO → BACKWARD is sufficient

🚫 WHEN NOT TO USE

1. WHEN USING JSON WITHOUT SCHEMA REGISTRY
   → JSON has no built-in schema evolution mechanism
   → Consumers must handle missing/extra fields manually
   → Schema Registry can enforce JSON Schema compatibility
   → But Avro provides MUCH better evolution support

2. WHEN TOPIC IS SHORT-LIVED (HOURS)
   → Temporary processing topics with no long-term consumers
   → Schema evolution overhead not worth it
   → Just use the schema and delete the topic when done

3. WHEN SCHEMA CHANGES ARE TRULY BREAKING
   → Sometimes business requirements demand incompatible changes
   → Don't fight the compatibility system — use migration strategy
   → Create NEW topic with new schema
   → Migrate consumers from old topic to new topic
   → Dual-write during transition period
   → Delete old topic after migration complete

4. WHEN ALL PRODUCERS AND CONSUMERS DEPLOY TOGETHER
   → Monolithic application producing and consuming same topic
   → No version mismatch possible
   → Schema evolution still good practice but not critical

5. WHEN PERFORMANCE IS EXTREME PRIORITY
   → Schema Registry lookup adds ~1ms per new schema (cached after)
   → For sub-millisecond requirements: pre-compile schemas
   → Use auto.register.schemas=false + manual registration

SCHEMA REGISTRY INTERNALS:

  → Schemas stored in internal Kafka topic: _schemas

  → Multiple instances for HA — all read same topic

COMPATIBILITY CHECK ALGORITHM:

  When new schema is registered, Schema Registry:

  BACKWARD check:
    → Can schema N (new reader) read data written by schema N-1 (old writer)?
    → Avro: resolve(reader=new, writer=old) — does it succeed?
    → If new schema has field X without default that old schema doesn't have
      → Resolution fails → INCOMPATIBLE

  FORWARD check:
    → Can schema N-1 (old reader) read data written by schema N (new writer)?
    → Avro: resolve(reader=old, writer=new) — does it succeed?
    → If new schema removes field X that old reader expects
      → Resolution fails → INCOMPATIBLE

  FULL check:
    → Both BACKWARD and FORWARD must pass

  TRANSITIVE check:
    → Instead of checking only against version N-1
    → Check against ALL previous versions (1, 2, ..., N-1)
    → Much stricter — prevents "version hop" incompatibilities


AVRO SCHEMA RESOLUTION RULES:

  When reader schema ≠ writer schema, Avro applies these rules:

  1. FIELD IN BOTH:
     → Use writer's value, cast to reader's type if compatible
     → int → long ✅ (promotion)
     → string → int ❌ (incompatible)

  2. FIELD IN WRITER BUT NOT READER:
     → Reader ignores it (field is skipped)
     → This is why FORWARD works for adding fields

  3. FIELD IN READER BUT NOT WRITER:
     → Use reader's default value
     → If no default → FAIL (this is the critical rule)
     → This is why BACKWARD requires defaults on new fields

  4. FIELD TYPE PROMOTIONS (always safe):
     → int → long
     → int → float
     → int → double
     → long → float
     → long → double
     → float → double
     → string → bytes (and vice versa)

  5. UNION TYPE RULES:
     → Writer union must be subset of reader union
     → ["null", "string"] writer → ["null", "string", "int"] reader ✅
     → ["null", "string", "int"] writer → ["null", "string"] reader ❌

  6. ENUM RULES:
     → BACKWARD: reader can have FEWER enum values (subset)
     → FORWARD: reader can have MORE enum values (superset)
     → FULL: enum values must be identical

      {"name": "currency", "type": "string", "default": "USD"}  ← HAS DEFAULT
    ]
  }

  → Old message (v1): {id:"P1", amount:150.0, customer_id:"C1"}
  → New consumer (v2 reader): needs {id, amount, customer_id, currency}
  → Avro resolution: id✅, amount✅, customer_id✅, currency→default "USD" ✅
  → Result: {id:"P1", amount:150.0, customer_id:"C1", currency:"USD"}

  Version 3 ATTEMPT (Remove amount — BACKWARD INCOMPATIBLE):
  {
    "type": "record",
    "name": "Payment",
    "fields": [
      {"name": "id", "type": "string"},
      {"name": "total", "type": "double"},           ← RENAMED from 'amount'
      {"name": "customer_id", "type": "string"},
      {"name": "currency", "type": "string", "default": "USD"}
    ]
  }

  Resolution scenario:
  → Old message (v2): {id:"P1", amount:150.0, customer_id:"C1", currency:"EUR"}
  → New consumer (v3 reader): needs {id, total, customer_id, currency}
  → Avro resolution: id✅, total→NOT in writer, NO default → ❌ FAIL
  → Schema Registry REJECTS v3 registration

  Version 3 CORRECT (Add alias — BACKWARD COMPATIBLE):
  {
    "type": "record",
    "name": "Payment",
    "fields": [
      {"name": "id", "type": "string"},
      {"name": "total", "type": "double", "aliases": ["amount"]},  ← ALIAS
      {"name": "customer_id", "type": "string"},
      {"name": "currency", "type": "string", "default": "USD"}
    ]
  }

  Resolution with alias:
  → Old message (v2): has 'amount' field
  → New reader (v3): looks for 'total', also accepts 'amount' via alias
  → Avro resolution: maps writer's 'amount' to reader's 'total' ✅

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: ADDING FIELD WITHOUT DEFAULT
  Symptom: Schema Registry rejects registration
  Error: "Schema being registered is incompatible with an earlier schema"
  Fix: Add default value to every new field
  Prevention: CI/CD compatibility check before merge

PITFALL 2: TRANSITIVE INCOMPATIBILITY
  Scenario:
    v1: {name, amount}
    v2: {name, amount, currency} with default ← compatible with v1
    v3: {name, total, currency} ← compatible with v2 (has alias)
    BUT: v3 is NOT compatible with v1 (v1 has 'amount', v3 reader has 'total')
  Symptom: Works fine with BACKWARD (checks only v2 → v3)
           Fails with BACKWARD_TRANSITIVE (checks v1 → v3 too)
  Fix: Use BACKWARD_TRANSITIVE for topics with long retention
  Prevention: Always check against ALL versions, not just latest

PITFALL 3: ENUM VALUE ADDITION IN BACKWARD MODE
  Scenario:
    v1 enum: ["PENDING", "CONFIRMED", "SHIPPED"]
    v2 enum: ["PENDING", "CONFIRMED", "SHIPPED", "CANCELLED"]
  Symptom: Schema Registry REJECTS v2 in BACKWARD mode
  Reason: Old reader (v1) encounters "CANCELLED" → doesn't know it → fails
  Fix: Use FORWARD mode for enums, OR use string type instead of enum
  Prevention: Prefer string over enum for evolving value sets

PITFALL 4: OPTIONAL FIELD CONFUSION (UNION TYPES)
  Scenario: Adding optional field
    WRONG:  {"name": "notes", "type": "string", "default": ""}
    RIGHT:  {"name": "notes", "type": ["null", "string"], "default": null}
  Reason: Avro "optional" = union with null, not string with empty default
  Symptom: Unexpected behavior when field is truly absent vs empty
  Prevention: Use ["null", "type"] pattern for optional fields

PITFALL 5: SCHEMA CACHE STALENESS
  Scenario: Schema updated in registry, consumer has old version cached
  Symptom: Consumer uses stale schema, misinterprets new messages
  Fix: Configure schema cache TTL
  Prevention:
    → Producer: auto.register.schemas=false (use pre-registered schemas)
    → Consumer: schema cache refreshes on unknown schema ID automatically

PITFALL 6: SUBJECT NAMING MISMATCH
  Scenario: Producer registers under "payments-value"
            Consumer looks up "payment-value" (singular)
  Symptom: Consumer can't find schema → deserialization fails
  Fix: Standardize subject naming convention
  Prevention: Use TopicNameStrategy (default) — derived from topic name automatically

📈 SCALABILITY CONSIDERATIONS

SCHEMA REGISTRY SCALING:
  → Schema Registry is stateless — reads from _schemas topic
  → Multiple instances behind load balancer for HA
  → Only ONE instance is "primary" (handles writes)
  → Others are "secondary" (handle reads)
  → On primary failure, secondary promoted automatically

SCHEMA COUNT:
  → Typical enterprise: 500–5,000 schemas
  → Schema Registry handles 100,000+ schemas easily
  → Storage is Kafka topic — scales with Kafka

SCHEMA LOOKUP PERFORMANCE:
  → First lookup: ~5ms (network call to registry)
  → Subsequent lookups: <0.1ms (local cache)
  → Cache key: schema ID (integer)
  → Cache invalidation: on new schema ID encountered

VERSIONING MATH:
  → 200 topics × average 5 versions = 1,000 schema versions
  → Each version: ~1KB (Avro schema JSON)
  → Total storage: ~1MB — negligible
  → Compatibility check: O(fields × versions) — milliseconds

MULTI-REGION:
  → Schema Registry per region with shared _schemas topic
  → Or: MirrorMaker 2 (Project 27) replicating _schemas topic
  → Schema IDs must be globally unique across regions
  → Use Schema Registry cluster linking for multi-DC setups

import requests
import json
import sys


class SchemaRegistryManager:
    """
    Comprehensive Schema Registry management.
    Handles: registration, compatibility checks, version management,
    and evolution validation.
    """

    def __init__(self, url='http://localhost:8081'):
        self.url = url
        self.headers = {
            'Content-Type': 'application/vnd.schemaregistry.v1+json'
        }

    # ========================================================
    # HEALTH AND CONFIGURATION
    # ========================================================

    def check_health(self) -> bool:
        """Verify Schema Registry is running"""
        try:
            resp = requests.get(f"{self.url}/config")
            if resp.status_code == 200:
                config = resp.json()
                print(f"  ✅ Schema Registry is healthy")
                print(f"     Global compatibility: {config.get('compatibilityLevel', 'UNKNOWN')}")
                return True
        except requests.ConnectionError:
            print(f"  ❌ Schema Registry not reachable at {self.url}")
        return False

    def get_global_compatibility(self) -> str:
        """Get global compatibility level"""
        resp = requests.get(f"{self.url}/config")
        return resp.json().get('compatibilityLevel', 'UNKNOWN')

        """
        Set global default compatibility level.
        Options: BACKWARD, BACKWARD_TRANSITIVE, FORWARD, FORWARD_TRANSITIVE,
                 FULL, FULL_TRANSITIVE, NONE
        """
        valid_levels = [
            'BACKWARD', 'BACKWARD_TRANSITIVE',
            'FORWARD', 'FORWARD_TRANSITIVE',
            'FULL', 'FULL_TRANSITIVE',
            'NONE'
        ]
        if level not in valid_levels:
            print(f"  ❌ Invalid level: {level}")
            print(f"     Valid: {', '.join(valid_levels)}")
            return

        resp = requests.put(
            f"{self.url}/config",
            headers=self.headers,
            data=json.dumps({"compatibility": level})
        )
        if resp.status_code == 200:
            print(f"  ✅ Global compatibility set to: {level}")

    # ========================================================
    # SCHEMA REGISTRATION
    # ========================================================

    def register_schema(self, subject: str, schema: dict) -> dict:
        """
        Register a new schema version.
        Schema Registry will:
        1. Check compatibility against existing versions
        2. If compatible → assign schema ID and version
        3. If incompatible → reject with 409 Conflict
        """
        payload = {"schema": json.dumps(schema)}

        resp = requests.post(
            f"{self.url}/subjects/{subject}/versions",
            headers=self.headers,
            data=json.dumps(payload)
        )

        if resp.status_code == 200:
            result = resp.json()
            schema_id = result.get('id')
            print(f"  ✅ Schema registered: subject='{subject}', id={schema_id}")
            return {'success': True, 'id': schema_id}
        elif resp.status_code == 409:
            print(f"  ❌ INCOMPATIBLE: Schema rejected by compatibility check!")
            error = resp.json()
            print(f"     Error: {error.get('message', resp.text)}")
            return {'success': False, 'error': error}
        elif resp.status_code == 422:
            print(f"  ❌ INVALID SCHEMA: {resp.json().get('message', resp.text)}")
            return {'success': False, 'error': resp.json()}
        else:
            print(f"  ❌ Registration failed: {resp.status_code} {resp.text}")
            return {'success': False, 'error': resp.text}

    def test_compatibility(self, subject: str, schema: dict,
                           version: str = 'latest') -> dict:
        """
        Test if a schema is compatible WITHOUT registering it.
        This is for CI/CD pipelines — check before deploy.
        """
        payload = {"schema": json.dumps(schema)}

        resp = requests.post(
            f"{self.url}/compatibility/subjects/{subject}/versions/{version}",
            headers=self.headers,
            data=json.dumps(payload)
        )

        if resp.status_code == 200:
            result = resp.json()
            is_compatible = result.get('is_compatible', False)
            if is_compatible:
                print(f"  ✅ COMPATIBLE: Schema passes compatibility check against {version}")
            else:
                print(f"  ❌ INCOMPATIBLE: Schema fails compatibility check against {version}")
                messages = result.get('messages', [])
                for msg in messages:
                    print(f"     → {msg}")
            return result
        elif resp.status_code == 404:
            print(f"  ⚠️  Subject '{subject}' not found — first schema is always compatible")
            return {'is_compatible': True}
        else:
            print(f"  ❌ Compatibility check failed: {resp.status_code} {resp.text}")
            return {'is_compatible': False, 'error': resp.text}

    # ========================================================

    def diff_versions(self, subject: str, v1: int, v2: int):
        """Show differences between two schema versions"""
        schema1 = self.get_schema(subject, str(v1))
        schema2 = self.get_schema(subject, str(v2))

        if not schema1 or not schema2:
            print(f"  ❌ Could not fetch schemas for comparison")
            return

        fields1 = {f['name']: f for f in schema1['schema'].get('fields', [])}
        fields2 = {f['name']: f for f in schema2['schema'].get('fields', [])}

        added = set(fields2.keys()) - set(fields1.keys())
        removed = set(fields1.keys()) - set(fields2.keys())
        common = set(fields1.keys()) & set(fields2.keys())

        print(f"\n  📊 SCHEMA DIFF: {subject} v{v1} → v{v2}")
        print(f"  {'─' * 50}")

        if added:
            for field_name in added:
                field = fields2[field_name]
                has_default = 'default' in field
                default_val = field.get('default', 'NONE')
                print(f"  ➕ ADDED:   {field_name} (type: {field['type']}, "
                      f"default: {default_val})")

        if removed:
            for field_name in removed:
                field = fields1[field_name]
                print(f"  ➖ REMOVED: {field_name} (type: {field['type']})")

        for field_name in common:
            f1 = fields1[field_name]
            f2 = fields2[field_name]
            if f1['type'] != f2['type']:
                print(f"  🔄 CHANGED: {field_name} type: {f1['type']} → {f2['type']}")
            if f1.get('default') != f2.get('default'):
                print(f"  🔄 CHANGED: {field_name} default: "
                      f"{f1.get('default', 'NONE')} → {f2.get('default', 'NONE')}")

        if not added and not removed:
            type_changes = any(
                fields1[f]['type'] != fields2[f]['type'] for f in common
            )
            if not type_changes:
                print(f"  ℹ️  No field-level differences detected")

    # ========================================================
    # FULL REPORT
    # ========================================================

    def print_full_report(self, subject: str):
        """Print complete schema evolution report for a subject"""
        versions = self.get_versions(subject)
        compat = self.get_subject_compatibility(subject)

        print(f"\n{'=' * 60}")
        print(f"📋 SCHEMA EVOLUTION REPORT: {subject}")
        print(f"{'=' * 60}")
        print(f"  Compatibility Level: {compat}")
        print(f"  Total Versions: {len(versions)}")

        for version in versions:
            schema_info = self.get_schema(subject, str(version))
            schema = schema_info.get('schema', {})
            fields = schema.get('fields', [])

            print(f"\n  📄 Version {version} (ID: {schema_info.get('id')})")
            print(f"     Name: {schema.get('name', '?')}")
            print(f"     Fields ({len(fields)}):")
            for field in fields:
                field_type = field['type']
                default = field.get('default', 'REQUIRED')
                aliases = field.get('aliases', [])
                extras = []
                if default != 'REQUIRED':
                    extras.append(f"default={default}")
                if aliases:
                    extras.append(f"aliases={aliases}")
                extra_str = f" ({', '.join(extras)})" if extras else ""
                print(f"       → {field['name']}: {field_type}{extra_str}")

        # Show diffs between consecutive versions
        if len(versions) > 1:
            print(f"\n  📊 CHANGES BETWEEN VERSIONS:")
            for i in range(len(versions) - 1):
                self.diff_versions(subject, versions[i], versions[i + 1])

        print(f"\n{'=' * 60}")

class PaymentProducerV1:
    """
    Simulates the ORIGINAL producer running schema v1.
    This producer might still be running during upgrade window.
    Messages from this producer must be readable by v2 and v3 consumers.
    """

    def __init__(self):
        schema_registry_client = SchemaRegistryClient({
            'url': SCHEMA_REGISTRY_URL
        })

        self.serializer = AvroSerializer(
            schema_registry_client,
            SCHEMA_V1,
            payment_to_dict,
            conf={'auto.register.schemas': True}
        )

        self.producer = Producer({
            'bootstrap.servers': KAFKA_BROKER,
            'acks': 'all',
            'enable.idempotence': True
        })

        print("  🔵 PaymentProducerV1 initialized (schema: id, amount, customer_id, status)")

    def produce(self, count=5):
        """Produce v1 messages — no currency, no timestamp"""
        statuses = ['PENDING', 'CONFIRMED', 'SHIPPED']

        for i in range(count):
            payment = {
                'id': f"PAY-V1-{uuid.uuid4().hex[:8]}",
                'amount': round(random.uniform(10.0, 500.0), 2),
                'customer_id': f"CUST-{random.randint(1, 100)}",
                'status': random.choice(statuses)
            }

            self.producer.produce(
                topic=TOPIC,
                key=payment['id'].encode('utf-8'),
                value=self.serializer(
                    payment,
                    SerializationContext(TOPIC, MessageField.VALUE)
                ),
                callback=lambda err, msg: None
            )
            print(f"    📤 V1 produced: {payment['id']} | ${payment['amount']} | {payment['status']}")
            time.sleep(0.3)

        self.producer.flush()

class EvolvingPaymentConsumer:
    """
    Consumer that reads messages from any schema version.

    KEY MECHANISM:
    → Consumer has its OWN schema (reader schema)
    → Each message carries its writer schema ID
    → Avro deserializer:
      1. Reads schema ID from message bytes
      2. Fetches writer schema from Schema Registry
      3. Resolves writer schema → reader schema
      4. Fills defaults for missing fields
      5. Ignores extra fields in writer

    This is HOW schema evolution works at the wire level.
    """

    def __init__(self, reader_schema_str: str, consumer_name: str,
                 group_suffix: str = ''):
        schema_registry_client = SchemaRegistryClient({
            'url': SCHEMA_REGISTRY_URL
        })

        # The reader schema determines what THIS consumer expects
        self.deserializer = AvroDeserializer(
            schema_registry_client,
            reader_schema_str,
            dict_to_payment
        )

        self.consumer = Consumer({
            'bootstrap.servers': KAFKA_BROKER,
            'group.id': f'payment-consumer-{group_suffix}-{int(time.time())}',
            'auto.offset.reset': 'earliest',
            'enable.auto.commit': True
        })
        self.consumer.subscribe([TOPIC])

        self.consumer_name = consumer_name
        self.running = True
        self.stats = {'total': 0, 'v1': 0, 'v2': 0, 'v3': 0, 'defaults_used': 0}

    def consume(self, max_messages=30, timeout_seconds=30):
        """Consume and display messages, showing schema resolution"""

        print(f"\n  👂 {self.consumer_name} listening...")

        start = time.time()

        while self.running and self.stats['total'] < max_messages:
            if time.time() - start > timeout_seconds:
                break

            msg = self.consumer.poll(timeout=1.0)
            if msg is None:
                continue

            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    continue
                print(f"  ❌ Error: {msg.error()}")
                continue

            try:
                payment = self.deserializer(
                    msg.value(),
                    SerializationContext(TOPIC, MessageField.VALUE)
                )

                if payment is None:
                    continue

                self.stats['total'] += 1

                # Detect which version produced this message
                pay_id = payment.get('id', '?')
                if 'PAY-V1' in pay_id:
                    self.stats['v1'] += 1
                    version_tag = 'V1'
                elif 'PAY-V2' in pay_id:
                    self.stats['v2'] += 1
                    version_tag = 'V2'
                elif 'PAY-V3' in pay_id:
                    self.stats['v3'] += 1
                    version_tag = 'V3'
                else:
                    version_tag = '??'

                # Check which fields used defaults
                defaults_info = []
                currency = payment.get('currency')
                if currency == 'USD' and 'V1' in version_tag:
                    defaults_info.append('currency=USD(default)')
                    self.stats['defaults_used'] += 1

                processed_at = payment.get('processed_at')
                if processed_at is None and 'V1' in version_tag or 'V2' in version_tag:
                    defaults_info.append('processed_at=null(default)')
                    self.stats['defaults_used'] += 1

                notes = payment.get('notes')
                if notes is None and version_tag in ('V1', 'V2'):
                    defaults_info.append('notes=null(default)')

                # Display
                defaults_str = f" | DEFAULTS: {', '.join(defaults_info)}" if defaults_info else ""
                print(f"    📥 [{self.consumer_name}] {version_tag} → "
                      f"{pay_id} | ${payment.get('amount', '?')} "
                      f"{payment.get('currency', '?')} | "
                      f"{payment.get('status', '?')}"
                      f"{defaults_str}")

            except Exception as e:
                print(f"    ❌ [{self.consumer_name}] Deserialization error: {e}")
                import traceback
                traceback.print_exc()

        self.consumer.close()

    def print_stats(self):
        print(f"\n  📊 {self.consumer_name} Statistics:")
        print(f"     Total messages:    {self.stats['total']}")
        print(f"     From V1 producer:  {self.stats['v1']}")
        print(f"     From V2 producer:  {self.stats['v2']}")
        print(f"     From V3 producer:  {self.stats['v3']}")
        print(f"     Defaults applied:  {self.stats['defaults_used']}")

    PaymentProducerV2().produce(count=2)
    PaymentProducerV3().produce(count=2)

    # Start consumers in parallel threads
    consumers = [
        EvolvingPaymentConsumer(SCHEMA_V1, "V1-Reader", "parallel-v1"),
        EvolvingPaymentConsumer(SCHEMA_V2, "V2-Reader", "parallel-v2"),
        EvolvingPaymentConsumer(SCHEMA_V3, "V3-Reader", "parallel-v3"),
    ]

    print("\n  📥 STARTING 3 PARALLEL CONSUMERS...")
    print("  " + "─" * 50)

    threads = []
    for consumer in consumers:
        t = threading.Thread(
            target=consumer.consume,
            kwargs={'max_messages': 6, 'timeout_seconds': 20},
            daemon=True
        )
        t.start()
        threads.append(t)

    # Wait for all to finish
    for t in threads:
        t.join(timeout=25)

    # Print stats
    print("\n" + "─" * 50)
    for consumer in consumers:
        consumer.print_stats()

import json
import sys
import os
from schema_registry_manager import SchemaRegistryManager


class SchemaCIChecker:
    """
    Automated schema compatibility checker for CI/CD pipelines.
    
    Integration points:
    → GitHub Actions / GitLab CI / Jenkins
    → Run after code review, before merge
    → Exit code 0 = compatible (proceed)
    → Exit code 1 = incompatible (block merge)
    
    Reads .avsc files from a directory and checks each
    against its corresponding subject in Schema Registry.
    """

    def __init__(self, registry_url='http://localhost:8081'):
        self.mgr = SchemaRegistryManager(registry_url)
        self.results = []

    def check_schema_file(self, filepath: str, subject: str) -> bool:
        """
        Check a single .avsc file against its subject in Schema Registry.
        Returns True if compatible, False if not.
        """
        print(f"\n  🔍 Checking: {filepath}")
        print(f"     Subject:  {subject}")

        # Read schema file
        try:
            with open(filepath, 'r') as f:
                schema = json.load(f)
        except FileNotFoundError:
            print(f"     ❌ File not found: {filepath}")
            self.results.append({
                'file': filepath, 'subject': subject,
                'compatible': False, 'reason': 'File not found'
            })
            return False
        except json.JSONDecodeError as e:
            print(f"     ❌ Invalid JSON: {e}")
            self.results.append({
                'file': filepath, 'subject': subject,
                'compatible': False, 'reason': f'Invalid JSON: {e}'
            })
            return False

        # Check if subject exists
        versions = self.mgr.get_versions(subject)
        if not versions:
            print(f"     ℹ️  Subject '{subject}' has no existing versions")
            print(f"     ✅ First schema — automatically compatible")
            self.results.append({
                'file': filepath, 'subject': subject,
                'compatible': True, 'reason': 'First version'
            })
            return True

        # Get current compatibility level
        compat_level = self.mgr.get_subject_compatibility(subject)
        print(f"     Compatibility: {compat_level}")
        print(f"     Existing versions: {len(versions)}")

        # Test compatibility against latest
        result = self.mgr.test_compatibility(subject, schema, 'latest')
        is_compatible = result.get('is_compatible', False)

        if is_compatible:
            print(f"     ✅ COMPATIBLE with latest version")

            # Also check transitive if applicable
            if 'TRANSITIVE' in compat_level:
                trans_result = self.mgr.test_compatibility_all_versions(subject, schema)
                if not trans_result.get('all_compatible', False):
                    print(f"     ❌ TRANSITIVE CHECK FAILED")
                    is_compatible = False
        else:
            messages = result.get('messages', [])
            print(f"     ❌ INCOMPATIBLE with latest version")
            for msg in messages:
                print(f"        → {msg}")

        # Show diff
        if versions:
            latest = self.mgr.get_schema(subject, 'latest')
            latest_schema = latest.get('schema', {})
            old_fields = {f['name'] for f in latest_schema.get('fields', [])}
            new_fields = {f['name'] for f in schema.get('fields', [])}

            added = new_fields - old_fields
            removed = old_fields - new_fields

            if added:
                for field_name in added:
                    field = next(f for f in schema['fields'] if f['name'] == field_name)
                    has_default = 'default' in field
                    default_icon = "✅" if has_default else "❌"
                    print(f"     ➕ Added: {field_name} (has default: {default_icon})")

            if removed:
                for field_name in removed:
                    print(f"     ➖ Removed: {field_name}")

        reason = 'Compatible' if is_compatible else 'Incompatible'
        self.results.append({
            'file': filepath, 'subject': subject,
            'compatible': is_compatible, 'reason': reason
        })

        return is_compatible

    def check_directory(self, schema_dir: str,
                        subject_mapping: dict = None) -> bool:
        """
        Check all .avsc files in a directory.
        
        subject_mapping: dict of filename → subject name
        If not provided, derives subject from filename:
          payment.avsc → payment-value
          order.avsc → order-value
        """
        print("\n" + "=" * 60)
        print("  🏗️  CI/CD SCHEMA COMPATIBILITY CHECK")
        print(f"  Directory: {schema_dir}")
        print("=" * 60)

        if not os.path.isdir(schema_dir):
            print(f"  ❌ Directory not found: {schema_dir}")
            return False

        avsc_files = [f for f in os.listdir(schema_dir) if f.endswith('.avsc')]

        if not avsc_files:
            print(f"  ℹ️  No .avsc files found in {schema_dir}")
            return True

        all_compatible = True

        for filename in sorted(avsc_files):
            filepath = os.path.join(schema_dir, filename)

            # Derive subject name
            if subject_mapping and filename in subject_mapping:
                subject = subject_mapping[filename]
            else:
                base_name = filename.replace('.avsc', '')
                subject = f"{base_name}-value"

            if not self.check_schema_file(filepath, subject):
                all_compatible = False

        # Print summary
        self.print_summary()

        return all_compatible

    def print_summary(self):
        """Print final CI/CD check summary"""
        print("\n" + "=" * 60)
        print("  📋 COMPATIBILITY CHECK SUMMARY")
        print("=" * 60)

        passed = sum(1 for r in self.results if r['compatible'])
        failed = sum(1 for r in self.results if not r['compatible'])

        for r in self.results:
            icon = "✅" if r['compatible'] else "❌"
            print(f"  {icon} {r['file']:<30s} → {r['subject']:<20s} ({r['reason']})")

        print(f"\n  Results: {passed} passed, {failed} failed")

        if failed > 0:
            print(f"\n  🚨 SCHEMA CHECK FAILED — DO NOT DEPLOY")
            print(f"     Fix incompatible schemas before merging")
        else:
            print(f"\n  ✅ ALL SCHEMAS COMPATIBLE — SAFE TO DEPLOY")

        print("=" * 60)


def create_sample_schemas():

# PURPOSE: When you MUST make an incompatible change,

# THIS IS THE LAST RESORT — when evolution rules cannot help

from confluent_kafka import Producer, Consumer, KafkaError
from confluent_kafka.admin import AdminClient, NewTopic
import json

class Bre[REDACTED:AWS_ACCESS_KEY]:
    """
    When a schema change is truly incompatible (rename field,
    change type, restructure entirely), you cannot evolve
    the existing topic.

    STRATEGY: Topic Migration (Blue-Green for Kafka)

    Step 1: Create NEW topic with new schema
    Step 2: Deploy BRIDGE that reads old topic → transforms → writes new topic
    Step 3: Deploy NEW consumers on new topic
    Step 4: Switch producers to write to new topic
    Step 5: Wait for old topic to drain
    Step 6: Decommission old topic

    This is the Kafka equivalent of database migration.
    """

    def __init__(self, broker='localhost:9092'):
        self.broker = broker
        self.admin = AdminClient({'bootstrap.servers': broker})

    def create_new_topic(self, topic_name: str, partitions: int = 6):
        """Step 1: Create the new topic"""
        topic = NewTopic(
            topic=topic_name,
            num_partitions=partitions,
            replication_factor=1
        )

        futures = self.admin.create_topics([topic])
        for name, future in futures.items():
            try:
                future.result()
                print(f"  ✅ Created new topic: {name}")
            except Exception as e:
                if 'already exists' in str(e):
                    print(f"  ⏭️  Topic already exists: {name}")
                else:
                    print(f"  ❌ Error: {e}")

    def run_bridge(self, old_topic: str, new_topic: str,
                   transform_fn, max_messages: int = 100):
        """
        Step 2: Bridge that reads old topic, transforms, writes to new topic.
        
        transform_fn: function that converts old format to new format
        Example: transform_fn({'amount': 150}) → {'total_amount': 150.0, 'currency': 'USD'}
        """
        consumer = Consumer({
            'bootstrap.servers': self.broker,
            'group.id': f'bridge-{old_topic}-to-{new_topic}',
            'auto.offset.reset': 'earliest',
            'enable.auto.commit': True
        })
        consumer.subscribe([old_topic])

        producer = Producer({
            'bootstrap.servers': self.broker,
            'acks': 'all',
            'enable.idempotence': True
        })

        print(f"\n  🌉 BRIDGE RUNNING: {old_topic} → {new_topic}")
        print(f"     Transform function: {transform_fn.__name__}")

        count = 0
        empty_polls = 0

        while count < max_messages and empty_polls < 10:
            msg = consumer.poll(timeout=2.0)
            if msg is None:
                empty_polls += 1
                continue
            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    empty_polls += 1
                    continue
                continue

            empty_polls = 0

            try:
                key = msg.key()
                old_value = json.loads(msg.value().decode('utf-8'))

                # TRANSFORM: old format → new format
                new_value = transform_fn(old_value)

                # Write to new topic
                producer.produce(
                    topic=new_topic,
                    key=key,
                    value=json.dumps(new_value).encode('utf-8')
                )
                count += 1

                if count % 10 == 0:
                    print(f"     Migrated {count} messages...")

            except Exception as e:
                print(f"     ❌ Transform error: {e}")

        producer.flush()
        consumer.close()

        print(f"  ✅ Bridge complete: {count} messages migrated")
        return count
