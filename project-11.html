PROJECT 11: THE BANKING LEDGER — DELIVERY GUARANTEES

🔴 HYPOTHETICAL PROBLEM

You are building a core banking transaction ledger system

Every single transaction — deposit, withdrawal, transfer — must be recorded

The business has three different subsystems:
  → Marketing analytics: "We don't care if we miss a few transactions"
  → Transaction notifications: "We'd rather send a duplicate SMS than miss one"
  → Core ledger: "Every transaction must appear EXACTLY once — no duplicates, no losses"

Three different reliability requirements
Three different delivery guarantee needs
One Kafka cluster serving all three

The CTO asks:
  "Can Kafka guarantee that no message is ever lost?"
  "Can Kafka guarantee that no message is ever duplicated?"
  "What happens if the broker crashes mid-write?"
  "What happens if the producer crashes after sending but before receiving acknowledgment?"

You need to understand delivery semantics at a fundamental level
Not just configuration — the WHY behind every guarantee

🔵 HOW TO APPROACH IT

Step 1: Understand what "delivery guarantee" actually means
  → It is NOT just about the producer sending
  → It is NOT just about the consumer receiving
  → It is about the END-TO-END contract between producer, broker, and consumer

Step 2: Recognize the three guarantee levels exist because of PHYSICS
  → Networks are unreliable
  → Machines crash
  → Acknowledgments can be lost even when data is safe
  → You CANNOT have perfect reliability with zero cost

Step 3: Map each business requirement to a guarantee level
  → Marketing analytics     → At-most-once   (fast, lossy is OK)
  → Transaction notifications → At-least-once  (duplicates OK, no loss)
  → Core ledger              → Exactly-once    (no loss, no duplicates)

Step 4: Understand that guarantees are configured at BOTH sides
  → Producer side: How many brokers must acknowledge before "success"
  → Consumer side: When offsets are committed relative to processing
  → Broker side: How many replicas must have the data

Step 5: Connect back to what we already know
  → Project 2: We produced messages continuously — but did we verify they arrived?
  → Project 6: We learned offsets — but what if we commit offset before processing?

🟡 WHAT COULD BE DONE

OPTION A: Use acks=0 for everything
  → Fastest possible throughput
  → Zero reliability
  → Producer doesn't wait for any confirmation
  → Message could vanish and you'd never know
  → ONLY suitable for metrics/analytics where loss is acceptable

OPTION B: Use acks=1 for everything
  → Leader broker confirms receipt
  → Good balance of speed and safety
  → BUT: if leader crashes AFTER ack but BEFORE replication → DATA LOST
  → Suitable for most use cases where occasional loss is tolerable

OPTION C: Use acks=all for everything
  → ALL in-sync replicas must confirm
  → Slowest but safest
  → Combined with min.insync.replicas=2 → survives single broker failure
  → Required for financial/critical data

OPTION D: Use acks=all + idempotent producer
  → Everything from Option C
  → PLUS: Kafka deduplicates retries on the broker side
  → Prevents the "ack lost, producer retries, message written twice" problem
  → Foundation for exactly-once semantics

✅ WHAT SHOULD DO

1. MATCH GUARANTEE LEVEL TO BUSINESS CRITICALITY
   → Not every topic needs acks=all
   → Over-guaranteeing wastes resources
   → Under-guaranteeing loses critical data

2. ALWAYS SET retries TO A HIGH VALUE WITH acks=all
   → Network blips happen
   → Default retries in modern Kafka is already 2147483647
   → But explicitly set it to show intent

3. SET min.insync.replicas ON CRITICAL TOPICS
   → acks=all means "all ISR members"
   → If ISR shrinks to 1, acks=all = acks=1 (DANGEROUS)
   → min.insync.replicas=2 prevents writes when safety is compromised

4. USE IDEMPOTENT PRODUCER FOR CRITICAL PATHS
   → enable.idempotence=true
   → Prevents duplicates caused by producer retries
   → Zero performance overhead in practice

5. ON CONSUMER SIDE — MATCH COMMIT STRATEGY TO GUARANTEE
   → At-most-once:  commit BEFORE processing
   → At-least-once:  commit AFTER processing
   → Exactly-once:   use transactions (Project 18)

6. SET delivery.timeout.ms APPROPRIATELY
   → This is the TOTAL time a producer will try to deliver
   → Includes retries, backoff, everything
   → Must be >= request.timeout.ms + linger.ms

❌ WHAT SHOULD NOT DO

1. DO NOT USE acks=0 FOR ANY DATA YOU CARE ABOUT
   → You literally get zero confirmation
   → The broker could be down and you'd keep "sending" into the void

2. DO NOT SET acks=all WITHOUT min.insync.replicas
   → acks=all with ISR of 1 is effectively acks=1
   → You THINK you're safe but you're NOT

3. DO NOT USE acks=all WITH replication.factor=1
   → There's only one copy — "all replicas" means one broker
   → Completely defeats the purpose

4. DO NOT COMMIT OFFSETS BEFORE PROCESSING IF YOU NEED AT-LEAST-ONCE
   → Consumer crashes after commit but before processing
   → Message is "consumed" but never processed
   → This silently becomes at-most-once

5. DO NOT ASSUME "DELIVERED" MEANS "PROCESSED"
   → Producer ack = broker received it
   → It does NOT mean consumer has processed it
   → End-to-end guarantee requires BOTH sides configured correctly

6. DO NOT DISABLE RETRIES WITH acks=all
   → Transient failures will cause permanent message loss
   → retries=0 with acks=all is contradictory intent

7. DO NOT CONFUSE IDEMPOTENT PRODUCER WITH EXACTLY-ONCE
   → Idempotent producer = no duplicates on SINGLE partition from SINGLE producer session
   → Exactly-once = cross-partition, cross-topic atomic writes (needs transactions — Project 18)

  → You need to detect production failures

  → You're using acks=0 (idempotence requires acks=all internally)

Scenario: replication.factor=3, min.insync.replicas=2, acks=all

HEALTHY STATE:
  ISR = [Broker1(Leader), Broker2, Broker3]  → 3 members
  acks=all requires ALL 3 to confirm
  min.insync.replicas=2 → at least 2 must be alive to accept writes
  ✅ Writes succeed

ONE BROKER DOWN:
  ISR = [Broker1(Leader), Broker2]  → 2 members
  acks=all requires BOTH to confirm
  min.insync.replicas=2 → exactly 2 alive → threshold MET
  ✅ Writes still succeed (degraded but safe)

TWO BROKERS DOWN:
  ISR = [Broker1(Leader)]  → 1 member
  min.insync.replicas=2 → only 1 alive → threshold NOT MET
  ❌ Writes REJECTED with NotEnoughReplicasException
  → This is CORRECT behavior — better to reject than to risk loss
  → Reads still work

AT-MOST-ONCE (Consumer):
  [Poll] ──► [Commit Offset] ──► [Process] ──► [Poll]
                    ↑                  ↑
                    │                  └── If crash HERE → message lost
                    └── Offset already moved forward

  [Poll] ──► [Process] ──► [Commit Offset] ──► [Poll]
                  ↑               ↑
                  │               └── If crash HERE → reprocess same msg
                  └── Processing done, but offset not saved yet

EXACTLY-ONCE (Consumer):
  [Poll] ──► [Process + Commit ATOMICALLY] ──► [Poll]
                         ↑
                         └── Requires Kafka Transactions (Project 18)
                             OR external idempotent processing

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "I set acks=all so I'm safe"
  → Did you set min.insync.replicas? No? Then ISR can shrink to 1
  → acks=all with ISR=1 is acks=1 in disguise
  → FIX: Always pair acks=all with min.insync.replicas >= 2

PITFALL 2: "My producer got a timeout — is the message lost?"
  → NOT NECESSARILY
  → Timeout means the producer didn't get ACK within request.timeout.ms
  → The broker MAY have written it
  → Without idempotence, retrying creates duplicates
  → With idempotence, retrying is safe (broker deduplicates)

PITFALL 3: "I enabled idempotence but still see duplicates"
  → Idempotent producer deduplicates within a SINGLE producer session
  → If producer process restarts → new PID → new session
  → Previous in-flight messages may be retried with new PID = duplicate
  → FIX: Use transactional producer for cross-session guarantees (Project 18)

PITFALL 4: "enable.auto.commit=true with at-least-once"
  → Auto-commit commits on a TIMER (every 5 seconds by default)
  → It doesn't know if you've finished processing
  → Can commit BEFORE processing completes → at-most-once
  → Can commit AFTER reprocessing on restart → at-least-once
  → It's UNPREDICTABLE — not suitable for either guarantee precisely
  → FIX: Use manual commit for deterministic behavior

PITFALL 5: "I set acks=all but my latency went through the roof"
  → acks=all waits for ALL ISR replicas
  → If one replica is slow (GC pause, disk I/O), EVERYTHING waits
  → FIX: Monitor ISR health, remove slow brokers, tune replica.lag.time.max.ms

PITFALL 6: "NotEnoughReplicasException everywhere"
  → min.insync.replicas is set higher than available brokers
  → Or too many brokers are down
  → This is a SAFETY mechanism, not a bug
  → FIX: Add brokers, or temporarily lower min.insync.replicas (risky)

PITFALL 7: "Messages arriving out of order with retries"
  → Without idempotence: retries can cause reordering
  → Batch 1 fails, Batch 2 succeeds, Batch 1 retry succeeds → [2, 1]
  → FIX: enable.idempotence=true automatically sets max.in.flight.requests.per.connection=5
         with proper sequence tracking — ordering preserved

📈 SCALABILITY CONSIDERATIONS

THROUGHPUT VS GUARANTEE LEVEL:
  ┌──────────┬────────────────┬───────────────────────────────────────┐
  │ acks     │ Throughput     │ Why                                   │
  ├──────────┼────────────────┼───────────────────────────────────────┤
  │ acks=0   │ ★★★★★ Highest │ No waiting, no overhead               │
  │ acks=1   │ ★★★★☆ High    │ Wait for leader only                  │
  │ acks=all │ ★★★☆☆ Medium  │ Wait for ALL ISR replicas             │
  │ +idempot │ ★★★☆☆ Medium  │ Tiny overhead for PID/seq tracking    │
  └──────────┴────────────────┴───────────────────────────────────────┘

LATENCY VS GUARANTEE LEVEL:
  acks=0   → ~0.5ms  (network send only)
  acks=1   → ~2-5ms  (leader write + ack)
  acks=all → ~5-15ms (leader write + replication + ack)
  These are approximate — actual depends on cluster, network, disk

REPLICATION FACTOR IMPACT:
  → Higher replication = more safety but more network I/O
  → replication.factor=3 is the industry standard for production
  → replication.factor=5 for ultra-critical data (rare)
  → Every message is written N times (N = replication factor)
  → Storage cost = message_size × replication_factor

PRODUCER BATCHING WITH acks=all:
  → Batching AMORTIZES the cost of acks=all
  → Instead of waiting for ACK per message, wait per BATCH
  → Set linger.ms=5-20 and batch.size=32768-65536 (as introduced in Project 2)
  → Throughput with acks=all + good batching can approach acks=1 throughput

MIN.INSYNC.REPLICAS SCALING:
  → Formula: min.insync.replicas = replication.factor - 1
  → This tolerates 1 broker failure while maintaining safety
  → For replication.factor=3 → min.insync.replicas=2 (standard)
  → For replication.factor=5 → min.insync.replicas=3 (high availability)

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY YOUR GUARANTEE IS WORKING:

1. PRODUCER SIDE — Check acks behavior
   → Produce with acks=all
   → Kill a follower broker
   → Produce should still succeed (ISR shrinks but min.isr met)
   → Kill another follower → produce should FAIL (min.isr not met)
   → This PROVES acks=all + min.insync.replicas is working

2. CONSUMER SIDE — Verify offset commit timing
   → Produce 10 messages
   → Consumer: process 5, crash before commit
   → Restart consumer → should reprocess from last committed offset
   → If at-least-once: you see messages 1-5 again (correct)
   → If at-most-once: you see messages 6-10 only (correct, 1-5 lost)

3. IDEMPOTENCE VERIFICATION
   → Produce with idempotence enabled
   → Inject network partition after send but before ack
   → Producer retries
   → Check topic: message should appear ONCE, not twice

4. MONITORING METRICS
   → record-error-rate: should be 0 for critical topics
   → record-retry-rate: shows how often retries happen
   → request-latency-avg: shows ack wait time
   → Producer metric: acks-latency shows time spent waiting for replicas

5. DESCRIBING TOPIC CONFIG
   → kafka-topics.sh --describe shows replication factor and ISR
   → If ISR < replication.factor → you're in degraded mode
   → If ISR < min.insync.replicas → writes are blocked

from confluent_kafka.admin import AdminClient, NewTopic
import time

admin = AdminClient({
    'bootstrap.servers': 'localhost:9092,localhost:9093,localhost:9094'
})

topics_to_create = [
    # ── Marketing Analytics: Low reliability, high speed ──
    NewTopic(
        topic='banking.analytics.clickstream',
        num_partitions=3,
        replication_factor=1,       # Single copy — loss acceptable
        config={
            'min.insync.replicas': '1'
        }
    ),

    # ── Notifications: Medium reliability ──
    NewTopic(
        topic='banking.notifications.sms',
        num_partitions=3,
        replication_factor=2,       # Two copies
        config={
            'min.insync.replicas': '1'
        }
    ),

    # ── Core Ledger: Maximum reliability ──
    NewTopic(
        topic='banking.ledger.transactions',
        num_partitions=3,
        replication_factor=3,       # Three copies
        config={
            'min.insync.replicas': '2'  # At least 2 must confirm
        }
    ),
]

futures = admin.create_topics(topics_to_create)

for topic_name, future in futures.items():
    try:
        future.result()
        print(f"✅ Created topic: {topic_name}")
    except Exception as e:
        print(f"⚠️  Topic {topic_name}: {e}")

# ── Verify topic configurations ──
time.sleep(2)
metadata = admin.list_topics(timeout=10)
for topic_name in ['banking.analytics.clickstream',
                    'banking.notifications.sms',
                    'banking.ledger.transactions']:
    topic = metadata.topics.get(topic_name)
    if topic:
        partitions = topic.partitions
        first_partition = partitions[0]
        print(f"\n📋 Topic: {topic_name}")
        print(f"   Partitions: {len(partitions)}")
        print(f"   Replicas (partition 0): {len(first_partition.replicas)}")
        print(f"   ISR (partition 0): {len(first_partition.isrs)}")

    # ═══════════════════════════════════════════════
    # IDEMPOTENCE: Prevent duplicates from retries
    # ═══════════════════════════════════════════════
    'enable.idempotence': True,

    # ── When idempotence is enabled, these are auto-configured: ──
    # 'acks': 'all'                              (forced)
    # 'max.in.flight.requests.per.connection': 5  (max allowed)
    # 'retries': 2147483647                       (max retries)

    # ── Explicit safety settings ──
    'delivery.timeout.ms': 120000,    # 2 minutes total delivery timeout
    'request.timeout.ms': 30000,      # 30 seconds per request
    'retry.backoff.ms': 100,          # 100ms between retries

    # ── Moderate batching (don't sacrifice latency too much) ──
    'batch.size': 16384,
    'linger.ms': 5,
})

# ── Detailed delivery tracking ──
results = {
    'success': 0,
    'failure': 0,
    'partitions': {},
    'latencies': []
}

def delivery_callback(err, msg):
    """
    With acks=all + idempotence:
      → This fires AFTER ALL ISR replicas confirm
      → Message is GUARANTEED to be durable (unless entire cluster dies)
      → If retried due to lost ACK, broker deduplicates
    """

        results['failure'] += 1
        print(f"❌ CRITICAL: Ledger write failed: {err}")
        print(f"   Transaction: {msg.key().decode('utf-8') if msg.key() else 'N/A'}")
        # ── In production: trigger alert, write to fallback store ──
    else:
        results['success'] += 1
        partition = msg.partition()
        offset = msg.offset()
        results['partitions'].setdefault(partition, []).append(offset)

        latency = msg.latency()
        if latency is not None:
            results['latencies'].append(latency)

def at_most_once_consumer():
    """
    COMMIT FIRST, THEN PROCESS
    → If crash during processing → message lost forever
    → Consumer won't re-read it because offset already advanced
    → Use for: analytics where missing data points is acceptable
    """
    print("\n═══ AT-MOST-ONCE CONSUMER ═══")
    print("Strategy: Commit offset BEFORE processing\n")

    consumer = Consumer({
        'bootstrap.servers': 'localhost:9092,localhost:9093,localhost:9094',
        'group.id': 'at-most-once-group',
        'auto.offset.reset': 'earliest',

        # ── Auto-commit ON — commits on a timer ──
        'enable.auto.commit': True,
        'auto.commit.interval.ms': 1000,  # Commits every 1 second
    })

    consumer.subscribe(['banking.analytics.clickstream'])

    processed = 0
    try:
        while running and processed < 50:
            msg = consumer.poll(timeout=2.0)
            if msg is None:
                continue
            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    continue
                print(f"Error: {msg.error()}")
                continue

            # ── Offset is auto-committed INDEPENDENTLY of processing ──
            # ── If we crash right HERE, the message is lost ──
            # ── Because auto-commit may have already moved the offset ──

            # Simulate processing
            data = json.loads(msg.value().decode('utf-8'))
            processed += 1
            print(f"  [{processed}] Processed: user={data.get('user_id', 'N/A')} "
                  f"(partition={msg.partition()}, offset={msg.offset()})")

    finally:
        consumer.close()
        print(f"\n  Total processed: {processed}")
        print(f"  ⚠️  Some messages may have been committed but NOT processed")
        print(f"  ⚠️  Those messages are LOST — at-most-once delivery\n")


def at_least_once_consumer():
    """
    PROCESS FIRST, THEN COMMIT
    → If crash after processing but before commit → reprocess on restart
    → Consumer will re-read because offset wasn't committed
    → Use for: notifications, order processing (idempotent downstream)
    """
    print("\n═══ AT-LEAST-ONCE CONSUMER ═══")
    print("Strategy: Process THEN commit offset manually\n")

    consumer = Consumer({
        'bootstrap.servers': 'localhost:9092,localhost:9093,localhost:9094',
        'group.id': 'at-least-once-group',
        'auto.offset.reset': 'earliest',

        # ═══════════════════════════════════════════
        # CRITICAL: Disable auto-commit
        # We commit MANUALLY after successful processing
        # ═══════════════════════════════════════════
        'enable.auto.commit': False,
    })

    consumer.subscribe(['banking.notifications.sms'])

    processed = 0
    batch = []
    BATCH_SIZE = 10

    try:
        while running and processed < 50:
            msg = consumer.poll(timeout=2.0)
            if msg is None:
                # ── Commit any remaining batch ──
                if batch:
                    consumer.commit(asynchronous=False)
                    print(f"  📌 Committed batch of {len(batch)} messages")
                    batch = []
                continue
            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    continue
                print(f"Error: {msg.error()}")
                continue

            # ── STEP 1: Process the message FIRST ──
            data = json.loads(msg.value().decode('utf-8'))
            processed += 1
            batch.append(msg)
            print(f"  [{processed}] Processed: txn={data.get('txn_id', 'N/A')} "
                  f"(partition={msg.partition()}, offset={msg.offset()})")

            # ── STEP 2: Commit AFTER processing a batch ──
            if len(batch) >= BATCH_SIZE:
                consumer.commit(asynchronous=False)
                print(f"  📌 Committed batch of {len(batch)} messages")
                batch = []

                # ── If crash happens HERE → batch is committed ──
                # ── If crash happens BEFORE commit → batch reprocessed ──
                # ── Hence: AT-LEAST-ONCE ──

    finally:
        # ── Final commit for remaining messages ──
        if batch:
            consumer.commit(asynchronous=False)
            print(f"  📌 Final commit: {len(batch)} messages")
        consumer.close()
        print(f"\n  Total processed: {processed}")
        print(f"  ✅ Every message processed AT LEAST once")
        print(f"  ⚠️  On crash/restart, some may be processed AGAIN\n")


def exactly_once_simulation_consumer():
    """
    PROCESS + COMMIT ATOMICALLY (simulated)
    → True exactly-once requires Kafka Transactions (Project 18)
    → OR: Make processing idempotent (same result if done twice)
    → This shows the PATTERN — true implementation in Project 18
    """
    print("\n═══ EXACTLY-ONCE CONSUMER (Simulated via Idempotency) ═══")
    print("Strategy: Manual commit + idempotent processing\n")

    consumer = Consumer({
        'bootstrap.servers': 'localhost:9092,localhost:9093,localhost:9094',
        'group.id': 'exactly-once-group',
        'auto.offset.reset': 'earliest',
        'enable.auto.commit': False,
    })

    consumer.subscribe(['banking.ledger.transactions'])

    # ── Simulate an idempotent store (like a database with unique constraints) ──
    processed_txn_ids = set()
    processed = 0
    duplicate_count = 0

    try:
        while running and processed < 50:
            msg = consumer.poll(timeout=2.0)
            if msg is None:
                continue
            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    continue
                print(f"Error: {msg.error()}")
                continue

            data = json.loads(msg.value().decode('utf-8'))
            txn_id = data.get('txn_id', '')

            # ═══════════════════════════════════════════════
            # IDEMPOTENT CHECK: Have we seen this transaction?
            # In production: use DB unique constraint or upsert
            # ═══════════════════════════════════════════════
            if txn_id in processed_txn_ids:
                duplicate_count += 1
                print(f"  ⏭️  SKIP duplicate: txn={txn_id}")
                # ── Still commit to move offset forward ──
                consumer.commit(message=msg, asynchronous=False)
                continue

            # ── Process the transaction ──
            # ── In production: INSERT INTO ledger ... ON CONFLICT DO NOTHING ──
            processed_txn_ids.add(txn_id)
            processed += 1
            print(f"  [{processed}] Ledger entry: txn={txn_id} "
                  f"amount=${data.get('amount', 0):.2f} "
                  f"(partition={msg.partition()}, offset={msg.offset()})")

            # ── Commit AFTER successful processing ──
            consumer.commit(message=msg, asynchronous=False)

    finally:
        consumer.close()
        print(f"\n  Total processed: {processed}")
        print(f"  Duplicates skipped: {duplicate_count}")
        print(f"  ✅ Each transaction processed EXACTLY once (via idempotency)")
        print(f"  ℹ️  True exactly-once with Kafka Transactions → Project 18\n")

from confluent_kafka import Producer
import json
import time

BOOTSTRAP = 'localhost:9092,localhost:9093,localhost:9094'
NUM_MESSAGES = 2000

def benchmark_producer(acks_level, topic, label):
    """
    Produce NUM_MESSAGES with a specific acks level and measure performance.
    """
    config = {
        'bootstrap.servers': BOOTSTRAP,
        'acks': str(acks_level),
        'batch.size': 16384,
        'linger.ms': 5,
    }

    if acks_level == 'all':
        config['enable.idempotence'] = True

    producer = Producer(config)
    latencies = []
    success_count = 0
    error_count = 0

    def callback(err, msg):
        nonlocal success_count, error_count
        if err:
            error_count += 1
        else:
            success_count += 1
            lat = msg.latency()
            if lat is not None:
                latencies.append(lat)

    start = time.time()

    for i in range(NUM_MESSAGES):
        msg = json.dumps({
            'id': i,
            'data': 'x' * 100,  # 100-byte payload
            'ts': time.time()
        }).encode('utf-8')

        producer.produce(
            topic=topic,
            key=str(i % 10).encode('utf-8'),
            value=msg,
            callback=callback
        )

        if i % 500 == 0:
            producer.poll(0)

    producer.flush(timeout=60)
    elapsed = time.time() - start

    avg_lat = sum(latencies) / len(latencies) if latencies else 0
    p99_lat = sorted(latencies)[int(len(latencies) * 0.99)] if latencies else 0

    return {
        'label': label,
        'acks': acks_level,
        'messages': NUM_MESSAGES,
        'success': success_count,
        'errors': error_count,
        'elapsed_sec': round(elapsed, 3),
        'throughput_msg_sec': round(NUM_MESSAGES / elapsed),
        'avg_latency_ms': round(avg_lat * 1000, 2),
        'p99_latency_ms': round(p99_lat * 1000, 2),
    }

def try_produce(producer, topic, message_id):
    """Attempt to produce a single message and report result."""
    result = {'success': False, 'error': None}

    def callback(err, msg):
        if err:
            result['error'] = str(err)
        else:
            result['success'] = True

    try:
        producer.produce(
            topic=topic,
            key=f'test-{message_id}'.encode('utf-8'),
            value=json.dumps({
                'test_id': message_id,
                'timestamp': time.time()
            }).encode('utf-8'),
            callback=callback
        )
        producer.flush(timeout=10)
    except Exception as e:
        result['error'] = str(e)

    return result

def get_producer_config(guarantee_level, bootstrap_servers):
    """
    Returns the correct producer configuration for a given guarantee level.
    
    This encapsulates EVERYTHING learned in this project into one function.
    """
    
    configs = {
        'at-most-once': {
            'bootstrap.servers': bootstrap_servers,
            'acks': '0',
            'retries': 0,
            'batch.size': 65536,
            'linger.ms': 10,
            'compression.type': 'snappy',
            # ── CHARACTERISTICS ──
            # Speed:      ★★★★★
            # Safety:     ★☆☆☆☆
            # Use case:   Metrics, clickstream, logs where loss is OK
            # Risk:       Messages can vanish silently
        },
        
        'at-least-once': {
            'bootstrap.servers': bootstrap_servers,
            'acks': '1',
            'retries': 5,
            'retry.backoff.ms': 100,
            'batch.size': 32768,
            'linger.ms': 5,
            # ── CHARACTERISTICS ──
            # Speed:      ★★★★☆
            # Safety:     ★★★☆☆
            # Use case:   Notifications, activity feeds
            # Risk:       Leader crash before replication = loss
            #             Retries without idempotence = duplicates
        },
        
        'at-least-once-strong': {
            'bootstrap.servers': bootstrap_servers,
            'acks': 'all',
            'enable.idempotence': True,
            # idempotence auto-sets:
            #   acks=all
            #   retries=MAX_INT
            #   max.in.flight.requests.per.connection=5
            'delivery.timeout.ms': 120000,
            'request.timeout.ms': 30000,
            'batch.size': 16384,
            'linger.ms': 5,
            # ── CHARACTERISTICS ──
            # Speed:      ★★★☆☆
            # Safety:     ★★★★★ (producer side)
            # Use case:   Financial transactions, orders, audit
            # Risk:       Consumer side still needs manual commit
            #             Cross-topic atomicity needs transactions (Project 18)
        },
        
        'exactly-once': {
            'bootstrap.servers': bootstrap_servers,
            'acks': 'all',
            'enable.idempotence': True,
            'transactional.id': 'your-unique-transactional-id',
            'delivery.timeout.ms': 120000,
            'request.timeout.ms': 30000,
            'batch.size': 16384,
            'linger.ms': 5,
            # ── CHARACTERISTICS ──
            # Speed:      ★★☆☆☆
            # Safety:     ★★★★★ (end-to-end)
            # Use case:   Consume-transform-produce pipelines
            # Requires:   Kafka Transactions API (Project 18)
            # Note:       transactional.id must be UNIQUE per producer instance
        },
    }
    
    if guarantee_level not in configs:
        raise ValueError(
            f"Unknown guarantee: {guarantee_level}. "
            f"Choose from: {list(configs.keys())}"
        )
    
    return configs[guarantee_level]


def get_consumer_config(guarantee_level, bootstrap_servers, group_id):
    """
    Returns the correct consumer configuration for a given guarantee level.
    """
    
    configs = {
        'at-most-once': {
            'bootstrap.servers': bootstrap_servers,
            'group.id': group_id,
            'auto.offset.reset': 'latest',
            'enable.auto.commit': True,
            'auto.commit.interval.ms': 1000,
            # ── PROCESSING PATTERN ──
            # [Poll] → [Auto-Commit] → [Process]
            # Crash during process = message LOST
        },
        
        'at-least-once': {
            'bootstrap.servers': bootstrap_servers,
            'group.id': group_id,
            'auto.offset.reset': 'earliest',
            'enable.auto.commit': False,
            # ── PROCESSING PATTERN ──
            # [Poll] → [Process] → [Manual Commit]
            # Crash before commit = message REPROCESSED
        },
        
        'exactly-once': {
            'bootstrap.servers': bootstrap_servers,
            'group.id': group_id,
            'auto.offset.reset': 'earliest',
            'enable.auto.commit': False,
            'isolation.level': 'read_committed',
            # ── PROCESSING PATTERN ──
            # [Poll] → [Process + Commit in Transaction]
            # Requires transactional producer on write side
            # Consumer only reads COMMITTED transactional messages
            # Full implementation → Project 18
        },
    }
    
    if guarantee_level not in configs:
        raise ValueError(
            f"Unknown guarantee: {guarantee_level}. "
            f"Choose from: {list(configs.keys())}"
        )
    
    return configs[guarantee_level]

# FILE: 09_e2e_verification.py
# PURPOSE: Prove delivery guarantees work end-to-end
# PRODUCES: N messages, CONSUMES them, VERIFIES completeness

from confluent_kafka import Producer, Consumer, KafkaError
import json
import time
import uuid

BOOTSTRAP = 'localhost:9092,localhost:9093,localhost:9094'
TOPIC = 'banking.ledger.transactions'
NUM_MESSAGES = 200
UNIQUE_RUN_ID = str(uuid.uuid4())[:8]


def produce_verified_messages():
    """Produce messages with acks=all and track every confirmed delivery."""
    
    producer = Producer({
        'bootstrap.servers': BOOTSTRAP,
        'acks': 'all',
        'enable.idempotence': True,
    })
    
    confirmed_ids = set()
    failed_ids = set()
    
    def callback(err, msg):
        msg_id = msg.key().decode('utf-8')
        if err:
            failed_ids.add(msg_id)
        else:
            confirmed_ids.add(msg_id)
    
    print(f"📤 Producing {NUM_MESSAGES} messages (run={UNIQUE_RUN_ID})...")
    
    for i in range(NUM_MESSAGES):
        msg_id = f"{UNIQUE_RUN_ID}-{i:04d}"
        producer.produce(
            topic=TOPIC,
            key=msg_id.encode('utf-8'),
            value=json.dumps({
                'msg_id': msg_id,
                'run_id': UNIQUE_RUN_ID,
                'sequence': i,
                'amount': round(100 + i * 1.5, 2),
                'timestamp': time.time()
            }).encode('utf-8'),
            callback=callback
        )
        if i % 50 == 0:
            producer.poll(0)
    
    producer.flush(timeout=30)
    
    print(f"   ✅ Confirmed: {len(confirmed_ids)}")
    print(f"   ❌ Failed:    {len(failed_ids)}")
    
    return confirmed_ids


def consume_and_verify(expected_ids):
    """Consume messages and verify we get EXACTLY what was produced."""
    
    consumer = Consumer({
        'bootstrap.servers': BOOTSTRAP,
        'group.id': f'verification-{UNIQUE_RUN_ID}',
        'auto.offset.reset': 'earliest',
        'enable.auto.commit': False,
    })
    
    consumer.subscribe([TOPIC])
    
    received_ids = set()
    duplicate_count = 0
    
    print(f"\n📥 Consuming messages (looking for run={UNIQUE_RUN_ID})...")
    
    empty_polls = 0
    MAX_EMPTY_POLLS = 20
    
    while empty_polls < MAX_EMPTY_POLLS:
        msg = consumer.poll(timeout=1.0)
        
        if msg is None:
            empty_polls += 1
            continue
        
        if msg.error():
            if msg.error().code() == KafkaError._PARTITION_EOF:
                empty_polls += 1
                continue
            continue
        
        empty_polls = 0
        
        try:
            data = json.loads(msg.value().decode('utf-8'))
        except (json.JSONDecodeError, UnicodeDecodeError):
            continue
            
        if data.get('run_id') != UNIQUE_RUN_ID:
            continue
        
        msg_id = data['msg_id']
        
        if msg_id in received_ids:
            duplicate_count += 1
        else:
            received_ids.add(msg_id)
    
    consumer.close()
    
    # ── VERIFICATION ──
    print(f"\n{'='*60}")
    print(f"  END-TO-END VERIFICATION RESULTS")
    print(f"{'='*60}")
    
    missing = expected_ids - received_ids
    extra = received_ids - expected_ids
    
    print(f"  Expected messages:  {len(expected_ids)}")
    print(f"  Received unique:    {len(received_ids)}")
    print(f"  Duplicates seen:    {duplicate_count}")
    print(f"  Missing:            {len(missing)}")
    print(f"  Unexpected extra:   {len(extra)}")
    
    if len(missing) == 0 and len(extra) == 0:
        print(f"\n  ✅ VERIFICATION PASSED: Every confirmed message was consumed")
        print(f"     Producer acks=all guaranteed broker durability")
        print(f"     Consumer read every message from the committed log")
    else:
        print(f"\n  ❌ VERIFICATION FAILED:")
        if missing:
            print(f"     Missing message IDs: {sorted(list(missing))[:10]}...")
        if extra:
            print(f"     Extra message IDs: {sorted(list(extra))[:10]}...")
    
    if duplicate_count > 0:
        print(f"\n  ℹ️  Duplicates were seen at consumer level")
        print(f"     This is expected with at-least-once consumer semantics")
        print(f"     Idempotent processing (as shown in Part 5) handles this")
        print(f"     True deduplication via transactions → Project 18")
    
    return len(missing) == 0
