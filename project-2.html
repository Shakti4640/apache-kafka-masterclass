PROJECT 2: THE NEWS TICKER — PRODUCING A CONTINUOUS STREAM

🎯 HYPOTHETICAL PROBLEM

You work at "ByteWire News Agency"

→ Breaking news events happen CONTINUOUSLY throughout the day
→ A reporter files a story every few seconds
→ A news ticker display on a TV screen must show headlines IN ORDER
→ Multiple TV screens across the building show the SAME ticker

HERE IS THE PROBLEM:
→ In Project 1, the waiter sent ONE order and stopped
→ But news NEVER stops — it flows continuously
→ If a headline arrives out of order, the ticker looks broken
  Example:
    CORRECT:  "Election Results In" → "Markets React" → "New Policy Announced"
    BROKEN:   "New Policy Announced" → "Election Results In" → "Markets React"

→ If the ticker screen goes offline for 5 minutes and comes back,
  it must show ALL headlines it missed — in the CORRECT order
→ If a SECOND ticker screen is added, it must independently
  start from the beginning and catch up

YOU NEED:
→ A reporter that CONTINUOUSLY produces headlines
→ A ticker that CONTINUOUSLY consumes headlines IN ORDER
→ The ability to REPLAY all headlines from any point
→ Proof that Kafka preserves ordering

🔍 HOW TO APPROACH IT

WHAT WE ALREADY KNOW (from Project 1):
→ Producer sends messages to a topic ✅
→ Consumer reads messages from a topic ✅
→ Messages get sequential offsets ✅
→ Messages persist on disk ✅

WHAT IS NEW HERE:
→ The producer runs in an INFINITE LOOP (continuous stream)
→ The consumer runs in an INFINITE LOOP (continuous polling)
→ We need to PROVE ordering is maintained
→ We need to DEMONSTRATE replay capability
→ We need to understand WHY ordering works

APPROACH:
Step 1: Build a producer that generates headlines every 2 seconds
Step 2: Embed a SEQUENCE NUMBER in each message
Step 3: Build a consumer that validates ordering by checking sequence
Step 4: Stop the consumer, let messages pile up, restart it
Step 5: Observe that ALL missed messages arrive IN ORDER
Step 6: Start a SECOND consumer with a different group.id
Step 7: Observe it reads ALL messages independently from offset 0

🛠️ HOW CAN WE SOLVE IT

SOLUTION DESIGN:

  Reporter (Producer)                    Kafka Broker
  ┌──────────────────┐                 ┌────────────────────────────┐
  │ while True:      │                 │ Topic: breaking-news       │
  │   headline = ... │  ──produce──→   │                            │
  │   sleep(2 sec)   │                 │ [0] [1] [2] [3] [4] [5]...│
  │                  │                 │  ↑                     ↑   │
  └──────────────────┘                 │  first             latest  │
                                       └─────────────┬──────────────┘
                                                     │
                                          ┌──────────┴──────────┐
                                          │                     │
                                    ┌─────▼─────┐        ┌─────▼─────┐
                                    │ Ticker #1  │        │ Ticker #2  │
                                    │ group:     │        │ group:     │
                                    │ "ticker-a" │        │ "ticker-b" │
                                    │ offset: 5  │        │ offset: 2  │
                                    └────────────┘        └────────────┘
                                    (caught up)           (catching up)

KEY INSIGHT:
→ Each consumer group has its OWN offset pointer
→ Ticker #1 might be at offset 5 (fully caught up)
→ Ticker #2 might be at offset 2 (still catching up)
→ They do NOT interfere with each other
→ The topic log is SHARED but offset tracking is INDEPENDENT

✅ WHAT COULD BE DONE

Option A: Producer sends all headlines in one batch
          → PROBLEM: Not realistic — news arrives continuously
          → PROBLEM: No way to see real-time streaming behavior

Option B: Producer sends one message and consumer polls once
          → THIS IS WHAT PROJECT 1 DID — we need to go beyond

Option C: Producer loops continuously, consumer loops continuously ✅
          → REALISTIC: Mirrors real-world streaming systems
          → DEMONSTRATES: Ordering, replay, independent consumers
          → PROVES: Kafka's append-only log is the foundation of everything

✅ WHAT SHOULD DO

1. ALWAYS include a logical sequence number or timestamp IN your message
   → Kafka guarantees offset ordering
   → But YOUR application should also validate ordering independently
   → This is your SAFETY NET for detecting anomalies

2. ALWAYS use producer.poll(0) inside a producing loop
   → This triggers delivery callbacks without blocking
   → Without this, callbacks queue up and memory grows
   → Project 1 used flush() at the end — in a loop, poll(0) is better

3. ALWAYS use try/finally with consumer.close()
   → If the consumer crashes without close(), offsets may not commit
   → The finally block ensures clean shutdown

4. ALWAYS design messages to be SELF-DESCRIBING
   → Include enough metadata to understand the message WITHOUT context
   → A headline message should carry: sequence, timestamp, category, text

5. ALWAYS test replay by restarting the consumer with a NEW group.id
   → This proves messages are still on disk
   → This proves the append-only log works as expected

❌ WHAT SHOULD NOT DO

1. DO NOT use time.sleep() inside the consumer poll loop
   → The poll(timeout) already has a built-in wait
   → Adding sleep ON TOP of poll creates unnecessary latency
   → poll(1.0) = "wait up to 1 second for a message" — that IS your sleep

2. DO NOT assume ordering across MULTIPLE partitions
   → In this project we use 1 partition → ordering guaranteed
   → In Project 3, we will add partitions → ordering guarantee changes
   → For now: 1 partition = total order guaranteed

3. DO NOT flush() inside a tight producing loop
   → flush() BLOCKS until all messages are sent
   → In a loop, this kills throughput
   → Use poll(0) instead — non-blocking callback processing

4. DO NOT ignore the delivery callback in continuous producing
   → If the broker is down, messages silently fail without callbacks
   → The callback is your ONLY way to know if delivery succeeded

5. DO NOT create a new Producer/Consumer instance for each message
   → Create ONCE, reuse forever
   → Instance creation involves TCP handshake, metadata fetch — expensive

🔀 WHEN TO USE WHAT

┌────────────────────────────────────┬────────────────────────────────────────┐
│ USE CONTINUOUS PRODUCING WHEN      │ USE BATCH/ONE-SHOT PRODUCING WHEN      │
├────────────────────────────────────┼────────────────────────────────────────┤
│ Data arrives as an ongoing stream  │ Data arrives as a finite batch         │
│ (sensor data, logs, events, news)  │ (daily file import, one-time migration)│
├────────────────────────────────────┼────────────────────────────────────────┤
│ You need real-time processing      │ Latency does not matter                │
│ downstream                         │ (nightly ETL job)                      │
├────────────────────────────────────┼────────────────────────────────────────┤
│ Producer runs as a long-lived      │ Producer runs as a short-lived         │
│ service/daemon                     │ script/cron job                        │
└────────────────────────────────────┴────────────────────────────────────────┘

┌────────────────────────────────────┬────────────────────────────────────────┐
│ USE poll(0) IN PRODUCER LOOP       │ USE flush() IN PRODUCER                │
├────────────────────────────────────┼────────────────────────────────────────┤
│ Inside a continuous loop           │ At the END of a batch/script           │
│ (non-blocking callback trigger)    │ (blocking — ensure everything sent)    │
├────────────────────────────────────┼────────────────────────────────────────┤
│ When throughput matters            │ When completeness matters              │
│ (don't want to block the loop)    │ (must guarantee all sent before exit)  │
└────────────────────────────────────┴────────────────────────────────────────┘

🚫 WHEN NOT TO USE

→ DO NOT use continuous producing if your data source is finite
  (e.g., reading a CSV file — use batch produce + flush instead)

→ DO NOT use continuous consuming without a shutdown signal
  (always handle KeyboardInterrupt or SIGTERM for clean exit)

→ DO NOT use a single partition if ordering across
  INDEPENDENT categories is needed
  (e.g., sports news vs politics news can be parallel — Project 3)

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

═══════════════════════════════════════════════════════════════
WHY DOES KAFKA GUARANTEE ORDERING?
═══════════════════════════════════════════════════════════════

FUNDAMENTAL TRUTH:
→ A Kafka topic partition is an APPEND-ONLY LOG
→ "Append-only" means: new data is ALWAYS written at the END
→ There is NO insert, NO update, NO reorder
→ This is identical to writing in a notebook:
  Page 1, Page 2, Page 3... you never go back and insert Page 1.5

WHAT HAPPENS ON DISK:

  food-orders-0/00000000000000000000.log
  ┌──────────┬──────────┬──────────┬──────────┬──────────┐
  │ offset=0 │ offset=1 │ offset=2 │ offset=3 │ offset=4 │
  │ "news A" │ "news B" │ "news C" │ "news D" │ "news E" │
  └──────────┴──────────┴──────────┴──────────┴──────────┘
  ↑                                                      ↑
  START of file                                    END of file
  (oldest)                                         (newest)

→ The broker APPENDS at the end → offset increments
→ The consumer READS from left to right → offset 0, 1, 2, 3, 4
→ This is WHY ordering is guaranteed within a single partition


═══════════════════════════════════════════════════════════════
WHAT HAPPENS DURING CONTINUOUS PRODUCING
═══════════════════════════════════════════════════════════════

  Time    Producer Action              Broker Log State
  ─────   ─────────────────────        ──────────────────────────
  T+0s    produce("headline A")   →    [0:"headline A"]
  T+2s    produce("headline B")   →    [0:"A"][1:"headline B"]
  T+4s    produce("headline C")   →    [0:"A"][1:"B"][2:"headline C"]
  T+6s    produce("headline D")   →    [0:"A"][1:"B"][2:"C"][3:"headline D"]

  CRITICAL: Each produce() call:
  1. Serializes message to bytes
  2. Places bytes in producer's INTERNAL BUFFER (in memory)
  3. A background SENDER THREAD batches buffered messages
  4. Sender thread sends batch to broker via TCP
  5. Broker appends to partition log file
  6. Broker returns offset in response
  7. Producer's callback fires with offset confirmation

  ┌──────────────────────────────────────────────────────────┐
  │              PRODUCER INTERNAL ARCHITECTURE               │
  │                                                          │
  │  Your Code                                               │
  │     │                                                    │
  │     ▼                                                    │
  │  ┌──────────┐     ┌──────────────┐     ┌─────────────┐  │
  │  │ produce() │ ──→ │ Record       │ ──→ │ Sender      │  │
  │  │ (buffer)  │     │ Accumulator  │     │ Thread      │  │
  │  └──────────┘     │ (batches by  │     │ (TCP send)  │  │
  │                   │  topic+part)  │     └──────┬──────┘  │
  │                   └──────────────┘            │         │
  │                                               ▼         │
  │                                         To Broker       │
  │                                         via TCP         │
  │                                                          │
  │  poll(0) ──→ triggers delivery callbacks from responses  │
  └──────────────────────────────────────────────────────────┘

  WHY poll(0) MATTERS IN A LOOP:
  → The sender thread sends messages and receives responses
  → Responses sit in an internal queue
  → poll(0) processes that queue and fires YOUR callbacks
  → Without poll(0), callbacks never fire until flush()


═══════════════════════════════════════════════════════════════
WHAT HAPPENS DURING CONTINUOUS CONSUMING
═══════════════════════════════════════════════════════════════

  Consumer                              Broker
  ────────                              ──────
  poll(1.0)                        
    → FetchRequest(offset=0, max=500)  → reads log from offset 0
    ← FetchResponse([msg0, msg1])      ← returns batch of messages
    → process msg0, msg1
    → auto-commit offset=2

  poll(1.0)
    → FetchRequest(offset=2, max=500)  → reads log from offset 2
    ← FetchResponse([msg2, msg3, msg4])← returns batch
    → process msg2, msg3, msg4
    → auto-commit offset=5

  poll(1.0)
    → FetchRequest(offset=5, max=500)  → reads log from offset 5
    ← FetchResponse([])                ← no new messages
    → returns None to your code
    → your code prints "waiting..."

  CRITICAL INSIGHT:
  → The consumer ALWAYS tells the broker "give me from offset X"
  → The broker ALWAYS reads sequentially from the log file
  → This is a SEQUENTIAL READ — the fastest type of disk I/O
  → This is why Kafka can serve millions of messages per second


═══════════════════════════════════════════════════════════════
HOW REPLAY WORKS (Consumer restart / new consumer group)
═══════════════════════════════════════════════════════════════

  SCENARIO: Consumer "ticker-a" read up to offset 5, then crashed

  Kafka's internal __consumer_offsets topic stores:
  ┌────────────────────────────────────────────┐
  │ group="ticker-a", topic="breaking-news",   │
  │ partition=0, committed_offset=5            │
  └────────────────────────────────────────────┘

  CASE A: Consumer "ticker-a" restarts
  → Kafka says: "your last committed offset was 5"
  → Consumer resumes from offset 5
  → NO REPLAY — picks up where it left off

  CASE B: NEW consumer group "ticker-b" starts
  → Kafka says: "I have no record of group ticker-b"
  → auto.offset.reset = 'earliest'
  → Consumer starts from offset 0
  → FULL REPLAY — reads every message from the beginning

  CASE C: NEW consumer group with auto.offset.reset = 'latest'
  → Kafka says: "I have no record, and you said latest"
  → Consumer starts from the CURRENT END of the log
  → NO HISTORY — only sees new messages going forward

  THIS IS THE POWER OF THE APPEND-ONLY LOG:
  → Data is NEVER deleted after consumption
  → Any consumer can read from ANY offset at ANY time
  → Replay is a FIRST-CLASS feature, not a hack

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Producer loop runs but callbacks never fire"
├── CAUSE: Never calling poll(0) inside the loop
├── WHY: Callbacks are queued internally — poll(0) processes them
├── SYMPTOM: No "✅ delivered" messages printed during the loop
└── FIX: Add producer.poll(0) after each produce() call

PITFALL 2: "Consumer seems stuck — not receiving messages"
├── CAUSE A: Consumer started AFTER producer finished with auto.offset.reset='latest'
│   └── FIX: Use auto.offset.reset='earliest' for replay
├── CAUSE B: Consumer is in the SAME group.id as a previous run
│   └── WHY: Previous run already committed offset past all messages
│   └── FIX: Use a new group.id OR reset offsets (Project 8 CLI tools)
└── CAUSE C: Subscribed to wrong topic name (typo)
    └── FIX: Verify with kafka-topics --list

PITFALL 3: "Messages arrive but sequence numbers are wrong"
├── CAUSE: You are using multiple partitions
├── WHY: Ordering is only guaranteed WITHIN a single partition
├── SYMPTOM: seq 1, seq 3, seq 2, seq 5, seq 4
└── FIX: For this project, use exactly 1 partition (Project 3 explains multi-partition)

PITFALL 4: "Producer throws BufferError in a fast loop"
├── CAUSE: Producing faster than the sender thread can send
├── WHY: Internal buffer (default 32MB) is full
├── SYMPTOM: "Local: Queue full" error
└── FIX: Add producer.poll(0) to let sender thread drain
         OR increase buffer.memory (Project 39 performance tuning)

PITFALL 5: "Consumer reads duplicate messages after restart"
├── CAUSE: Auto-commit had not fired before crash
├── WHY: Default auto-commit interval is 5 seconds
│         If consumer crashes between commits, it re-reads from last committed
├── THIS IS EXPECTED BEHAVIOR — not a bug
└── DEEP DIVE: Project 6 (Offsets) and Project 11 (Delivery Guarantees)

📈 SCALABILITY CONSIDERATIONS

CURRENT STATE (1 producer, 1 partition, 1 consumer):

  Throughput ceiling:
  → Producer: ~50,000–100,000 messages/sec (single partition, no batching tuning)
  → Consumer: ~100,000–200,000 messages/sec (sequential read is fast)
  → Bottleneck: Single partition = single log file = single disk thread

WHAT IF NEWS ARRIVES FASTER?
  → Producer can buffer and batch (linger.ms — Project 39)
  → But single partition has a hard ceiling

WHAT IF ONE CONSUMER CANNOT KEEP UP?
  → With 1 partition, you CANNOT add a second consumer in the same group
  → 1 partition = maximum 1 consumer per group
  → To parallelize, you need MORE partitions → Project 3

WHAT IF THE BROKER DISK FILLS UP?
  → Messages accumulate based on retention.ms (default 7 days)
  → Old messages are deleted by Kafka's log cleaner
  → Increase disk OR reduce retention for high-volume topics

FOR NOW: This project proves the PATTERN works
         Scaling the pattern is the job of Projects 3, 5, 9, and 39

🔍 INTERPRETABILITY CONSIDERATIONS

HOW DO YOU VERIFY THE STREAM IS WORKING?

1. SEQUENCE NUMBERS IN MESSAGES
   → Embed seq=1, seq=2, seq=3... in your message payload
   → Consumer checks: current_seq == previous_seq + 1
   → If not → you have an ordering violation (should not happen with 1 partition)

2. OFFSET MONITORING
   → Each message has a Kafka-assigned offset
   → Offsets should be strictly incrementing: 0, 1, 2, 3...
   → If you see gaps → a message was skipped (investigate!)

3. TIMESTAMP TRACKING
   → Kafka attaches a timestamp to every message
   → CreateTime: when the producer created the message
   → LogAppendTime: when the broker wrote it to disk
   → Compare these to detect producer-to-broker latency

4. LAG MONITORING
   → "Lag" = latest offset in topic MINUS consumer's committed offset
   → Lag = 0 means consumer is fully caught up
   → Lag > 0 means consumer is behind
   → Lag growing means consumer is falling behind
   → This is THE most important metric in Kafka (Project 28)

  Example:
  Topic latest offset: 1000
  Consumer committed:   950
  Lag:                   50  (consumer is 50 messages behind)

THE REPORTER — Continuous Producer

# ═══════════════════════════════════════════════════════════
# file: reporter_producer.py
# THE REPORTER — Continuously files breaking news headlines
# ═══════════════════════════════════════════════════════════

from confluent_kafka import Producer
import json
import time
import random

# ─────────────────────────────────────────────────────────
# CONFIGURATION
# Same as Project 1 — bootstrap.servers points to broker
# ─────────────────────────────────────────────────────────

config = {
    'bootstrap.servers': 'localhost:9092',
}

# ─────────────────────────────────────────────────────────
# DELIVERY CALLBACK (same pattern as Project 1)
# But now we track SUCCESS and FAILURE counts
# ─────────────────────────────────────────────────────────

stats = {'delivered': 0, 'failed': 0}

def delivery_callback(err, msg):
    if err is not None:
        stats['failed'] += 1
        print(f'❌ DELIVERY FAILED [seq unknown]: {err}')
    else:
        stats['delivered'] += 1
        # Decode the message to show the sequence number
        payload = json.loads(msg.value().decode('utf-8'))
        print(
            f'✅ Delivered | '
            f'seq={payload["sequence"]:>4} | '
            f'offset={msg.offset():>4} | '
            f'{payload["headline"][:50]}'
        )

# ─────────────────────────────────────────────────────────
# SIMULATED NEWS HEADLINES
# ─────────────────────────────────────────────────────────

categories = ['POLITICS', 'SPORTS', 'TECH', 'BUSINESS', 'WEATHER']

headlines_pool = [
    "Breaking: Major policy reform announced by government",
    "Championship finals set for this weekend",
    "New AI model breaks performance records",
    "Stock market hits all-time high amid tech rally",
    "Severe storm warning issued for coastal regions",
    "International trade agreement signed by 12 nations",
    "Olympic committee reveals host city for 2036 games",
    "Quantum computing breakthrough at research lab",
    "Central bank announces interest rate decision",
    "Heatwave expected to last through next week",
    "Emergency summit called over diplomatic tensions",
    "World cup qualifier results shock the nation",
    "Major cybersecurity vulnerability discovered",
    "Startup raises record-breaking funding round",
    "Flooding displaces thousands in river delta region",
]

# ─────────────────────────────────────────────────────────
# CREATE PRODUCER
# ─────────────────────────────────────────────────────────

producer = Producer(config)
topic_name = 'breaking-news'

print("📰 ByteWire News Agency — Reporter Station")
print(f"   Producing to topic: {topic_name}")
print(f"   Press Ctrl+C to stop\n")
print("=" * 70)

# ─────────────────────────────────────────────────────────
# CONTINUOUS PRODUCING LOOP
# ─────────────────────────────────────────────────────────

sequence = 0

try:
    while True:
        sequence += 1

        # Build the news message with rich metadata
        message = {
            "sequence": sequence,
            "category": random.choice(categories),
            "headline": random.choice(headlines_pool),
            "reporter": f"reporter-{random.randint(1, 5)}",
            "priority": random.choice(["HIGH", "MEDIUM", "LOW"]),
            "produced_at": time.time(),
            "produced_at_human": time.strftime('%Y-%m-%d %H:%M:%S'),
        }

        # ─────────────────────────────────────────────────
        # PRODUCE — puts message in internal buffer
        # ─────────────────────────────────────────────────
        producer.produce(
            topic=topic_name,
            value=json.dumps(message).encode('utf-8'),
            callback=delivery_callback
        )

        # ─────────────────────────────────────────────────
        # poll(0) — NON-BLOCKING callback processing
        # ─────────────────────────────────────────────────
        # THIS IS CRITICAL IN A LOOP
        # Without this, callbacks queue up forever
        # poll(0) means: "process any pending callbacks, don't wait"
        producer.poll(0)

        # Simulate real-world pacing: one headline every 2 seconds
        time.sleep(2)

except KeyboardInterrupt:
    print("\n\n🛑 Reporter signing off...")

finally:
    # ─────────────────────────────────────────────────────
    # FLUSH — ensure all buffered messages are sent
    # ─────────────────────────────────────────────────────
    # This is the FINAL flush — blocks until complete
    # In the loop we used poll(0), here we use flush()
    remaining = producer.flush(timeout=10)

    print(f"\n📊 Session Stats:")
    print(f"   Total produced : {sequence}")
    print(f"   Delivered      : {stats['delivered']}")
    print(f"   Failed         : {stats['failed']}")
    print(f"   Still in buffer: {remaining}")
    print("📰 Reporter station closed.")

THE TICKER — Continuous Consumer with Ordering Validation

# ═══════════════════════════════════════════════════════════
# file: ticker_consumer.py
# THE NEWS TICKER — Continuously displays headlines in order
# Validates that ordering is preserved
# ═══════════════════════════════════════════════════════════

from confluent_kafka import Consumer, KafkaError
import json
import time

# ─────────────────────────────────────────────────────────
# CONFIGURATION
# group.id and auto.offset.reset explained in Project 1
# ─────────────────────────────────────────────────────────

config = {
    'bootstrap.servers': 'localhost:9092',
    'group.id': 'ticker-display-v1',
    'auto.offset.reset': 'earliest',
}

# ─────────────────────────────────────────────────────────
# CREATE CONSUMER AND SUBSCRIBE
# ─────────────────────────────────────────────────────────

consumer = Consumer(config)
consumer.subscribe(['breaking-news'])

print("📺 ByteWire News Ticker — Display Screen #1")
print(f"   Consumer Group: {config['group.id']}")
print(f"   Reading from  : breaking-news")
print(f"   Press Ctrl+C to stop\n")
print("=" * 70)

# ─────────────────────────────────────────────────────────
# ORDERING VALIDATION STATE
# ─────────────────────────────────────────────────────────

last_sequence = 0
last_offset = -1
messages_received = 0
ordering_violations = 0

# ─────────────────────────────────────────────────────────
# CONTINUOUS CONSUMPTION LOOP
# ─────────────────────────────────────────────────────────

try:
    while True:
        # ─────────────────────────────────────────────────
        # poll(1.0) — wait up to 1 second for a message
        # NO additional sleep() needed — poll IS the wait
        # ─────────────────────────────────────────────────
        msg = consumer.poll(timeout=1.0)

        # No message within timeout
        if msg is None:
            continue

        # Error handling
        if msg.error():
            if msg.error().code() == KafkaError._PARTITION_EOF:
                # Reached end of available messages — not an error
                print(f"   📭 Caught up! Waiting for new headlines...")
            else:
                print(f"   ❌ Error: {msg.error()}")
            continue

        # ─────────────────────────────────────────────────
        # VALID MESSAGE — Deserialize and display
        # ─────────────────────────────────────────────────
        messages_received += 1
        payload = json.loads(msg.value().decode('utf-8'))

        current_sequence = payload['sequence']
        current_offset = msg.offset()

        # ─────────────────────────────────────────────────
        # ORDERING VALIDATION
        # ─────────────────────────────────────────────────
        # Check 1: Kafka offset must be strictly incrementing
        offset_ok = current_offset == last_offset + 1 or last_offset == -1

        # Check 2: Our sequence number must be strictly incrementing
        sequence_ok = current_sequence == last_sequence + 1 or last_sequence == 0

        order_status = "✅"
        if not offset_ok or not sequence_ok:
            order_status = "⚠️ ORDER VIOLATION"
            ordering_violations += 1

        # ─────────────────────────────────────────────────
        # DISPLAY THE TICKER
        # ─────────────────────────────────────────────────
        print(f"\n🔔 [{payload['category']:>8}] [{payload['priority']:>6}] "
              f"seq={current_sequence:>4} | offset={current_offset:>4} | "
              f"{order_status}")
        print(f"   📰 {payload['headline']}")
        print(f"   👤 {payload['reporter']} | "
              f"🕐 {payload['produced_at_human']}")

        # Calculate consumption lag (time between produce and consume)
        produce_time = payload['produced_at']
        consume_time = time.time()
        lag_seconds = consume_time - produce_time
        print(f"   ⏱️  Consumption lag: {lag_seconds:.2f}s")

        # Update tracking state
        last_sequence = current_sequence
        last_offset = current_offset

except KeyboardInterrupt:
    print("\n\n🛑 Ticker display shutting down...")

finally:
    consumer.close()
    print(f"\n📊 Ticker Session Stats:")
    print(f"   Messages received   : {messages_received}")
    print(f"   Ordering violations : {ordering_violations}")
    print(f"   Last sequence       : {last_sequence}")
    print(f"   Last offset         : {last_offset}")

    if ordering_violations == 0 and messages_received > 0:
        print(f"\n   ✅ PERFECT ORDERING CONFIRMED across {messages_received} messages!")
    elif ordering_violations > 0:
        print(f"\n   ⚠️  {ordering_violations} ordering violations detected!")

    print("📺 Ticker display closed.")

THE REPLAY PROOF — Second Independent Consumer

# ═══════════════════════════════════════════════════════════
# file: ticker_replay.py
# A SECOND ticker screen that proves REPLAY from offset 0
# Uses a DIFFERENT group.id → gets its own independent offset
# ═══════════════════════════════════════════════════════════

from confluent_kafka import Consumer, KafkaError
import json

# ─────────────────────────────────────────────────────────
# DIFFERENT group.id = independent offset tracking
# ─────────────────────────────────────────────────────────

config = {
    'bootstrap.servers': 'localhost:9092',
    'group.id': 'ticker-replay-screen',     # <-- DIFFERENT from ticker_consumer.py
    'auto.offset.reset': 'earliest',         # Start from the very beginning
}

consumer = Consumer(config)
consumer.subscribe(['breaking-news'])

print("📺 ByteWire News Ticker — REPLAY SCREEN")
print("   This screen reads ALL messages from the beginning")
print("   Proving that Kafka's append-only log retains everything\n")
print("=" * 70)

messages = []

try:
    empty_count = 0
    while empty_count < 5:
        msg = consumer.poll(timeout=1.0)

        if msg is None:
            empty_count += 1
            continue

        if msg.error():
            if msg.error().code() == KafkaError._PARTITION_EOF:
                print("   📭 Reached end of log")
                break
            continue

        empty_count = 0
        payload = json.loads(msg.value().decode('utf-8'))
        messages.append({
            'offset': msg.offset(),
            'sequence': payload['sequence'],
            'headline': payload['headline'][:40],
        })

except KeyboardInterrupt:
    pass

finally:
    consumer.close()

    # ─────────────────────────────────────────────────────
    # DISPLAY REPLAY SUMMARY
    # ─────────────────────────────────────────────────────
    print(f"\n📋 REPLAY SUMMARY — {len(messages)} messages recovered\n")
    print(f"{'Offset':>6} | {'Seq':>4} | Headline")
    print("-" * 60)

    for m in messages:
        print(f"{m['offset']:>6} | {m['sequence']:>4} | {m['headline']}")

    # ─────────────────────────────────────────────────────
    # VERIFY ORDERING IN REPLAYED MESSAGES
    # ─────────────────────────────────────────────────────
    if len(messages) > 1:
        offsets_ordered = all(
            messages[i]['offset'] < messages[i+1]['offset']
            for i in range(len(messages) - 1)
        )
        sequences_ordered = all(
            messages[i]['sequence'] < messages[i+1]['sequence']
            for i in range(len(messages) - 1)
        )

        print(f"\n✅ Offsets strictly ordered   : {offsets_ordered}")
        print(f"✅ Sequences strictly ordered : {sequences_ordered}")

        if offsets_ordered and sequences_ordered:
            print("\n🎉 REPLAY PROOF: All messages recovered in perfect order!")
            print("   The append-only log preserved every single message.")
        else:
            print("\n⚠️  Ordering issue detected — investigate!")
