PROJECT 23: THE DATABASE SYNC PIPELINE — JDBC SOURCE CONNECTOR

🗄️ HYPOTHETICAL PROBLEM

A retail chain operates 500 stores with a centralized PostgreSQL database

SCENARIO: Real-Time Database-to-Kafka Synchronization
→ Store operations system writes to PostgreSQL tables:
    → "products" table: 500,000 products with prices, inventory
    → "orders" table: 2 million new orders per day
    → "customers" table: 5 million customers with profile updates
→ Multiple downstream systems need this data in REAL-TIME:
    → Kafka Streams analytics pipeline (Projects 19-22)
    → Search index (Elasticsearch)
    → Data warehouse (Snowflake)
    → Mobile app push notifications
    → Fraud detection system

THE CRISIS (Week 1 — Manual Producer Approach):

  Developer writes a Python script to poll PostgreSQL:

  while True:
      cursor.execute("SELECT * FROM orders WHERE id > %s", last_id)
      for row in cursor.fetchall():
          producer.produce("orders", json.dumps(row))
          last_id = row['id']
      time.sleep(5)

  Problems:
  → OFFSET TRACKING: Where to store last_id? File? Database?
    → If script crashes between reading and storing last_id → DUPLICATES
    → If stored in file → not atomic with Kafka produce
  
  → UPDATE DETECTION: How to detect CHANGED rows?
    → "SELECT * FROM orders WHERE id > last_id" finds NEW rows only
    → Product price changed from $299 to $249 → NOT detected
    → Customer updated email → NOT detected
    → Need: "SELECT * WHERE updated_at > last_timestamp"
    → But what if both id AND updated_at change? Need BOTH modes
  
  → SCHEMA CHANGES: Table adds new column
    → Script hardcodes column names → breaks
    → No automatic schema detection
  
  → SCALABILITY: Single-threaded polling
    → 500,000 products + 2M orders + 5M customers
    → One SELECT per table every 5 seconds
    → Database overwhelmed during peak hours
    → No parallelism — one table at a time
  
  → FAULT TOLERANCE: Script crashes
    → No automatic restart
    → No progress checkpoint
    → Manual intervention required
    → Data gap until script restarts

  → CONNECTION MANAGEMENT:
    → Connection pooling? Timeout handling? Retry logic?
    → SSL/TLS to database? Credential rotation?
    → All manual, all error-prone

THE CRISIS (Week 2 — Multiple Scripts):
  → Three scripts: one per table
  → Each has its own bugs, its own offset tracking
  → Adding new table = writing new script
  → 15 scripts running → ops nightmare
  → No centralized monitoring, no health checks

THE QUESTION:
  → Is there a DECLARATIVE way to sync databases to Kafka?
  → Where adding a new table = adding a JSON config?
  → With automatic offset tracking, fault tolerance, scalability?
  → No code — just configuration?

THE ANSWER: KAFKA CONNECT + JDBC SOURCE CONNECTOR

🔍 HOW TO APPROACH IT

Step 1: Recall Kafka Connect from Project 15
         → Project 15 introduced Kafka Connect with FileStreamSource/Sink
         → Standalone mode — single worker process
         → This project upgrades to:
           → JDBC Source Connector (production-grade)
           → Distributed mode (multi-worker, fault-tolerant)
           → Multiple change detection modes

Step 2: Understand Kafka Connect architecture
         
         KAFKA CONNECT is a FRAMEWORK with:
         → Workers: JVM processes that run connectors
         → Connectors: Plugins that define HOW to move data
         → Tasks: Parallel units of work within a connector
         → Converters: Serialize/deserialize data (JSON, Avro, Protobuf)
         
         Two deployment modes:
         → Standalone: Single worker, config from files (Project 15)
         → Distributed: Multiple workers, config via REST API, fault-tolerant

Step 3: Understand JDBC Source Connector specifically
         
         → Polls database tables at configurable intervals
         → Detects new/changed rows using three modes:
           → Incrementing: auto-increment column (new rows only)
           → Timestamp: updated_at column (changed rows only)
           → Timestamp+Incrementing: both columns (new AND changed rows)
         → Automatically tracks offsets (last seen id/timestamp)
         → Creates Kafka topics automatically (one per table)
         → Handles schema detection and evolution

Step 4: Understand the three detection modes
         
         MODE 1 — INCREMENTING:
         → Uses auto-increment primary key (e.g., id SERIAL)
         → Detects NEW rows only (id > last_seen_id)
         → Cannot detect UPDATES (same id, changed data)
         → Best for: append-only tables (orders, events, logs)
         
         MODE 2 — TIMESTAMP:
         → Uses timestamp column (e.g., updated_at TIMESTAMP)
         → Detects UPDATED rows (updated_at > last_seen_timestamp)
         → Can MISS rows if multiple rows have same timestamp
         → Best for: tables with reliable update timestamps
         
         MODE 3 — TIMESTAMP+INCREMENTING (RECOMMENDED):
         → Uses BOTH id AND updated_at columns
         → Detects NEW rows (by incrementing id)
         → Detects UPDATED rows (by timestamp)
         → Handles same-timestamp rows via incrementing tiebreaker
         → Best for: most production use cases

Step 5: Plan the database schema and connector configuration
         → Three tables → three connector instances (or one with table whitelist)
         → Each table → one Kafka topic
         → Configure polling interval, batch size, detection mode per table

✅ WHAT COULD BE DONE

Option A: Manual Polling Scripts (Current crisis)
  → Custom Python/Java scripts polling database
  → PROBLEM: Offset tracking, fault tolerance, scalability all manual
  → PROBLEM: Each new table = new script
  → PROBLEM: No centralized management

Option B: Database Triggers + Custom Producer
  → PostgreSQL triggers fire on INSERT/UPDATE
  → Trigger calls external producer via pg_notify or HTTP
  → PROBLEM: Triggers add latency to database writes
  → PROBLEM: Trigger failures can block database transactions
  → PROBLEM: Complex to maintain across schema changes

Option C: Change Data Capture (Debezium)
  → Reads PostgreSQL WAL (Write-Ahead Log) directly
  → Captures ALL changes including DELETEs
  → ZERO polling overhead on database
  → PROBLEM: More complex setup (logical replication, WAL configuration)
  → NOTE: This is the BEST solution for many cases → Project 37
  → For this project: JDBC connector is simpler to understand first

Option D: Kafka Connect JDBC Source Connector ← THIS PROJECT
  → Declarative JSON configuration
  → Automatic offset tracking in Kafka internal topics
  → Distributed mode for fault tolerance
  → Multiple detection modes (incrementing, timestamp, both)
  → Handles schema detection automatically
  → Scales with tasks
  → No custom code needed

✅ WHAT SHOULD DO

1. USE DISTRIBUTED MODE IN PRODUCTION
   → Standalone mode (Project 15) = single point of failure
   → Distributed mode = multiple workers, automatic failover
   → Workers form a consumer group → tasks redistributed on failure
   → Config stored in Kafka topics (not files) → survives restarts
   → REST API for connector management

2. USE TIMESTAMP+INCREMENTING MODE FOR MOST TABLES
   → Catches both NEW rows (incrementing) and UPDATED rows (timestamp)
   → Handles same-timestamp collisions via incrementing tiebreaker
   → Requires: auto-increment primary key + updated_at timestamp column
   → If table lacks updated_at → use incrementing mode (new rows only)

3. SET APPROPRIATE POLL INTERVAL
   → poll.interval.ms controls how often connector queries database
   → Too low (100ms): hammers database with constant queries
   → Too high (60000ms): 1-minute delay before changes appear in Kafka
   → Sweet spot: 1000-5000ms for most use cases
   → Balance between freshness and database load

4. USE TOPIC PREFIX FOR NAMESPACE ISOLATION
   → topic.prefix = "db." → tables become "db.products", "db.orders"
   → Prevents collision with other Kafka topics
   → Makes it clear which topics come from database sync
   → Connects to multi-tenant naming in Project 42

5. CONFIGURE PROPER BATCH SIZE
   → batch.max.rows controls maximum rows per poll
   → Too small (10): many small queries, high overhead
   → Too large (100000): single query locks table, memory pressure
   → Default 100 is good for most cases
   → Increase for high-volume tables with infrequent polls

6. SET UP SCHEMA HANDLING
   → value.converter = JsonConverter for simple JSON output
   → value.converter = AvroConverter for schema-governed output (Project 16)
   → Include schemas.enable=true for self-describing messages
   → Schema changes in database auto-detected on next poll

7. MONITOR CONNECTOR HEALTH VIA REST API
   → GET /connectors/{name}/status — check running state
   → GET /connectors/{name}/tasks/{id}/status — check task state
   → Set up alerting on FAILED status
   → This feeds into monitoring (Project 28)

❌ WHAT SHOULD NOT DO

1. DO NOT USE STANDALONE MODE IN PRODUCTION
   → Single worker = single point of failure
   → No automatic task redistribution on crash
   → Config in files = not version-controlled easily
   → Use standalone only for development/testing

2. DO NOT POLL TOO FREQUENTLY ON LARGE TABLES
   → poll.interval.ms=100 on a 5M row table
   → Generates: SELECT * FROM customers WHERE updated_at > ? every 100ms
   → Database CPU spikes, connection pool exhausted
   → Other applications sharing the database suffer
   → Start with 5000ms, tune based on monitoring

3. DO NOT USE INCREMENTING MODE FOR TABLES WITH UPDATES
   → Incrementing only detects NEW rows (id > last_seen)
   → Product price change: same id, different data → MISSED
   → Customer email update: same id → MISSED
   → Use timestamp+incrementing for tables with updates

4. DO NOT FORGET THE updated_at COLUMN IN DATABASE SCHEMA
   → Timestamp mode requires a timestamp column
   → Column must be updated on EVERY change (not just inserts)
   → PostgreSQL: use trigger or DEFAULT NOW() on update
   → Missing column → connector config fails or misses updates

5. DO NOT RUN JDBC CONNECTOR FOR HIGH-FREQUENCY CHANGE TABLES
   → JDBC connector POLLS — introduces inherent latency
   → For tables with 1000+ changes per second → use Debezium (Project 37)
   → Debezium reads WAL directly → sub-second latency
   → JDBC connector is best for: moderate change rates (< 100/sec)

6. DO NOT IGNORE CONNECTOR OFFSET TOPIC CONFIGURATION
   → Distributed mode stores offsets in: connect-offsets topic
   → If this topic is deleted → connector restarts from beginning
   → Configure proper replication for this topic
   → Never delete connect-offsets, connect-configs, connect-status topics

7. DO NOT CREATE TOO MANY TASKS FOR SMALL TABLES
   → tasks.max=10 for a table with 1000 rows → wasteful
   → Each task = separate database connection
   → More tasks than needed = connection pool pressure
   → Rule: tasks.max = number of tables being polled

🚫 WHEN NOT TO USE

1. WHEN DELETE DETECTION IS REQUIRED
   → JDBC Source Connector CANNOT detect deleted rows
   → DELETE FROM products WHERE id = 42 → invisible to JDBC connector
   → Product still exists in Kafka topic
   → Use Debezium CDC (Project 37) for delete detection

2. WHEN SUB-SECOND LATENCY IS REQUIRED
   → JDBC polls at configurable intervals (minimum ~100ms practical)
   → Even at 100ms: effective latency = poll_interval + query_time
   → For < 100ms latency: use Debezium WAL-based CDC

3. WHEN DATABASE CANNOT HANDLE POLLING LOAD
   → Each poll = SELECT query on the table
   → Large tables with frequent polls = significant database load
   → If database is already at capacity: use Debezium (reads WAL, not table)

4. WHEN TABLES LACK INCREMENTING OR TIMESTAMP COLUMNS
   → No auto-increment PK and no updated_at column
   → Only option: bulk mode (full table dump every poll)
   → Bulk mode on large tables = extremely wasteful
   → Better: add the columns or use Debezium

MECHANISM 1: DISTRIBUTED MODE ARCHITECTURE
═══════════════════════════════════════════════════════════════════

  Distributed Kafka Connect uses THREE internal Kafka topics:

  1. connect-configs (compacted)
     → Stores connector configurations as JSON
     → Key: connector name
     → Value: full connector config JSON
     → Replicated across all workers

  2. connect-offsets (compacted)
     → Stores connector progress (offsets)
     → Key: connector + partition identifier
     → Value: last processed offset (id/timestamp)
     → This is HOW connector resumes after restart

  3. connect-status (compacted)
     → Stores connector and task status
     → Key: connector/task identifier
     → Value: status JSON (RUNNING, PAUSED, FAILED)

  Worker coordination:
  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
  │  Worker 1    │  │  Worker 2    │  │  Worker 3    │
  │              │  │              │  │              │
  │ ┌──────────┐│  │ ┌──────────┐│  │              │
  │ │ Task 0   ││  │ │ Task 1   ││  │ (standby)   │
  │ │(products)││  │ │(orders)  ││  │              │
  │ └──────────┘│  │ └──────────┘│  │              │
  │ ┌──────────┐│  │              │  │              │
  │ │ Task 2   ││  │              │  │              │
  │ │(customers││  │              │  │              │
  │ └──────────┘│  │              │  │              │
  └──────────────┘  └──────────────┘  └──────────────┘

  → Workers form a consumer group (group.id in connect config)
  → Tasks distributed across workers
  → Worker 2 crashes → Tasks 1 redistributed to Workers 1 or 3
  → Connector resumes from last committed offset → no data loss

MECHANISM 2: POLLING AND OFFSET TRACKING
═══════════════════════════════════════════════════════════════════

  INCREMENTING MODE:
  ┌──────────────────────────────────────────────────────────┐
  │ Poll 1: SELECT * FROM orders                             │
  │         WHERE id > 0                                     │
  │         ORDER BY id ASC                                  │
  │         LIMIT 100                                        │
  │                                                          │
  │ Results: id=1, id=2, ... id=100                          │
  │ Offset stored: {incrementing: 100}                       │
  │                                                          │
  │ Poll 2: SELECT * FROM orders                             │
  │         WHERE id > 100                                   │
  │         ORDER BY id ASC                                  │
  │         LIMIT 100                                        │
  │                                                          │
  │ Results: id=101, id=102, ... id=200                      │
  │ Offset stored: {incrementing: 200}                       │
  └──────────────────────────────────────────────────────────┘

  TIMESTAMP+INCREMENTING MODE:
  ┌──────────────────────────────────────────────────────────┐
  │ Poll 1: SELECT * FROM products                           │
  │         WHERE updated_at > '1970-01-01'                  │
  │         AND NOT (updated_at = '1970-01-01' AND id <= 0)  │
  │         ORDER BY updated_at ASC, id ASC                  │
  │         LIMIT 100                                        │
  │                                                          │
  │ Results: rows ordered by (updated_at, id)                │
  │ Offset stored: {                                         │
  │   timestamp: 1699876543000,                              │
  │   incrementing: 42                                       │
  │ }                                                        │
  │                                                          │
  │ Poll 2: SELECT * FROM products                           │
  │         WHERE updated_at > '2023-11-13 10:15:43'         │
  │         AND NOT (updated_at = '2023-11-13 10:15:43'      │
  │                  AND id <= 42)                            │
  │         ORDER BY updated_at ASC, id ASC                  │
  │         LIMIT 100                                        │
  │                                                          │
  │ The AND NOT clause handles same-timestamp tiebreaking:   │
  │ → If multiple rows have same updated_at                  │
  │ → Only return rows with id > last_seen_id                │
  │ → Prevents missing rows with identical timestamps        │
  └──────────────────────────────────────────────────────────┘

MECHANISM 4: TASK ASSIGNMENT AND TABLE DISTRIBUTION
═══════════════════════════════════════════════════════════════════

  With table.whitelist = "products,orders,customers"
  And tasks.max = 3:

  → Connector creates 3 tasks
  → Each task assigned one or more tables
  → Distribution: round-robin by table name

  Task 0: polls "customers" table
  Task 1: polls "orders" table  
  Task 2: polls "products" table

  Each task:
  → Has its own JDBC connection
  → Runs its own poll loop
  → Tracks its own offset
  → Can run on any worker

  With tasks.max = 1:
  → Single task polls ALL three tables sequentially
  → Less parallelism but fewer database connections

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: MISSING updated_at COLUMN
─────────────────────────────────────
  Symptom: Connector config fails with "No timestamp column found"
  Root Cause: Table doesn't have the column specified in timestamp.column.name
  Fix: Add column: ALTER TABLE products ADD COLUMN updated_at TIMESTAMP DEFAULT NOW();
       Create trigger to auto-update on changes

PITFALL 2: TIMESTAMP PRECISION MISMATCH
─────────────────────────────────────
  Symptom: Connector misses some updates or reprocesses old rows
  Root Cause: Database timestamp has microsecond precision but connector
              stores millisecond offset → rounds down → re-reads
  Fix: Ensure timestamp column has consistent precision
       Use timestamp+incrementing to handle tiebreakers

PITFALL 3: CONNECTOR FALLS BEHIND
─────────────────────────────────────
  Symptom: Kafka topics contain stale data, hours behind database
  Root Cause: poll.interval.ms too high or batch.max.rows too low
  Fix: Decrease poll interval, increase batch size
       Add more tasks for parallel polling
       Consider Debezium for high-throughput tables

PITFALL 4: DATABASE CONNECTION EXHAUSTION
─────────────────────────────────────
  Symptom: "too many connections" errors from PostgreSQL
  Root Cause: Too many tasks × multiple connectors = too many connections
  Fix: Reduce tasks.max per connector
       Increase PostgreSQL max_connections
       Use connection pooler (PgBouncer)

PITFALL 5: TOPIC AUTO-CREATION NAMING ISSUES
─────────────────────────────────────
  Symptom: Topics created with unexpected names
  Root Cause: Default topic naming = topic.prefix + table_name
              If prefix includes dots or special characters → issues
  Fix: Set explicit topic.prefix with clean naming
       Or use transforms to rename topics (Project 24 SMTs)

PITFALL 6: SCHEMA EVOLUTION BREAKING CONSUMERS
─────────────────────────────────────
  Symptom: Downstream consumers fail after table column added/removed
  Root Cause: Schema change detected → new schema in messages
              Consumer using old schema → deserialization failure
  Fix: Use Avro + Schema Registry (Project 16) for governed evolution
       Or use JSON with ignoreUnknown = true
       Schema evolution strategies → Project 38

📈 SCALABILITY CONSIDERATIONS

POLLING LOAD ON DATABASE
─────────────────────────

  Each poll = one SELECT query per task per table
  
  ┌─────────────────────────────────┬────────────────────────┐
  │ Configuration                   │ Queries per minute     │
  ├─────────────────────────────────┼────────────────────────┤
  │ 3 tables, poll=5000ms, tasks=3 │ 3 × (60/5) = 36       │
  │ 3 tables, poll=1000ms, tasks=3 │ 3 × (60/1) = 180      │
  │ 10 tables, poll=1000ms, tasks=5│ 10 × 60 = 600         │
  │ 10 tables, poll=100ms, tasks=10│ 10 × 600 = 6,000 ⚠️   │
  └─────────────────────────────────┴────────────────────────┘

  Rule: queries_per_minute = num_tables × (60000 / poll_interval_ms)

KAFKA TOPIC THROUGHPUT
─────────────────────────

CREATE TABLE IF NOT EXISTS products (
    id SERIAL PRIMARY KEY,                          -- incrementing column
    name VARCHAR(255) NOT NULL,
    category VARCHAR(100) NOT NULL,
    price DECIMAL(10, 2) NOT NULL,
    in_stock BOOLEAN DEFAULT true,
    inventory_count INTEGER DEFAULT 0,
    description TEXT,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()              -- timestamp column
);

-- Auto-update updated_at on any change
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER products_updated_at
    BEFORE UPDATE ON products
    FOR EACH ROW
    EXECUTE FUNCTION update_updated_at();

-- ────────────────────────────────────────────────
-- TABLE 2: orders
-- Mode: incrementing (append-only, new orders only)
-- Orders are never updated — only inserted
-- ────────────────────────────────────────────────

def wait_for_connect():
    """Wait for Kafka Connect REST API to be available"""
    print('  Waiting for Kafka Connect REST API...')

        except requests.ConnectionError:
            pass
        time.sleep(2)
        print(f'    Attempt {attempt + 1}/30...')
    print('  ❌ Kafka Connect not available')
    return False


def register_connector(name, config):
    """Register a connector via REST API"""
    print(f'\n  Registering connector: {name}')

    # Check if connector already exists
    response = requests.get(f'{CONNECT_REST_URL}/connectors/{name}')
    if response.status_code == 200:
        print(f'    ⚠️  Connector {name} already exists — updating...')
        response = requests.put(
            f'{CONNECT_REST_URL}/connectors/{name}/config',
            headers={'Content-Type': 'application/json'},
            data=json.dumps(config),
        )
    else:
        response = requests.post(
            f'{CONNECT_REST_URL}/connectors',
            headers={'Content-Type': 'application/json'},
            data=json.dumps({'name': name, 'config': config}),
        )

    if response.status_code in (200, 201):
        print(f'    ✅ Connector {name} registered successfully')
        return True
    else:
        print(f'    ❌ Failed: {response.status_code} — {response.text}')
        return False


def check_connector_status(name):
    """Check connector status"""
    response = requests.get(f'{CONNECT_REST_URL}/connectors/{name}/status')
    if response.status_code == 200:
        status = response.json()
        connector_state = status['connector']['state']
        tasks = status.get('tasks', [])
        task_states = [t['state'] for t in tasks]
        print(f'    Connector: {connector_state}')
        for i, task in enumerate(tasks):
            print(f'    Task {i}: {task["state"]}')
            if task['state'] == 'FAILED':
                print(f'      Error: {task.get("trace", "unknown")[:200]}')
        return connector_state
    return 'UNKNOWN'

def get_db_counts():

def check_topics_exist():

def consume_and_verify():
    """Consume from all DB topics and analyze records"""
    print('\n📋 CONSUMING FROM DATABASE TOPICS:')
    print('-' * 70)

    config = {
        'bootstrap.servers': BOOTSTRAP_SERVERS,
        'group.id': f'sync-verifier-{int(time.time())}',
        'auto.offset.reset': 'earliest',
        'enable.auto.commit': False,
    }

    consumer = Consumer(config)
    consumer.subscribe(DB_TOPICS)

    # Track records per topic
    topic_records = defaultdict(list)
    topic_ids = defaultdict(set)
    total_records = 0

    start_time = time.time()
    timeout_seconds = 20
    empty_polls = 0
    max_empty = 10

    while (time.time() - start_time) < timeout_seconds:
        msg = consumer.poll(timeout=1.0)

        if msg is None:
            empty_polls += 1
            if empty_polls >= max_empty and total_records > 0:
                break
            continue

        empty_polls = 0

        if msg.error():
            if msg.error().code() == KafkaError._PARTITION_EOF:
                continue
            print(f'  ❌ Error: {msg.error()}')
            continue

        topic = msg.topic()

        # ─── Parse record ───
        try:
            value = json.loads(msg.value().decode('utf-8'))
        except (json.JSONDecodeError, AttributeError):
            value = {'_raw': msg.value().decode('utf-8', errors='replace')}

        # Extract payload if schema+payload format
        if 'payload' in value and 'schema' in value:
            payload = value['payload']
            schema = value['schema']
        else:
            payload = value
            schema = None

        # Track record
        topic_records[topic].append(payload)
        total_records += 1

        # Track unique IDs
        record_id = payload.get('id')
        if record_id is not None:
            topic_ids[topic].add(record_id)

        # Print first few records per topic
        if len(topic_records[topic]) <= 3:
            # Compact display
            if topic == 'db.products':
                name = payload.get('name', '?')
                price = payload.get('price', '?')
                category = payload.get('category', '?')
                print(f'  📦 {topic} id={record_id} "{name}" ${price} cat={category}')
            elif topic == 'db.orders':
                amount = payload.get('total_amount', '?')
                status = payload.get('status', '?')
                print(f'  📋 {topic} id={record_id} amount=${amount} status={status}')
            elif topic == 'db.customers':
                email = payload.get('email', '?')
                tier = payload.get('loyalty_tier', '?')
                print(f'  👤 {topic} id={record_id} email={email} tier={tier}')

        if len(topic_records[topic]) == 4:
            print(f'  ... (more {topic} records)')

    consumer.close()

    return topic_records, topic_ids
