PROJECT 18: THE FUND TRANSFER SYSTEM — KAFKA TRANSACTIONS

🏦 HYPOTHETICAL PROBLEM

The same national bank from Project 17 now faces a NEW crisis

SCENARIO: Inter-Account Fund Transfer System
→ Customer requests: "Transfer $5,000 from Savings (Account A) to Checking (Account B)"
→ System must produce TWO events:
    → DEBIT event to topic "account-debits" (partition based on Account A)
    → CREDIT event to topic "account-credits" (partition based on Account B)

THE CRISIS (Week 1 — Using Idempotent Producer from Project 17):
→ Transfer service produces DEBIT to "account-debits" → SUCCESS
→ Transfer service produces CREDIT to "account-credits" → BROKER CRASHES mid-write
→ Customer's savings debited $5,000 ✓
→ Customer's checking NEVER credited $5,000 ✗
→ $5,000 VANISHED from the customer's perspective
→ 47 transfers affected in one hour = $230,000 in limbo

THE CRISIS (Week 2 — Retry Logic Added):
→ Transfer service detects credit failure → retries BOTH debit and credit
→ Idempotent producer prevents duplicate WITHIN same session (Project 17)
→ BUT service crashed between debit and credit
→ On restart: NEW PID assigned (Project 17 explained why)
→ Retry produces SECOND debit with new PID → broker treats as NEW
→ Customer debited TWICE ($10,000) but credited ONCE ($5,000)
→ $5,000 overcharged

THE CRISIS (Week 3 — Consume-Process-Produce Pipeline):
→ Fraud detection service reads from "account-debits"
→ Processes the event → produces alert to "fraud-alerts"
→ Commits consumer offset for "account-debits"
→ CRASH between produce and offset commit
→ On restart: re-reads same debit → produces DUPLICATE alert
→ Customer account FROZEN twice → support calls spike 400%

ROOT CAUSE ANALYSIS:
→ Idempotent producer (Project 17) solved SINGLE-write deduplication
→ It CANNOT solve:
    Problem 1: Atomic MULTI-topic writes (debit + credit together)
    Problem 2: Cross-session identity (PID changes on restart)
    Problem 3: Atomic consume-process-produce (read + write + commit as one unit)
→ ALL THREE require KAFKA TRANSACTIONS

🔍 HOW TO APPROACH IT

Step 1: Identify the ATOMICITY requirement
         → Multiple writes must either ALL succeed or ALL fail
         → Partial writes = data corruption
         → This is the classic "all-or-nothing" problem from database ACID

Step 2: Recognize that idempotence (Project 17) is NECESSARY but NOT SUFFICIENT
         → Idempotence = per-partition, per-session deduplication
         → Transactions = multi-partition, cross-session atomic commits
         → Transactions BUILD ON idempotence — they don't replace it
         → enable.idempotence is AUTO-ENABLED when using transactions

Step 3: Understand the THREE capabilities Kafka Transactions provide
         
         CAPABILITY 1: STABLE PRODUCER IDENTITY
         → transactional.id = permanent badge number
         → Broker maps transactional.id → PID
         → Restart gets SAME PID → broker deduplicates across sessions
         → Solves Project 17's "PID lost on restart" problem

         CAPABILITY 2: ATOMIC MULTI-PARTITION WRITES
         → beginTransaction() → produce to topic A → produce to topic B → commitTransaction()
         → Either BOTH writes visible or NEITHER visible
         → Consumers with isolation.level=read_committed see only committed data

         CAPABILITY 3: ATOMIC CONSUME-PROCESS-PRODUCE
         → Read from input topic
         → Process
         → Write to output topic
         → Commit input offsets
         → ALL wrapped in ONE transaction
         → Crash at ANY point → entire transaction rolled back

Step 4: Learn the Transaction Coordinator
         → Special broker component managing transaction state
         → Uses internal topic: __transaction_state
         → Similar to how consumer groups use __consumer_offsets (Project 6)
         → Coordinates two-phase commit protocol

Step 5: Understand consumer isolation levels
         → read_uncommitted (default): sees ALL messages including uncommitted
         → read_committed: sees ONLY messages from committed transactions
         → Critical for downstream correctness

🛠️ HOW CAN WE SOLVE IT

SOLUTION: Kafka Transactional Producer + Committed Consumer

✅ WHAT COULD BE DONE

Option A: Application-Level Two-Phase Commit
  → Maintain a "transfer status" database table
  → Write debit → update status → write credit → update status
  → On failure: check status table and compensate
  → PROBLEM: Distributed state management nightmare
  → PROBLEM: Race conditions between Kafka and database
  → PROBLEM: Status table becomes single point of failure

Option B: Saga Pattern with Compensating Events
  → Write debit event
  → If credit fails → write "reverse debit" compensating event
  → Eventually consistent
  → PROBLEM: Complex to implement correctly
  → PROBLEM: Temporary inconsistency visible to consumers
  → PROBLEM: Compensating logic can itself fail
  → NOTE: Saga IS appropriate for cross-SERVICE transactions (Project 36)
          But NOT needed for within-Kafka atomic writes

Option C: Database + Kafka Outbox Pattern
  → Write both events to database in one DB transaction
  → Use CDC (Debezium) to stream from database to Kafka
  → Atomicity guaranteed by database
  → PROBLEM: Adds database as dependency
  → PROBLEM: Latency increase from DB → CDC → Kafka path
  → NOTE: Valid pattern explored in Project 37

Option D: Kafka Transactions ← CORRECT SOLUTION FOR THIS PROBLEM
  → Native Kafka protocol-level atomicity
  → No external systems required
  → Sub-millisecond overhead
  → Works for produce-only AND consume-process-produce patterns
  → Built on top of idempotence (Project 17)

✅ WHAT SHOULD DO

1. ASSIGN A STABLE AND UNIQUE transactional.id
   → This is the producer's PERMANENT identity
   → Must be unique per producer INSTANCE
   → Must be STABLE across restarts of the SAME instance
   → Pattern: "{application-name}-{instance-id}"
   → Example: "transfer-service-instance-01"
   → DO NOT use random UUIDs — defeats the purpose

2. CALL initTransactions() EXACTLY ONCE AT STARTUP
   → Registers producer with Transaction Coordinator
   → Fences any previous zombie instance with same transactional.id
   → Must be called BEFORE any beginTransaction()
   → Blocking call — waits for coordinator response

3. WRAP ALL RELATED WRITES IN beginTransaction()...commitTransaction()
   → Every produce() between begin and commit is part of the transaction
   → Include send_offsets_to_transaction() for consume-process-produce
   → Keep transactions SHORT — don't include external I/O inside

4. HANDLE abortTransaction() IN CATCH BLOCKS
   → If ANY produce fails within a transaction → abort
   → Aborting makes all writes in the transaction invisible
   → After abort, can start a new transaction immediately

5. SET CONSUMER isolation.level=read_committed
   → Without this, consumers see UNCOMMITTED messages
   → Defeats the entire purpose of transactions
   → Default is read_uncommitted — you MUST change it explicitly

6. USE send_offsets_to_transaction() FOR CONSUME-PROCESS-PRODUCE
   → This ties consumer offset commit to the transaction
   → Offset committed ONLY when transaction commits
   → Prevents the "processed but offset not committed" gap

7. ENSURE enable.idempotence IS TRUE
   → Automatically set when transactional.id is configured
   → But be explicit for documentation purposes
   → All Project 17 requirements (acks=all, max.in.flight≤5) auto-enforced

❌ WHAT SHOULD NOT DO

1. DO NOT USE RANDOM/DYNAMIC transactional.id
   → transactional.id = f"transfer-{uuid.uuid4()}"  ← WRONG
   → Every restart gets new ID → no zombie fencing
   → No cross-session deduplication
   → Broker accumulates abandoned transaction state
   → Eventually hits transactional.id cleanup overhead

2. DO NOT SHARE transactional.id ACROSS MULTIPLE PRODUCER INSTANCES
   → Two producers with same transactional.id = one FENCES the other
   → Fenced producer gets ProducerFencedException on next produce
   → Each concurrent producer instance needs UNIQUE transactional.id
   → Pattern: "transfer-service-{partition-range}" for partitioned processing

3. DO NOT PUT SLOW OPERATIONS INSIDE TRANSACTIONS
   → beginTransaction()
   → ... HTTP call to external API (3 seconds) ...  ← WRONG
   → ... Database query (500ms) ...                  ← WRONG
   → commitTransaction()
   → transaction.timeout.ms default = 60 seconds
   → Long transactions hold resources and block consumers

4. DO NOT FORGET isolation.level ON CONSUMERS
   → Producer uses transactions but consumer uses default (read_uncommitted)
   → Consumer sees ALL messages including aborted transaction messages
   → Those aborted messages appear as normal messages
   → Consumer processes data that was supposed to be rolled back
   → THIS IS THE #1 MISTAKE in Kafka transactions

5. DO NOT MIX TRANSACTIONAL AND NON-TRANSACTIONAL PRODUCES
   → Once a producer has transactional.id, ALL produces must be in transactions
   → Producing outside beginTransaction()...commitTransaction() throws error
   → If you need non-transactional produces, use a SEPARATE producer instance

6. DO NOT CALL initTransactions() MULTIPLE TIMES
   → Call once at startup
   → Calling again resets the producer's epoch → fences itself
   → Existing in-flight transactions aborted
   → Can cause data loss if not handled

7. DO NOT ASSUME TRANSACTIONS WORK ACROSS KAFKA CLUSTERS
   → Transactions are LOCAL to a single Kafka cluster
   → Cross-cluster atomicity requires MirrorMaker patterns (Project 27)
   → Or external coordination (Saga pattern — Project 36)

🚫 WHEN NOT TO USE

1. HIGH-THROUGHPUT LOG/METRICS INGESTION
   → Millions of events/second where occasional duplicates are acceptable
   → Transaction overhead (coordinator round-trips) reduces throughput 20-40%
   → Idempotent producer (Project 17) is sufficient
   → Or even acks=1 without idempotence for fire-and-forget logs

2. SINGLE-PARTITION SINGLE-SESSION DEDUP ONLY
   → If producer never crashes and writes to one partition
   → Idempotent producer is enough
   → Transactions add unnecessary coordinator overhead

3. WHEN EXTERNAL SYSTEMS ARE PART OF THE ATOMIC UNIT
   → "Write to Kafka AND update PostgreSQL atomically"
   → Kafka transactions CANNOT include external systems
   → Use Outbox Pattern (Project 37) or Saga (Project 36) instead

4. WHEN CONSUMER DOESN'T NEED read_committed
   → If downstream consumers process with their own dedup logic
   → Transaction markers add ~50 bytes per transaction to the log
   → Consumers with read_uncommitted ignore markers anyway
   → Transactions become pure overhead with no benefit

5. VERY LOW LATENCY REQUIREMENTS (< 5ms end-to-end)
   → Transaction commit requires round-trip to Transaction Coordinator
   → Adds 5-20ms to produce latency
   → For ultra-low-latency use cases, use idempotent producer + application-level dedup

COMPONENT 1: THE TRANSACTION COORDINATOR
═══════════════════════════════════════════════════════════════════

  Every Kafka broker can act as a Transaction Coordinator
  
  Which broker coordinates a specific transactional.id?
  → hash(transactional.id) % num_partitions(__transaction_state) = partition
  → Leader of that __transaction_state partition = coordinator for this transactional.id
  
  __transaction_state topic:
  → Internal compacted topic (similar to __consumer_offsets from Project 6)
  → Default 50 partitions
  → Stores: transactional.id → (PID, epoch, state, partitions, timeout)
  → Survives broker restarts (replicated)

STEP-BY-STEP: COMPLETE TRANSACTION LIFECYCLE
═══════════════════════════════════════════════════════════════════

  ┌─────────────────────────────────────────────────────────────┐
  │ PHASE A: INITIALIZATION (once at startup)                   │
  │                                                             │
  │ Producer → Transaction Coordinator:                         │
  │   "I am transactional.id='transfer-svc-01'"                │
  │                                                             │
  │ Coordinator:                                                │
  │   1. Look up transfer-svc-01 in __transaction_state         │
  │   2. If exists: bump epoch (3 → 4), fence old epoch        │
  │   3. If new: assign PID=7, epoch=0                          │
  │   4. Write to __transaction_state                           │
  │   5. Return PID=7, epoch=4 to producer                      │
  │                                                             │
  │ ZOMBIE FENCING:                                             │
  │   → Old producer with epoch=3 tries to produce              │
  │   → Broker sees epoch=3 < current epoch=4                   │
  │   → Returns ProducerFencedException                         │
  │   → Old producer is DEAD — cannot corrupt data              │
  └─────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────┐
  │ PHASE B: BEGIN TRANSACTION (client-side only)               │
  │                                                             │
  │ producer.beginTransaction()                                 │
  │   → Sets internal state to "in transaction"                 │
  │   → NO network call — purely local state change             │
  │   → Coordinator doesn't know yet                            │
  │   → Transaction "starts" when first produce() is sent       │
  └─────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────┐
  │ PHASE C: PRODUCING WITHIN TRANSACTION                       │
  │                                                             │
  │ producer.produce("account-debits", partition=2, ...)        │
  │                                                             │
  │ Step C1: AddPartitionsToTxn Request                         │
  │   Producer → Coordinator:                                   │
  │     "Add account-debits-2 to my transaction"                │
  │   Coordinator:                                              │
  │     → Records partition in transaction metadata             │
  │     → Writes to __transaction_state                         │
  │     → State: Ongoing, partitions: [account-debits-2]        │
  │                                                             │
  │ Step C2: Produce Request (same as normal, with PID+epoch)   │
  │   Producer → Partition Leader (account-debits-2):           │
  │     Record batch: PID=7, epoch=4, seq=0, data="DEBIT $5000"│
  │   Partition Leader:                                         │
  │     → Writes to log (message IS in the log)                 │
  │     → But marked as part of ongoing transaction             │
  │     → read_committed consumers CANNOT see it yet            │
  │     → read_uncommitted consumers CAN see it                 │
  │                                                             │
  │ Step C3: Second produce (different topic/partition)          │
  │   producer.produce("account-credits", partition=5, ...)     │
  │   → AddPartitionsToTxn: add account-credits-5               │
  │   → Produce: PID=7, epoch=4, seq=0, data="CREDIT $5000"    │
  │   → Coordinator state: partitions=[debits-2, credits-5]     │
  └─────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────┐
  │ PHASE D: SENDING CONSUMER OFFSETS (consume-process-produce) │
  │                                                             │
  │ producer.send_offsets_to_transaction(                       │
  │   offsets={TopicPartition("input-topic", 0): offset=150},  │
  │   group_metadata=consumer.consumer_group_metadata()         │
  │ )                                                           │
  │                                                             │
  │ Step D1: AddOffsetsToTxn Request                            │
  │   Producer → Coordinator:                                   │
  │     "My transaction also includes consumer group offsets"   │
  │     "Group: fraud-detector-group"                           │
  │   Coordinator:                                              │
  │     → Finds partition in __consumer_offsets for this group  │
  │     → Adds that partition to the transaction                │
  │                                                             │
  │ Step D2: TxnOffsetCommit Request                            │
  │   Producer → Group Coordinator:                             │
  │     "Write offset=150 for input-topic-0, group=fraud-det"  │
  │     "But DON'T make it visible until my transaction commits"│
  │   Group Coordinator:                                        │
  │     → Writes offset with PID+epoch tag                     │
  │     → Offset NOT visible to consumer group until commit     │
  └─────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────┐
  │ PHASE E: COMMIT TRANSACTION (Two-Phase Commit)              │
  │                                                             │
  │ producer.commitTransaction()                                │
  │                                                             │
  │ Step E1: EndTxn Request (Phase 1 — Prepare)                 │
  │   Producer → Coordinator:                                   │
  │     "Commit my transaction"                                 │
  │   Coordinator:                                              │
  │     → Writes PrepareCommit to __transaction_state           │
  │     → This write is REPLICATED (durable)                    │
  │     → If coordinator crashes HERE, new coordinator          │
  │       reads PrepareCommit and continues the commit          │
  │                                                             │
  │ Step E2: WriteTxnMarkers (Phase 2 — Execute)                │
  │   Coordinator → ALL partition leaders in transaction:       │
  │     "Write COMMIT marker to account-debits-2"              │
  │     "Write COMMIT marker to account-credits-5"             │
  │     "Write COMMIT marker to __consumer_offsets-X"          │
  │   Each Partition Leader:                                    │
  │     → Appends COMMIT control record to the partition log    │
  │     → This control record is NOT a data message             │
  │     → It tells consumers: "PID=7 epoch=4 is COMMITTED"     │
  │                                                             │
  │ Step E3: Complete                                           │
  │   Coordinator:                                              │
  │     → Writes CompleteCommit to __transaction_state          │
  │     → Transaction is DONE                                   │
  │     → Returns success to producer                           │
  │                                                             │
  │ NOW:                                                        │
  │   → read_committed consumers can see the debit message      │
  │   → read_committed consumers can see the credit message     │
  │   → Consumer group offset is advanced                       │
  │   → ALL happened atomically                                 │
  └─────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────┐
  │ PHASE F: ABORT TRANSACTION (Rollback)                       │
  │                                                             │
  │ producer.abortTransaction()                                 │
  │                                                             │
  │ Same two-phase process but with ABORT markers:              │
  │   → Coordinator writes PrepareAbort                         │
  │   → Coordinator sends ABORT markers to all partitions       │
  │   → Each partition appends ABORT control record             │
  │   → Coordinator writes CompleteAbort                        │
  │                                                             │
  │ NOW:                                                        │
  │   → Messages from this transaction ARE in the log           │
  │   → BUT followed by ABORT marker                            │
  │   → read_committed consumers SKIP these messages            │
  │   → read_uncommitted consumers still see them               │
  │   → Log compaction eventually removes aborted records       │
  └─────────────────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════════════
HOW read_committed CONSUMERS WORK
═══════════════════════════════════════════════════════════════════

  Consumer with isolation.level=read_committed:

  The log on partition account-debits-2:
  
  Offset 100: [PID=7, epoch=4] "DEBIT $5000"     ← transactional message
  Offset 101: [PID=9, epoch=0] "DEBIT $200"       ← non-transactional
  Offset 102: [PID=7, epoch=4] COMMIT_MARKER      ← control record
  Offset 103: [PID=11, epoch=1] "DEBIT $800"      ← another transaction
  Offset 104: [PID=11, epoch=1] ABORT_MARKER       ← aborted!
  
  What read_committed consumer sees:
  ✅ Offset 100: "DEBIT $5000"  → committed (marker at 102 confirms)
  ✅ Offset 101: "DEBIT $200"   → non-transactional, always visible
  ❌ Offset 102: COMMIT_MARKER  → filtered out (control record)
  ❌ Offset 103: "DEBIT $800"   → ABORTED (marker at 104 says abort)
  ❌ Offset 104: ABORT_MARKER   → filtered out (control record)
  
  LSO (Last Stable Offset):
  → The offset of the first message in an ONGOING (not yet committed) transaction
  → read_committed consumers cannot read PAST the LSO
  → If PID=7 has an ongoing transaction with a message at offset 50
  → Consumer blocks at offset 50 even if offsets 51-100 are committed
  → This is why LONG TRANSACTIONS ARE DANGEROUS — they block consumers

═══════════════════════════════════════════════════════════════════
ZOMBIE FENCING — THE SAFETY NET
═══════════════════════════════════════════════════════════════════

  SCENARIO: Producer instance A (transactional.id="transfer-01") crashes
            Producer instance B starts with SAME transactional.id
  
  Timeline:
  T1: Instance A: PID=7, epoch=3, beginTransaction(), produce(...)
  T2: Instance A: CRASHES (network partition, OOM, etc.)
  T3: Instance B: initTransactions() with transactional.id="transfer-01"
  T4: Coordinator: epoch 3 → 4, fence epoch 3
  T5: Coordinator: aborts A's ongoing transaction (sends ABORT markers)
  T6: Instance B: gets PID=7, epoch=4
  T7: Instance B: can safely begin new transactions
  
  IF Instance A somehow recovers at T8:
  T8: Instance A: tries produce() with PID=7, epoch=3
  T9: Broker: epoch 3 < current epoch 4 → ProducerFencedException
  T10: Instance A: MUST shut down — it's a zombie
  
  Result: NO duplicate writes, NO conflicting transactions

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: FORGETTING isolation.level=read_committed ON CONSUMERS
─────────────────────────────────────────────────────────────────
  Symptom: Consumer processes messages from aborted transactions
  Root Cause: Default isolation.level is read_uncommitted
  Impact: Data corruption — consumer acts on rolled-back data
  Fix: ALWAYS set isolation.level=read_committed when consuming
       from topics that transactional producers write to
  Detection: Consumer processes events that have no matching
             committed state in downstream systems

PITFALL 2: TRANSACTION TIMEOUT (transaction.timeout.ms)
─────────────────────────────────────────────────────────────────
  Symptom: ProducerFencedException during commitTransaction()
  Root Cause: Transaction took longer than timeout (default 60s)
  What happens:
    → Coordinator considers transaction abandoned
    → Coordinator bumps epoch → fences the slow producer
    → Producer gets fenced when it finally tries to commit
  Fix: Keep transactions short (< 30 seconds)
       Increase timeout if processing genuinely takes long
       Never put external API calls inside transactions

PITFALL 3: LONG TRANSACTIONS BLOCKING CONSUMERS (LSO STALL)
─────────────────────────────────────────────────────────────────
  Symptom: read_committed consumers stop receiving messages
  Root Cause: An ongoing transaction has a message at a low offset
  Example:
    → Transaction A starts, produces at offset 1000
    → Transactions B, C, D produce and commit at offsets 1001-1050
    → Transaction A is still ongoing
    → LSO stuck at 1000 → consumers can't read past 1000
    → Even though 1001-1050 are committed
  Fix: Keep transactions short
       Monitor LSO lag
       Set reasonable transaction.timeout.ms

PITFALL 4: transactional.id COLLISION ACROSS APPLICATIONS
─────────────────────────────────────────────────────────────────
  Symptom: ProducerFencedException on one of the producers
  Root Cause: Two DIFFERENT applications using same transactional.id
  Example:
    → Transfer service: transactional.id="service-01"
    → Payment service: transactional.id="service-01"  ← SAME!
    → One fences the other on every restart
  Fix: Use application-specific prefix in transactional.id
       "transfer-svc-01", "payment-svc-01"

PITFALL 5: NOT HANDLING ProducerFencedException
─────────────────────────────────────────────────────────────────
  Symptom: Application hangs or enters infinite retry loop
  Root Cause: Fenced producer keeps trying to produce
  What happens: Every produce() throws ProducerFencedException
  Fix: Catch ProducerFencedException → close producer → shut down
       Let orchestrator (Kubernetes, systemd) restart the process
       New instance will get new epoch and work correctly

PITFALL 6: MIXING TRANSACTIONAL AND NON-TRANSACTIONAL MESSAGES ON SAME TOPIC
─────────────────────────────────────────────────────────────────
  Symptom: read_committed consumers see gaps in data
  Root Cause: Non-transactional messages don't have transaction markers
  What happens:
    → Non-transactional messages always visible to read_committed
    → Transactional messages only visible after commit
    → Consumer sees interleaved committed/non-transactional data
    → Aborted transactional messages create "holes"
  Fix: Either ALL producers to a topic are transactional
       Or ALL are non-transactional
       Don't mix on the same topic

PITFALL 7: CONSUMER OFFSET REGRESSION AFTER ABORT
─────────────────────────────────────────────────────────────────
  Symptom: Consumer re-processes messages after transaction abort
  Root Cause: send_offsets_to_transaction() included in aborted transaction
  What happens:
    → Offsets committed as part of transaction
    → Transaction aborted → offset commit ALSO rolled back
    → Consumer restarts → reads from old (pre-transaction) offset
    → Re-processes messages
  This is CORRECT BEHAVIOR — exactly what we want!
  → But application must be prepared for reprocessing
  → Idempotent downstream writes prevent duplicate side effects

  KEY INSIGHT: Transaction overhead is PER-TRANSACTION, not per-message
  → Batching many messages in one transaction amortizes overhead
  → 1 message per transaction = worst case (25ms overhead)
  → 100 messages per transaction = nearly as fast as idempotent

TRANSACTION COORDINATOR LOAD
─────────────────────────────────────
  → Each transactional.id maps to one coordinator partition
  → 50 partitions in __transaction_state = 50 possible coordinators
  → Each coordinator handles:
      → InitProducerIdRequest (once per startup)
      → AddPartitionsToTxn (once per new partition in transaction)
      → EndTxn (once per commit/abort)
      → WriteTxnMarkers (once per partition per commit/abort)
  → For 1000 producers committing every second:
      → ~1000 EndTxn/s distributed across 50 coordinators
      → ~20 EndTxn/s per coordinator — easily handled

__transaction_state TOPIC SIZING
─────────────────────────────────────
  → Compacted topic — only latest state per transactional.id retained
  → 10,000 unique transactional.ids × ~200 bytes each = ~2MB
  → Negligible storage overhead
  → Replication factor should match your availability requirements

CONSUMER-SIDE SCALING WITH read_committed
─────────────────────────────────────
  → read_committed consumers buffer messages until transaction resolves
  → Memory usage increases with:
      → Number of concurrent open transactions
      → Duration of open transactions
      → Message size within transactions
  → For fast transactions (< 1s): minimal buffer overhead
  → For slow transactions (> 30s): significant buffer + LSO stall

PARTITION COUNT AND TRANSACTION SIZE
─────────────────────────────────────
  → Each partition added to transaction = one AddPartitionsToTxn RPC
  → Transaction touching 100 partitions = 100 RPCs + 100 commit markers
  → Keep partition count in single transaction reasonable (< 50)
  → For consume-process-produce: usually 1 input + 1 output = 2 partitions
  → For fan-out patterns: consider if transactions are truly needed

SCALING transactional.id ASSIGNMENT
─────────────────────────────────────
  Pattern for horizontal scaling:

  ┌──────────────────────────────────────────────────────────┐
  │ Instance 1: transactional.id = "transfer-svc-part-0-2"  │
  │   → Handles partitions 0, 1, 2                          │
  │                                                          │
  │ Instance 2: transactional.id = "transfer-svc-part-3-5"  │
  │   → Handles partitions 3, 4, 5                          │
  │                                                          │
  │ Instance 3: transactional.id = "transfer-svc-part-6-8"  │
  │   → Handles partitions 6, 7, 8                          │
  └──────────────────────────────────────────────────────────┘

  → Each instance has unique, stable transactional.id
  → Partition assignment encoded in the ID

"""
Fund Transfer Service — Transactional Producer
Atomically writes DEBIT + CREDIT events for each transfer.

Builds on:
  → Project 17: Idempotent producer (PID + Sequence)
  → Project 12: Python Kafka client
  → Project 13: Message design
  → Project 4:  Key-based routing
"""

import json
import time
import uuid
import random
import signal
from datetime import datetime, timezone
from confluent_kafka import Producer, KafkaError, KafkaException


# ============================================================
# TRANSACTIONAL PRODUCER CONFIGURATION
# ============================================================

def create_transactional_producer(instance_id):
    """
    Create a transactional producer with a STABLE transactional.id.

    KEY DIFFERENCE FROM PROJECT 17:
    → Project 17: enable.idempotence=true (per-session dedup)
    → Project 18: transactional.id set (cross-session dedup + atomic writes)
    → Setting transactional.id AUTO-ENABLES idempotence
    """

    config = {
        'bootstrap.servers': 'localhost:9092',

        # ─── THE CRITICAL NEW SETTING ───
        # This MUST be:
        #   1. Unique per producer instance
        #   2. Stable across restarts of the SAME instance
        #   3. NOT a random UUID
        'transactional.id': f'transfer-service-{instance_id}',

        # ─── AUTO-ENABLED BY transactional.id ───
        # Listed explicitly for documentation (from Project 17)
        'enable.idempotence': True,
        'acks': 'all',
        'max.in.flight.requests.per.connection': 5,

        # ─── TRANSACTION TIMEOUT ───
        # If transaction not committed/aborted within this time,
        # coordinator aborts it and fences the producer
        'transaction.timeout.ms': 60000,  # 60 seconds

        # ─── BATCHING ───
        'batch.size': 16384,
        'linger.ms': 5,
        'compression.type': 'snappy',

        # ─── CLIENT ID ───
        'client.id': f'transfer-producer-{instance_id}',
    }

    producer = Producer(config)
    return producer


# ============================================================
# TRANSFER EVENT GENERATOR
# ============================================================

class TransferGenerator:
    """Generates realistic inter-account fund transfer events"""

    ACCOUNTS = [
        {'id': f'ACCT-{str(i).zfill(6)}', 'type': t, 'holder': f'Customer-{i}'}
        for i, t in enumerate(
            ['SAVINGS', 'CHECKING', 'SAVINGS', 'CHECKING', 'MONEY_MARKET',
             'SAVINGS', 'CHECKING', 'SAVINGS', 'CHECKING', 'MONEY_MARKET',
             'SAVINGS', 'CHECKING', 'SAVINGS', 'CHECKING', 'CHECKING',
             'SAVINGS', 'CHECKING', 'SAVINGS', 'CHECKING', 'MONEY_MARKET'],
        )
    ]

    def generate_transfer(self):
        # Pick two DIFFERENT accounts
        source = random.choice(self.ACCOUNTS)
        target = random.choice([a for a in self.ACCOUNTS if a['id'] != source['id']])
        amount = round(random.uniform(100, 10000), 2)

        transfer_id = f"TXF-{datetime.now(timezone.utc).strftime('%Y%m%d%H%M%S')}-{uuid.uuid4().hex[:8]}"

        return {
            'transfer_id': transfer_id,
            'source_account': source['id'],
            'target_account': target['id'],
            'source_type': source['type'],
            'target_type': target['type'],
            'amount': amount,
            'currency': 'USD',
            'timestamp_utc': datetime.now(timezone.utc).isoformat(),
            'initiated_by': source['holder'],
        }


# ============================================================
# TRANSACTIONAL TRANSFER EXECUTION
# ============================================================

class TransferService:
    """
    Executes fund transfers using Kafka transactions.

    For each transfer:
    1. beginTransaction()
    2. Produce DEBIT event to "account-debits"
    3. Produce CREDIT event to "account-credits"
    4. Produce RESULT event to "transfer-results"
    5. commitTransaction() — all three writes become visible atomically

    If ANY step fails → abortTransaction() → none visible
    """

    def __init__(self, instance_id='01'):
        self.instance_id = instance_id
        self.producer = create_transactional_producer(instance_id)
        self.generator = TransferGenerator()

        # Statistics
        self.committed_count = 0
        self.aborted_count = 0
        self.total_transferred = 0.0

        # ─── INITIALIZE TRANSACTIONS ───
        # This MUST be called exactly ONCE before any transaction
        # It registers with the Transaction Coordinator
        # And fences any zombie instance with the same transactional.id
        print(f'  Initializing transactions for: transfer-service-{instance_id}')
        self.producer.init_transactions()
        print(f'  ✓ Transactions initialized — ready to process transfers')

    def execute_transfer(self, transfer):
        """
        Execute a single fund transfer as an atomic Kafka transaction.
        """

        transfer_id = transfer['transfer_id']
        source = transfer['source_account']
        target = transfer['target_account']
        amount = transfer['amount']

        print(f'\n  ┌─ TRANSFER {transfer_id}')
        print(f'  │  {source} → {target}: ${amount:,.2f}')

        try:
            # ═══════════════════════════════════════════
            # STEP 1: BEGIN TRANSACTION
            # ═══════════════════════════════════════════
            # This is a LOCAL operation — no network call
            # Transaction officially starts on first produce()
            self.producer.begin_transaction()
            print(f'  │  ✓ Transaction begun')

            # ═══════════════════════════════════════════
            # STEP 2: PRODUCE DEBIT EVENT
            # ═══════════════════════════════════════════
            debit_event = {
                'transfer_id': transfer_id,
                'event_type': 'DEBIT',
                'account_id': source,
                'amount': -amount,  # Negative for debit
                'currency': transfer['currency'],
                'counterparty_account': target,
                'timestamp_utc': transfer['timestamp_utc'],
            }

            self.producer.produce(
                topic='account-debits',
                key=source.encode('utf-8'),  # Key = source account (Project 4)
                value=json.dumps(debit_event).encode('utf-8'),
                headers=[
                    ('transfer_id', transfer_id.encode('utf-8')),
                    ('event_type', b'DEBIT'),
                    ('transactional', b'true'),
                ],
            )
            print(f'  │  ✓ DEBIT produced: {source} -${amount:,.2f}')

            # ═══════════════════════════════════════════
            # STEP 3: PRODUCE CREDIT EVENT
            # ═══════════════════════════════════════════
            credit_event = {
                'transfer_id': transfer_id,
                'event_type': 'CREDIT',
                'account_id': target,
                'amount': amount,  # Positive for credit
                'currency': transfer['currency'],
                'counterparty_account': source,
                'timestamp_utc': transfer['timestamp_utc'],
            }

            self.producer.produce(
                topic='account-credits',
                key=target.encode('utf-8'),  # Key = target account
                value=json.dumps(credit_event).encode('utf-8'),
                headers=[
                    ('transfer_id', transfer_id.encode('utf-8')),
                    ('event_type', b'CREDIT'),
                    ('transactional', b'true'),
                ],
            )
            print(f'  │  ✓ CREDIT produced: {target} +${amount:,.2f}')

            # ═══════════════════════════════════════════
            # STEP 4: PRODUCE TRANSFER RESULT EVENT
            # ═══════════════════════════════════════════
            result_event = {
                'transfer_id': transfer_id,
                'status': 'COMPLETED',
                'source_account': source,
                'target_account': target,
                'amount': amount,
                'currency': transfer['currency'],
                'completed_at': datetime.now(timezone.utc).isoformat(),
            }

            self.producer.produce(
                topic='transfer-results',
                key=transfer_id.encode('utf-8'),
                value=json.dumps(result_event).encode('utf-8'),
                headers=[
                    ('transfer_id', transfer_id.encode('utf-8')),
                    ('status', b'COMPLETED'),
                ],
            )
            print(f'  │  ✓ RESULT produced: COMPLETED')

            # ═══════════════════════════════════════════
            # STEP 5: COMMIT TRANSACTION
            # ═══════════════════════════════════════════
            # This is the CRITICAL step:
            # → Coordinator writes PrepareCommit
            # → Coordinator sends COMMIT markers to all partitions
            # → All three messages become visible atomically
            self.producer.commit_transaction()

            self.committed_count += 1
            self.total_transferred += amount
            print(f'  │  ✅ TRANSACTION COMMITTED')
            print(f'  └─ All 3 events (DEBIT + CREDIT + RESULT) are now visible')

        except KafkaException as e:
            # ═══════════════════════════════════════════
            # TRANSACTION FAILURE HANDLING
            # ═══════════════════════════════════════════

            error = e.args[0]
            print(f'  │  ❌ Transaction error: {error}')

            if error.code() == KafkaError._FENCED:
                # ─── ZOMBIE FENCING ───
                # Another instance with same transactional.id took over
                # This instance must SHUT DOWN
                print(f'  │  🧟 FENCED — another instance has taken over')
                print(f'  └─ This producer must shut down immediately')
                raise  # Re-raise — caller must handle shutdown

            elif error.code() == KafkaError.INVALID_TXN_STATE:
                # ─── INVALID STATE ───
                # Transaction in unexpected state
                print(f'  │  ⚠️  Invalid transaction state')
                try:
                    self.producer.abort_transaction()
                    print(f'  │  ✓ Transaction aborted')
                except Exception:
                    print(f'  │  ⚠️  Abort also failed — producer may need restart')
                self.aborted_count += 1
                print(f'  └─ Transfer ROLLED BACK — no partial writes visible')

            else:
                # ─── GENERAL ERROR ───
                # Attempt to abort the transaction
                print(f'  │  Attempting abort...')
                try:
                    self.producer.abort_transaction()
                    self.aborted_count += 1
                    print(f'  │  ✓ Transaction aborted successfully')
                except KafkaException as abort_err:
                    print(f'  │  ❌ Abort failed: {abort_err}')
                    print(f'  │  Producer may need restart')
                print(f'  └─ Transfer ROLLED BACK')

        except BufferError:
            # ─── BUFFER FULL ───
            print(f'  │  ⚠️  Producer buffer full')
            try:
                self.producer.abort_transaction()
                self.aborted_count += 1
            except Exception:
                pass
            print(f'  └─ Transfer ROLLED BACK due to buffer pressure')

    def simulate_failed_transfer(self):
        """
        Demonstrate abort behavior — produce debit but abort before credit.
        read_committed consumers should NEVER see the orphaned debit.

        transfer = self.generator.generate_transfer()

        # Producer cleanup

def process_batch(producer, consumer, detector, batch, stats):
    """
    Process a batch of messages in a SINGLE Kafka transaction.

    THE ATOMIC UNIT:
    → For each message in batch:
        → Analyze for fraud
        → If suspicious: produce alert to fraud-alerts
    → Commit consumer offsets for all messages in batch
    → ALL of the above = ONE transaction
    → Commit or abort as a unit
    """

    if not batch:
        return

    print(f'\n  ┌─ TRANSACTION BATCH ({len(batch)} messages)')

    try:
        # ═══════════════════════════════════════════
        # STEP 1: BEGIN TRANSACTION
        # ═══════════════════════════════════════════
        producer.begin_transaction()

        alerts_in_batch = 0
        offsets_to_commit = {}

        for msg in batch:
            partition = msg.partition()
            offset = msg.offset()

            # ─── DESERIALIZE ───
            try:
                value = json.loads(msg.value().decode('utf-8'))
            except (json.JSONDecodeError, AttributeError):
                print(f'  │  ⚠️  Skip bad message at P{partition}|O{offset}')
                # Still track offset to advance past bad message
                offsets_to_commit[TopicPartition(msg.topic(), partition)] = offset + 1
                continue

            transfer_id = value.get('transfer_id', 'unknown')
            amount = value.get('amount', 0.0)

            # ═══════════════════════════════════════════
            # STEP 2: PROCESS (Fraud Detection)
            # ═══════════════════════════════════════════
            alert = detector.analyze(value)

            if alert:
                # ═══════════════════════════════════════════
                # STEP 3: PRODUCE ALERT (within transaction)
                # ═══════════════════════════════════════════
                producer.produce(
                    topic='fraud-alerts',
                    key=alert['account_id'].encode('utf-8'),
                    value=json.dumps(alert).encode('utf-8'),
                    headers=[
                        ('alert_level', alert['alert_level'].encode('utf-8')),
                        ('source_transfer', transfer_id.encode('utf-8')),
                        ('exactly_once', b'true'),
                    ],
                )
                alerts_in_batch += 1
                stats['alerts_produced'] += 1
                print(f'  │  🚨 ALERT [{alert["alert_level"]}] '
                      f'Account={alert["account_id"]} '
                      f'Amount=${abs(amount):,.2f} '
                      f'Transfer={transfer_id[:20]}...')
            else:
                stats['no_alert_needed'] += 1
                print(f'  │  ✓  OK     P{partition}|O{offset} '
                      f'Amount=${abs(amount):,.2f} — no fraud detected')

            stats['processed'] += 1

            # ─── TRACK HIGHEST OFFSET PER PARTITION ───
            tp = TopicPartition(msg.topic(), partition)
            current_max = offsets_to_commit.get(tp, 0)
            offsets_to_commit[tp] = max(current_max, offset + 1)

        # ═══════════════════════════════════════════
        # STEP 4: SEND CONSUMER OFFSETS TO TRANSACTION
        # ═══════════════════════════════════════════
        # THIS IS THE KEY TO EXACTLY-ONCE:
        # → Consumer offsets are committed AS PART of the transaction
        # → If transaction commits: offsets advance + alerts written
        # → If transaction aborts: offsets NOT advanced + alerts NOT written
        # → On restart: consumer re-reads from old offset → re-processes
        # → But re-produce with same PID → broker deduplicates

        # Convert offsets dict to list format
        offset_list = [
            TopicPartition(tp.topic, tp.partition, off)
            for tp, off in offsets_to_commit.items()
        ]

        # Get consumer group metadata for the transaction
        group_metadata = consumer.consumer_group_metadata()

        producer.send_offsets_to_transaction(
            offset_list,
            group_metadata,
        )

        print(f'  │  📎 Offsets attached to transaction:')
        for tp, off in offsets_to_commit.items():
            print(f'  │     {tp.topic} P{tp.partition} → offset {off}')

        # ═══════════════════════════════════════════
        # STEP 5: COMMIT TRANSACTION
        # ═══════════════════════════════════════════
        # Atomic commit of:
        #   → All fraud alerts produced in this batch
        #   → All consumer offset advances
        # Either ALL happen or NONE happen
        producer.commit_transaction()

        stats['transactions_committed'] += 1
        print(f'  │  ✅ TRANSACTION COMMITTED')
        print(f'  │     {len(batch)} messages processed, {alerts_in_batch} alerts produced')
        print(f'  └─ Offsets + alerts committed atomically')

    except KafkaException as e:
        error = e.args[0]
        print(f'  │  ❌ Transaction error: {error}')

        if error.code() == KafkaError._FENCED:
            print(f'  └─ 🧟 FENCED — must shut down')
            raise  # Propagate to main loop

        # ─── ABORT TRANSACTION ───
        try:
            producer.abort_transaction()
            stats['transactions_aborted'] += 1
            print(f'  │  ✓ Transaction aborted')
            print(f'  │  → Alerts NOT written, offsets NOT advanced')
            print(f'  └─ Messages will be re-read on next poll')
        except KafkaException as abort_err:
            print(f'  └─ ❌ Abort failed: {abort_err} — producer needs restart')
            raise


# ============================================================
# ENTRY POINT
# ============================================================

def consume_messages(isolation_level, timeout=10):
    """Consume all messages with specified isolation level"""
    config = {
        'bootstrap.servers': BOOTSTRAP_SERVERS,
        'group.id': f'verify-{isolation_level}-{uuid.uuid4().hex[:8]}',
        'auto.offset.reset': 'earliest',
        'enable.auto.commit': False,
        'isolation.level': isolation_level,
    }
    consumer = Consumer(config)
    consumer.subscribe([TEST_TOPIC])

    messages = []
    start = time.time()
    while time.time() - start < timeout:
        msg = consumer.poll(1.0)
        if msg is None:
            if messages:
                break
            continue
        if msg.error():
            if msg.error().code() == KafkaError._PARTITION_EOF:
                break
            continue
        messages.append(json.loads(msg.value().decode('utf-8')))

    consumer.close()
    return messages
