PROJECT 4: THE HOTEL FRONT DESK — KEY-BASED MESSAGE ROUTING

🎯 HYPOTHETICAL PROBLEM

You manage "ByteStay Grand Hotel" — a luxury hotel with 200 rooms

→ Guests check in, order room service, request housekeeping,
  complain about noise, extend their stay, and check out
→ ALL of these events are recorded in a central system
→ Each event must be processed IN ORDER per guest

HERE IS THE PROBLEM:

Guest "Mr. Anderson" (Room 412) does the following:
  T=1: Checks in
  T=2: Orders room service (steak dinner)
  T=3: Complains about cold food
  T=4: Gets a replacement meal
  T=5: Checks out with a discount applied

If these events are processed OUT OF ORDER:
  → System tries to apply discount BEFORE check-in → ERROR
  → System delivers replacement BEFORE complaint → CONFUSION
  → System processes checkout BEFORE room service charge → REVENUE LOSS

YOU NEED:
→ All events for Mr. Anderson processed IN ORDER
→ All events for Ms. Chen (Room 208) processed IN ORDER
→ BUT Mr. Anderson's events and Ms. Chen's events
  can be processed IN PARALLEL (independent guests)

ADDITIONAL COMPLEXITY:
→ Some events use "guest name" as identifier
→ Some events use "room number" as identifier
→ Some events have NO identifier (system-wide announcements)
→ Peak season: 200 guests × ~10 events each = 2000 events/hour
→ Off season: 20 guests × ~3 events each = 60 events/hour

CHOOSING THE WRONG KEY:
→ key="hotel" → ALL events to ONE partition → no parallelism
→ key=None → events scattered → per-guest ordering LOST
→ key="room-412" → what if guest changes rooms?
→ key="mr-anderson" → what if two guests share same name?

WHAT IS THE RIGHT KEY?

🔍 HOW TO APPROACH IT

WHAT WE ALREADY KNOW:
→ Partitions are independent logs with own offsets (Project 3) ✅
→ hash(key) % partition_count = target partition (Project 3) ✅
→ Same key always goes to same partition (Project 3) ✅
→ No key = round-robin/sticky distribution (Project 3) ✅

WHAT IS NEW:
→ HOW to choose the right key for a business problem
→ WHAT happens with different key strategies
→ WHY key cardinality matters for partition balance
→ WHAT is the difference between null key, empty key, and meaningful key
→ HOW to prove determinism survives producer restarts
→ HOW hot partitions form and how to detect them

APPROACH:
Step 1: Model the hotel events with different key strategies
Step 2: Measure partition distribution for each strategy
Step 3: Prove per-key ordering across multiple produce sessions
Step 4: Demonstrate the hot partition problem
Step 5: Establish key selection guidelines for real-world systems

🛠️ HOW CAN WE SOLVE IT

SOLUTION: Use GUEST ID as the message key

WHY GUEST ID?

  ┌─────────────────────┬────────────────┬────────────────────────────────────┐
  │ Key Candidate        │ Cardinality    │ Problem                            │
  ├─────────────────────┼────────────────┼────────────────────────────────────┤
  │ key=None             │ N/A            │ No per-guest ordering              │
  │ key="hotel"          │ 1              │ All events → 1 partition (useless) │
  │ key="vip" / "normal" │ 2              │ 2 partitions used, rest empty      │
  │ key=room_number      │ ~200           │ Guest changes room → key changes   │
  │ key=guest_name       │ ~200           │ Name collisions (two "John Smith") │
  │ key=guest_id  ✅     │ ~200 (unique)  │ Immutable, unique, high cardinality│
  └─────────────────────┴────────────────┴────────────────────────────────────┘

GUEST ID is the right key because:
→ It is UNIQUE per guest (no collisions)
→ It is IMMUTABLE (does not change if guest switches rooms)
→ It has HIGH CARDINALITY (200 unique values → good distribution)
→ All events for one guest share the same key → same partition → ordered

DESIGN:
  ┌───────────────┐
  │ Event:        │
  │  key: G-1042  │───→ hash("G-1042") % 4 = Partition 2
  │  value: {     │
  │    guest_id   │     Partition 2: [checkin] [room-svc] [complaint] [checkout]
  │    event_type │                   ↑ all G-1042 events, IN ORDER
  │    details    │
  │    timestamp  │
  │  }            │
  └───────────────┘

✅ WHAT SHOULD DO

1. ALWAYS choose a key that represents the ENTITY whose ordering matters
   → Ordering per guest → key = guest_id
   → Ordering per device → key = device_id
   → Ordering per transaction → key = transaction_id
   → The key answers: "for WHOM must events be ordered?"

2. ALWAYS use IMMUTABLE identifiers as keys
   → guest_id ✅ (never changes)
   → room_number ❌ (guest can switch rooms)
   → email ⚠️ (user might change email)
   → UUID ✅ (generated once, never changes)

3. ALWAYS prefer HIGH CARDINALITY keys
   → 200 unique guest IDs → messages spread across all 4 partitions
   → 2 unique categories → messages pile up in 2 partitions only
   → Cardinality = number of distinct key values
   → Higher cardinality → more even distribution

4. ALWAYS encode keys as UTF-8 strings consistently
   → key="G-1042" everywhere (not "g-1042" or "G1042" or " G-1042 ")
   → Hash of "G-1042" ≠ hash of "g-1042" → different partitions!
   → Inconsistent key encoding BREAKS ordering guarantee

5. ALWAYS validate key consistency at the producer level
   → Log the key + assigned partition for debugging
   → Detect key format anomalies before they reach Kafka

❌ WHAT SHOULD NOT DO

1. DO NOT use LOW CARDINALITY keys unless you accept skew
   → key="vip" for all VIP guests → ONE partition gets all VIP traffic
   → key="country" with 80% traffic from "US" → hot partition

2. DO NOT use MUTABLE attributes as keys
   → key=email → user changes email → new key → different partition
   → Events before and after email change go to DIFFERENT partitions
   → Ordering for that user is BROKEN

3. DO NOT use empty string "" as a key thinking it means "no key"
   → "" is a VALID key → hash("") produces a specific partition
   → ALL messages with key="" go to the SAME partition
   → This is NOT the same as key=None (no key)

   ┌──────────────┬──────────────────────────────────────────┐
   │ key=None     │ No key → round-robin/sticky distribution  │
   │ key=""       │ Empty string key → hash("") → ONE partition│
   │ key="guest"  │ Meaningful key → hash("guest") → ONE part │
   └──────────────┴──────────────────────────────────────────┘

4. DO NOT change partition count after establishing key routing
   → Covered in Project 3 — hash(key) % NEW_count ≠ old mapping
   → All established key-to-partition assignments BREAK

5. DO NOT use composite keys without careful thought
   → key="G-1042_room-412" → changes if room changes
   → key="G-1042" alone → stable and sufficient
   → Composite keys reduce cardinality if one part is low-cardinality

6. DO NOT use random UUIDs as keys when ordering does not matter
   → Random keys → perfect distribution BUT no semantic meaning
   → If you do not need per-entity ordering, use key=None instead
   → Keys consume storage (stored in the log alongside value)

🔀 WHEN TO USE WHAT

┌──────────────────────────────────┬──────────────────────────────────────────┐
│ USE key=None WHEN                │ USE key=<entity_id> WHEN                 │
├──────────────────────────────────┼──────────────────────────────────────────┤
│ You want maximum even            │ You need ordering per entity             │
│ distribution across partitions   │ (per user, per device, per order)        │
├──────────────────────────────────┼──────────────────────────────────────────┤
│ Messages are independent         │ Messages are related and sequence        │
│ (each log line is standalone)    │ matters (event chain for one entity)     │
├──────────────────────────────────┼──────────────────────────────────────────┤
│ No consumer needs all messages   │ One consumer must see all events         │
│ for one entity together          │ for one entity in order                  │
└──────────────────────────────────┴──────────────────────────────────────────┘

┌──────────────────────────────────┬──────────────────────────────────────────┐
│ HIGH CARDINALITY KEY             │ LOW CARDINALITY KEY                      │
│ (user_id, order_id, device_id)   │ (country, plan_type, status)             │
├──────────────────────────────────┼──────────────────────────────────────────┤
│ ✅ Even distribution             │ ⚠️ Skewed distribution                   │
│ ✅ All partitions utilized       │ ⚠️ Some partitions overloaded            │
│ ✅ Parallelism maximized         │ ⚠️ Parallelism limited                   │
├──────────────────────────────────┼──────────────────────────────────────────┤
│ USE when entity count >> parts   │ USE only when you truly need ordering    │
│                                  │ per that low-cardinality attribute       │
└──────────────────────────────────┴──────────────────────────────────────────┘

┌──────────────────────────────────┬──────────────────────────────────────────┐
│ SINGLE-FIELD KEY                 │ COMPOSITE KEY                            │
│ key="G-1042"                     │ key="G-1042_room-412"                    │
├──────────────────────────────────┼──────────────────────────────────────────┤
│ ✅ Stable if field is immutable  │ ⚠️ Unstable if any component changes     │
│ ✅ Simple to manage              │ ⚠️ Higher cardinality (may be good/bad)  │
│ ✅ Sufficient for most cases     │ ✅ Useful for sub-entity ordering         │
│                                  │   (per user per session)                 │
└──────────────────────────────────┴──────────────────────────────────────────┘

🚫 WHEN NOT TO USE

→ DO NOT use keyed messages for system-wide broadcast events
  (e.g., "maintenance window starting" should go to all partitions
   or use a dedicated single-partition topic)

→ DO NOT use keyed messages when the key has cardinality of 1
  (all messages go to one partition → worse than no key at all)

→ DO NOT use keyed messages when distribution matters more than ordering
  (log aggregation: each log line is independent → key=None is better)

→ DO NOT use keyed messages for time-series data where ordering
  is by timestamp across ALL sources
  (use 1 partition or implement cross-partition merging — Project 22)

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

═══════════════════════════════════════════════════════════════
THE COMPLETE KEY ROUTING PIPELINE
═══════════════════════════════════════════════════════════════

When you call: producer.produce(topic="hotel-events", key="G-1042", value=...)

STEP 1: Key Serialization
─────────────────────────
  → Your key string "G-1042" is converted to bytes
  → Using the configured key serializer (default: StringSerializer)
  → "G-1042" → bytes: [71, 45, 49, 48, 52, 50]

STEP 2: Partition Assignment (inside the Producer)
──────────────────────────────────────────────────
  → The producer's PARTITIONER function is called
  → Default partitioner checks: is key null?

  IF key IS null:
    → Use Sticky Partitioner (Kafka 2.4+)
    → Fill current partition's batch until batch.size or linger.ms
    → Then switch to next partition
    → Result: roughly even distribution over time

  IF key IS NOT null:
    → Compute: murmur2(key_bytes)
    → Apply: abs(hash) % partition_count
    → This is DETERMINISTIC — always same result for same key

  ┌──────────────────────────────────────────────────────────┐
  │                  PARTITIONER DECISION TREE                │
  │                                                          │
  │  produce(key, value)                                     │
  │       │                                                  │
  │       ▼                                                  │
  │  ┌─────────────┐                                         │
  │  │ key == null? │                                         │
  │  └──────┬──────┘                                         │
  │    YES  │    NO                                          │
  │    │    │                                                │
  │    ▼    ▼                                                │
  │  Sticky  murmur2(key_bytes)                              │
  │  Batch   abs(hash) % num_partitions                      │
  │  Mode    → deterministic partition                       │
  │    │         │                                           │
  │    ▼         ▼                                           │
  │  [some    [specific                                      │
  │   part]    partition]                                     │
  └──────────────────────────────────────────────────────────┘

STEP 3: Record Accumulator Buffering
─────────────────────────────────────
  → Message placed in the batch for (topic, partition)
  → Producer maintains one batch per topic-partition pair
  → Batch accumulates until batch.size or linger.ms triggers send

STEP 4: Network Send
─────────────────────
  → Sender thread picks up full batches
  → Sends ProduceRequest to the broker that LEADS that partition
  → Broker appends to the partition's log file
  → Broker responds with assigned offset

STEP 5: Callback
────────────────
  → Response arrives → callback fires with (topic, partition, offset)
  → You can verify: same key → same partition every time


═══════════════════════════════════════════════════════════════
THE MURMUR2 HASH IN DETAIL
═══════════════════════════════════════════════════════════════

  Why murmur2?
  → Fast (non-cryptographic hash — speed over security)
  → Good distribution (minimizes collisions)
  → Java-compatible (Kafka is written in Java/Scala)
  → Deterministic across all platforms (same bytes → same hash)

  Properties:
  → murmur2("G-1042")  → always the same integer
  → murmur2("G-1043")  → completely different integer
  → murmur2("g-1042")  → different from "G-1042" (case sensitive!)
  → murmur2("")         → specific non-zero integer (empty string IS hashed)

  Collision behavior:
  → With 200 guests and 4 partitions, ~50 guests per partition
  → Distribution is NOT perfectly even — hash functions have variance
  → Some partitions might get 45 guests, others 55
  → This is acceptable — perfect balance is not guaranteed or needed


═══════════════════════════════════════════════════════════════
NULL KEY vs EMPTY KEY vs MEANINGFUL KEY — BYTE-LEVEL TRUTH
═══════════════════════════════════════════════════════════════

  ┌──────────────────────────────────────────────────────────────────┐
  │ What You Pass        │ Bytes Stored   │ Partition Logic           │
  ├──────────────────────┼────────────────┼───────────────────────────┤
  │ key=None             │ No key bytes   │ Sticky/Round-robin        │
  │                      │ (null in log)  │ (no hash computed)        │
  ├──────────────────────┼────────────────┼───────────────────────────┤
  │ key=b""              │ 0 bytes stored │ murmur2(empty) % parts    │
  │ (empty bytes)        │ (key exists    │ → SPECIFIC partition      │
  │                      │  but empty)    │ → ALL empty keys → same   │
  ├──────────────────────┼────────────────┼───────────────────────────┤
  │ key=b"G-1042"        │ 6 bytes stored │ murmur2("G-1042") % parts│
  │ (meaningful key)     │                │ → SPECIFIC partition      │
  │                      │                │ → deterministic per key   │
  └──────────────────────┴────────────────┴───────────────────────────┘

  STORED IN THE LOG:

  Each message record on disk:
  ┌──────────┬──────────┬───────────────┬────────────────┬──────────┐
  │ Offset   │ Timestamp│ Key Length    │ Key Bytes      │ Value    │
  ├──────────┼──────────┼───────────────┼────────────────┼──────────┤
  │ 0        │ 17100000 │ 6             │ "G-1042"       │ {...}    │
  │ 1        │ 17100001 │ -1 (null)     │ (none)         │ {...}    │
  │ 2        │ 17100002 │ 0             │ (empty)        │ {...}    │
  └──────────┴──────────┴───────────────┴────────────────┴──────────┘

  Key Length = -1 means NULL key (no key at all)
  Key Length = 0 means EMPTY key (key exists but has no bytes)
  Key Length > 0 means MEANINGFUL key


═══════════════════════════════════════════════════════════════
HOW DETERMINISM SURVIVES ACROSS RESTARTS
═══════════════════════════════════════════════════════════════

  QUESTION: If I stop the producer and restart it,
            does "G-1042" still go to the same partition?

  ANSWER: YES — because the hash function is STATELESS

  The partitioner does NOT store any mapping.
  It computes: murmur2("G-1042") % 4 → every single time.
  This computation depends ONLY on:
    1. The key bytes (always same for "G-1042")
    2. The partition count (does not change for existing topic)

  No state, no memory, no lookup table.
  Pure mathematics → always same result.

  WHAT WOULD BREAK IT:
  → Changing partition count (hash % NEW_count ≠ hash % OLD_count)
  → Changing the key string ("G-1042" vs "g-1042")
  → Using a different hash function (custom partitioner — Project 31)

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Guest events are out of order even though I used a key"
├── CAUSE A: Key is inconsistent — sometimes "G-1042", sometimes "g-1042"
│   └── FIX: Normalize keys (lowercase, strip whitespace) before producing
├── CAUSE B: Multiple partitions AND topic partition count was changed
│   └── FIX: Do not change partition count on keyed topics
├── CAUSE C: Producer retries caused duplicate with reordering
│   └── FIX: Enable idempotent producer (Project 17)
└── CAUSE D: Using key=None accidentally for some messages
    └── FIX: Validate key is not None before producing

PITFALL 2: "One partition has 80% of messages, others are nearly empty"
├── CAUSE: Hot key — one key value dominates traffic
├── EXAMPLE: key="walk-in" for all non-registered guests
│   → 80% of guests are walk-ins → 80% of messages to ONE partition
├── DETECTION: Check partition watermark offsets (partition_inspector.py from Project 3)
└── FIX: Choose higher cardinality key
         OR use custom partitioner to spread hot keys (Project 31)

PITFALL 3: "Key is correct but consumer processes events out of order"
├── CAUSE: Consumer is in a group with multiple members
│   → Partition was REASSIGNED during rebalance (Project 10)
│   → New consumer starts from last committed offset
│   → Some events processed twice if offset commit was delayed
├── THIS IS NOT A KEY PROBLEM — it is a consumer group problem
└── FIX: Understand consumer groups (Project 5) and rebalancing (Project 10)

PITFALL 4: "Using guest email as key — events split after email change"
├── CAUSE: Mutable attribute used as key
├── SCENARIO:
│   Before: key="anderson@mail.com" → Partition 2
│   After:  key="anderson@newmail.com" → Partition 0
│   Events for same guest now in TWO partitions
└── FIX: Use immutable identifier (guest_id, UUID)

PITFALL 5: "Empty string key causes all unkeyed messages to go to one partition"
├── CAUSE: Using key="" instead of key=None
├── WHY: "" is a valid key → hash("") → specific partition
│   All messages with key="" → SAME partition
├── SYMPTOM: Looks like a hot partition but no obvious hot key
└── FIX: Use key=None when you truly want no key

PITFALL 6: "Composite key too specific — cardinality explosion"
├── CAUSE: key="G-1042_2024-03-10_session-abc"
│   → Every message gets a unique key → every message to different partition
│   → Per-entity ordering is LOST because no two messages share a key
├── SYMPTOM: Behaves like key=None but wastes key storage
└── FIX: Key should represent the ENTITY, not the EVENT
         key="G-1042" groups all events for that guest

📈 SCALABILITY CONSIDERATIONS

KEY CARDINALITY vs PARTITION COUNT:

  Ideal: key_cardinality >> partition_count
  → 200 unique guest IDs, 4 partitions → ~50 guests per partition
  → Good distribution, all partitions utilized

  Problematic: key_cardinality ≈ partition_count
  → 4 unique keys, 4 partitions → might not be 1:1
  → Hash function does NOT guarantee perfect distribution
  → Could be 2 keys → Partition 0, 1 key → Partition 2, 1 key → Partition 3

  Worst: key_cardinality < partition_count
  → 2 unique keys, 4 partitions → 2 partitions used, 2 empty
  → Wasted parallelism, unbalanced load

KEY SIZE IMPACT ON STORAGE:
  → Key is stored WITH every message on disk
  → key="G-1042" = 6 bytes per message
  → key="guest-registration-id-mr-anderson-room-412" = 46 bytes per message
  → At 1 million messages/day: 6MB vs 46MB just for keys
  → Keep keys SHORT but MEANINGFUL

HOT KEY SCALING STRATEGIES:
  → If one key dominates (e.g., "system-events"):
    Option A: Split into sub-keys ("system-events-shard-1", "system-events-shard-2")
    Option B: Use custom partitioner (Project 31)
    Option C: Move hot-key events to a dedicated topic
    Option D: Accept the skew if it is within throughput limits

🔍 INTERPRETABILITY CONSIDERATIONS

HOW DO YOU DEBUG KEY ROUTING?

1. ALWAYS LOG KEY + PARTITION IN PRODUCER CALLBACK
   → "key=G-1042 → partition=2, offset=47"
   → This is your audit trail for routing decisions

2. BUILD A KEY-TO-PARTITION MAP
   → Collect (key, partition) pairs from callback
   → Verify: each key maps to exactly ONE partition
   → Detect anomalies: same key appearing in multiple partitions

3. MEASURE PARTITION BALANCE
   → Count messages per partition over time
   → Alert if any partition has > 2x average message count
   → This detects hot partitions early

4. TEST DETERMINISM
   → Produce messages with known keys
   → Stop producer, restart, produce same keys
   → Verify: same keys → same partitions (they must!)

5. INSPECT KEYS IN CONSUMED MESSAGES
   → Consumer receives msg.key() alongside msg.value()
   → Verify key matches expected value
   → Detect null keys or corrupted keys

THE HOTEL EVENT PRODUCER — Key Strategy Comparison

# ═══════════════════════════════════════════════════════════
# file: hotel_producer.py
# Demonstrates key-based routing with multiple strategies
# ═══════════════════════════════════════════════════════════

from confluent_kafka import Producer
import json
import time
import random

config = {
    'bootstrap.servers': 'localhost:9092',
}

# ─────────────────────────────────────────────────────────
# KEY ROUTING TRACKER
# ─────────────────────────────────────────────────────────

key_partition_map = {}     # key → set of partitions seen
partition_counts = {}      # partition → message count
delivery_log = []          # full audit trail

def delivery_callback(err, msg):
    if err:
        print(f'❌ Failed: {err}')
        return
    
    key_str = msg.key().decode('utf-8') if msg.key() else 'NULL'
    p = msg.partition()
    o = msg.offset()
    
    # Track key-to-partition mapping
    if key_str not in key_partition_map:
        key_partition_map[key_str] = set()
    key_partition_map[key_str].add(p)
    
    # Track partition counts
    partition_counts[p] = partition_counts.get(p, 0) + 1
    
    # Audit log
    payload = json.loads(msg.value().decode('utf-8'))
    delivery_log.append({
        'key': key_str,
        'partition': p,
        'offset': o,
        'event_type': payload['event_type'],
        'guest_id': payload.get('guest_id', 'N/A'),
    })
    
    print(
        f'   ✅ key="{key_str:>12}" → '
        f'Partition {p} (offset {o:>3}) | '
        f'{payload["event_type"]}'
    )

producer = Producer(config)
topic = 'hotel-events'

# ─────────────────────────────────────────────────────────
# GUEST DATABASE (simulated)
# ─────────────────────────────────────────────────────────

guests = [
    {"guest_id": "G-1042", "name": "Mr. Anderson", "room": 412, "tier": "vip"},
    {"guest_id": "G-1043", "name": "Ms. Chen",     "room": 208, "tier": "normal"},
    {"guest_id": "G-1044", "name": "Dr. Smith",    "room": 315, "tier": "vip"},
    {"guest_id": "G-1045", "name": "Mrs. Johnson", "room": 101, "tier": "normal"},
    {"guest_id": "G-1046", "name": "Mr. Garcia",   "room": 507, "tier": "normal"},
    {"guest_id": "G-1047", "name": "Ms. Lee",      "room": 620, "tier": "vip"},
    {"guest_id": "G-1048", "name": "Mr. Brown",    "room": 412, "tier": "normal"},  # same room as Anderson later
    {"guest_id": "G-1049", "name": "Ms. Davis",    "room": 103, "tier": "normal"},
    {"guest_id": "G-1050", "name": "Mr. Wilson",   "room": 225, "tier": "normal"},
    {"guest_id": "G-1051", "name": "Mrs. Taylor",  "room": 318, "tier": "vip"},
]

# Event sequence for each guest (must be processed IN ORDER)
event_sequence = [
    "CHECK_IN",
    "ROOM_SERVICE_ORDER",
    "HOUSEKEEPING_REQUEST",
    "MINIBAR_CHARGE",
    "CHECK_OUT",
]

# ─────────────────────────────────────────────────────────
# STRATEGY 1: Using guest_id as key (CORRECT approach)
# ─────────────────────────────────────────────────────────

print("=" * 70)
print("📋 STRATEGY 1: key = guest_id (CORRECT)")
print("   Expectation: all events per guest → same partition")
print("=" * 70 + "\n")

key_partition_map.clear()
partition_counts.clear()
delivery_log.clear()

seq = 0
for guest in guests:
    for event_type in event_sequence:
        seq += 1
        message = {
            'sequence': seq,
            'guest_id': guest['guest_id'],
            'guest_name': guest['name'],
            'room': guest['room'],
            'tier': guest['tier'],
            'event_type': event_type,
            'details': f'{event_type} for {guest["name"]} in room {guest["room"]}',
            'timestamp': time.time(),
        }
        
        # ─────────────────────────────────────────────────
        # KEY = guest_id → deterministic, immutable, unique
        # ─────────────────────────────────────────────────
        producer.produce(
            topic=topic,
            key=guest['guest_id'].encode('utf-8'),
            value=json.dumps(message).encode('utf-8'),
            callback=delivery_callback,
        )
        producer.poll(0)
        time.sleep(0.05)

producer.flush()

# ─────────────────────────────────────────────────────────

HOT PARTITION DETECTOR

# ═══════════════════════════════════════════════════════════
# file: hot_partition_detector.py
# Detects uneven partition distribution caused by key skew
# ═══════════════════════════════════════════════════════════

from confluent_kafka import Producer
import json
import time
import random

config = {
    'bootstrap.servers': 'localhost:9092',
}

partition_counts = {}

def delivery_callback(err, msg):
    if err:
        return
    p = msg.partition()
    partition_counts[p] = partition_counts.get(p, 0) + 1

producer = Producer(config)

# ─────────────────────────────────────────────────────────
# SIMULATE: 80% of traffic comes from "walk-in" guests
# (a common real-world scenario: one key dominates)
# ─────────────────────────────────────────────────────────

print("🔥 HOT PARTITION SIMULATION")
print("   80% of guests are 'walk-in' (same key)")
print("   20% are registered (unique keys)\n")

num_messages = 200

for i in range(num_messages):
    if random.random() < 0.80:
        # 80% walk-in guests — all share same key
        key = "walk-in"
        guest_type = "walk-in"
    else:
        # 20% registered guests — unique keys
        key = f"G-{random.randint(1000, 9999)}"
        guest_type = "registered"
    
    message = {
        'guest_type': guest_type,
        'guest_key': key,
        'event': 'CHECK_IN',
        'timestamp': time.time(),
    }
    
    producer.produce(
        topic='hotel-events',
        key=key.encode('utf-8'),
        value=json.dumps(message).encode('utf-8'),
        callback=delivery_callback,
    )
    producer.poll(0)

producer.flush()

# ─────────────────────────────────────────────────────────
# HOT PARTITION ANALYSIS
# ─────────────────────────────────────────────────────────

print(f"\n📊 PARTITION DISTRIBUTION ({num_messages} messages):\n")

total = sum(partition_counts.values())
avg = total / max(len(partition_counts), 1)

hot_partitions = []
cold_partitions = []

for p in range(4):
    count = partition_counts.get(p, 0)
    pct = (count / total * 100) if total > 0 else 0
    bar = "█" * int(count / 2)
    
    if count > avg * 1.5:
        status = "🔥 HOT"
        hot_partitions.append(p)
    elif count < avg * 0.5:
        status = "🧊 COLD"
        cold_partitions.append(p)
    else:
        status = "✅ OK"
    
    print(f"   Partition {p}: {count:>4} messages ({pct:>5.1f}%) {bar} {status}")

print(f"\n   Average per partition: {avg:.1f}")
print(f"   Hot partitions:  {hot_partitions if hot_partitions else 'None'}")
print(f"   Cold partitions: {cold_partitions if cold_partitions else 'None'}")

if hot_partitions:
    print(f"""
   ⚠️  HOT PARTITION DETECTED!
   
   ROOT CAUSE: key="walk-in" used for 80% of messages
   → hash("walk-in") % 4 → always same partition
   → That partition receives ~80% of all traffic
   
   SOLUTIONS:
   1. Use unique guest ID even for walk-ins (generate UUID at check-in)
   2. Add shard suffix: "walk-in-1", "walk-in-2", "walk-in-3"
   3. Custom partitioner to spread hot keys (Project 31)
   4. Accept the skew if throughput is within limits
""")
else:
    print(f"\n   ✅ No hot partitions detected — distribution is healthy")

    key_len = len(msg.key()) if msg.key() is not None else -1

        'key_length': key_len,
