PROJECT 5: THE CALL CENTER QUEUE — CONSUMER GROUPS

🎯 HYPOTHETICAL PROBLEM

You run "ByteCall Support Center" — a customer support hotline

→ Thousands of customers call every hour with issues
→ Each call is logged as a support ticket in a Kafka topic
→ Support agents process tickets one by one

CURRENT SETUP (from Projects 1-4):
→ ONE support agent handles ALL tickets
→ Topic "support-tickets" has 4 partitions (from Project 3 learning)
→ Tickets are keyed by customer_id (from Project 4 learning)
→ One agent reads from all 4 partitions

HERE IS THE PROBLEM:

→ Agent Alice processes 50 tickets/hour
→ 200 tickets/hour are arriving
→ Alice falls behind by 150 tickets/hour
→ By end of day: 1200 ticket BACKLOG
→ Customers are furious — 24-hour response time

THE MANAGER'S SOLUTION:
→ Hire 3 more agents: Bob, Charlie, Diana
→ 4 agents total, 4 partitions total
→ Each agent handles ONE partition

  Partition 0 → Alice   (handles ~50 tickets/hour)
  Partition 1 → Bob     (handles ~50 tickets/hour)
  Partition 2 → Charlie (handles ~50 tickets/hour)
  Partition 3 → Diana   (handles ~50 tickets/hour)
  
  Total capacity: 200 tickets/hour ✅ matches incoming rate

BUT NEW QUESTIONS:
→ What if Diana calls in sick? Who handles Partition 3?
→ What if we hire a 5th agent, Eve? What does she do?
→ What if Alice quits and rejoins? Does she get the same partition?
→ How do agents COORDINATE without talking to each other?
→ Can a SECOND team (e.g., quality auditors) read the SAME tickets?

THIS IS THE CONSUMER GROUP PROBLEM.

🔍 HOW TO APPROACH IT

WHAT WE ALREADY KNOW:
→ Topics have multiple partitions (Project 3) ✅
→ Keys route messages to specific partitions (Project 4) ✅
→ One consumer can read from all partitions (Projects 1-4) ✅
→ Offsets track consumption progress per partition (Project 2) ✅

WHAT IS NEW:
→ MULTIPLE consumers can form a GROUP to share the workload
→ Kafka AUTOMATICALLY assigns partitions to consumers in a group
→ Each partition is assigned to EXACTLY ONE consumer in a group
→ If a consumer dies, its partitions are REASSIGNED to survivors
→ DIFFERENT groups read the SAME topic INDEPENDENTLY

MAPPING:
  Call Center Team    = Consumer Group
  Support Agent       = Consumer (within the group)
  Phone Line          = Partition
  Ticket              = Message
  Agent's Ticket Log  = Committed Offset

  Team "support-agents" (group.id):
    Alice  → assigned Partition 0
    Bob    → assigned Partition 1
    Charlie → assigned Partition 2
    Diana  → assigned Partition 3

  Team "quality-audit" (different group.id):
    Auditor-1 → assigned Partitions 0, 1
    Auditor-2 → assigned Partitions 2, 3

  BOTH teams read the SAME messages independently

APPROACH:
Step 1: Create 4 consumers with the SAME group.id
Step 2: Observe automatic partition assignment
Step 3: Kill one consumer — observe partition reassignment
Step 4: Add a 5th consumer — observe it sits idle
Step 5: Create a second group — observe independent reading
Step 6: Prove offset independence between groups

🛠️ HOW CAN WE SOLVE IT

SOLUTION DESIGN:

  Topic: support-tickets (4 partitions)
  
  ┌──────────────────────────────────────────────────────────────────────┐
  │                         KAFKA BROKER                                 │
  │                                                                      │
  │  Partition 0: [t0] [t1] [t2] [t3] [t4] ...                          │
  │  Partition 1: [t0] [t1] [t2] [t3] ...                               │
  │  Partition 2: [t0] [t1] [t2] [t3] [t4] [t5] ...                     │
  │  Partition 3: [t0] [t1] [t2] ...                                     │
  │                                                                      │
  └────────┬──────────┬──────────┬──────────┬────────────────────────────┘
           │          │          │          │
           ▼          ▼          ▼          ▼
  ┌─────────────────────────────────────────────────────┐
  │     CONSUMER GROUP: "support-agents"                │
  │                                                     │
  │  Alice      Bob        Charlie     Diana            │
  │  (Part 0)   (Part 1)   (Part 2)   (Part 3)         │
  │  offset=4   offset=3   offset=5   offset=2         │
  └─────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────┐
  │     CONSUMER GROUP: "quality-audit"                 │
  │                                                     │
  │  Auditor-1           Auditor-2                      │
  │  (Part 0, Part 1)    (Part 2, Part 3)               │
  │  offset=4, offset=3  offset=5, offset=2             │
  └─────────────────────────────────────────────────────┘

  KEY RULES:
  → Within a group: each partition → exactly ONE consumer
  → Within a group: each consumer → one or more partitions
  → Across groups: completely independent (own offsets, own assignment)
  → Max useful consumers per group = partition count

✅ WHAT COULD BE DONE

Option A: Run 4 independent consumers (each with unique group.id)
          → Each reads ALL 4 partitions independently
          → No load sharing — 4x redundant work
          → NOT what we want for parallel processing

Option B: Run 4 consumers with SAME group.id ✅
          → Kafka automatically splits partitions among them
          → Each processes ~25% of the workload
          → Automatic failover if one dies
          → THIS IS THE CONSUMER GROUP PATTERN

Option C: Manually assign partitions to consumers
          → Possible via assign() instead of subscribe()
          → No automatic rebalancing — you manage everything
          → Use only for special cases (Project 10 discusses this)

✅ WHAT SHOULD DO

1. ALWAYS use the SAME group.id for consumers that should SHARE work
   → group.id="support-agents" for all 4 agents
   → Kafka uses group.id to coordinate partition assignment

2. ALWAYS use DIFFERENT group.id for independent consumption
   → group.id="support-agents" for processing
   → group.id="quality-audit" for auditing
   → Each group maintains its OWN offset position

3. ALWAYS plan: partition count ≥ maximum expected consumers
   → 4 partitions → maximum 4 useful consumers per group
   → If you might need 8 agents later → create 8 partitions now
   → You CAN increase partitions later (but see Project 3 warnings about keys)

4. ALWAYS let Kafka handle partition assignment (use subscribe())
   → subscribe() enables automatic rebalancing
   → Kafka's group coordinator manages everything
   → Manual assignment (assign()) disables this — avoid unless needed

5. ALWAYS give each consumer instance a UNIQUE client.id (optional but helpful)
   → client.id="agent-alice", client.id="agent-bob"
   → Appears in broker logs for debugging
   → NOT the same as group.id (group.id is shared, client.id is unique)

6. ALWAYS handle the consumer lifecycle: subscribe → poll loop → close
   → close() triggers a graceful leave from the group
   → Without close(), Kafka waits for session timeout before reassigning

❌ WHAT SHOULD NOT DO

1. DO NOT create more consumers than partitions in the same group
   → 5 consumers for 4 partitions → 1 consumer sits IDLE
   → That idle consumer wastes resources — it receives ZERO messages
   → This is the MOST common consumer group mistake

   ┌──────────────────────────────────────────────────────┐
   │ 4 partitions, 5 consumers in same group:             │
   │                                                      │
   │ Alice   → Partition 0  ✅ working                     │
   │ Bob     → Partition 1  ✅ working                     │
   │ Charlie → Partition 2  ✅ working                     │
   │ Diana   → Partition 3  ✅ working                     │
   │ Eve     → (nothing)    ❌ IDLE — wasting resources    │
   └──────────────────────────────────────────────────────┘

2. DO NOT use the same group.id for unrelated applications
   → If "support-agents" and "billing-system" share a group.id
   → They SPLIT the partitions — each sees only HALF the messages
   → This is a silent, devastating bug

3. DO NOT expect a consumer to get the SAME partition after restart
   → Partition assignment is DYNAMIC
   → Alice might get Partition 0 first time, Partition 3 next time
   → Your application must NOT depend on specific partition assignment

4. DO NOT forget to call consumer.close() on shutdown
   → Without close(): Kafka waits for session.timeout.ms (default 45s)
   → During that wait, the dead consumer's partitions are unprocessed
   → With close(): Kafka immediately reassigns — near-zero downtime

5. DO NOT mix subscribe() and assign() on the same consumer
   → subscribe() = automatic group management
   → assign() = manual partition control (no group coordination)
   → Using both causes unpredictable behavior

6. DO NOT assume partition assignment is instantaneous
   → When a consumer joins/leaves, a REBALANCE occurs
   → During rebalance, ALL consumers in the group PAUSE briefly
   → This is called "stop-the-world" rebalance (improved in Project 10)

🔀 WHEN TO USE WHAT

🚫 WHEN NOT TO USE

→ DO NOT use consumer groups when every consumer needs ALL messages
  (use different group.ids instead — each group gets all messages)

→ DO NOT use consumer groups for exactly-once processing without
  additional guarantees (consumer groups provide at-least-once by default)
  (exactly-once covered in Projects 17-18)

→ DO NOT use consumer groups when you need to control WHICH consumer
  reads WHICH partition permanently (use assign() for static assignment)
  
→ DO NOT use consumer groups with a single partition
  (only 1 consumer can be active — no parallelism benefit)

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

═══════════════════════════════════════════════════════════════
THE GROUP COORDINATOR — THE BRAIN OF CONSUMER GROUPS
═══════════════════════════════════════════════════════════════

Every consumer group has a GROUP COORDINATOR:
→ A specific Kafka broker that manages the group
→ Selected based on: hash(group.id) % __consumer_offsets partitions

The coordinator handles:
  1. Consumer membership (who is in the group?)
  2. Partition assignment (who reads what?)
  3. Offset storage (how far has each consumer read?)
  4. Heartbeat monitoring (is each consumer still alive?)

HOW THE COORDINATOR IS SELECTED:

  __consumer_offsets topic has 50 partitions (default)
  
  coordinator_partition = abs(hash("support-agents")) % 50
  → Let's say = partition 23
  
  The broker that LEADS partition 23 of __consumer_offsets
  becomes the group coordinator for "support-agents"


═══════════════════════════════════════════════════════════════
CONSUMER LIFECYCLE: JOIN → ASSIGN → CONSUME → LEAVE
═══════════════════════════════════════════════════════════════

STEP 1: FIND COORDINATOR
─────────────────────────
  Consumer sends FindCoordinatorRequest(group="support-agents")
  → Any broker can handle this
  → Returns: "Broker 2 is your coordinator"

STEP 2: JOIN GROUP
──────────────────
  Consumer sends JoinGroupRequest to coordinator
  → "Hi, I'm agent-alice, I want to join support-agents"
  → Coordinator collects all JoinGroupRequests
  → Waits for a short period (group.initial.rebalance.delay.ms)
  → Then initiates a REBALANCE

  ┌─────────────────────────────────────────────────────────────┐
  │  COORDINATOR receives JoinGroupRequests:                    │
  │                                                             │
  │  agent-alice:   "I can handle topics: [support-tickets]"    │
  │  agent-bob:     "I can handle topics: [support-tickets]"    │
  │  agent-charlie: "I can handle topics: [support-tickets]"    │
  │  agent-diana:   "I can handle topics: [support-tickets]"    │
  │                                                             │
  │  Coordinator picks ONE consumer as GROUP LEADER              │
  │  (usually the first to join)                                │
  │                                                             │
  │  → agent-alice becomes GROUP LEADER                          │
  └─────────────────────────────────────────────────────────────┘

STEP 3: PARTITION ASSIGNMENT (by the group leader)
──────────────────────────────────────────────────
  → Coordinator sends member list to the GROUP LEADER
  → The GROUP LEADER (not the coordinator!) decides assignment
  → Leader uses a PARTITION ASSIGNOR strategy

  Default strategy: RangeAssignor
  
  Topic "support-tickets" has partitions: [0, 1, 2, 3]
  Members: [alice, bob, charlie, diana]
  
  RangeAssignor logic:
    partitions_per_consumer = 4 / 4 = 1
    alice   → [Partition 0]
    bob     → [Partition 1]
    charlie → [Partition 2]
    diana   → [Partition 3]

  Leader sends SyncGroupRequest with this assignment to coordinator
  Coordinator forwards assignment to ALL members

STEP 4: SYNC AND START CONSUMING
─────────────────────────────────
  Each consumer receives its SyncGroupResponse:
  → "alice, you are assigned Partition 0"
  → "bob, you are assigned Partition 1"
  → etc.

  Each consumer:
  → Fetches committed offset for assigned partition(s)
  → Starts polling from that offset
  → Begins processing messages

STEP 5: HEARTBEAT LOOP
───────────────────────
  While consuming, each consumer sends periodic HEARTBEATS:
  → HeartbeatRequest every heartbeat.interval.ms (default 3s)
  → Coordinator responds with HeartbeatResponse
  → If coordinator says "REBALANCE_IN_PROGRESS" → consumer rejoins

  ┌──────────────┐     heartbeat (3s)     ┌─────────────┐
  │   Consumer   │ ─────────────────────→ │ Coordinator │
  │   (alice)    │ ←───────────────────── │             │
  │              │     "OK" or            │             │
  │              │     "REBALANCE"        │             │
  └──────────────┘                        └─────────────┘

STEP 6: CONSUMER LEAVES OR DIES
───────────────────────────────

  GRACEFUL LEAVE (consumer.close()):
  → Consumer sends LeaveGroupRequest to coordinator
  → Coordinator immediately triggers rebalance
  → Remaining consumers get reassigned partitions
  → Downtime: milliseconds to seconds

  UNGRACEFUL DEATH (crash/network failure):
  → Heartbeats stop arriving
  → Coordinator waits for session.timeout.ms (default 45s)
  → After timeout: coordinator marks consumer as dead
  → Triggers rebalance — remaining consumers take over
  → Downtime: up to 45 seconds!

  THIS IS WHY consumer.close() IS CRITICAL
  → Graceful: near-instant failover
  → Crash: up to 45 seconds of stalled consumption


═══════════════════════════════════════════════════════════════
PARTITION ASSIGNMENT STRATEGIES (Built-in)
═══════════════════════════════════════════════════════════════

STRATEGY 1: RangeAssignor (Default)
────────────────────────────────────
  → Sorts partitions and consumers alphabetically
  → Assigns contiguous ranges

  4 partitions, 3 consumers:
    alice   → [Partition 0, Partition 1]  ← gets extra
    bob     → [Partition 2]
    charlie → [Partition 3]

  4 partitions, 2 consumers:
    alice   → [Partition 0, Partition 1]
    bob     → [Partition 2, Partition 3]

  Problem: First consumer(s) always get extra partitions
           across multiple topics → unfair

STRATEGY 2: RoundRobinAssignor
──────────────────────────────
  → Assigns partitions in round-robin across consumers

  4 partitions, 3 consumers:
    alice   → [Partition 0, Partition 3]
    bob     → [Partition 1]
    charlie → [Partition 2]

  More balanced than Range across multiple topics

STRATEGY 3: CooperativeStickyAssignor (Recommended — Project 10)
────────────────────────────────────────────────────────────────
  → Tries to keep previous assignments stable
  → Only moves partitions that NEED to move
  → Does NOT stop all consumers during rebalance
  → Incremental rebalancing — much better for production


═══════════════════════════════════════════════════════════════
REBALANCE SCENARIOS
═══════════════════════════════════════════════════════════════

SCENARIO A: New consumer joins
──────────────────────────────
  BEFORE:                          AFTER:
  alice   → [P0, P1]              alice   → [P0]
  bob     → [P2, P3]              bob     → [P1]
                                  charlie → [P2]    ← NEW
                                  diana   → [P3]    ← NEW

  All consumers PAUSE during rebalance
  New assignment distributed
  All consumers RESUME

SCENARIO B: Consumer leaves gracefully
──────────────────────────────────────
  BEFORE:                          AFTER:
  alice   → [P0]                  alice   → [P0, P3]  ← took over
  bob     → [P1]                  bob     → [P1]
  charlie → [P2]                  charlie → [P2]
  diana   → [P3]  ← LEAVES

  diana.close() → LeaveGroupRequest → immediate rebalance

SCENARIO C: Consumer crashes
────────────────────────────
  BEFORE:                          AFTER (45 seconds later):
  alice   → [P0]                  alice   → [P0, P3]  ← took over
  bob     → [P1]                  bob     → [P1]
  charlie → [P2]                  charlie → [P2]
  diana   → [P3]  ← CRASHES

  Heartbeats stop → session.timeout.ms expires → rebalance

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "I have 4 consumers but only 1 is receiving messages"
├── CAUSE A: All 4 have DIFFERENT group.ids (not sharing work)
│   └── FIX: Ensure ALL consumers use the SAME group.id
├── CAUSE B: Topic has only 1 partition
│   └── FIX: Create topic with more partitions (≥ consumer count)
└── CAUSE C: Other 3 consumers subscribed to wrong topic name
    └── FIX: Verify topic name is identical across all consumers

PITFALL 2: "5th consumer is not receiving any messages"
├── CAUSE: 4 partitions, 5 consumers → 1 must be idle
├── THIS IS CORRECT BEHAVIOR
├── FIX A: Increase partitions to 5+ (careful with key routing — Project 4)
└── FIX B: Remove the 5th consumer (don't waste resources)

PITFALL 3: "Consumer rejoins but gets different partition than before"
├── CAUSE: Partition assignment is dynamic — no guarantees
├── THIS IS BY DESIGN
├── FIX: Do NOT store partition-specific state locally
│         OR use CooperativeStickyAssignor for stable assignment
└── DEEP DIVE: Project 10 (Rebalancing)

PITFALL 4: "Two different applications accidentally share group.id"
├── CAUSE: Copy-pasted configuration without changing group.id
├── SYMPTOM: Each application sees only HALF the messages
│   → App A gets Partitions 0, 1
│   → App B gets Partitions 2, 3
│   → Both think they are missing data
├── THIS IS SILENT AND DEVASTATING
└── FIX: Use descriptive, unique group.ids per application
         "support-ticket-processor-v1", "support-ticket-auditor-v1"

PITFALL 5: "Long rebalance delays when consumer crashes"
├── CAUSE: Default session.timeout.ms = 45 seconds
├── WHY: Kafka waits 45s for heartbeats before declaring death
├── SYMPTOM: Partition 3 unprocessed for 45 seconds after crash
└── FIX: Reduce session.timeout.ms (minimum 6s with default heartbeat 3s)
         TRADEOFF: Too low → false positives (slow GC pauses trigger timeout)

PITFALL 6: "All consumers stop processing during rebalance"
├── CAUSE: Default "eager" rebalance protocol
├── WHY: All partitions revoked, then reassigned
├── SYMPTOM: Brief processing gap every time a consumer joins/leaves
└── FIX: Use CooperativeStickyAssignor (Project 10) — incremental rebalance

📈 SCALABILITY CONSIDERATIONS

CONSUMER GROUP SCALING MODEL:

  ┌───────────────────────────────────────────────────────────────────┐
  │ Partitions │ Consumers │ Assignment            │ Efficiency       │
  ├────────────┼───────────┼───────────────────────┼──────────────────┤
  │ 4          │ 1         │ 1 consumer gets all 4 │ No parallelism   │
  │ 4          │ 2         │ Each gets 2           │ 2x throughput    │
  │ 4          │ 4         │ Each gets 1           │ 4x throughput ✅ │
  │ 4          │ 6         │ 4 get 1, 2 IDLE      │ Wasted resources │
  │ 8          │ 4         │ Each gets 2           │ Room to grow     │
  │ 8          │ 8         │ Each gets 1           │ 8x throughput ✅ │
  │ 12         │ 4         │ Each gets 3           │ Room to grow     │
  └───────────────────────────────────────────────────────────────────┘

PLANNING GUIDELINE:
  → partition_count = max_expected_consumers × 1.5 (growth room)
  → If you expect up to 8 consumers → create 12 partitions
  → Overprovisioning partitions is safer than underprovisioning

MULTIPLE GROUPS ON SAME TOPIC:
  → Each group adds independent read load on the broker
  → 3 groups × 4 consumers = 12 consumer connections
  → Broker handles fetch requests from all 12
  → Data is read from PAGE CACHE (not disk) if recently written
  → Multiple groups reading same data is CHEAP (data read once into cache)

GROUP COORDINATION OVERHEAD:
  → Heartbeats: 1 request per consumer every 3 seconds
  → 100 consumers = ~33 heartbeat requests/second
  → Negligible for broker (network overhead is minimal)
  → Rebalance overhead scales with number of consumers
  → 100 consumers rebalancing = noticeable pause (~5-15 seconds)

🔍 INTERPRETABILITY CONSIDERATIONS

HOW DO YOU MONITOR CONSUMER GROUPS?

1. kafka-consumer-groups CLI (most important tool)
   → Shows all groups, members, assigned partitions
   → Shows LAG per partition (how far behind each consumer is)
   → docker exec -it kafka kafka-consumer-groups \
       --bootstrap-server localhost:9092 \
       --describe --group support-agents

   Output:
   GROUP          TOPIC            PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG
   support-agents support-tickets  0          47              50              3
   support-agents support-tickets  1          32              32              0
   support-agents support-tickets  2          28              35              7
   support-agents support-tickets  3          15              15              0

   → LAG = LOG-END-OFFSET - CURRENT-OFFSET
   → LAG 0 = consumer caught up ✅
   → LAG growing = consumer falling behind ⚠️

2. GROUP STATE
   → Stable: all consumers active, partitions assigned
   → Rebalancing: consumers joining/leaving, partitions being reassigned
   → Empty: no active consumers (all dead or stopped)
   → Dead: group has no members and no offsets

3. CONSUMER ASSIGNMENT VERIFICATION
   → Check that each consumer has expected partition count
   → Check that no partitions are unassigned
   → Check that no consumer is idle (0 partitions)
