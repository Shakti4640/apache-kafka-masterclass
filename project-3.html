PROJECT 3: THE MAILBOX SYSTEM — UNDERSTANDING PARTITIONS

🎯 HYPOTHETICAL PROBLEM

You manage a large apartment building called "ByteView Towers"
→ 500 residents live here
→ Every day, hundreds of letters and packages arrive
→ There is ONE mailbox counter with ONE mail sorter

HERE IS THE PROBLEM:
→ One mail sorter handles ALL 500 residents
→ During holiday season, 2000 letters arrive per hour
→ The single sorter becomes a BOTTLENECK
→ Residents wait hours to get their mail
→ The line at the counter wraps around the building

THE MANAGER'S IDEA:
→ Install 4 separate mailbox counters (A, B, C, D)
→ Hire 4 mail sorters — one per counter
→ Assign residents to counters by FLOOR:
    Floor 1-3   → Counter A
    Floor 4-6   → Counter B
    Floor 7-9   → Counter C
    Floor 10-12 → Counter D

→ Now 4 sorters work IN PARALLEL
→ Each handles ~125 residents instead of 500
→ Throughput is 4x higher

BUT WAIT — NEW QUESTIONS:
→ If Alice (Floor 2) sends two letters, do they arrive in order?
   → YES — both go to Counter A → order preserved within counter
→ If Alice (Floor 2) and Bob (Floor 7) send letters simultaneously,
   which arrives first?
   → UNKNOWN — they go to different counters (A vs C)
   → Order across counters is NOT guaranteed
→ Can we add a 5th sorter if traffic grows?
   → Only if we add a 5th counter — sorter without counter is useless

THIS IS EXACTLY HOW KAFKA PARTITIONS WORK.

🔍 HOW TO APPROACH IT

WHAT WE ALREADY KNOW:
→ A topic is an append-only log (Project 1) ✅
→ Messages within a single log are ordered (Project 2) ✅
→ Offsets increment sequentially (Project 2) ✅

WHAT IS NEW:
→ A topic can be SPLIT into multiple append-only logs
→ Each split is called a PARTITION
→ Each partition has its OWN independent offset sequence
→ Messages are DISTRIBUTED across partitions
→ Ordering is guaranteed ONLY within a partition, NOT across

MAPPING:
  Apartment Building = Kafka Topic
  Mailbox Counter    = Partition
  Mail Sorter        = Consumer (one per partition per group)
  Resident's Floor   = Message Key (determines which partition)
  Letter             = Message

APPROACH:
Step 1: Create a topic with 4 partitions
Step 2: Produce messages WITHOUT keys → observe round-robin distribution
Step 3: Produce messages WITH keys → observe deterministic partition assignment
Step 4: Consume from all partitions → observe per-partition ordering
Step 5: Prove that cross-partition ordering is NOT guaranteed
Step 6: Visualize the offset-per-partition model

🛠️ HOW CAN WE SOLVE IT

SOLUTION DESIGN:

  Topic: mailbox-deliveries (4 partitions)

  ┌──────────────────────────────────────────────────────────┐
  │                   KAFKA BROKER                            │
  │                                                          │
  │  Partition 0:  [off=0] [off=1] [off=2] [off=3]          │
  │  Partition 1:  [off=0] [off=1]                           │
  │  Partition 2:  [off=0] [off=1] [off=2]                   │
  │  Partition 3:  [off=0] [off=1] [off=2] [off=3] [off=4]  │
  │                                                          │
  └──────────────────────────────────────────────────────────┘

  CRITICAL OBSERVATIONS:
  → Each partition has its OWN offset starting from 0
  → Partition 0 has 4 messages (offsets 0-3)
  → Partition 1 has 2 messages (offsets 0-1)
  → Partition 3 has 5 messages (offsets 0-4)
  → Offsets are NOT shared across partitions
  → "Offset 2 in Partition 0" and "Offset 2 in Partition 2"
    are COMPLETELY DIFFERENT messages

  TO UNIQUELY IDENTIFY A MESSAGE:
  → You need THREE things: (topic, partition, offset)
  → NOT just offset alone — that was only true with 1 partition

✅ WHAT COULD BE DONE

Option A: Keep using 1 partition forever
          → Ordering is simple and guaranteed
          → BUT: Maximum 1 consumer per group — no parallelism
          → BUT: Single disk bottleneck — limited throughput

Option B: Create many partitions (e.g., 100)
          → Maximum parallelism — 100 consumers can work simultaneously
          → BUT: More metadata overhead on broker
          → BUT: More file handles open on disk
          → BUT: Ordering becomes harder to reason about

Option C: Choose partition count based on expected throughput ✅
          → Rule of thumb: partitions ≈ expected consumer count
          → 4 consumers needed → 4 partitions
          → This balances parallelism with manageability
          → Can increase later (but NOT decrease — Kafka limitation)

✅ WHAT SHOULD DO

1. ALWAYS choose partition count THOUGHTFULLY at topic creation
   → You CAN increase partitions later
   → You CANNOT decrease partitions — ever
   → Start with a reasonable number (4-12 for most use cases)

2. ALWAYS use message KEYS when ordering matters for a specific entity
   → All messages with the same key go to the SAME partition
   → Same partition = ordering guaranteed for that key
   → Example: all mail for "Floor-3" always goes to Partition 0

3. ALWAYS think of offset as (partition, offset) pair — NOT just offset
   → "Offset 5" is meaningless without knowing WHICH partition
   → This mental model prevents confusion in every future project

4. ALWAYS understand: partition count = maximum parallelism
   → 4 partitions = maximum 4 consumers in one group
   → 10 partitions = maximum 10 consumers in one group
   → More consumers than partitions → extra consumers sit idle

5. ALWAYS verify partition distribution after producing
   → Use kafka-console-consumer with --partition flag
   → Or use the admin API to check partition message counts

❌ WHAT SHOULD NOT DO

1. DO NOT assume offset is globally unique across partitions
   → Offset 3 in Partition 0 ≠ Offset 3 in Partition 1
   → This is the #1 confusion for Kafka beginners

2. DO NOT create thousands of partitions "just in case"
   → Each partition = directory on disk + open file handles
   → Each partition = metadata tracked by controller
   → Excessive partitions slow down broker startup and recovery
   → Rule: start small, scale when needed

3. DO NOT expect ordering across partitions
   → If you produce seq=1 to Partition 0 and seq=2 to Partition 1
   → Consumer might read seq=2 BEFORE seq=1
   → This is BY DESIGN — not a bug

4. DO NOT try to decrease partition count after creation
   → Kafka does not support this
   → You would need to create a new topic with fewer partitions
     and migrate data

5. DO NOT ignore the key-to-partition relationship
   → Without a key: round-robin (or sticky) distribution
   → With a key: hash(key) % partition_count = target partition
   → Changing partition count CHANGES where keys land (rebalance chaos)

🔀 WHEN TO USE WHAT

┌─────────────────────────────────┬─────────────────────────────────────────┐
│ USE 1 PARTITION WHEN            │ USE MULTIPLE PARTITIONS WHEN            │
├─────────────────────────────────┼─────────────────────────────────────────┤
│ Strict GLOBAL ordering is       │ You need PARALLEL consumption           │
│ absolutely required             │ (multiple consumers in one group)       │
├─────────────────────────────────┼─────────────────────────────────────────┤
│ Throughput is low               │ Throughput needs to scale beyond        │
│ (< 1000 msgs/sec)              │ single partition limits                 │
├─────────────────────────────────┼─────────────────────────────────────────┤
│ Only 1 consumer ever needed     │ Multiple consumers needed               │
│ per consumer group              │ per consumer group                      │
└─────────────────────────────────┴─────────────────────────────────────────┘

┌─────────────────────────────────┬─────────────────────────────────────────┐
│ PRODUCE WITHOUT KEY WHEN        │ PRODUCE WITH KEY WHEN                   │
├─────────────────────────────────┼─────────────────────────────────────────┤
│ You want EVEN distribution      │ You need ordering per entity            │
│ across partitions (load balance)│ (all orders for user-123 → same part)  │
├─────────────────────────────────┼─────────────────────────────────────────┤
│ No entity-level ordering needed │ Related messages must stay together     │
│ (independent log events)        │ (user events, device readings)          │
└─────────────────────────────────┴─────────────────────────────────────────┘

🚫 WHEN NOT TO USE

→ DO NOT use multiple partitions when you need a single total order
  of ALL messages (use 1 partition — accept the throughput limit)

→ DO NOT use keyed messages when even distribution matters more
  than ordering (keys can cause SKEW — some partitions get more messages)

→ DO NOT increase partitions on a keyed topic without understanding
  that ALL key-to-partition mappings will CHANGE
  (existing key "user-123" may now go to a different partition)

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

═══════════════════════════════════════════════════════════════
PARTITION = INDEPENDENT APPEND-ONLY LOG
═══════════════════════════════════════════════════════════════

When you create a topic with 4 partitions:

  kafka-topics --create --topic mailbox --partitions 4

Kafka creates 4 SEPARATE directories on disk:

  /tmp/kafka-logs/
  ├── mailbox-0/           ← Partition 0
  │   ├── 00000000000000000000.log
  │   ├── 00000000000000000000.index
  │   └── 00000000000000000000.timeindex
  ├── mailbox-1/           ← Partition 1
  │   ├── 00000000000000000000.log
  │   ├── 00000000000000000000.index
  │   └── 00000000000000000000.timeindex
  ├── mailbox-2/           ← Partition 2
  │   └── ...
  └── mailbox-3/           ← Partition 3
      └── ...

Each directory IS a partition.
Each partition IS an independent log.
Each log HAS its own offset counter starting from 0.


═══════════════════════════════════════════════════════════════
HOW MESSAGES ARE ASSIGNED TO PARTITIONS
═══════════════════════════════════════════════════════════════

CASE 1: NO KEY provided (key=None)
──────────────────────────────────

  Kafka uses the DEFAULT PARTITIONER strategy:
  
  → Kafka 2.4+: "Sticky Partitioner"
    • Fills one partition's batch first
    • Then switches to next partition
    • Reduces number of broker requests
    • Messages are roughly evenly distributed over time
  
  → Before Kafka 2.4: "Round Robin"
    • Message 1 → Partition 0
    • Message 2 → Partition 1
    • Message 3 → Partition 2
    • Message 4 → Partition 3
    • Message 5 → Partition 0 (wraps around)

  RESULT: Even distribution, NO ordering guarantee for any entity


CASE 2: KEY provided (key="floor-3")
─────────────────────────────────────

  Kafka computes: partition = hash(key) % number_of_partitions

  STEP BY STEP:
  1. Serialize the key to bytes: "floor-3" → bytes
  2. Apply murmur2 hash: murmur2(bytes) → integer (e.g., 1234567)
  3. Take absolute value: abs(1234567) → 1234567
  4. Modulo partition count: 1234567 % 4 → 3
  5. Target partition: Partition 3

  CRITICAL PROPERTIES:
  → DETERMINISTIC: same key ALWAYS produces same hash
  → "floor-3" will ALWAYS go to Partition 3 (as long as partition count = 4)
  → "floor-3" today, tomorrow, next year → same partition
  → This is WHY ordering per key is guaranteed

  THE FORMULA:
  ┌─────────────────────────────────────────────────────────┐
  │  target_partition = abs(murmur2(key_bytes)) % num_parts │
  └─────────────────────────────────────────────────────────┘

  EXAMPLE WITH 4 PARTITIONS:
  ┌──────────────┬──────────────────┬─────────────────────┐
  │ Key          │ murmur2 hash     │ hash % 4 = Partition│
  ├──────────────┼──────────────────┼─────────────────────┤
  │ "floor-1"    │ ...some number   │ 2                   │
  │ "floor-2"    │ ...some number   │ 0                   │
  │ "floor-3"    │ ...some number   │ 3                   │
  │ "floor-4"    │ ...some number   │ 1                   │
  │ "floor-5"    │ ...some number   │ 2  ← same as floor-1│
  └──────────────┴──────────────────┴─────────────────────┘

  NOTE: Multiple keys CAN map to the same partition
  → "floor-1" and "floor-5" both → Partition 2
  → This is normal — hash collisions are expected
  → Their messages will be INTERLEAVED within Partition 2
  → But ALL "floor-1" messages stay in Partition 2 (ordered)
  → And ALL "floor-5" messages stay in Partition 2 (ordered)


═══════════════════════════════════════════════════════════════
WHAT HAPPENS WHEN PARTITION COUNT CHANGES
═══════════════════════════════════════════════════════════════

  BEFORE: 4 partitions
  "floor-3" → hash % 4 → Partition 3

  AFTER: 6 partitions (added 2 more)
  "floor-3" → hash % 6 → Partition 1  ← DIFFERENT!

  THIS BREAKS KEY-TO-PARTITION CONSISTENCY
  → Old messages for "floor-3" are in Partition 3
  → New messages for "floor-3" go to Partition 1
  → A consumer reading Partition 3 misses new "floor-3" messages
  → A consumer reading Partition 1 misses old "floor-3" messages

  THIS IS WHY:
  → Choose partition count carefully upfront
  → Increasing partitions on a keyed topic is DANGEROUS
  → Some systems create topics with extra partitions for future growth


═══════════════════════════════════════════════════════════════
ORDERING PROOF: WITHIN PARTITION vs ACROSS PARTITIONS
═══════════════════════════════════════════════════════════════

  Producer sends in this order:
  T=1: msg-A (key=null) → Partition 0, offset 0
  T=2: msg-B (key=null) → Partition 1, offset 0
  T=3: msg-C (key=null) → Partition 0, offset 1
  T=4: msg-D (key=null) → Partition 1, offset 1

  Partition 0: [msg-A (off=0)] [msg-C (off=1)]
  Partition 1: [msg-B (off=0)] [msg-D (off=1)]

  WITHIN Partition 0: msg-A ALWAYS comes before msg-C ✅
  WITHIN Partition 1: msg-B ALWAYS comes before msg-D ✅

  ACROSS Partitions: Consumer might read in any interleaving:
  → msg-A, msg-B, msg-C, msg-D  (lucky — matches produce order)
  → msg-B, msg-A, msg-D, msg-C  (different — but valid!)
  → msg-A, msg-C, msg-B, msg-D  (Partition 0 fully, then Partition 1)

  ALL of these are VALID consumption orders
  → Kafka makes NO guarantee about cross-partition ordering


═══════════════════════════════════════════════════════════════
HOW CONSUMER READS FROM MULTIPLE PARTITIONS
═══════════════════════════════════════════════════════════════

  Single consumer subscribed to a 4-partition topic:

  Consumer maintains 4 INDEPENDENT fetch positions:
  ┌──────────────────────────────────────────────┐
  │ Consumer Internal State:                     │
  │   Partition 0 → next_offset = 5              │
  │   Partition 1 → next_offset = 3              │
  │   Partition 2 → next_offset = 7              │
  │   Partition 3 → next_offset = 2              │
  └──────────────────────────────────────────────┘

  Each poll() call:
  → Fetches batches from ALL assigned partitions
  → Returns messages from potentially MULTIPLE partitions
  → Messages within each partition batch are ordered
  → But the interleaving between partition batches is arbitrary

  This is why poll() returns ONE message at a time (confluent-kafka)
  → Under the hood, it pre-fetches batches
  → Delivers them to you one by one
  → Within-partition order is maintained in delivery

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "I added partitions but my keyed messages are going to wrong partitions"
├── CAUSE: hash(key) % NEW_partition_count ≠ hash(key) % OLD_partition_count
├── WHY: Explained in internal mechanics above
└── FIX: Do NOT increase partitions on keyed topics without a migration plan

PITFALL 2: "Some partitions have way more messages than others"
├── CAUSE: Key skew — some keys are more frequent than others
├── EXAMPLE: key="premium" has 80% of traffic, key="free" has 20%
│   → One partition gets 80% of all messages
├── NAME: "Hot partition" problem
└── FIX: Choose keys with high cardinality (user_id > plan_type)
         Or use custom partitioner (Project 31)

PITFALL 3: "Consumer is not reading from all partitions"
├── CAUSE: Multiple consumers in same group — partitions are SPLIT
├── WHY: With 4 partitions and 2 consumers, each gets 2 partitions
│   → Consumer A reads Partition 0, 1
│   → Consumer B reads Partition 2, 3
│   → This is BY DESIGN — explained in Project 5
└── FIX: Use only 1 consumer per group if you want ALL partitions

PITFALL 4: "I see offset 0 in multiple partitions — is that a bug?"
├── CAUSE: Each partition has its OWN offset sequence
├── THIS IS CORRECT BEHAVIOR
└── FIX: Always reference messages as (topic, partition, offset) triple

PITFALL 5: "I created topic with 1 partition, now I want 4"
├── CAUSE: Underestimated parallelism needs
├── FIX: kafka-topics --alter --partitions 4
├── WARNING: If using keys, existing key mappings WILL change
└── SAFE ONLY: If not using keys, or if you can tolerate remapping

PITFALL 6: "I want to reduce partitions from 8 to 4"
├── CAUSE: Over-provisioned partitions
├── KAFKA DOES NOT SUPPORT THIS
└── FIX: Create new topic with 4 partitions, migrate data, switch consumers

📈 SCALABILITY CONSIDERATIONS

PARTITION COUNT DIRECTLY CONTROLS SCALABILITY:

  ┌──────────────┬────────────────────┬─────────────────────────────────┐
  │ Partitions   │ Max Consumers/Group│ Throughput Potential             │
  ├──────────────┼────────────────────┼─────────────────────────────────┤
  │ 1            │ 1                  │ ~100K msgs/sec (single disk)    │
  │ 4            │ 4                  │ ~400K msgs/sec (4 parallel)     │
  │ 12           │ 12                 │ ~1.2M msgs/sec (12 parallel)    │
  │ 50           │ 50                 │ ~5M msgs/sec (50 parallel)      │
  │ 100+         │ 100+               │ Diminishing returns + overhead  │
  └──────────────┴────────────────────┴─────────────────────────────────┘

GUIDELINES:
→ Small system (dev/testing): 1-4 partitions
→ Medium system (production): 6-12 partitions
→ Large system (high throughput): 12-50 partitions
→ Very large (massive scale): 50-100+ (requires careful monitoring)

FORMULA FOR ESTIMATION:
  desired_partitions = max(
      target_throughput / single_partition_throughput,
      expected_consumer_count
  )

OVERHEAD PER PARTITION:
→ ~10MB memory on broker for index caching
→ 1 directory + 3 files on disk
→ Metadata tracked by controller (ZooKeeper/KRaft)
→ Leader election time during failover scales with partition count

🔍 INTERPRETABILITY CONSIDERATIONS

HOW DO YOU INSPECT PARTITION STATE?

1. TOPIC DESCRIBE COMMAND
   → Shows partition count, leader, replicas per partition
   → docker exec -it kafka kafka-topics --describe --topic mailbox

2. PER-PARTITION CONSUMPTION
   → Read from a specific partition only:
   → kafka-console-consumer --partition 2 --offset 0

3. OFFSET TRACKING
   → Always think in (partition, offset) pairs
   → Log both in your application for debugging
   → "Message processed: partition=2, offset=47"

4. PARTITION DISTRIBUTION VERIFICATION
   → After producing, check how many messages per partition
   → Use kafka-run-class kafka.tools.GetOffsetShell
   → Or use the admin API (shown in code below)

5. KEY-TO-PARTITION MAPPING VERIFICATION
   → Log which partition each keyed message lands in
   → Verify same key → same partition consistently
   → Detect hot partitions early

EXPERIMENT 1: Producing WITHOUT Keys (Round-Robin / Sticky Distribution)

# ═══════════════════════════════════════════════════════════
# file: mailbox_no_keys.py
# Produces messages WITHOUT keys to observe distribution
# ═══════════════════════════════════════════════════════════

from confluent_kafka import Producer
import json
import time

config = {
    'bootstrap.servers': 'localhost:9092',
}

# ─────────────────────────────────────────────────────────
# Track which partition each message lands in
# ─────────────────────────────────────────────────────────

partition_counts = {0: 0, 1: 0, 2: 0, 3: 0}
partition_messages = {0: [], 1: [], 2: [], 3: []}

def delivery_callback(err, msg):
    if err:
        print(f'❌ Failed: {err}')
        return
    
    p = msg.partition()
    o = msg.offset()
    partition_counts[p] += 1
    payload = json.loads(msg.value().decode('utf-8'))
    partition_messages[p].append({
        'offset': o,
        'seq': payload['sequence'],
        'resident': payload['resident'],
    })
    print(
        f'✅ seq={payload["sequence"]:>3} → '
        f'Partition {p} (offset {o:>3}) | '
        f'resident={payload["resident"]}'
    )

producer = Producer(config)
topic = 'mailbox-deliveries'

print("📬 ByteView Towers — Mail Delivery (NO KEYS)")
print("   Observing how Kafka distributes messages without keys\n")
print("=" * 65)

# ─────────────────────────────────────────────────────────
# Produce 20 messages WITHOUT keys
# ─────────────────────────────────────────────────────────

residents = [
    "Alice-Floor1", "Bob-Floor3", "Charlie-Floor5",
    "Diana-Floor7", "Eve-Floor9", "Frank-Floor11",
]

for seq in range(1, 21):
    message = {
        'sequence': seq,
        'resident': residents[(seq - 1) % len(residents)],
        'item': f'Letter #{seq}',
        'timestamp': time.time(),
    }
    
    # ─────────────────────────────────────────────────────
    # KEY IS NONE — Kafka decides which partition
    # ─────────────────────────────────────────────────────
    producer.produce(
        topic=topic,
        key=None,                                  # <── NO KEY
        value=json.dumps(message).encode('utf-8'),
        callback=delivery_callback,
    )
    producer.poll(0)
    time.sleep(0.3)

producer.flush()

# ─────────────────────────────────────────────────────────
# DISTRIBUTION ANALYSIS
# ─────────────────────────────────────────────────────────

print("\n" + "=" * 65)
print("\n📊 PARTITION DISTRIBUTION (No Keys):\n")

for p in range(4):
    bar = "█" * partition_counts[p]
    print(f"   Partition {p}: {partition_counts[p]:>3} messages  {bar}")

print(f"\n   Total: {sum(partition_counts.values())} messages across 4 partitions")

# Check if distribution is roughly even
max_count = max(partition_counts.values())
min_count = min(partition_counts.values())
skew = max_count - min_count

if skew <= 3:
    print(f"   ✅ Distribution is fairly even (skew={skew})")
else:
    print(f"   ⚠️  Distribution is skewed (skew={skew})")
    print(f"      This can happen with sticky partitioner batching")

# ─────────────────────────────────────────────────────────
# ORDERING ANALYSIS WITHIN EACH PARTITION
# ─────────────────────────────────────────────────────────

print(f"\n📋 MESSAGES PER PARTITION:\n")

for p in range(4):
    if partition_messages[p]:
        seqs = [m['seq'] for m in partition_messages[p]]
        offsets = [m['offset'] for m in partition_messages[p]]
        
        offsets_ordered = all(offsets[i] < offsets[i+1] for i in range(len(offsets)-1))
        
        print(f"   Partition {p}: sequences={seqs}")
        print(f"                offsets  ={offsets}")
        print(f"                offsets ordered: {'✅' if offsets_ordered else '❌'}")
    else:
        print(f"   Partition {p}: (empty)")

# ─────────────────────────────────────────────────────────
# KEY INSIGHT: Same resident can appear in DIFFERENT partitions
# ─────────────────────────────────────────────────────────

print(f"\n🔍 WHERE DID EACH RESIDENT'S MAIL GO?\n")

resident_partitions = {}
for p in range(4):
    for m in partition_messages[p]:
        name = m['resident']
        if name not in resident_partitions:
            resident_partitions[name] = set()
        resident_partitions[name].add(p)

for name, parts in sorted(resident_partitions.items()):
    status = "✅ single partition" if len(parts) == 1 else "⚠️  SCATTERED across partitions"
    print(f"   {name:>20} → Partitions {sorted(parts)} {status}")

print(f"\n💡 WITHOUT keys, same resident's mail can go to DIFFERENT partitions")
print(f"   This means ordering per resident is NOT guaranteed!")

EXPERIMENT 2: Producing WITH Keys (Deterministic Partition Assignment)

# ═══════════════════════════════════════════════════════════
# file: mailbox_with_keys.py
# Produces messages WITH keys to observe deterministic routing
# ═══════════════════════════════════════════════════════════

from confluent_kafka import Producer
import json
import time

config = {
    'bootstrap.servers': 'localhost:9092',
}

partition_counts = {0: 0, 1: 0, 2: 0, 3: 0}
partition_messages = {0: [], 1: [], 2: [], 3: []}

def delivery_callback(err, msg):
    if err:
        print(f'❌ Failed: {err}')
        return
    
    p = msg.partition()
    o = msg.offset()
    partition_counts[p] += 1
    payload = json.loads(msg.value().decode('utf-8'))
    partition_messages[p].append({
        'offset': o,
        'seq': payload['sequence'],
        'floor': payload['floor'],
    })
    print(
        f'✅ seq={payload["sequence"]:>3} | '
        f'key="{payload["floor"]:>10}" → '
        f'Partition {p} (offset {o:>3})'
    )

producer = Producer(config)
topic = 'mailbox-deliveries'

print("📬 ByteView Towers — Mail Delivery (WITH KEYS)")
print("   Observing deterministic partition assignment by floor\n")
print("=" * 65)

# ─────────────────────────────────────────────────────────
# Produce 24 messages WITH keys (floor number as key)
# Each floor sends multiple letters
# ─────────────────────────────────────────────────────────

floors = ["floor-1", "floor-2", "floor-3", "floor-4", "floor-5", "floor-6"]

seq = 0
for round_num in range(4):        # 4 rounds of deliveries
    for floor in floors:           # each floor gets a letter per round
        seq += 1
        message = {
            'sequence': seq,
            'floor': floor,
            'item': f'Letter #{seq} for {floor}',
            'round': round_num + 1,
            'timestamp': time.time(),
        }

        # ─────────────────────────────────────────────────
        # KEY IS SET — floor name determines partition
        # hash("floor-3") % 4 → always same partition
        # ─────────────────────────────────────────────────
        producer.produce(
            topic=topic,
            key=floor.encode('utf-8'),              # <── KEY SET
            value=json.dumps(message).encode('utf-8'),
            callback=delivery_callback,
        )
        producer.poll(0)
        time.sleep(0.2)

producer.flush()

# ─────────────────────────────────────────────────────────
# DISTRIBUTION ANALYSIS
# ─────────────────────────────────────────────────────────

print("\n" + "=" * 65)
print("\n📊 PARTITION DISTRIBUTION (With Keys):\n")

for p in range(4):
    bar = "█" * partition_counts[p]
    print(f"   Partition {p}: {partition_counts[p]:>3} messages  {bar}")

# ─────────────────────────────────────────────────────────
# KEY-TO-PARTITION MAPPING
# ─────────────────────────────────────────────────────────

print(f"\n🔑 KEY-TO-PARTITION MAPPING:\n")

floor_to_partition = {}
for p in range(4):
    for m in partition_messages[p]:
        floor = m['floor']
        if floor not in floor_to_partition:
            floor_to_partition[floor] = set()
        floor_to_partition[floor].add(p)

for floor, parts in sorted(floor_to_partition.items()):
    parts_list = sorted(parts)
    if len(parts_list) == 1:
        print(f"   {floor:>10} → ALWAYS Partition {parts_list[0]} ✅ DETERMINISTIC")
    else:
        print(f"   {floor:>10} → Partitions {parts_list} ❌ SCATTERED (should not happen)")

# ─────────────────────────────────────────────────────────
# ORDERING PROOF PER KEY WITHIN PARTITION
# ─────────────────────────────────────────────────────────

print(f"\n📋 ORDERING PROOF PER FLOOR:\n")

floor_sequences = {}
for p in range(4):
    for m in partition_messages[p]:
        floor = m['floor']
        if floor not in floor_sequences:
            floor_sequences[floor] = []
        floor_sequences[floor].append(m['seq'])

for floor, seqs in sorted(floor_sequences.items()):
    is_ordered = all(seqs[i] < seqs[i+1] for i in range(len(seqs)-1))
    status = "✅ ORDERED" if is_ordered else "❌ OUT OF ORDER"
    print(f"   {floor:>10}: sequences = {seqs} {status}")

print(f"\n💡 WITH keys, same floor's mail ALWAYS goes to SAME partition")
print(f"   Ordering per floor is GUARANTEED!")
print(f"   This is the power of keyed messages in multi-partition topics")

from confluent_kafka import Consumer, KafkaError
import json

config = {
    'bootstrap.servers': 'localhost:9092',
    'group.id': 'mailroom-reader-v1',
    'auto.offset.reset': 'earliest',
}

consumer = Consumer(config)
consumer.subscribe(['mailbox-deliveries'])

print("📭 ByteView Towers — Mailroom Reader")
print("   Reading from ALL partitions\n")
print("=" * 70)

messages = []

try:
    empty_count = 0
    while empty_count < 5:
        msg = consumer.poll(timeout=1.0)
        
        if msg is None:
            empty_count += 1
            continue
        
        if msg.error():
            if msg.error().code() == KafkaError._PARTITION_EOF:
                continue
            print(f"Error: {msg.error()}")
            continue
        
        empty_count = 0
        payload = json.loads(msg.value().decode('utf-8'))
        
        record = {
            'partition': msg.partition(),
            'offset': msg.offset(),
            'seq': payload.get('sequence', '?'),
            'floor': payload.get('floor', 'N/A'),
            'read_order': len(messages) + 1,
        }
        messages.append(record)
        
        print(
            f"   Read #{record['read_order']:>3} | "
            f"Partition {record['partition']} | "
            f"Offset {record['offset']:>3} | "
            f"seq={record['seq']:>3} | "
            f"floor={record['floor']}"
        )

except KeyboardInterrupt:
    pass

finally:
    consumer.close()

# ─────────────────────────────────────────────────────────
# ANALYSIS: Cross-partition ordering
# ─────────────────────────────────────────────────────────

print(f"\n" + "=" * 70)
print(f"\n📊 CONSUMPTION ORDER ANALYSIS:\n")

# Check if global sequence is ordered
global_seqs = [m['seq'] for m in messages if isinstance(m['seq'], int)]
globally_ordered = all(global_seqs[i] <= global_seqs[i+1] for i in range(len(global_seqs)-1))

print(f"   Global sequence ordered?  : {'✅ Yes' if globally_ordered else '❌ No (EXPECTED!)'}")

if not globally_ordered:
    print(f"   ↳ This is NORMAL with multiple partitions")
    print(f"   ↳ Kafka does NOT guarantee ordering ACROSS partitions")

# Check per-partition ordering
print(f"\n   Per-partition ordering:")
for p in range(4):
    p_seqs = [m['seq'] for m in messages if m['partition'] == p and isinstance(m['seq'], int)]
    if len(p_seqs) > 1:
        p_ordered = all(p_seqs[i] < p_seqs[i+1] for i in range(len(p_seqs)-1))
        print(f"   Partition {p}: {p_seqs} → {'✅ Ordered' if p_ordered else '❌ NOT ordered'}")
    elif len(p_seqs) == 1:
        print(f"   Partition {p}: {p_seqs} → ✅ Single message (trivially ordered)")
    else:
        print(f"   Partition {p}: (empty)")

# ─────────────────────────────────────────────────────────
# VISUAL: Show how partitions were interleaved during consumption
# ─────────────────────────────────────────────────────────

print(f"\n📋 CONSUMPTION INTERLEAVING PATTERN:")
print(f"   (shows which partition each consecutive read came from)\n")

pattern = [m['partition'] for m in messages[:40]]  # first 40
print(f"   Partitions: {pattern}")
print(f"   Notice: consumer reads from multiple partitions in arbitrary order")

# ─────────────────────────────────────────────────────────
# Method 2: Get watermark offsets (low and high) per partition
# ─────────────────────────────────────────────────────────
# Watermarks tell you:
#   low = earliest available offset (might not be 0 if messages expired)
#   high = next offset to be written (= count of messages if low=0)

consumer_config = {
    'bootstrap.servers': 'localhost:9092',
    'group.id': 'inspector-temp',
}

consumer = Consumer(consumer_config)

print(f"\n📊 PARTITION WATERMARKS (message counts):\n")

total_messages = 0
for part_id in sorted(topic_meta.partitions.keys()):
    tp = TopicPartition(topic_name, part_id)
    low, high = consumer.get_watermark_offsets(tp, timeout=5)
    count = high - low
    total_messages += count
    bar = "█" * count
    print(f"   Partition {part_id}: low={low:>4}, high={high:>4}, "
          f"messages={count:>4}  {bar}")

print(f"\n   Total messages across all partitions: {total_messages}")

consumer.close()

# Method 3: Simulate key-to-partition mapping
# ─────────────────────────────────────────────────────────

print(f"\n🔑 KEY-TO-PARTITION SIMULATION:")
print(f"   (Using Python's murmur2 approximation)\n")

import struct

def murmur2(data):
    """
    Java-compatible murmur2 hash (same as Kafka's default partitioner)
    """
    length = len(data)
    seed = 0x9747b28c
    m = 0x5bd1e995
    r = 24
    
    h = seed ^ length
    length4 = length // 4
    
    for i in range(length4):
        i4 = i * 4
        k = (data[i4 + 0] & 0xff) + \
            ((data[i4 + 1] & 0xff) << 8) + \
            ((data[i4 + 2] & 0xff) << 16) + \
            ((data[i4 + 3] & 0xff) << 24)
        k &= 0xffffffff
        k = (k * m) & 0xffffffff
        k ^= (k >> r)
        k = (k * m) & 0xffffffff
        h = (h * m) & 0xffffffff
        h ^= k
    
    extra = length % 4
    offset = length4 * 4
    if extra >= 3:
        h ^= (data[offset + 2] & 0xff) << 16
    if extra >= 2:
        h ^= (data[offset + 1] & 0xff) << 8
    if extra >= 1:
        h ^= (data[offset + 0] & 0xff)
        h = (h * m) & 0xffffffff
    
    h ^= (h >> 13)
    h = (h * m) & 0xffffffff
    h ^= (h >> 15)
    
    # Convert to signed 32-bit integer (Java compatibility)
    if h >= 0x80000000:
        h -= 0x100000000
    
    return h

num_partitions = 4
test_keys = [
    "floor-1", "floor-2", "floor-3", "floor-4",
    "floor-5", "floor-6", "floor-7", "floor-8",
    "floor-9", "floor-10", "floor-11", "floor-12",
]

for key in test_keys:
    key_bytes = key.encode('utf-8')
    hash_val = murmur2(key_bytes)
    partition = abs(hash_val) % num_partitions
    print(f"   key=\"{key:>10}\" → hash={hash_val:>12} → "
          f"abs % {num_partitions} = Partition {partition}")

print(f"\n💡 Same key ALWAYS maps to same partition (deterministic hash)")
print(f"   If partition count changes, mapping changes!")
