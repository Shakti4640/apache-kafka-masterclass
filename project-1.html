PROJECT 1: THE RESTAURANT ORDER SLIP — YOUR FIRST KAFKA MESSAGE

🎯 HYPOTHETICAL PROBLEM

You own a small restaurant called "ByteBite Diner"

→ A customer walks in and places an order: "1x Burger, extra cheese"
→ The waiter writes the order on a slip of paper
→ The slip goes to a counter (a shared board)
→ The kitchen picks up the slip and starts cooking

RIGHT NOW this works fine with 1 waiter and 1 kitchen

BUT here is the problem:
→ The waiter CANNOT walk to the kitchen every single time
→ The kitchen CANNOT keep asking "any new orders?"
→ If the kitchen is busy, the waiter is STUCK waiting
→ If the waiter is fast, orders pile up and get LOST

You need a SYSTEM where:
→ The waiter DROPS the order slip and walks away immediately
→ The kitchen PICKS UP slips whenever it is ready
→ No order is EVER lost
→ The waiter and kitchen NEVER need to know about each other

🔍 HOW TO APPROACH IT

Step 1: Identify the ACTORS
         → Waiter = the one PRODUCING information
         → Kitchen = the one CONSUMING information
         → Order Slip = the MESSAGE
         → Counter/Board = the CHANNEL where slips are placed

Step 2: Identify the DECOUPLING NEED
         → Waiter should NOT wait for kitchen
         → Kitchen should NOT wait for waiter
         → They communicate through an INTERMEDIARY

Step 3: Identify the GUARANTEE NEED
         → Every order must be stored safely
         → Even if the kitchen is temporarily offline
         → The order must still be there when kitchen comes back

Step 4: Map this to Kafka
         → Waiter         = PRODUCER
         → Kitchen         = CONSUMER
         → Order Slip      = MESSAGE (a record)
         → Counter/Board   = TOPIC (named channel)
         → Restaurant      = BROKER (the Kafka server holding everything)

🛠️ HOW CAN WE SOLVE IT

SOLUTION: Install Apache Kafka and simulate this exact flow

→ Start a Kafka BROKER (the restaurant's message board system)
→ Create a TOPIC called "food-orders" (the counter/board)
→ Write a PRODUCER that sends an order message (the waiter)
→ Write a CONSUMER that reads the order message (the kitchen)

THE FLOW:
  Producer → writes "1x Burger, extra cheese" → to topic "food-orders"
  Consumer → reads from topic "food-orders" → gets "1x Burger, extra cheese"

THEY NEVER TALK TO EACH OTHER DIRECTLY

✅ WHAT COULD BE DONE

Option A: Use a simple file on disk
          → Waiter writes to a file, kitchen reads from it
          → PROBLEM: No ordering guarantee, no concurrent access safety, no scalability

Option B: Use a database table as a queue
          → Insert rows, poll for new rows
          → PROBLEM: Polling is expensive, locking issues, not real-time

Option C: Use a traditional message queue (RabbitMQ, ActiveMQ)
          → Works for simple cases
          → PROBLEM: Message is DELETED after consumption — no replay, no history

Option D: Use Apache Kafka ✅
          → Messages are STORED in an append-only log
          → Messages are NOT deleted after consumption
          → Multiple consumers can read the SAME message independently
          → Messages are ORDERED within their storage unit
          → Built for HIGH THROUGHPUT from day one

✅ WHAT SHOULD DO

1. ALWAYS start ZooKeeper (or KRaft controller) BEFORE starting Kafka broker
   → Kafka depends on coordination metadata

2. ALWAYS create the topic BEFORE producing messages
   → Even though Kafka can auto-create topics, explicit creation gives you control

3. ALWAYS name topics with clear, lowercase, hyphen-separated names
   → "food-orders" ✅
   → "FoodOrders" ❌
   → "food_orders" ⚠️ (works but convention prefers hyphens)

4. ALWAYS specify the bootstrap server address when connecting
   → This is how your producer/consumer finds the Kafka cluster

5. ALWAYS shut down Kafka BEFORE ZooKeeper when stopping
   → Reverse order of startup

6. ALWAYS use a unique group.id for each logical consumer application
   → This is how Kafka tracks what has been read

❌ WHAT SHOULD NOT DO

1. DO NOT start Kafka broker without ZooKeeper/KRaft running
   → Kafka will fail to start or behave unpredictably

2. DO NOT assume messages are deleted after reading
   → Kafka is NOT a traditional queue — messages persist based on retention

3. DO NOT hardcode "localhost:9092" in production
   → Use configuration files or environment variables

4. DO NOT skip understanding the BROKER concept
   → Everything in Kafka revolves around the broker — it is the server

5. DO NOT create hundreds of topics for testing
   → Each topic has metadata overhead — start with ONE topic, understand it deeply

6. DO NOT ignore the log output during startup
   → Kafka logs tell you EXACTLY what is happening — port binding, topic creation, errors

🔀 WHEN TO USE WHAT

┌─────────────────────────────┬──────────────────────────────────────────┐
│ USE KAFKA WHEN              │ USE SOMETHING ELSE WHEN                  │
├─────────────────────────────┼──────────────────────────────────────────┤
│ You need to DECOUPLE        │ You need REQUEST-RESPONSE (use HTTP)     │
│ producer from consumer      │                                          │
├─────────────────────────────┼──────────────────────────────────────────┤
│ You need MESSAGE REPLAY     │ You need one-time delivery and DELETE    │
│ (re-read old messages)      │ after read (use RabbitMQ)                │
├─────────────────────────────┼──────────────────────────────────────────┤
│ You need HIGH THROUGHPUT    │ You have 10 messages/day (overkill)      │
│ (thousands/sec or more)     │                                          │
├─────────────────────────────┼──────────────────────────────────────────┤
│ You need MULTIPLE consumers │ Only ONE consumer will ever read the     │
│ reading same data           │ data (simpler tools suffice)             │
├─────────────────────────────┼──────────────────────────────────────────┤
│ You need DURABILITY         │ You are OK losing messages (use UDP)     │
│ (data must survive restarts)│                                          │
└─────────────────────────────┴──────────────────────────────────────────┘

🚫 WHEN NOT TO USE

→ When your system needs SYNCHRONOUS request-response
  (Kafka is asynchronous by design)

→ When total data volume is tiny and latency is not critical
  (a simple database table or REST API is simpler)

→ When you need COMPLEX ROUTING logic per message
  (RabbitMQ has built-in exchange routing — Kafka routes only by topic/partition)

→ When you have no operational capacity to manage a Kafka cluster
  (Kafka needs disk, memory, monitoring — it is infrastructure)

STEP-BY-STEP: What happens when a message is produced and consumed


═══════════════════════════════════════════════════════════════
LAYER 1: THE BROKER
═══════════════════════════════════════════════════════════════

→ A Kafka BROKER is a single Kafka server process
→ It listens on a port (default: 9092)
→ It STORES messages on disk in a structured directory
→ It ACCEPTS connections from producers and consumers
→ Think of it as the RESTAURANT BUILDING itself

  ┌──────────────────────────────────────────┐
  │              KAFKA BROKER                 │
  │           (localhost:9092)                │
  │                                          │
  │   ┌──────────────────────────────────┐   │
  │   │        TOPIC: food-orders        │   │
  │   │                                  │   │
  │   │  [msg0] [msg1] [msg2] [msg3]     │   │
  │   │   ↑                        ↑     │   │
  │   │   oldest              newest     │   │
  │   └──────────────────────────────────┘   │
  └──────────────────────────────────────────┘


═══════════════════════════════════════════════════════════════
LAYER 2: THE TOPIC
═══════════════════════════════════════════════════════════════

→ A TOPIC is a NAMED category/feed
→ It is an APPEND-ONLY LOG on disk
→ New messages are ALWAYS added at the END
→ Old messages are NEVER modified
→ Messages stay for a configurable RETENTION period (default: 7 days)
→ Think of it as the ORDER COUNTER with a long paper roll

  Timeline of writes:
  ────────────────────────────────────────────→ time
  [offset 0] [offset 1] [offset 2] [offset 3]
  "burger"   "pizza"    "salad"    "pasta"

→ Each message gets a sequential number called OFFSET
→ Offset 0 is the FIRST message ever written
→ Offsets NEVER reset, NEVER reuse, ALWAYS increment


═══════════════════════════════════════════════════════════════
LAYER 3: THE PRODUCER
═══════════════════════════════════════════════════════════════

→ The producer is a CLIENT application
→ It connects to the broker via TCP
→ It SERIALIZES your data (converts string/object → bytes)
→ It sends a PRODUCE REQUEST to the broker
→ The broker APPENDS the message to the topic's log file
→ The broker returns an ACKNOWLEDGMENT

  Producer Flow:
  ┌──────────┐    TCP     ┌──────────┐
  │ PRODUCER │ ─────────→ │  BROKER  │
  │          │  "burger"  │          │
  │          │ ←───────── │  ack: ok │
  └──────────┘            └──────────┘


═══════════════════════════════════════════════════════════════
LAYER 4: THE CONSUMER
═══════════════════════════════════════════════════════════════

→ The consumer is a CLIENT application
→ It connects to the broker via TCP
→ It sends a FETCH REQUEST: "give me messages from offset X"
→ The broker reads from the log file and returns messages
→ The consumer DESERIALIZES bytes back to string/object
→ The consumer PROCESSES the message
→ The consumer's offset position is tracked by Kafka

  Consumer Flow:
  ┌──────────┐    TCP     ┌──────────┐
  │ CONSUMER │ ─────────→ │  BROKER  │
  │          │ "fetch     │          │
  │          │  from 0"   │          │
  │          │ ←───────── │ "burger" │
  └──────────┘            └──────────┘

  CRITICAL INSIGHT:
  → The consumer PULLS data from the broker
  → The broker does NOT push data to the consumer
  → This is a PULL-BASED model
  → This means the consumer controls its own speed


═══════════════════════════════════════════════════════════════
LAYER 5: THE DISK STORAGE (What the log looks like on disk)
═══════════════════════════════════════════════════════════════

→ When you create topic "food-orders"
→ Kafka creates a DIRECTORY on disk:

  /tmp/kafka-logs/food-orders-0/
  ├── 00000000000000000000.log      ← actual message bytes
  ├── 00000000000000000000.index    ← offset-to-position mapping
  └── 00000000000000000000.timeindex ← timestamp-to-offset mapping

→ The ".log" file is the APPEND-ONLY file
→ Messages are written sequentially (like writing in a notebook)
→ This sequential write is WHY Kafka is fast
  (sequential disk I/O is faster than random access)


═══════════════════════════════════════════════════════════════
LAYER 6: COORDINATION (ZooKeeper / KRaft)
═══════════════════════════════════════════════════════════════

→ Kafka needs a METADATA MANAGER
→ Historically: Apache ZooKeeper (separate process)
→ Modern (3.3+): KRaft mode (built-in, no ZooKeeper needed)

→ This metadata manager tracks:
  • Which brokers are alive
  • Which topics exist
  • Which broker is responsible for which topic
  • Consumer group offsets (in older versions)

→ For this project: We use ZooKeeper (widely documented)
→ KRaft is the future, but ZooKeeper is still in most tutorials


═══════════════════════════════════════════════════════════════
COMPLETE INTERNAL FLOW: Producing + Consuming ONE message
═══════════════════════════════════════════════════════════════

  1. ZooKeeper starts → port 2181
  2. Kafka Broker starts → registers with ZooKeeper → port 9092
  3. Topic "food-orders" created → directory created on disk
  4. Producer connects to broker
  5. Producer serializes "1x Burger, extra cheese" → bytes
  6. Producer sends ProduceRequest(topic=food-orders, value=bytes)
  7. Broker appends bytes to food-orders-0/00000000.log
  8. Broker assigns offset 0 to this message
  9. Broker sends ProduceResponse(offset=0, status=success)
  10. Consumer connects to broker with group.id="kitchen-app"
  11. Consumer sends FetchRequest(topic=food-orders, offset=0)
  12. Broker reads from food-orders-0/00000000.log at offset 0
  13. Broker sends FetchResponse(offset=0, value=bytes)
  14. Consumer deserializes bytes → "1x Burger, extra cheese"
  15. Consumer commits offset 0 → "I have read up to offset 0"
  16. Next fetch will start from offset 1

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Kafka won't start"
├── CAUSE: ZooKeeper is not running
├── SYMPTOM: "Connection refused" in Kafka logs
└── FIX: Always start ZooKeeper first, wait 5 seconds, then start Kafka

PITFALL 2: "Producer sends but consumer sees nothing"
├── CAUSE: Consumer started with --from-beginning NOT set
├── SYMPTOM: Consumer is waiting for NEW messages only
├── WHY: Default consumer behavior starts from LATEST offset
└── FIX: Use --from-beginning flag OR set auto.offset.reset=earliest

PITFALL 3: "Port 9092 already in use"
├── CAUSE: Previous Kafka instance not shut down properly
├── FIX: Kill the process → lsof -i :9092 → kill -9 <PID>

PITFALL 4: "Topic already exists error"
├── CAUSE: Re-running topic creation command
├── FIX: Use --if-not-exists flag OR check with --list first

PITFALL 5: "Messages seem to disappear after a week"
├── CAUSE: Default retention is 168 hours (7 days)
├── THIS IS NORMAL: Kafka auto-deletes old messages based on retention
└── FIX: Increase retention if needed (covered in later projects)

PITFALL 6: "Consumer reads same messages again after restart"
├── CAUSE: Offset was not committed before shutdown
├── WHY: Kafka does not know how far the consumer got
└── FIX: Understand offset commit (deep dive in Project 6)

📈 SCALABILITY CONSIDERATIONS

AT THIS STAGE (single broker, single partition):
→ Maximum throughput ≈ limited by single disk I/O
→ Maximum consumers reading same partition = 1 per consumer group
→ Maximum producers = unlimited (but all write to same partition)

SCALING QUESTIONS FOR LATER PROJECTS:
→ What if one broker cannot handle the load?
  → Add more brokers (multi-broker cluster) — Project 40
→ What if one partition is a bottleneck?
  → Add more partitions — Project 3
→ What if one consumer is too slow?
  → Add more consumers in same group — Project 5

FOR NOW: Understand that this single-broker single-partition setup
is the ATOMIC UNIT of Kafka — everything else is scaling THIS up

🔍 INTERPRETABILITY CONSIDERATIONS

HOW DO YOU KNOW WHAT IS HAPPENING?

1. BROKER LOGS
   → Location: kafka-install-dir/logs/server.log
   → Shows: startup, topic creation, connection events
   → Read this FIRST when something goes wrong

2. CLI TOOLS
   → kafka-topics.sh --describe → shows topic configuration
   → kafka-console-consumer.sh → see messages in real-time
   → These are your EYES into Kafka

3. MESSAGE OFFSETS
   → Every message has a number (offset)
   → You can ALWAYS point to a specific message by its offset
   → This makes debugging traceable

4. DATA DIRECTORY
   → Check /tmp/kafka-logs/ → see actual files on disk
   → You can verify messages are PHYSICALLY stored
   → This builds trust in the system

KEY INSIGHT: Kafka is NOT a black box
→ Every message is a file on disk
→ Every offset is a number
→ Every log file is readable
→ Transparency is built into the design

STEP 4: THE PRODUCER (The Waiter)

# ═══════════════════════════════════════════════════════════
# file: producer_waiter.py
# THE WAITER — Takes an order and drops it on the counter
# ═══════════════════════════════════════════════════════════

from confluent_kafka import Producer
import json
import time

# ─────────────────────────────────────────────────────────
# CONFIGURATION
# ─────────────────────────────────────────────────────────
# bootstrap.servers = the address of the Kafka broker
# This is how the producer FINDS the Kafka cluster
# Think of it as the restaurant's street address

config = {
    'bootstrap.servers': 'localhost:9092',
}

# ─────────────────────────────────────────────────────────
# DELIVERY CALLBACK
# ─────────────────────────────────────────────────────────
# This function is called AFTER the broker acknowledges
# Think: the counter confirms "order slip received"

def delivery_callback(err, msg):
    if err is not None:
        print(f'❌ DELIVERY FAILED: {err}')
    else:
        print(f'✅ ORDER DELIVERED!')
        print(f'   Topic    : {msg.topic()}')
        print(f'   Partition: {msg.partition()}')
        print(f'   Offset   : {msg.offset()}')
        print(f'   Value    : {msg.value().decode("utf-8")}')

# ─────────────────────────────────────────────────────────
# CREATE PRODUCER INSTANCE
# ─────────────────────────────────────────────────────────
# This establishes a TCP connection to the broker
# The producer is now READY to send messages

producer = Producer(config)
print("🍔 ByteBite Diner — Waiter Station Ready\n")

# ─────────────────────────────────────────────────────────
# PREPARE THE ORDER (THE MESSAGE)
# ─────────────────────────────────────────────────────────

order = {
    "order_id": 1,
    "customer": "Alice",
    "item": "1x Burger, extra cheese",
    "table": 5,
    "timestamp": time.time()
}

# Convert the order dict to a JSON string, then to bytes
# Kafka deals in BYTES — it does not understand Python dicts
order_bytes = json.dumps(order).encode('utf-8')

# ─────────────────────────────────────────────────────────
# PRODUCE THE MESSAGE
# ─────────────────────────────────────────────────────────
# .produce() puts the message in an INTERNAL BUFFER
# It does NOT immediately send to broker
# Think: waiter writes the slip but hasn't walked to counter yet

topic_name = 'food-orders'

producer.produce(
    topic=topic_name,
    value=order_bytes,
    callback=delivery_callback
)

print(f"📝 Order written to buffer: {order['item']}")

# ─────────────────────────────────────────────────────────
# FLUSH — Force send all buffered messages
# ─────────────────────────────────────────────────────────
# .flush() blocks until ALL messages in the buffer
# have been delivered (or failed)
# Think: waiter walks to counter and places ALL slips

producer.flush()

print("\n✨ All orders have been sent to the kitchen counter!")

STEP 5: THE CONSUMER (The Kitchen)

# ═══════════════════════════════════════════════════════════
# file: consumer_kitchen.py
# THE KITCHEN — Picks up order slips from the counter
# ═══════════════════════════════════════════════════════════

from confluent_kafka import Consumer, KafkaError
import json

# ─────────────────────────────────────────────────────────
# CONFIGURATION
# ─────────────────────────────────────────────────────────
# bootstrap.servers = where to find the Kafka broker
# group.id = a UNIQUE name for this consumer application
#   → Kafka uses this to track which messages have been read
#   → Think: kitchen shift team name — "morning-shift" vs "evening-shift"
# auto.offset.reset = where to start reading if NO prior offset exists
#   → 'earliest' = start from the VERY FIRST message in the topic
#   → 'latest'   = start from ONLY new messages going forward

config = {
    'bootstrap.servers': 'localhost:9092',
    'group.id': 'kitchen-app',
    'auto.offset.reset': 'earliest',
}

# ─────────────────────────────────────────────────────────
# CREATE CONSUMER INSTANCE
# ─────────────────────────────────────────────────────────

consumer = Consumer(config)
print("👨‍🍳 ByteBite Diner — Kitchen Station Ready\n")

# ─────────────────────────────────────────────────────────
# SUBSCRIBE TO TOPIC
# ─────────────────────────────────────────────────────────
# Tell the consumer WHICH topic(s) to read from
# Think: kitchen team says "we handle food-orders counter"

consumer.subscribe(['food-orders'])
print("👀 Watching the food-orders counter...\n")

# ─────────────────────────────────────────────────────────
# POLL LOOP — The core consumption pattern
# ─────────────────────────────────────────────────────────
# The consumer continuously ASKS the broker for new messages
# .poll(timeout) = wait up to `timeout` seconds for a message
# If no message arrives in that time, returns None
# Think: kitchen staff checking the counter every second

try:
    message_count = 0
    max_messages = 5           # Stop after 5 messages (for demo)
    empty_polls = 0
    max_empty_polls = 10       # Stop after 10 empty polls (no more data)

    while message_count < max_messages and empty_polls < max_empty_polls:

        # ─────────────────────────────────────────────────
        # POLL for a single message
        # timeout=1.0 means wait up to 1 second
        # ─────────────────────────────────────────────────
        msg = consumer.poll(timeout=1.0)

        # Case 1: No message received within timeout
        if msg is None:
            empty_polls += 1
            print(f"⏳ No new orders... (waiting {empty_polls}/{max_empty_polls})")
            continue

        # Case 2: An error occurred
        if msg.error():
            if msg.error().code() == KafkaError._PARTITION_EOF:
                # This means we have read ALL messages in the partition
                print(f"📭 Reached end of partition {msg.partition()}")
            else:
                print(f"❌ Error: {msg.error()}")
            continue

        # Case 3: Valid message received!
        empty_polls = 0     # Reset empty poll counter
        message_count += 1

        # ─────────────────────────────────────────────────
        # DESERIALIZE the message
        # Kafka gives us bytes → we decode to string → parse JSON
        # ─────────────────────────────────────────────────
        raw_value = msg.value().decode('utf-8')
        order = json.loads(raw_value)

        print(f"🔔 NEW ORDER RECEIVED!")
        print(f"   Order ID : {order['order_id']}")
        print(f"   Customer : {order['customer']}")
        print(f"   Item     : {order['item']}")
        print(f"   Table    : {order['table']}")
        print(f"   ─── Kafka Metadata ───")
        print(f"   Topic    : {msg.topic()}")
        print(f"   Partition: {msg.partition()}")
        print(f"   Offset   : {msg.offset()}")
        print(f"   Timestamp: {msg.timestamp()}")
        print()

except KeyboardInterrupt:
    print("\n🛑 Kitchen shutting down...")

finally:
    # ─────────────────────────────────────────────────────
    # CLOSE the consumer cleanly
    # ─────────────────────────────────────────────────────
    # This commits final offsets and leaves the consumer group
    # Think: kitchen shift ends, they mark where they stopped
    consumer.close()
    print("👋 Kitchen station closed. All offsets committed.")

STEP 7: VERIFY — Send Multiple Orders

# ═══════════════════════════════════════════════════════════
# file: producer_multiple_orders.py
# Send multiple orders to see offset incrementing
# ═══════════════════════════════════════════════════════════

from confluent_kafka import Producer
import json
import time

config = {
    'bootstrap.servers': 'localhost:9092',
}

def delivery_callback(err, msg):
    if err:
        print(f'❌ FAILED: {err}')
    else:
        print(f'✅ Offset {msg.offset()} → {msg.value().decode("utf-8")[:50]}...')

producer = Producer(config)

# ─────────────────────────────────────────────────────────
# Multiple orders — watch offsets increment: 0, 1, 2, 3, 4
# ─────────────────────────────────────────────────────────
orders = [
    {"order_id": 1, "customer": "Alice",   "item": "1x Burger, extra cheese",  "table": 5},
    {"order_id": 2, "customer": "Bob",     "item": "2x Pizza Margherita",      "table": 3},
    {"order_id": 3, "customer": "Charlie", "item": "1x Caesar Salad",          "table": 7},
    {"order_id": 4, "customer": "Diana",   "item": "3x Fish & Chips",          "table": 1},
    {"order_id": 5, "customer": "Eve",     "item": "1x Veggie Wrap, no onion", "table": 12},
]

print("🍔 Sending 5 orders to food-orders topic...\n")

for order in orders:
    order['timestamp'] = time.time()
    producer.produce(
        topic='food-orders',
        value=json.dumps(order).encode('utf-8'),
        callback=delivery_callback
    )
    # producer.poll(0) triggers callback processing
    # without this, callbacks queue up until flush()
    producer.poll(0)
    time.sleep(0.5)       # small delay to simulate real-time ordering

# Send all remaining buffered messages
producer.flush()

print("\n✨ All 5 orders sent!")

# EXPECTED OUTPUT:
# 🍔 Sending 5 orders to food-orders topic...
#
# ✅ Offset 1 → {"order_id": 1, "customer": "Alice", "item": "1x Bu...

# ✅ Offset 3 → {"order_id": 3, "customer": "Charlie", "item": "1x C...
# ✅ Offset 4 → {"order_id": 4, "customer": "Diana", "item": "3x Fis...

# ═══════════════════════════════════════════════════════════
# file: verify_setup.py
# Confirms broker is reachable and topic exists

from confluent_kafka.admin import AdminClient, NewTopic

admin_config = {
    'bootstrap.servers': 'localhost:9092',
}

admin = AdminClient(admin_config)

# ─────────────────────────────────────────────────────────
# Check 1: Broker connectivity
# ─────────────────────────────────────────────────────────
cluster_metadata = admin.list_topics(timeout=5)

print("═══ KAFKA CLUSTER STATUS ═══\n")
print(f"Broker count: {len(cluster_metadata.brokers)}")
for broker_id, broker in cluster_metadata.brokers.items():
    print(f"  Broker {broker_id}: {broker}")

# ─────────────────────────────────────────────────────────
# Check 2: Topic existence
# ─────────────────────────────────────────────────────────
print(f"\nTopic count: {len(cluster_metadata.topics)}")
for topic_name, topic_metadata in cluster_metadata.topics.items():
    if not topic_name.startswith('_'):    # Skip internal topics
        print(f"  📋 {topic_name}")
        print(f"     Partitions: {len(topic_metadata.partitions)}")
        for part_id, part_meta in topic_metadata.partitions.items():
            print(f"     Partition {part_id}: Leader=Broker {part_meta.leader}")

print("\n✅ Kafka is running and food-orders topic exists!")

WHAT YOU LEARNED IN PROJECT 1:
═══════════════════════════════

✅ Kafka is a DISTRIBUTED APPEND-ONLY LOG system
✅ A BROKER is the server that stores and serves messages
✅ A TOPIC is a named channel where messages are written
✅ A PRODUCER writes messages to a topic
✅ A CONSUMER reads messages from a topic (pull-based)
✅ Every message gets an OFFSET — a sequential, immutable number
✅ Messages are NOT deleted after reading (unlike traditional queues)
✅ Producer and Consumer are FULLY DECOUPLED — they never talk to each other

THE CORE MENTAL MODEL:
  Producer → Topic (append-only log on broker) → Consumer
  
  This NEVER changes — every single Kafka pattern
  is a variation of this flow.
