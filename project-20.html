PROJECT 20: THE CONTENT FILTER — STATELESS STREAM TRANSFORMATIONS

🛡️ HYPOTHETICAL PROBLEM

The same social media analytics company from Project 19 now faces a content crisis

SCENARIO: Real-Time Content Moderation Pipeline
→ 50 million posts/day flow through "social-posts" topic
→ Posts contain: text, user metadata, language, category tags
→ Legal, marketing, and safety teams each need DIFFERENT filtered views

REQUIREMENTS FROM EACH TEAM:

  SAFETY TEAM:
  → "Show me ONLY posts containing banned/toxic words"
  → "Flag them with severity levels (LOW, MEDIUM, HIGH)"
  → "Route HIGH severity to immediate-review topic"
  → "Route LOW/MEDIUM to batch-review topic"

  MARKETING TEAM:
  → "Show me posts mentioning our brand or competitor brands"
  → "Enrich each post with sentiment score (POSITIVE/NEGATIVE/NEUTRAL)"
  → "Strip out personally identifiable information (PII)"
  → "Route to different topics by detected language"

  LEGAL TEAM:
  → "Show me posts from users in EU regions (GDPR scope)"
  → "Redact email addresses and phone numbers"
  → "Add compliance headers to every message"
  → "Merge all compliance-relevant streams into one audit topic"

THE CRISIS WITH PROJECT 19 APPROACH:

  Developer tries to build everything in ONE topology:
  
  builder.stream("social-posts")
      .flatMapValues(split)
      .groupBy(word)
      .count()
      .to("word-counts");

  Problems:
  → How do I FILTER posts by content? (filter)
  → How do I TRANSFORM the structure? (map/mapValues)
  → How do I add/change the KEY? (selectKey)
  → How do I split ONE stream into MULTIPLE? (branch)
  → How do I MERGE multiple streams into ONE? (merge)
  → How do I observe without modifying? (peek)
  → How do I generate MULTIPLE output records from one input? (flatMap)

  ALL of these are STATELESS operations:
  → No state store needed
  → No changelog topic
  → No RocksDB
  → Cheapest operations in Kafka Streams
  → MUST master these before adding stateful complexity (Project 21)

🔍 HOW TO APPROACH IT

Step 1: Categorize all Kafka Streams operations
         
         STATELESS (this project — no state store):
         → filter(), filterNot()
         → map(), mapValues()
         → flatMap(), flatMapValues()
         → selectKey()
         → branch() / split()
         → merge()
         → peek()
         → through() / repartition()
         → groupBy() — technically triggers repartition but no local state
         
         STATEFUL (Project 21 — requires state store):
         → count(), reduce(), aggregate()
         → joins (Project 32)
         → windowed operations (Project 22)

Step 2: Understand the KEY distinction
         → Operations on VALUES only (mapValues, flatMapValues, filter by value):
           → Do NOT trigger repartition
           → Key remains unchanged → partition assignment unchanged
           → CHEAPEST possible operations
         
         → Operations that CHANGE the KEY (map, flatMap, selectKey, groupBy):
           → MAY trigger repartition if followed by stateful operation
           → New key → different partition → data must be shuffled
           → More expensive — use only when necessary

Step 3: Map each team's requirement to an operation
         
         Safety:  filter → mapValues (add severity) → branch (by severity)
         Marketing: filter → mapValues (add sentiment) → mapValues (strip PII) → branch (by language)
         Legal: filter → mapValues (redact) → map (add headers) → merge

Step 4: Build the topology as a DAG
         → Single source stream splits into multiple processing paths
         → Each path applies different transformations
         → Paths may converge (merge) or diverge (branch)
         → This is the power of the DSL

🛠️ HOW CAN WE SOLVE IT

SOLUTION: Multi-path stateless topology with branch, filter, map, and merge

✅ WHAT COULD BE DONE

Option A: Separate Consumer Applications per Team
  → Safety team runs their own consumer with filter logic
  → Marketing team runs their own consumer with transform logic
  → Legal team runs their own consumer with redaction logic
  → PROBLEM: Same data read 3 times from source topic
  → PROBLEM: Each team duplicates deserialization overhead
  → PROBLEM: No coordination — each team's logic drifts independently

Option B: Single Monolithic Consumer (Project 12-18 style)
  → One consumer reads everything
  → Giant if/else block routes to different producers
  → All transformation logic in one massive function
  → PROBLEM: Unmaintainable code
  → PROBLEM: No parallelism within the processing
  → PROBLEM: Adding new team requirements = risky changes

Option C: Kafka Streams Stateless Topology ← CORRECT
  → Single source stream, multiple processing branches
  → Each transformation is a composable DSL operation
  → Automatic parallelism via partition-based tasks
  → Adding new branches = adding new DSL chain
  → No state store overhead for pure transformations

✅ WHAT SHOULD DO

1. PREFER mapValues() OVER map() WHEN KEY DOESN'T CHANGE
   → mapValues() preserves the key → no repartition possible
   → map() creates new KeyValue → marks stream as "requires repartition"
   → Even if your map() returns the SAME key, Kafka Streams doesn't know that
   → Always use mapValues() for value-only transformations

2. PREFER flatMapValues() OVER flatMap() WHEN KEY DOESN'T CHANGE
   → Same reasoning as above
   → flatMap() can change key → potential repartition
   → flatMapValues() keeps key → no repartition

3. USE branch()/split() TO AVOID MULTIPLE filter() CALLS
   → BAD: stream.filter(isHigh) + stream.filter(isMedium) + stream.filter(isLow)
     → Processes EVERY record THREE times
   → GOOD: stream.split().branch(isHigh).branch(isMedium).defaultBranch()
     → Processes each record ONCE, routes to correct branch

4. USE peek() FOR LOGGING/MONITORING, NEVER FOR SIDE EFFECTS
   → peek() is meant for observation only
   → DO NOT write to database in peek()
   → DO NOT make HTTP calls in peek()
   → Kafka Streams may call peek() multiple times (retries)
   → Side effects would be duplicated

5. USE Named.as() FOR EVERY OPERATION
   → Names appear in topology description and metrics
   → Without names: auto-generated like "KSTREAM-FILTER-0000000003"
   → With names: "filter-toxic-posts" — much easier to debug
   → Required for meaningful monitoring (Project 28)

6. CHAIN OPERATIONS EFFICIENTLY
   → Each operation creates a new processing node
   → Combine related transformations to minimize nodes
   → But don't sacrifice readability for micro-optimization

❌ WHAT SHOULD NOT DO

1. DO NOT USE map() WHEN mapValues() SUFFICES
   → map() marks stream as "needs repartition"
   → If followed by groupByKey() → UNNECESSARY repartition
   → Wastes network, disk, and CPU
   → This is the #1 performance mistake in Kafka Streams

2. DO NOT PUT BLOCKING I/O IN TRANSFORMATION FUNCTIONS
   → filter predicate calls HTTP API → blocks stream thread
   → mapValues calls database → blocks stream thread
   → Stream thread blocked → consumer times out → rebalance triggered
   → All transformations must be FAST and PURE
   → Pre-load reference data into KTable (Project 32) instead

3. DO NOT USE filter() MULTIPLE TIMES FOR MUTUAL EXCLUSION
   → stream.filter(x -> x > 10) — processes ALL records
   → stream.filter(x -> x <= 10) — processes ALL records AGAIN
   → Use branch()/split() — processes each record ONCE
   → Especially important at high throughput

4. DO NOT MODIFY INPUT OBJECTS IN mapValues()
   → Kafka Streams may reuse objects internally
   → Always create NEW objects in map/mapValues
   → BAD:  mapValues(v -> { v.setField("x"); return v; })
   → GOOD: mapValues(v -> new MyObject(v.getField(), "x"))

5. DO NOT IGNORE THE SERDE MISMATCH PROBLEM
   → If mapValues() changes value TYPE (String → JSON → Avro)
   → Downstream operations need matching Serde
   → Explicitly specify Serdes when types change
   → Runtime ClassCastException is the symptom of Serde mismatch

6. DO NOT CREATE OVERLY DEEP TOPOLOGY CHAINS
   → 20+ chained operations become hard to debug
   → Break into named sub-topologies
   → Use intermediate variables for readability
   → Each branch should be a clear, named processing path

DECISION TREE FOR CHOOSING OPERATION:

  "Do I need to remove records?"
  │ YES → filter() or filterNot()
  │
  "Do I need to transform the value?"
  │ YES → "Does the key change?"
  │        │ NO  → mapValues() ← PREFER THIS
  │        │ YES → map()
  │
  "Do I need one input → many outputs?"
  │ YES → "Does the key change?"
  │        │ NO  → flatMapValues() ← PREFER THIS
  │        │ YES → flatMap()
  │
  "Do I need to change ONLY the key?"
  │ YES → selectKey()
  │
  "Do I need to split into multiple paths?"
  │ YES → branch() / split()
  │
  "Do I need to combine multiple paths?"
  │ YES → merge()
  │
  "Do I need to observe without changing?"
  │ YES → peek()

🚫 WHEN NOT TO USE

1. WHEN YOU NEED AGGREGATION
   → filter + map cannot maintain running totals
   → That's count(), reduce(), aggregate() → Project 21

2. WHEN YOU NEED TO JOIN WITH ANOTHER STREAM/TABLE
   → Stateless operations work on ONE stream at a time
   → Enriching post with user profile data → KStream-KTable join → Project 32
   → Cannot be done with mapValues() alone (no access to external state)

3. WHEN YOU NEED WINDOWED PROCESSING
   → "Posts in the last 5 minutes" requires windowing
   → Stateless operations have no concept of time windows
   → Windowed operations → Project 22

4. WHEN TRANSFORMATION REQUIRES EXTERNAL STATE
   → "Look up user reputation score from database"
   → Cannot call database in mapValues() (blocking I/O)
   → Load data into KTable via Kafka Connect, then join → Project 32

═══════════════════════════════════════════════════════════════════
MECHANISM 1: PROCESSOR NODE CHAIN
═══════════════════════════════════════════════════════════════════

  Each DSL operation creates a PROCESSOR NODE in the topology:

  builder.stream("social-posts")           → SourceNode
      .filter(predicate)                   → FilterNode
      .mapValues(transformer)              → MapValuesNode
      .flatMapValues(splitter)             → FlatMapValuesNode
      .to("output-topic");                 → SinkNode

  Internal representation:
  
  ┌────────────────┐
  │ SourceNode     │ → Wraps a Kafka Consumer for "social-posts"
  │ (auto-named)   │    Polls records, forwards to child nodes
  └───────┬────────┘
          │ forward(key, value)
          ▼
  ┌────────────────┐
  │ FilterNode     │ → Calls predicate.test(key, value)
  │ "filter-toxic" │    If true: forward to next node
  │                │    If false: DROP record (no forwarding)
  └───────┬────────┘
          │ forward(key, value) — only if predicate passed
          ▼
  ┌────────────────┐
  │ MapValuesNode  │ → Calls mapper.apply(value)
  │ "add-severity" │    Returns new value
  │                │    Forwards (SAME key, NEW value)
  └───────┬────────┘
          │ forward(key, newValue)
          ▼
  ┌─────────────────┐
  │ FlatMapValuesNode│ → Calls mapper.apply(value)
  │ "split-tags"     │    Returns Iterable<NewValue>
  │                  │    Forwards (SAME key, each newValue) — MULTIPLE forwards
  └───────┬──────────┘
          │ forward(key, val1), forward(key, val2), ...
          ▼
  ┌────────────────┐
  │ SinkNode       │ → Wraps a Kafka Producer for "output-topic"
  │ (auto-named)   │    Serializes and produces each forwarded record
  └────────────────┘

  KEY INSIGHT: 
  → All nodes run in the SAME THREAD for a given task
  → No inter-thread communication for stateless operations
  → Record flows through entire chain synchronously
  → This is why stateless operations are FAST

═══════════════════════════════════════════════════════════════════
MECHANISM 2: filter() INTERNALS
═══════════════════════════════════════════════════════════════════

  public class KStreamFilter<K, V> implements Processor<K, V> {
      private final Predicate<K, V> predicate;
      private ProcessorContext context;

      @Override
      public void process(Record<K, V> record) {
          if (predicate.test(record.key(), record.value())) {
              // PASS — forward to next node
              context.forward(record);
          }
          // FAIL — record is DROPPED (not forwarded)
          // No error, no dead letter — simply gone
      }
  }

  Cost: One predicate evaluation per record
  Memory: Zero (no state)
  I/O: Zero (no disk, no network)

MECHANISM 3: mapValues() vs map() — THE CRITICAL DIFFERENCE
═══════════════════════════════════════════════════════════════════

  mapValues():
  ┌────────────────────────────────────────────────────┐
  │ Input:  (key="user-1", value="hello world")       │
  │ Apply:  v -> v.toUpperCase()                       │
  │ Output: (key="user-1", value="HELLO WORLD")       │
  │                                                    │
  │ Internal flag: KEY_NOT_CHANGED = true              │
  │ → Kafka Streams KNOWS key is preserved             │
  │ → No repartition needed before stateful operations │
  └────────────────────────────────────────────────────┘

  map():
  ┌────────────────────────────────────────────────────┐
  │ Input:  (key="user-1", value="hello world")       │
  │ Apply:  (k,v) -> KeyValue.pair(k, v.toUpperCase())│
  │ Output: (key="user-1", value="HELLO WORLD")       │
  │                                                    │
  │ Internal flag: KEY_NOT_CHANGED = false             │
  │ → Kafka Streams ASSUMES key MAY have changed       │
  │ → Even though we returned the SAME key             │
  │ → Repartition WILL be triggered before groupByKey()│
  └────────────────────────────────────────────────────┘

  This is why the rule is:
  → If you're only changing the value → ALWAYS use mapValues()
  → map() should ONLY be used when key actually changes

═══════════════════════════════════════════════════════════════════
MECHANISM 4: branch()/split() — SINGLE-PASS ROUTING
═══════════════════════════════════════════════════════════════════

  How branch works internally:

  stream.split(Named.as("route-"))
      .branch(isHigh,    Branched.as("high"))      // predicate 1
      .branch(isMedium,  Branched.as("medium"))    // predicate 2
      .defaultBranch(    Branched.as("low"));       // catch-all

  For each record:
  ┌──────────────────────────────────────────────────┐
  │ 1. Test predicate 1 (isHigh)                     │
  │    → If TRUE: forward to "high" branch → DONE    │
  │    → If FALSE: continue                           │
  │                                                   │
  │ 2. Test predicate 2 (isMedium)                   │
  │    → If TRUE: forward to "medium" branch → DONE  │
  │    → If FALSE: continue                           │
  │                                                   │
  │ 3. Default branch: forward to "low" → DONE       │
  └──────────────────────────────────────────────────┘

  → Record is tested against predicates IN ORDER
  → FIRST matching predicate wins
  → Record goes to EXACTLY ONE branch
  → Single pass through the data (unlike multiple filter() calls)

  vs Multiple filter() approach:
  ┌──────────────────────────────────────────────────┐
  │ stream.filter(isHigh)   → tests ALL records      │
  │ stream.filter(isMedium) → tests ALL records AGAIN │
  │ stream.filter(isLow)    → tests ALL records AGAIN │
  │                                                   │
  │ 3x the predicate evaluations!                    │
  │ AND: if predicates overlap, record goes to       │
  │ MULTIPLE streams (may or may not be desired)     │
  └──────────────────────────────────────────────────┘

MECHANISM 5: merge() — COMBINING STREAMS
═══════════════════════════════════════════════════════════════════

  streamA.merge(streamB).merge(streamC)

  Internal representation:
  
  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ Stream A │  │ Stream B │  │ Stream C │
  └────┬─────┘  └────┬─────┘  └────┬─────┘
       │              │              │
       └──────────────┼──────────────┘
                      │
                ┌─────┴─────┐
                │ MergeNode │ → Forwards records from ALL inputs
                └─────┬─────┘    to the SAME downstream chain
                      │
                      ▼
                 (next node)

  → No ordering guarantee across merged streams
  → Records from A, B, C are interleaved
  → Same key from different streams may arrive in any order
  → Merge is LOGICAL — no actual data movement

MECHANISM 6: REPARTITION TRIGGER TRACKING
═══════════════════════════════════════════════════════════════════

  Kafka Streams internally tracks whether a stream "needs repartitioning":

  KStream<String, String> original = builder.stream("input");
  // repartitionRequired = false

  KStream<String, String> filtered = original.filter(...);
  // repartitionRequired = false (filter doesn't change key)

  KStream<String, String> mappedValues = filtered.mapValues(...);
  // repartitionRequired = false (mapValues doesn't change key)

  KStream<String, String> mapped = filtered.map(...);
  // repartitionRequired = TRUE (map MIGHT change key)

  KStream<String, String> selected = filtered.selectKey(...);
  // repartitionRequired = TRUE (selectKey changes key)

  // If we now do:
  mapped.groupByKey().count();
  // Kafka Streams sees repartitionRequired = true
  // → Creates repartition topic AUTOMATICALLY
  // → Shuffles data through repartition topic
  // → Then performs count() on repartitioned data

  // But if we do:
  mappedValues.groupByKey().count();
  // Kafka Streams sees repartitionRequired = false
  // → NO repartition topic created
  // → Directly performs count()
  // → MUCH more efficient

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: USING map() WHEN mapValues() WOULD WORK
─────────────────────────────────────────────────────
  Symptom: Unexpected repartition topic, increased latency
  Root Cause: map() marks stream as "key potentially changed"
  Example:
    // BAD — triggers repartition flag
    stream.map((k, v) -> KeyValue.pair(k, v.toUpperCase()))
    
    // GOOD — no repartition flag
    stream.mapValues(v -> v.toUpperCase())
  
  Detection: Check topology description for repartition nodes
  Impact: 2x latency for the repartition step + extra topic

PITFALL 2: MULTIPLE filter() INSTEAD OF branch()
─────────────────────────────────────────────────────
  Symptom: High CPU usage, each record processed multiple times
  Root Cause: N filter() calls = N passes over the data
  Example:
    // BAD — 3 passes
    KStream<> high = stream.filter((k,v) -> severity(v) == HIGH);
    KStream<> med  = stream.filter((k,v) -> severity(v) == MEDIUM);
    KStream<> low  = stream.filter((k,v) -> severity(v) == LOW);
    
    // GOOD — 1 pass
    Map<String, KStream<>> branches = stream.split()
        .branch((k,v) -> severity(v) == HIGH, Branched.as("high"))
        .branch((k,v) -> severity(v) == MEDIUM, Branched.as("med"))
        .defaultBranch(Branched.as("low"));

PITFALL 3: MUTATING INPUT OBJECTS
─────────────────────────────────────────────────────
  Symptom: Mysterious data corruption, non-deterministic behavior
  Root Cause: Kafka Streams may reuse record objects
  Example:
    // BAD — mutates the input
    stream.mapValues(post -> {
        post.setText(post.getText().toUpperCase());
        return post;  // Same object, mutated
    });
    
    // GOOD — creates new object
    stream.mapValues(post -> {
        return new Post(post.getText().toUpperCase(), post.getUser());
    });

PITFALL 4: BLOCKING I/O IN TRANSFORMATION FUNCTIONS
─────────────────────────────────────────────────────
  Symptom: Consumer timeout, rebalancing, processing stalls
  Root Cause: HTTP call or DB query blocks the stream thread
  Example:
    // BAD — blocks stream thread
    stream.mapValues(post -> {
        String sentiment = httpClient.get("/api/sentiment?text=" + post);
        return post + "|" + sentiment;
    });
    
    // GOOD — use KTable join for enrichment (Project 32)
    stream.leftJoin(sentimentTable, (post, sentiment) -> post + "|" + sentiment);

PITFALL 5: flatMap PRODUCING TOO MANY RECORDS
─────────────────────────────────────────────────────
  Symptom: Memory pressure, downstream topic overwhelmed
  Root Cause: flatMap generates unbounded output per input
  Example:
    // DANGEROUS — could produce millions of records per input
    stream.flatMapValues(post -> generateAllNGrams(post));
    
  Fix: Add limits in flatMap logic
       Filter immediately after flatMap to reduce volume

PITFALL 6: SERDE MISMATCH AFTER TYPE CHANGE
─────────────────────────────────────────────────────
  Symptom: ClassCastException at runtime
  Root Cause: mapValues changes value type but Serde not updated
  Example:
    // Input: KStream<String, String>
    stream.mapValues(v -> Integer.parseInt(v))
    // Now: KStream<String, Integer>
    // But default value Serde is still String!
    .to("output");  // ClassCastException when serializing Integer as String
    
  Fix: Specify Serde explicitly:
    .to("output", Produced.with(Serdes.String(), Serdes.Integer()));

  │ filter()           │ Predicate  │ Zero      │ Zero         │
  │ mapValues()        │ Transform  │ Zero      │ Zero         │
  │ flatMapValues()    │ Transform  │ Temp list │ Zero         │
  │ peek()             │ Minimal    │ Zero      │ Zero         │
  │ branch()/split()   │ Predicates │ Zero      │ Zero         │
  │ merge()            │ Zero       │ Zero      │ Zero         │
  │ selectKey()        │ Transform  │ Zero      │ Repartition* │
  │ map()              │ Transform  │ Zero      │ Repartition* │
  │ flatMap()          │ Transform  │ Temp list │ Repartition* │
  └────────────────────┴────────────┴───────────┴──────────────┘
  
  * Repartition only triggered when followed by stateful operation

THROUGHPUT BENCHMARKS (stateless only):
  
  Single instance, 6 partitions:
  → filter only:                    800K records/sec
  → filter + mapValues:             700K records/sec
  → filter + mapValues + branch:    650K records/sec
  → filter + map (with rekey):      350K records/sec (repartition overhead)
  → filter + flatMapValues (1→5):   400K records/sec

SCALING STATELESS TOPOLOGIES:
  → Stateless operations scale LINEARLY with instances
  → 1 instance = 650K/sec → 3 instances = ~1.9M/sec
  → No state to transfer on rebalance → instant rebalancing
  → This is the CHEAPEST topology to scale

MEMORY CONSIDERATIONS:
  → Stateless operations use negligible memory
  → No RocksDB, no state store, no changelog
  → Main memory usage: record buffers and deserialized objects
  → flatMap with large output can temporarily spike memory

package com.kafkamastery;

import com.fasterxml.jackson.databind.ObjectMapper;
import org.apache.kafka.common.serialization.Deserializer;
import org.apache.kafka.common.serialization.Serde;
import org.apache.kafka.common.serialization.Serializer;

/**
 * Generic JSON Serde for any Java class.
 * Bridges Jackson JSON processing with Kafka Streams Serde interface.
 *
 * Builds on: Project 7 (serialization concepts), Project 16 (schema concepts)
 */
public class JsonSerde<T> implements Serde<T> {

    private final ObjectMapper mapper = new ObjectMapper();
    private final Class<T> targetType;

    public JsonSerde(Class<T> targetType) {
        this.targetType = targetType;
    }

    @Override
    public Serializer<T> serializer() {
        return (topic, data) -> {
            if (data == null) return null;
            try {
                return mapper.writeValueAsBytes(data);
            } catch (Exception e) {
                throw new RuntimeException(
                    "Error serializing " + targetType.getSimpleName() + " to JSON", e);
            }
        };
    }

    @Override
    public Deserializer<T> deserializer() {
        return (topic, data) -> {
            if (data == null) return null;
            try {
                return mapper.readValue(data, targetType);
            } catch (Exception e) {
                System.err.printf("  ⚠️  Failed to deserialize from topic %s: %s%n", 
                    topic, e.getMessage());
                return null;
            }
        };
    }
}

    static Topology buildTopology() {
        StreamsBuilder builder = new StreamsBuilder();

        // ═══════════════════════════════════════════════════════
        // SOURCE: Read from content-posts topic
        // ═══════════════════════════════════════════════════════
        // Input: Key=user_id (String), Value=JSON ContentPost
        KStream<String, ContentPost> posts = builder.stream(
            INPUT_TOPIC,
            Consumed.with(Serdes.String(), POST_SERDE)
                .withName("source-content-posts")
        );

        // ═══════════════════════════════════════════════════════
        // PEEK: Log all incoming posts (observation only)
        // ═══════════════════════════════════════════════════════
        // peek() does NOT modify the stream
        // Used for logging, metrics, monitoring
        // NEVER put side effects (DB writes, HTTP calls) here
        KStream<String, ContentPost> observed = posts.peek(
            (userId, post) -> {
                System.out.printf("  📥 INCOMING: user=%s post_id=%s lang=%s region=%s%n",
                    userId,
                    post.getPostId(),
                    post.getLanguage(),
                    post.getRegion());
            },
            Named.as("peek-incoming-posts")
        );

        // ═══════════════════════════════════════════════════════
        // FILTER: Remove null/empty posts
        // ═══════════════════════════════════════════════════════
        // filter() drops records that don't match the predicate
        // Zero cost — no state, no I/O
        KStream<String, ContentPost> validPosts = observed.filter(
            (userId, post) -> post != null
                && post.getText() != null
                && !post.getText().trim().isEmpty()
                && post.getPostId() != null,
            Named.as("filter-valid-posts")
        );

        // ═══════════════════════════════════════════════════════
        // PATH 1: SAFETY PIPELINE
        // filter → mapValues → branch
        // ═══════════════════════════════════════════════════════

        // STEP 1A: filter() — Keep only posts with toxic content
        KStream<String, ContentPost> toxicPosts = validPosts.filter(
            (userId, post) -> containsToxicContent(post.getText()),
            Named.as("filter-toxic-content")
        );

        // STEP 1B: mapValues() — Add severity level
        // mapValues() because we're ONLY changing the value, NOT the key
        // This is CRITICAL — using map() here would be wrong (see pitfalls)
        KStream<String, ContentPost> severityTagged = toxicPosts.mapValues(
            (readOnlyKey, post) -> {
                String severity = calculateSeverity(post.getText());
                return post.withSeverity(severity).withProcessingStage("safety-review");
            },
            Named.as("mapvalues-add-severity")
        );

        // STEP 1C: branch()/split() — Route by severity
        // Single pass: each record goes to exactly ONE branch
        // Much more efficient than multiple filter() calls
        Map<String, KStream<String, ContentPost>> severityBranches = severityTagged
            .split(Named.as("branch-severity-"))
            .branch(
                (userId, post) -> "HIGH".equals(post.getSeverity()),
                Branched.withConsumer(
                    stream -> stream
                        .peek((k, v) -> System.out.printf(
                            "  🚨 HIGH SEVERITY: %s → immediate-review%n", v.getPostId()),
                            Named.as("peek-high-severity"))
                        .to(IMMEDIATE_REVIEW,
                            Produced.with(Serdes.String(), POST_SERDE)
                                .withName("sink-immediate-review")),
                    "high"
                )
            )
            .defaultBranch(
                Branched.withConsumer(
                    stream -> stream
                        .peek((k, v) -> System.out.printf(
                            "  ⚠️  %s SEVERITY: %s → batch-review%n",
                            v.getSeverity(), v.getPostId()),
                            Named.as("peek-batch-severity"))
                        .to(BATCH_REVIEW,
                            Produced.with(Serdes.String(), POST_SERDE)
                                .withName("sink-batch-review")),
                    "default"
                )
            );

        // ═══════════════════════════════════════════════════════
        // PATH 2: MARKETING PIPELINE
        // filter → mapValues (sentiment) → mapValues (PII) → branch (language)
        // ═══════════════════════════════════════════════════════

        // STEP 2A: filter() — Keep posts mentioning brand keywords
        KStream<String, ContentPost> brandPosts = validPosts.filter(
            (userId, post) -> containsBrandMention(post.getText()),
            Named.as("filter-brand-mentions")
        );

        // STEP 2B: mapValues() — Add sentiment analysis
        KStream<String, ContentPost> sentimentTagged = brandPosts.mapValues(
            (readOnlyKey, post) -> {
                String sentiment = analyzeSentiment(post.getText());
                return post.withSentiment(sentiment).withProcessingStage("marketing-analysis");
            },
            Named.as("mapvalues-add-sentiment")
        );

        // STEP 2C: mapValues() — Strip PII (emails, phone numbers)
        // Chaining mapValues — each creates a NEW ContentPost instance
        KStream<String, ContentPost> piiRedacted = sentimentTagged.mapValues(
            (readOnlyKey, post) -> {
                String redacted = redactPII(post.getText());
                return post.withRedactedText(redacted);
            },
            Named.as("mapvalues-redact-pii")
        );

            .split(Named.as("branch-language-"))

                (userId, post) -> "en".equalsIgnoreCase(post.getLanguage()),

                    "english"

                    stream -> stream.to(BRAND_MENTIONS_INTL,

                    "international"
                )

        // ═══════════════════════════════════════════════════════

        KStream<String, ContentPost> euPosts = validPosts.filter(
            (userId, post) -> EU_REGIONS.contains(post.getRegion()),

        );

        KStream<String, ContentPost> euRedacted = euPosts.mapValues(

    /**
     * Check if text contains toxic/banned words.
     * In production: use ML model or external service via KTable join.

    private static boolean containsToxicContent(String text) {
        if (text == null) return false;
        String lower = text.toLowerCase(Locale.ROOT);
        return TOXIC_WORDS.stream().anyMatch(lower::contains);
    }

    /**
     * Calculate severity based on toxic word count.
     */
    private static String calculateSeverity(String text) {
        if (text == null) return "LOW";
        String lower = text.toLowerCase(Locale.ROOT);
        long toxicCount = TOXIC_WORDS.stream().filter(lower::contains).count();
        if (toxicCount >= 3) return "HIGH";
        if (toxicCount >= 2) return "MEDIUM";
        return "LOW";
    }

    /**
     * Simple sentiment analysis (placeholder for ML model).
     */
    private static String analyzeSentiment(String text) {
        if (text == null) return "NEUTRAL";
        String lower = text.toLowerCase(Locale.ROOT);
        Set<String> positive = new HashSet<>(Arrays.asList(
            "great", "amazing", "love", "excellent", "fantastic", "awesome", "incredible"
        ));
        Set<String> negative = new HashSet<>(Arrays.asList(
            "terrible", "awful", "hate", "worst", "horrible", "disappointing", "bad"
        ));

        long posCount = positive.stream().filter(lower::contains).count();
        long negCount = negative.stream().filter(lower::contains).count();

        if (posCount > negCount) return "POSITIVE";
        if (negCount > posCount) return "NEGATIVE";
        return "NEUTRAL";
    }

    /**
     * Redact PII (emails and phone numbers) from text.
     */
    private static String redactPII(String text) {
        if (text == null) return text;
        String result = EMAIL_PATTERN.matcher(text).replaceAll("[EMAIL_REDACTED]");
        result = PHONE_PATTERN.matcher(result).replaceAll("[PHONE_REDACTED]");
        return result;
    }

    /**
     * Determine compliance flags based on post content and metadata.
     */
    private static String determineComplianceFlags(ContentPost post) {
        List<String> flags = new ArrayList<>();
        if (EU_REGIONS.contains(post.getRegion())) {

            flags.add("CONTAINS_EMAIL");

            flags.add("CONTAINS_PHONE");
        }
        if (containsToxicContent(post.getText())) {
            flags.add("TOXIC_CONTENT");
        }
        return flags.isEmpty() ? "NONE" : String.join(",", flags);
    }

def consume_topic(topic, max_messages=100, timeout=8):
    """Consume messages from a single topic"""
    config = {
        'bootstrap.servers': BOOTSTRAP_SERVERS,
        'group.id': f'verifier-{topic}-{int(time.time())}',
        'auto.offset.reset': 'earliest',
        'enable.auto.commit': False,
    }
    consumer = Consumer(config)
    consumer.subscribe([topic])

    messages = []
    start = time.time()

    while len(messages) < max_messages and (time.time() - start) < timeout:
        msg = consumer.poll(timeout=1.0)
        if msg is None:
            if messages:
                break
            continue
        if msg.error():
            if msg.error().code() == KafkaError._PARTITION_EOF:
                if messages:
                    break
                continue
            continue

        try:
            key = msg.key().decode('utf-8') if msg.key() else None
            # Try JSON first
            try:
                value = json.loads(msg.value().decode('utf-8'))
            except (json.JSONDecodeError, UnicodeDecodeError):
                # Might be plain string (from flatMapValues tags)
                value = msg.value().decode('utf-8')
        except Exception:
            value = "(binary)"
            key = "(binary)"

        messages.append({
            'key': key,
            'value': value,
            'partition': msg.partition(),
            'offset': msg.offset(),
        })

    consumer.close()
    return messages

THE CRITICAL RULE LEARNED:
  ┌──────────────────────────────────────────────────────────────┐
  │ PREFER VALUE-ONLY OPERATIONS OVER KEY-CHANGING OPERATIONS   │
  │                                                              │
  │ mapValues()      > map()        → No repartition           │
  │ flatMapValues()  > flatMap()    → No repartition           │
  │ filter()         > selectKey()  → No repartition           │
  │                                                              │
  │ Key changes trigger repartition = extra topic + I/O + latency│
  └──────────────────────────────────────────────────────────────┘
