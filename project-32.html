🔴 HYPOTHETICAL PROBLEM

You run a large e-commerce platform with these data flows:

STREAM 1: "orders" topic (high velocity — 50,000 orders/minute)
  Key: customer_id
  Value: {order_id, customer_id, product_id, quantity, amount, timestamp}

STREAM 2: "customers" topic (low velocity — updates when profile changes)
  Key: customer_id
  Value: {customer_id, name, email, tier, country, credit_limit}

STREAM 3: "payments" topic (medium velocity — payment events)
  Key: order_id
  Value: {payment_id, order_id, method, status, paid_at}

STREAM 4: "products" topic (very low velocity — product catalog)
  Key: product_id
  Value: {product_id, name, category, price, warehouse_region}

THE BUSINESS REQUIREMENTS:
─────────────────────────
1. ENRICHED ORDER VIEW:
   → Every order must include customer name, tier, and country
   → "Order #12345 from John (Gold tier, USA) for $99.99"
   → Requires: JOIN orders WITH customers ON customer_id

2. ORDER-PAYMENT CORRELATION:
   → Match each order with its payment within 30 minutes
   → "Order #12345 paid via Credit Card at 14:32"
   → Requires: JOIN orders WITH payments ON order_id
   → BUT: orders keyed by customer_id, payments keyed by order_id
   → KEYS DON'T MATCH → cannot join directly

3. FULL ORDER DETAIL:
   → Enrich order with product name, category, warehouse
   → "Order #12345: iPhone 15 (Electronics) from West Coast warehouse"
   → Requires: JOIN orders WITH products ON product_id
   → BUT: orders keyed by customer_id, products keyed by product_id
   → KEYS DON'T MATCH AGAIN

4. CUSTOMER TIER CHANGES:
   → When customer tier changes, update all active enrichment
   → Gold → Platinum should reflect immediately
   → Requires: TABLE semantics (latest value per key)

THE CRISIS WITHOUT JOINS:
─────────────────────────
→ Option A: Query database for each order → 50,000 DB queries/minute
            → Database overwhelmed, latency spikes to 5 seconds
→ Option B: Cache everything in memory → stale data, cache invalidation nightmare
→ Option C: Batch join hourly → 1-hour delay, not real-time
→ Option D: Kafka Streams Joins → Real-time, no external DB, automatically updated ✓

THE PARTITIONING NIGHTMARE (connects to Project 31):
─────────────────────────────────────────────────────
→ "orders" topic: 12 partitions, keyed by customer_id
→ "customers" topic: 6 partitions, keyed by customer_id
→ "products" topic: 4 partitions, keyed by product_id
→ "payments" topic: 12 partitions, keyed by order_id

→ KStream-KTable join REQUIRES co-partitioning:
    ① Same number of partitions
    ② Same key
    ③ Same partitioning strategy

→ orders (12 partitions) JOIN customers (6 partitions)
  → FAILS: partition count mismatch
→ orders (keyed by customer_id) JOIN payments (keyed by order_id)
  → FAILS: key mismatch
→ orders (keyed by customer_id) JOIN products (keyed by product_id)
  → FAILS: key mismatch

→ HOW DO WE SOLVE ALL THREE VIOLATIONS?

🔵 HOW TO APPROACH IT

Step 1: UNDERSTAND THE FOUR JOIN TYPES IN KAFKA STREAMS
  ┌──────────────────────┬────────────────────────────────────┐
  │ Join Type            │ What It Does                       │
  ├──────────────────────┼────────────────────────────────────┤
  │ KStream-KTable       │ Enrich stream with latest table    │
  │                      │ value (LEFT JOIN or INNER JOIN)    │
  ├──────────────────────┼────────────────────────────────────┤
  │ KStream-KStream      │ Correlate two event streams within │
  │                      │ a TIME WINDOW                      │
  ├──────────────────────┼────────────────────────────────────┤
  │ KTable-KTable        │ Join two materialized views        │
  │                      │ (always latest value per key)      │
  ├──────────────────────┼────────────────────────────────────┤
  │ KStream-GlobalKTable │ Enrich stream with table that is   │
  │                      │ FULLY replicated on every instance │
  │                      │ NO co-partitioning required        │
  └──────────────────────┴────────────────────────────────────┘

Step 2: MAP EACH REQUIREMENT TO A JOIN TYPE
  Requirement 1 (order + customer) → KStream-KTable join
    → Same key (customer_id) ✓
    → Must fix partition count mismatch
  
  Requirement 2 (order + payment)  → KStream-KStream windowed join
    → Different keys → must REKEY orders to order_id first
    → Time window: 30 minutes
  
  Requirement 3 (order + product)  → KStream-GlobalKTable join
    → Different keys → GlobalKTable avoids co-partitioning
    → Products are small catalog → fits in memory
  
  Requirement 4 (customer updates) → KTable semantics
    → Automatic: KTable always reflects latest value per key

Step 3: UNDERSTAND CO-PARTITIONING (connects to Project 31)
  → For KStream-KTable and KStream-KStream joins:
      Both sides MUST have:
        ① Same number of partitions
        ② Same key type and value
        ③ Same partitioning logic (same key → same partition number)
  → WHY: Join happens LOCALLY on each Streams instance
      Instance processing Partition 5 of Topic A
      can only see Partition 5 of Topic B
      If matching keys are on different partitions → MISSED JOINS

Step 4: SOLVE CO-PARTITIONING VIOLATIONS
  → Kafka Streams can AUTOMATICALLY repartition via:
      stream.selectKey(...).through("repartition-topic")
  → This creates an intermediate topic with correct partitioning
  → Performance cost: extra network hop + disk write
  → GlobalKTable: alternative that avoids repartitioning entirely

🟡 WHAT COULD BE DONE

Option A: Application-Level Joins (Manual)
  → For each order, query customer DB, product DB, payment DB
  → Assemble enriched record in application code
  → Problem: Massive external I/O per message
  → Problem: Database becomes bottleneck at 50K msg/min
  → Problem: Network latency multiplied by join count

Option B: Kafka Streams Joins (✓ CORRECT APPROACH)
  → Join happens IN-PROCESS using local state stores
  → Customer table materialized locally (RocksDB — Project 21)
  → No external database queries
  → Automatic state store synchronization via changelog topics
  → Horizontally scalable (add more instances)

Option C: ksqlDB Joins
  → SQL-like syntax: SELECT * FROM orders JOIN customers ON ...
  → Easier for SQL-familiar teams
  → Same underlying Kafka Streams engine
  → Less control over topology optimization
  → Good for prototyping, may need Streams for production tuning

Option D: Flink / Spark Structured Streaming Joins
  → External stream processing framework
  → More powerful for complex joins (multi-way, temporal)
  → Heavier infrastructure footprint
  → Overkill for standard enrichment patterns
  → Consider for Project 50 bridge topic

Option E: Pre-Join at Source (Denormalization)
  → Producer sends already-enriched records
  → Problem: Source system must know all join targets
  → Problem: Customer name changes → must re-emit all orders
  → Problem: Tight coupling between producers

✅ WHAT SHOULD DO

1. ENSURE CO-PARTITIONING BEFORE JOINING
   → Verify BOTH topics have same partition count
   → Verify BOTH topics use same key and partitioning strategy
   → If not → either repartition or use GlobalKTable
   → NEVER assume co-partitioning — always verify

2. CHOOSE THE RIGHT JOIN TYPE
   → Stream enrichment with reference data → KStream-KTable
   → Event correlation with time window → KStream-KStream
   → Small lookup table, different keys → KStream-GlobalKTable
   → Materialized view combination → KTable-KTable

3. SET APPROPRIATE WINDOW SIZES FOR KStream-KStream
   → Too small → miss legitimate matches
   → Too large → excessive state, memory pressure
   → Match business SLA (payment within 30 min → 30 min window)

4. USE SERDES CONSISTENTLY
   → Both sides of join must use compatible serialization
   → Key serde must match exactly
   → Value serde for each side can differ
   → Output serde must handle combined record

5. HANDLE NULL VALUES IN JOINS
   → LEFT JOIN: left side always emitted, right may be null
   → INNER JOIN: both sides must exist
   → Use leftJoin() for enrichment where reference may not exist yet
   → Always null-check right side in ValueJoiner

6. MONITOR STATE STORE SIZE
   → KTable state stores grow with unique keys
   → KStream-KStream window stores grow with window size × throughput
   → Set retention/TTL appropriately
   → Monitor RocksDB metrics (Project 21 state stores)

7. NAME YOUR STATE STORES AND REPARTITION TOPICS
   → Kafka Streams auto-generates names if you don't
   → Auto-generated names change if topology changes → state loss
   → Always use .join(..., Materialized.as("store-name"))

❌ WHAT SHOULD NOT DO

1. DO NOT JOIN WITHOUT VERIFYING CO-PARTITIONING
   → Kafka Streams will NOT throw an error
   → It will silently produce WRONG results
   → Records that should match will be on different instances
   → Joins will return null where they should return data
   → THIS IS THE #1 MISTAKE IN KAFKA STREAMS

2. DO NOT USE KStream-KStream JOIN FOR LOOKUP ENRICHMENT
   → KStream-KStream requires a TIME WINDOW
   → If customer record arrived 2 hours ago and window is 30 min
     → Join misses the customer entirely
   → Use KTable for lookup/reference data (no window needed)

3. DO NOT USE GlobalKTable FOR HIGH-VOLUME DATA
   → GlobalKTable replicates ALL data to EVERY instance
   → 100 million customer records × 10 instances = 1 billion copies
   → Memory explosion
   → Use only for small reference data (products, config, codes)

4. DO NOT FORGET ABOUT REPARTITION TOPIC OVERHEAD
   → selectKey() triggers automatic repartition
   → Creates intermediate topic + writes + reads
   → Doubles latency for that join step
   → Plan capacity for repartition topics

5. DO NOT CHAIN MULTIPLE REKEYS WITHOUT AWARENESS
   → Each rekey = new repartition topic
   → orders.selectKey(order_id).join(payments)
          .selectKey(product_id).join(products)
   → Two repartition topics, two extra hops
   → Consider GlobalKTable to avoid second rekey

6. DO NOT IGNORE GRACE PERIOD IN WINDOWED JOINS
   → Late-arriving events after window closes → dropped
   → Set grace period: JoinWindows.ofTimeDifferenceWithNoGrace()
     vs JoinWindows.ofTimeDifferenceAndGrace()
   → Balance completeness vs state store size

7. DO NOT MIX CUSTOM PARTITIONER WITH DEFAULT IN JOINED TOPICS
   → Project 31 taught custom partitioning
   → If orders uses GeographicPartitioner but customers uses default
   → Same customer_id maps to DIFFERENT partitions on each topic
   → Join produces WRONG RESULTS
   → Both topics MUST use identical partitioning strategy

🚫 WHEN NOT TO USE

1. WHEN DATA VOLUME ON BOTH SIDES IS MASSIVE
   → KStream-KStream windowed join stores BOTH sides in state
   → 50K orders/min × 30 min window = 1.5M records in state
   → If payments also 50K/min → 3M total in window state
   → Consider: reduce window, use KTable instead, or external store

2. WHEN JOIN KEY REQUIRES COMPLEX EXTRACTION
   → If matching requires regex, fuzzy match, or ML similarity
   → Kafka Streams joins are EXACT KEY MATCH only
   → Use external enrichment service instead

3. WHEN JOIN RESULT MUST BE TRANSACTIONALLY CONSISTENT
   → Kafka Streams joins are eventually consistent
   → Customer update + order may not join in same processing cycle
   → If strict consistency needed → use database transactions

4. WHEN REFERENCE DATA CHANGES FASTER THAN STREAM DATA
   → KTable updates trigger re-emission of joined results
   → If customer profile changes 100 times/sec
   → Every change re-joins with all buffered orders
   → Downstream gets flooded with duplicate enrichments

5. WHEN YOU NEED MULTI-KEY JOINS
   → Standard joins work on ONE key at a time
   → JOIN orders ON customer_id AND product_id simultaneously
   → Not natively supported → chain two separate joins
   → Or use GlobalKTable for one of the lookups

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

═══════════════════════════════════════════════════════════════
          KStream-KTable JOIN — INTERNAL MECHANICS
═══════════════════════════════════════════════════════════════

PREREQUISITE: CO-PARTITIONING
  orders topic:    Partition 0  Partition 1  Partition 2  ...
  customers topic: Partition 0  Partition 1  Partition 2  ...
  
  customer_id="C100" → hash % 12 = Partition 3 (BOTH topics)
  
  Streams Instance A processes: orders P0,P1,P2 + customers P0,P1,P2
  Streams Instance B processes: orders P3,P4,P5 + customers P3,P4,P5
  
  Instance B has customer C100 in its LOCAL state store
  Instance B receives order from C100 on orders P3
  → JOIN SUCCEEDS because both are on same instance

WHAT HAPPENS ON EACH RECORD:

  When CUSTOMER record arrives (KTable side):
  ┌─────────────────────────────────────────────────┐
  │ 1. Record arrives: key=C100, value={name:"John"}│
  │ 2. Upsert into local RocksDB state store        │
  │    state_store["C100"] = {name:"John", tier:"G"} │
  │ 3. Changelog topic updated (for recovery)        │
  │ 4. NO join triggered (KTable side is passive)    │
  │    → Table just maintains latest state           │
  └─────────────────────────────────────────────────┘

  When ORDER record arrives (KStream side):
  ┌─────────────────────────────────────────────────┐
  │ 1. Record arrives: key=C100, value={order:12345}│
  │ 2. Lookup key C100 in local state store          │
  │    found = state_store.get("C100")               │
  │ 3a. IF found (INNER JOIN):                       │
  │     → Call ValueJoiner.apply(order, customer)    │
  │     → Emit enriched record downstream            │
  │ 3b. IF found (LEFT JOIN):                        │
  │     → Same as 3a                                 │
  │ 3c. IF NOT found (INNER JOIN):                   │
  │     → Record is DROPPED (no match)               │
  │ 3d. IF NOT found (LEFT JOIN):                    │
  │     → Call ValueJoiner.apply(order, NULL)         │
  │     → Emit with null customer info               │
  │ 4. Result forwarded to next topology node         │
  └─────────────────────────────────────────────────┘

  TIMING MATTERS:
  ┌─────────────────────────────────────────────────┐
  │ T=0: Customer C100 arrives → stored in table     │
  │ T=1: Order from C100 arrives → JOIN SUCCEEDS ✓   │
  │                                                   │
  │ BUT IF:                                           │
  │ T=0: Order from C100 arrives → customer NOT in    │
  │      table yet → INNER JOIN drops, LEFT JOIN      │
  │      emits with null                              │
  │ T=1: Customer C100 arrives → stored in table      │
  │      → TOO LATE for the order (already processed) │
  │                                                   │
  │ KStream-KTable join is NOT retroactive            │
  │ Table must be populated BEFORE stream events      │
  │ arrive for INNER JOIN to work correctly            │
  └─────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════════
          KStream-KStream JOIN — INTERNAL MECHANICS
═══════════════════════════════════════════════════════════════

WINDOW STATE STORES (both sides):
  
  orders-window-store:
  ┌───────────────────────────────────────────────┐
  │ Key: order_id  │ Window: [T, T+30min]         │
  │ ORD-001        │ {order_id, amount, ...}       │
  │ ORD-002        │ {order_id, amount, ...}       │
  │ ORD-003        │ {order_id, amount, ...}       │
  │ ... (all orders in last 30 minutes)            │
  └───────────────────────────────────────────────┘

  payments-window-store:
  ┌───────────────────────────────────────────────┐
  │ Key: order_id  │ Window: [T, T+30min]         │
  │ ORD-001        │ {payment_id, method, ...}     │
  │ ORD-002        │ {payment_id, method, ...}     │
  │ ... (all payments in last 30 minutes)          │
  └───────────────────────────────────────────────┘

  When ORDER arrives:
  ┌─────────────────────────────────────────────────┐
  │ 1. Store in orders-window-store                  │
  │ 2. Scan payments-window-store for matching key   │
  │    within [order.timestamp - 30min,              │
  │            order.timestamp + 30min]              │
  │ 3a. IF match found → emit joined result          │
  │ 3b. IF no match → wait (payment may arrive later)│
  └─────────────────────────────────────────────────┘

  When PAYMENT arrives:
  ┌─────────────────────────────────────────────────┐
  │ 1. Store in payments-window-store                │
  │ 2. Scan orders-window-store for matching key     │
  │    within window range                           │
  │ 3a. IF match found → emit joined result          │
  │ 3b. IF no match → wait (order may arrive later)  │
  └─────────────────────────────────────────────────┘

  SYMMETRIC: Both sides trigger lookups on the other side
  
  WINDOW EXPIRY:
  → After 30 minutes + grace period, records are evicted
  → If payment arrives 31 minutes after order → NO JOIN
  → Lost forever (unless you have a DLQ pattern — Project 25)

═══════════════════════════════════════════════════════════════
          GlobalKTable — INTERNAL MECHANICS
═══════════════════════════════════════════════════════════════

  NORMAL KTable (partitioned):
  ┌───────────────────────────────────────────────┐
  │ Instance A sees: products P0, P1, P2          │
  │ Instance B sees: products P3, P4, P5          │
  │ Instance C sees: products P6, P7, P8          │
  │ Each instance has PARTIAL view                 │
  │ Join only works if keys co-partitioned         │
  └───────────────────────────────────────────────┘

  GlobalKTable (fully replicated):
  ┌───────────────────────────────────────────────┐
  │ Instance A sees: ALL products (P0–P11)         │
  │ Instance B sees: ALL products (P0–P11)         │
  │ Instance C sees: ALL products (P0–P11)         │
  │ Each instance has COMPLETE view                │
  │ Join works with ANY key — no co-partitioning   │
  └───────────────────────────────────────────────┘

  HOW IT WORKS:
  1. On startup, each instance reads ENTIRE products topic
  2. Materializes ALL records into local RocksDB
  3. New product records → ALL instances update their local copy
  4. Join: extract product_id from order → lookup in local store
  5. ALWAYS finds the product (if it exists anywhere)

  COST:
  → N instances × full dataset = N copies in memory/disk
  → Every product update → N network messages
  → Only viable for SMALL datasets (< millions of records)

═══════════════════════════════════════════════════════════════
          REPARTITIONING — INTERNAL MECHANICS
═══════════════════════════════════════════════════════════════

  PROBLEM:
  orders topic: keyed by customer_id
  payments topic: keyed by order_id
  → Cannot join because keys don't match

  SOLUTION: selectKey() + automatic repartition

  orders.selectKey((k, v) -> v.order_id)
  
  WHAT KAFKA STREAMS DOES INTERNALLY:
  ┌─────────────────────────────────────────────────┐
  │ 1. Creates internal topic:                       │
  │    "app-id-orders-KSTREAM-KEY-SELECT-repartition"│
  │    Same partition count as original               │
  │                                                   │
  │ 2. For each order record:                         │
  │    Old key: customer_id="C100"                    │
  │    New key: order_id="ORD-12345"                  │
  │    Produces to repartition topic with new key     │
  │    New partition = hash(ORD-12345) % numPartitions│
  │                                                   │
  │ 3. Downstream topology reads from repartition     │
  │    topic (not original orders topic)              │
  │                                                   │
  │ 4. Now both sides keyed by order_id               │
  │    → Co-partitioning satisfied → join works       │
  └─────────────────────────────────────────────────┘

  PERFORMANCE IMPACT:
  → Extra write to repartition topic (network + disk)
  → Extra read from repartition topic
  → Roughly 2x latency for the rekeyed stream
  → Additional storage for repartition topic
  → Repartition topic has same retention as source

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: SILENT CO-PARTITIONING VIOLATION
  → orders: 12 partitions, customers: 6 partitions
  → Kafka Streams starts without error
  → customer_id="C100" → orders Partition 7, customers Partition 1
  → Instance processing orders P7 does NOT have customers P1
  → INNER JOIN: order dropped (no customer found)
  → LEFT JOIN: order emitted with null customer
  → NO ERROR MESSAGE — just wrong results
  → FIX: Always verify partition counts match before deploying
  → FIX: Add validation in application startup

PITFALL 2: RACE CONDITION — TABLE NOT READY
  → Application starts → immediately processes orders
  → KTable for customers hasn't fully loaded yet
  → First 10,000 orders join with null customer
  → FIX: Use LEFT JOIN and handle null gracefully
  → FIX: Pre-populate KTable before sending orders
  → FIX: Add startup delay or readiness check

PITFALL 3: WINDOWED JOIN STATE EXPLOSION
  → KStream-KStream join with 1-hour window
  → 50,000 orders/min × 60 min = 3 million records in state
  → RocksDB memory pressure → slow lookups → consumer lag
  → FIX: Use smallest window that meets business needs
  → FIX: Increase state store cache size
  → FIX: Monitor state store metrics

PITFALL 4: REPARTITION TOPIC NOT CLEANED UP
  → Internal repartition topics created automatically
  → Application deleted but repartition topics remain
  → Disk space consumed indefinitely
  → FIX: Use kafka-streams application reset tool
  → FIX: Set retention on internal topics

PITFALL 5: DUPLICATE JOINS IN OUTER JOIN
  → KStream-KStream OUTER JOIN:
      Order arrives → emits (order, null) — no payment yet
      Payment arrives → emits (order, payment) — matched
  → Downstream receives TWO records for same order
  → FIX: Use downstream deduplication
  → FIX: Use INNER JOIN if both sides guaranteed

PITFALL 6: GlobalKTable MEMORY OVERFLOW
  → Used GlobalKTable for customers (100 million records)
  → Each instance loads ALL 100 million into local store
  → 10 instances × 100M × 1KB = 1TB total memory
  → Application OOM kills across cluster
  → FIX: Use regular KTable with co-partitioning instead
  → FIX: GlobalKTable only for < 1 million records

PITFALL 7: KEY TYPE MISMATCH
  → orders key serialized as String: "12345"
  → payments key serialized as Long: 12345L
  → Same logical value but different bytes
  → Hash produces different partition assignment
  → Join never matches
  → FIX: Ensure identical key serde on both topics

📈 SCALABILITY CONSIDERATIONS

DIMENSION 1: STATE STORE SIZE

"""
PROJECT 32: Order Enrichment Join Engine
==========================================
Demonstrates all four join types using confluent-kafka Consumer/Producer.

NOTE ON IMPLEMENTATION APPROACH:
─────────────────────────────────
Kafka Streams (Java) has native join operators:
  stream.join(table, valueJoiner)
  stream.join(otherStream, valueJoiner, JoinWindows.of(Duration))

Python does NOT have a native Kafka Streams library with join support.
Options:
  A) Faust (Python stream processing) — has join support but limited
  B) Manual join implementation — full control, educational
  C) ksqlDB — SQL interface to Kafka Streams

We implement MANUAL JOINS to understand the EXACT internal mechanics
that Kafka Streams performs automatically. This teaches:
  - Why co-partitioning matters (you'll see it in the code)
  - How state stores work (we build them explicitly)
  - How windowed joins manage time (we implement the window logic)
  - Why GlobalKTable is different (we load all partitions)

In production Java: use Kafka Streams DSL
In production Python: use Faust or ksqlDB
"""

import json
import time
import threading
import signal
from collections import defaultdict
from confluent_kafka import Consumer, Producer, TopicPartition


# ═══════════════════════════════════════════════════════════════
#  JOIN TYPE 1: KStream-KTable JOIN (Order + Customer Enrichment)
# ═══════════════════════════════════════════════════════════════

class OrderCustomerJoin:
    """
    Implements KStream-KTable LEFT JOIN manually.
    
    Internal equivalent of Kafka Streams:
      KStream<String, Order> orders = builder.stream("orders");
      KTable<String, Customer> customers = builder.table("customers");
      orders.leftJoin(customers, (order, customer) -> enrich(order, customer))
            .to("enriched-orders");
    
    HOW IT WORKS:
    1. Consumer Group A reads "customers" topic → builds local state store (dict)
    2. Consumer Group B reads "orders" topic → for each order, lookup customer in store
    3. If found → emit enriched record
    4. If not found → emit with null customer (LEFT JOIN behavior)
    
    CO-PARTITIONING:
    Both topics have 6 partitions, keyed by customer_id.
    This consumer reads BOTH topics with same partition assignment.
    customer_id="CUST-00050" → Partition 3 on BOTH topics
    This instance processes Partition 3 of both → JOIN SUCCEEDS
    """

    def __init__(self, bootstrap_servers: str = "localhost:9092"):
        self.bootstrap = bootstrap_servers
        self.running = True

        # ── LOCAL STATE STORE (equivalent of RocksDB in Kafka Streams) ──
        # In real Kafka Streams: backed by RocksDB with changelog topic
        # Here: in-memory dict (sufficient for demonstration)
        self.customer_store = {}
        self.store_lock = threading.Lock()

        # ── METRICS ──
        self.orders_processed = 0
        self.joins_matched = 0
        self.joins_missed = 0

        # ── OUTPUT PRODUCER ──
        self.producer = Producer({
            "bootstrap.servers": bootstrap_servers,
            "acks": "all",
        })

        signal.signal(signal.SIGINT, self._shutdown)

    def _shutdown(self, *args):
        self.running = False

    # ── PHASE 1: Materialize customer KTable ──

    def _materialize_customer_table(self):
        """
        Read ALL existing customer records to build initial state store.
        
        In Kafka Streams, this happens automatically when KTable is created.
        The KTable consumer reads from earliest offset and populates
        the local RocksDB state store before processing begins.
        
        This is why KTable topics should use cleanup.policy=compact
        (Project 14) — only latest value per key is retained.
        """
        print("\n  ── Phase 1: Materializing Customer KTable ──")

        table_consumer = Consumer({
            "bootstrap.servers": self.bootstrap,
            "group.id": "customer-table-materializer",
            "auto.offset.reset": "earliest",
            "enable.auto.commit": True,
        })
        table_consumer.subscribe(["customers"])

        # Read until no new messages for 3 seconds
        last_message_time = time.time()
        count = 0

        while time.time() - last_message_time < 3:
            msg = table_consumer.poll(timeout=1.0)

            if msg is None:
                continue
            if msg.error():
                continue

            try:
                customer = json.loads(msg.value().decode("utf-8"))
                customer_id = msg.key().decode("utf-8")

                # UPSERT: always store latest value for this key
                # This is exactly what KTable does internally
                with self.store_lock:
                    self.customer_store[customer_id] = customer

                count += 1
                last_message_time = time.time()

            except (json.JSONDecodeError, UnicodeDecodeError):
                continue

        table_consumer.close()
        print(f"  ✅ Customer table materialized: {count} records, "
              f"{len(self.customer_store)} unique customers")

    # ── PHASE 2: Process order stream with table lookups ──

    def _process_order_stream(self, max_orders: int = None):
        """
        Read orders and JOIN with customer state store.
        
        For each order:
        1. Extract customer_id from key
        2. Lookup in local customer_store
        3. Call value_joiner to create enriched record
        4. Produce enriched record to output topic
        
        This is EXACTLY what Kafka Streams does in:
          orders.leftJoin(customers, valueJoiner)
        """
        print("\n  ── Phase 2: Processing Order Stream with Table Lookups ──")

        stream_consumer = Consumer({
            "bootstrap.servers": self.bootstrap,
            "group.id": "order-customer-join-processor",
            "auto.offset.reset": "earliest",
            "enable.auto.commit": True,
        })
        stream_consumer.subscribe(["orders"])

        while self.running:
            msg = stream_consumer.poll(timeout=1.0)

            if msg is None:
                if max_orders and self.orders_processed >= max_orders:
                    break
                continue
            if msg.error():
                continue

            try:
                order = json.loads(msg.value().decode("utf-8"))
                customer_id = msg.key().decode("utf-8")

                # ── TABLE LOOKUP (the actual JOIN operation) ──
                with self.store_lock:
                    customer = self.customer_store.get(customer_id)

                # ── VALUE JOINER ──
                enriched = self._value_joiner_order_customer(order, customer)

                # ── EMIT TO OUTPUT TOPIC ──
                self.producer.produce(
                    topic="enriched-orders",
                    key=customer_id.encode("utf-8"),
                    value=json.dumps(enriched).encode("utf-8"),
                )

                self.orders_processed += 1
                if customer:
                    self.joins_matched += 1
                else:
                    self.joins_missed += 1

                # ── SAMPLE LOGGING ──
                if self.orders_processed % 50 == 1:
                    match_status = "✅ MATCHED" if customer else "⚠️ NO CUSTOMER"
                    tier = customer.get("tier", "N/A") if customer else "N/A"
                    print(
                        f"    [{self.orders_processed}] "
                        f"Order {order['order_id']} | "
                        f"Customer {customer_id} | "
                        f"Tier: {tier} | "
                        f"{match_status}"
                    )

                if self.orders_processed % 100 == 0:
                    self.producer.poll(0)

                if max_orders and self.orders_processed >= max_orders:
                    break

            except (json.JSONDecodeError, UnicodeDecodeError):
                continue

        self.producer.flush()
        stream_consumer.close()

    def _value_joiner_order_customer(self, order: dict, customer: dict) -> dict:
        """
        The ValueJoiner function — combines order with customer data.
        
        In Kafka Streams Java:
          ValueJoiner<Order, Customer, EnrichedOrder> joiner = 
              (order, customer) -> new EnrichedOrder(order, customer);
        
        LEFT JOIN: customer may be None
        INNER JOIN: customer is guaranteed non-None
        """
        enriched = {
            # Order fields
            "order_id": order["order_id"],
            "customer_id": order["customer_id"],
            "product_id": order["product_id"],
            "quantity": order["quantity"],
            "amount": order["amount"],
            "currency": order.get("currency", "USD"),
            "order_timestamp": order.get("timestamp"),

            # Customer fields (may be null for LEFT JOIN)
            "customer_name": customer["name"] if customer else None,
            "customer_tier": customer["tier"] if customer else None,
            "customer_country": customer["country"] if customer else None,
            "customer_email": customer["email"] if customer else None,
            "credit_limit": customer.get("credit_limit") if customer else None,

            # Join metadata
            "join_type": "KStream-KTable-LEFT",
            "join_matched": customer is not None,
            "enriched_at": int(time.time() * 1000),
        }
        return enriched

    # ── PHASE 3: Handle KTable updates (customer tier changes) ──

    def _watch_table_updates(self):
        """
        Background thread that watches for customer updates.
        
        In Kafka Streams, this is automatic:
        - KTable consumer continuously reads new records
        - State store updated in real-time
        - Next order from that customer gets NEW tier value
        
        We simulate this with a background consumer thread.
        """

class OrderPaymentWindowedJoin:
    """
    Implements KStream-KStream WINDOWED JOIN manually.

    Internal equivalent of Kafka Streams:
      KStream<String, Order> orders = ...;
      KStream<String, Payment> payments = ...;
      
      // REKEY orders from customer_id to order_id first!
      KStream<String, Order> ordersRekeyed = orders
          .selectKey((k, v) -> v.getOrderId());
      
      ordersRekeyed.join(
          payments,
          (order, payment) -> combine(order, payment),
          JoinWindows.ofTimeDifferenceAndGrace(
              Duration.ofMinutes(30),
              Duration.ofMinutes(5)
          )
      ).to("order-payment-matched");

    KEY CHALLENGE:
    orders keyed by customer_id, payments keyed by order_id
    → Must REKEY orders to order_id before joining
    → In Kafka Streams: selectKey() triggers automatic repartition topic
    → Here: we extract order_id and use it as lookup key
    """

    def __init__(self, bootstrap_servers: str = "localhost:9092"):
        self.bootstrap = bootstrap_servers
        self.running = True

        # ── WINDOW STATE STORES ──
        # In Kafka Streams: WindowStore backed by RocksDB
        # Here: in-memory dicts with timestamp tracking
        # Format: {order_id: [(timestamp, record), ...]}
        self.order_window_store = defaultdict(list)
        self.payment_window_store = defaultdict(list)
        self.store_lock = threading.Lock()

        # ── WINDOW CONFIGURATION ──
        self.window_size_ms = 30 * 60 * 1000  # 30 minutes
        self.grace_period_ms = 5 * 60 * 1000  # 5 minutes grace

        # ── METRICS ──
        self.orders_received = 0
        self.payments_received = 0
        self.joins_emitted = 0
        self.orders_expired_unmatched = 0

        # ── OUTPUT PRODUCER ──
        self.producer = Producer({
            "bootstrap.servers": bootstrap_servers,
            "acks": "all",
        })

        # ── ALREADY EMITTED (prevent duplicate joins) ──
        self.emitted_pairs = set()

        signal.signal(signal.SIGINT, self._shutdown)

    def _shutdown(self, *args):
        self.running = False

    def _is_within_window(self, timestamp_a: int, timestamp_b: int) -> bool:
        """Check if two timestamps fall within the join window"""
        return abs(timestamp_a - timestamp_b) <= self.window_size_ms

    def _evict_expired_records(self, current_time_ms: int):
        """
        Remove records outside the window + grace period.
        
        In Kafka Streams, this is handled by:
        - WindowStore retention period
        - Segment-based cleanup
        - Grace period before final eviction
        """
        cutoff = current_time_ms - self.window_size_ms - self.grace_period_ms

        with self.store_lock:
            # Evict expired orders
            for order_id in list(self.order_window_store.keys()):
                self.order_window_store[order_id] = [
                    (ts, record) for ts, record in self.order_window_store[order_id]
                    if ts > cutoff
                ]
                if not self.order_window_store[order_id]:
                    del self.order_window_store[order_id]
                    self.orders_expired_unmatched += 1

            # Evict expired payments
            for order_id in list(self.payment_window_store.keys()):
                self.payment_window_store[order_id] = [
                    (ts, record) for ts, record in self.payment_window_store[order_id]
                    if ts > cutoff
                ]
                if not self.payment_window_store[order_id]:
                    del self.payment_window_store[order_id]

    def _process_order(self, order: dict):
        """
        Process an incoming order record.
        
        STEP 1: REKEY — extract order_id (simulates selectKey())
        STEP 2: Store in order window store
        STEP 3: Probe payment window store for matching order_id
        STEP 4: If match found → emit joined result
        """
        order_id = order["order_id"]
        order_ts = order.get("timestamp", int(time.time() * 1000))

        with self.store_lock:
            # Store in window
            self.order_window_store[order_id].append((order_ts, order))

            # Probe other side for matches
            matching_payments = self.payment_window_store.get(order_id, [])
            for pay_ts, payment in matching_payments:
                if self._is_within_window(order_ts, pay_ts):
                    pair_key = (order_id, payment.get("payment_id", ""))
                    if pair_key not in self.emitted_pairs:
                        self._emit_joined_result(order, payment, order_id)
                        self.emitted_pairs.add(pair_key)

        self.orders_received += 1

    def _process_payment(self, payment: dict):
        """
        Process an incoming payment record.
        
        SYMMETRIC to _process_order:
        STEP 1: Extract order_id from payment (already the key)
        STEP 2: Store in payment window store
        STEP 3: Probe order window store for matching order_id
        STEP 4: If match found → emit joined result
        """
        order_id = payment["order_id"]
        pay_ts = payment.get("paid_at", int(time.time() * 1000))

        with self.store_lock:
            # Store in window
            self.payment_window_store[order_id].append((pay_ts, payment))

            # Probe other side for matches
            matching_orders = self.order_window_store.get(order_id, [])
            for order_ts, order in matching_orders:
                if self._is_within_window(order_ts, pay_ts):
                    pair_key = (order_id, payment.get("payment_id", ""))
                    if pair_key not in self.emitted_pairs:
                        self._emit_joined_result(order, payment, order_id)
                        self.emitted_pairs.add(pair_key)

        self.payments_received += 1

    def _emit_joined_result(self, order: dict, payment: dict, order_id: str):
        """Emit the joined order-payment record"""
        joined = {
            "order_id": order_id,
            "customer_id": order.get("customer_id"),
            "product_id": order.get("product_id"),
            "order_amount": order.get("amount"),
            "order_timestamp": order.get("timestamp"),
            "payment_id": payment.get("payment_id"),
            "payment_method": payment.get("method"),
            "payment_status": payment.get("status"),
            "payment_amount": payment.get("amount"),
            "paid_at": payment.get("paid_at"),
            "join_type": "KStream-KStream-WINDOWED",
            "window_size_minutes": self.window_size_ms // 60000,
            "join_latency_ms": abs(
                payment.get("paid_at", 0) - order.get("timestamp", 0)
            ),
            "joined_at": int(time.time() * 1000),
        }

        self.producer.produce(
            topic="order-payment-matched",
            key=order_id.encode("utf-8"),
            value=json.dumps(joined).encode("utf-8"),
        )
        self.joins_emitted += 1

        if self.joins_emitted % 50 == 1:
            print(
                f"    [{self.joins_emitted}] MATCHED: "
                f"Order {order_id} + Payment {payment.get('payment_id')} "
                f"| Method: {payment.get('method')} "
                f"| Status: {payment.get('status')}"
            )

    def run(self, max_records: int = 1000):
        """
        Execute the KStream-KStream windowed join.
        
        Reads BOTH topics simultaneously using two threads,
        similar to how Kafka Streams processes multiple
        source topics in a single topology.
        """
        print(f"\n{'═' * 65}")
        print("  JOIN TYPE 2: KStream-KStream WINDOWED JOIN")
        print("  orders (rekeyed to order_id) JOIN payments (keyed by order_id)")
        print(f"  Window: {self.window_size_ms // 60000} minutes "
              f"+ {self.grace_period_ms // 60000} min grace")
        print(f"{'═' * 65}")

        start_time = time.time()

        # ── Thread 1: Read orders ──
        def consume_orders():
            consumer = Consumer({
                "bootstrap.servers": self.bootstrap,
                "group.id": "order-payment-join-orders",
                "auto.offset.reset": "earliest",
                "enable.auto.commit": True,
            })
            consumer.subscribe(["orders"])

            idle_count = 0
            while self.running and idle_count < 5:
                msg = consumer.poll(timeout=1.0)
                if msg is None:
                    idle_count += 1
                    continue
                if msg.error():
                    continue
                idle_count = 0

                try:
                    order = json.loads(msg.value().decode("utf-8"))
                    self._process_order(order)
                except (json.JSONDecodeError, UnicodeDecodeError):
                    continue

            consumer.close()

        # ── Thread 2: Read payments ──
        def consume_payments():
            consumer = Consumer({
                "bootstrap.servers": self.bootstrap,
                "group.id": "order-payment-join-payments",
                "auto.offset.reset": "earliest",

            })
            consumer.subscribe(["payments"])

            while self.running and idle_count < 8:

                    continue

                    continue

            consumer.close()

        # ── Launch both threads ──
        order_thread = threading.Thread(
            target=consume_orders, daemon=True, name="order-reader"
        )
        payment_thread = threading.Thread(
            target=consume_payments, daemon=True, name="payment-reader"

        order_thread.start()
        payment_thread.start()

        order_thread.join(timeout=60)
        payment_thread.join(timeout=60)

        self.producer.flush()

        print(f"    Orders Received:   {self.orders_received}")

        return {

            "joined": self.joins_emitted,
            "match_rate": match_rate,
        }

class CoPartitioningValidator:
    """
    Validates co-partitioning between topics before allowing joins.

    In Kafka Streams (Java), this validation happens automatically
    at topology build time. If co-partitioning is violated,
    Kafka Streams throws TopologyException.

    In our manual implementation, we must check ourselves.
    """

    def __init__(self, bootstrap_servers: str = "localhost:9092"):
        self.admin_consumer = Consumer({
            "bootstrap.servers": bootstrap_servers,
            "group.id": "co-partition-validator-temp",
            "enable.auto.commit": False,
        })

    def validate(self, topic_a: str, topic_b: str) -> dict:
        """
        Check if two topics are co-partitioned.

        Requirements for co-partitioning:
        1. Same number of partitions
        2. Same key type (cannot verify here — application responsibility)
        3. Same partitioning strategy (default hash assumed)
        """
        metadata_a = self.admin_consumer.list_topics(topic=topic_a, timeout=10)
        metadata_b = self.admin_consumer.list_topics(topic=topic_b, timeout=10)

        partitions_a = len(metadata_a.topics[topic_a].partitions)
        partitions_b = len(metadata_b.topics[topic_b].partitions)

        is_valid = partitions_a == partitions_b

        result = {
            "topic_a": topic_a,
            "topic_b": topic_b,
            "partitions_a": partitions_a,
            "partitions_b": partitions_b,
            "co_partitioned": is_valid,
            "message": (
                f"✅ Co-partitioned ({partitions_a} partitions each)"
                if is_valid
                else f"❌ NOT co-partitioned ({topic_a}={partitions_a}, "
                     f"{topic_b}={partitions_b})"
            ),
        }

        return result

    def validate_for_join(
        self,
        stream_topic: str,
        table_topic: str,
        join_type: str
    ) -> bool:
        """
        Validate and report co-partitioning for a specific join.
        Returns True if join is safe to proceed.
        """
        if join_type == "GlobalKTable":
            print(f"  ⏩ {stream_topic} JOIN {table_topic} — "
                  f"GlobalKTable (no co-partitioning check needed)")
            return True

        result = self.validate(stream_topic, table_topic)
        print(f"  {result['message']}")

        if not result["co_partitioned"]:
            print(f"  ⚠️  JOIN WILL PRODUCE WRONG RESULTS!")
            print(f"  ⚠️  Fix: Ensure both topics have {max(result['partitions_a'], result['partitions_b'])} partitions")
            print(f"  ⚠️  Or: Use GlobalKTable instead of KTable")
            return False

        return True

"""
PROJECT 32: Repartitioning Demonstration
==========================================
Shows EXACTLY what happens when selectKey() triggers
automatic repartition in Kafka Streams.

This is the mechanism that FIXES co-partitioning violations.
"""

import json
import time
import hashlib
from confluent_kafka import Producer, Consumer
from confluent_kafka.admin import AdminClient, NewTopic


class RepartitionDemonstrator:
    """
    Demonstrates the internal repartition mechanism.
    
    SCENARIO:
    orders topic: keyed by customer_id (6 partitions)
    payments topic: keyed by order_id (6 partitions)
    
    To join these, orders must be REKEYED to order_id.
    
    WHAT KAFKA STREAMS DOES INTERNALLY:
    1. Creates topic: "app-KSTREAM-KEY-SELECT-0000000001-repartition"
    2. For each order: re-produces with new key = order_id
    3. Downstream topology reads from repartition topic
    4. Now both sides keyed by order_id → co-partitioned
    """

    def __init__(self, bootstrap_servers: str = "localhost:9092"):
        self.bootstrap = bootstrap_servers
        self.admin = AdminClient({"bootstrap.servers": bootstrap_servers})
        self.producer = Producer({
            "bootstrap.servers": bootstrap_servers,
            "acks": "all",
        })

    def demonstrate_repartition(self):
        """Show how repartitioning changes partition assignments"""
        print(f"\n{'═' * 65}")
        print("  REPARTITION DEMONSTRATION")
        print("  Showing how selectKey() changes partition assignments")
        print(f"{'═' * 65}")

        # ── Step 1: Create repartition topic ──
        repartition_topic = "orders-rekeyed-by-order-id"
        try:
            futures = self.admin.create_topics([
                NewTopic(repartition_topic, num_partitions=6, replication_factor=1)
            ])
            for t, f in futures.items():
                try:
                    f.result()
                except Exception:
                    pass
        except Exception:
            pass

        # ── Step 2: Read orders and show BEFORE repartition ──
        print("\n  BEFORE REPARTITION (keyed by customer_id):")
        print(f"  {'─' * 55}")

        consumer = Consumer({
            "bootstrap.servers": self.bootstrap,
            "group.id": "repartition-demo-reader",
            "auto.offset.reset": "earliest",
            "enable.auto.commit": True,
        })
        consumer.subscribe(["orders"])

        samples = []
        idle_count = 0
        while idle_count < 3 and len(samples) < 20:
            msg = consumer.poll(timeout=1.0)
            if msg is None:
                idle_count += 1
                continue
            if msg.error():
                continue
            idle_count = 0

            try:
                order = json.loads(msg.value().decode("utf-8"))
                original_key = msg.key().decode("utf-8")
                original_partition = msg.partition()

                samples.append({
                    "order": order,
                    "original_key": original_key,
                    "original_partition": original_partition,
                })
            except (json.JSONDecodeError, UnicodeDecodeError):
                continue

        consumer.close()

        # Show original assignments
        print(f"  {'Order ID':<14} {'Original Key':<16} {'Orig Partition':<16} "
              f"{'New Key':<14} {'New Partition':<14}")
        print(f"  {'─' * 75}")

        # ── Step 3: Rekey and show AFTER repartition ──
        for sample in samples[:15]:
            order = sample["order"]
            new_key = order["order_id"]

            # Simulate default partition assignment with new key
            # murmur2-like hash (using md5 as approximation)
            key_bytes = new_key.encode("utf-8")
            hash_val = int(hashlib.md5(key_bytes).hexdigest(), 16)
            new_partition = hash_val % 6  # 6 partitions

            moved = "← MOVED" if new_partition != sample["original_partition"] else ""

            print(
                f"  {order['order_id']:<14} "
                f"{sample['original_key']:<16} "
                f"P-{sample['original_partition']:<14} "
                f"{new_key:<14} "
                f"P-{new_partition:<10} {moved}"
            )

            # Actually produce to repartition topic
            self.producer.produce(
                topic=repartition_topic,
                key=new_key.encode("utf-8"),
                value=json.dumps(order).encode("utf-8"),
            )

        self.producer.flush()

        # ── Step 4: Show partition distribution change ──
        moved_count = 0
        for sample in samples:
            order = sample["order"]
            new_key = order["order_id"]
            key_bytes = new_key.encode("utf-8")
            hash_val = int(hashlib.md5(key_bytes).hexdigest(), 16)
            new_partition = hash_val % 6
            if new_partition != sample["original_partition"]:
                moved_count += 1

        total = len(samples)
        print(f"\n  REPARTITION SUMMARY:")
        print(f"    Total Records:     {total}")
        print(f"    Records Moved:     {moved_count} ({moved_count/total*100:.0f}%)")
        print(f"    Records Same Part: {total - moved_count}")
        print(f"\n    This is NORMAL — different keys hash to different partitions")
        print(f"    The repartition topic ensures orders are now co-partitioned")
        print(f"    with payments (both keyed by order_id)")
        print(f"{'═' * 65}\n")

"""
PROJECT 32: Co-Partitioning Fixer
====================================
Detects and fixes co-partitioning violations automatically.
Connects Project 31 (custom partitioner) with join requirements.

When two topics have different partition counts,
this tool repartitions data from one topic to match the other.
"""

import json
import time
from confluent_kafka import Consumer, Producer, TopicPartition
from confluent_kafka.admin import AdminClient, NewTopic


class CoPartitioningFixer:
    """
    Detects co-partitioning violations and provides automated repair.
    
    Three repair strategies:
    1. REPARTITION: Create new topic with correct partition count, copy data
    2. REKEY: Create new topic with different key, copy data
    3. REPORT: Just report the issue (no automated fix)
    """

    def __init__(self, bootstrap_servers: str = "localhost:9092"):
        self.bootstrap = bootstrap_servers
        self.admin = AdminClient({"bootstrap.servers": bootstrap_servers})
        self.producer = Producer({
            "bootstrap.servers": bootstrap_servers,
            "acks": "all",
            "linger.ms": 10,
            "batch.size": 65536,
        })

    def diagnose(self, topic_a: str, topic_b: str) -> dict:
        """
        Comprehensive diagnosis of co-partitioning between two topics.
        Returns detailed report of all violations found.
        """
        print(f"\n{'═' * 65}")
        print(f"  CO-PARTITIONING DIAGNOSIS")
        print(f"  Topic A: {topic_a}")
        print(f"  Topic B: {topic_b}")
        print(f"{'═' * 65}\n")

        metadata = self.admin.list_topics(timeout=10)

        # Check topics exist
        if topic_a not in metadata.topics:
            return {"error": f"Topic '{topic_a}' does not exist"}
        if topic_b not in metadata.topics:
            return {"error": f"Topic '{topic_b}' does not exist"}

        partitions_a = len(metadata.topics[topic_a].partitions)
        partitions_b = len(metadata.topics[topic_b].partitions)

        diagnosis = {
            "topic_a": {"name": topic_a, "partitions": partitions_a},
            "topic_b": {"name": topic_b, "partitions": partitions_b},
            "violations": [],
            "recommendations": [],
        }

        # Check 1: Partition count match
        if partitions_a != partitions_b:
            diagnosis["violations"].append({
                "type": "PARTITION_COUNT_MISMATCH",
                "detail": f"{topic_a} has {partitions_a}, {topic_b} has {partitions_b}",
                "severity": "CRITICAL",
            })
            diagnosis["recommendations"].append({
                "action": "REPARTITION",
                "detail": (
                    f"Create new topic with {max(partitions_a, partitions_b)} partitions "
                    f"and copy data from the smaller topic"
                ),
                "target_topic": topic_a if partitions_a < partitions_b else topic_b,
                "target_partitions": max(partitions_a, partitions_b),
            })

        # Check 2: Sample key distribution
        key_check = self._check_key_distribution(topic_a, topic_b)
        if key_check.get("mismatch"):
            diagnosis["violations"].append({
                "type": "KEY_DISTRIBUTION_MISMATCH",
                "detail": key_check["detail"],
                "severity": "HIGH",
            })
            diagnosis["recommendations"].append({
                "action": "REKEY",
                "detail": "Ensure both topics use same key for same entities",
            })

        # Print diagnosis
        if not diagnosis["violations"]:
            print("  ✅ No co-partitioning violations detected")
            print(f"  Both topics have {partitions_a} partitions")
        else:
            for v in diagnosis["violations"]:
                severity_icon = "🔴" if v["severity"] == "CRITICAL" else "🟡"
                print(f"  {severity_icon} {v['type']}: {v['detail']}")

            print(f"\n  Recommendations:")
            for i, r in enumerate(diagnosis["recommendations"], 1):
                print(f"    {i}. [{r['action']}] {r['detail']}")

        print(f"\n{'═' * 65}\n")
        return diagnosis

    def _check_key_distribution(self, topic_a: str, topic_b: str) -> dict:
        """Sample keys from both topics and check for overlap"""
        keys_a = self._sample_keys(topic_a, max_samples=100)
        keys_b = self._sample_keys(topic_b, max_samples=100)

        overlap = keys_a & keys_b
        only_a = keys_a - keys_b
        only_b = keys_b - keys_a

        result = {
            "keys_sampled_a": len(keys_a),
            "keys_sampled_b": len(keys_b),
            "overlap": len(overlap),
            "only_in_a": len(only_a),
            "only_in_b": len(only_b),
            "mismatch": False,
            "detail": "",
        }

        if len(overlap) == 0 and len(keys_a) > 0 and len(keys_b) > 0:
            result["mismatch"] = True
            result["detail"] = (
                f"Zero key overlap between topics. "
                f"Sample keys in {topic_a}: {list(keys_a)[:3]}... "
                f"Sample keys in {topic_b}: {list(keys_b)[:3]}..."
            )

        return result

    def _sample_keys(self, topic: str, max_samples: int = 100) -> set:
        """Sample unique keys from a topic"""
        consumer = Consumer({
            "bootstrap.servers": self.bootstrap,
            "group.id": f"key-sampler-{topic}-{int(time.time())}",
            "auto.offset.reset": "earliest",
            "enable.auto.commit": False,
        })
        consumer.subscribe([topic])

        keys = set()
        idle_count = 0

        while idle_count < 3 and len(keys) < max_samples:
            msg = consumer.poll(timeout=1.0)
            if msg is None:
                idle_count += 1
                continue
            if msg.error():
                continue
            idle_count = 0

            if msg.key():
                try:
                    keys.add(msg.key().decode("utf-8"))
                except UnicodeDecodeError:
                    keys.add(str(msg.key()))

        consumer.close()
        return keys

    def fix_partition_mismatch(
        self,
        source_topic: str,
        target_partition_count: int,
        new_topic_suffix: str = "-repartitioned",
        key_extractor=None
    ) -> dict:
        """
        Fix co-partitioning by creating a new topic with correct
        partition count and copying all data.
        
        Optionally rekeys data using key_extractor function.
        
        This is the MANUAL equivalent of what Kafka Streams does
        automatically with selectKey() + through().
        """
        new_topic_name = f"{source_topic}{new_topic_suffix}"

        print(f"\n{'─' * 60}")
        print(f"  FIXING: {source_topic} → {new_topic_name}")
        print(f"  Target partitions: {target_partition_count}")
        print(f"  Rekey: {'YES' if key_extractor else 'NO (preserve original key)'}")
        print(f"{'─' * 60}\n")

        # Step 1: Create new topic
        try:
            futures = self.admin.create_topics([
                NewTopic(
                    new_topic_name,
                    num_partitions=target_partition_count,
                    replication_factor=1
                )
            ])
            for t, f in futures.items():
                try:
                    f.result()
                    print(f"  ✅ Created topic: {t} ({target_partition_count} partitions)")
                except Exception as e:
                    if "already exists" in str(e).lower():
                        print(f"  ⏩ Topic exists: {t}")
                    else:
                        raise
        except Exception as e:
            print(f"  ❌ Failed to create topic: {e}")
            return {"error": str(e)}

        # Step 2: Copy data with optional rekeying
        consumer = Consumer({
            "bootstrap.servers": self.bootstrap,
            "group.id": f"repartitioner-{source_topic}-{int(time.time())}",
            "auto.offset.reset": "earliest",
            "enable.auto.commit": True,
        })
        consumer.subscribe([source_topic])

        copied = 0
        rekeyed = 0
        idle_count = 0

        while idle_count < 5:
            msg = consumer.poll(timeout=1.0)
            if msg is None:
                idle_count += 1
                continue
            if msg.error():
                continue
            idle_count = 0

            # Determine key
            original_key = msg.key()
            if key_extractor and msg.value():
                try:
                    value = json.loads(msg.value().decode("utf-8"))
                    new_key = key_extractor(value)
                    if new_key:
                        new_key_bytes = new_key.encode("utf-8")
                        rekeyed += 1
                    else:
                        new_key_bytes = original_key
                except (json.JSONDecodeError, UnicodeDecodeError, KeyError):
                    new_key_bytes = original_key
            else:
                new_key_bytes = original_key

            # Produce to new topic (let default partitioner assign partition)
            self.producer.produce(
                topic=new_topic_name,
                key=new_key_bytes,
                value=msg.value(),
                headers=msg.headers(),
            )

            copied += 1
            if copied % 500 == 0:
                self.producer.poll(0)
                print(f"    Copied {copied} records...")

        self.producer.flush()
        consumer.close()

        print(f"\n  ✅ Repartition complete:")
        print(f"    Records copied: {copied}")
        print(f"    Records rekeyed: {rekeyed}")
        print(f"    Source: {source_topic}")
        print(f"    Target: {new_topic_name} ({target_partition_count} partitions)")

        return {
            "source": source_topic,
            "target": new_topic_name,
            "records_copied": copied,
            "records_rekeyed": rekeyed,
            "target_partitions": target_partition_count,
        }

    def close(self):
        self.producer.flush()
