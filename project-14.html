PROJECT 14: THE USER PROFILE STORE — LOG COMPACTION

🔴 HYPOTHETICAL PROBLEM

You are building a user profile management system for "MegaShop" (from Project 13)

The platform has 10 million users
Each user's profile gets updated frequently:
  → Name changes (marriage, legal name change)
  → Address updates (moved to new city)
  → Preference updates (notification settings, language)
  → Loyalty tier changes (bronze → silver → gold)
  → Last login timestamp (every session)

The Profile Service publishes every update to Kafka:
  → Topic: "user.profiles"
  → Key: user_id
  → Value: full profile snapshot after the change

Current problem with TIME-BASED retention:
  → retention.ms=604800000 (7 days)
  → After 7 days, ALL old profile events are DELETED
  → A new microservice joins — needs to read current state of ALL users
  → But profiles older than 7 days are GONE
  → User "alice" updated her name 30 days ago, never updated since
  → Alice's profile is DELETED from Kafka — new service can't see it

The team asks:
  "Can Kafka keep ONLY the latest profile for each user?"
  "We have 10M users but 500M profile update events"
  "We want to keep 10M records (latest per user), not 500M"
  "We want new consumers to always get the CURRENT state"
  "How does this work internally?"
  "What happens if we DELETE a user? Can we remove their data?"

The questions:
  → What is log compaction?
  → How does it differ from time-based retention?
  → How does the compaction process actually work internally?
  → What are tombstone records and why do they matter?
  → How does key design (Project 13) affect compaction?
  → What are the performance implications?

🔵 HOW TO APPROACH IT

Step 1: Understand the TWO retention strategies in Kafka
  → DELETE policy: Remove segments older than retention.ms (time-based)
    → What we've been using in all previous projects
    → Good for event streams where history expires
  → COMPACT policy: Keep only the LATEST value per key
    → NEW in this project
    → Good for state/snapshot data where only latest matters
  → COMPACT,DELETE: Both — compact AND delete old segments
    → Hybrid approach

Step 2: Recognize that compaction is about KEYS
  → Every message with the SAME KEY overwrites the previous one
  → Key design from Project 13 DIRECTLY determines what gets compacted
  → user_id as key → one record per user retained
  → If key is null → message is NEVER compacted (no key to deduplicate)

Step 3: Understand tombstones (deletion markers)
  → To DELETE a key entirely, produce a message with value=null
  → This is called a "tombstone"
  → Tombstone tells compaction: "remove this key completely"
  → Tombstone itself is retained for delete.retention.ms, then removed

Step 4: Connect to what we already know
  → Project 4: Key determines partition — now key also determines compaction
  → Project 6: Offsets are sequential — compaction removes some offsets (gaps)
  → Project 11: Delivery guarantees — compacted topics need acks=all too
  → Project 13: Message design — key choice is critical for compaction

🟡 WHAT COULD BE DONE

OPTION A: Time-based retention with very long retention
  → Set retention.ms to 365 days
  → Keeps all history for a year
  → Problem: 500M records stored, not 10M
  → Problem: new consumer must read ALL 500M to get current state
  → Problem: storage cost grows linearly with update frequency
  → NOT SCALABLE

OPTION B: Log compaction (cleanup.policy=compact)
  → Kafka keeps ONLY the latest value per key
  → 500M updates → compacted to ~10M records (one per user)
  → New consumers read 10M records to get complete current state
  → Storage is proportional to UNIQUE KEYS, not total events
  → ✅ THIS IS THE SOLUTION

OPTION C: External database + Kafka for streaming
  → Store profiles in PostgreSQL
  → Use Kafka only for change notifications
  → New services query the database, not Kafka
  → Problem: tight coupling to the database
  → Problem: database becomes bottleneck
  → Valid but loses Kafka-as-source-of-truth benefit

OPTION D: Compact + Delete hybrid (cleanup.policy=compact,delete)
  → Compact: keep latest per key
  → Delete: also remove segments older than retention.ms
  → Use when: you want compaction but also want to limit total storage
  → Example: keep latest profile per user, but delete users inactive >90 days

✅ WHAT SHOULD DO

1. USE cleanup.policy=compact FOR STATE/SNAPSHOT TOPICS
   → User profiles, product catalog, configuration, account balances
   → Any data where "latest value per key" is the meaningful state
   → Consumers starting from beginning get complete current state

2. ALWAYS USE A MEANINGFUL KEY FOR COMPACTED TOPICS
   → Key determines WHAT gets compacted
   → user_id → one record per user
   → product_id → one record per product
   → No key (null) → message is NEVER compacted

3. USE TOMBSTONES TO DELETE KEYS
   → Produce message with key=user_id, value=null
   → Compaction will eventually remove the key entirely
   → Set delete.retention.ms long enough for consumers to see the tombstone
   → Default: 86400000 (24 hours)

4. ALWAYS PRODUCE FULL STATE SNAPSHOTS, NOT DELTAS
   → Compaction keeps ONLY the latest message per key
   → If latest message is a delta: {"name_changed": "Alice"}
   → Consumer gets only the delta, not the full profile
   → Always produce the COMPLETE current state:
     {"name": "Alice", "address": "NYC", "tier": "gold"}

5. SET segment.ms AND segment.bytes APPROPRIATELY
   → Compaction works on CLOSED (inactive) segments
   → The ACTIVE segment is NEVER compacted
   → segment.ms controls when a segment is rolled (closed)
   → Smaller segments → more frequent compaction → more up-to-date
   → But: more small files → more I/O overhead

6. MONITOR COMPACTION LAG
   → Compaction runs in the background — not instantaneous
   → There's always a window where old values still exist
   → Monitor: max.compaction.lag.ms
   → Consumers reading historical data may see duplicate keys temporarily

7. USE min.cleanable.dirty.ratio TO CONTROL COMPACTION AGGRESSIVENESS
   → 0.0 = compact as aggressively as possible
   → 0.5 = compact when 50% of log is "dirty" (uncompacted)
   → Lower ratio → more CPU for compaction, less storage
   → Higher ratio → less CPU, more storage temporarily

❌ WHAT SHOULD NOT DO

1. DO NOT USE COMPACTED TOPICS FOR EVENT STREAMS
   → Event streams: every event matters (order placed, payment received)
   → Compaction would DELETE intermediate events
   → "User placed 5 orders" → compaction keeps ONLY the last order event
   → Use cleanup.policy=delete for event streams

2. DO NOT USE NULL KEYS ON COMPACTED TOPICS
   → Null keys cannot be compacted (no key to deduplicate)
   → They accumulate forever until segment deletion
   → This defeats the entire purpose of compaction

3. DO NOT PRODUCE DELTA/PARTIAL UPDATES TO COMPACTED TOPICS
   → If you produce {"name_changed": "Alice"} (delta)
   → Then produce {"address_changed": "NYC"} (another delta)
   → Compaction keeps ONLY {"address_changed": "NYC"}
   → The name change is LOST
   → Always produce FULL STATE: {"name": "Alice", "address": "NYC"}

4. DO NOT EXPECT IMMEDIATE COMPACTION
   → Compaction is a BACKGROUND process
   → Active segment is NEVER compacted
   → There's always a delay between produce and compaction
   → Don't design systems that depend on instant compaction

5. DO NOT SET segment.ms TOO LOW
   → Very small segments (e.g., 1 minute) create many files
   → Each file has overhead: file handles, index files
   → Can overwhelm the filesystem
   → Minimum practical: 300000 (5 minutes) for aggressive compaction

6. DO NOT CONFUSE COMPACTION WITH DEDUPLICATION
   → Compaction removes older values for the SAME key
   → It does NOT deduplicate within the active segment
   → Two messages with same key in the same segment → both exist until compaction
   → Consumers may see "duplicates" before compaction runs

7. DO NOT FORGET TOMBSTONE RETENTION
   → Tombstone (value=null) is kept for delete.retention.ms
   → If a consumer is offline longer than delete.retention.ms
   → They'll never see the tombstone
   → They'll still have the old value in their local state
   → Set delete.retention.ms >= max expected consumer downtime

🚫 WHEN NOT TO USE

DO NOT USE LOG COMPACTION WHEN:
  → Every event in the history matters (event sourcing — Project 34)
  → You need to replay the FULL sequence of changes
  → You need time-series data (all readings, not just latest)
  → Events don't have a natural "entity key"
  → You need strict FIFO processing of all events

DO NOT USE cleanup.policy=compact,delete WHEN:
  → You need ALL keys preserved indefinitely
  → Your retention.ms might delete keys that are still valid
  → Use pure compact if every key must be retained forever

DO NOT USE COMPACTION AS A DATABASE REPLACEMENT WHEN:
  → You need random-access reads by key (Kafka is sequential)
  → You need complex queries (joins, aggregations)
  → You need transactions across multiple keys
  → You need sub-millisecond read latency
  → Use compacted topic as EVENT SOURCE, materialize into a database

              COMPACTION INTERNAL PROCESS
═══════════════════════════════════════════════════════════════

The Log Cleaner is a background thread pool that performs compaction.

STEP 1: IDENTIFY DIRTY VS CLEAN PORTIONS

  ┌──────────────────────┬──────────────────────────────┐
  │    CLEAN PORTION     │        DIRTY PORTION          │
  │  (already compacted) │  (not yet compacted)          │
  │                      │                                │
  │  Segment 1 (closed)  │  Segment 2 (closed)          │
  │  [A=v2, B=v1, C=v1]  │  [A=v3, B=v2, D=v1, A=v4]  │
  │                      │                                │
  │  cleaner checkpoint──►│◄── dirty portion starts here │
  │                      │                                │
  │                      │  Segment 3 (ACTIVE)           │
  │                      │  [B=v3, C=v2]                 │
  │                      │  ← NEVER compacted (active)   │
  └──────────────────────┴──────────────────────────────┘

  dirty ratio = size(dirty) / size(total)
  If dirty ratio > min.cleanable.dirty.ratio → trigger compaction


STEP 2: BUILD OFFSET MAP (in-memory)

  The cleaner reads the DIRTY portion and builds a map:
  
  ┌──────────────────────────────────────┐
  │  OFFSET MAP (key → latest offset)    │
  │                                        │
  │  Key A → offset 8  (latest for A)    │
  │  Key B → offset 9  (latest for B)    │
  │  Key C → offset 7  (latest for C)    │
  │  Key D → offset 6  (latest for D)    │
  │                                        │
  │  Memory used: num_unique_keys × 24B  │
  │  (key hash + offset)                  │
  └──────────────────────────────────────┘

  IMPORTANT: The offset map uses a HASH of the key, not the full key
  → Memory usage: ~24 bytes per unique key
  → 10M unique keys ≈ 240MB of memory for offset map
  → Controlled by: log.cleaner.dedupe.buffer.size


STEP 3: REWRITE SEGMENTS

  Read each segment in the dirty portion:
    → For each record, check if its offset == latest offset for that key
    → If YES → keep the record (write to new segment)
    → If NO → skip the record (it's an older version)

  Original Segment:
    [A=v1 @0] [B=v1 @1] [A=v2 @2] [C=v1 @3] [B=v2 @4] [A=v3 @5]

  After compaction:
    [C=v1 @3] [B=v2 @4] [A=v3 @5]
    
  (offsets 0, 1, 2 removed — their keys have newer values)


STEP 4: SWAP SEGMENTS

  → Old segment file is replaced with new (smaller) segment
  → This is an ATOMIC file swap
  → Consumers are NOT interrupted
  → New segment has the same offset range but fewer records


═══════════════════════════════════════════════════════════════
              TOMBSTONE MECHANICS
═══════════════════════════════════════════════════════════════

To DELETE a key, produce a record with value=null:

  producer.produce(topic='profiles', key='user-123', value=None)

This creates a TOMBSTONE record:

  Offset: 10
  Key:    user-123
  Value:  null  ← THIS IS THE TOMBSTONE

Tombstone lifecycle:

  TIME 0:   Tombstone produced at offset 10
             Previous values for user-123 still exist
             
  TIME 1:   Compaction runs
             → Removes all older values for user-123
             → KEEPS the tombstone (consumers need to see it)
             
  TIME 2:   delete.retention.ms passes (default 24 hours)
             → Tombstone is now eligible for removal
             
  TIME 3:   Next compaction runs
             → Removes the tombstone itself
             → Key "user-123" is completely gone from the log

WHY keep the tombstone temporarily?
  → Downstream consumers may be offline
  → When they come back, they need to see the deletion
  → If tombstone was removed immediately:
    → Consumer still has old value in their local state
    → They never learn the key was deleted
    → Their state is STALE


═══════════════════════════════════════════════════════════════
              ACTIVE SEGMENT — THE COMPACTION BLIND SPOT
═══════════════════════════════════════════════════════════════

  Segment 1 (closed)    Segment 2 (closed)    Segment 3 (ACTIVE)
  ┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
  │ CAN be compacted │  │ CAN be compacted │  │ NEVER compacted  │
  │                  │  │                  │  │                  │
  │ A=v1 (removed)   │  │ A=v3 (removed)   │  │ A=v5 (kept)     │
  │ B=v1 (kept)      │  │ B=v2 (kept)      │  │ A=v6 (kept too!)│
  │ A=v2 (removed)   │  │ C=v1 (kept)      │  │                  │
  └──────────────────┘  └──────────────────┘  └──────────────────┘
                                                      ↑
                                               Both A=v5 and A=v6
                                               exist in active segment
                                               until segment rolls

  To control when segments roll:
    segment.ms = 300000      (roll every 5 minutes)
    segment.bytes = 1073741824  (roll at 1GB, default)

  Smaller segment.ms → faster compaction → more up-to-date
  But: more small files → more file handles → more I/O

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "I set cleanup.policy=compact but nothing is getting compacted"
  → Compaction only works on CLOSED segments
  → The active segment is NEVER compacted
  → If your topic has low volume, segments may never close
  → FIX: Set segment.ms to a reasonable value (e.g., 300000 = 5 min)
  → Or set segment.bytes to a smaller value

PITFALL 2: "I produced a tombstone but the key is still there"
  → Tombstone doesn't delete immediately
  → Compaction must run first to remove older values
  → Tombstone itself stays for delete.retention.ms
  → FIX: Wait for compaction cycle + delete.retention.ms

PITFALL 3: "Consumer sees offset gaps — thinks messages are lost"
  → After compaction, offsets have gaps (e.g., 0, 5, 8, 12)
  → Offsets 1-4, 6-7, 9-11 were compacted away
  → This is NORMAL for compacted topics
  → Consumer auto-seeks to next available offset
  → FIX: Don't assume contiguous offsets on compacted topics

PITFALL 4: "Compaction is using too much memory"
  → The offset map (key hash → latest offset) lives in memory
  → log.cleaner.dedupe.buffer.size controls this (default 128MB)
  → If buffer is too small, compaction processes in chunks (slower)
  → FIX: Increase buffer for topics with many unique keys
  → 10M keys × 24 bytes = 240MB needed

PITFALL 5: "I'm producing deltas and losing data after compaction"
  → Produce: {key=user-123, value={"name": "Alice"}}
  → Produce: {key=user-123, value={"address": "NYC"}}
  → After compaction: ONLY {"address": "NYC"} remains
  → Name is LOST because it was in the older message
  → FIX: Always produce FULL STATE snapshots

PITFALL 6: "Compaction isn't keeping up — topic keeps growing"
  → Too many unique keys being produced too fast
  → Compaction thread pool is undersized
  → FIX: Increase log.cleaner.threads (default 1)
  → FIX: Decrease min.cleanable.dirty.ratio (more aggressive compaction)
  → FIX: Increase log.cleaner.dedupe.buffer.size

PITFALL 7: "I used null key and messages aren't being compacted"
  → Null keys CANNOT be compacted — there's nothing to deduplicate
  → They accumulate until segment-level deletion
  → FIX: Never use null keys on compacted topics
  → (Key design from Project 4 and Project 13 is critical here)

📈 SCALABILITY CONSIDERATIONS

STORAGE IMPACT:

  Without compaction (delete policy):
    Storage = message_rate × avg_size × retention_time
    10K updates/sec × 500B × 7 days = ~3 TB

  With compaction:
    Storage ≈ unique_keys × avg_latest_size × (1 + dirty_ratio)
    10M users × 500B × 1.5 = ~7.5 GB
    
  SAVINGS: 3 TB → 7.5 GB (400x reduction!)

COMPACTION RESOURCE USAGE:
  ┌────────────────────────────┬─────────────────────────────────┐
  │ Resource                   │ Impact                          │
  ├────────────────────────────┼─────────────────────────────────┤
  │ CPU                        │ Reading + rewriting segments    │
  │                            │ Proportional to dirty data size │
  │                            │ Controlled by cleaner threads   │
  ├────────────────────────────┼─────────────────────────────────┤
  │ Memory                     │ Offset map: ~24B per unique key │
  │                            │ dedupe.buffer.size controls max │
  │                            │ 10M keys ≈ 240MB               │
  ├────────────────────────────┼─────────────────────────────────┤
  │ Disk I/O                   │ Read dirty segments             │
  │                            │ Write new compacted segments    │
  │                            │ ~2x read amplification          │
  ├────────────────────────────┼─────────────────────────────────┤
  │ Network                    │ Zero — compaction is LOCAL      │
  │                            │ Runs on each broker for its     │
  │                            │ own partitions                  │
  └────────────────────────────┴─────────────────────────────────┘

CONSUMER BOOTSTRAP TIME:
  → New consumer reading compacted topic from beginning
  → Reads unique_keys records (not total_produced records)
  → 10M records × 500B = ~5GB to read
  → At 100MB/sec consumer throughput ≈ 50 seconds
  → Without compaction: 500M records ≈ 5000 seconds (83 minutes)

PARTITION COUNT FOR COMPACTED TOPICS:
  → More partitions → more parallel compaction
  → But: each partition has its own active segment (uncompacted window)
  → Balance: enough partitions for parallelism, not too many for overhead
  → Rule of thumb: same partitioning as regular topics

   → kafka.log:type=LogCleaner,name=max-clean-time-secs

4. VERIFY COMPACTION BY CONSUMING FROM BEGINNING
   → Start new consumer with auto.offset.reset=earliest
   → Count unique keys vs total messages
   → After compaction: unique keys ≈ total messages
   → Before compaction: unique keys << total messages

5. VERIFY TOMBSTONES
   → Produce tombstone for a key
   → Wait for compaction cycle
   → Consume from beginning
   → Key should be absent (after delete.retention.ms)

6. CHECK FOR OFFSET GAPS
   → Consume and track offsets
   → Compacted topic will have non-contiguous offsets
   → This confirms compaction has run

class UserProfile:
    """
    Represents a complete user profile state.
    
    CRITICAL FOR COMPACTION:
      Every produce must contain the FULL current state.
      Not deltas. Not patches. The COMPLETE profile.
      Because after compaction, only the LATEST message survives.
    """

                 address_country, language, loyalty_tier,

        self.loyalty_tier = loyalty_tier

        self.last_login = last_login or datetime.now(timezone.utc).isoformat()
        self.updated_at = datetime.now(timezone.utc).isoformat()
        self.version = 1
    
    def to_dict(self):
        """
        Serialize to dictionary — this is the FULL STATE SNAPSHOT.
        Every field is included every time.
        """
        return {
            'user_id': self.user_id,

        }

class ProfileProducer:
    """
    Produces user profile snapshots to the compacted topic.
    Also produces change events to the non-compacted events topic.
    """
    
    def __init__(self, bootstrap_servers):
        self.running = True
        self.stats = defaultdict(int)
        
        # ── Producer with strong guarantees (Project 11) ──
        self.producer = Producer({
            'bootstrap.servers': bootstrap_servers,
            'acks': 'all',
            'enable.idempotence': True,
            'batch.size': 16384,
            'linger.ms': 5,
            'compression.type': 'lz4',
        })
        
        signal.signal(signal.SIGINT, lambda s, f: setattr(self, 'running', False))
    
    def _delivery_cb(self, err, msg):
        if err:
            self.stats['errors'] += 1
        else:
            self.stats['delivered'] += 1
            self.stats[f'delivered_{msg.topic()}'] += 1
    
    def publish_profile(self, profile):
        """
        Publish a FULL STATE SNAPSHOT to the compacted topic.
        
        KEY = user_id → compaction keeps latest per user
        VALUE = complete profile → after compaction, this is the "current" state
        """
        key = profile.user_id
        value = profile.to_dict()
        
        # ── Headers for metadata (Project 13 pattern) ──
        headers = {
            'event_type': 'profile.updated',
            'content_type': 'application/json',
            'profile_version': str(value['profile_version']),
        }
        
        header_list = [
            (k, v.encode('utf-8') if isinstance(v, str) else v)
            for k, v in headers.items()
        ]
        
        # ── Produce to COMPACTED topic (latest state per key) ──
        self.producer.produce(
            topic='user.profiles',
            key=key.encode('utf-8'),
            value=json.dumps(value).encode('utf-8'),
            headers=header_list,
            callback=self._delivery_cb,
        )
        
        # ── Also produce to NON-COMPACTED topic (full history) ──
        # This lets event-sourcing consumers see ALL changes
        change_event = {
            'event_id': str(uuid.uuid4()),
            'event_type': 'profile.updated',
            'event_time': value['updated_at'],
            'data': value,
            'metadata': {
                'source_service': 'profile-service',
                'profile_version': value['profile_version'],
            }
        }
        
        self.producer.produce(
            topic='user.profile.events',
            key=key.encode('utf-8'),
            value=json.dumps(change_event).encode('utf-8'),
            callback=self._delivery_cb,
        )
        
        self.stats['published'] += 1
        self.producer.poll(0)
    
    def delete_profile(self, user_id):
        """
        Delete a user profile by sending a TOMBSTONE.
        
        ══════════════════════════════════════════════════
        TOMBSTONE: key=user_id, value=None
        This tells the compaction process to DELETE this key.
        ══════════════════════════════════════════════════
        """
        # ── Tombstone to compacted topic ──
        self.producer.produce(
            topic='user.profiles',
            key=user_id.encode('utf-8'),
            value=None,                    # ← THIS IS THE TOMBSTONE
            headers=[
                ('event_type', b'profile.deleted'),
            ],
            callback=self._delivery_cb,
        )
        
        # ── Deletion event to events topic ──
        deletion_event = {
            'event_id': str(uuid.uuid4()),
            'event_type': 'profile.deleted',
            'event_time': datetime.now(timezone.utc).isoformat(),
            'data': {'user_id': user_id, 'deleted': True},
            'metadata': {'source_service': 'profile-service'},
        }
        
        self.producer.produce(
            topic='user.profile.events',
            key=user_id.encode('utf-8'),
            value=json.dumps(deletion_event).encode('utf-8'),
            callback=self._delivery_cb,
        )
        
        self.stats['tombstones'] += 1
        self.producer.poll(0)
        print(f"  🪦 Tombstone sent for {user_id}")
    
    def shutdown(self):
        remaining = self.producer.flush(timeout=15)
        print(f"\n  📊 Producer Stats:")
        for k, v in sorted(self.stats.items()):
            print(f"     {k}: {v}")
        if remaining > 0:
            print(f"  ⚠️  {remaining} unflushed")

# ============================================================
# FILE: 03_compaction_verifier.py
# PURPOSE: Verify compaction by consuming from beginning
# TEACHES: How to observe compaction effects, offset gaps,
#          tombstone handling, and state reconstruction
# ============================================================

from confluent_kafka import Consumer, KafkaError
import json
import time
import signal
from collections import defaultdict

BOOTSTRAP = 'localhost:9092,localhost:9093,localhost:9094'


class CompactionVerifier:
    """
    Reads a compacted topic from beginning and analyzes:
      → How many records per key (should be 1 after full compaction)
      → Offset gaps (evidence of compaction)
      → Tombstone presence
      → State reconstruction accuracy
    """
    
    def __init__(self, bootstrap_servers, topic):
        self.topic = topic
        self.running = True
        
        # ── Fresh consumer group to read from beginning ──
        self.consumer = Consumer({
            'bootstrap.servers': bootstrap_servers,
            'group.id': f'compaction-verifier-{int(time.time())}',
            'auto.offset.reset': 'earliest',
            'enable.auto.commit': False,
        })
        
        self.consumer.subscribe([topic])
        signal.signal(signal.SIGINT, lambda s, f: setattr(self, 'running', False))
        
        # ── Tracking ──
        self.records_per_key = defaultdict(int)
        self.latest_value_per_key = {}
        self.tombstones = set()
        self.offsets_seen = []
        self.total_records = 0
        self.null_key_records = 0
        self.partition_records = defaultdict(int)
    
    def consume_all(self):
        """Read all available records from the topic."""
        print(f"  📥 Reading topic '{self.topic}' from beginning...\n")
        
        empty_polls = 0
        MAX_EMPTY = 30
        
        while self.running and empty_polls < MAX_EMPTY:
            msg = self.consumer.poll(timeout=0.5)
            
            if msg is None:
                empty_polls += 1
                continue
            
            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    empty_polls += 1
                    continue
                print(f"  ❌ Error: {msg.error()}")
                continue
            
            empty_polls = 0
            self.total_records += 1
            
            offset = msg.offset()
            partition = msg.partition()
            self.offsets_seen.append((partition, offset))
            self.partition_records[partition] += 1
            
            # ── Handle key ──
            if msg.key() is None:
                self.null_key_records += 1
                continue
            
            key = msg.key().decode('utf-8')
            
            # ── Handle tombstone ──
            if msg.value() is None or len(msg.value()) == 0:
                self.tombstones.add(key)
                self.records_per_key[key] += 1
                if key in self.latest_value_per_key:
                    del self.latest_value_per_key[key]
                continue
            
            # ── Handle normal record ──
            self.records_per_key[key] += 1
            
            try:
                value = json.loads(msg.value().decode('utf-8'))
                self.latest_value_per_key[key] = value
                
                # ── Remove from tombstones if we see a new value ──
                if key in self.tombstones:
                    self.tombstones.discard(key)
            except (json.JSONDecodeError, UnicodeDecodeError):
                pass
        
        self.consumer.close()
    
    def analyze_compaction(self):
        """Analyze the state of compaction."""
        print(f"\n  ╔══════════════════════════════════════════════════════╗")
        print(f"  ║           COMPACTION ANALYSIS REPORT                  ║")
        print(f"  ╠══════════════════════════════════════════════════════╣")
        
        unique_keys = len(self.records_per_key)
        
        print(f"  ║                                                        ║")
        print(f"  ║  RECORD COUNTS:                                        ║")
        print(f"  ║    Total records read:     {self.total_records:>8}                  ║")
        print(f"  ║    Unique keys:            {unique_keys:>8}                  ║")
        print(f"  ║    Active profiles:        {len(self.latest_value_per_key):>8}"
              f"                  ║")
        print(f"  ║    Tombstones:             {len(self.tombstones):>8}                  ║")
        print(f"  ║    Null key records:       {self.null_key_records:>8}                  ║")
        
        # ── Compaction ratio ──
        if unique_keys > 0:
            avg_records_per_key = self.total_records / unique_keys
            compaction_ratio = unique_keys / self.total_records * 100
            
            print(f"  ║                                                        ║")
            print(f"  ║  COMPACTION EFFECTIVENESS:                              ║")
            print(f"  ║    Avg records per key:   {avg_records_per_key:>8.1f}"
                  f"                  ║")
            print(f"  ║    Compaction ratio:       {compaction_ratio:>7.1f}%"
                  f"                  ║")
            
            if avg_records_per_key <= 1.1:
                print(f"  ║    Status:                ✅ FULLY COMPACTED"
                      f"           ║")
            elif avg_records_per_key <= 2.0:
                print(f"  ║    Status:                ⏳ PARTIALLY COMPACTED"
                      f"       ║")
            else:
                print(f"  ║    Status:                ⚠️  NEEDS COMPACTION"
                      f"         ║")
                print(f"  ║    (Active segment records are not yet compacted)"
                      f"   ║")
        
        # ── Records per key distribution ──
        print(f"  ║                                                        ║")
        print(f"  ║  RECORDS PER KEY DISTRIBUTION:                         ║")
        
        count_distribution = defaultdict(int)
        for key, count in self.records_per_key.items():
            count_distribution[count] += 1
        
        for num_records, num_keys in sorted(count_distribution.items()):
            bar = "█" * min(num_keys, 30)
            print(f"  ║    {num_records:>3} records: {num_keys:>5} keys  {bar:<30}║")
        
        # ── Offset gap analysis ──
        print(f"  ║                                                        ║")
        print(f"  ║  OFFSET GAP ANALYSIS:                                  ║")
        
        for partition in sorted(self.partition_records.keys()):
            partition_offsets = sorted([
                off for p, off in self.offsets_seen if p == partition
            ])
            
            if len(partition_offsets) >= 2:
                expected_contiguous = partition_offsets[-1] - partition_offsets[0] + 1
                actual_count = len(partition_offsets)
                gaps = expected_contiguous - actual_count
                
                gap_status = "✅ No gaps" if gaps == 0 else f"🔍 {gaps} offsets removed"
                print(f"  ║    P{partition}: {actual_count:>5} records, "
                      f"offsets {partition_offsets[0]}-{partition_offsets[-1]}"
                      f"  {gap_status:<15}║")
            elif len(partition_offsets) == 1:
                print(f"  ║    P{partition}: {1:>5} record, "
                      f"offset {partition_offsets[0]}"
                      f"                            ║")
        
        # ── Tombstone details ──
        if self.tombstones:
            print(f"  ║                                                        ║")
            print(f"  ║  TOMBSTONES (deleted keys):                            ║")
            for key in sorted(list(self.tombstones)[:10]):
                print(f"  ║    🪦 {key:<50}║")
            if len(self.tombstones) > 10:
                print(f"  ║    ... and {len(self.tombstones)-10} more"
                      f"                                ║")
        
        print(f"  ║                                                        ║")
        print(f"  ╚══════════════════════════════════════════════════════╝")
        
        return {
            'total_records': self.total_records,
            'unique_keys': unique_keys,
            'active_profiles': len(self.latest_value_per_key),

from confluent_kafka import Consumer, KafkaError

class ProfileStateStore:
    """
    In-memory state store that materializes user profiles
    from the compacted Kafka topic.
    
    This simulates what a new microservice does on startup:
      1. Read compacted topic from beginning
      2. Build complete in-memory state
      3. Switch to "live" mode for ongoing updates
    
    WHY compaction matters here:
      → Without compaction: must read 500M records to build state
      → With compaction: read ~10M records (one per user)
      → Bootstrap time: minutes instead of hours
    """
    
    def __init__(self):
        self.profiles = {}          # user_id → profile dict
        self.deleted_users = set()  # user_ids that were tombstoned
        self.stats = defaultdict(int)
    
    def apply_record(self, key, value, offset, partition):
        """
        Apply a single record to the state store.
        
        Handles:
          → Normal record: upsert the profile
          → Tombstone (value=None): delete the profile
          → Duplicate key: overwrite with latest (compaction may not be complete)
        """
        if value is None:
            # ── Tombstone: delete this user ──
            if key in self.profiles:
                del self.profiles[key]
            self.deleted_users.add(key)
            self.stats['tombstones_applied'] += 1
            return 'deleted'
        
        try:
            profile = json.loads(value.decode('utf-8'))
            
            # ── Check if this is newer than what we have ──
            existing = self.profiles.get(key)
            if existing:
                existing_version = existing.get('profile_version', 0)
                new_version = profile.get('profile_version', 0)
                
                if new_version < existing_version:
                    self.stats['stale_skipped'] += 1
                    return 'skipped'
                
                self.stats['updates_applied'] += 1
            else:
                self.stats['new_profiles'] += 1
            
            self.profiles[key] = profile
            
            # ── Remove from deleted set if reappears ──
            self.deleted_users.discard(key)
            
            return 'applied'
            
        except (json.JSONDecodeError, UnicodeDecodeError):
            self.stats['parse_errors'] += 1
            return 'error'
    
    def get_profile(self, user_id):
        return self.profiles.get(user_id)
    
    def get_all_profiles(self):
        return dict(self.profiles)
    
    def get_stats(self):
        return {
            **dict(self.stats),
            'active_profiles': len(self.profiles),
            'deleted_users': len(self.deleted_users),
        }


class StateBootstrapper:
    """
    Bootstraps a state store from a compacted topic.
    
    Two-phase approach:
      PHASE 1: "Catch-up" — Read from beginning to end (bulk load)
      PHASE 2: "Live" — Continue reading new updates in real-time
    """
    
    def __init__(self, bootstrap_servers, topic):
        self.topic = topic
        self.bootstrap_servers = bootstrap_servers
        self.running = True
        self.state_store = ProfileStateStore()
        
        signal.signal(signal.SIGINT, lambda s, f: setattr(self, 'running', False))
    
    def bootstrap_phase(self):
        """
        PHASE 1: Read ALL existing records from compacted topic.
        
        This builds the complete current state.
        After compaction, this reads ~1 record per unique key.
        """
        print(f"  ┌────────────────────────────────────────────────────┐")
        print(f"  │  PHASE 1: BOOTSTRAP (reading historical state)      │")
        print(f"  └────────────────────────────────────────────────────┘\n")
        
        consumer = Consumer({
            'bootstrap.servers': self.bootstrap_servers,
            'group.id': f'state-bootstrap-{int(time.time())}',
            'auto.offset.reset': 'earliest',
            'enable.auto.commit': False,
        })
        
        consumer.subscribe([self.topic])
        
        start_time = time.time()
        records_read = 0
        empty_polls = 0
        MAX_EMPTY = 30
        
        while self.running and empty_polls < MAX_EMPTY:
            msg = consumer.poll(timeout=0.5)
            
            if msg is None:
                empty_polls += 1
                continue
            
            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    empty_polls += 1
                    continue
                continue
            
            empty_polls = 0
            records_read += 1
            
            if msg.key() is None:
                continue
            
            key = msg.key().decode('utf-8')
            self.state_store.apply_record(
                key=key,
                value=msg.value(),
                offset=msg.offset(),
                partition=msg.partition()
            )
            
            if records_read % 100 == 0:
                elapsed = time.time() - start_time
                rate = records_read / elapsed if elapsed > 0 else 0
                profiles = len(self.state_store.profiles)
                print(f"  📥 Read {records_read} records → "
                      f"{profiles} profiles ({rate:.0f} rec/s)")
        
        consumer.close()
        
        elapsed = time.time() - start_time
        stats = self.state_store.get_stats()
        
        print(f"\n  ✅ Bootstrap complete!")
        print(f"     Time:             {elapsed:.1f}s")
        print(f"     Records read:     {records_read}")
        print(f"     Active profiles:  {stats['active_profiles']}")
        print(f"     New profiles:     {stats['new_profiles']}")
        print(f"     Updates applied:  {stats['updates_applied']}")
        print(f"     Stale skipped:    {stats['stale_skipped']}")
        print(f"     Tombstones:       {stats['tombstones_applied']}")
        print(f"     Deleted users:    {stats['deleted_users']}")
        
        if records_read > 0:
            efficiency = stats['active_profiles'] / records_read * 100
            print(f"     Efficiency:       {efficiency:.1f}% "
                  f"(records → unique profiles)")
            
            if efficiency > 80:
                print(f"     📊 Topic is well-compacted")
            elif efficiency > 50:
                print(f"     📊 Topic is partially compacted")
            else:
                print(f"     📊 Topic needs compaction (many duplicates)")
        
        return elapsed, records_read
    
    def live_phase(self, duration=20):
        """
        PHASE 2: Continue reading new updates in real-time.
        
        After bootstrap, this keeps the state store current.
        """
        print(f"\n  ┌────────────────────────────────────────────────────┐")
        print(f"  │  PHASE 2: LIVE MODE (listening for updates)         │")
        print(f"  └────────────────────────────────────────────────────┘\n")
        
        consumer = Consumer({
            'bootstrap.servers': self.bootstrap_servers,
            'group.id': f'state-live-{int(time.time())}',
            'auto.offset.reset': 'latest',   # Only new messages
            'enable.auto.commit': False,
        })
        
        consumer.subscribe([self.topic])
        
        start_time = time.time()
        live_updates = 0
        
        while self.running and (time.time() - start_time) < duration:
            msg = consumer.poll(timeout=1.0)
            
            if msg is None:
                continue
            
            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    continue
                continue
            
            if msg.key() is None:
                continue
            
            key = msg.key().decode('utf-8')
            result = self.state_store.apply_record(
                key=key,
                value=msg.value(),
                offset=msg.offset(),
                partition=msg.partition()
            )
            
            live_updates += 1
            action = "🪦 DELETED" if result == 'deleted' else "📝 UPDATED"
            print(f"  {action}: {key} (P{msg.partition()}:{msg.offset()})")
        
        consumer.close()
        
        print(f"\n  📊 Live phase: {live_updates} updates received")
        print(f"     Active profiles: {len(self.state_store.profiles)}")
    
    def query_state(self):
        """

        """

        print(f"  │  STATE QUERIES (materialized from compacted topic)   │")
        print(f"  └────────────────────────────────────────────────────┘\n")
        
        profiles = self.state_store.get_all_profiles()
        
        if not profiles:
            print(f"  ⚠️  No profiles in state store")
            return
        
        # ── Query 1: Lookup specific user ──
        sample_keys = list(profiles.keys())[:3]
        print(f"  🔍 Sample profile lookups:")
        for key in sample_keys:
            profile = self.state_store.get_profile(key)
            if profile:
                print(f"     {key}:")
                print(f"       Name:    {profile.get('name', 'N/A')}")
                print(f"       City:    {profile.get('address', {}).get('city', 'N/A')}")
                print(f"       Tier:    {profile.get('loyalty_tier', 'N/A')}")
                print(f"       Version: {profile.get('profile_version', 'N/A')}")

        # ── Query 2: Aggregate by loyalty tier ──

        print(f"  📊 Users by loyalty tier:")
        for tier, count in sorted(tier_counts.items()):
            bar = "█" * (count // 2)
            print(f"     {tier:<12} {count:>5} users  {bar}")
        
        # ── Query 3: Aggregate by country ──

        for country, count in sorted(country_counts.items(),
                                      key=lambda x: -x[1])[:10]:
            bar = "█" * (count // 2)
            print(f"     {country:<5} {count:>5} users  {bar}")

class CompactionMonitor:
    """
    Periodically reads the compacted topic and measures:
      → Total records vs unique keys (compaction ratio)
      → Change over time (is compaction running?)
      → Estimated time until full compaction
    """
    
    def __init__(self, bootstrap_servers, topic):
        self.bootstrap_servers = bootstrap_servers
        self.topic = topic
        self.running = True
        self.history = []
        
        signal.signal(signal.SIGINT, lambda s, f: setattr(self, 'running', False))
    
    def measure_once(self):
        """Take a single measurement of topic state."""
        consumer = Consumer({
            'bootstrap.servers': self.bootstrap_servers,
            'group.id': f'compaction-monitor-{int(time.time())}-{id(self)}',
            'auto.offset.reset': 'earliest',
            'enable.auto.commit': False,
        })
        
        consumer.subscribe([self.topic])
        
        total_records = 0
        unique_keys = set()
        tombstones = 0
        total_bytes = 0
        empty_polls = 0
        
        while empty_polls < 15:
            msg = consumer.poll(timeout=0.3)
            
            if msg is None:
                empty_polls += 1
                continue
            
            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    empty_polls += 1
                    continue
                continue
            
            empty_polls = 0
            total_records += 1
            
            if msg.key():
                unique_keys.add(msg.key().decode('utf-8'))
            
            if msg.value() is None or len(msg.value()) == 0:
                tombstones += 1
            else:
                total_bytes += len(msg.value())
            
            if msg.key():
                total_bytes += len(msg.key())
        
        consumer.close()
        
        measurement = {
            'timestamp': time.time(),
            'total_records': total_records,
            'unique_keys': len(unique_keys),
            'tombstones': tombstones,
            'total_bytes': total_bytes,
            'ratio': total_records / len(unique_keys) if unique_keys else 0,
        }
        
        self.history.append(measurement)
        return measurement
    
    def monitor_loop(self, interval=30, max_iterations=10):
        """
        Monitor compaction progress over time.
        
        Run this AFTER producing data to watch compaction happen.
        """
        print("╔══════════════════════════════════════════════════╗")
        print("║    COMPACTION PROGRESS MONITOR                    ║")
        print("║    Watching topic compaction in real-time          ║")
        print("║    Press Ctrl+C to stop                           ║")
        print("╚══════════════════════════════════════════════════╝\n")
        
        print(f"  Topic: {self.topic}")
        print(f"  Interval: {interval}s between measurements")
        print(f"  Max iterations: {max_iterations}\n")
        
        print(f"  {'#':<4} {'Time':<10} {'Records':<10} {'Unique':<10} "
              f"{'Ratio':<8} {'Size MB':<10} {'Status':<15}")
        print(f"  {'─'*70}")
        
        iteration = 0
        
        while self.running and iteration < max_iterations:
            m = self.measure_once()
            
            size_mb = m['total_bytes'] / (1024 * 1024)
            ratio = m['ratio']
            
            if ratio <= 1.1:
                status = "✅ COMPACTED"
            elif ratio <= 2.0:
                status = "⏳ PARTIAL"
            else:
                status = "⚠️  DIRTY"
            
            time_str = time.strftime('%H:%M:%S')
            
            print(f"  {iteration+1:<4} {time_str:<10} {m['total_records']:<10} "
                  f"{m['unique_keys']:<10} {ratio:<8.2f} {size_mb:<10.2f} {status}")
            
            # ── Check for improvement ──
            if len(self.history) >= 2:
                prev = self.history[-2]
                curr = self.history[-1]
                
                if curr['total_records'] < prev['total_records']:
                    removed = prev['total_records'] - curr['total_records']
                    print(f"       ↑ Compaction removed {removed} records!")
                elif curr['ratio'] < prev['ratio']:
                    improvement = prev['ratio'] - curr['ratio']
                    print(f"       ↑ Ratio improved by {improvement:.2f}")
            
            iteration += 1
            
            if iteration < max_iterations:
                # ── Wait with interrupt checking ──
                for _ in range(interval * 10):
                    if not self.running:
                        break
                    time.sleep(0.1)
        
        self.display_summary()
    
    def display_summary(self):
        """Show compaction progress summary."""
        if len(self.history) < 2:
            print(f"\n  ⚠️  Not enough measurements for trend analysis")
            return
        
        first = self.history[0]
        last = self.history[-1]
        
        print(f"\n  ┌────────────────────────────────────────────────────┐")
        print(f"  │  COMPACTION PROGRESS SUMMARY                        │")
        print(f"  ├────────────────────────────────────────────────────┤")
        print(f"  │  Measurements taken:  {len(self.history):>6}                       │")
        print(f"  │                                                      │")
        print(f"  │  FIRST measurement:                                  │")

# ============================================================
# FILE: 06_tombstone_demo.py
# PURPOSE: Demonstrate the complete lifecycle of tombstones
# TEACHES: How key deletion works in compacted topics
# ============================================================

from confluent_kafka import Producer, Consumer, KafkaError
import json
import time
import uuid

BOOTSTRAP = 'localhost:9092,localhost:9093,localhost:9094'

DEMO_TOPIC = 'user.profiles'


def demonstrate_tombstone_lifecycle():
    """
    Step-by-step demonstration of tombstone mechanics.
    
    Flow:
      1. Produce a profile for a user
      2. Verify the profile exists
      3. Send a tombstone (value=None)
      4. Verify the tombstone exists alongside the profile
      5. Explain what happens after compaction + delete.retention.ms
    """
    
    print("╔══════════════════════════════════════════════════╗")
    print("║    TOMBSTONE LIFECYCLE DEMONSTRATION              ║")
    print("╚══════════════════════════════════════════════════╝\n")
    
    test_user = f'tombstone-demo-{uuid.uuid4().hex[:6]}'
    
    producer = Producer({
        'bootstrap.servers': BOOTSTRAP,
        'acks': 'all',
        'enable.idempotence': True,
    })
    
    delivery_results = []
    
    def cb(err, msg):
        delivery_results.append({
            'error': str(err) if err else None,
            'topic': msg.topic(),
            'partition': msg.partition(),
            'offset': msg.offset(),
            'key': msg.key().decode('utf-8') if msg.key() else None,
            'is_tombstone': msg.value() is None or len(msg.value()) == 0,
        })
    
    # ══════════════════════════════════════════════
    # STEP 1: Create the user profile
    # ══════════════════════════════════════════════
    print(f"  STEP 1: Creating profile for {test_user}\n")
    
    profile = {
        'user_id': test_user,
        'name': 'Tombstone Test User',
        'email': f'{test_user}@test.com',
        'loyalty_tier': 'gold',
        'address': {'city': 'Test City', 'country': 'TC'},
    }
    
    producer.produce(
        topic=DEMO_TOPIC,
        key=test_user.encode('utf-8'),
        value=json.dumps(profile).encode('utf-8'),
        callback=cb,
    )
    producer.flush(timeout=10)
    
    print(f"    ✅ Profile produced:")
    print(f"       Key:   {test_user}")
    print(f"       Value: {json.dumps(profile)[:80]}...")
    if delivery_results:
        r = delivery_results[-1]
        print(f"       Partition: {r['partition']}, Offset: {r['offset']}")
    
    # ══════════════════════════════════════════════
    # STEP 2: Update the profile (creates second version)
    # ══════════════════════════════════════════════
    print(f"\n  STEP 2: Updating profile (new address)\n")
    
    profile['address'] = {'city': 'Updated City', 'country': 'UC'}
    profile['name'] = 'Updated Tombstone User'
    
    producer.produce(
        topic=DEMO_TOPIC,
        key=test_user.encode('utf-8'),
        value=json.dumps(profile).encode('utf-8'),
        callback=cb,
    )
    producer.flush(timeout=10)
    
    print(f"    ✅ Updated profile produced:")
    if delivery_results:
        r = delivery_results[-1]
        print(f"       Partition: {r['partition']}, Offset: {r['offset']}")
    
    # ══════════════════════════════════════════════
    # STEP 3: Send TOMBSTONE (delete the user)
    # ══════════════════════════════════════════════
    print(f"\n  STEP 3: Sending TOMBSTONE (deleting user)\n")
    
    producer.produce(
        topic=DEMO_TOPIC,
        key=test_user.encode('utf-8'),
        value=None,                          # ← TOMBSTONE
        callback=cb,
    )
    producer.flush(timeout=10)
    
    print(f"    🪦 Tombstone produced:")
    print(f"       Key:   {test_user}")
    print(f"       Value: None (tombstone)")
    if delivery_results:
        r = delivery_results[-1]
        print(f"       Partition: {r['partition']}, Offset: {r['offset']}")
    
    # ══════════════════════════════════════════════
    # STEP 4: Read back all records for this key
    # ══════════════════════════════════════════════
    print(f"\n  STEP 4: Reading all records for key={test_user}\n")
    
    time.sleep(2)
    
    consumer = Consumer({
        'bootstrap.servers': BOOTSTRAP,
        'group.id': f'tombstone-reader-{int(time.time())}',
        'auto.offset.reset': 'earliest',
        'enable.auto.commit': False,
    })
    
    consumer.subscribe([DEMO_TOPIC])
    
    found_records = []
    empty_polls = 0
    
    while empty_polls < 20:
        msg = consumer.poll(timeout=0.5)
        
        if msg is None:
            empty_polls += 1
            continue
        
        if msg.error():
            if msg.error().code() == KafkaError._PARTITION_EOF:
                empty_polls += 1
                continue
            continue
        
        empty_polls = 0
        
        if msg.key() and msg.key().decode('utf-8') == test_user:
            is_tombstone = msg.value() is None or len(msg.value()) == 0
            
            record = {
                'offset': msg.offset(),
                'partition': msg.partition(),
                'is_tombstone': is_tombstone,
                'value': None if is_tombstone else json.loads(msg.value().decode('utf-8')),
            }
            found_records.append(record)
    
    consumer.close()
    
    print(f"    Records found for {test_user}: {len(found_records)}\n")
    
    for i, rec in enumerate(found_records):
        if rec['is_tombstone']:
            print(f"    [{i+1}] 🪦 TOMBSTONE at P{rec['partition']}:{rec['offset']}")
            print(f"        Value: None")
        else:
            name = rec['value'].get('name', 'N/A')
            city = rec['value'].get('address', {}).get('city', 'N/A')
            print(f"    [{i+1}] 📝 RECORD at P{rec['partition']}:{rec['offset']}")
            print(f"        Name: {name}, City: {city}")
    
    # ══════════════════════════════════════════════
    # STEP 5: Explain what happens next
    # ══════════════════════════════════════════════
    print(f"\n  ┌────────────────────────────────────────────────────────┐")
    print(f"  │  TOMBSTONE LIFECYCLE TIMELINE                            │")
    print(f"  ├────────────────────────────────────────────────────────┤")
    print(f"  │                                                          │")
    print(f"  │  NOW:                                                    │")
    print(f"  │    → {len(found_records)} records exist for this key     │")
    print(f"  │    → Profile versions + tombstone all coexist           │")
    print(f"  │                                                          │")
    print(f"  │  AFTER SEGMENT ROLLS (segment.ms = 5 min):              │")
    print(f"  │    → Compaction becomes eligible for closed segments    │")
    print(f"  │    → Old profile versions will be REMOVED              │")
    print(f"  │    → Tombstone will be KEPT                            │")
    print(f"  │    → Result: only tombstone remains for this key       │")
    print(f"  │                                                          │")
    print(f"  │  AFTER delete.retention.ms (24 hours default):          │")
    print(f"  │    → Tombstone itself becomes eligible for removal     │")
    print(f"  │    → Next compaction removes the tombstone             │")
    print(f"  │    → Key is COMPLETELY gone from the topic             │")
    print(f"  │    → Any consumer reading from beginning won't see it  │")
    print(f"  │                                                          │")
    print(f"  │  ⚠️  WARNING: If a consumer is offline for longer than   │")
    print(f"  │     delete.retention.ms, they'll miss the tombstone    │")
    print(f"  │     and still have stale data in their local state     │")
    print(f"  └────────────────────────────────────────────────────────┘")
    
    print(f"\n  📊 Delivery Summary:")
    for r in delivery_results:
        tomb_label = "🪦 TOMBSTONE" if r['is_tombstone'] else "📝 RECORD"
        err_label = f"❌ {r['error']}" if r['error'] else "✅"
        print(f"     {err_label} P{r['partition']}:{r['offset']} "
              f"key={r['key']} {tomb_label}")


if __name__ == '__main__':
    demonstrate_tombstone_lifecycle()
