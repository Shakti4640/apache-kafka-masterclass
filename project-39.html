🔴 HYPOTHETICAL PROBLEM

StreamMetrics is an IoT analytics platform ingesting sensor data
from 500,000 devices. Each device sends a reading every second.

CURRENT THROUGHPUT REQUIREMENT:
  → 500,000 messages/second ingest
  → Average message size: 500 bytes
  → Total throughput: ~250 MB/second
  → End-to-end latency target: < 500ms (p99)

CURRENT PERFORMANCE (before tuning):
  → Producer throughput: 12,000 messages/second ❌ (need 500K)
  → Consumer throughput: 8,000 messages/second ❌
  → End-to-end latency: 2,400ms p99 ❌ (need < 500ms)
  → Broker disk usage: 4.2 TB/day ❌ (budget: 1 TB/day)
  → Producer CPU: 95% ❌ (serialization bottleneck)
  → Network bandwidth: 85% saturated ❌

SPECIFIC SYMPTOMS:

  Symptom 1: PRODUCER BOTTLENECK
    → Each produce() call blocks waiting for broker acknowledgment
    → 12,000 synchronous calls/second = ceiling
    → CPU spent 70% on serialization, 30% on network I/O
    → Buffer fills up → BufferExhaustedError → messages dropped

  Symptom 2: CONSUMER LAG GROWING
    → Consumer processes one message at a time
    → Each poll() returns only 500 records
    → Network round-trip per poll: 15ms
    → 500 records / 15ms = 33,333 records/second maximum
    → Lag grows by 466,667 messages/second

  Symptom 3: DISK EXPLOSION
    → 500 bytes × 500,000 msg/sec × 86,400 sec/day = 21.6 TB/day raw
    → With replication factor 3 = 64.8 TB/day
    → No compression = maximum disk and network usage

  Symptom 4: NETWORK SATURATION
    → Each message sent individually = maximum overhead
    → TCP header (40 bytes) + Kafka protocol header (70 bytes)
    → For 500-byte payload: 110/500 = 22% overhead
    → Network carrying 22% protocol overhead = wasted bandwidth

CTO: "We benchmarked Kafka at 2 million messages/second.
      Why are we stuck at 12,000?"

ANSWER: Default Kafka configurations are SAFE, not FAST.
        Performance tuning requires understanding the tradeoffs
        between throughput, latency, durability, and resource usage.

→ Kafka's default configs prioritize SAFETY and COMPATIBILITY

→ Every tuning knob involves a TRADEOFF
→ There is no "best config" — only "best config for YOUR workload"

THREE TUNING LAYERS:

  Layer 1: PRODUCER TUNING
    → How messages are batched, compressed, and sent
    → Controls: throughput ceiling, latency floor, network efficiency

  Layer 2: CONSUMER TUNING
    → How messages are fetched, processed, and committed
    → Controls: consumption rate, processing latency, rebalance impact

  Layer 3: BROKER TUNING
    → How messages are stored, replicated, and served
    → Controls: disk I/O, replication speed, topic throughput

  WE WILL TUNE ALL THREE LAYERS with measurable benchmarks.

TUNING METHODOLOGY:
  Step 1: MEASURE baseline (before any changes)
  Step 2: IDENTIFY bottleneck (producer? consumer? broker? network?)
  Step 3: TUNE one parameter at a time
  Step 4: MEASURE again (did it improve?)
  Step 5: REPEAT until target met
  Step 6: STRESS TEST under sustained load
  
  NEVER tune blindly. ALWAYS measure before and after.

✅ WHAT SHOULD DO

1. ALWAYS BENCHMARK BEFORE AND AFTER
   → Record: messages/sec, bytes/sec, latency p50/p95/p99, CPU, memory
   → Change ONE parameter at a time
   → Run benchmark for minimum 60 seconds (avoid warmup effects)
   → Use consistent message size and count for fair comparison

2. INCREASE BATCH SIZE FOR THROUGHPUT
   → Default 16KB is small for high-throughput scenarios
   → 64KB–256KB significantly improves throughput
   → Larger batches = better compression ratio = less network
   → But: larger batches = more memory per partition

3. SET LINGER.MS > 0 FOR BATCHING
   → Default 0 = send immediately (minimum batching)
   → linger.ms=5 → wait up to 5ms to fill batch
   → At 500K msg/sec: 5ms = 2,500 messages per batch
   → Dramatic throughput improvement for small trade in latency

4. ENABLE COMPRESSION
   → lz4 for balanced CPU/ratio (default choice)
   → zstd for best compression ratio (disk-constrained)
   → Compression happens on full batch = better ratio than per-message
   → Broker stores compressed → disk savings
   → Consumer decompresses → small CPU cost

5. SIZE BUFFERS APPROPRIATELY
   → buffer.memory: total bytes producer can buffer
   → If producing faster than broker can accept → buffer fills → block/error
   → 64MB–128MB for high-throughput producers
   → Monitor: buffer-available-bytes metric

6. TUNE CONSUMER FETCH SIZE
   → fetch.min.bytes > 1 → broker waits for more data before responding
   → Fewer round-trips = higher throughput
   → max.poll.records > 500 → more records per poll
   → Balance with max.poll.interval.ms to avoid rebalance

❌ WHAT SHOULD NOT DO

1. NEVER TUNE WITHOUT MEASURING
   → "I set batch.size to 1MB, should be faster" → maybe, maybe not
   → Larger batch.size with low message rate → messages sit in buffer
   → Throughput may DECREASE if batch never fills before linger.ms
   → ALWAYS benchmark before and after

2. NEVER SET ACKS=0 FOR CRITICAL DATA
   → acks=0 gives maximum throughput (no waiting for broker)
   → But: messages can be LOST (broker crash before write)
   → For IoT sensor data (StreamMetrics): acks=1 is acceptable
   → For financial data: acks=all is mandatory (Project 11)
   → Understand the tradeoff before changing

3. NEVER IGNORE CONSUMER MAX.POLL.INTERVAL
   → max.poll.records=10000 with slow processing → poll takes > 5 minutes
   → Consumer exceeds max.poll.interval.ms → rebalance triggered
   → Rebalance → all consumers pause → throughput drops to zero
   → Rule: max.poll.records × processing_time < max.poll.interval.ms

4. NEVER COMPRESS ALREADY-COMPRESSED DATA
   → Images, videos, encrypted payloads → already compressed
   → Compression adds CPU cost with near-zero size reduction
   → Use compression.type=none for binary/encrypted payloads

5. NEVER USE GZIP IN HIGH-THROUGHPUT SCENARIOS
   → gzip has best compression ratio
   → But: 5-10x slower than lz4
   → At 500K msg/sec: gzip becomes CPU bottleneck
   → Use lz4 or zstd for high throughput

6. NEVER INCREASE BATCH SIZE WITHOUT INCREASING BUFFER.MEMORY
   → batch.size × number_of_partitions ≤ buffer.memory
   → 256KB batch × 200 partitions = 50MB minimum buffer
   → If buffer.memory < total batch needs → producer blocks

🚫 WHEN NOT TO USE

1. WHEN WORKLOAD IS ALREADY WITHIN TARGETS
   → "It works fine" → don't tune
   → Premature optimization wastes time and adds complexity
   → Only tune when metrics show a gap between current and target

2. WHEN BOTTLENECK IS NOT KAFKA
   → Consumer processing logic takes 100ms per message
   → Tuning Kafka won't help — optimize processing code first
   → Profile: is time spent in poll()? in processing? in commit()?
   → Fix the actual bottleneck

3. WHEN USING EXTREME LINGER.MS FOR REAL-TIME
   → linger.ms=100 adds 100ms minimum latency to every message
   → For real-time dashboards needing <50ms → linger.ms must be < 50
   → Don't apply high-throughput tuning to low-latency workloads

4. WHEN COMPRESSION RATIO IS < 1.1x
   → Pre-compressed data, random binary data, encrypted data
   → Compression wastes CPU for no benefit
   → Test: produce 1000 messages, compare compressed vs uncompressed size
   → If < 10% reduction → disable compression

5. WHEN BROKER IS THE BOTTLENECK
   → Producer/consumer tuning won't help if broker disk I/O is saturated
   → Check: broker request latency, disk utilization, under-replicated partitions
   → Fix: add brokers, faster disks, increase partitions
   → This is broker-level tuning (not covered in this project — see Project 40)

⚙️ HOW IT ONLY WORKS (INTERNAL MECHANICS)

PRODUCER BATCHING INTERNALS:

  When you call producer.produce():

  Step 1: MESSAGE ENTERS ACCUMULATOR
    → Message serialized (Avro/JSON — Project 16/7)
    → Message added to RecordAccumulator
    → RecordAccumulator maintains ONE batch per partition
    → Message routed to partition batch based on key hash (Project 4)

    ┌──────────────────────────────────────────────┐
    │           RECORD ACCUMULATOR                 │
    │                                              │
    │  Partition 0: [msg1, msg2, msg3, ...]  batch │
    │  Partition 1: [msg4, msg5, ...]         batch│
    │  Partition 2: [msg6, ...]               batch│
    │  ...                                         │
    │  Partition N: [...]                     batch│
    │                                              │
    │  Total memory ≤ buffer.memory                │
    └──────────────────────────────────────────────┘

  Step 2: BATCH DECISION — SEND OR WAIT?
    A batch is sent when ANY of these conditions is true:

    Condition A: Batch size reached
      → batch bytes ≥ batch.size (default 16KB)
      → Send immediately — batch is "full"

    Condition B: Linger time expired
      → Time since first message in batch ≥ linger.ms
      → Even if batch isn't full — time to send
      → linger.ms=0 → send as soon as possible (next sender cycle)

    Condition C: Buffer memory pressure
      → buffer.memory nearly full
      → Must send existing batches to free memory

    Condition D: Explicit flush()
      → Application calls producer.flush()
      → All accumulated batches sent immediately

  Step 3: COMPRESSION (before network send)
    → Entire batch compressed as one unit
    → NOT per-message — batch-level compression
    → This is why larger batches compress better
    → 100 messages × 500 bytes = 50KB batch
    → Compression finds patterns across messages
    → Similar messages (same schema) compress very well

    Compression flow:
    ┌────────────┐     ┌──────────────┐     ┌────────────┐
    │ Raw batch  │────▶│ Compress     │────▶│ Compressed │
    │ 50 KB      │     │ (lz4/zstd)  │     │ batch 15KB │
    │ 100 msgs   │     │              │     │ 100 msgs   │
    └────────────┘     └──────────────┘     └────────────┘

  Step 4: NETWORK SEND
    → Compressed batch sent as single ProduceRequest
    → One TCP round-trip for entire batch
    → Without batching: 100 TCP round-trips for 100 messages
    → With batching: 1 TCP round-trip for 100 messages
    → 100x reduction in network overhead

  Step 5: BROKER RECEIVES
    → Broker stores compressed batch as-is on disk
    → NO decompression on broker → minimal broker CPU
    → Disk write: 15KB instead of 50KB → 70% disk savings

  Step 6: ACKNOWLEDGMENT
    → Broker sends ack back to producer
    → acks=0: no ack (fire and forget)
    → acks=1: leader ack (leader wrote to disk)
    → acks=all: all ISR ack (all replicas confirmed)

    max.in.flight.requests.per.connection:
    → Default 5: up to 5 batches unacknowledged simultaneously
    → Higher → more throughput (pipeline more batches)
    → But: if retry needed → possible reordering
    → With enable.idempotence=true (Project 17): safe up to 5


CONSUMER FETCH INTERNALS:

  When you call consumer.poll():

  Step 1: CHECK LOCAL BUFFER
    → Consumer maintains local buffer of pre-fetched records
    → If buffer has records → return immediately (no network)
    → If buffer empty → send FetchRequest to broker

  Step 2: FETCH REQUEST TO BROKER
    ┌──────────────────────────────────────────────┐
    │ FetchRequest:                                │
    │   fetch.min.bytes: 1 (default)               │
    │   fetch.max.wait.ms: 500 (default)           │
    │   fetch.max.bytes: 52,428,800                │
    │   partitions: [                              │
    │     {topic: "sensors", partition: 0,          │
    │      offset: 15234, max_bytes: 1048576},     │
    │     {topic: "sensors", partition: 1,          │
    │      offset: 8921, max_bytes: 1048576},      │
    │     ...                                      │
    │   ]                                          │
    └──────────────────────────────────────────────┘

  Step 3: BROKER PROCESSES FETCH
    → Broker checks if fetch.min.bytes of data available
    → If yes → respond immediately
    → If no → wait up to fetch.max.wait.ms for more data
    → This is the latency-throughput tradeoff for consumers

    fetch.min.bytes=1 (default):
      → Respond immediately with whatever is available
      → LOW LATENCY but many small responses → high overhead

    fetch.min.bytes=50000 (50KB):
      → Wait until 50KB available (or timeout)
      → HIGHER LATENCY but fewer, larger responses → less overhead
      → Better throughput for high-volume topics

  Step 4: DECOMPRESSION
    → Broker sends compressed batches
    → Consumer decompresses on receive
    → CPU cost on consumer side
    → But: network transfer was 3-5x smaller

  Step 5: RECORD DELIVERY
    → Decompressed records added to local buffer
    → poll() returns up to max.poll.records from buffer
    → Remaining records served from buffer on next poll()
    → No network call needed until buffer exhausted

    max.poll.records=500 (default):
      → At most 500 records per poll() call
      → Small batches → more poll() calls → more overhead

    max.poll.records=5000:
      → 10x more records per poll() call
      → Fewer poll() calls → less overhead
      → BUT: processing 5000 records takes longer
      → Must complete before max.poll.interval.ms


BATCH SIZE vs LINGER.MS INTERACTION:

  Scenario: 10,000 messages/second, 500 bytes each, 10 partitions

  Config A: batch.size=16KB, linger.ms=0
    → 1000 msg/sec per partition
    → 500 bytes × 1000 = 500KB/sec per partition
    → Batch fills 16KB in 32ms (32 messages)
    → ~31 batches/sec per partition
    → 310 ProduceRequests/sec total
    → High network overhead

  Config B: batch.size=64KB, linger.ms=5
    → Wait 5ms → collect 5 messages (2.5KB) → batch not full but linger expired
    → Send small batches every 5ms
    → 200 batches/sec per partition
    → WORSE than Config A! (linger too short for batch to fill)

  Config C: batch.size=64KB, linger.ms=50
    → Wait 50ms → collect 50 messages (25KB) → linger expired
    → Send 25KB batches every 50ms
    → 20 batches/sec per partition
    → 200 ProduceRequests/sec total
    → Much better compression, much less overhead
    → But: 50ms minimum latency added

  Config D: batch.size=256KB, linger.ms=100
    → Wait 100ms → collect 100 messages (50KB) → linger expired
    → Or: batch reaches 256KB after 512 messages (256ms)
    → Whichever comes first
    → 10 batches/sec per partition at most
    → 100 ProduceRequests/sec total
    → MAXIMUM throughput, but 100ms latency floor

  THE KEY INSIGHT:
    linger.ms sets the LATENCY FLOOR
    batch.size sets the THROUGHPUT CEILING
    The actual behavior depends on message arrival rate

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: BUFFER EXHAUSTION
  Symptom: BufferExhaustedError or producer.produce() blocks
  Cause: Producing faster than broker can accept
  Config: buffer.memory too small for production rate
  Fix:
    → Increase buffer.memory (64MB → 128MB → 256MB)
    → Increase batch.size (more efficient sends)
    → Enable compression (smaller batches)
    → Add partitions (more parallel sends)
  Monitor: buffer-available-bytes JMX metric

PITFALL 2: CONSUMER REBALANCE STORM
  Symptom: Constant rebalances, throughput drops to near-zero
  Cause: max.poll.records too high, processing too slow
  Timeline:
    → poll() returns 10,000 records
    → Processing takes 6 minutes
    → max.poll.interval.ms = 300,000 (5 min)
    → Consumer exceeds interval → coordinator thinks it's dead
    → Rebalance triggered → all consumers pause
  Fix:
    → Reduce max.poll.records to match processing speed
    → Increase max.poll.interval.ms (but don't exceed 10 min)
    → Optimize processing code
    → Rule: max.poll.records × per_record_time < 80% of max.poll.interval.ms

PITFALL 3: COMPRESSION MISMATCH
  Symptom: Consumer CPU spikes, unexpected decompression errors
  Cause: Producer uses zstd, consumer's library doesn't support it
  Fix:
    → Ensure all clients support the chosen compression codec
    → lz4 has widest support across client libraries
    → Broker can transcode between codecs (but adds CPU)

PITFALL 4: FETCH.MIN.BYTES TOO HIGH ON LOW-VOLUME TOPIC
  Symptom: Consumer receives messages in bursts with long delays
  Cause: fetch.min.bytes=1MB on topic with 10 msg/sec (5KB/sec)
  Timeline:
    → Broker waits for 1MB of data
    → At 5KB/sec: waits 200 seconds
    → fetch.max.wait.ms=500ms triggers first → small response
    → But repeated: mostly empty fetches with occasional bursts
  Fix:
    → Set fetch.min.bytes proportional to expected volume
    → Low volume: fetch.min.bytes=1 (default)
    → High volume: fetch.min.bytes=50KB–1MB

PITFALL 5: OVER-TUNING FOR THROUGHPUT WHEN LATENCY MATTERS
  Symptom: Throughput is great, but latency is 500ms+
  Cause: linger.ms=100, batch.size=1MB, fetch.min.bytes=1MB
  Fix:
    → For latency-sensitive: linger.ms ≤ 5, batch.size ≤ 32KB
    → For throughput-sensitive: linger.ms ≤ 100, batch.size ≤ 256KB
    → You CANNOT have both minimum latency AND maximum throughput
    → Pick your priority, then optimize the other as secondary

PITFALL 6: IGNORING SERIALIZATION OVERHEAD
  Symptom: CPU is bottleneck, not network or disk
  Cause: JSON serialization at 500K msg/sec = heavy CPU
  Fix:
    → Switch from JSON to Avro or Protobuf
    → Avro: 2-5x smaller, 3-10x faster serialization
    → Protobuf: similar to Avro
    → This connects to Project 16 (Avro) and Project 38 (Schema Evolution)
    → Serialization format is often the #1 CPU bottleneck

📈 SCALABILITY CONSIDERATIONS

PRODUCER SCALING:
  → Single producer instance: 100K-500K msg/sec (tuned)
  → Multiple producer instances: linear scaling
  → Key: ensure each instance targets different partitions
  → 5 producers × 100K = 500K msg/sec

CONSUMER SCALING:
  → Single consumer: 50K-200K msg/sec (tuned)
  → Consumer group with N consumers: up to N × single throughput
  → Maximum parallelism = number of partitions (Project 5)
  → 10 consumers × 50K = 500K msg/sec (if 10+ partitions)

PARTITION SCALING:
  → More partitions = more parallelism
  → But: more partitions = more broker overhead
  → Sweet spot: 6-50 partitions per topic for most workloads
  → High-throughput: 100+ partitions

COMPRESSION SCALING:
  → Compression is CPU-bound on producer
  → Multi-threaded compression: lz4 and zstd support it
  → More CPU cores → higher compression throughput
  → Broker: stores compressed → decompression only on consumer

END-TO-END THROUGHPUT FORMULA:
  Throughput = min(
    producer_throughput,
    broker_write_throughput,
    broker_replication_throughput,
    consumer_throughput
  )

  → The SLOWEST component determines overall throughput
  → Identify and fix the bottleneck FIRST

SIZING CALCULATOR:
  Given: 500K msg/sec, 500 bytes/msg, RF=3, retention=7 days

  Network (ingress):
    → 500K × 500B = 250 MB/sec raw
    → With lz4 compression: ~100 MB/sec
    → With RF=3 replication: 300 MB/sec inter-broker

  Disk (daily):
    → 250 MB/sec × 86,400 sec = 21.6 TB/day raw
    → With lz4: ~8.6 TB/day
    → With RF=3: 25.8 TB/day
    → 7-day retention: 180.6 TB total
    → With zstd: ~120 TB total (33% savings over lz4)

  Brokers:
    → Each broker: ~100 MB/sec write throughput (SSD)
    → Need: 300 MB/sec (with replication) / 100 = 3 brokers minimum

@dataclass
class ProducerBenchmarkResult:
    config_name: str
    total_messages: int
    total_bytes: int
    duration_seconds: float
    latencies_ms: List[float] = field(default_factory=list)
    errors: int = 0
    config: Dict = field(default_factory=dict)

    @property
    def throughput_msg_sec(self) -> float:
        return self.total_messages / self.duration_seconds if self.duration_seconds > 0 else 0

    @property
    def throughput_mb_sec(self) -> float:
        return (self.total_bytes / (1024 * 1024)) / self.duration_seconds if self.duration_seconds > 0 else 0

    @property
    def latency_p50(self) -> float:
        return self._percentile(50) if self.latencies_ms else 0

    @property
    def latency_p95(self) -> float:
        return self._percentile(95) if self.latencies_ms else 0

    @property
    def latency_p99(self) -> float:
        return self._percentile(99) if self.latencies_ms else 0

    def _percentile(self, p: float) -> float:
        if not self.latencies_ms:
            return 0
        sorted_lat = sorted(self.latencies_ms)
        idx = int(len(sorted_lat) * p / 100)
        idx = min(idx, len(sorted_lat) - 1)
        return sorted_lat[idx]

    def print_report(self):
        print(f"\n  {'='*60}")
        print(f"  📊 PRODUCER BENCHMARK: {self.config_name}")
        print(f"  {'='*60}")
        print(f"  Messages:        {self.total_messages:,}")
        print(f"  Total Bytes:     {self.total_bytes:,} ({self.total_bytes/(1024*1024):.1f} MB)")

        pass

    topic = NewTopic(
        topic=topic_name,

        replication_factor=1,
        config={
            'min.insync.replicas': '1',
            'retention.ms': '600000',  # 10 min retention (benchmark data)
        }
    )

    futures = admin.create_topics([topic])
    for name, future in futures.items():
        try:
            future.result()
            print(f"  ✅ Created benchmark topic: {name} ({partitions} partitions)")
        except Exception as e:
            if 'already exists' in str(e):
                print(f"  ⏭️  Topic exists: {name}")
            else:
                print(f"  ❌ Error: {e}")

    time.sleep(1)

    return msg[:size_bytes]

class ProducerBenchmark:
    """
    Benchmark producer throughput and latency
    with different configuration profiles.
    """

    def __init__(self, broker='localhost:9092'):
        self.broker = broker
        self.topic = 'benchmark-producer'

    def run_benchmark(self, config_name: str, producer_config: dict,
                      num_messages: int = 100000) -> ProducerBenchmarkResult:
        """
        Run a single producer benchmark with given configuration.
        Returns detailed results for comparison.
        """
        setup_benchmark_topic(self.topic, partitions=12, broker=self.broker)

        # Merge base config with test config
        full_config = {
            'bootstrap.servers': self.broker,
            'client.id': f'benchmark-{config_name}',
        }
        full_config.update(producer_config)

        producer = Producer(full_config)

        latencies = []
        errors = 0
        total_bytes = 0
        delivered = 0
        delivery_times = {}

        def delivery_callback(err, msg):
            nonlocal errors, delivered
            msg_id = msg.key()
            if err:
                errors += 1
            else:
                delivered += 1
                if msg_id and msg_id in delivery_times:
                    latency = (time.time() - delivery_times[msg_id]) * 1000
                    latencies.append(latency)

        print(f"\n  🏃 Running benchmark: {config_name}")
        print(f"     Messages: {num_messages:,}")
        print(f"     Config: {producer_config}")

        # Warm-up: send 1000 messages to prime connections
        for i in range(min(1000, num_messages // 10)):
            msg = PREGENERATED_MESSAGES[i % len(PREGENERATED_MESSAGES)]
            producer.produce(
                topic=self.topic,
                value=msg,
                callback=lambda e, m: None
            )
        producer.flush()
        time.sleep(1)

        # Actual benchmark
        start_time = time.time()

        for i in range(num_messages):
            msg = PREGENERATED_MESSAGES[i % len(PREGENERATED_MESSAGES)]
            msg_key = f"{i}".encode('utf-8')

            # Track delivery time for latency measurement (sample 10%)
            if i % 10 == 0:
                delivery_times[msg_key] = time.time()

            try:
                producer.produce(
                    topic=self.topic,
                    key=msg_key,
                    value=msg,
                    callback=delivery_callback
                )
                total_bytes += len(msg)
            except BufferError:
                # Buffer full — poll to free space and retry
                producer.poll(0.1)
                try:
                    producer.produce(
                        topic=self.topic,
                        key=msg_key,
                        value=msg,
                        callback=delivery_callback
                    )
                    total_bytes += len(msg)
                except Exception:
                    errors += 1

            # Periodic poll to trigger callbacks
            if i % 1000 == 0:
                producer.poll(0)

            # Progress
            if i % 25000 == 0 and i > 0:
                elapsed = time.time() - start_time
                rate = i / elapsed
                print(f"     Progress: {i:,}/{num_messages:,} "
                      f"({rate:,.0f} msg/sec)")

        # Flush remaining messages
        producer.flush(timeout=30)
        end_time = time.time()

        duration = end_time - start_time

        result = ProducerBenchmarkResult(
            config_name=config_name,
            total_messages=num_messages,
            total_bytes=total_bytes,
            duration_seconds=duration,
            latencies_ms=latencies,
            errors=errors,
            config=producer_config
        )

        result.print_report()
        return result

class ConsumerBenchmark:
    """
    Benchmark consumer throughput with different fetch configurations.
    First fills topic with data, then measures consumption speed.
    """

    def __init__(self, broker='localhost:9092'):
        self.broker = broker
        self.topic = 'benchmark-consumer'

    def _fill_topic(self, num_messages: int):
        """Pre-fill topic with messages for consumer benchmark"""
        print(f"  📤 Pre-filling topic with {num_messages:,} messages...")

        producer = Producer({
            'bootstrap.servers': self.broker,
            'batch.size': 262144,
            'linger.ms': 50,
            'compression.type': 'lz4',
            'acks': '1',
            'buffer.memory': 134217728,
        })

        for i in range(num_messages):
            msg = PREGENERATED_MESSAGES[i % len(PREGENERATED_MESSAGES)]
            try:
                producer.produce(
                    topic=self.topic,
                    value=msg,
                    callback=lambda e, m: None
                )
            except BufferError:
                producer.poll(0.1)
                producer.produce(
                    topic=self.topic,
                    value=msg,
                    callback=lambda e, m: None
                )

            if i % 1000 == 0:
                producer.poll(0)

            if i % 50000 == 0 and i > 0:
                print(f"     Filled {i:,}/{num_messages:,}...")

        producer.flush(timeout=30)
        print(f"  ✅ Topic filled with {num_messages:,} messages")
        time.sleep(2)

    def run_benchmark(self, config_name: str, consumer_config: dict,
                      num_messages: int = 100000) -> ConsumerBenchmarkResult:
        """Run a single consumer benchmark"""

        setup_benchmark_topic(self.topic, partitions=12, broker=self.broker)
        self._fill_topic(num_messages)

        # Build consumer config
        full_config = {
            'bootstrap.servers': self.broker,
            'group.id': f'benchmark-consumer-{config_name}-{int(time.time())}',
            'auto.offset.reset': 'earliest',
            'enable.auto.commit': True,
            'client.id': f'bench-consumer-{config_name}',
        }
        full_config.update(consumer_config)

        consumer = Consumer(full_config)
        consumer.subscribe([self.topic])

        print(f"\n  🏃 Running consumer benchmark: {config_name}")
        print(f"     Target messages: {num_messages:,}")
        print(f"     Config: {consumer_config}")

        total_messages = 0
        total_bytes = 0
        poll_count = 0
        empty_polls = 0

        start_time = time.time()

        while total_messages < num_messages and empty_polls < 30:
            msg = consumer.poll(timeout=1.0)
            poll_count += 1

            if msg is None:
                empty_polls += 1
                continue

            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    empty_polls += 1
                    continue
                continue

            empty_polls = 0
            total_messages += 1
            total_bytes += len(msg.value()) if msg.value() else 0

            if total_messages % 25000 == 0:
                elapsed = time.time() - start_time
                rate = total_messages / elapsed
                print(f"     Progress: {total_messages:,}/{num_messages:,} "
                      f"({rate:,.0f} msg/sec)")

        end_time = time.time()
        consumer.close()

        duration = end_time - start_time

        result = ConsumerBenchmarkResult(
            config_name=config_name,
            total_messages=total_messages,
            total_bytes=total_bytes,
            duration_seconds=duration,
            poll_count=poll_count,
            config=consumer_config
        )

        result.print_report()
        return result

# ============================================================
# FILE: e2e_latency_benchmark.py
# PURPOSE: Measure true end-to-end latency
#          (produce timestamp → consumer receive timestamp)
#          This is the metric users ACTUALLY care about
# ============================================================

import time
import json
import uuid
import threading
import statistics
import sys
from confluent_kafka import Producer, Consumer, KafkaError
from kafka_benchmark import setup_benchmark_topic


class EndToEndBenchmark:
    """
    Measures the REAL end-to-end latency:
    → Time from producer.produce() to consumer.poll() receiving the message
    → Includes: batching delay + network + broker write + replication +
                fetch delay + network back + deserialization
    
    This is the metric that matters for real-time applications.
    Throughput is meaningless if latency is unacceptable.
    """

    def __init__(self, broker='localhost:9092'):
        self.broker = broker
        self.topic = 'benchmark-e2e'
        self.latencies = []
        self.consumer_running = True

    def run_benchmark(self, config_name: str,
                      producer_config: dict,
                      consumer_config: dict,
                      num_messages: int = 10000,
                      produce_rate: int = 1000):
        """
        Run end-to-end latency benchmark.
        
        produce_rate: messages per second (controls load)
        Lower rate → more realistic for latency measurement
        Higher rate → tests behavior under load
        """

        setup_benchmark_topic(self.topic, partitions=6, broker=self.broker)
        self.latencies = []
        self.consumer_running = True

        # Build configs
        full_producer_config = {
            'bootstrap.servers': self.broker,
            'client.id': f'e2e-producer-{config_name}',
        }
        full_producer_config.update(producer_config)

        full_consumer_config = {
            'bootstrap.servers': self.broker,
            'group.id': f'e2e-consumer-{config_name}-{int(time.time())}',
            'auto.offset.reset': 'latest',
            'enable.auto.commit': True,
            'client.id': f'e2e-consumer-{config_name}',
        }
        full_consumer_config.update(consumer_config)

        print(f"\n  {'='*60}")
        print(f"  🏃 E2E LATENCY BENCHMARK: {config_name}")
        print(f"  {'='*60}")
        print(f"  Messages:     {num_messages:,}")
        print(f"  Produce rate: {produce_rate:,} msg/sec")
        print(f"  Producer:     {producer_config}")
        print(f"  Consumer:     {consumer_config}")

        # Start consumer FIRST (needs to be subscribed before producing)
        consumer = Consumer(full_consumer_config)
        consumer.subscribe([self.topic])

        # Pre-subscribe warmup
        for _ in range(5):
            consumer.poll(timeout=0.5)

        time.sleep(1)

        # Start consumer thread
        received_timestamps = {}
        consumer_errors = 0

        def consumer_loop():
            nonlocal consumer_errors
            while self.consumer_running:
                msg = consumer.poll(timeout=0.5)
                if msg is None:
                    continue
                if msg.error():
                    if msg.error().code() == KafkaError._PARTITION_EOF:
                        continue
                    consumer_errors += 1
                    continue

                receive_time = time.time()
                try:
                    value = json.loads(msg.value().decode('utf-8'))
                    msg_id = value.get('msg_id')
                    send_time = value.get('send_time', 0)

                    if msg_id and send_time:
                        latency_ms = (receive_time - send_time) * 1000
                        self.latencies.append(latency_ms)
                except Exception:
                    consumer_errors += 1

        consumer_thread = threading.Thread(target=consumer_loop, daemon=True)
        consumer_thread.start()

        # Produce messages at specified rate
        producer = Producer(full_producer_config)
        interval = 1.0 / produce_rate if produce_rate > 0 else 0
        produce_errors = 0

        print(f"  📤 Producing at {produce_rate} msg/sec...")

        start_time = time.time()

        for i in range(num_messages):
            send_time = time.time()
            message = {
                'msg_id': f"e2e-{i}",
                'send_time': send_time,
                'sequence': i,
                'data': 'X' * 200  # ~300 byte payload
            }

            try:
                producer.produce(
                    topic=self.topic,
                    key=f"{i}".encode('utf-8'),
                    value=json.dumps(message).encode('utf-8'),
                    callback=lambda e, m: None
                )
            except BufferError:
                producer.poll(0.01)
                try:
                    producer.produce(
                        topic=self.topic,
                        key=f"{i}".encode('utf-8'),
                        value=json.dumps(message).encode('utf-8'),
                        callback=lambda e, m: None
                    )
                except Exception:
                    produce_errors += 1

            # Rate limiting
            if interval > 0 and i % 10 == 0:
                producer.poll(0)
                elapsed = time.time() - start_time
                expected_elapsed = (i + 1) * interval
                sleep_time = expected_elapsed - elapsed
                if sleep_time > 0:
                    time.sleep(sleep_time)

            if i % 2500 == 0 and i > 0:
                current_latencies = len(self.latencies)
                print(f"     Produced: {i:,}, Consumed: {current_latencies:,}")

        producer.flush(timeout=30)

        # Wait for consumer to catch up
        print(f"  ⏳ Waiting for consumer to catch up...")
        wait_start = time.time()
        while len(self.latencies) < num_messages * 0.95:
            if time.time() - wait_start > 30:
                print(f"     Timeout waiting for consumer (got {len(self.latencies)}/{num_messages})")
                break
            time.sleep(0.5)

        self.consumer_running = False
        consumer_thread.join(timeout=5)
        consumer.close()

        # Calculate results
        total_duration = time.time() - start_time

        self._print_latency_report(
            config_name, num_messages, produce_rate,
            total_duration, produce_errors, consumer_errors,
            producer_config, consumer_config
        )

    def _print_latency_report(self, config_name, num_messages, produce_rate,
                               duration, prod_errors, cons_errors,
                               prod_config, cons_config):
        """Print detailed latency analysis"""

        if not self.latencies:
            print(f"\n  ❌ No latency data collected!")
            return

        sorted_lat = sorted(self.latencies)
        count = len(sorted_lat)

        p50 = sorted_lat[int(count * 0.50)]
        p75 = sorted_lat[int(count * 0.75)]
        p90 = sorted_lat[int(count * 0.90)]
        p95 = sorted_lat[int(count * 0.95)]
        p99 = sorted_lat[min(int(count * 0.99), count - 1)]
        p999 = sorted_lat[min(int(count * 0.999), count - 1)]
        avg = statistics.mean(self.latencies)
        std = statistics.stdev(self.latencies) if count > 1 else 0

        print(f"\n  {'='*60}")
        print(f"  📊 E2E LATENCY REPORT: {config_name}")
        print(f"  {'='*60}")
        print(f"  Messages sent:      {num_messages:,}")
        print(f"  Messages received:  {count:,} ({count/num_messages*100:.1f}%)")
        print(f"  Produce rate:       {produce_rate:,} msg/sec")
        print(f"  Duration:           {duration:.2f}s")
        print(f"  Producer errors:    {prod_errors}")
        print(f"  Consumer errors:    {cons_errors}")

        print(f"\n  📏 LATENCY DISTRIBUTION:")
        print(f"     Min:    {sorted_lat[0]:>10.2f} ms")
        print(f"     p50:    {p50:>10.2f} ms")
        print(f"     p75:    {p75:>10.2f} ms")
        print(f"     p90:    {p90:>10.2f} ms")
        print(f"     p95:    {p95:>10.2f} ms")
        print(f"     p99:    {p99:>10.2f} ms   ← MOST IMPORTANT")
        print(f"     p99.9:  {p999:>10.2f} ms")
        print(f"     Max:    {sorted_lat[-1]:>10.2f} ms")
        print(f"     Avg:    {avg:>10.2f} ms")
        print(f"     StdDev: {std:>10.2f} ms")

        # Histogram
        print(f"\n  📊 LATENCY HISTOGRAM:")
        buckets = [1, 2, 5, 10, 20, 50, 100, 200, 500, 1000, 5000]
        prev = 0
        for bucket in buckets:
            in_bucket = sum(1 for l in self.latencies if prev < l <= bucket)
            pct = in_bucket / count * 100
            bar = '█' * int(pct / 2)
            if in_bucket > 0:
                print(f"     {prev:>5d}-{bucket:>5d}ms: {bar:<30s} {pct:>5.1f}% ({in_bucket:,})")
            prev = bucket
        remaining = sum(1 for l in self.latencies if l > buckets[-1])
        if remaining > 0:
            pct = remaining / count * 100
            print(f"     >{buckets[-1]:>5d}ms: {'█' * int(pct / 2):<30s} {pct:>5.1f}% ({remaining:,})")

        # Latency budget breakdown
        print(f"\n  ⏱️  LATENCY BUDGET ESTIMATE:")
        linger = prod_config.get('linger.ms', 0)
        fetch_wait = cons_config.get('fetch.max.wait.ms', 500)
        print(f"     Producer linger.ms:      {linger} ms (batching delay)")
        print(f"     Network (est):           ~2-5 ms (producer→broker→consumer)")
        print(f"     Broker write:            ~1-3 ms (disk write + replication)")
        print(f"     Consumer fetch.wait:     up to {fetch_wait} ms (if no data)")
        print(f"     Measured p50:            {p50:.1f} ms")
        print(f"     Measured p99:            {p99:.1f} ms")

        print(f"  {'='*60}")

import time
import json
import sys
from confluent_kafka import Producer
from kafka_benchmark import (
    setup_benchmark_topic, generate_message, PREGENERATED_MESSAGES
)


class CompressionBenchmark:
    """
    Compare compression algorithms head-to-head.
    For each algorithm: measure throughput, ratio, and CPU impact.
    """

    def __init__(self, broker='localhost:9092'):
        self.broker = broker

    def benchmark_compression(self, algorithm: str,
                               num_messages: int = 50000) -> dict:
        """Benchmark a single compression algorithm"""

        topic = f'benchmark-compress-{algorithm}'
        setup_benchmark_topic(topic, partitions=6, broker=self.broker)

        config = {
            'bootstrap.servers': self.broker,
            'batch.size': 131072,          # 128KB — large enough for good compression
            'linger.ms': 20,
            'compression.type': algorithm,
            'acks': '1',
            'buffer.memory': 67108864,
        }

        producer = Producer(config)

        total_raw_bytes = 0
        delivered_count = 0
        errors = 0

        def callback(err, msg):
            nonlocal delivered_count, errors
            if err:
                errors += 1
            else:
                delivered_count += 1

        print(f"  🗜️  Benchmarking: {algorithm}")

        start = time.time()

        for i in range(num_messages):
            msg = PREGENERATED_MESSAGES[i % len(PREGENERATED_MESSAGES)]
            total_raw_bytes += len(msg)

            try:
                producer.produce(
                    topic=topic,
                    value=msg,
                    callback=callback
                )
            except BufferError:
                producer.poll(0.1)
                producer.produce(
                    topic=topic,
                    value=msg,
                    callback=callback
                )

            if i % 1000 == 0:
                producer.poll(0)

        producer.flush(timeout=30)
        duration = time.time() - start

        throughput = num_messages / duration
        bandwidth_raw = (total_raw_bytes / (1024 * 1024)) / duration

        result = {
            'algorithm': algorithm,
            'messages': num_messages,
            'raw_bytes': total_raw_bytes,
            'duration': duration,
            'throughput_msg_sec': throughput,
            'bandwidth_raw_mb_sec': bandwidth_raw,
            'errors': errors,
            'delivered': delivered_count,
        }

        print(f"     Throughput: {throughput:,.0f} msg/sec | "
              f"Raw bandwidth: {bandwidth_raw:.1f} MB/sec | "
              f"Duration: {duration:.2f}s")

        return result

    def run_full_comparison(self, num_messages=50000):
        """Compare all compression algorithms"""

        algorithms = ['none', 'lz4', 'snappy', 'zstd', 'gzip']
        results = []

        print("\n" + "=" * 70)
        print("  🗜️  COMPRESSION ALGORITHM COMPARISON")
        print(f"  Messages: {num_messages:,} × {len(PREGENERATED_MESSAGES[0])} bytes each")
        print("=" * 70)

        for algo in algorithms:
            result = self.benchmark_compression(algo, num_messages)
            results.append(result)
            time.sleep(2)

        # Print comparison
        print("\n" + "=" * 90)
        print("  📊 COMPRESSION COMPARISON RESULTS")
        print("=" * 90)

        baseline = next(r for r in results if r['algorithm'] == 'none')

# FILE: tuning_advisor.py
# PURPOSE: Interactive tuning advisor that recommends
#          configurations based on workload characteristics
# CONNECTS: All benchmarking results to actionable config

import json
import sys


class TuningAdvisor:
    """
    Given workload characteristics, recommends optimal
    Kafka producer and consumer configurations.
    
    This encodes all the knowledge from our benchmarks
    into an automated recommendation engine.
    """

    def analyze_workload(self, workload: dict) -> dict:
        """
        Given workload characteristics, produce configuration recommendations.
        
        workload dict keys:
        → messages_per_sec: target throughput
        → message_size_bytes: average message size
        → latency_target_ms: p99 latency target
        → durability: 'best_effort', 'standard', 'critical'
        → data_type: 'json', 'avro', 'binary', 'compressed'
        → num_partitions: number of topic partitions
        → retention_days: how long to keep data
        → consumer_count: number of consumer instances
        """

        msg_rate = workload.get('messages_per_sec', 1000)
        msg_size = workload.get('message_size_bytes', 500)
        latency_target = workload.get('latency_target_ms', 100)
        durability = workload.get('durability', 'standard')
        data_type = workload.get('data_type', 'json')
        num_partitions = workload.get('num_partitions', 12)
        retention_days = workload.get('retention_days', 7)
        consumer_count = workload.get('consumer_count', 3)

        # Calculate derived metrics
        raw_bandwidth_mb = (msg_rate * msg_size) / (1024 * 1024)
        msgs_per_partition = msg_rate / num_partitions
        daily_storage_gb = (msg_rate * msg_size * 86400) / (1024 ** 3)

        # Determine workload profile
        if latency_target <= 10:
            profile = 'low_latency'
        elif msg_rate > 100000:
            profile = 'high_throughput'
        elif durability == 'critical':
            profile = 'max_durability'
        else:
            profile = 'balanced'

        # ====================================================
        # PRODUCER RECOMMENDATIONS
        # ====================================================

        producer_config = {}

        # batch.size
        if profile == 'low_latency':
            producer_config['batch.size'] = 16384
        elif profile == 'high_throughput':
            producer_config['batch.size'] = min(262144, msg_size * 500)
        else:
            producer_config['batch.size'] = 65536

        # linger.ms
        if profile == 'low_latency':
            producer_config['linger.ms'] = 0
        elif profile == 'high_throughput':
            optimal_linger = min(50, max(5, int(1000 / msgs_per_partition * 10)))
            producer_config['linger.ms'] = optimal_linger
        else:
            producer_config['linger.ms'] = 5

        # compression.type
        if data_type == 'compressed' or data_type == 'binary':
            producer_config['compression.type'] = 'none'
            compression_reason = 'Data already compressed/binary — compression adds CPU with no benefit'
        elif profile == 'low_latency':
            producer_config['compression.type'] = 'lz4'
            compression_reason = 'lz4: minimal CPU overhead for low-latency workload'
        elif daily_storage_gb > 100:
            producer_config['compression.type'] = 'zstd'
            compression_reason = 'zstd: best ratio for high storage volume'
        else:
            producer_config['compression.type'] = 'lz4'
            compression_reason = 'lz4: best balance of speed and compression'

        # acks
        if durability == 'best_effort':
            producer_config['acks'] = '1'
            acks_reason = 'acks=1: leader only — acceptable for best-effort data'
        elif durability == 'critical':
            producer_config['acks'] = 'all'
            producer_config['enable.idempotence'] = True
            producer_config['min.insync.replicas'] = 2
            acks_reason = 'acks=all + idempotence: maximum durability for critical data'
        else:
            producer_config['acks'] = 'all'
            acks_reason = 'acks=all: standard durability'

        # buffer.memory
        min_buffer = producer_config['batch.size'] * num_partitions * 2
        recommended_buffer = max(min_buffer, 67108864)
        if profile == 'high_throughput':
            recommended_buffer = max(recommended_buffer, 134217728)
        producer_config['buffer.memory'] = recommended_buffer

        # ====================================================
        # CONSUMER RECOMMENDATIONS
        # ====================================================

        consumer_config = {}

        # fetch.min.bytes
        if profile == 'low_latency':
            consumer_config['fetch.min.bytes'] = 1
        elif profile == 'high_throughput':
            consumer_config['fetch.min.bytes'] = min(102400, msg_size * 100)
        else:
            consumer_config['fetch.min.bytes'] = max(1024, msg_size * 10)

        # fetch.max.wait.ms
        if profile == 'low_latency':
            consumer_config['fetch.max.wait.ms'] = 50
        elif profile == 'high_throughput':
            consumer_config['fetch.max.wait.ms'] = 200
        else:
            consumer_config['fetch.max.wait.ms'] = 100

        # max.poll.records
        if profile == 'low_latency':
            consumer_config['max.poll.records'] = 100
        elif profile == 'high_throughput':
            consumer_config['max.poll.records'] = 5000
        else:
            consumer_config['max.poll.records'] = 1000

        # max.partition.fetch.bytes
        if profile == 'high_throughput':
            consumer_config['max.partition.fetch.bytes'] = 4194304
        else:
            consumer_config['max.partition.fetch.bytes'] = 1048576

        # ====================================================
        # BUILD REPORT
        # ====================================================

        compression_ratio = {
            'none': 1.0, 'lz4': 2.5, 'snappy': 2.3,
            'zstd': 3.5, 'gzip': 3.8
        }
        est_ratio = compression_ratio.get(
            producer_config['compression.type'], 1.0
        )
        compressed_bandwidth = raw_bandwidth_mb / est_ratio
        compressed_daily = daily_storage_gb / est_ratio

        report = {
            'workload': workload,
            'profile': profile,
            'producer_config': producer_config,
            'consumer_config': consumer_config,
            'analysis': {
                'raw_bandwidth_mb_sec': round(raw_bandwidth_mb, 1),
                'compressed_bandwidth_mb_sec': round(compressed_bandwidth, 1),
                'daily_storage_raw_gb': round(daily_storage_gb, 1),
                'daily_storage_compressed_gb': round(compressed_daily, 1),
                'total_retention_storage_gb': round(compressed_daily * retention_days * 3, 1),
                'compression_reason': compression_reason,
                'acks_reason': acks_reason,
                'msgs_per_partition_per_sec': round(msgs_per_partition, 0),
                'consumers_needed': max(
                    consumer_count,
                    int(msg_rate / 50000) + 1
                ),
            }
        }

        return report

    def print_recommendation(self, report: dict):
        """Print formatted recommendation"""

        w = report['workload']
        a = report['analysis']
        pc = report['producer_config']
        cc = report['consumer_config']

        print("\n" + "=" * 70)
        print("  🎯 KAFKA TUNING RECOMMENDATION")
        print("=" * 70)

        print(f"\n  📋 WORKLOAD ANALYSIS:")
        print(f"     Messages/sec:         {w.get('messages_per_sec', '?'):,}")

        print(f"     Durability:           {w.get('durability', '?')}")

        print(f"     Partitions:           {w.get('num_partitions', '?')}")
        print(f"     Retention:            {w.get('retention_days', '?')} days")

        print(f"\n  🏷️  DETECTED PROFILE: {report['profile'].upper()}")

                print(f"     {key:<40s} = {human}")

                print(f"     {key:<40s} = {value}")
        print(f"     Compression reason: {a['compression_reason']}")

        for key, value in cc.items():
            if isinstance(value, int) and value > 10000:

            else:
                print(f"     {key:<40s} = {value}")

        print(f"\n  📊 CAPACITY ESTIMATES:")
        print(f"     Raw bandwidth:        {a['raw_bandwidth_mb_sec']} MB/sec")
