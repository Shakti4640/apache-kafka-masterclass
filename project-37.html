🔴 HYPOTHETICAL PROBLEM

MegaRetail is a large e-commerce platform with 200+ microservices.
Each service owns its database (PostgreSQL, MySQL, MongoDB).

CURRENT SITUATION:

The Order Service writes to PostgreSQL:
  → INSERT INTO orders (id, customer_id, total, status) VALUES (...)
  → UPDATE orders SET status = 'SHIPPED' WHERE id = 1234

Other services NEED to know about these changes:
  → Inventory Service: decrement stock when order placed
  → Analytics Service: update dashboards in real-time
  → Search Service: update product availability index
  → Notification Service: send email when order ships
  → Saga Orchestrator (Project 36): coordinate multi-step flows

APPROACH 1 (Current — Dual Write):
  Order Service code:
    def place_order(order):
        db.insert(order)                    # Write to database
        kafka_producer.send("orders", order) # Write to Kafka

  PROBLEMS:
    → Database write succeeds, Kafka write fails → data inconsistency
    → Kafka write succeeds, database write fails → phantom event
    → Network partition between DB and Kafka → which one wins?
    → Developer forgets to add Kafka publish → silent data loss
    → 200 services × every write operation = thousands of dual-write points

APPROACH 2 (Project 23 — JDBC Polling):
  Kafka Connect JDBC Source polls database every N seconds:
    SELECT * FROM orders WHERE updated_at > last_poll_time

  PROBLEMS:
    → Polling interval creates latency (seconds to minutes)
    → Heavy database load from constant polling
    → Cannot capture DELETE operations
    → Misses rapid updates between polls
    → Requires every table to have updated_at column
    → Cannot capture the BEFORE state of an update

APPROACH 3 (What we need — CDC):
  → Read the database's own TRANSACTION LOG
  → PostgreSQL WAL (Write-Ahead Log)
  → MySQL Binlog (Binary Log)
  → MongoDB Oplog (Operations Log)
  → Every INSERT, UPDATE, DELETE captured in real-time
  → Zero application code changes
  → Millisecond latency
  → Captures BEFORE and AFTER state of every change

  This is CHANGE DATA CAPTURE (CDC).
  The tool: DEBEZIUM — the industry standard CDC platform for Kafka.

CEO: "We spent 6 months adding Kafka producers to 50 services.
      200 services remaining. We need a better approach."

Answer: Debezium reads the database log. ZERO code changes needed.

🟡 HOW TO APPROACH IT

CORE INSIGHT:
→ Every database already maintains a transaction log
→ This log is HOW databases implement ACID properties
→ PostgreSQL WAL: sequential record of every change
→ MySQL Binlog: binary log of every modification
→ This log IS the source of truth — the tables are just a materialized view
→ This connects directly to Event Sourcing (Project 34):
    Database log = event store
    Table state = projection/materialization

DEBEZIUM WORKFLOW:

  Step 1: INITIAL SNAPSHOT
    → On first start, Debezium reads entire table
    → Produces one Kafka message per existing row
    → This gives consumers a complete starting point

  Step 2: STREAMING CHANGES
    → After snapshot, switches to reading WAL/Binlog
    → Every INSERT → Kafka message with "op": "c" (create)
    → Every UPDATE → Kafka message with "op": "u" (update) + before/after
    → Every DELETE → Kafka message with "op": "d" (delete) + before state

  Step 3: CONTINUOUS OPERATION
    → Debezium tracks its position in the WAL (LSN — Log Sequence Number)
    → On restart, resumes from last committed position
    → No data loss, no duplicates (with exactly-once via Project 18)

✅ WHAT SHOULD DO

1. ENABLE LOGICAL REPLICATION IN PostgreSQL
   → Set wal_level = logical in postgresql.conf
   → Create a replication slot for Debezium
   → Debezium needs a user with REPLICATION privilege
   → Without this, Debezium cannot read the WAL

2. USE LOGICAL DECODING OUTPUT PLUGIN
   → pgoutput (built into PostgreSQL 10+) — RECOMMENDED
   → decoderbufs (Debezium's own, requires extension install)
   → wal2json (JSON-based output)
   → pgoutput requires zero extensions — simplest path

3. SET APPROPRIATE TABLE PRIMARY KEYS
   → Debezium uses primary key as Kafka message key
   → Primary key → message key → partition assignment
   → All changes for same row → same partition → ordering guaranteed
   → Tables without primary keys: Debezium uses entire row as key (messy)

4. CONFIGURE SNAPSHOT MODE CAREFULLY
   → initial: Full snapshot on first run, then stream (RECOMMENDED for new setup)
   → schema_only: No data snapshot, only schema — start streaming immediately
   → never: No snapshot at all — only stream new changes
   → always: Full snapshot on every restart (expensive, rarely needed)

5. MONITOR REPLICATION LAG
   → Track distance between WAL write position and Debezium read position
   → Growing lag = Debezium falling behind = stale data downstream
   → Alert when lag exceeds threshold

6. USE SCHEMA HISTORY TOPIC
   → Debezium tracks table schema changes (ALTER TABLE)
   → Stores schema history in dedicated Kafka topic
   → On restart, replays schema history to understand current structure
   → Without this: schema mismatch errors after DDL changes

❌ WHAT SHOULD NOT DO

1. NEVER SKIP WAL CONFIGURATION
   → Default PostgreSQL: wal_level = replica (insufficient)
   → Debezium REQUIRES wal_level = logical
   → Changing wal_level requires PostgreSQL RESTART
   → Plan this during maintenance window, not in production emergency

2. NEVER USE DEBEZIUM WITHOUT MONITORING REPLICATION SLOTS
   → Inactive replication slot → WAL files accumulate on disk
   → Debezium down for hours → gigabytes of WAL accumulate
   → Disk fills up → PostgreSQL CRASHES (refuses writes)
   → Monitor slot activity, set max_slot_wal_keep_size

3. NEVER IGNORE TOMBSTONE EVENTS
   → DELETE produces two events:
     a. Delete event with before state (op: "d")
     b. Tombstone: same key, null value
   → Tombstone enables log compaction (Project 14) to remove key
   → If you filter out tombstones → compacted topics grow forever

4. NEVER PROCESS CDC EVENTS WITHOUT IDEMPOTENCY
   → Debezium guarantees at-least-once delivery
   → Connector restart may replay recent events
   → Consumers MUST handle duplicate events
   → Use event LSN (Log Sequence Number) for deduplication

5. NEVER USE DEBEZIUM FOR HIGH-FREQUENCY METRICS TABLES
   → Table updated 10,000 times/second → 10,000 CDC events/second
   → Consider: does downstream actually need every intermediate value?
   → For metrics: use periodic aggregation, not CDC
   → CDC is for business state changes, not telemetry

6. NEVER MODIFY DATABASE DIRECTLY TO FIX CDC ISSUES
   → "Just UPDATE the row to re-trigger CDC" → creates phantom event
   → Fix Debezium connector, not the data
   → Use snapshot mode for recovery if events were missed

  → Existing databases that need Kafka integration WITHOUT code changes
  → Need real-time (<100ms) event propagation
  → Need DELETE capture and before/after state
  → Microservice database → event bus integration
  → Saga pattern (Project 36) needs reliable event emission
  → CQRS (Project 35) read model needs to sync from write database

WHEN JDBC POLLING (Project 23) IS SUFFICIENT:
  → Batch analytics that can tolerate minutes of delay
  → Simple append-only tables (logs, metrics)
  → Database doesn't support logical replication
  → Quick proof-of-concept before CDC investment

🚫 WHEN NOT TO USE

1. WHEN DATABASE DOESN'T SUPPORT LOG-BASED REPLICATION
   → Some embedded databases (SQLite, H2) have no WAL access
   → Some cloud databases restrict replication features
   → Fall back to JDBC polling (Project 23)

2. WHEN ALL YOU NEED IS A ONE-TIME DATA MIGRATION
   → CDC is for continuous streaming
   → One-time migration: use pg_dump → kafka-console-producer
   → Or JDBC connector in bulk mode

3. WHEN THE DATABASE IS EPHEMERAL
   → Test databases, dev databases spun up/down frequently
   → Debezium replication slot overhead not worth it
   → Use application-level events instead

4. WHEN YOU CONTROL ALL APPLICATION CODE
   → If you own every service and can add Kafka producers
   → Direct application events are simpler (no Debezium infra)
   → BUT: dual-write problem remains (see Outbox Pattern below)

5. WHEN DOWNSTREAM DOESN'T NEED ROW-LEVEL CHANGES
   → Dashboard that shows daily totals → batch query is fine
   → CDC provides granularity that may be unnecessary
   → Over-engineering: don't use a fire hose when a garden hose works

6. WHEN DATABASE WRITE VOLUME IS EXTREME
   → 100K+ writes/second to a single table
   → CDC event volume may overwhelm Kafka topic
   → Consider: aggregation layer between Debezium and consumers

⚙️ HOW IT ONLY WORKS (INTERNAL MECHANICS)

POSTGRESQL WAL (Write-Ahead Log) — THE FOUNDATION:

  Every PostgreSQL write follows this sequence:

  Application: INSERT INTO orders (id, customer, total) VALUES (1, 'Alice', 150.00)

  Step 1: PostgreSQL writes to WAL FIRST
    WAL Record:
    ┌──────────────────────────────────────────────┐
    │ LSN: 0/1644A30                               │
    │ Transaction ID: 5823                          │
    │ Type: INSERT                                  │
    │ Relation: public.orders                       │
    │ New tuple: {id:1, customer:'Alice', total:150}│
    │ Timestamp: 2025-07-15T10:23:45.123Z          │
    └──────────────────────────────────────────────┘

  Step 2: THEN writes to actual table data files
    → If crash between Step 1 and Step 2: WAL replays to recover
    → This is why it's called "Write-AHEAD Log"

  Step 3: WAL segment files accumulate
    → Each segment: 16MB by default
    → Named: 000000010000000000000001, ...00000002, etc.
    → Retained until all replication slots have consumed them


LOGICAL REPLICATION — HOW DEBEZIUM READS WAL:

  PostgreSQL supports two replication modes:

  PHYSICAL REPLICATION:
    → Sends raw WAL bytes to replica
    → Used for standby databases
    → Cannot be selectively filtered
    → Not useful for Debezium

  LOGICAL REPLICATION:
    → Decodes WAL into logical change events
    → Can filter by table, schema
    → Output plugins transform to readable format
    → THIS is what Debezium uses

  Debezium creates a REPLICATION SLOT:
    → Named: debezium_slot (configurable)
    → PostgreSQL guarantees WAL retained until slot confirms consumption
    → Debezium reads decoded changes via streaming replication protocol
    → Tracks position via LSN (Log Sequence Number)

  ┌────────────┐    Logical      ┌────────────┐    Kafka     ┌────────┐
  │ PostgreSQL │    Replication   │  Debezium  │    Connect   │ Kafka  │
  │    WAL     │───────────────▶│  Connector │────────────▶│ Topics │
  │            │    Protocol     │            │    Protocol  │        │
  │  LSN: 0/A │    (streaming)  │ Reads from │   (produce)  │        │
  │  LSN: 0/B │                 │ LSN: 0/A   │              │        │
  │  LSN: 0/C │                 │ Advances   │              │        │
  │            │                 │ to 0/B,0/C │              │        │
  └────────────┘                 └────────────┘              └────────┘


DEBEZIUM EVENT STRUCTURE:

  Every CDC event has this structure:

  {
    "schema": { ... },        // Kafka Connect schema (for Schema Registry)
    "payload": {
      "before": { ... },      // Row state BEFORE the change (null for INSERT)
      "after": { ... },       // Row state AFTER the change (null for DELETE)
      "source": {
        "version": "2.4.0",
        "connector": "postgresql",
        "name": "megastore",
        "ts_ms": 1689412345123,   // Database timestamp
        "snapshot": "false",
        "db": "megastore",
        "schema": "public",
        "table": "orders",
        "txId": 5823,
        "lsn": 23456789,         // WAL position
        "xmin": null
      },
      "op": "u",              // Operation: c=create, u=update, d=delete, r=read(snapshot)
      "ts_ms": 1689412345456, // Debezium processing timestamp
      "transaction": {
        "id": "5823",
        "total_order": 1,
        "data_collection_order": 1
      }
    }
  }

  OPERATION TYPES:
    "op": "r" → Snapshot read (initial load)
    "op": "c" → Create (INSERT)
    "op": "u" → Update (UPDATE) — has both before AND after
    "op": "d" → Delete (DELETE) — has before, after is null
    (followed by tombstone: key present, value null)


SNAPSHOT PROCESS — FIRST STARTUP:

  ┌─────────────────────────────────────────────────────────┐
  │ Phase 1: LOCK (brief)                                   │
  │   → Debezium takes a snapshot lock on tables             │
  │   → Reads current WAL position (LSN = 0/ABC)            │
  │   → Records table schemas                                │
  │   → Releases lock                                        │
  │                                                         │
  │ Phase 2: READ (may take hours for large tables)         │
  │   → SELECT * FROM orders                                │
  │   → Each row → Kafka event with "op": "r"              │
  │   → Produces in chunks (configurable batch size)        │
  │                                                         │
  │ Phase 3: STREAM                                         │
  │   → Switches to WAL reading from LSN 0/ABC             │
  │   → All changes AFTER snapshot position are captured    │
  │   → Changes DURING snapshot are also captured           │
  │     (because we recorded LSN before reading)            │
  │   → Deduplication ensures no double-counting            │
  │                                                         │
  │ Phase 4: CONTINUOUS                                     │
  │   → Reads WAL in real-time                              │
  │   → Commits LSN position back to replication slot       │
  │   → On restart: resume from last committed LSN          │
  └─────────────────────────────────────────────────────────┘

THE OUTBOX PATTERN — SOLVING DUAL-WRITE:

  PROBLEM (as stated above):
    db.insert(order)          # What if this succeeds...
    kafka.send("orders", order)  # ...but this fails?

  OUTBOX SOLUTION:
    Instead of writing to Kafka directly, write to an OUTBOX TABLE
    in the SAME database transaction:

    BEGIN TRANSACTION;
      INSERT INTO orders (id, customer, total) VALUES (1, 'Alice', 150);
      INSERT INTO outbox (
        aggregate_type, aggregate_id, event_type, payload
      ) VALUES (
        'Order', '1', 'OrderCreated',
        '{"id":1,"customer":"Alice","total":150}'
      );
    COMMIT;

    → Both writes are in SAME transaction → atomic
    → Either BOTH succeed or BOTH fail → no inconsistency
    → Debezium reads the outbox table via CDC
    → Routes outbox events to appropriate Kafka topics
    → Debezium Outbox Event Router SMT (Project 24) handles routing

  FLOW:
    ┌─────────┐   ACID tx    ┌─────────────┐   CDC    ┌─────────┐   route   ┌──────────┐
    │  App    │─────────────▶│ PostgreSQL  │────────▶│Debezium │──────────▶│  Kafka   │
    │ Service │  write both  │ orders +    │  read   │         │  publish  │  Topics  │
    │         │  in one tx   │ outbox      │  WAL    │         │  events   │          │
    └─────────┘              └─────────────┘         └─────────┘          └──────────┘

  WHY THIS MATTERS FOR SAGA (Project 36):
    → Saga services write to their database AND outbox in one transaction
    → Debezium captures outbox events → Kafka → Orchestrator
    → No dual-write problem
    → Service code NEVER directly produces to Kafka
    → Reliable event emission without Kafka client in business logic

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: WAL DISK EXPLOSION
  Problem: Debezium connector goes down
  Result: Replication slot retains ALL WAL since last consumed position
  Timeline:
    → Debezium down for 4 hours
    → Database writes 50GB of WAL during that time
    → 50GB of WAL files accumulated on disk
    → If disk fills → PostgreSQL STOPS accepting writes
  Solution:
    → Set max_slot_wal_keep_size = 10GB in postgresql.conf (PG 13+)
    → Monitor pg_replication_slots view
    → Alert when slot lag exceeds threshold
    → If slot falls too far behind → drop and recreate (triggers re-snapshot)

PITFALL 2: SNAPSHOT TAKES FOREVER
  Problem: Table has 500 million rows
  Result: Initial snapshot takes 12+ hours
  Timeline:
    → During snapshot, new changes accumulate in WAL
    → Consumers see stale snapshot data mixed with current state
  Solution:
    → Use snapshot.mode = schema_only if you only need future changes
    → For large tables: snapshot.select.statement.overrides to limit rows
    → Increase snapshot.fetch.size for faster reads
    → Run initial snapshot during low-traffic period

PITFALL 3: SCHEMA CHANGE BREAKS CONNECTOR
  Problem: DBA runs ALTER TABLE orders ADD COLUMN discount DECIMAL
  Result: Debezium detects schema change mid-stream
  Timeline:
    → Events before ALTER have old schema (no discount column)
    → Events after ALTER have new schema (with discount column)
    → Consumers parsing with fixed schema BREAK
  Solution:
    → Debezium publishes schema to schema history topic
    → Use Schema Registry (Project 16) with Avro for evolution
    → Schema compatibility (Project 38) handles additions gracefully
    → Backward compatible changes (add optional columns) are safe

PITFALL 4: TRANSACTION ORDERING ACROSS TABLES
  Problem: Single database transaction writes to orders AND order_items
  Result: Debezium may produce order_items event BEFORE orders event
  Timeline:
    → Consumer receives order_item for order_id=5
    → But order_id=5 hasn't arrived yet on orders topic
    → Foreign key lookup fails
  Solution:
    → Debezium 2.0+: transaction metadata support
    → Groups events by transaction ID
    → Consumer waits for transaction boundary marker
    → Alternative: eventual consistency — retry with backoff

PITFALL 5: DUPLICATE EVENTS ON CONNECTOR RESTART
  Problem: Debezium commits offset to Kafka Connect periodically (not per event)
  Result: On restart, replays events since last committed offset
  Timeline:
    → Debezium processed LSN 100–150, committed at LSN 100
    → Crash at LSN 150
    → Restart: replays LSN 100–150 → duplicates
  Solution:
    → Consumers MUST be idempotent (as stated in Should Not Do)
    → Use source LSN as deduplication key
    → event.source.lsn is monotonically increasing and unique

PITFALL 6: OUTBOX TABLE GROWS UNBOUNDED
  Problem: Outbox rows accumulate forever
  Result: Table bloat, slow queries, backup size explosion
  Solution:
    → Delete outbox rows AFTER Debezium captures them
    → Use Debezium outbox table heartbeat to track consumed position
    → Scheduled cleanup: DELETE FROM outbox WHERE created_at < NOW() - INTERVAL '1 day'
    → Or use table partitioning and drop old partitions

📈 SCALABILITY CONSIDERATIONS

SINGLE CONNECTOR THROUGHPUT:
  → Debezium PostgreSQL: ~10,000–30,000 events/second per task
  → Bottleneck: WAL decoding speed, not Kafka producing
  → Single replication slot = single reader thread
  → PostgreSQL does NOT support parallel logical replication from one slot

SCALING STRATEGIES:

  1. MULTIPLE CONNECTORS FOR MULTIPLE DATABASES
     → Each microservice database gets its own Debezium connector
     → Independent replication slots
     → Independent Kafka topics
     → Scales linearly with number of databases

     ┌───────────┐     ┌──────────────┐     ┌─────────┐
     │ orders_db │────▶│ connector-1  │────▶│ Kafka   │
     └───────────┘     └──────────────┘     │         │
     ┌───────────┐     ┌──────────────┐     │         │
     │ users_db  │────▶│ connector-2  │────▶│         │
     └───────────┘     └──────────────┘     │         │
     ┌───────────┐     ┌──────────────┐     │         │
     │ inventory │────▶│ connector-3  │────▶│         │
     └───────────┘     └──────────────┘     └─────────┘

  2. TABLE FILTERING
     → Don't capture ALL tables — only business-critical ones
     → table.include.list = "public.orders,public.customers"
     → Reduces event volume and WAL processing load

  3. COLUMN FILTERING
     → Don't capture ALL columns — exclude large BLOBs

🔍 INTERPRETABILITY CONSIDERATIONS

CDC EVENT READABILITY:
  → Every event contains complete context:
    • What table changed (source.table)
    • What operation (op: c/u/d/r)
    • Before state (for updates and deletes)
    • After state (for inserts and updates)
    • Database transaction ID (source.txId)
    • Exact WAL position (source.lsn)
    • Database timestamp (source.ts_ms)
    • Debezium processing timestamp (ts_ms)

LATENCY TRACKING:
  → source.ts_ms = when database wrote the change
  → ts_ms = when Debezium processed the event
  → Consumer processing time = when consumer reads event
  → End-to-end latency = consumer time - source.ts_ms
  → Typical: 50–200ms end-to-end

# ============================================================
# FILE: cdc_consumer.py
# PURPOSE: Consume and display CDC events from Debezium topics
#          Shows real-time database changes as Kafka events
# ============================================================

from confluent_kafka import Consumer, KafkaError
import json
import time
import sys


class CDCEventConsumer:
    """
    Consumes Debezium CDC events and displays them in a readable format.
    Demonstrates how downstream services would process database changes.
    """

    def __init__(self, topics=None, broker='localhost:9092'):
        self.topics = topics or [
            'megastore.public.orders',
            'megastore.public.customers',
            'megastore.public.products'
        ]

        self.consumer = Consumer({
            'bootstrap.servers': broker,
            'group.id': 'cdc-viewer-group',
            'auto.offset.reset': 'earliest',
            'enable.auto.commit': True
        })
        self.consumer.subscribe(self.topics)

        # Statistics
        self.stats = {
            'total': 0,
            'by_operation': {'c': 0, 'u': 0, 'd': 0, 'r': 0},
            'by_table': {}
        }

        self.running = True
        # For idempotency — track processed LSNs
        self.processed_lsns = set()

    def _format_operation(self, op: str) -> str:
        """Human-readable operation type"""
        ops = {
            'c': '🆕 INSERT',
            'u': '✏️  UPDATE',
            'd': '🗑️  DELETE',
            'r': '📸 SNAPSHOT'
        }
        return ops.get(op, f'❓ UNKNOWN ({op})')

    def _extract_table(self, source: dict) -> str:
        """Extract table name from source metadata"""
        schema = source.get('schema', '?')
        table = source.get('table', '?')
        return f"{schema}.{table}"

    def _format_event(self, value: dict) -> str:
        """Format a CDC event for display"""
        payload = value if 'source' in value else value.get('payload', value)

        op = payload.get('op', '?')
        source = payload.get('source', {})
        before = payload.get('before')
        after = payload.get('after')

        table = self._extract_table(source)
        lsn = source.get('lsn', '?')
        tx_id = source.get('txId', '?')
        db_ts = source.get('ts_ms', 0)

        # Calculate latency
        debezium_ts = payload.get('ts_ms', 0)
        now_ms = int(time.time() * 1000)
        if db_ts > 0 and debezium_ts > 0:
            capture_latency = debezium_ts - db_ts
            e2e_latency = now_ms - db_ts
        else:
            capture_latency = '?'
            e2e_latency = '?'

        lines = [
            f"\n  {'='*60}",
            f"  {self._format_operation(op)} on {table}",
            f"  LSN: {lsn} | TxID: {tx_id} | Capture Latency: {capture_latency}ms | E2E: {e2e_latency}ms",
        ]

        if before:
            lines.append(f"  BEFORE: {json.dumps(before, default=str)}")
        if after:
            lines.append(f"  AFTER:  {json.dumps(after, default=str)}")

        # For updates: highlight what changed
        if op == 'u' and before and after:
            changes = []
            for key in after:
                if key in before and before[key] != after[key]:
                    changes.append(f"    {key}: {before[key]} → {after[key]}")
            if changes:
                lines.append(f"  CHANGES:")
                lines.extend(changes)

        return '\n'.join(lines)

    def consume(self):
        """Main consumption loop"""
        print("\n" + "=" * 60)
        print("👁️  CDC EVENT CONSUMER — WATCHING DATABASE CHANGES")
        print(f"   Topics: {', '.join(self.topics)}")
        print("   Waiting for events... (make changes in another terminal)")
        print("=" * 60)

        while self.running:
            msg = self.consumer.poll(timeout=1.0)
            if msg is None:
                continue

            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    continue
                print(f"  ❌ Error: {msg.error()}")
                continue

            try:
                topic = msg.topic()
                key_raw = msg.key()
                value_raw = msg.value()

                # Handle tombstone (null value after delete)
                if value_raw is None:
                    key_str = key_raw.decode('utf-8') if key_raw else '?'
                    print(f"\n  ⚰️  TOMBSTONE for key: {key_str} on {topic}")
                    print(f"     (Enables log compaction to remove this key)")
                    continue

                key = json.loads(key_raw.decode('utf-8')) if key_raw else None
                value = json.loads(value_raw.decode('utf-8'))

                # Idempotency check using LSN
                payload = value if 'source' in value else value.get('payload', value)
                source = payload.get('source', {})
                lsn = source.get('lsn')
                if lsn and lsn in self.processed_lsns:
                    print(f"  ⏭️  Duplicate LSN {lsn} — skipping (idempotency)")
                    continue
                if lsn:
                    self.processed_lsns.add(lsn)

                # Display formatted event
                print(self._format_event(value))

                # Update statistics
                op = payload.get('op', '?')
                table = self._extract_table(source)
                self.stats['total'] += 1
                self.stats['by_operation'][op] = self.stats['by_operation'].get(op, 0) + 1
                self.stats['by_table'][table] = self.stats['by_table'].get(table, 0) + 1

            except Exception as e:
                print(f"  ❌ Parse error: {e}")
                import traceback
                traceback.print_exc()

    def print_stats(self):
        """Print consumption statistics"""
        print(f"\n  📊 CDC Consumer Statistics:")
        print(f"     Total events: {self.stats['total']}")
        print(f"     By operation: {json.dumps(self.stats['by_operation'])}")
        print(f"     By table:     {json.dumps(self.stats['by_table'])}")

    def shutdown(self):
        self.running = False
        self.print_stats()
        self.consumer.close()
        print("  🛑 CDC Consumer stopped")


if __name__ == '__main__':
    # Optional: specify topics as arguments
    topics = sys.argv[1:] if len(sys.argv) > 1 else None
    consumer = CDCEventConsumer(topics=topics)

    try:
        consumer.consume()
    except KeyboardInterrupt:
        consumer.shutdown()

    """
    Generate continuous database changes.
    Run this in one terminal while cdc_consumer.py watches in another.
    """

    weighted_ops = []
    for name, func, weight in operations:
        weighted_ops.extend([(name, func)] * weight)

    print("\n" + "=" * 60)
    print(f"🔄 CONTINUOUS DATABASE SIMULATION")
    print(f"   Interval: {interval_seconds}s | Duration: {duration_seconds}s")
    print(f"   Operations: {len(operations)} types, weighted")

        try:

            print(f"\n  [{count}] Operation: {name}")

            # Periodic outbox cleanup

            print(f"  ❌ Database error: {e}")

            time.sleep(1)
        except Exception as e:
            print(f"  ❌ Error: {e}")
            import traceback
            traceback.print_exc()
            time.sleep(1)

    elapsed = time.time() - start
    print(f"\n  ✅ Simulation complete: {count} operations in {elapsed:.1f}s")
    conn.close()

# ============================================================
# FILE: outbox_consumer.py
# PURPOSE: Consume events from the outbox table CDC topic
#          This is how downstream services receive reliable events
#          WITHOUT the source service ever touching Kafka
#
# CONNECTS TO PROJECT 36 (Saga):
#   → Saga orchestrator would consume from this topic
#   → Instead of services publishing to reply topics directly
#   → Services write to outbox table → Debezium → Kafka → Orchestrator
# ============================================================

from confluent_kafka import Consumer, KafkaError
import json
import time


class OutboxEventProcessor:
    """
    Processes events from the outbox CDC topic.
    
    In production, this would be:
    → Saga Orchestrator (Project 36) processing service replies
    → Analytics pipeline ingesting business events
    → Notification service sending emails on order status changes
    → Search service updating indexes
    
    The KEY insight: the Order Service never imported confluent_kafka.
    It just wrote to its own database. Debezium did the rest.
    """

    def __init__(self, broker='localhost:9092'):
        self.consumer = Consumer({
            'bootstrap.servers': broker,
            'group.id': 'outbox-processor-group',
            'auto.offset.reset': 'earliest',
            'enable.auto.commit': True
        })
        # Subscribe to the outbox CDC topic
        self.consumer.subscribe(['megastore.public.outbox'])

        self.event_handlers = {
            'OrderCreated': self._handle_order_created,
            'OrderStatusChanged': self._handle_order_status_changed,
            'HEARTBEAT': self._handle_heartbeat,
        }

        self.processed_events = set()
        self.running = True

        print("=" * 60)
        print("📦 OUTBOX EVENT PROCESSOR")
        print("   Consuming: megastore.public.outbox")
        print("   This simulates downstream services receiving events")
        print("   WITHOUT source services ever touching Kafka")
        print("=" * 60)

    def _extract_outbox_event(self, cdc_value: dict) -> dict:
        """
        Extract the business event from the CDC wrapper.
        
        CDC event structure for outbox table:
        {
            "before": null,
            "after": {
                "id": "uuid-...",
                "aggregate_type": "Order",
                "aggregate_id": "42",
                "event_type": "OrderCreated",
                "payload": "{\"order_id\": 42, ...}",
                "created_at": 1689412345123
            },
            "op": "c",
            "source": { ... }
        }
        
        We extract the business event from the "after" field.
        """
        payload = cdc_value if 'op' in cdc_value else cdc_value.get('payload', cdc_value)
        op = payload.get('op', '?')

        # Only process INSERTs to outbox (new events)
        # Ignore deletes (cleanup) and snapshot reads
        if op not in ('c', 'r'):
            return None

        after = payload.get('after')
        if not after:
            return None

        # Parse the business event from outbox row
        event = {
            'outbox_id': after.get('id'),
            'aggregate_type': after.get('aggregate_type'),
            'aggregate_id': after.get('aggregate_id'),
            'event_type': after.get('event_type'),
            'created_at': after.get('created_at'),
        }

        # Parse the JSON payload
        raw_payload = after.get('payload')
        if isinstance(raw_payload, str):
            try:
                event['data'] = json.loads(raw_payload)
            except json.JSONDecodeError:
                event['data'] = raw_payload
        elif isinstance(raw_payload, dict):
            event['data'] = raw_payload
        else:
            event['data'] = {}

        return event

    def _handle_order_created(self, event: dict):
        """
        Handle OrderCreated event.
        In production: notify inventory, analytics, search services.
        For Saga (Project 36): this would be the service reply.
        """
        data = event.get('data', {})
        order_id = data.get('order_id')
        customer_id = data.get('customer_id')
        product = data.get('product_name', '?')
        total = data.get('total_amount', 0)

        print(f"\n  📬 ORDER CREATED EVENT RECEIVED")
        print(f"     Order ID:    #{order_id}")
        print(f"     Customer:    {customer_id}")
        print(f"     Product:     {product}")
        print(f"     Total:       ${total:.2f}")
        print(f"     → Inventory Service: decrement stock")
        print(f"     → Analytics Service: update dashboard")
        print(f"     → Saga Orchestrator: process as service reply")

    def _handle_order_status_changed(self, event: dict):
        """
        Handle OrderStatusChanged event.
        In production: send notification, update tracking, etc.
        """
        data = event.get('data', {})
        order_id = data.get('order_id')
        old_status = data.get('previous_status')
        new_status = data.get('new_status')

        print(f"\n  📬 ORDER STATUS CHANGED EVENT RECEIVED")
        print(f"     Order ID:      #{order_id}")
        print(f"     Status Change: {old_status} → {new_status}")

        if new_status == 'SHIPPED':
            print(f"     → Notification: Send shipping confirmation email")
        elif new_status == 'DELIVERED':
            print(f"     → Notification: Send delivery confirmation")
            print(f"     → Review Service: Request product review")

    def _handle_heartbeat(self, event: dict):
        """Heartbeat events keep the replication slot alive"""
        pass  # Silently ignore heartbeats

    def consume(self):
        """Main event loop"""
        print("\n👂 Listening for outbox events...\n")

        while self.running:
            msg = self.consumer.poll(timeout=1.0)
            if msg is None:
                continue

            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    continue
                print(f"  ❌ Error: {msg.error()}")
                continue

            try:
                # Handle tombstones
                if msg.value() is None:
                    continue

                value = json.loads(msg.value().decode('utf-8'))

                # Extract business event from CDC wrapper
                event = self._extract_outbox_event(value)
                if not event:
                    continue

                # Idempotency check
                outbox_id = event.get('outbox_id')
                if outbox_id in self.processed_events:
                    print(f"  ⏭️  Duplicate outbox event {outbox_id} — skipping")
                    continue
                if outbox_id:
                    self.processed_events.add(outbox_id)

                # Route to appropriate handler
                event_type = event.get('event_type', 'UNKNOWN')
                handler = self.event_handlers.get(event_type)

                if handler:
                    handler(event)
                else:
                    print(f"\n  ❓ Unknown event type: {event_type}")
                    print(f"     Data: {json.dumps(event, indent=2, default=str)}")

            except Exception as e:
                print(f"  ❌ Processing error: {e}")
                import traceback
                traceback.print_exc()

    def shutdown(self):
        self.running = False
        self.consumer.close()
        print("\n  🛑 Outbox processor stopped")
        print(f"     Processed {len(self.processed_events)} unique events")


if __name__ == '__main__':
    processor = OutboxEventProcessor()
    try:
        processor.consume()
    except KeyboardInterrupt:
        processor.shutdown()

# ============================================================
# FILE: monitor_replication.py
# PURPOSE: Monitor PostgreSQL replication slots and WAL usage
#          Prevent the #1 Debezium pitfall: WAL disk explosion
# BUILDS ON: Project 28 (Monitoring with JMX)
# ============================================================

import psycopg2
import time
import json
import sys


class ReplicationMonitor:
    """
    Monitors PostgreSQL replication slots used by Debezium.
    
    CRITICAL MONITORING:
    → Replication slot lag = how far Debezium is behind
    → WAL disk usage = risk of disk filling up
    → Slot activity = is Debezium actively consuming?
    """

    def __init__(self):
        self.conn = psycopg2.connect(
            host='localhost',
            port=5432,
            dbname='megastore',
            user='postgres',
            password='postgres'
        )
        self.conn.autocommit = True

    def check_wal_level(self):
        """Verify WAL level is logical"""
        cursor = self.conn.cursor()
        cursor.execute("SHOW wal_level;")
        level = cursor.fetchone()[0]
        status = "✅" if level == 'logical' else "❌"
        print(f"  {status} WAL Level: {level}")
        return level == 'logical'

    def check_replication_slots(self):
        """
        Check all replication slots.
        KEY METRICS:
        → active: Is Debezium connected and consuming?
        → restart_lsn: Where Debezium will resume from
        → confirmed_flush_lsn: Last position Debezium confirmed
        → pg_current_wal_lsn(): Current WAL write position
        → Difference = replication lag
        """
        cursor = self.conn.cursor()

        cursor.execute("""
            SELECT
                slot_name,
                plugin,
                slot_type,
                active,
                active_pid,
                restart_lsn,
                confirmed_flush_lsn,
                pg_current_wal_lsn() AS current_lsn,
                pg_wal_lsn_diff(
                    pg_current_wal_lsn(),
                    COALESCE(confirmed_flush_lsn, restart_lsn)
                ) AS lag_bytes
            FROM pg_replication_slots;
        """)

        columns = [desc[0] for desc in cursor.description]
        rows = cursor.fetchall()

        if not rows:
            print("  ⚠️  No replication slots found")
            print("     Debezium connector may not have started yet")
            return []

        print(f"\n  📊 REPLICATION SLOTS ({len(rows)} found):")
        print(f"  {'─' * 70}")

        results = []
        for row in rows:
            slot = dict(zip(columns, row))
            results.append(slot)

            active_icon = "🟢" if slot['active'] else "🔴"
            lag_mb = slot['lag_bytes'] / (1024 * 1024) if slot['lag_bytes'] else 0

            print(f"  {active_icon} Slot: {slot['slot_name']}")
            print(f"     Plugin:          {slot['plugin']}")
            print(f"     Active:          {slot['active']} (PID: {slot['active_pid']})")
            print(f"     Restart LSN:     {slot['restart_lsn']}")
            print(f"     Confirmed LSN:   {slot['confirmed_flush_lsn']}")
            print(f"     Current WAL LSN: {slot['current_lsn']}")
            print(f"     Lag:             {slot['lag_bytes']} bytes ({lag_mb:.2f} MB)")

            # Alert thresholds
            if lag_mb > 100:
                print(f"     🚨 CRITICAL: Lag > 100MB — risk of WAL disk explosion!")
            elif lag_mb > 10:
                print(f"     ⚠️  WARNING: Lag > 10MB — Debezium may be falling behind")
            else:
                print(f"     ✅ Healthy: Lag within normal range")

        return results

    def check_wal_size(self):
        """Check total WAL disk usage"""
        cursor = self.conn.cursor()

        cursor.execute("""
            SELECT
                pg_wal_lsn_diff(pg_current_wal_lsn(), '0/0') AS total_wal_bytes,
                pg_size_pretty(
                    pg_wal_lsn_diff(pg_current_wal_lsn(), '0/0')
                ) AS total_wal_pretty
        """)

        row = cursor.fetchone()
        if row:
            print(f"\n  💾 WAL Size: {row[1]}")

    def check_table_stats(self):
        """Check CDC-relevant table statistics"""
        cursor = self.conn.cursor()

        cursor.execute("""
            SELECT
                schemaname || '.' || relname AS table_name,
                n_tup_ins AS inserts,
                n_tup_upd AS updates,
                n_tup_del AS deletes,
                n_live_tup AS live_rows
            FROM pg_stat_user_tables
            WHERE schemaname = 'public'
            ORDER BY relname;
        """)

        print(f"\n  📋 TABLE STATISTICS:")
        print(f"  {'Table':<25s} {'Inserts':<10s} {'Updates':<10s} {'Deletes':<10s} {'Live Rows':<10s}")
        print(f"  {'─'*25} {'─'*10} {'─'*10} {'─'*10} {'─'*10}")

        for row in cursor.fetchall():
            print(f"  {row[0]:<25s} {row[1]:<10d} {row[2]:<10d} {row[3]:<10d} {row[4]:<10d}")

    def check_outbox_size(self):
        """Monitor outbox table size — should stay small"""
        cursor = self.conn.cursor()
        cursor.execute("SELECT COUNT(*) FROM outbox;")

    def drop_inactive_slot(self, slot_name):
        """
        Emergency: Drop an inactive replication slot to prevent WAL explosion.
        WARNING: This will cause Debezium to lose its position.
        It will need to re-snapshot on next start.
        """
        cursor = self.conn.cursor()

        cursor.execute(
            "SELECT active FROM pg_replication_slots WHERE slot_name = %s",
            (slot_name,)
        )
        row = cursor.fetchone()

        if not row:
            print(f"  ❌ Slot '{slot_name}' not found")
            return

        if row[0]:
            print(f"  ⚠️  Slot '{slot_name}' is ACTIVE — cannot drop while in use")
            print(f"     Stop Debezium connector first")
            return

        cursor.execute(
            "SELECT pg_drop_replication_slot(%s);",
            (slot_name,)
        )
        print(f"  ✅ Dropped replication slot: {slot_name}")
        print(f"     ⚠️  Debezium will need to re-snapshot on next start")
