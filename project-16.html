🔵 HOW TO APPROACH IT

Step 1: Understand WHY JSON falls short at scale
  → No schema enforcement: producer can send anything
  → Verbose: field names repeated in every message
  → No compatibility checking: any change can break consumers
  → Slow parsing: text-based, must scan for field boundaries
  → No centralized schema documentation

Step 2: Understand Avro as a BINARY serialization format
  → Schema defines the structure (like a database table definition)
  → Data is encoded WITHOUT field names (just values in order)
  → Reader and writer can have DIFFERENT but compatible schemas
  → Schema is NOT embedded in every message (stored in Registry)

Step 3: Understand Schema Registry as a CENTRAL SCHEMA STORE
  → Stores Avro (and Protobuf, JSON Schema) schemas
  → Assigns unique ID to each schema version
  → Enforces compatibility rules between versions
  → Provides REST API for schema lookup
  → Producer registers schema → gets ID
  → Consumer looks up schema by ID → knows how to deserialize

Step 4: Connect to what we already know
  → Project 7: JSON serialization — now we upgrade to Avro
  → Project 13: Envelope pattern — becomes a formal Avro schema
  → Project 15: Connect converters — AvroConverter replaces JsonConverter
  → Project 12: Python producer/consumer — now uses SerializingProducer

🟡 WHAT COULD BE DONE

OPTION A: Stay with JSON + add JSON Schema validation
  → Use JSON Schema to define message structure
  → Validate at producer side before sending
  → Problem: validation is client-side only (no central enforcement)
  → Problem: JSON is still verbose and slow
  → Problem: no compatibility checking between versions

OPTION B: Use Protocol Buffers (Protobuf)
  → Binary format, schema-based, fast
  → Requires .proto files compiled to language-specific code
  → Schema Registry supports Protobuf
  → Good choice if your team already uses gRPC
  → Slightly more complex than Avro for Kafka use cases

OPTION C: Use Apache Avro with Schema Registry
  → Binary format, compact, fast
  → Schema defined in JSON (easy to read and write)
  → Schema Registry provides central governance
  → Native Kafka integration (Confluent ecosystem)
  → Schema evolution with compatibility rules
  → ✅ THIS IS THE INDUSTRY STANDARD FOR KAFKA

OPTION D: Use JSON Schema with Schema Registry
  → Keep JSON format but add schema enforcement
  → Schema Registry supports JSON Schema
  → Less compact than Avro but more readable on wire
  → Good migration path from unstructured JSON
  → Suitable when binary format is not required

✅ WHAT SHOULD DO

1. DEFINE SCHEMAS BEFORE WRITING CODE
   → Schema is the CONTRACT between producer and consumer
   → Design it like a database table definition
   → Include documentation (doc field in Avro)
   → Version from the start (even if you think it won't change)

2. USE SCHEMA REGISTRY FOR ALL AVRO TOPICS
   → Every Avro-serialized topic MUST have schemas in Registry
   → Without Registry, consumers can't find the schema to deserialize
   → Registry is the SINGLE SOURCE OF TRUTH for message structure

3. SET COMPATIBILITY MODE PER SUBJECT
   → BACKWARD (default): new schema can read old data
   → FORWARD: old schema can read new data
   → FULL: both backward and forward compatible
   → NONE: no compatibility checking (dangerous)
   → BACKWARD is the safest default

4. USE UNION TYPES WITH NULL FOR OPTIONAL FIELDS
   → Avro: ["null", "string"] means "nullable string"
   → Always put "null" FIRST in the union
   → Always provide a DEFAULT for new fields
   → This enables backward-compatible schema evolution

5. USE SUBJECT NAMING STRATEGY
   → TopicNameStrategy (default): {topic}-key, {topic}-value
   → RecordNameStrategy: {record-name} (share schema across topics)
   → TopicRecordNameStrategy: {topic}-{record-name}
   → Stick with TopicNameStrategy unless you have a specific reason

6. CACHE SCHEMAS ON THE CLIENT SIDE
   → confluent-kafka serializers cache schemas automatically
   → First produce: register schema → get ID → cache
   → Subsequent produces: use cached ID (no Registry call)
   → First consume: lookup schema by ID → cache
   → Subsequent consumes: use cached schema

7. USE SerializingProducer AND DeserializingConsumer
   → These handle Avro serialization/deserialization automatically
   → They manage Schema Registry communication
   → They handle the wire format (magic byte + schema ID + data)

❌ WHAT SHOULD NOT DO

1. DO NOT EMBED THE FULL SCHEMA IN EVERY MESSAGE
   → That's what JSON does — repeats field names every time
   → Avro + Schema Registry: schema stored ONCE in Registry
   → Message contains only 4-byte schema ID + binary data
   → Huge size savings at scale

2. DO NOT CHANGE FIELD TYPES WITHOUT COMPATIBILITY CHECK
   → Changing "amount" from double to string breaks ALL consumers
   → Schema Registry will REJECT incompatible changes
   → This is a FEATURE — it prevents the silent corruption from our problem

3. DO NOT REMOVE REQUIRED FIELDS
   → Removing "currency" breaks consumers that expect it
   → Instead: make it optional with a default
   → Or: add a new field and deprecate the old one

4. DO NOT USE COMPATIBILITY=NONE IN PRODUCTION
   → NONE means any schema change is allowed
   → Completely defeats the purpose of Schema Registry
   → Only use NONE for development/testing topics

5. DO NOT SKIP SCHEMA REGISTRATION
   → Producing Avro without Schema Registry = manual schema management
   → Consumers won't know which schema to use for deserialization
   → Schema Registry is NOT optional for production Avro

6. DO NOT USE AVRO FOR SIMPLE STRING MESSAGES
   → Log lines, simple text events → StringSerializer is fine
   → Avro adds overhead (Registry lookup, binary encoding)
   → Use Avro for STRUCTURED data with defined schemas

7. DO NOT REGISTER SCHEMAS MANUALLY IN PRODUCTION
   → Let the serializer auto-register on first produce
   → Or use CI/CD pipeline to register schemas before deployment
   → Manual registration via REST API → human error risk

8. DO NOT IGNORE THE doc FIELD IN AVRO SCHEMAS
   → doc is self-documentation for every field
   → Future developers (and your future self) will thank you
   → Schema Registry UI shows doc fields for discovery

⚙️ HOW IT ONLY WORKS (INTERNAL MECHANICS)

═══════════════════════════════════════════════════════════════
              AVRO ENCODING — HOW DATA BECOMES BINARY
═══════════════════════════════════════════════════════════════

Schema:
  {
    "type": "record",
    "name": "Order",
    "fields": [
      {"name": "order_id", "type": "string"},
      {"name": "amount", "type": "double"},
      {"name": "currency", "type": "string"}
    ]
  }

JSON representation (what we used in Projects 7-15):
  {"order_id": "ORD-001", "amount": 99.99, "currency": "USD"}
  
  Size: 56 bytes
  Contains: field names + values + syntax characters ({, ", :, ,)

Avro binary representation:
  [0E] [4F 52 44 2D 30 30 31]  [40 58 FF 5C 28 F5 C3]  [06] [55 53 44]
   │         │                          │                    │      │
   │    "ORD-001"                    99.99                   │   "USD"
   │    (length-prefixed)         (IEEE 754 double)          │  (3 chars)
   length=7                                               length=3
  
  Size: 18 bytes
  Contains: ONLY values (no field names, no syntax)
  Savings: 56 → 18 bytes = 68% reduction!

HOW AVRO KNOWS FIELD ORDER:
  → Fields are written in the ORDER defined in the schema
  → Reader uses the SAME schema to read in the same order
  → No field names needed in the data itself
  → This is why schema is REQUIRED for deserialization


═══════════════════════════════════════════════════════════════
              SCHEMA REGISTRY — THE CENTRAL BRAIN
═══════════════════════════════════════════════════════════════

Schema Registry is a separate service that:
  → Stores schemas (Avro, Protobuf, JSON Schema)
  → Assigns unique numeric IDs to each schema version
  → Organizes schemas into "subjects"
  → Enforces compatibility rules between versions

REST API:
  POST /subjects/{subject}/versions     → Register new schema
  GET  /schemas/ids/{id}                → Get schema by ID
  GET  /subjects/{subject}/versions     → List versions
  GET  /subjects/{subject}/versions/{v} → Get specific version
  POST /compatibility/subjects/{s}/versions/{v} → Check compatibility

Subject naming (TopicNameStrategy):
  Topic: ecommerce.orders
  Key subject:   ecommerce.orders-key
  Value subject: ecommerce.orders-value


═══════════════════════════════════════════════════════════════
              WIRE FORMAT — WHAT'S ACTUALLY IN KAFKA
═══════════════════════════════════════════════════════════════

Every Avro-serialized Kafka message has this format:

  Byte 0:     Magic byte (always 0x00)
  Bytes 1-4:  Schema ID (4-byte big-endian integer)
  Bytes 5+:   Avro binary-encoded data

  ┌──────┬──────────────────┬──────────────────────────────────┐
  │ 0x00 │ 0x00 0x00 0x00 │ Avro binary data                  │
  │      │ 0x04            │ (encoded using schema ID 4)       │
  │      │                  │                                    │
  │magic │  schema ID = 4  │  [0E][4F 52 44 2D 30 30 31]...   │
  └──────┴──────────────────┴──────────────────────────────────┘
  1 byte   4 bytes           variable length

  Total overhead: 5 bytes per message (vs hundreds in JSON field names)


PRODUCER FLOW:
  1. Producer has data: {"order_id": "ORD-001", "amount": 99.99, ...}
  2. Serializer checks: "Do I have schema ID cached for this schema?"
     → YES: use cached ID (skip to step 5)
     → NO: continue to step 3
  3. Serializer sends schema to Registry: POST /subjects/{topic}-value/versions
  4. Registry checks compatibility, stores schema, returns ID
  5. Serializer encodes: [0x00][schema_id_bytes][avro_binary_data]
  6. Producer sends bytes to Kafka

CONSUMER FLOW:
  1. Consumer reads bytes from Kafka
  2. Deserializer reads first 5 bytes: magic=0x00, schema_id=4
  3. Deserializer checks: "Do I have schema ID 4 cached?"
     → YES: use cached schema (skip to step 5)
     → NO: continue to step 4
  4. Deserializer fetches: GET /schemas/ids/4
  5. Deserializer decodes Avro binary data using the schema
  6. Consumer receives deserialized record


═══════════════════════════════════════════════════════════════
              SCHEMA EVOLUTION — THE MAGIC
═══════════════════════════════════════════════════════════════

BACKWARD COMPATIBLE EVOLUTION:

  Schema V1 (writer):                Schema V2 (reader):
  {                                  {
    "name": "Order",                   "name": "Order",
    "fields": [                        "fields": [
      {"name":"order_id",               {"name":"order_id",
       "type":"string"},                  "type":"string"},
      {"name":"amount",                 {"name":"amount",
       "type":"double"},                  "type":"double"},
      {"name":"currency",              {"name":"currency",
       "type":"string"}                   "type":"string"},
    ]                                   {"name":"discount",
  }                                      "type":["null","double"],
                                          "default":null}
                                       ]
                                     }

  When V2 reader reads V1 data:
    → order_id: read from data ✅
    → amount: read from data ✅
    → currency: read from data ✅
    → discount: NOT in data → use default (null) ✅

  This is BACKWARD compatible:
    → New reader (V2) can read old data (V1)
    → The "discount" field gets its default value

  When V1 reader reads V2 data:
    → order_id: read from data ✅
    → amount: read from data ✅
    → currency: read from data ✅
    → discount: IGNORED (V1 doesn't know about it) ✅

  This is also FORWARD compatible (V2 → V1 works too)
  Together: FULL compatibility


═══════════════════════════════════════════════════════════════
              SCHEMA RESOLUTION — READER VS WRITER SCHEMA
═══════════════════════════════════════════════════════════════

Avro uses TWO schemas during deserialization:

  WRITER SCHEMA: The schema used to encode the data
    → Stored in Schema Registry with the ID in the message
    → Tells deserializer: "the data was written in THIS format"

  READER SCHEMA: The schema the consumer EXPECTS
    → Defined in the consumer's code
    → Tells deserializer: "I want the data in THIS format"

  Avro RESOLVES the differences:
    → Field in both: read from data
    → Field in reader but not writer: use reader's default
    → Field in writer but not reader: skip (ignore)
    → Field type changed incompatibly: ERROR

  This resolution is what enables schema evolution!

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Schema Registry is unreachable — all producers/consumers fail"
  → Schema Registry is a CRITICAL dependency
  → If it's down, serializers can't register or look up schemas
  → CACHED schemas still work (existing IDs)
  → But NEW schemas can't be registered
  → FIX: Run Schema Registry with high availability (multiple instances)
  → FIX: Use schema caching to survive short outages

PITFALL 2: "Compatibility check failed — can't register new schema"
  → You tried to make an incompatible change
  → e.g., removed a required field, changed a field type
  → Schema Registry REJECTED the registration
  → This is CORRECT behavior — it prevented a breaking change
  → FIX: Make compatible changes (add optional fields with defaults)
  → FIX: If you MUST break compatibility → create a new topic

PITFALL 3: "Consumer gets 'unknown magic byte' error"
  → Consumer is trying to deserialize non-Avro data as Avro
  → The topic contains JSON messages (from before migration)
  → Avro deserializer expects magic byte 0x00 → gets '{' (0x7B)
  → FIX: Don't mix Avro and non-Avro messages in the same topic
  → FIX: For migration: use a new topic for Avro data

PITFALL 4: "New optional field always returns null even when producer sends a value"
  → Reader schema might have the wrong type for the union
  → Avro union order matters: ["null", "double"] ≠ ["double", "null"]
  → Default must match the FIRST type in the union
  → FIX: Always put "null" first: ["null", "actualType"]

PITFALL 5: "Schema ID not found in Registry"
  → Consumer reads a message with schema ID that doesn't exist
  → Could be: Registry data was lost, or pointing to wrong Registry
  → FIX: Schema Registry must have persistent storage
  → FIX: Verify SCHEMA_REGISTRY_URL is correct

PITFALL 6: "Auto-registration disabled — producer fails"
  → auto.register.schemas=false in producer config
  → Producer can't register new schemas
  → FIX: Register schemas via CI/CD before deployment
  → OR: Set auto.register.schemas=true (default)

PITFALL 7: "Performance degraded after switching to Avro"
  → First message per schema triggers Registry HTTP call
  → Subsequent messages use cache (fast)
  → If many unique schemas → many Registry calls
  → FIX: Warm up caches on startup
  → FIX: Ensure Registry has low latency (same datacenter)

SCHEMA REGISTRY SCALING:
  → Single Registry instance: handles ~1000 requests/second
  → Schema lookups are CACHED by clients (one lookup per schema ID)
  → At 500K msg/sec with 10 schemas: ~0 Registry calls (all cached)
  → Registry stores schemas in Kafka topic (_schemas — compacted!)
  → Multiple Registry instances share the same _schemas topic
  → Leader-follower: one leader handles writes, all handle reads

SCHEMA CACHE BEHAVIOR:
  → Producer: caches schema → ID mapping (register once, use forever)
  → Consumer: caches ID → schema mapping (lookup once per ID)
  → Cache is per-process, in-memory
  → Restart = cold cache = first messages hit Registry
  → At scale: pre-warm cache on startup

import requests
import json
import time

SCHEMA_REGISTRY_URL = 'http://localhost:8081'


class SchemaRegistryClient:
    """
    Python client for the Confluent Schema Registry REST API.
    
    This demonstrates the API that serializers use internally.
    In production, the confluent-kafka serializer handles this
    automatically — this is for learning and administration.
    """

    def __init__(self, url):
        self.url = url.rstrip('/')
        self.headers = {
            'Content-Type': 'application/vnd.schemaregistry.v1+json'
        }

    def _get(self, path):
        resp = requests.get(f'{self.url}{path}', headers=self.headers, timeout=10)
        if resp.status_code == 200:
            return resp.json()
        return None

    def _post(self, path, data):
        resp = requests.post(f'{self.url}{path}', headers=self.headers,
                             json=data, timeout=10)
        return resp.status_code, resp.json()

    def _put(self, path, data):
        resp = requests.put(f'{self.url}{path}', headers=self.headers,
                            json=data, timeout=10)
        return resp.status_code, resp.json()

    # ── Subject Management ──

    def list_subjects(self):
        return self._get('/subjects') or []

    def get_versions(self, subject):
        return self._get(f'/subjects/{subject}/versions') or []

    def get_schema_by_version(self, subject, version='latest'):
        return self._get(f'/subjects/{subject}/versions/{version}')

    def delete_subject(self, subject):
        resp = requests.delete(f'{self.url}/subjects/{subject}',
                               headers=self.headers, timeout=10)
        return resp.status_code

    # ── Schema Registration ──

    def register_schema(self, subject, schema_dict):
        """
        Register a new schema version under a subject.
        
        Returns: schema ID (globally unique)
        """
        payload = {'schema': json.dumps(schema_dict)}
        status, result = self._post(f'/subjects/{subject}/versions', payload)

        if status == 200:
            return result.get('id')
        else:
            print(f"  ❌ Registration failed ({status}): {result}")
            return None

    # ── Schema Lookup ──

    def get_schema_by_id(self, schema_id):
        """
        Get schema by its global ID.
        This is what the consumer deserializer calls.
        """
        return self._get(f'/schemas/ids/{schema_id}')

    # ── Compatibility ──

    def get_compatibility(self, subject=None):
        if subject:
            result = self._get(f'/config/{subject}')
        else:
            result = self._get('/config')
        return result.get('compatibilityLevel') if result else None

    def set_compatibility(self, subject, level):
        """
        Set compatibility level for a subject.
        Levels: BACKWARD, FORWARD, FULL, NONE,
                BACKWARD_TRANSITIVE, FORWARD_TRANSITIVE, FULL_TRANSITIVE
        """
        status, result = self._put(f'/config/{subject}',
                                    {'compatibility': level})
        return status == 200

    def test_compatibility(self, subject, schema_dict, version='latest'):
        """
        Test if a new schema is compatible with existing versions.
        Returns True if compatible.
        """
        payload = {'schema': json.dumps(schema_dict)}
        status, result = self._post(
            f'/compatibility/subjects/{subject}/versions/{version}',
            payload
        )
        return result.get('is_compatible', False)

class AvroOrderProducer:
    """
    Produces order events serialized with Avro.
    
    KEY DIFFERENCES from JSON producer (Project 12):
      → Uses SerializingProducer instead of Producer
      → Uses AvroSerializer instead of manual json.dumps()
      → Schema is registered automatically on first produce
      → Messages are binary (not human-readable JSON)
      → Schema ID is prepended to every message (5-byte header)
    """

    def __init__(self, bootstrap_servers, schema_registry_url):
        self.running = True
        self.stats = defaultdict(int)

        # ── Schema Registry client ──
        schema_registry_conf = {'url': schema_registry_url}
        schema_registry_client = SchemaRegistryClient(schema_registry_conf)

        # ── Avro serializer for message values ──
        # This handles:
        #   → Schema registration (auto)
        #   → Avro binary encoding
        #   → Wire format (magic byte + schema ID + data)
        avro_serializer = AvroSerializer(
            schema_registry_client,
            json.dumps(ORDER_SCHEMA_V1),      # Schema as JSON string
            to_dict=self._order_to_dict,       # Convert object → dict
            conf={
                'auto.register.schemas': True,  # Auto-register on first use
            }
        )

        # ── String serializer for message keys ──
        string_serializer = StringSerializer('utf_8')

        # ── SerializingProducer combines producer + serializers ──
        self.producer = SerializingProducer({
            'bootstrap.servers': bootstrap_servers,
            'acks': 'all',
            'enable.idempotence': True,
            'key.serializer': string_serializer,
            'value.serializer': avro_serializer,
        })

        signal.signal(signal.SIGINT, lambda s, f: setattr(self, 'running', False))

    def _order_to_dict(self, order, ctx):
        """
        Convert an order object to a dict matching the Avro schema.
        
        This function is called by AvroSerializer before encoding.
        The dict MUST match the schema field names and types exactly.
        
        ctx: SerializationContext (contains topic, headers, field)
        """
        return order  # Already a dict in our case

    def _delivery_cb(self, err, msg):
        if err:
            self.stats['errors'] += 1
            print(f"  ❌ Delivery failed: {err}")
        else:
            self.stats['delivered'] += 1

    def produce_order(self, order_data):
        """
        Produce an Avro-serialized order event.
        
        IMPORTANT: The data dict must match the Avro schema exactly.
        Wrong types → AvroSerializer raises an exception.
        Missing required fields → AvroSerializer raises an exception.
        This is the ENFORCEMENT that JSON lacked!
        """
        key = order_data['order_id']

        try:
            self.producer.produce(
                topic='avro.orders',
                key=key,
                value=order_data,
                on_delivery=self._delivery_cb,
            )
            self.stats['produced'] += 1
            self.producer.poll(0)

        except ValueError as e:
            # ── Schema validation error! ──
            # This is what PREVENTS the incidents from our problem statement
            self.stats['schema_errors'] += 1
            print(f"  🚨 SCHEMA VALIDATION ERROR: {e}")
            print(f"     Data: {json.dumps(order_data)[:100]}...")

        except BufferError:
            self.stats['buffer_errors'] += 1
            self.producer.poll(1.0)

    def shutdown(self):
        remaining = self.producer.flush(timeout=15)
        print(f"\n  📊 Avro Producer Stats:")
        for k, v in sorted(self.stats.items()):
            print(f"     {k}: {v}")
        if remaining > 0:
            print(f"  ⚠️  {remaining} unflushed")

# PURPOSE: Consume Avro-serialized messages from Kafka

# BUILDS ON: Project 12 (IoTDashboardConsumer → now with Avro)

from confluent_kafka import DeserializingConsumer
from confluent_kafka.serialization import StringDeserializer
from confluent_kafka.schema_registry import SchemaRegistryClient
from confluent_kafka.schema_registry.avro import AvroDeserializer
from confluent_kafka import KafkaError
import json
import time
import signal
from collections import defaultdict

from schema_definitions_01 import ORDER_SCHEMA_V1

BOOTSTRAP = 'localhost:9092,localhost:9093'
SCHEMA_REGISTRY_URL = 'http://localhost:8081'


class AvroOrderConsumer:
    """
    Consumes Avro-serialized order events.
    
    KEY DIFFERENCES from JSON consumer (Project 12):
      → Uses DeserializingConsumer instead of Consumer
      → Uses AvroDeserializer instead of manual json.loads()
      → Schema is looked up from Registry by ID in message header
      → Deserialized data arrives as a dict (same as JSON consumer)
      → Schema evolution is handled AUTOMATICALLY
    """

    def __init__(self, bootstrap_servers, schema_registry_url, group_id):
        self.running = True
        self.stats = defaultdict(int)

        # ── Schema Registry client ──
        schema_registry_conf = {'url': schema_registry_url}
        schema_registry_client = SchemaRegistryClient(schema_registry_conf)

        # ── Avro deserializer ──
        # Optional: provide reader schema for schema evolution
        # If not provided, uses the writer's schema (from Registry)
        avro_deserializer = AvroDeserializer(
            schema_registry_client,
            json.dumps(ORDER_SCHEMA_V1),      # Reader schema (optional)
            from_dict=self._dict_to_order,     # Convert dict → object
        )

        string_deserializer = StringDeserializer('utf_8')

        # ── DeserializingConsumer combines consumer + deserializers ──
        self.consumer = DeserializingConsumer({
            'bootstrap.servers': bootstrap_servers,
            'group.id': group_id,
            'auto.offset.reset': 'earliest',
            'enable.auto.commit': False,
            'key.deserializer': string_deserializer,
            'value.deserializer': avro_deserializer,
        })

        signal.signal(signal.SIGINT, lambda s, f: setattr(self, 'running', False))

    def _dict_to_order(self, data, ctx):
        """
        Convert deserialized dict to application object.
        
        AvroDeserializer calls this after Avro → dict conversion.
        You can return any object (dict, dataclass, custom class).
        
        For simplicity, we return the dict as-is.
        """
        return data

    def consume_orders(self, topic='avro.orders', max_messages=100):
        """Consume and process Avro order events."""

        self.consumer.subscribe([topic])

        print(f"  📡 Consuming Avro orders from '{topic}'...\n")

        processed = []
        messages_since_commit = 0
        COMMIT_BATCH = 20
        empty_polls = 0

        while self.running and len(processed) < max_messages and empty_polls < 20:
            try:
                msg = self.consumer.poll(timeout=1.0)
            except Exception as e:
                # ── Deserialization errors surface here ──
                self.stats['deserialize_errors'] += 1
                print(f"  ❌ Deserialization error: {e}")
                continue

            if msg is None:
                empty_polls += 1
                if messages_since_commit > 0:
                    self.consumer.commit(asynchronous=False)
                    messages_since_commit = 0
                continue

            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    empty_polls += 1
                    continue
                self.stats['consume_errors'] += 1
                continue

            empty_polls = 0

            # ══════════════════════════════════════════════
            # msg.value() is ALREADY a Python dict!
            # AvroDeserializer handled:
            #   → Reading magic byte + schema ID
            #   → Looking up schema from Registry
            #   → Decoding Avro binary → Python dict
            # ══════════════════════════════════════════════
            key = msg.key()
            order = msg.value()

            if order is None:
                continue

            self.stats['consumed'] += 1
            messages_since_commit += 1

            # ── Process the order (already typed correctly!) ──
            order_id = order.get('order_id', 'N/A')
            customer = order.get('customer_id', 'N/A')
            total = order.get('total_amount', 0)
            currency = order.get('currency', 'USD')
            items = order.get('items', [])
            item_count = len(items)

            processed.append({
                'order_id': order_id,
                'customer_id': customer,
                'total_amount': total,
                'currency': currency,
                'item_count': item_count,
                'partition': msg.partition(),
                'offset': msg.offset(),
            })

            if len(processed) % 10 == 0:
                print(f"  📦 [{len(processed)}] order={order_id}, "
                      f"customer={customer}, "
                      f"total={currency} {total:.2f}, "
                      f"items={item_count}, "
                      f"P{msg.partition()}:{msg.offset()}")

            if messages_since_commit >= COMMIT_BATCH:
                self.consumer.commit(asynchronous=False)
                self.stats['commits'] += 1
                messages_since_commit = 0

        # ── Final commit ──
        if messages_since_commit > 0:
            self.consumer.commit(asynchronous=False)

        self.consumer.close()

        return processed

    def display_stats(self, processed):
        """Display consumption statistics."""
        print(f"\n  ┌────────────────────────────────────────────────────┐")
        print(f"  │  AVRO CONSUMER STATISTICS                            │")
        print(f"  ├────────────────────────────────────────────────────┤")
        print(f"  │  Messages consumed:    {self.stats.get('consumed', 0):>6}                     │")
        print(f"  │  Deserialize errors:   {self.stats.get('deserialize_errors', 0):>6}"
              f"                     │")
        print(f"  │  Commits:              {self.stats.get('commits', 0):>6}                     │")

        if processed:
            total_revenue = sum(p['total_amount'] for p in processed)
            currencies = defaultdict(float)
            for p in processed:
                currencies[p['currency']] += p['total_amount']

            print(f"  ├────────────────────────────────────────────────────┤")
            print(f"  │  BUSINESS METRICS:                                  │")
            print(f"  │  Total orders:         {len(processed):>6}                     │")

            for curr, amount in sorted(currencies.items()):
                print(f"  │  Revenue ({curr}):      {amount:>10.2f}                 │")

            unique_customers = len(set(p['customer_id'] for p in processed))
            print(f"  │  Unique customers:     {unique_customers:>6}                     │")

        print(f"  └────────────────────────────────────────────────────┘")

from confluent_kafka import Consumer, KafkaError
import struct
import json
import time
import requests
from collections import defaultdict

BOOTSTRAP = 'localhost:9092,localhost:9093'
SCHEMA_REGISTRY_URL = 'http://localhost:8081'


class WireFormatInspector:
    """
    Reads RAW bytes from Kafka (no Avro deserializer)
    to inspect the wire format of Avro messages.
    """

    def __init__(self, bootstrap_servers, schema_registry_url):
        self.bootstrap_servers = bootstrap_servers
        self.sr_url = schema_registry_url

        # ── Use RAW consumer (no deserializer) to see raw bytes ──
        self.consumer = Consumer({
            'bootstrap.servers': bootstrap_servers,
            'group.id': f'wire-inspector-{int(time.time())}',
            'auto.offset.reset': 'earliest',
            'enable.auto.commit': False,
        })

    def inspect_topic(self, topic, max_messages=20):
        """Read raw bytes and decode the wire format manually."""

        self.consumer.subscribe([topic])

        print(f"  📡 Inspecting raw wire format of '{topic}'...\n")

        messages = []
        empty_polls = 0
        schema_cache = {}

        while empty_polls < 15 and len(messages) < max_messages:
            msg = self.consumer.poll(timeout=1.0)

            if msg is None:
                empty_polls += 1
                continue

            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    empty_polls += 1
                    continue
                continue

            empty_polls = 0

            raw_value = msg.value()
            raw_key = msg.key()

            if not raw_value or len(raw_value) < 5:
                continue

            # ══════════════════════════════════════════════
            # MANUALLY DECODE THE WIRE FORMAT
            # Byte 0:     Magic byte (should be 0x00)
            # Bytes 1-4:  Schema ID (big-endian 4-byte int)
            # Bytes 5+:   Avro binary data
            # ══════════════════════════════════════════════

            magic_byte = raw_value[0]
            schema_id = struct.unpack('>I', raw_value[1:5])[0]
            avro_data = raw_value[5:]

            # ── Lookup schema from Registry ──
            schema_name = "unknown"
            if schema_id not in schema_cache:
                try:
                    resp = requests.get(
                        f'{self.sr_url}/schemas/ids/{schema_id}',
                        timeout=5
                    )
                    if resp.status_code == 200:
                        schema_json = json.loads(resp.json().get('schema', '{}'))
                        schema_name = (
                            f"{schema_json.get('namespace', '')}"
                            f".{schema_json.get('name', '?')}"
                        )
                        field_names = [
                            f['name'] for f in schema_json.get('fields', [])
                        ]
                        schema_cache[schema_id] = {
                            'name': schema_name,
                            'fields': field_names,
                        }
                except Exception:
                    pass
            else:
                schema_name = schema_cache[schema_id]['name']

            key_str = raw_key.decode('utf-8') if raw_key else 'null'

            record = {
                'key': key_str,
                'magic_byte': f'0x{magic_byte:02X}',
                'schema_id': schema_id,
                'schema_name': schema_name,
                'total_bytes': len(raw_value),
                'header_bytes': 5,
                'data_bytes': len(avro_data),
                'key_bytes': len(raw_key) if raw_key else 0,
                'avro_hex_preview': avro_data[:20].hex(),
                'partition': msg.partition(),
                'offset': msg.offset(),
            }

            messages.append(record)

        self.consumer.close()
        return messages, schema_cache

    def display_inspection(self, messages, schema_cache):
        """Display wire format analysis."""

        print(f"  ╔══════════════════════════════════════════════════════╗")
        print(f"  ║         AVRO WIRE FORMAT INSPECTION                   ║")
        print(f"  ╠══════════════════════════════════════════════════════╣")

        for i, m in enumerate(messages[:10]):
            print(f"  ║                                                        ║")
            print(f"  ║  Message {i+1}:                                         ║")
            print(f"  ║    Key:        {m['key']:<40}  ║")
            print(f"  ║    Magic:      {m['magic_byte']}"
                  f" ({'✅ valid' if m['magic_byte'] == '0x00' else '❌ invalid'})"
                  f"                          ║")
            print(f"  ║    Schema ID:  {m['schema_id']:<40}  ║")
            print(f"  ║    Schema:     {m['schema_name'][:40]:<40}  ║")
            print(f"  ║    Total size: {m['total_bytes']:>5} bytes"
                  f" (header=5 + data={m['data_bytes']})"
                  f"        ║")
            print(f"  ║    Avro hex:   {m['avro_hex_preview'][:40]:<40}  ║")

        # ── Size analysis ──
        if messages:
            sizes = [m['total_bytes'] for m in messages]
            key_sizes = [m['key_bytes'] for m in messages]
            avg_size = sum(sizes) / len(sizes)
            avg_key = sum(key_sizes) / len(key_sizes)

            print(f"  ║                                                        ║")
            print(f"  ╠══════════════════════════════════════════════════════╣")
            print(f"  ║  SIZE ANALYSIS:                                        ║")
            print(f"  ║    Messages inspected: {len(messages):>6}                       ║")
            print(f"  ║    Avg message size:   {avg_size:>6.0f} bytes (Avro)            ║")
            print(f"  ║    Avg key size:       {avg_key:>6.0f} bytes                    ║")
            print(f"  ║    Min message size:   {min(sizes):>6} bytes                    ║")
            print(f"  ║    Max message size:   {max(sizes):>6} bytes                    ║")

            # ── Estimate JSON equivalent ──
            # JSON overhead: field names, quotes, colons, commas, braces
            # Rough estimate: Avro is ~30-40% of JSON size
            estimated_json = avg_size * 2.8
            savings_pct = (1 - avg_size / estimated_json) * 100

            print(f"  ║                                                        ║")
            print(f"  ║  ESTIMATED JSON COMPARISON:                             ║")
            print(f"  ║    Est. JSON size:     ~{estimated_json:>5.0f} bytes              ║")
            print(f"  ║    Avro size:           {avg_size:>5.0f} bytes              ║")
            print(f"  ║    Savings:            ~{savings_pct:>4.0f}%                       ║")

        # ── Schema summary ──
        if schema_cache:
            print(f"  ║                                                        ║")
            print(f"  ║  SCHEMAS ENCOUNTERED:                                   ║")
            for sid, info in sorted(schema_cache.items()):
                print(f"  ║    ID={sid}: {info['name'][:40]:<42}  ║")
                fields_str = ', '.join(info['fields'][:5])
                if len(info['fields']) > 5:
                    fields_str += f', ... (+{len(info["fields"])-5})'
                print(f"  ║           fields: {fields_str[:42]:<42}  ║")

        print(f"  ║                                                        ║")
        print(f"  ╚══════════════════════════════════════════════════════╝")


# ══════════════════════════════════════════
# MAIN
# ══════════════════════════════════════════

if __name__ == '__main__':
    print("╔══════════════════════════════════════════════════╗")
