PROJECT 28: THE KAFKA HEALTH DASHBOARD — MONITORING WITH JMX

🏗️ HYPOTHETICAL PROBLEM

SCENARIO: E-Commerce Platform Operations Team

→ A large e-commerce platform runs Kafka at scale:
  - 3 Kafka clusters (from Project 27: US-East, EU-West, APAC)
  - 47 topics across all clusters
  - 120+ consumer groups
  - 15 Kafka Connect connectors (Projects 15, 23, 24)
  - 4 MirrorMaker 2.0 replication flows (Project 27)
  - Retry topic infrastructure (Project 26)
  - DLQ topics (Project 25)
  - Processing 2.3 million events/hour at peak

→ THE INCIDENTS THAT EXPOSED THE MONITORING GAP:

  INCIDENT 1 — "The Silent Disk Death" (Week 1)
  ──────────────────────────────────────────────
  → Broker 2 disk filled to 98% over 3 days
  → Nobody noticed until broker started rejecting writes
  → 45 minutes of partial outage
  → Root cause: topic retention misconfigured (30 days instead of 7)
  → COULD HAVE BEEN PREVENTED: disk usage metric + alert at 80%

  INCIDENT 2 — "The Consumer Lag Avalanche" (Week 2)
  ──────────────────────────────────────────────────
  → Order processing consumer fell behind by 500,000 records
  → Customers complained: "My order shows as pending for 3 hours"
  → Support team discovered lag by manually checking consumer groups
  → Root cause: downstream database connection pool exhausted
  → COULD HAVE BEEN PREVENTED: consumer lag metric + alert at 10,000

  INCIDENT 3 — "The Partition Imbalance" (Week 3)
  ───────────────────────────────────────────────
  → Broker 1 handling 70% of all partition leaders
  → Broker 1 CPU at 95%, Brokers 2-3 at 30%
  → Throughput degraded by 60%
  → Root cause: preferred leader election not triggered after maintenance
  → COULD HAVE BEEN PREVENTED: leader imbalance metric + auto-rebalance

  INCIDENT 4 — "The Under-Replicated Horror" (Week 4)
  ──────────────────────────────────────────────────
  → 12 partitions became under-replicated
  → No alert — nobody knew
  → Broker 3 crashed during this period
  → 12 partitions had NO replica to promote
  → DATA LOSS for those 12 partitions
  → COULD HAVE BEEN PREVENTED: under-replicated partition alert

→ OPERATIONS TEAM REALIZATION:
  "We're running a mission-critical distributed system
   with ZERO operational visibility. We're flying blind."

→ REQUIREMENT:
  1. Real-time visibility into broker health (CPU, disk, network)
  2. Consumer lag tracking across ALL 120+ consumer groups
  3. Under-replicated partition alerting
  4. Producer performance metrics (batch efficiency, errors)
  5. MirrorMaker replication lag (cross-cluster from Project 27)
  6. Historical trending (is lag growing? is disk filling?)
  7. Automated alerting (PagerDuty/Slack integration)
  8. Single dashboard for ALL clusters

WHAT IS JMX?
═════════════
→ Java Management Extensions — built into every JVM
→ Kafka is written in Java/Scala → exposes metrics via JMX
→ MBeans (Managed Beans) = metric objects registered in JVM
→ Each MBean has: ObjectName, Attributes, Operations
→ Example: kafka.server:type=BrokerTopicMetrics,name=MessagesInPerSec

✅ WHAT SHOULD DO

1. MONITOR THE "BIG FIVE" BROKER METRICS FIRST
   → UnderReplicatedPartitions (MUST be 0)
   → OfflinePartitionsCount (MUST be 0)
   → ActiveControllerCount (MUST be exactly 1 across cluster)
   → MessagesInPerSec (throughput baseline)
   → RequestHandlerAvgIdlePercent (SHOULD be > 30%)
   → These five catch 90% of production issues

2. ALERT ON UNDER-REPLICATED PARTITIONS IMMEDIATELY
   → UnderReplicatedPartitions > 0 for > 30 seconds = CRITICAL
   → This means: one or more replicas are NOT in sync
   → If the leader fails during this time = DATA LOSS
   → Fastest indicator of broker health issues

3. TRACK CONSUMER LAG AS PRIMARY APPLICATION METRIC
   → Lag = how far behind consumers are from producers
   → Low lag = real-time processing
   → Growing lag = consumer can't keep up
   → Alert thresholds depend on SLA:
     - Real-time pipeline: alert at lag > 100
     - Analytics pipeline: alert at lag > 100,000
     - Batch pipeline: alert at lag > 1,000,000

4. SET UP HISTORICAL RETENTION IN PROMETHEUS
   → Default: 15 days retention
   → Recommended: 30-90 days for trend analysis
   → "Is disk usage growing 2% per week?" needs 4+ weeks of data
   → Storage: ~1-2 bytes per sample per metric

5. USE GRAFANA VARIABLES FOR MULTI-CLUSTER DASHBOARDS
   → Variable: $cluster = [us-east, eu-west, apac]
   → Variable: $broker = [broker-1, broker-2, broker-3]
   → Variable: $topic = [all topics from selected cluster]
   → One dashboard serves ALL clusters

6. SEPARATE OPERATIONAL vs BUSINESS DASHBOARDS
   → Operational: broker health, disk, CPU, under-replicated
     Audience: infrastructure team
   → Business: consumer lag, throughput, error rate
     Audience: application team
   → Executive: availability %, SLA compliance
     Audience: management

7. INCLUDE KAFKA CONNECT METRICS
   → Connector status (RUNNING/FAILED/PAUSED)
   → Task count and status
   → Source connector: poll rate, record count
   → Sink connector: put rate, error rate
   → SMT processing time (from Project 24)

8. MONITOR JVM METRICS ALONGSIDE KAFKA METRICS
   → Heap usage and GC pauses
   → Thread count
   → File descriptor count
   → These affect Kafka performance directly

❌ WHAT SHOULD NOT DO

1. DO NOT COLLECT ALL 400+ METRICS
   → Kafka exposes hundreds of MBeans
   → Most are low-value for daily operations
   → Collecting all = storage waste + dashboard clutter
   → START with 20-30 critical metrics
   → Add more as specific issues arise

2. DO NOT SET SCRAPE INTERVAL TOO LOW
   → Prometheus scrape interval < 10 seconds
   → Creates load on brokers (JMX queries are not free)
   → 15-30 second intervals sufficient for most metrics
   → Exception: consumer lag can be 10-second interval

3. DO NOT ALERT ON EVERY METRIC
   → 50 alerts = alert fatigue = alerts get ignored
   → START with 5-7 critical alerts
   → UnderReplicatedPartitions > 0
   → OfflinePartitionsCount > 0
   → ActiveControllerCount != 1
   → Consumer lag > threshold
   → Disk usage > 80%
   → Add more alerts ONLY after these are tuned

4. DO NOT IGNORE JVM GARBAGE COLLECTION METRICS
   → Long GC pauses = broker becomes unresponsive
   → Followers think leader is dead → unnecessary leader election
   → Producers get timeout errors
   → Monitor: GC pause time, GC frequency, heap usage

5. DO NOT MONITOR ONLY AVERAGES
   → Average latency = 5ms (looks great!)
   → P99 latency = 500ms (terrible for 1% of requests)
   → Always track: avg, p50, p95, p99, max
   → Percentiles reveal real user experience

6. DO NOT FORGET TO MONITOR THE MONITORING SYSTEM
   → If Prometheus goes down, you have zero visibility
   → Alert on: Prometheus up/down, scrape failures
   → Grafana health check
   → Meta-monitoring: "who watches the watchers?"

7. DO NOT HARDCODE THRESHOLDS WITHOUT BASELINING
   → "Alert if throughput < 1000 records/sec"
   → But normal throughput at 3AM is 50 records/sec
   → Baseline first: observe for 1-2 weeks
   → Set thresholds based on observed patterns
   → Use percentage-based anomaly detection where possible

═══════════════════════════════════════════════════════════
CONSUMER LAG — HOW IT'S CALCULATED
═══════════════════════════════════════════════════════════

  FOR EACH PARTITION:
    lag = log_end_offset - consumer_committed_offset

  EXAMPLE:
    Topic: orders, Partition: 0
    
    Log end offset (latest produced):    15,847
    Consumer committed offset:           15,200
    LAG = 15,847 - 15,200 = 647 records

  WHERE THE DATA COMES FROM:
    log_end_offset → Broker knows this (partition high watermark)
    committed_offset → Stored in __consumer_offsets topic
    
  THREE WAYS TO GET LAG:

  1. kafka-consumer-groups.sh (Project 8)
     → CLI tool, manual, point-in-time
     → Queries __consumer_offsets + broker watermarks

  2. Consumer JMX metrics
     → records-lag per partition
     → records-lag-max across all partitions
     → Requires consumer to be running

  3. External lag monitoring (Burrow / custom)
     → Queries __consumer_offsets + broker API
     → Doesn't require consumer to be running
     → Can detect STOPPED consumers (lag growing, no commits)

  LAG INTERPRETATION:

  ┌────────────┬──────────────────────────────────────────┐
  │ Lag Pattern │ What It Means                           │
  ├────────────┼──────────────────────────────────────────┤
  │ Lag = 0    │ Consumer fully caught up                 │
  │ Lag stable │ Consumer keeping up (constant small lag) │
  │ Lag growing│ Consumer falling behind (PROBLEM)        │
  │ Lag shrinks│ Consumer catching up after delay         │
  │ Lag = huge │ Consumer stopped or very slow            │
  │ No commits │ Consumer dead or not started             │
  └────────────┴──────────────────────────────────────────┘

UNDER-REPLICATED PARTITIONS — WHAT TRIGGERS IT
═══════════════════════════════════════════════════════════

  Normal state (replication factor = 3):
    Partition 0: Leader=Broker1, ISR=[Broker1, Broker2, Broker3]
    UnderReplicatedPartitions = 0 ✅

  Under-replicated (follower falls behind):
    Partition 0: Leader=Broker1, ISR=[Broker1, Broker2]
    Broker3 is BEHIND — removed from ISR
    UnderReplicatedPartitions = 1 ⚠️

  CAUSES OF UNDER-REPLICATION:
  → Broker overloaded (can't replicate fast enough)
  → Network issue between brokers
  → Disk I/O bottleneck on follower
  → GC pause on follower JVM
  → Broker recently restarted (catching up)

  WHY IT'S CRITICAL:
  → With replication factor 3 and ISR = [Broker1, Broker2]:
    - If Broker1 (leader) fails → Broker2 becomes leader ✅
    - But now ISR = [Broker2] only
    - If Broker2 fails → NO available replica → DATA LOSS ❌
  → Under-replication = reduced fault tolerance
  → Duration matters: brief (during restart) = OK
    Sustained (> 5 minutes) = investigate immediately

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: JMX PORT CONFLICT
═════════════════════════════
  → Multiple brokers on same machine
  → Both configured with JMX_PORT=9999
  → Second broker fails to start: "Address already in use"
  
  FIX: Unique JMX port per broker
       Broker 1: JMX_PORT=9999
       Broker 2: JMX_PORT=9998
       Broker 3: JMX_PORT=9997


PITFALL 2: JMX EXPORTER CAUSES GC PRESSURE
═══════════════════════════════════════════
  → JMX Exporter collects ALL MBeans by default
  → Kafka has 400+ MBeans
  → Each scrape creates temporary objects
  → With 10-second scrape interval: significant GC pressure
  
  FIX: Configure JMX Exporter with whitelist rules
       Only export metrics you actually use
       Use blacklist for known high-cardinality metrics


PITFALL 3: HIGH-CARDINALITY METRIC EXPLOSION
═════════════════════════════════════════════
  → Per-topic metrics × 47 topics = 47 time series per metric
  → Per-partition metrics × 47 topics × 6 partitions = 282 series
  → Per-consumer-group × 120 groups = 120 series
  → Total: thousands of time series
  → Prometheus storage grows rapidly
  
  FIX: Aggregate at topic level, not partition level (for dashboards)
       Use recording rules in Prometheus for pre-aggregation
       Set appropriate retention period
       Monitor Prometheus storage usage itself


PITFALL 4: CONSUMER LAG METRIC SHOWS ZERO BUT CONSUMER IS DEAD
═══════════════════════════════════════════════════════════════
  → Consumer crashed 2 hours ago
  → Consumer JMX metrics: no data (consumer not running)
  → Prometheus: last known lag was 0
  → Alert doesn't fire (lag = 0)
  → But no records are being processed!
  
  FIX: Monitor ABSENCE of metrics, not just values
       Alert if: consumer_records_consumed_rate == 0 for > 5 minutes
       AND source topic has active producers
       Use Burrow-style "consumer status" evaluation


PITFALL 5: ALERT STORM DURING MAINTENANCE
══════════════════════════════════════════
  → Broker restart for upgrade
  → UnderReplicatedPartitions spikes to 50
  → ISR shrink rate spikes
  → Leader election count spikes
  → 15 alerts fire simultaneously
  → Team ignores alerts (planned maintenance)
  → REAL alert mixed in gets ignored too
  
  FIX: Implement maintenance windows in alerting
       "Silence alerts for broker-2 from 02:00-04:00"
       Use Alertmanager inhibition rules
       Or Grafana's "Pause" feature on alerts


PITFALL 6: MONITORING ONLY BROKERS, FORGETTING CLIENTS
═══════════════════════════════════════════════════════
  → Broker metrics all green
  → But producer batch.size too small → inefficient
  → Consumer max.poll.records too low → can't keep up
  → Connect tasks failing silently
  → Broker looks healthy, but applications suffer
  
  FIX: Monitor ALL layers:
       Broker + Producer + Consumer + Connect + Application


PITFALL 7: DASHBOARD SHOWS AVERAGES THAT HIDE SPIKES
═════════════════════════════════════════════════════
  → Dashboard shows: avg request latency = 5ms
  → Reality: 99% of requests at 2ms, 1% at 500ms
  → Those 1% timeouts cause producer retries
  → Retries cause duplicates
  → Average hides the problem
  
  FIX: Always display percentiles (p50, p95, p99, p999)
       Alert on p99, not average
       Use Grafana heatmaps for latency distribution

📈 SCALABILITY CONSIDERATIONS

PROMETHEUS STORAGE:
═══════════════════
  → Each metric time series: ~1-2 bytes per sample
  → 500 metrics × 3 brokers × 4 samples/min = 6000 samples/min
  → 6000 × 2 bytes × 60 min × 24 hr × 30 days = ~500MB/month
  → Manageable for small clusters
  → Large clusters (50+ brokers): consider Thanos or Cortex for long-term storage

SCRAPE OVERHEAD:
════════════════
  → Each scrape: JMX Exporter queries MBean Server
  → 200 metrics × 15s interval = ~13 queries/second per broker
  → CPU overhead: < 1% of broker CPU
  → Network: ~50KB per scrape response
  → NEGLIGIBLE for normal operations
  → BUT: with 1000+ metrics enabled: can reach 5% CPU overhead

# ============================================================
# FILE: lag_exporter.py
#
# Exports consumer lag metrics to Prometheus format
# Runs as HTTP server that Prometheus scrapes
#
# Consumer lag is NOT available via broker JMX alone
# This fills that gap using admin client API
# ============================================================

from confluent_kafka import Consumer, TopicPartition
from confluent_kafka.admin import AdminClient
from http.server import HTTPServer, BaseHTTPRequestHandler
import threading
import time
import json

BOOTSTRAP = 'localhost:9092'
EXPORTER_PORT = 8000
SCRAPE_INTERVAL = 15  # seconds

class LagCollector:
    """Collects consumer lag for all consumer groups"""
    
    def __init__(self):
        self.admin = AdminClient({'bootstrap.servers': BOOTSTRAP})
        self.metrics = {}
        self.last_update = 0
        self._lock = threading.Lock()
    
    def collect(self):
        """Collect lag for all consumer groups"""
        
        try:
            # List all consumer groups
            group_metadata = self.admin.list_groups(timeout=10)
            
            group_names = [g.id for g in group_metadata 
                          if not g.id.startswith('_') 
                          and not g.id.startswith('mm2-')
                          and not g.id.startswith('connect-')]
            
            all_metrics = []
            
            for group_name in group_names:
                try:
                    lag_metrics = self._get_group_lag(group_name)
                    all_metrics.extend(lag_metrics)
                except Exception as e:
                    all_metrics.append({
                        'name': 'kafka_consumer_group_error',
                        'labels': {'group': group_name},
                        'value': 1,
                        'help': 'Error collecting lag for consumer group'
                    })
            
            with self._lock:
                self.metrics = all_metrics
                self.last_update = time.time()
            
            return all_metrics
        
        except Exception as e:
            print(f"  Error collecting metrics: {e}")
            return []
    
    def _get_group_lag(self, group_name):
        """Get lag for a specific consumer group"""
        
        consumer = Consumer({
            'bootstrap.servers': BOOTSTRAP,
            'group.id': group_name,
            'auto.offset.reset': 'latest',
            'enable.auto.commit': False
        })
        
        metrics = []
        
        try:
            # Get committed offsets for this group
            # First, find which topics this group consumes
            metadata = self.admin.list_topics(timeout=10)
            
            for topic_name, topic_meta in metadata.topics.items():
                if topic_name.startswith('__') or topic_name.startswith('connect-'):
                    continue
                
                partitions = [
                    TopicPartition(topic_name, pid) 
                    for pid in topic_meta.partitions
                ]
                
                # Get committed offsets
                committed = consumer.committed(partitions, timeout=5)
                
                topic_total_lag = 0
                has_commits = False
                
                for tp in committed:
                    if tp.offset < 0:
                        continue  # No committed offset for this partition
                    
                    has_commits = True
                    
                    # Get high watermark
                    _, high = consumer.get_watermark_offsets(
                        TopicPartition(tp.topic, tp.partition), timeout=5
                    )
                    
                    lag = max(0, high - tp.offset)
                    topic_total_lag += lag
                    
                    # Per-partition lag
                    metrics.append({
                        'name': 'kafka_consumer_group_partition_lag',
                        'labels': {
                            'group': group_name,
                            'topic': tp.topic,
                            'partition': str(tp.partition)
                        },
                        'value': lag,
                        'help': 'Consumer group partition lag'
                    })
                
                if has_commits:
                    # Per-topic total lag
                    metrics.append({
                        'name': 'kafka_consumer_group_topic_lag',
                        'labels': {
                            'group': group_name,
                            'topic': topic_name
                        },
                        'value': topic_total_lag,
                        'help': 'Consumer group total lag for topic'
                    })
        
        except Exception as e:
            metrics.append({
                'name': 'kafka_consumer_group_error',
                'labels': {'group': group_name, 'error': str(e)[:50]},
                'value': 1,
                'help': 'Error collecting group lag'
            })
        
        finally:
            consumer.close()
        
        return metrics
    
    def get_prometheus_output(self):
        """Format metrics in Prometheus exposition format"""
        
        with self._lock:
            metrics = self.metrics
        
        lines = []
        seen_helps = set()
        
        for m in metrics:
            name = m['name']
            
            # Add HELP and TYPE once per metric name
            if name not in seen_helps:
                lines.append(f"# HELP {name} {m.get('help', '')}")
                lines.append(f"# TYPE {name} gauge")
                seen_helps.add(name)
            
            # Format labels
            label_parts = [f'{k}="{v}"' for k, v in m['labels'].items()]
            label_str = '{' + ','.join(label_parts) + '}' if label_parts else ''
            
            lines.append(f"{name}{label_str} {m['value']}")
        
        # Add meta-metrics
        lines.append(f"# HELP kafka_lag_exporter_last_scrape_timestamp_seconds Last successful scrape")
        lines.append(f"# TYPE kafka_lag_exporter_last_scrape_timestamp_seconds gauge")
        lines.append(f"kafka_lag_exporter_last_scrape_timestamp_seconds {self.last_update}")
        
        lines.append(f"# HELP kafka_lag_exporter_metrics_count Number of metrics collected")
        lines.append(f"# TYPE kafka_lag_exporter_metrics_count gauge")
        lines.append(f"kafka_lag_exporter_metrics_count {len(metrics)}")
        
        return '\n'.join(lines) + '\n'


# Global collector instance
collector = LagCollector()

class MetricsHandler(BaseHTTPRequestHandler):
    """HTTP handler for Prometheus scraping"""
    
    def do_GET(self):
        if self.path == '/metrics':
            output = collector.get_prometheus_output()
            
            self.send_response(200)
            self.send_header('Content-Type', 'text/plain; version=0.0.4')
            self.end_headers()
            self.wfile.write(output.encode('utf-8'))
        
        elif self.path == '/health':
            self.send_response(200)
            self.send_header('Content-Type', 'application/json')
            self.end_headers()
            health = {
                'status': 'healthy',
                'last_update': collector.last_update,
                'metric_count': len(collector.metrics)
            }
            self.wfile.write(json.dumps(health).encode('utf-8'))
        
        else:
            self.send_response(404)
            self.end_headers()
    
    def log_message(self, format, *args):
        # Suppress default logging for cleaner output
        pass

def background_collector():
    """Background thread that collects metrics periodically"""
    while True:
        try:
            metrics = collector.collect()
            print(f"  Collected {len(metrics)} lag metrics at {time.strftime('%H:%M:%S')}")
        except Exception as e:
            print(f"  Collection error: {e}")
        
        time.sleep(SCRAPE_INTERVAL)

    def _parse_prometheus_metrics(self, text):
        """Parse Prometheus exposition format into dict"""
        metrics = {}
        
        for line in text.split('\n'):
            line = line.strip()
            
            if not line or line.startswith('#'):
                continue
            
            try:
                # Handle metrics with labels
                if '{' in line:
                    name_part = line.split('{')[0]
                    value_part = line.split('}')[-1].strip()
                    
                    # Extract labels
                    labels_str = line.split('{')[1].split('}')[0]
                    
                    try:
                        value = float(value_part)
                    except ValueError:
                        continue
                    
                    # Store with labels as sub-dict
                    if name_part not in metrics:
                        metrics[name_part] = {}
                    
                    # Parse label key-value pairs
                    label_dict = {}
                    for pair in labels_str.split(','):
                        if '=' in pair:
                            k, v = pair.split('=', 1)
                            label_dict[k.strip()] = v.strip('"')
                    
                    # Use first label value as sub-key for simple access
                    if label_dict:
                        first_val = list(label_dict.values())[0]
                        if isinstance(metrics[name_part], dict):
                            metrics[name_part][first_val] = value
                        
                else:
                    # Simple metric without labels
                    parts = line.split()
                    if len(parts) >= 2:
                        name = parts[0]
                        try:
                            value = float(parts[1])
                            metrics[name] = value
                        except ValueError:
                            continue
            
            except Exception:
                continue
        
        return metrics

    def _get_group_total_lag(self, group_name):
        """Calculate total lag for a consumer group"""
        consumer = Consumer({
            'bootstrap.servers': BOOTSTRAP,
            'group.id': group_name,
            'auto.offset.reset': 'latest',
            'enable.auto.commit': False
        })
        
        total_lag = 0
        has_commits = False
        
        try:
            metadata = self.admin.list_topics(timeout=5)
            
            for topic_name, topic_meta in metadata.topics.items():
                if topic_name.startswith('__'):
                    continue
                
                partitions = [
                    TopicPartition(topic_name, pid) 
                    for pid in topic_meta.partitions
                ]
                
                committed = consumer.committed(partitions, timeout=3)
                
                for tp in committed:
                    if tp.offset < 0:
                        continue
                    
                    has_commits = True
                    _, high = consumer.get_watermark_offsets(
                        TopicPartition(tp.topic, tp.partition), timeout=3
                    )
                    
                    lag = max(0, high - tp.offset)
                    total_lag += lag
        
        finally:
            consumer.close()
        
        return total_lag if has_commits else -1

                key=event['user_id'].encode('utf-8'),
                value=json.dumps(event).encode('utf-8'),
                callback=delivery_cb
            )
        
        producer.flush(timeout=1)
        
        if stats['produced'] % 100 == 0 and stats['produced'] > 0:
            elapsed_s = time.time() - start
            rate = stats['produced'] / elapsed_s
            print(f"    Produced: {stats['produced']:,} ({rate:.0f}/sec)")
        
        time.sleep(0.1)
    
    producer.flush()
    
    print(f"\n  ✅ Production complete: {stats['produced']:,} records, "
          f"{stats['errors']} errors")
    
    return stats

def consume_with_lag(topic, group_name, speed='normal', duration_seconds=120):
    """
    Consume from topic at specified speed to create observable lag
    speed: 'normal' (keeps up), 'slow' (creates lag), 'stopped' (max lag)
    """
    
    if speed == 'stopped':
        print(f"  ⏸️  Consumer '{group_name}' STOPPED (creating max lag)")
        return
    
    consumer = Consumer({
        'bootstrap.servers': BOOTSTRAP,
        'group.id': group_name,
        'auto.offset.reset': 'earliest',
        'enable.auto.commit': True,
        'auto.commit.interval.ms': 5000,
        'max.poll.records': 10 if speed == 'slow' else 500
    })
    
    consumer.subscribe([topic])
    consumed = 0
    
    print(f"  📥 Consumer '{group_name}' started ({speed} speed)")
    
    start = time.time()
    
    try:
        while time.time() - start < duration_seconds:
            msg = consumer.poll(1.0)
            
            if msg is None:
                continue
            
            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    continue
                continue
            
            consumed += 1
            
            # Slow consumer: artificial delay
            if speed == 'slow':
                time.sleep(0.1)  # 100ms per record — much slower than production
            
            if consumed % 200 == 0:
                print(f"    {group_name}: consumed {consumed:,}")
    
    except KeyboardInterrupt:
        pass
    finally:
        consumer.close()
    
    print(f"  ✅ {group_name}: consumed {consumed:,} records")
