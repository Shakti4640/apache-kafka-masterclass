🎯 HYPOTHETICAL PROBLEM

You manage "ByteFactory Assembly Line" — a 24/7 manufacturing plant

→ 8 assembly stations (partitions) run continuously
→ 3 shifts of workers rotate throughout the day:
    Morning shift:   6 workers (6:00 AM - 2:00 PM)
    Afternoon shift: 4 workers (2:00 PM - 10:00 PM)
    Night shift:     2 workers (10:00 PM - 6:00 AM)

SHIFT CHANGE PROBLEMS:

PROBLEM A: The Handover Gap
→ At 2:00 PM, morning shift (6 workers) leaves
→ Afternoon shift (4 workers) arrives
→ During handover, ALL stations STOP for 5 minutes
→ No products assembled during this gap
→ 5 minutes × 3 shifts/day = 15 minutes of DAILY DOWNTIME

PROBLEM B: The Sick Call
→ Worker Charlie calls in sick at 10:30 AM
→ His 2 stations (P3, P4) go UNPROCESSED
→ Factory waits 45 seconds before realizing Charlie is gone
→ Then ALL workers stop, reassign, restart
→ Products pile up at P3 and P4 for nearly a minute

PROBLEM C: The New Hire
→ New worker Eve joins at 11:00 AM
→ To give Eve a station, ALL workers must stop
→ Stations are reshuffled — Alice loses P0, gets P5 instead
→ Alice had been building state about P0 for hours
→ That state is now WASTED — she starts fresh on P5

PROBLEM D: The Sticky Worker
→ Bob has been working station P2 for 6 hours
→ He knows P2's quirks — built up expertise (cached state)
→ During shift change, Bob gets reassigned to P7
→ Bob's P2 expertise is LOST
→ The new P2 worker starts from scratch

YOU NEED:
→ Minimize the "all stop" duration during shift changes
→ Keep workers on their SAME stations when possible
→ Detect absent workers QUICKLY (not 45-second wait)
→ Add new workers WITHOUT disrupting existing ones
→ Handle graceful departures vs sudden crashes differently

SOLUTION: Use Cooperative Sticky Assignor with proper rebalance handling

THE TWO REBALANCE PROTOCOLS:

═══ EAGER REBALANCE (Default — "Stop the World") ═══

  BEFORE:                    DURING:                   AFTER:
  Alice → [P0, P1]          Alice → []   STOPPED!     Alice → [P0, P1]
  Bob   → [P2, P3]          Bob   → []   STOPPED!     Bob   → [P2]
  Charlie → [P4, P5]        Charlie → [] STOPPED!     Charlie → [P4, P5]
                                                       Diana → [P3]  NEW!
  
  Timeline:
  ─────────────────────────────────────────────────────────────
  T=0     Diana joins
  T=0.1s  Coordinator says "REBALANCE"
  T=0.2s  ALL consumers REVOKE all partitions → STOP processing
  T=0.5s  Leader computes new assignment
  T=0.7s  ALL consumers get new assignment → RESUME processing
  
  TOTAL PAUSE: ~0.5-2 seconds (ALL consumers stopped)
  PROBLEM: Alice and Charlie did NOT change — but they STILL stopped!


═══ COOPERATIVE REBALANCE (Incremental — "Minimal Disruption") ═══

  BEFORE:                    DURING:                   AFTER:
  Alice → [P0, P1]          Alice → [P0, P1] RUNNING! Alice → [P0, P1]
  Bob   → [P2, P3]          Bob   → [P2]    partial   Bob   → [P2]
  Charlie → [P4, P5]        Charlie → [P4, P5] RUN!   Charlie → [P4, P5]
                                                       Diana → [P3]  NEW!
  
  Timeline:
  ─────────────────────────────────────────────────────────────
  T=0     Diana joins
  T=0.1s  Coordinator says "REBALANCE"
  T=0.2s  ONLY Bob revokes P3 (the partition being moved)
          Alice and Charlie CONTINUE processing — no interruption!
  T=0.5s  P3 assigned to Diana
  T=0.6s  Diana starts processing P3
  
  TOTAL PAUSE: ~0.3s for Bob's P3 ONLY
  Alice: ZERO downtime
  Charlie: ZERO downtime
  Bob: Lost P3 but KEPT P2 (no unnecessary shuffle)

✅ WHAT SHOULD DO

1. ALWAYS use CooperativeStickyAssignor in production
   → Minimizes rebalance impact
   → Keeps existing assignments stable
   → Only moves partitions that NEED to move
   → Set via: 'partition.assignment.strategy': 'cooperative-sticky'

2. ALWAYS commit offsets in on_revoke callback
   → When a partition is revoked, you lose it
   → If you processed messages but didn't commit, they'll be reprocessed
   → Commit in on_revoke = smallest possible reprocessing window

3. ALWAYS clean up partition-specific state in on_revoke
   → If you cache data per partition (e.g., aggregation buffers)
   → Flush and persist that state before the partition is taken away
   → The next consumer for that partition needs a clean start

4. ALWAYS tune session.timeout.ms based on your infrastructure
   → Default: 45 seconds (very conservative)
   → Recommended: 10-30 seconds for most applications
   → Lower = faster crash detection, but risk of false positives
   → Must be > 3x heartbeat.interval.ms

5. ALWAYS set max.poll.interval.ms based on your processing time
   → Default: 300 seconds (5 minutes)
   → If your processing takes >5 minutes between poll() calls
   → Kafka thinks you're dead → triggers rebalance
   → Set higher if processing is legitimately slow

6. ALWAYS handle rebalance as a NORMAL event, not an error
   → Rebalances happen routinely (deployments, scaling, crashes)
   → Your code must be rebalance-resilient by design
   → Test rebalance scenarios explicitly

❌ WHAT SHOULD NOT DO

1. DO NOT use eager rebalance in production if you have >4 consumers
   → Every rebalance stops ALL consumers
   → With 20 consumers, a single join/leave pauses ALL 20
   → Impact grows linearly with consumer count

2. DO NOT set session.timeout.ms too low
   → Below 6 seconds with default heartbeat (3s)
   → GC pauses, network blips trigger false "consumer dead" detection
   → Causes unnecessary rebalances → cascading instability
   → Minimum safe: 2x heartbeat.interval.ms + buffer

3. DO NOT do heavy processing inside on_revoke callback
   → on_revoke should be FAST (commit offsets, flush buffers)
   → Heavy processing delays the rebalance for ALL consumers
   → Other consumers wait until on_revoke completes

4. DO NOT ignore max.poll.interval.ms
   → If time between poll() calls exceeds this value
   → Consumer is considered "stuck" → kicked from group → rebalance
   → Common cause: processing a single message takes too long
   → Common fix: increase max.poll.interval.ms or process faster

5. DO NOT depend on specific partition assignment
   → Even with sticky assignor, partitions CAN move
   → Your code must handle ANY partition assignment
   → Never hardcode logic like "if partition == 0, do special thing"

6. DO NOT mix assignor strategies within the same consumer group
   → All consumers in a group MUST use the same assignor
   → Mixing causes: "Assignor mismatch" error
   → Coordinate strategy changes during rolling deployments

🚫 WHEN NOT TO USE

→ DO NOT use cooperative rebalance if ALL consumers must restart simultaneously
  (rare case: some stream processing frameworks require eager for correctness)

→ DO NOT tune session.timeout.ms below 6 seconds in cloud environments
  (network jitter in cloud → false positives → cascading rebalances)

→ DO NOT use rebalance listeners for long-running operations
  (database transactions, API calls should complete BEFORE on_revoke)
  (or be designed to be safely interrupted)

═══════════════════════════════════════════════════════════════
THE HEARTBEAT MECHANISM — HOW KAFKA DETECTS LIVENESS
═══════════════════════════════════════════════════════════════

Every consumer in a group runs a BACKGROUND HEARTBEAT THREAD:

  Consumer Process
  ┌────────────────────────────────────────────────────┐
  │                                                    │
  │  MAIN THREAD          HEARTBEAT THREAD             │
  │  ┌──────────────┐     ┌──────────────────┐         │
  │  │ while True:  │     │ while True:      │         │
  │  │   poll()     │     │   send_heartbeat()│         │
  │  │   process()  │     │   sleep(3s)      │         │
  │  │   commit()   │     │   check_response()│        │
  │  └──────────────┘     └──────────────────┘         │
  │                                                    │
  └────────────────────────────────────────────────────┘
                              │
                    HeartbeatRequest (every 3s)
                              │
                              ▼
                    ┌──────────────────┐
                    │ GROUP COORDINATOR│
                    │ (Broker)         │
                    │                  │
                    │ Tracks:          │
                    │ - Last heartbeat │
                    │ - Session timeout│
                    │ - Member list    │
                    └──────────────────┘

  COORDINATOR LOGIC:
  
  for each member in group:
      time_since_heartbeat = now - member.last_heartbeat
      if time_since_heartbeat > session.timeout.ms:
          mark member as DEAD
          trigger REBALANCE

  HEARTBEAT RESPONSE can contain:
  → "OK" — everything normal
  → "REBALANCE_IN_PROGRESS" — consumer must rejoin
  → "UNKNOWN_MEMBER_ID" — consumer was kicked out


═══════════════════════════════════════════════════════════════
TWO TIMEOUT MECHANISMS — session.timeout vs max.poll.interval
═══════════════════════════════════════════════════════════════

MECHANISM 1: session.timeout.ms (heartbeat-based)
─────────────────────────────────────────────────
  → Detected by: GROUP COORDINATOR (broker-side)
  → Triggered when: Heartbeats stop arriving
  → Cause: Consumer process crashed, network failure, host down
  → Detection time: session.timeout.ms (default 45s)
  
  Timeline of a CRASH:
  T=0s     Consumer crashes (process killed)
  T=3s     Missed heartbeat #1 (coordinator notices)
  T=6s     Missed heartbeat #2
  T=9s     Missed heartbeat #3
  ...
  T=45s    session.timeout.ms expires → REBALANCE triggered
  
  → 45 seconds of dead consumer before detection!
  → Reduce to 10s: detection in ~10 seconds


MECHANISM 2: max.poll.interval.ms (poll-based)
──────────────────────────────────────────────
  → Detected by: CONSUMER CLIENT (client-side)
  → Triggered when: Time between poll() calls exceeds limit
  → Cause: Slow message processing, blocked thread, infinite loop
  → Detection time: max.poll.interval.ms (default 300s = 5 min)
  
  Timeline of STUCK PROCESSING:
  T=0s     poll() returns message
  T=0.1s   Start processing (e.g., calling slow API)
  T=60s    Still processing... heartbeats still sending!
  T=120s   Still processing... coordinator thinks consumer is alive
  T=300s   max.poll.interval.ms expires
           → Client LEAVES group voluntarily
           → Heartbeat thread sends LeaveGroupRequest
           → REBALANCE triggered immediately
  
  KEY INSIGHT:
  → Heartbeats continue even when processing is stuck
  → Coordinator thinks consumer is alive (heartbeats arriving)
  → BUT consumer is not making progress (not calling poll())
  → max.poll.interval.ms catches this case

  ┌────────────────────────────────────────────────────────────┐
  │ session.timeout.ms    → Detects DEAD consumers             │
  │                         (no heartbeats at all)             │
  │                                                            │
  │ max.poll.interval.ms  → Detects STUCK consumers            │
  │                         (heartbeats fine, but not polling) │
  └────────────────────────────────────────────────────────────┘


═══════════════════════════════════════════════════════════════
EAGER REBALANCE PROTOCOL — STEP BY STEP
═══════════════════════════════════════════════════════════════

  INITIAL STATE:
  Alice → [P0, P1, P2]
  Bob   → [P3, P4, P5]

  TRIGGER: Charlie joins

  STEP 1: Coordinator detects new member
  ──────────────────────────────────────
  Charlie sends JoinGroupRequest
  Coordinator marks group as "REBALANCING"
  
  STEP 2: Coordinator notifies ALL members via heartbeat response
  ─────────────────────────────────────────────────────────────
  Alice's heartbeat → response: "REBALANCE_IN_PROGRESS"
  Bob's heartbeat   → response: "REBALANCE_IN_PROGRESS"
  
  STEP 3: ALL members REVOKE ALL partitions (EAGER — everything revoked!)
  ────────────────────────────────────────────────────────────────────────
  Alice: on_revoke([P0, P1, P2]) → commits offsets, stops processing
  Bob:   on_revoke([P3, P4, P5]) → commits offsets, stops processing
  
  ★ ALL PROCESSING STOPS ★
  
  STEP 4: ALL members send JoinGroupRequest
  ──────────────────────────────────────────
  Alice: "I want to rejoin, I can handle [warehouse-topic]"
  Bob:   "I want to rejoin, I can handle [warehouse-topic]"
  Charlie: "I'm new, I can handle [warehouse-topic]"
  
  STEP 5: Leader (Alice) computes new assignment
  ───────────────────────────────────────────────
  RangeAssignor:
    Alice   → [P0, P1]
    Bob     → [P2, P3]
    Charlie → [P4, P5]
  
  STEP 6: Leader sends SyncGroupRequest with assignment
  ─────────────────────────────────────────────────────
  Coordinator distributes assignment to all members
  
  STEP 7: ALL members receive assignment → on_assign fires
  ────────────────────────────────────────────────────────
  Alice:   on_assign([P0, P1])   → resumes processing
  Bob:     on_assign([P2, P3])   → resumes processing
  Charlie: on_assign([P4, P5])   → starts processing
  
  ★ PROCESSING RESUMES ★

  TOTAL PAUSE: Steps 3-7 = typically 0.5-5 seconds
  ALL consumers paused even if their assignment didn't change!


═══════════════════════════════════════════════════════════════
COOPERATIVE REBALANCE PROTOCOL — STEP BY STEP
═══════════════════════════════════════════════════════════════

  INITIAL STATE:
  Alice → [P0, P1, P2]
  Bob   → [P3, P4, P5]

  TRIGGER: Charlie joins

  STEP 1: Same as eager — coordinator detects new member
  
  STEP 2: Same — notify via heartbeat response
  
  STEP 3: COOPERATIVE — members revoke ONLY moved partitions
  ─────────────────────────────────────────────────────────
  
  FIRST REBALANCE ROUND:
  → Leader computes: what NEEDS to move?
  → Alice currently has [P0, P1, P2] → should have [P0, P1]
  → Only P2 needs to move from Alice
  → Bob currently has [P3, P4, P5] → should have [P3, P4]
  → Only P5 needs to move from Bob
  
  Alice: on_revoke([P2]) → commits offset for P2 ONLY
         CONTINUES processing P0, P1!  ← KEY DIFFERENCE
  Bob:   on_revoke([P5]) → commits offset for P5 ONLY
         CONTINUES processing P3, P4!
  
  SECOND REBALANCE ROUND (automatic):
  → P2 and P5 are now unassigned
  → Leader assigns them to Charlie
  
  Charlie: on_assign([P2, P5]) → starts processing
  
  ★ RESULT ★
  Alice:   NEVER stopped processing P0, P1 (zero downtime!)
  Bob:     NEVER stopped processing P3, P4 (zero downtime!)
  Charlie: Started P2, P5 after brief delay
  
  ONLY the partitions that MOVED experienced any pause


═══════════════════════════════════════════════════════════════
COOPERATIVE STICKY — THE "STICKY" PART
═══════════════════════════════════════════════════════════════

  "Sticky" means: try to keep the SAME assignment as before

  SCENARIO: Bob leaves the group

  WITHOUT STICKY (RangeAssignor):
  ────────────────────────────────
  BEFORE:                   AFTER:
  Alice   → [P0, P1]       Alice   → [P0, P1, P2]   ← added P2
  Bob     → [P2, P3]       (Bob gone)
  Charlie → [P4, P5]       Charlie → [P3, P4, P5]   ← CHANGED! was P4,P5

  Charlie LOST P4, P5 and GOT P3, P4, P5
  → Charlie's cached state for "old P4" is at different offset now
  → Unnecessary shuffle!

  WITH STICKY (CooperativeStickyAssignor):
  ─────────────────────────────────────────
  BEFORE:                   AFTER:
  Alice   → [P0, P1]       Alice   → [P0, P1, P2]   ← added P2 (Bob's)
  Bob     → [P2, P3]       (Bob gone)
  Charlie → [P4, P5]       Charlie → [P3, P4, P5]   ← added P3 (Bob's)
                                                       KEPT P4, P5!

  Wait — same result? The DIFFERENCE is:
  → Sticky TRIES to minimize movement
  → With more consumers, the difference is dramatic:

  8 consumers, 1 leaves:
  → RangeAssignor: potentially shuffles ALL 7 remaining consumers
  → StickyAssignor: only moves the 1 departed consumer's partitions
    → Other 7 consumers keep their EXACT assignment

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Constant rebalancing — consumers keep joining and leaving"
├── CAUSE A: session.timeout.ms too low + GC pauses
│   → Consumer pauses for GC → misses heartbeat → kicked → rejoins → rebalance
│   └── FIX: Increase session.timeout.ms to 30s, tune JVM GC
├── CAUSE B: max.poll.interval.ms too low + slow processing
│   → Processing takes 6 minutes, limit is 5 minutes → kicked
│   └── FIX: Increase max.poll.interval.ms or process faster
└── CAUSE C: Consumer crashes and restarts in a loop
    └── FIX: Fix the crash cause — check application logs

PITFALL 2: "Rebalance takes 45 seconds after consumer crash"
├── CAUSE: Default session.timeout.ms = 45000
├── WHY: Coordinator waits 45s for heartbeats before declaring dead
├── FIX: Reduce to 10000-15000 for faster detection
└── TRADEOFF: Too low → false positives during GC/network blips

PITFALL 3: "All consumers stop during rebalance"
├── CAUSE: Using eager rebalance protocol (default in older configs)
├── FIX: Switch to CooperativeStickyAssignor
│   config = {'partition.assignment.strategy': 'cooperative-sticky'}
└── NOTE: All consumers in group must use same strategy

PITFALL 4: "Consumer loses partition state after rebalance"
├── CAUSE: Cached/in-memory state not persisted before revocation
├── EXAMPLE: Aggregation buffer for P3 lost when P3 moves to another consumer
├── FIX: Persist state in on_revoke callback
│   → Write to external store (database, state store)
│   → Or commit state to a Kafka compacted topic
└── DEEP DIVE: Project 21 (stateful aggregations), Project 33 (interactive queries)

PITFALL 5: "Duplicate messages after rebalance"
├── CAUSE: Messages processed but offset not committed before revocation
├── WHY: Consumer processed offsets 45-50 but committed only up to 45
│   → Partition revoked → new consumer starts at 45 → reprocesses 45-50
├── FIX: Commit offsets in on_revoke callback
│   → consumer.commit(asynchronous=False) inside on_revoke
└── DEEP DIVE: Project 6 (offset management), Project 17 (idempotent producer)

PITFALL 6: "Rolling deployment causes cascading rebalances"
├── CAUSE: Restarting consumers one by one → each restart triggers rebalance
├── SCENARIO: 8 consumers, restart one at a time
│   → 8 leave events + 8 join events = up to 16 rebalances
├── FIX A: Use group.instance.id for static membership (no rebalance on restart)
├── FIX B: Use CooperativeStickyAssignor (minimizes impact per rebalance)
└── FIX C: Batch restarts with group.initial.rebalance.delay.ms

📈 SCALABILITY CONSIDERATIONS

REBALANCE IMPACT SCALES WITH CONSUMER COUNT:

  ┌───────────────┬──────────────────────┬──────────────────────────┐
  │ Consumers     │ Eager Rebalance      │ Cooperative Rebalance    │
  ├───────────────┼──────────────────────┼──────────────────────────┤
  │ 4             │ ~0.5s pause (all 4)  │ ~0.3s (1-2 affected)    │
  │ 20            │ ~2s pause (all 20)   │ ~0.3s (1-2 affected)    │
  │ 100           │ ~5-15s pause (all!)  │ ~0.5s (1-2 affected)    │
  │ 500           │ ~30-60s pause (!!!)  │ ~1s (1-2 affected)      │
  └───────────────┴──────────────────────┴──────────────────────────┘

  → Eager rebalance: impact grows LINEARLY with consumer count
  → Cooperative: impact is CONSTANT regardless of consumer count
  → This is why cooperative is MANDATORY for large consumer groups

STATIC MEMBERSHIP (group.instance.id):
  → Assigns a persistent identity to each consumer
  → When consumer restarts with SAME instance.id:
    → No rebalance! Consumer gets same partitions back
    → Only triggers rebalance after session.timeout.ms expires
  → Perfect for rolling deployments
  → Reduces rebalances from N (one per restart) to 0

        self.last_revoke_time = None
        self.messages_processed = 0
        self.start_time = time.time()
        self._lock = threading.Lock()
    
    def on_revoke(self, consumer, partitions):
        """Called when partitions are REVOKED"""
        revoke_start = time.time()
        
        revoked_parts = sorted([p.partition for p in partitions])
        
        with self._lock:
            self.rebalance_count += 1
            self.last_revoke_time = revoke_start
            
            self.events.append({
                'time': time.strftime('%H:%M:%S'),
                'elapsed': revoke_start - self.start_time,
                'type': 'REVOKE',
                'partitions': revoked_parts,
                'rebalance_num': self.rebalance_count,
            })
        
        # ─────────────────────────────────────────────────
        # CRITICAL: Commit offsets for revoked partitions
        # This minimizes reprocessing after rebalance
        # (Project 6 — commit before losing partition)
        # ─────────────────────────────────────────────────
        try:
            if partitions:
                consumer.commit(asynchronous=False)
        except Exception as e:
            pass  # May fail if no offsets to commit
        
        revoke_duration = (time.time() - revoke_start) * 1000
        self.total_revoke_time += revoke_duration
        
        self.current_partitions -= set(revoked_parts)
        
        print(
            f"   [{time.strftime('%H:%M:%S')}] 🔄 {self.name:>12} "
            f"REVOKE  ← P{revoked_parts} "
            f"({revoke_duration:.1f}ms) "
            f"[rebalance #{self.rebalance_count}]"
        )
    
    def on_assign(self, consumer, partitions):
        """Called when partitions are ASSIGNED"""
        assign_start = time.time()
        
        assigned_parts = sorted([p.partition for p in partitions])
        
        # Calculate rebalance duration (time between revoke and assign)
        rebalance_duration = None
        if self.last_revoke_time:
            rebalance_duration = (assign_start - self.last_revoke_time) * 1000
        
        with self._lock:
            self.events.append({
                'time': time.strftime('%H:%M:%S'),
                'elapsed': assign_start - self.start_time,
                'type': 'ASSIGN',
                'partitions': assigned_parts,
                'rebalance_duration_ms': rebalance_duration,
            })
        
        self.current_partitions = set(assigned_parts)
        
        assign_duration = (time.time() - assign_start) * 1000
        self.total_assign_time += assign_duration
        
        duration_str = f" rebalance={rebalance_duration:.0f}ms" if rebalance_duration else ""
        
        print(
            f"   [{time.strftime('%H:%M:%S')}] 🔄 {self.name:>12} "
            f"ASSIGN  → P{assigned_parts} "
            f"({assign_duration:.1f}ms){duration_str}"
        )
    
    def record_message(self):
        with self._lock:
            self.messages_processed += 1
    
    def print_summary(self):
        elapsed = time.time() - self.start_time
        rate = self.messages_processed / max(elapsed, 0.001)
        
        print(f"\n   📊 {self.name} Summary:")
        print(f"      Duration           : {elapsed:.1f}s")
        print(f"      Messages processed : {self.messages_processed:,}")
        print(f"      Processing rate    : {rate:.0f} msgs/sec")
        print(f"      Rebalances         : {self.rebalance_count}")
        print(f"      Total revoke time  : {self.total_revoke_time:.1f}ms")
        print(f"      Total assign time  : {self.total_assign_time:.1f}ms")
        print(f"      Current partitions : P{sorted(self.current_partitions)}")
        
        # Assignment stability analysis
        assign_events = [e for e in self.events if e['type'] == 'ASSIGN']
        if len(assign_events) >= 2:
            first = set(assign_events[0]['partitions'])
            last = set(assign_events[-1]['partitions'])
            kept = first & last
            lost = first - last
            gained = last - first
            
            print(f"      Assignment changes :")
            print(f"         First assignment : P{sorted(first)}")
            print(f"         Final assignment : P{sorted(last)}")
            print(f"         Kept throughout  : P{sorted(kept)}")
            print(f"         Lost             : P{sorted(lost)}")
            print(f"         Gained           : P{sorted(gained)}")
            
            stability = len(kept) / max(len(first), 1) * 100
            print(f"         Stability score  : {stability:.0f}%")


def create_rebalance_consumer(name, group_id, strategy='range',
                                session_timeout=10000, max_poll_interval=300000):
    """

    """
    
    # Map strategy names to Kafka config values

        'range': 'range',
        'roundrobin': 'roundrobin',
        'cooperative-sticky': 'cooperative-sticky',
    }
    
    config = {
        'bootstrap.servers': 'localhost:9092',
        'group.id': group_id,
        'client.id': f'worker-{name}',
        'auto.offset.reset': 'earliest',
        'enable.auto.commit': False,
        'session.timeout.ms': session_timeout,
        'heartbeat.interval.ms': 3000,
        'max.poll.interval.ms': max_poll_interval,
        'partition.assignment.strategy': strategy_map.get(strategy, strategy),
    }
    
    consumer = Consumer(config)
    tracker = RebalanceTracker(name)

        ['assembly-line'],
        on_assign=lambda c, p: tracker.on_assign(c, p),
        on_revoke=lambda c, p: tracker.on_revoke(c, p),
    )
    
    return consumer, tracker

# ═══════════════════════════════════════════════════════════
# file: resilient_consumer.py
# Production-grade consumer that handles rebalances gracefully:
# - Commits offsets in on_revoke
# - Persists partition state in on_revoke
# - Restores state in on_assign
# - Tracks rebalance metrics
# ═══════════════════════════════════════════════════════════

from confluent_kafka import Consumer, KafkaError, TopicPartition
import json
import time
import sys

class ResilientConsumer:
    """
    Production-grade rebalance-resilient consumer
    Manages per-partition state across rebalances
    """
    
    def __init__(self, name, group_id):
        self.name = name
        self.group_id = group_id
        
        self.config = {
            'bootstrap.servers': 'localhost:9092',
            'group.id': group_id,
            'client.id': f'resilient-{name}',
            'auto.offset.reset': 'earliest',
            'enable.auto.commit': False,
            'session.timeout.ms': 15000,
            'heartbeat.interval.ms': 3000,
            'max.poll.interval.ms': 60000,
            'partition.assignment.strategy': 'cooperative-sticky',
        }
        
        self.consumer = Consumer(self.config)
        
        # Per-partition state (simulated aggregation buffers)
        self.partition_state = {}
        
        # Rebalance metrics
        self.rebalance_count = 0
        self.rebalance_events = []
        self.messages_processed = 0
        self.commits = 0
        self.state_saves = 0
        self.state_restores = 0
        
        # External state store (simulated — in production use DB/Redis)
        self.external_store = {}
        
        self.consumer.subscribe(
            ['assembly-line'],
            on_assign=self._on_assign,
            on_revoke=self._on_revoke,
        )
    
    def _on_revoke(self, consumer, partitions):
        """
        Called BEFORE partitions are taken away
        CRITICAL: This is your last chance to save state
        """
        revoke_start = time.time()
        revoked = sorted([p.partition for p in partitions])
        self.rebalance_count += 1
        
        print(f"\n   [{time.strftime('%H:%M:%S')}] ⚠️  {self.name} REVOKE P{revoked}")
        
        # ─────────────────────────────────────────────────
        # STEP 1: Commit offsets for revoked partitions
        # (Project 6 — commit before losing partition)
        # ─────────────────────────────────────────────────
        try:
            consumer.commit(asynchronous=False)
            self.commits += 1
            print(f"   [{time.strftime('%H:%M:%S')}]    ✅ Offsets committed")
        except Exception as e:
            print(f"   [{time.strftime('%H:%M:%S')}]    ⚠️  Commit failed: {e}")
        
        # ─────────────────────────────────────────────────
        # STEP 2: Save per-partition state to external store
        # In production: write to database, Redis, or state topic
        # ─────────────────────────────────────────────────
        for p in revoked:
            if p in self.partition_state:
                state = self.partition_state[p]
                self.external_store[p] = state.copy()
                self.state_saves += 1
                
                print(
                    f"   [{time.strftime('%H:%M:%S')}]    💾 Saved state for P{p}: "
                    f"count={state.get('count', 0)}, "
                    f"last_seq={state.get('last_seq', 'N/A')}"
                )
                
                # Clean up in-memory state
                del self.partition_state[p]
        
        revoke_duration = (time.time() - revoke_start) * 1000
        
        self.rebalance_events.append({
            'type': 'REVOKE',
            'time': time.strftime('%H:%M:%S'),
            'partitions': revoked,
            'duration_ms': revoke_duration,
        })
        
        print(f"   [{time.strftime('%H:%M:%S')}]    ⏱️  Revoke completed in {revoke_duration:.1f}ms")
    
    def _on_assign(self, consumer, partitions):
        """
        Called AFTER partitions are assigned
        Restore state for newly assigned partitions
        """
        assign_start = time.time()
        assigned = sorted([p.partition for p in partitions])
        
        print(f"\n   [{time.strftime('%H:%M:%S')}] ✅ {self.name} ASSIGN P{assigned}")
        
        # ─────────────────────────────────────────────────
        # Restore state for assigned partitions
        # ─────────────────────────────────────────────────
        for p in assigned:
            if p in self.external_store:
                # Restore from external store
                self.partition_state[p] = self.external_store[p].copy()
                self.state_restores += 1
                
                print(
                    f"   [{time.strftime('%H:%M:%S')}]    🔄 Restored state for P{p}: "
                    f"count={self.partition_state[p].get('count', 0)}"
                )
            elif p not in self.partition_state:
                # Initialize fresh state
                self.partition_state[p] = {
                    'count': 0,
                    'last_seq': None,
                    'first_assigned': time.time(),
                    'products': {},
                }
                print(f"   [{time.strftime('%H:%M:%S')}]    🆕 New state for P{p}")
        
        assign_duration = (time.time() - assign_start) * 1000
        
        self.rebalance_events.append({
            'type': 'ASSIGN',
            'time': time.strftime('%H:%M:%S'),
            'partitions': assigned,
            'duration_ms': assign_duration,
        })
        
        print(f"   [{time.strftime('%H:%M:%S')}]    ⏱️  Assign completed in {assign_duration:.1f}ms\n")
    
    def process_message(self, msg):
        """Process a single message and update partition state"""
        partition = msg.partition()
        
        try:
            payload = json.loads(msg.value().decode('utf-8'))
        except (json.JSONDecodeError, UnicodeDecodeError):
            return
        
        # Update partition-specific state
        if partition not in self.partition_state:
            self.partition_state[partition] = {
                'count': 0,
                'last_seq': None,
                'first_assigned': time.time(),
                'products': {},
            }
        
        state = self.partition_state[partition]
        state['count'] += 1
        state['last_seq'] = payload.get('seq', None)
        
        product = payload.get('product', 'unknown')
        state['products'][product] = state['products'].get(product, 0) + 1
        
        self.messages_processed += 1
    
    def run(self, duration=30, commit_interval=50):
        """Main processing loop"""
        
        print(f"   🏭 {self.name} starting...")
        print(f"      Group: {self.group_id}")
        print(f"      Strategy: cooperative-sticky")
        print(f"      Duration: {duration}s")
        print(f"      Commit every: {commit_interval} messages\n")
        
        start = time.time()
        batch_count = 0
        
        try:
            while (time.time() - start) < duration:
                msg = self.consumer.poll(timeout=1.0)
                
                if msg is None:
                    continue
                
                if msg.error():
                    if msg.error().code() == KafkaError._PARTITION_EOF:
                        continue
                    print(f"   ❌ Error: {msg.error()}")
                    continue
                
                # Process
                self.process_message(msg)
                batch_count += 1
                
                # Periodic commit (Project 6 batch commit pattern)
                if batch_count >= commit_interval:
                    self.consumer.commit(asynchronous=False)
                    self.commits += 1
                    batch_count = 0
                
                # Progress logging
                if self.messages_processed % 200 == 0:
                    elapsed = time.time() - start
                    rate = self.messages_processed / max(elapsed, 0.001)
                    parts = sorted(self.partition_state.keys())
                    print(
                        f"   [{time.strftime('%H:%M:%S')}] {self.name:>10}: "
                        f"{self.messages_processed:>5} msgs ({rate:.0f}/s) "
                        f"partitions=P{parts}"
                    )
        
        except KeyboardInterrupt:
            print(f"\n   🛑 {self.name} interrupted")
        
        finally:
            # Final commit
            if batch_count > 0:
                try:
                    self.consumer.commit(asynchronous=False)
                    self.commits += 1
                except Exception:
                    pass
            
            self.consumer.close()
    
    def print_report(self):
        """Print comprehensive session report"""
        print(f"\n   {'═' * 60}")
        print(f"   📊 {self.name} — SESSION REPORT")
        print(f"   {'═' * 60}")
        
        print(f"\n   PROCESSING:")
        print(f"      Messages processed : {self.messages_processed:,}")
        print(f"      Commits            : {self.commits}")
        
        print(f"\n   REBALANCES:")
        print(f"      Total rebalances   : {self.rebalance_count}")
        print(f"      State saves        : {self.state_saves}")
        print(f"      State restores     : {self.state_restores}")
        
        print(f"\n   REBALANCE TIMELINE:")
        for event in self.rebalance_events:
            icon = "📤" if event['type'] == 'REVOKE' else "📥"
            print(
                f"      {event['time']} {icon} {event['type']:>6} "
                f"P{event['partitions']} ({event['duration_ms']:.1f}ms)"
            )
        
        print(f"\n   PARTITION STATE:")
        for p, state in sorted(self.partition_state.items()):
            print(f"      P{p}: {state['count']} msgs, last_seq={state['last_seq']}")
            if state['products']:
                top_products = sorted(
                    state['products'].items(), key=lambda x: -x[1]
                )[:3]
                products_str = ", ".join(f"{k}={v}" for k, v in top_products)
                print(f"           top products: {products_str}")
        
        print(f"\n   EXTERNAL STORE (persisted state):")
        for p, state in sorted(self.external_store.items()):
            print(f"      P{p}: {state['count']} msgs (saved)")
