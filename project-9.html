  │  STRATEGY 3: No key, no partition (default partitioner)          │

  │                                                                   │
  │  STRATEGY 4: Weighted distribution (custom logic)                │

  └───────────────────────────────────────────────────────────────────┘

✅ WHAT SHOULD DO

1. ALWAYS understand the partition selection precedence
   → Explicit partition > Key hash > Default partitioner
   → If you pass partition=3, the key does NOT affect routing
   → The key is STILL stored in the message for consumer use

2. ALWAYS verify actual distribution after producing
   → Use watermark offsets to count messages per partition (Project 8 CLI)
   → Build distribution monitoring into your producer
   → Alert on unexpected skew

3. ALWAYS use explicit partition ONLY when business logic requires it
   → Priority routing (express vs standard)
   → Testing specific partitions under load
   → Regulatory requirements (data locality per partition)
   → For most cases, key-based routing (Project 4) is sufficient

4. ALWAYS account for partition count in your explicit routing
   → If topic has 6 partitions, valid values are 0-5
   → Producing to partition 6 → ERROR (out of range)
   → Dynamically discover partition count via metadata API

5. ALWAYS combine explicit partition with meaningful keys when needed
   → partition=0, key="express-cust-123"
   → Consumer reads from partition 0 and sees the key
   → Key is metadata — partition controls placement

❌ WHAT SHOULD NOT DO

1. DO NOT hardcode partition numbers without checking topic metadata
   → Topic might have fewer partitions than you expect
   → Produces to non-existent partition → exception
   → Always check partition count first

2. DO NOT use explicit partition for entity ordering
   → key=customer_id is the right tool for per-entity ordering (Project 4)
   → Explicit partition ties you to a specific number
   → If partition count changes, your logic breaks

3. DO NOT mix explicit partition and key-based routing carelessly
   → If some producers use explicit partition and others use key
   → Same key might end up in DIFFERENT partitions
   → Consumer expecting all key-X messages in one partition is confused

4. DO NOT use round-robin when per-entity ordering matters
   → Round-robin spreads messages across ALL partitions
   → Same entity's messages scatter everywhere
   → Per-entity ordering LOST (Project 4 explained this)

5. DO NOT assume sticky partitioner gives perfect distribution
   → Sticky fills one partition's batch, then moves to next
   → Short-lived producers may not rotate enough
   → Long-running producers achieve good distribution over time

6. DO NOT produce to partition 0 for ALL special cases
   → "Just send everything important to partition 0"
   → Partition 0 becomes a hot partition
   → Other partitions underutilized

→ DO NOT use explicit partition when you might change partition count later
  (your hardcoded partition numbers become invalid or suboptimal)

→ DO NOT use weighted distribution when even distribution is sufficient
  (adds complexity without benefit — KISS principle)

→ DO NOT use round-robin for stateful processing
  (consumer building state per entity needs all entity events — use keys)

⚙️ HOW IT ONLY WORKS — INTERNAL MECHANICS

═══════════════════════════════════════════════════════════════
THE PRODUCE() CALL — COMPLETE PARTITION SELECTION PATH
═══════════════════════════════════════════════════════════════

When you call:
  producer.produce(topic="warehouse", key=b"cust-1", value=b"...", partition=2)

STEP 1: PARTITION PARAMETER CHECK
──────────────────────────────────
  → Producer checks: was partition explicitly provided?
  → partition=2 → YES
  → SKIP key hashing entirely
  → SKIP default partitioner entirely
  → Target partition = 2

  → If partition was NOT provided (partition=None or omitted):
    → Check key...

STEP 2: KEY CHECK (only if partition not specified)
───────────────────────────────────────────────────
  → key=b"cust-1" → NOT null
  → Compute: murmur2(b"cust-1") % num_partitions
  → Target partition = hash result

  → If key is also None:
    → Use default partitioner (sticky/round-robin)

STEP 3: RECORD ACCUMULATOR
───────────────────────────
  → Message placed in batch for (topic="warehouse", partition=2)
  → Batch accumulates until batch.size or linger.ms triggers
  → Key "cust-1" is STORED in the message record
  → But it was NOT used for partition selection (partition=2 was explicit)


═══════════════════════════════════════════════════════════════
STICKY PARTITIONER — HOW IT REALLY WORKS
═══════════════════════════════════════════════════════════════

When key=None and partition=None (default partitioner):

  Kafka 2.4+ uses the "Sticky Partitioner":

  CONCEPT:
  → Pick a partition
  → Keep sending to THAT partition until the batch is full or linger.ms expires
  → Then pick a DIFFERENT partition
  → Repeat

  WHY "STICKY"?
  → Traditional round-robin: msg1→P0, msg2→P1, msg3→P2, msg4→P0...
    → Every message in a different batch → many small batches → many requests
  
  → Sticky: msg1→P0, msg2→P0, msg3→P0 (batch fills), msg4→P1, msg5→P1...
    → Messages batched together → fewer, larger batches → fewer requests

  TIMELINE:

  ┌────────────────────────────────────────────────────────────────┐
  │ ROUND-ROBIN (old):                                             │
  │ msg1→P0 | msg2→P1 | msg3→P2 | msg4→P0 | msg5→P1 | msg6→P2   │
  │ Batches: [msg1]P0 [msg2]P1 [msg3]P2 [msg4]P0 [msg5]P1 [msg6]P2│
  │ = 6 batches for 6 messages (1 per batch)                       │
  │ = 6 network requests                                           │
  │                                                                │
  │ STICKY (new):                                                  │
  │ msg1→P0 | msg2→P0 | msg3→P0 | msg4→P1 | msg5→P1 | msg6→P1   │
  │ Batches: [msg1,msg2,msg3]P0 [msg4,msg5,msg6]P1                │
  │ = 2 batches for 6 messages (3 per batch)                       │
  │ = 2 network requests (3x more efficient!)                      │
  └────────────────────────────────────────────────────────────────┘

  WHEN DOES STICKY SWITCH PARTITIONS?
  → When batch for current partition reaches batch.size (default 16KB)
  → When linger.ms timer expires (default 0ms)
  → When flush() or poll(0) is called

  IMPLICATION FOR SHORT-LIVED PRODUCERS:
  → If you produce 10 messages and immediately flush()
  → Sticky might put ALL 10 in ONE partition
  → Over time (long-running producer), distribution evens out
  → For testing, this creates UNEXPECTED skew


═══════════════════════════════════════════════════════════════
EXPLICIT PARTITION — WHAT THE BROKER SEES
═══════════════════════════════════════════════════════════════

  ProduceRequest from producer to broker:

  ┌──────────────────────────────────────────────────────────┐
  │ ProduceRequest:                                          │
  │   topic: "warehouse"                                     │
  │   partition: 2          ← explicitly set by producer     │
  │   records: [                                             │
  │     {key: "cust-1", value: "package-data", ...}          │
  │   ]                                                      │
  └──────────────────────────────────────────────────────────┘

  → Broker receives request for partition 2
  → Broker checks: is partition 2 valid? (exists and has a leader?)
  → If valid: appends to partition 2's log → returns offset
  → If invalid: returns error (UnknownTopicOrPartitionException)

  THE BROKER DOES NOT RE-ROUTE:
  → Broker trusts the producer's partition choice
  → No second-guessing, no key re-hashing
  → Producer said partition 2 → broker writes to partition 2
  → This is why explicit partition is POWERFUL but DANGEROUS

PARTITION COUNT DISCOVERY
═══════════════════════════════════════════════════════════════

  Before producing to explicit partitions, discover valid range:

  1. PRODUCER METADATA REFRESH
     → Producer automatically fetches topic metadata on first produce
     → Metadata includes partition count
     → Cached and refreshed periodically (metadata.max.age.ms)

  2. ADMIN API
     → AdminClient.list_topics() returns partition info
     → Use this before starting production

  3. WHAT HAPPENS IF PARTITION COUNT CHANGES MID-PRODUCTION?
     → Producer's metadata cache becomes stale
     → Producer might not know about new partitions
     → metadata.max.age.ms (default 5 min) controls refresh interval
     → Explicit partition=7 on a topic that grew from 6→8:
       → Works if metadata refreshed
       → Fails if metadata stale (still thinks 6 partitions)

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "KafkaException: partition 6 does not exist"
├── CAUSE: Topic has fewer partitions than the explicit value
├── EXAMPLE: Topic has 4 partitions (0-3), you specified partition=6
├── FIX: Check partition count via metadata before producing
└── PREVENTION: Always use partition % num_partitions for safety

PITFALL 2: "Sticky partitioner puts everything in one partition"
├── CAUSE: Short-lived producer with small batch count
├── WHY: Sticky fills one partition's batch before moving to next
│         If you produce 5 messages and flush, all 5 may go to P0
├── SYMPTOM: Distribution looks completely skewed in tests
└── FIX: For testing, use explicit round-robin or produce many messages
         For production, long-running producers distribute well over time

PITFALL 3: "Key is set but messages go to wrong partition"
├── CAUSE: Explicit partition parameter OVERRIDES key routing
├── WHY: Precedence is partition > key > default
├── EXAMPLE: produce(key="cust-1", partition=3)
│   → Goes to partition 3 regardless of hash("cust-1")
│   → Key "cust-1" is stored but NOT used for routing
└── FIX: Don't pass both key and partition unless intentional
         If using key routing, omit partition parameter

PITFALL 4: "Weighted distribution is uneven after short test"
├── CAUSE: Statistical variance with small sample sizes
├── WHY: 100 messages with 60/40 split might show 55/45 or 65/35
├── FIX: Produce 10,000+ messages for reliable distribution measurement
└── STATS: Standard deviation decreases with √N (more samples = more accurate)

PITFALL 5: "Consumer group rebalance after partition count increase"
├── CAUSE: New partitions trigger consumer group rebalance (Project 5)
├── SCENARIO: Topic goes from 4→8 partitions
│   → Producers can write to new partitions immediately
│   → Consumers need rebalance to discover new partitions
│   → Brief pause in consumption during rebalance
└── FIX: Coordinate partition increase with consumer restart plan

PITFALL 6: "Explicit partition breaks after topic recreation"
├── CAUSE: Recreated topic might have different partition count
├── EXAMPLE: Deleted topic with 6 partitions, recreated with 4
│   → Code still produces to partition 5 → ERROR
└── FIX: Always discover partition count dynamically, never hardcode

📈 SCALABILITY CONSIDERATIONS

EXPLICIT PARTITION OVERHEAD:
→ Zero additional overhead vs key-based or default partitioning
→ Partition selection happens in producer memory (no network call)
→ Actually slightly FASTER than key-based (no hash computation)

STICKY vs ROUND-ROBIN THROUGHPUT:
  ┌──────────────────────┬──────────────────────┬──────────────────────┐
  │ Strategy             │ Batching Efficiency  │ Network Requests     │
  ├──────────────────────┼──────────────────────┼──────────────────────┤
  │ Round-robin          │ Poor (1 msg/batch)   │ High (1 per message) │
  │ Sticky               │ Excellent (N/batch)  │ Low (1 per batch)    │
  │ Explicit partition    │ Depends on pattern   │ Depends on pattern   │
  └──────────────────────┴──────────────────────┴──────────────────────┘

  → Sticky partitioner improves throughput by 2-5x over round-robin
  → Explicit partition with uniform distribution ≈ round-robin performance
  → Explicit partition with skewed distribution → hot partition risk

WEIGHTED DISTRIBUTION AT SCALE:
→ 70/30 split across 6 partitions at 100K msgs/sec:
  → Express (P0-P1): ~35K msgs/sec each
  → Standard (P2-P5): ~7.5K msgs/sec each
  → Express partitions need 4.5x consumer capacity vs standard
  → Plan consumer scaling accordingly (Project 5 — consumer per partition)

🔍 INTERPRETABILITY CONSIDERATIONS

HOW DO YOU VERIFY DISTRIBUTION?

1. WATERMARK OFFSET COMPARISON
   → Check high watermark for each partition
   → Even distribution: all partitions have similar high watermarks
   → Skewed distribution: some partitions much higher than others

2. DISTRIBUTION METRICS IN PRODUCER
   → Track partition assignment in delivery callback
   → Compute: count per partition, percentage, skew coefficient
   → Alert when skew exceeds threshold

3. CLI VERIFICATION (Project 8 tools)
   → kafka-consumer-groups --describe shows LOG-END-OFFSET per partition
   → Compare across partitions for distribution analysis

4. STATISTICAL MEASURES
   → Standard deviation of message counts across partitions
   → Coefficient of variation (CV) = std_dev / mean
   → CV < 0.1 = very even, CV > 0.3 = significantly skewed

5. REAL-TIME MONITORING
   → Track distribution over sliding windows (last 1000 messages)
   → Detect distribution drift as patterns change

# ═══════════════════════════════════════════════════════════
# file: distribution_tracker.py
# Reusable distribution tracking and analysis utility
# Used by all experiments in this project
# ═══════════════════════════════════════════════════════════

import math
import time

class DistributionTracker:
    """
    Tracks message distribution across partitions
    and computes statistical measures of evenness
    """
    
    def __init__(self, num_partitions):
        self.num_partitions = num_partitions
        self.counts = {p: 0 for p in range(num_partitions)}
        self.total = 0
        self.start_time = time.time()
    
    def record(self, partition):
        """Record a message delivered to a partition"""
        self.counts[partition] = self.counts.get(partition, 0) + 1
        self.total += 1
    
    def get_percentages(self):
        """Get percentage distribution per partition"""
        if self.total == 0:
            return {p: 0.0 for p in range(self.num_partitions)}
        return {p: (self.counts[p] / self.total) * 100
                for p in range(self.num_partitions)}
    
    def get_expected_percentage(self):
        """Expected even distribution percentage"""
        return 100.0 / self.num_partitions
    
    def get_std_deviation(self):
        """Standard deviation of counts (lower = more even)"""
        if self.total == 0:
            return 0.0
        mean = self.total / self.num_partitions
        variance = sum((self.counts[p] - mean) ** 2
                       for p in range(self.num_partitions)) / self.num_partitions
        return math.sqrt(variance)
    
    def get_coefficient_of_variation(self):
        """CV = std_dev / mean (0 = perfect, >0.3 = skewed)"""
        if self.total == 0:
            return 0.0
        mean = self.total / self.num_partitions
        if mean == 0:
            return 0.0
        return self.get_std_deviation() / mean
    
    def get_skew_score(self):
        """
        Skew score: ratio of max to min partition count
        1.0 = perfect, >2.0 = significantly skewed
        """
        if self.total == 0:
            return 1.0
        non_zero = [c for c in self.counts.values() if c > 0]
        if not non_zero:
            return 1.0
        return max(non_zero) / max(min(non_zero), 1)
    
    def print_report(self, title="DISTRIBUTION REPORT"):
        """Print a formatted distribution report"""
        pcts = self.get_percentages()
        expected = self.get_expected_percentage()
        elapsed = time.time() - self.start_time
        rate = self.total / max(elapsed, 0.001)
        
        print(f"\n📊 {title}")
        print(f"   Total messages: {self.total:,} ({rate:.0f} msgs/sec)")
        print(f"   Expected even : {expected:.1f}% per partition\n")
        
        max_count = max(self.counts.values()) if self.counts else 1
        bar_scale = 40 / max(max_count, 1)
        
        print(f"   {'Part':>4} | {'Count':>8} | {'Pct':>6} | {'Deviation':>9} | Distribution")
        print(f"   {'─' * 65}")
        
        for p in range(self.num_partitions):
            count = self.counts[p]
            pct = pcts[p]
            deviation = pct - expected
            bar_len = int(count * bar_scale)
            bar = "█" * bar_len
            
            dev_icon = "✅" if abs(deviation) < 3 else "⚠️" if abs(deviation) < 10 else "🔥"
            
            print(
                f"   P{p:>2}  | {count:>8,} | {pct:>5.1f}% | "
                f"{deviation:>+7.1f}% {dev_icon} | {bar}"
            )
        
        print(f"\n   STATISTICAL MEASURES:")
        print(f"   Std Deviation    : {self.get_std_deviation():.2f}")
        print(f"   Coeff. Variation : {self.get_coefficient_of_variation():.4f} "
              f"({'even' if self.get_coefficient_of_variation() < 0.1 else 'skewed'})")
        print(f"   Skew Score       : {self.get_skew_score():.2f} "
              f"({'balanced' if self.get_skew_score() < 1.5 else 'imbalanced'})")
        
        # Overall assessment
        cv = self.get_coefficient_of_variation()
        if cv < 0.05:
            print(f"\n   ✅ EXCELLENT: Near-perfect distribution")
        elif cv < 0.1:
            print(f"\n   ✅ GOOD: Acceptable distribution")
        elif cv < 0.3:
            print(f"\n   ⚠️  MODERATE: Some skew detected")
        else:
            print(f"\n   🔥 SEVERE: Significant skew — investigate!")

EXPERIMENT 4: Safe Partition Producer with Metadata Discovery

# ═══════════════════════════════════════════════════════════
# file: safe_partition_producer.py
# Production-grade explicit partition producer with:
# - Dynamic partition count discovery
# - Boundary checking
# - Fallback strategies
# - Distribution monitoring
# ═══════════════════════════════════════════════════════════

from confluent_kafka import Producer, KafkaException
from distribution_tracker import DistributionTracker
import json
import time
import random

class SafePartitionProducer:
    """
    Production-grade producer with safe explicit partition assignment
    Discovers partitions dynamically and validates all assignments
    """
    
    def __init__(self, bootstrap_servers, topic):
        self.config = {
            'bootstrap.servers': bootstrap_servers,
        }
        self.topic = topic
        self.producer = Producer(self.config)
        self.num_partitions = self._discover_partitions()
        self.tracker = DistributionTracker(self.num_partitions)
        self.error_count = 0
        self.success_count = 0
    
    def _discover_partitions(self):
        """Dynamically discover partition count for the topic"""
        try:
            metadata = self.producer.list_topics(topic=self.topic, timeout=10)
            topic_meta = metadata.topics.get(self.topic)
            
            if topic_meta is None:
                raise ValueError(f"Topic '{self.topic}' not found")
            
            if topic_meta.error is not None:
                raise ValueError(f"Topic error: {topic_meta.error}")
            
            count = len(topic_meta.partitions)
            print(f"   📡 Discovered {count} partitions for topic '{self.topic}'")
            return count
            
        except Exception as e:
            print(f"   ❌ Failed to discover partitions: {e}")
            raise
    
    def _validate_partition(self, partition):
        """Validate partition number is within valid range"""
        if partition is None:
            return True
        if not isinstance(partition, int):
            return False
        return 0 <= partition < self.num_partitions
    
    def _safe_partition(self, partition):
        """
        Make partition safe:
        - If valid, return as-is
        - If out of range, wrap with modulo
        - If None, return None (use default partitioner)
        """
        if partition is None:
            return None
        if self._validate_partition(partition):
            return partition
        # Wrap to valid range
        safe = partition % self.num_partitions
        print(f"   ⚠️  Partition {partition} out of range → wrapped to {safe}")
        return safe
    
    def _delivery_callback(self, err, msg):
        if err:
            self.error_count += 1
        else:
            self.success_count += 1
            self.tracker.record(msg.partition())
    
    def produce(self, key=None, value=None, partition=None, headers=None):
        """
        Produce with safe partition validation
        """
        safe_partition = self._safe_partition(partition)
        
        produce_args = {
            'topic': self.topic,
            'value': value,
            'callback': self._delivery_callback,
        }
        
        if key is not None:
            produce_args['key'] = key
        
        if safe_partition is not None:
            produce_args['partition'] = safe_partition
        
        if headers is not None:
            produce_args['headers'] = headers
        
        try:
            self.producer.produce(**produce_args)
            self.producer.poll(0)
        except KafkaException as e:
            self.error_count += 1
            print(f"   ❌ Produce error: {e}")
    
    def flush(self):
        self.producer.flush()
    
    def print_stats(self):
        print(f"\n📊 PRODUCER STATS:")
        print(f"   Successes : {self.success_count:,}")
        print(f"   Errors    : {self.error_count:,}")
        print(f"   Partitions: {self.num_partitions}")
        self.tracker.print_report("PARTITION DISTRIBUTION")
    
    def refresh_partitions(self):
        """Re-discover partitions (call if topic was altered)"""
        old_count = self.num_partitions
        self.num_partitions = self._discover_partitions()
        if self.num_partitions != old_count:
            print(f"   🔄 Partition count changed: {old_count} → {self.num_partitions}")
            self.tracker = DistributionTracker(self.num_partitions)

            if msg.error().code() == KafkaError._PARTITION_EOF:
                continue
            continue
        
        empty_polls = 0
        messages_read += 1
        tracker.record(msg.partition())
        
        # Try to parse strategy from message
        try:
            payload = json.loads(msg.value().decode('utf-8'))
            strategy = payload.get('strategy', payload.get('priority', 'unknown'))
            strategy_counts[strategy] = strategy_counts.get(strategy, 0) + 1
        except (json.JSONDecodeError, UnicodeDecodeError):
            strategy_counts['unparseable'] = strategy_counts.get('unparseable', 0) + 1

except KeyboardInterrupt:
    pass

finally:
    consumer.close()

    pct = count / max(messages_read, 1) * 100
    bar = "█" * int(pct / 2)
    print(f"   {strategy:<20} {count:>6} ({pct:>5.1f}%)  {bar}")

def run_strategy(name, produce_fn):
    """
    Run a strategy and return distribution tracker
    produce_fn(producer, topic, i) → called for each message
    """
    config = {
        'bootstrap.servers': 'localhost:9092',
        'linger.ms': 5,
    }
    
    producer = Producer(config)
    tracker = DistributionTracker(num_partitions)
    
    def callback(err, msg):
        if err:
            return
        tracker.record(msg.partition())
    
    start = time.perf_counter()
    
    for i in range(messages_per_test):
        produce_fn(producer, topic, i, callback)
        producer.poll(0)
    
    producer.flush()
    elapsed = time.perf_counter() - start
    
    return tracker, elapsed

def strategy_sticky(producer, topic, i, callback):
    """No key, no partition → sticky partitioner"""
    producer.produce(
        topic=topic,
        value=json.dumps({'s': 'sticky', 'i': i}).encode('utf-8'),
        callback=callback,
    )

def strategy_key_based(producer, topic, i, callback):
    """Key-based routing (Project 4 approach)"""
    key = f'customer-{i % 50}'.encode('utf-8')
    producer.produce(
        topic=topic,
        key=key,
        value=json.dumps({'s': 'key', 'i': i}).encode('utf-8'),
        callback=callback,
    )

def strategy_roundrobin(producer, topic, i, callback):
    """Explicit round-robin across all partitions"""
    producer.produce(
        topic=topic,
        value=json.dumps({'s': 'rr', 'i': i}).encode('utf-8'),
        partition=i % num_partitions,
        callback=callback,
    )

def strategy_weighted(producer, topic, i, callback):
    """Weighted: 40% to P0-P1, 60% to P2-P5"""
    if random.random() < 0.40:
        partition = random.choice([0, 1])
    else:
        partition = random.choice([2, 3, 4, 5])
    
    producer.produce(
        topic=topic,
        value=json.dumps({'s': 'weighted', 'i': i}).encode('utf-8'),
        partition=partition,
        callback=callback,
    )

def strategy_single(producer, topic, i, callback):
    """All messages to partition 0 (stress test / hot partition)"""
    producer.produce(
        topic=topic,
        value=json.dumps({'s': 'single', 'i': i}).encode('utf-8'),
        partition=0,
        callback=callback,
    )
