🔵 HOW TO APPROACH IT

Step 1: Understand the problem Kafka Connect solves
  → Moving data between Kafka and external systems is EXTREMELY common
  → Every company needs: DB → Kafka, Kafka → DB, Files → Kafka, etc.
  → Writing custom producers/consumers (Project 12) works but:
    → You must handle offset tracking yourself
    → You must handle failure recovery yourself
    → You must handle serialization yourself
    → You must handle scaling yourself
    → Every integration reinvents the wheel

Step 2: Recognize Kafka Connect as a FRAMEWORK, not a tool
  → It provides the infrastructure (workers, offset management, config)
  → CONNECTORS provide the integration logic (read from X, write to Y)
  → You configure connectors — you don't write producers/consumers
  → 200+ pre-built connectors exist for databases, files, cloud, etc.

Step 3: Understand the two deployment modes
  → STANDALONE: Single worker process, local offset storage
    → Good for development, testing, simple pipelines
  → DISTRIBUTED: Multiple worker processes, Kafka-based offset storage
    → Good for production, fault-tolerant, scalable

Step 4: Connect to what we already know
  → Project 12: We wrote Python producers/consumers manually
    → Connect replaces that for standard integrations
  → Project 13: We designed structured messages
    → Connect uses converters (JSON, Avro) for serialization
  → Project 14: We learned about compacted topics
    → Connect uses internal compacted topics for offset storage
  → Project 11: Delivery guarantees
    → Connect provides at-least-once by default, exactly-once possible

🟡 WHAT COULD BE DONE

OPTION A: Write custom producers/consumers (Project 12 approach)
  → Full control over every detail
  → Must handle offset tracking, recovery, serialization
  → Must build monitoring and management yourself
  → Every integration is custom code to maintain
  → Good for: unique integrations, complex transformations
  → Bad for: standard file/DB ingestion (reinventing the wheel)

OPTION B: Kafka Connect Standalone Mode
  → Single worker process on one machine
  → Offsets stored in local file
  → Configuration via properties files
  → Simple to set up and understand
  → No fault tolerance (worker dies → pipeline stops)
  → ✅ GOOD FOR: Development, testing, simple pipelines

OPTION C: Kafka Connect Distributed Mode
  → Multiple worker processes across machines
  → Offsets stored in Kafka topics (compacted — Project 14 knowledge!)
  → Configuration via REST API
  → Automatic fault tolerance (task failover)
  → Automatic load balancing
  → ✅ BEST FOR: Production deployments

OPTION D: Kafka Connect with managed service (Confluent Cloud, AWS MSK Connect)
  → Fully managed — no infrastructure to maintain
  → Auto-scaling, monitoring built in
  → Pay per connector/task
  → Good for: cloud-native deployments

✅ WHAT SHOULD DO

1. USE KAFKA CONNECT FOR STANDARD INTEGRATIONS
   → Files, databases, S3, Elasticsearch, etc.
   → Don't write custom code when a connector exists
   → 200+ connectors available on Confluent Hub

2. START WITH STANDALONE MODE FOR DEVELOPMENT
   → Simpler to understand and debug
   → Properties file configuration
   → Run on your local machine
   → Graduate to distributed mode for production

3. UNDERSTAND THE CONNECTOR-TASK MODEL
   → Connector: defines the integration (what to read/write)
   → Task: executes the work (one or more per connector)
   → Multiple tasks = parallel processing
   → Tasks are distributed across workers in distributed mode

4. USE JSON CONVERTER FOR DEVELOPMENT, AVRO FOR PRODUCTION
   → JsonConverter: human-readable, easy to debug
   → AvroConverter: compact, schema-enforced (Project 16)
   → StringConverter: raw strings (simplest)
   → Configure key.converter and value.converter separately

5. MONITOR CONNECTOR STATUS
   → Use REST API: GET /connectors/{name}/status
   → Check for FAILED tasks
   → Set up alerts on task failures

6. USE APPROPRIATE TRANSFORMS (SMTs)
   → Single Message Transforms modify messages in-flight
   → Add fields, rename fields, route to different topics
   → Lightweight transformations only (heavy → use Kafka Streams)
   → Covered in depth in Project 24

7. TEST WITH SMALL FILES FIRST
   → Verify serialization, topic routing, offset tracking
   → Then scale to production volumes

❌ WHAT SHOULD NOT DO

1. DO NOT USE KAFKA CONNECT FOR COMPLEX TRANSFORMATIONS
   → Connect is for MOVING data, not transforming it
   → Simple transforms (SMTs) are OK: rename field, add timestamp
   → Complex logic (joins, aggregations, filtering) → Kafka Streams (Project 19+)

2. DO NOT USE STANDALONE MODE IN PRODUCTION
   → No fault tolerance — worker death = pipeline death
   → No REST API for management
   → No automatic task redistribution
   → Use distributed mode for anything that matters

3. DO NOT FORGET TO CONFIGURE CONVERTERS
   → Default converter may not match your data format
   → key.converter and value.converter are INDEPENDENT
   → Mismatched converters = deserialization errors everywhere

4. DO NOT RUN TOO MANY TASKS PER WORKER
   → Each task uses memory and CPU
   → Overloaded worker → slow processing → consumer lag
   → Monitor worker resource usage

5. DO NOT IGNORE OFFSET MANAGEMENT
   → Source connectors track "where they left off" via offsets
   → Offset storage must be reliable (Kafka topics in distributed mode)
   → Losing offsets = reprocessing all data from the beginning

6. DO NOT USE FileStreamSource/Sink IN PRODUCTION
   → They are DEMO connectors — minimal features
   → No file rotation, no glob patterns, no proper error handling
   → For production file processing: use SpoolDir connector or custom
   → We use them here to LEARN the Connect framework

7. DO NOT MODIFY CONNECT INTERNAL TOPICS
   → connect-offsets, connect-configs, connect-status
   → These are managed by the Connect framework
   → Manual modifications can corrupt connector state

🚫 WHEN NOT TO USE

DO NOT USE KAFKA CONNECT WHEN:
  → You need complex event processing (use Kafka Streams)
  → You need custom business logic between read and write
  → The source/sink has no connector and building one is overkill
  → You need sub-millisecond latency (Connect adds overhead)
  → You're doing a one-time data migration (use a script)

DO NOT USE FileStreamSource/Sink WHEN:
  → Production workloads (use SpoolDir, S3, JDBC connectors instead)
  → You need file rotation handling
  → You need to process multiple files
  → You need exactly-once file processing
  → They exist for LEARNING and DEMO purposes only

DO NOT USE STANDALONE MODE WHEN:
  → You need fault tolerance
  → You need to manage connectors via API
  → You have multiple workers
  → You're running in production

═══════════════════════════════════════════════════════════════
              OFFSET TRACKING FOR SOURCE CONNECTORS
═══════════════════════════════════════════════════════════════

Source connectors need to track "where they left off" in the
external system. This is DIFFERENT from consumer offsets.

Example: FileStreamSource tracking position in a file

  ┌────────────────────────────────────────────────┐
  │ OFFSET STORAGE (connect-offsets topic)          │
  │                                                  │
  │ Key:   {"filename": "/data/transactions.csv"}   │
  │ Value: {"position": 4096}                       │
  │                                                  │
  │ This is a COMPACTED topic (Project 14!)         │
  │ Only the LATEST position per file is retained   │
  └────────────────────────────────────────────────┘

  On restart:
    1. Read connect-offsets topic
    2. Find last position for this file
    3. Seek to that position
    4. Resume reading from there

  This is why Connect provides AT-LEAST-ONCE:
    → Read data from source
    → Send to Kafka
    → Commit offset
    → If crash between send and commit → re-read and re-send
    → Duplicates possible (same as Project 11 at-least-once pattern)


═══════════════════════════════════════════════════════════════
              CONVERTER PIPELINE
═══════════════════════════════════════════════════════════════

Source path:
  Raw data → SourceRecord (Java object) → Converter → bytes → Kafka

Sink path:
  Kafka → bytes → Converter → SinkRecord (Java object) → External system

Converter is configured INDEPENDENTLY for key and value:

  key.converter=org.apache.kafka.connect.storage.StringConverter
  value.converter=org.apache.kafka.connect.json.JsonConverter

Common converters:
  StringConverter:    raw string (no schema)
  JsonConverter:      JSON with optional embedded schema
  AvroConverter:      Avro with Schema Registry (Project 16)
  ProtobufConverter:  Protocol Buffers with Schema Registry
  ByteArrayConverter: raw bytes (pass-through)

JsonConverter with schemas.enable=true produces:
  {
    "schema": {"type": "struct", "fields": [...]},
    "payload": {"field1": "value1", ...}
  }

JsonConverter with schemas.enable=false produces:
  {"field1": "value1", ...}

The schemas.enable=false format is what we've been using
in Projects 12-14 (plain JSON).

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "Connector keeps producing messages with schema wrapper"
  → JsonConverter has schemas.enable=true by default
  → Produces: {"schema":{...}, "payload":{...}}
  → Your consumers expect: {"field1":"value1",...}
  → FIX: Set value.converter.schemas.enable=false

PITFALL 2: "FileStreamSource doesn't detect new data in the file"
  → FileStreamSource reads from a single file
  → It does NOT watch for new files or file rotation
  → It reads continuously as data is APPENDED
  → FIX: For file watching, use SpoolDir connector
  → FileStreamSource is a DEMO connector

PITFALL 3: "Connector status shows FAILED but no error message"
  → Check Connect worker logs (not just REST API status)
  → Common cause: converter mismatch (expecting JSON, got Avro)
  → Common cause: topic doesn't exist and auto-create is disabled
  → FIX: Check worker logs: docker logs <connect-container>

PITFALL 4: "Standalone mode loses offsets on restart"
  → Offset file may be in a temporary directory
  → Container restart = file gone = reprocess everything
  → FIX: Mount offset file to persistent volume
  → Or use distributed mode (offsets in Kafka)

PITFALL 5: "Data appears duplicated in sink"
  → Connect provides at-least-once by default
  → Source crash/restart may re-send some records
  → FIX: Make sink idempotent (upsert instead of insert)
  → Or use exactly-once support (Kafka 3.3+)

PITFALL 6: "Connect worker runs out of memory"
  → Too many connectors/tasks on one worker
  → Large messages filling internal buffers
  → FIX: Increase worker heap size
  → FIX: Reduce tasks.max per connector
  → FIX: Limit max.message.bytes

PITFALL 7: "Can't find the connector plugin"
  → Connector JAR not in the plugin path
  → plugin.path must point to directory containing connector JARs
  → FIX: Verify plugin.path in worker config
  → FIX: Check that JAR files are in the correct directory structure

📈 SCALABILITY CONSIDERATIONS

TASK PARALLELISM:
  → tasks.max controls how many tasks run per connector
  → Source connector: each task reads from different source partition
    (e.g., different file, different DB table)
  → Sink connector: each task reads from different Kafka partitions
    → tasks.max should be ≤ number of topic partitions
    → More tasks than partitions = idle tasks

WORKER SCALING:
  Standalone mode:
    → Single worker, single machine
    → Scale up: bigger machine (vertical)
    → No horizontal scaling

  Distributed mode:
    → Multiple workers across machines
    → Tasks auto-distributed across workers
    → Add worker → tasks rebalance → more throughput
    → Remove worker → tasks failover to remaining workers

THROUGHPUT EXPECTATIONS:
  ┌──────────────────────────┬──────────────────────────────┐
  │ Connector Type           │ Typical Throughput            │
  ├──────────────────────────┼──────────────────────────────┤
  │ FileStreamSource         │ ~10K-50K records/sec          │
  │ JDBC Source              │ ~5K-20K records/sec           │
  │ Debezium CDC             │ ~10K-50K records/sec          │
  │ S3 Sink                  │ ~50K-200K records/sec         │
  │ Elasticsearch Sink       │ ~10K-50K records/sec          │
  │ JDBC Sink                │ ~5K-20K records/sec           │
  └──────────────────────────┴──────────────────────────────┘
  (Approximate — depends on message size, network, hardware)

CONNECT INTERNAL TOPICS:
  → connect-offsets: 25 partitions default (compacted)
  → connect-configs: 1 partition (compacted)
  → connect-status: 5 partitions (compacted)
  → These use compacted topics (Project 14 knowledge!)
  → Size is proportional to number of connectors, not data volume

🔍 INTERPRETABILITY CONSIDERATIONS

HOW TO VERIFY CONNECT IS WORKING:

1. REST API STATUS CHECK (Distributed mode)
   GET /connectors                      → List all connectors
   GET /connectors/{name}/status        → Connector and task status
   GET /connectors/{name}/config        → Current configuration
   GET /connectors/{name}/tasks         → Task details

2. WORKER LOG INSPECTION
   → Connect worker logs show all internal activity
   → Look for: "Starting connector", "Task started", errors
   → Log level: INFO shows lifecycle, DEBUG shows data flow

3. TOPIC VERIFICATION
   → Check target topic has data: kafka-console-consumer
   → Check connect-offsets topic for position tracking
   → Check connect-status topic for connector health

4. CONSUMER GROUP MONITORING
   → Sink connectors use consumer groups: connect-{connector-name}
   → Monitor lag using tools from Project 12 (lag monitor)
   → High lag = sink can't keep up with source

5. METRICS (JMX)
   → source-record-poll-rate: records read per second
   → source-record-write-rate: records sent to Kafka per second
   → sink-record-read-rate: records consumed per second
   → sink-record-send-rate: records written to external system

import requests
import json

class ConnectClient:
    """
    Python client for the Kafka Connect REST API.
    
    This is how you manage connectors in distributed mode.
    Standalone mode uses properties files instead.
    """
    
    def __init__(self, base_url):
        self.base_url = base_url.rstrip('/')
    
    def _request(self, method, path, data=None):
        """Make an HTTP request to Connect REST API."""
        url = f'{self.base_url}{path}'
        headers = {'Content-Type': 'application/json'}
        
        try:
            if method == 'GET':
                resp = requests.get(url, headers=headers, timeout=10)
            elif method == 'POST':
                resp = requests.post(url, headers=headers,
                                     json=data, timeout=10)
            elif method == 'PUT':
                resp = requests.put(url, headers=headers,
                                    json=data, timeout=10)
            elif method == 'DELETE':
                resp = requests.delete(url, headers=headers, timeout=10)
            else:
                raise ValueError(f"Unknown method: {method}")
            
            if resp.status_code >= 400:
                print(f"  ❌ {method} {path}: {resp.status_code}")
                print(f"     {resp.text[:200]}")
                return None
            
            return resp.json() if resp.text else {}
            
        except requests.ConnectionError:
            print(f"  ❌ Cannot connect to {url}")
            print(f"     Is Kafka Connect running?")
            return None
    
    # ── Cluster Info ──
    def get_cluster_info(self):
        return self._request('GET', '/')
    
    # ── Connector Plugins ──
    def list_plugins(self):
        return self._request('GET', '/connector-plugins')
    
    # ── Connector CRUD ──
    def list_connectors(self):
        return self._request('GET', '/connectors')
    
    def create_connector(self, name, config):
        return self._request('POST', '/connectors', {
            'name': name,
            'config': config,
        })
    
    def get_connector(self, name):
        return self._request('GET', f'/connectors/{name}')
    
    def get_connector_config(self, name):
        return self._request('GET', f'/connectors/{name}/config')
    
    def update_connector_config(self, name, config):
        return self._request('PUT', f'/connectors/{name}/config', config)
    
    def delete_connector(self, name):
        return self._request('DELETE', f'/connectors/{name}')
    
    def get_connector_status(self, name):
        return self._request('GET', f'/connectors/{name}/status')
    
    def get_connector_tasks(self, name):
        return self._request('GET', f'/connectors/{name}/tasks')
    
    def restart_connector(self, name):
        return self._request('POST', f'/connectors/{name}/restart')
    
    def restart_task(self, name, task_id):
        return self._request('POST',
                             f'/connectors/{name}/tasks/{task_id}/restart')
    
    def pause_connector(self, name):
        return self._request('PUT', f'/connectors/{name}/pause')
    
    def resume_connector(self, name):
        return self._request('PUT', f'/connectors/{name}/resume')

def create_sample_csv():
    """
    Create a sample CSV file that the source connector will read.
    
    This simulates the Finance team's payment processor export.
    """
    filepath = os.path.join(DATA_DIR, 'transactions.csv')
    
    print(f"  📝 Creating sample CSV: {filepath}\n")
    
    # ── Generate transaction data ──
    transactions = []
    for i in range(50):
        txn = {
            'txn_id': f'TXN-{i+1:06d}',
            'timestamp': datetime.now(timezone.utc).isoformat(),
            'customer_id': f'CUST-{random.randint(1000, 9999)}',
            'amount': round(random.uniform(10.0, 5000.0), 2),
            'currency': random.choice(['USD', 'EUR', 'GBP']),
            'status': random.choice(['completed', 'pending', 'refunded']),
            'payment_method': random.choice(['credit_card', 'debit_card',
                                              'bank_transfer', 'paypal']),
        }
        transactions.append(txn)
    
    # ── Write as line-delimited JSON (FileStreamSource reads lines) ──
    # NOTE: FileStreamSource reads the file line by line
    # Each line becomes one Kafka message
    with open(filepath, 'w') as f:
        for txn in transactions:
            f.write(json.dumps(txn) + '\n')
    
    print(f"     Written {len(transactions)} transactions")
    print(f"     Sample: {json.dumps(transactions[0])[:80]}...")
    
    return filepath, len(transactions)


def create_source_connector():
    """
    Create a FileStreamSource connector via REST API.
    
    This connector reads lines from a file and produces
    them as messages to a Kafka topic.
    """
    connector_config = {
        'name': 'finance-csv-source',
        'config': {
            # ── Connector class ──
            'connector.class': 'org.apache.kafka.connect.file.FileStreamSourceConnector',
            
            # ── Source file path (inside the container) ──
            'file': '/data/transactions.csv',
            
            # ── Target Kafka topic ──
            'topic': 'finance.transactions.raw',
            
            # ── Number of tasks (FileStreamSource supports only 1) ──
            'tasks.max': '1',
            
            # ══════════════════════════════════════════════
            # CONVERTER CONFIGURATION
            # 
            # key.converter: How to serialize the message key
            # value.converter: How to serialize the message value
            #
            # FileStreamSource produces:
            #   key = null (no key by default)
            #   value = line from file (string)
            #
            # We use StringConverter for both since file lines are strings
            # ══════════════════════════════════════════════
            'key.converter': 'org.apache.kafka.connect.storage.StringConverter',
            'value.converter': 'org.apache.kafka.connect.storage.StringConverter',
        }
    }
    
    print(f"\n  🔌 Creating source connector: {connector_config['name']}")
    print(f"     File: /data/transactions.csv")
    print(f"     Topic: finance.transactions.raw")
    print(f"     Converter: StringConverter (raw line per message)\n")
    
    resp = requests.post(
        f'{CONNECT_URL}/connectors',
        headers={'Content-Type': 'application/json'},
        json=connector_config,
        timeout=10,
    )
    
    if resp.status_code == 201:
        print(f"  ✅ Connector created successfully")
        return True
    elif resp.status_code == 409:
        print(f"  ℹ️  Connector already exists — updating config...")
        resp2 = requests.put(
            f'{CONNECT_URL}/connectors/{connector_config["name"]}/config',
            headers={'Content-Type': 'application/json'},
            json=connector_config['config'],
            timeout=10,
        )
        print(f"     Update status: {resp2.status_code}")
        return True
    else:
        print(f"  ❌ Failed: {resp.status_code} — {resp.text[:200]}")
        return False


def monitor_source_connector(timeout=30):
    """Monitor the source connector until it processes the file."""
    
    print(f"\n  📊 Monitoring source connector...\n")
    
    start = time.time()
    
    while time.time() - start < timeout:
        try:
            resp = requests.get(
                f'{CONNECT_URL}/connectors/finance-csv-source/status',
                timeout=5,
            )
            status = resp.json()
            
            conn_state = status.get('connector', {}).get('state', 'UNKNOWN')
            tasks = status.get('tasks', [])
            
            task_info = []
            for task in tasks:
                task_state = task.get('state', 'UNKNOWN')
                task_id = task.get('id', '?')
                task_info.append(f"task-{task_id}={task_state}")
            
            tasks_str = ', '.join(task_info) if task_info else 'no tasks'
            
            icon = {
                'RUNNING': '✅',
                'PAUSED': '⏸️',
                'FAILED': '❌',
                'UNASSIGNED': '⏳',
            }.get(conn_state, '❓')
            
            print(f"  {icon} Connector: {conn_state}, Tasks: [{tasks_str}]")
            
            if conn_state == 'RUNNING' and all(
                t.get('state') == 'RUNNING' for t in tasks
            ):
                print(f"\n  ✅ Source connector is running and processing")
                return True
            
            if conn_state == 'FAILED' or any(
                t.get('state') == 'FAILED' for t in tasks
            ):
                print(f"\n  ❌ Connector or task FAILED")
                for task in tasks:
                    if task.get('state') == 'FAILED':
                        trace = task.get('trace', 'No trace available')
                        print(f"     Task {task['id']} error: {trace[:200]}")
                return False
        
        except requests.ConnectionError:
            print(f"  ⏳ Waiting for Connect...")
        
        time.sleep(3)
    
    print(f"  ⏳ Timeout reached — connector may still be starting")
    return False


def verify_source_data():
    """
    Verify that data from the CSV file appeared in the Kafka topic.
    Uses our Python consumer from Project 12.
    """
    from confluent_kafka import Consumer, KafkaError
    
    print(f"\n  📥 Verifying data in Kafka topic...\n")
    
    consumer = Consumer({
        'bootstrap.servers': 'localhost:9092,localhost:9093',
        'group.id': f'source-verify-{int(time.time())}',
        'auto.offset.reset': 'earliest',
        'enable.auto.commit': False,
    })
    
    consumer.subscribe(['finance.transactions.raw'])
    
    messages = []
    empty_polls = 0
    
    while empty_polls < 15:
        msg = consumer.poll(timeout=1.0)
        
        if msg is None:
            empty_polls += 1
            continue
        
        if msg.error():
            if msg.error().code() == KafkaError._PARTITION_EOF:
                empty_polls += 1
                continue
            continue
        
        empty_polls = 0
        value = msg.value().decode('utf-8') if msg.value() else ''
        messages.append({
            'offset': msg.offset(),
            'partition': msg.partition(),
            'value': value,
        })
    
    consumer.close()
    
    print(f"  Found {len(messages)} messages in finance.transactions.raw\n")
    
    if messages:
        print(f"  Sample messages:")
        for m in messages[:5]:
            val_preview = m['value'][:70]
            print(f"    P{m['partition']}:{m['offset']} → {val_preview}...")
        
        if len(messages) > 5:
            print(f"    ... and {len(messages) - 5} more")
        
        # ── Try parsing as JSON ──
        parsed = 0
        for m in messages:
            try:
                json.loads(m['value'])
                parsed += 1
            except json.JSONDecodeError:
                pass
        
        print(f"\n  JSON parseable: {parsed}/{len(messages)}")
    
    return len(messages)

import requests
import json
import time

class ConnectorLifecycleManager:
    """
    Manages the full lifecycle of Kafka Connect connectors.
    
    Lifecycle states:
      UNASSIGNED → RUNNING → PAUSED → RUNNING → FAILED → (restart) → RUNNING
                                                            ↓
                                                         (delete)
    """
    
    def __init__(self, connect_url):
        self.url = connect_url.rstrip('/')
    
    def _get(self, path):
        try:
            resp = requests.get(f'{self.url}{path}', timeout=10)
            return resp.json() if resp.status_code == 200 else None
        except Exception as e:
            print(f"  ❌ GET {path}: {e}")
            return None
    
    def _post(self, path, data=None):
        try:
            resp = requests.post(f'{self.url}{path}',
                                 headers={'Content-Type': 'application/json'},
                                 json=data, timeout=10)
            return resp.status_code
        except Exception as e:
            print(f"  ❌ POST {path}: {e}")
            return None
    
    def _put(self, path, data=None):
        try:
            resp = requests.put(f'{self.url}{path}',
                                headers={'Content-Type': 'application/json'},
                                json=data, timeout=10)
            return resp.status_code
        except Exception as e:
            print(f"  ❌ PUT {path}: {e}")
            return None
    
    def _delete(self, path):
        try:
            resp = requests.delete(f'{self.url}{path}', timeout=10)
            return resp.status_code
        except Exception as e:
            print(f"  ❌ DELETE {path}: {e}")
            return None
    
    def list_all(self):
        """List all connectors with their status."""
        connectors = self._get('/connectors')
        if not connectors:
            print(f"  (no connectors)")
            return []
        
        results = []
        for name in connectors:
            status = self._get(f'/connectors/{name}/status')
            if status:
                conn_state = status.get('connector', {}).get('state', '?')
                tasks = status.get('tasks', [])
                task_states = [t.get('state', '?') for t in tasks]
                
                icon = {
                    'RUNNING': '✅',
                    'PAUSED': '⏸️',
                    'FAILED': '❌',
                    'UNASSIGNED': '⏳',
                }.get(conn_state, '❓')
                
                print(f"  {icon} {name}: connector={conn_state}, "
                      f"tasks={task_states}")
                
                results.append({
                    'name': name,
                    'state': conn_state,
                    'tasks': task_states,
                })
        
        return results
    
    def pause(self, name):
        """
        Pause a connector — stops all tasks but keeps configuration.
        
        Use when:
          → Maintenance on external system
          → Investigating issues
          → Temporary data freeze
        """
        status = self._put(f'/connectors/{name}/pause')
        if status == 202:
            print(f"  ⏸️  Paused: {name}")
            return True
        else:
            print(f"  ❌ Failed to pause {name}: {status}")
            return False
    
    def resume(self, name):
        """
        Resume a paused connector — restarts all tasks.
        
        Resumes from where it left off (offsets are preserved).
        """
        status = self._put(f'/connectors/{name}/resume')
        if status == 202:
            print(f"  ▶️  Resumed: {name}")
            return True
        else:
            print(f"  ❌ Failed to resume {name}: {status}")
            return False
    
    def restart(self, name, include_tasks=True):
        """
        Restart a connector and optionally its tasks.
        
        Use when:
          → Connector is in FAILED state
          → After fixing configuration issue
          → After external system recovery
        """
        # ── Restart connector ──
        status = self._post(f'/connectors/{name}/restart')
        if status == 200 or status == 204:
            print(f"  🔄 Restarted connector: {name}")
        else:
            print(f"  ⚠️  Connector restart returned: {status}")
        
        # ── Restart individual tasks ──
        if include_tasks:
            tasks = self._get(f'/connectors/{name}/tasks')
            if tasks:
                for task in tasks:
                    task_id = task.get('id', {}).get('task', 0)
                    t_status = self._post(
                        f'/connectors/{name}/tasks/{task_id}/restart'
                    )
                    if t_status == 200 or t_status == 204:
                        print(f"  🔄 Restarted task {task_id}")
                    else:
                        print(f"  ⚠️  Task {task_id} restart returned: {t_status}")
    
    def reconfigure(self, name, new_config):
        """
        Update connector configuration.
        
        Connect will:
          1. Stop existing tasks
          2. Apply new configuration
          3. Start tasks with new configuration
        
        This is a ROLLING update — minimal downtime.
        """
        status = self._put(f'/connectors/{name}/config', new_config)
        if status == 200:
            print(f"  ⚙️  Reconfigured: {name}")
            return True
        else:
            print(f"  ❌ Failed to reconfigure {name}: {status}")
            return False
    
    def delete(self, name):
        """
        Delete a connector — stops all tasks and removes configuration.
        
        IMPORTANT: This does NOT delete data in Kafka topics.
        The connector is just removed from Connect.
        """
        status = self._delete(f'/connectors/{name}')
        if status == 204:
            print(f"  🗑️  Deleted: {name}")
            return True
        else:
            print(f"  ❌ Failed to delete {name}: {status}")
            return False
    
    def get_config(self, name):
        """Get current configuration of a connector."""
        return self._get(f'/connectors/{name}/config')

def write_test_records(count=25):

            f.write(json.dumps(record) + '\n')
            test_records.append(record)
    
    print(f"  📝 Wrote {count} test records (batch={batch_id})")
    
    return batch_id, test_records


def wait_for_pipeline(batch_id, timeout=60):
    """Wait for records to flow through the pipeline."""
    from confluent_kafka import Consumer, KafkaError
    
    print(f"\n  ⏳ Waiting for pipeline to process batch {batch_id}...\n")
    
    consumer = Consumer({
        'bootstrap.servers': 'localhost:9092,localhost:9093',
        'group.id': f'e2e-verify-{int(time.time())}',
        'auto.offset.reset': 'earliest',
        'enable.auto.commit': False,
    })
    
    consumer.subscribe(['finance.transactions.raw'])
    
    found_records = []
    start = time.time()
    empty_polls = 0
    
    while time.time() - start < timeout and empty_polls < 30:
        msg = consumer.poll(timeout=1.0)
        
        if msg is None:
            empty_polls += 1
            continue
        
        if msg.error():
            if msg.error().code() == KafkaError._PARTITION_EOF:
                empty_polls += 1
                continue
            continue
        
        empty_polls = 0
        
        try:
            value = msg.value().decode('utf-8')
            data = json.loads(value)
            
            if data.get('batch_id') == batch_id:
                found_records.append({
                    'record_num': data.get('record_num'),
                    'txn_id': data.get('txn_id'),
                    'amount': data.get('amount'),
                    'partition': msg.partition(),
                    'offset': msg.offset(),
                })
        except (json.JSONDecodeError, UnicodeDecodeError):
            pass
    
    consumer.close()
    
    return found_records
