✅ WHAT SHOULD DO

1. ALWAYS use --describe after --create to verify topic settings
   → Trust but verify — creation can silently use defaults you did not expect

2. ALWAYS use --dry-run before --execute for offset resets
   → Offset reset is DESTRUCTIVE — preview changes first
   → One wrong reset can cause massive data reprocessing

3. ALWAYS specify --bootstrap-server in every command
   → No default assumed — every command needs it
   → Use environment variable or alias to avoid repetition

4. ALWAYS use --property print.key=true when consuming
   → Default console consumer hides keys
   → Keys are critical for debugging partition routing (Project 4)

5. ALWAYS check consumer group state before resetting offsets
   → Group must have NO active members during reset
   → Active consumers will conflict with the reset

6. ALWAYS use --if-not-exists when creating topics in scripts
   → Prevents errors on re-run
   → Idempotent topic creation for automation

❌ WHAT SHOULD NOT DO

1. DO NOT use --delete on topics without double-checking
   → Topic deletion is PERMANENT and IRREVERSIBLE
   → All messages in the topic are lost
   → All consumer group offsets for that topic become orphaned

2. DO NOT reset offsets while consumers are actively running
   → Results in conflict between running consumer and new offsets
   → Consumer may override your reset on next commit
   → ALWAYS stop consumers first, reset, then restart

3. DO NOT use kafka-console-consumer without --max-messages in production
   → Without a limit, it runs FOREVER consuming all messages
   → Can flood your terminal with millions of lines
   → Always set --max-messages for inspection

4. DO NOT alter partition count on keyed topics casually
   → Key routing changes (Project 4 explained why)
   → Use --alter --partitions only when you understand the impact

5. DO NOT ignore the --from-beginning flag behavior
   → Without it: console consumer shows only NEW messages
   → With it: shows ALL messages from offset 0
   → Forgetting this flag is the #1 cause of "I see nothing" confusion

6. DO NOT use CLI tools as your primary monitoring solution
   → CLI is for debugging and one-off operations
   → Production monitoring needs JMX + Grafana (Project 28)

═══════════════════════════════════════════════════════════════
TOPIC CONFIGURATION — WHAT LIVES WHERE
═══════════════════════════════════════════════════════════════

Topic configuration has TWO levels:

LEVEL 1: BROKER-LEVEL DEFAULTS (server.properties)
  → log.retention.hours = 168 (7 days)
  → num.partitions = 1
  → default.replication.factor = 1
  → log.segment.bytes = 1073741824 (1GB)
  → Applied to ALL new topics unless overridden

LEVEL 2: TOPIC-LEVEL OVERRIDES (per-topic config)
  → retention.ms = 3600000 (1 hour) — overrides broker default
  → segment.bytes = 104857600 (100MB) — overrides broker default
  → Applied to THIS topic only
  → Set via kafka-configs --alter

PRECEDENCE:
  Topic-level override > Broker-level default

  Example:
  Broker: log.retention.hours = 168 (7 days)
  Topic "temp-debug": retention.ms = 3600000 (1 hour)
  → "temp-debug" retains for 1 hour (topic override wins)
  → All other topics retain for 7 days (broker default)

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "kafka-topics --create says 'topic already exists'"
├── CAUSE: Topic was created before (maybe by auto.create.topics.enable)
├── FIX: Use --if-not-exists flag
└── CHECK: kafka-topics --list to see existing topics

PITFALL 2: "kafka-console-consumer shows nothing"
├── CAUSE A: Missing --from-beginning (only sees new messages)
├── CAUSE B: Topic is empty (no messages produced yet)
├── CAUSE C: Wrong topic name (typo)
├── CAUSE D: Consumer is in a group that already consumed everything
└── FIX: Use --from-beginning --max-messages 5 for quick test

PITFALL 3: "kafka-consumer-groups --reset-offsets fails"
├── CAUSE: Active consumers still running in the group
├── SYMPTOM: "Error: Assignments can only be reset if group is inactive"
├── FIX: Stop ALL consumers in the group first, then reset
└── VERIFY: --describe should show state=Empty before reset

PITFALL 4: "kafka-topics --alter --partitions 2 (decrease) fails"
├── CAUSE: Kafka does not support decreasing partitions
├── SYMPTOM: "Error: partition count can only be increased"
├── FIX: Create new topic with fewer partitions, migrate data
└── REFERENCE: Project 3 explained this limitation

PITFALL 5: "Retention change does not take effect immediately"
├── CAUSE: Kafka checks retention on segment roll or periodic cleanup
├── WHY: Log cleaner runs on a schedule (default every 5 minutes)
├── FIX: Wait for log cleaner cycle OR trigger segment roll
└── VERIFY: Check disk usage with kafka-log-dirs after waiting

PITFALL 6: "kafka-configs --alter rejected for unknown config"
├── CAUSE: Using broker config name instead of topic config name
├── EXAMPLE: log.retention.hours (broker) vs retention.ms (topic)
├── FIX: Use topic-level config names (see parameter table above)
└── LIST: kafka-configs --describe --all shows valid parameters

📈 SCALABILITY CONSIDERATIONS

CLI TOOLS IN LARGE CLUSTERS:

→ kafka-topics --list with 10,000 topics: may take several seconds
  → Filter with --topic or grep for faster results

→ kafka-consumer-groups --describe with 100 consumers: fast
  → But --describe --all-groups with 1000 groups: slow

→ kafka-console-consumer on high-volume topic: can overwhelm terminal
  → ALWAYS use --max-messages for inspection
  → Redirect to file for analysis: ... > output.txt

→ kafka-consumer-groups --reset-offsets across 100 partitions: fast
  → But impacts all consumers on restart — plan for reprocessing load

OPERATIONAL SCALING:
→ For 1-10 topics: CLI is sufficient for all management
→ For 10-100 topics: CLI + scripts for common operations
→ For 100+ topics: Admin API + automation tools (Terraform, Ansible)
→ For 1000+ topics: Self-service platform (Project 42, 50)

🔍 INTERPRETABILITY CONSIDERATIONS

HOW DO YOU BUILD A TROUBLESHOOTING WORKFLOW?

THE 5-MINUTE DIAGNOSIS:

  1. CLUSTER HEALTH (30 seconds)
     → kafka-metadata --snapshot (KRaft) or check ZooKeeper
     → Are all brokers up?

  2. TOPIC CHECK (30 seconds)
     → kafka-topics --describe --topic <name>
     → Partitions? Replicas? ISR? Under-replicated?

  3. CONSUMER LAG (1 minute)
     → kafka-consumer-groups --describe --group <name>
     → Any partition with high lag?
     → Is the group state Stable?

  4. MESSAGE INSPECTION (1 minute)
     → kafka-console-consumer --topic <name> --from-beginning --max-messages 3
     → Are messages being produced? What do they look like?

  5. CONFIGURATION CHECK (1 minute)
     → kafka-configs --describe --topic <name> --all
     → Retention correct? Max message size sufficient?

THIS WORKFLOW SOLVES 80% OF KAFKA ISSUES IN 5 MINUTES.

SECTION 1: kafka-topics — Topic Management

# ═══════════════════════════════════════════════════════════
# 1A: CREATE TOPICS
# ═══════════════════════════════════════════════════════════

# Basic topic creation
docker exec -it kafka kafka-topics --create \
  --topic orders \
  --bootstrap-server localhost:9092 \
  --partitions 4 \
  --replication-factor 1

# Create with --if-not-exists (idempotent — safe for scripts)
docker exec -it kafka kafka-topics --create \
  --topic orders \
  --bootstrap-server localhost:9092 \
  --partitions 4 \
  --replication-factor 1 \
  --if-not-exists

# Create with custom configuration overrides
docker exec -it kafka kafka-topics --create \
  --topic temp-debug \
  --bootstrap-server localhost:9092 \
  --partitions 2 \
  --replication-factor 1 \
  --config retention.ms=3600000 \
  --config segment.bytes=104857600 \
  --config max.message.bytes=5242880

# Create compacted topic (Project 14 preview)
docker exec -it kafka kafka-topics --create \
  --topic user-profiles \
  --bootstrap-server localhost:9092 \
  --partitions 4 \
  --replication-factor 1 \
  --config cleanup.policy=compact \
  --config min.cleanable.dirty.ratio=0.1

# ═══════════════════════════════════════════════════════════
# 1B: LIST TOPICS
# ═══════════════════════════════════════════════════════════

# List ALL topics (including internal topics like __consumer_offsets)
docker exec -it kafka kafka-topics --list \
  --bootstrap-server localhost:9092

# List and filter (using grep)
docker exec -it kafka kafka-topics --list \
  --bootstrap-server localhost:9092 | grep -v "^_"
# ↑ Excludes internal topics starting with underscore

# List topics matching a pattern
docker exec -it kafka kafka-topics --list \
  --bootstrap-server localhost:9092 | grep "postal"

# ═══════════════════════════════════════════════════════════
# 1C: DESCRIBE TOPICS (most important for debugging)
# ═══════════════════════════════════════════════════════════

# Describe a specific topic
docker exec -it kafka kafka-topics --describe \
  --topic orders \
  --bootstrap-server localhost:9092

# OUTPUT:
# Topic: orders   PartitionCount: 4   ReplicationFactor: 1   Configs:
#   Topic: orders   Partition: 0   Leader: 1   Replicas: 1   Isr: 1
#   Topic: orders   Partition: 1   Leader: 1   Replicas: 1   Isr: 1
#   Topic: orders   Partition: 2   Leader: 1   Replicas: 1   Isr: 1
#   Topic: orders   Partition: 3   Leader: 1   Replicas: 1   Isr: 1

# READING THE OUTPUT:
# Leader: 1     → Broker 1 is the leader for this partition
# Replicas: 1   → Only broker 1 has a copy (replication-factor=1)
# Isr: 1        → Broker 1 is in-sync (In-Sync Replicas)
# If Isr is SHORTER than Replicas → under-replicated! (Project 40)

# Describe ALL topics
docker exec -it kafka kafka-topics --describe \
  --bootstrap-server localhost:9092

# Describe topics with specific overrides (non-default configs)
docker exec -it kafka kafka-topics --describe \
  --bootstrap-server localhost:9092 \
  --topics-with-overrides

# Describe only under-replicated partitions (critical for monitoring)
docker exec -it kafka kafka-topics --describe \
  --bootstrap-server localhost:9092 \
  --under-replicated-partitions

# ═══════════════════════════════════════════════════════════
# 1D: ALTER TOPICS
# ═══════════════════════════════════════════════════════════

# Increase partitions (CANNOT decrease — Project 3)
docker exec -it kafka kafka-topics --alter \
  --topic orders \
  --bootstrap-server localhost:9092 \
  --partitions 8

# Verify the change
docker exec -it kafka kafka-topics --describe \
  --topic orders \
  --bootstrap-server localhost:9092

# ═══════════════════════════════════════════════════════════
# 1E: DELETE TOPICS
# ═══════════════════════════════════════════════════════════

# Delete a topic (PERMANENT — all data lost!)
docker exec -it kafka kafka-topics --delete \
  --topic temp-debug \
  --bootstrap-server localhost:9092

# Verify deletion
docker exec -it kafka kafka-topics --list \
  --bootstrap-server localhost:9092 | grep temp-debug
# Should return nothing

#!/bin/bash
# ═══════════════════════════════════════════════════════════
# file: kafka_troubleshoot.sh
# Complete troubleshooting script for Kafka operations
# Run: bash kafka_troubleshoot.sh
# ═══════════════════════════════════════════════════════════

BROKER="localhost:9092"
DOCKER_CMD="docker exec -it kafka"

echo "╔══════════════════════════════════════════════════════════╗"
echo "║       KAFKA CLUSTER TROUBLESHOOTING REPORT              ║"
echo "║       Generated: $(date)             ║"
echo "╚══════════════════════════════════════════════════════════╝"

# ─────────────────────────────────────────────────────────
# CHECK 1: Cluster connectivity
# ─────────────────────────────────────────────────────────
echo ""
echo "═══ CHECK 1: CLUSTER CONNECTIVITY ═══"
echo ""

$DOCKER_CMD kafka-broker-api-versions \
  --bootstrap-server $BROKER 2>/dev/null | head -5

if [ $? -eq 0 ]; then
    echo "✅ Broker is reachable at $BROKER"
else
    echo "❌ Cannot reach broker at $BROKER"
    exit 1
fi

# ─────────────────────────────────────────────────────────
# CHECK 2: Topic inventory
# ─────────────────────────────────────────────────────────
echo ""
echo "═══ CHECK 2: TOPIC INVENTORY ═══"
echo ""

TOPICS=$($DOCKER_CMD kafka-topics --list --bootstrap-server $BROKER 2>/dev/null)
USER_TOPICS=$(echo "$TOPICS" | grep -v "^_" | grep -v "^$")
INTERNAL_TOPICS=$(echo "$TOPICS" | grep "^_")

echo "User topics:"
echo "$USER_TOPICS" | while read topic; do
    if [ -n "$topic" ]; then
        echo "  📋 $topic"
    fi
done

USER_COUNT=$(echo "$USER_TOPICS" | grep -c ".")
INTERNAL_COUNT=$(echo "$INTERNAL_TOPICS" | grep -c ".")
echo ""
echo "Total user topics: $USER_COUNT"
echo "Total internal topics: $INTERNAL_COUNT"

# ─────────────────────────────────────────────────────────
# CHECK 3: Topic details
# ─────────────────────────────────────────────────────────
echo ""
echo "═══ CHECK 3: TOPIC DETAILS ═══"
echo ""

echo "$USER_TOPICS" | while read topic; do
    if [ -n "$topic" ]; then
        echo "--- $topic ---"
        $DOCKER_CMD kafka-topics --describe \
            --topic "$topic" \
            --bootstrap-server $BROKER 2>/dev/null | head -6
        echo ""
    fi
done

# ─────────────────────────────────────────────────────────
# CHECK 4: Under-replicated partitions
# ─────────────────────────────────────────────────────────
echo ""
echo "═══ CHECK 4: UNDER-REPLICATED PARTITIONS ═══"
echo ""

URP=$($DOCKER_CMD kafka-topics --describe \
    --bootstrap-server $BROKER \
    --under-replicated-partitions 2>/dev/null)

if [ -z "$URP" ]; then
    echo "✅ No under-replicated partitions"
else
    echo "⚠️  UNDER-REPLICATED PARTITIONS FOUND:"
    echo "$URP"
fi

# ─────────────────────────────────────────────────────────
# CHECK 5: Consumer groups
# ─────────────────────────────────────────────────────────
echo ""
echo "═══ CHECK 5: CONSUMER GROUPS ═══"
echo ""

GROUPS=$($DOCKER_CMD kafka-consumer-groups --list \
    --bootstrap-server $BROKER 2>/dev/null)

echo "$GROUPS" | while read group; do
    if [ -n "$group" ]; then
        echo "--- Group: $group ---"
        $DOCKER_CMD kafka-consumer-groups --describe \
            --group "$group" \
            --bootstrap-server $BROKER 2>/dev/null
        echo ""
    fi
done

# ─────────────────────────────────────────────────────────
# CHECK 6: Consumer lag summary
# ─────────────────────────────────────────────────────────
echo ""
echo "═══ CHECK 6: CONSUMER LAG SUMMARY ═══"
echo ""

echo "$GROUPS" | while read group; do
    if [ -n "$group" ]; then
        TOTAL_LAG=$($DOCKER_CMD kafka-consumer-groups --describe \
            --group "$group" \
            --bootstrap-server $BROKER 2>/dev/null | \
            grep -v "^$" | grep -v "GROUP" | grep -v "TOPIC" | \
            awk '{sum += $6} END {print sum}')
        
        if [ -n "$TOTAL_LAG" ] && [ "$TOTAL_LAG" -gt 0 ] 2>/dev/null; then
            if [ "$TOTAL_LAG" -gt 100 ]; then
                echo "🔥 $group: total lag = $TOTAL_LAG"
            elif [ "$TOTAL_LAG" -gt 0 ]; then
                echo "⚠️  $group: total lag = $TOTAL_LAG"
            fi
        else
            echo "✅ $group: no lag (or inactive)"
        fi
    fi
done

echo ""
echo "═══ TROUBLESHOOTING COMPLETE ═══"

# ═══════════════════════════════════════════════════════════
# file: kafka_admin_tool.py
# Python-based admin tool that mirrors CLI functionality
# Use when you need programmatic access to the same operations
# ═══════════════════════════════════════════════════════════

from confluent_kafka.admin import (
    AdminClient, NewTopic, ConfigResource, ResourceType
)
from confluent_kafka import Consumer, TopicPartition
import sys
import time

admin = AdminClient({'bootstrap.servers': 'localhost:9092'})

def list_topics():
    """Equivalent to: kafka-topics --list"""
    metadata = admin.list_topics(timeout=10)
    
    print("\n📋 TOPIC LIST:\n")
    for topic_name in sorted(metadata.topics.keys()):
        if not topic_name.startswith('_'):
            topic = metadata.topics[topic_name]
            print(f"   {topic_name:<30} partitions={len(topic.partitions)}")

def describe_topic(topic_name):
    """Equivalent to: kafka-topics --describe --topic <name>"""
    metadata = admin.list_topics(topic=topic_name, timeout=10)
    topic = metadata.topics.get(topic_name)
    
    if not topic:
        print(f"   ❌ Topic '{topic_name}' not found")
        return
    
    print(f"\n📋 TOPIC: {topic_name}")
    print(f"   Partitions: {len(topic.partitions)}")
    
    for part_id, part in sorted(topic.partitions.items()):
        print(
            f"   Partition {part_id}: leader={part.leader}, "
            f"replicas={part.replicas}, isr={part.isrs}"
        )

def create_topic(topic_name, num_partitions=4, replication_factor=1, configs=None):
    """Equivalent to: kafka-topics --create"""
    new_topic = NewTopic(
        topic_name,
        num_partitions=num_partitions,
        replication_factor=replication_factor,
        config=configs or {},
    )
    
    futures = admin.create_topics([new_topic])
    
    for topic, future in futures.items():
        try:
            future.result()
            print(f"   ✅ Topic '{topic}' created successfully")
        except Exception as e:
            print(f"   ❌ Failed to create '{topic}': {e}")

def describe_config(topic_name):
    """Equivalent to: kafka-configs --describe --entity-type topics"""
    resource = ConfigResource(ResourceType.TOPIC, topic_name)
    futures = admin.describe_configs([resource])
    
    for res, future in futures.items():
        try:
            configs = future.result()
            print(f"\n📋 CONFIGURATION: {topic_name}\n")
            
            # Separate overridden from default
            overridden = []
            defaults = []
            
            for name, config in sorted(configs.items()):
                if config.source.name == 'DYNAMIC_TOPIC_CONFIG':
                    overridden.append((name, config.value))
                else:
                    defaults.append((name, config.value))
            
            if overridden:
                print("   OVERRIDDEN (topic-specific):")
                for name, value in overridden:
                    print(f"      ✏️  {name} = {value}")
            
            print(f"\n   DEFAULT (from broker) — showing key ones:")
            important = [
                'retention.ms', 'retention.bytes', 'cleanup.policy',
                'segment.bytes', 'max.message.bytes', 'compression.type',
                'min.insync.replicas',
            ]
            for name, value in defaults:
                if name in important:
                    print(f"      📌 {name} = {value}")
                    
        except Exception as e:
            print(f"   ❌ Failed: {e}")

def describe_consumer_group(group_id):
    """Equivalent to: kafka-consumer-groups --describe --group <name>"""
    
    # Get group members
    groups = admin.list_groups(timeout=10)
    target_group = None
    for g in groups:
        if g.id == group_id:
            target_group = g
            break
    
    print(f"\n📋 CONSUMER GROUP: {group_id}")
    
    if target_group:
        print(f"   State    : {target_group.state}")
        print(f"   Protocol : {target_group.protocol}")
        print(f"   Members  : {len(target_group.members)}")
        
        for member in target_group.members:
            print(f"      → {member.client_id} (host: {member.client_host})")
    else:
        print(f"   State: Empty or not found")
    
    # Get lag information using a temporary consumer
    temp_config = {
        'bootstrap.servers': 'localhost:9092',
        'group.id': group_id,
    }
    temp_consumer = Consumer(temp_config)
    
    # Get topics this group subscribes to
    all_metadata = admin.list_topics(timeout=10)
    
    print(f"\n   LAG PER PARTITION:")
    print(f"   {'Topic':<25} {'Part':>4} {'Committed':>10} {'Latest':>8} {'Lag':>6}")
    print(f"   {'─' * 58}")
    
    total_lag = 0
    
    for topic_name, topic_meta in all_metadata.topics.items():
        if topic_name.startswith('_'):
            continue
        
        for part_id in topic_meta.partitions:
            tp = TopicPartition(topic_name, part_id)
            
            try:
                committed_list = temp_consumer.committed([tp], timeout=3)
                committed = committed_list[0].offset if committed_list[0].offset >= 0 else -1
                
                if committed < 0:
                    continue
                
                low, high = temp_consumer.get_watermark_offsets(tp, timeout=3)
                lag = high - committed
                total_lag += lag
                
                lag_icon = "✅" if lag == 0 else "⚠️" if lag < 10 else "🔥"
                
                print(
                    f"   {topic_name:<25} {part_id:>4} {committed:>10} "
                    f"{high:>8} {lag:>6} {lag_icon}"
                )
            except Exception:
                continue
    
    print(f"\n   TOTAL LAG: {total_lag}")
    temp_consumer.close()

# ─────────────────────────────────────────────────────────

if __name__ == '__main__':
    print("🔧 KAFKA ADMIN TOOL (Python)\n")
    print("=" * 60)
    
    # Run all inspections

    # Describe specific topics

    for topic_name in sorted(metadata.topics.keys()):
        if not topic_name.startswith('_'):
            describe_topic(topic_name)
    
    print("\n" + "=" * 60)
    print("\n📋 CONFIGURATION FOR KEY TOPICS:\n")
    
    for topic_name in sorted(metadata.topics.keys()):
        if not topic_name.startswith('_'):
            describe_config(topic_name)
    
    print("\n" + "=" * 60)
    print("\n📋 CONSUMER GROUPS:\n")
    
    groups = admin.list_groups(timeout=10)
    for g in groups:
        if not g.id.startswith('_'):
            describe_consumer_group(g.id)
    
    print(f"\n{'=' * 60}")
    print("✅ Admin inspection complete!")

# State:    kf-groups --describe --group GROUP_NAME --state
# Members:  kf-groups --describe --group GROUP_NAME --members --verbose

# ── OFFSET RESET (stop consumers first!) ──
# Preview:  kf-groups --group GRP --topic T --reset-offsets --to-earliest --dry-run
# Execute:  kf-groups --group GRP --topic T --reset-offsets --to-earliest --execute
# To time:  kf-groups --group GRP --topic T --reset-offsets --to-datetime "2024-03-10T00:00:00.000" --dry-run
# Shift:    kf-groups --group GRP --topic T --reset-offsets --shift-by -10 --dry-run
# Specific: kf-groups --group GRP --topic T:0 --reset-offsets --to-offset 25 --dry-run

# ── CONFIG ──
# Describe: kf-configs --describe --entity-type topics --entity-name NAME --all
# Alter:    kf-configs --alter --entity-type topics --entity-name NAME --add-config retention.ms=3600000
# Remove:   kf-configs --alter --entity-type topics --entity-name NAME --delete-config retention.ms

# LOG-END-OFFSET shows latest message
# LAG = difference (messages not yet consumed)

  --group cli-test-group \
  --bootstrap-server localhost:9092 \
  --state

# OBSERVE: State should be "Empty" (no active consumers)


# ═══════════════════════════════════════════════════════════
# STEP 6: Offset reset practice
# ═══════════════════════════════════════════════════════════

# ALWAYS dry-run first!

# Preview: reset to earliest
docker exec -it kafka kafka-consumer-groups \
  --group cli-test-group \
  --bootstrap-server localhost:9092 \
  --topic cli-test-1 \
  --reset-offsets --to-earliest \
  --dry-run

# OBSERVE: Shows NEW-OFFSET = 0 for all partitions

# Execute the reset
docker exec -it kafka kafka-consumer-groups \
  --group cli-test-group \
  --bootstrap-server localhost:9092 \
  --topic cli-test-1 \
  --reset-offsets --to-earliest \
  --execute

# Verify: describe group again
docker exec -it kafka kafka-consumer-groups --describe \
  --group cli-test-group \
  --bootstrap-server localhost:9092

# OBSERVE: CURRENT-OFFSET = 0, LAG = full message count

# Now consume again — it re-reads ALL messages
docker exec -it kafka kafka-console-consumer \
  --topic cli-test-1 \
  --bootstrap-server localhost:9092 \
  --group cli-test-group \
  --max-messages 5

# Preview: shift back by 2 messages
docker exec -it kafka kafka-consumer-groups \
  --group cli-test-group \
  --bootstrap-server localhost:9092 \
  --topic cli-test-1 \
  --reset-offsets --shift-by -2 \
  --dry-run

# Preview: reset to specific offset on partition 0
docker exec -it kafka kafka-consumer-groups \
  --group cli-test-group \
  --bootstrap-server localhost:9092 \
  --topic cli-test-1:0 \
  --reset-offsets --to-offset 1 \
  --dry-run

  --entity-type topics \
  --entity-name cli-test-2 \
  --bootstrap-server localhost:9092
