PROJECT 21: THE SALES LEADERBOARD — STATEFUL AGGREGATIONS

🏆 HYPOTHETICAL PROBLEM

A global e-commerce platform processes 2 million orders per day

SCENARIO: Real-Time Sales Leaderboard
→ Every purchase generates an order event to "order-events" topic
→ Business needs FIVE real-time metrics:

  METRIC 1: "Total orders per product category"
    → Electronics: 45,230 orders
    → Clothing: 38,102 orders
    → Books: 22,891 orders

  METRIC 2: "Total revenue per seller"
    → Seller-001: $1,245,600
    → Seller-002: $982,300
    → Seller-003: $756,100

  METRIC 3: "Running average order value per region"
    → North America: $127.45
    → Europe: $98.30
    → Asia Pacific: $64.20

  METRIC 4: "Highest single order per category"
    → Electronics: $4,299 (order #ORD-88231)
    → Clothing: $1,850 (order #ORD-77442)

  METRIC 5: "Combined leaderboard score per seller"
    → Custom formula: (total_revenue * 0.4) + (order_count * 0.3) + (avg_rating * 0.3)

THE CRISIS WITH PROJECT 20 (STATELESS) APPROACH:

  Developer tries to build with stateless operations only:

  orders.mapValues(order -> order.getCategory() + ":" + order.getAmount())
        .to("category-totals");

  Problems:
  → mapValues produces ONE output per input — no running total
  → "Electronics: $299" then "Electronics: $499" — two separate records
  → WHERE is the accumulated total of $798?
  → Stateless operations have NO MEMORY of previous records
  → Cannot count, sum, average, or track maximum
  → NEED: operations that REMEMBER and UPDATE state

  Developer tries manual state management:

  Map<String, Long> counts = new HashMap<>();  // In-memory
  orders.mapValues(order -> {
      counts.merge(order.getCategory(), 1L, Long::sum);
      return counts.get(order.getCategory());
  });

  Problems:
  → HashMap is in-memory → lost on crash
  → No changelog → no recovery
  → Multiple tasks = multiple HashMaps → inconsistent counts
  → Not thread-safe across stream threads
  → Kafka Streams PROHIBITS external mutable state in DSL operations

THE ANSWER: STATEFUL AGGREGATIONS
  → groupByKey() / groupBy() — organize records by key
  → count() — count records per key
  → reduce() — combine records per key with binary function
  → aggregate() — flexible accumulation with custom initializer and adder
  → ALL backed by RocksDB state stores + changelog topics automatically

🔍 HOW TO APPROACH IT

Step 1: Understand the stateful operation pipeline
         → EVERY stateful operation follows this pattern:
         
         KStream → group → aggregate → KTable
         
         stream.groupByKey()    → KGroupedStream
               .count()         → KTable<Key, Long>
         
         stream.groupBy(...)    → KGroupedStream (with repartition)
               .reduce(...)     → KTable<Key, Value>
         
         stream.groupByKey()    → KGroupedStream
               .aggregate(...)  → KTable<Key, CustomType>

Step 2: Understand KTable vs KStream (introduced in Project 19, deep dive now)
         
         KStream = FACTS (events that happened)
         → "Order #123 placed for $299"
         → "Order #124 placed for $499"
         → APPEND-ONLY — every record is meaningful
         → Reading ALL records gives complete history
         
         KTable = STATE (current value per key)
         → "Electronics → 45,230 orders"
         → "Clothing → 38,102 orders"
         → UPSERT — new value REPLACES old value for same key
         → Reading LATEST value per key gives current state
         → Backed by state store (RocksDB)

Step 3: Map each metric to an aggregation operation
         
         Metric 1 (order count per category):
         → groupBy(category) → count()
         
         Metric 2 (revenue per seller):
         → groupBy(seller) → reduce(sum amounts)
         
         Metric 3 (avg order value per region):
         → groupBy(region) → aggregate(sum + count → average)
         
         Metric 4 (highest order per category):
         → groupBy(category) → reduce(keep maximum)
         
         Metric 5 (composite score per seller):
         → groupBy(seller) → aggregate(custom accumulator)

Step 4: Understand state store mechanics
         → Each aggregation creates a STATE STORE
         → State store = RocksDB embedded database on local disk
         → Every update also written to CHANGELOG TOPIC
         → Changelog is compacted (Project 14) — only latest value per key
         → On crash: state rebuilt by replaying changelog

✅ WHAT COULD BE DONE

Option A: External Database for Aggregation
  → Consumer reads orders → writes to PostgreSQL → SQL aggregation
  → PROBLEM: Added latency (Kafka → consumer → DB → query)
  → PROBLEM: DB becomes bottleneck at high throughput
  → PROBLEM: Not real-time — query-time aggregation

Option B: In-Memory State in Consumer Application
  → HashMap in consumer code maintains running totals
  → PROBLEM: Lost on crash — no durability
  → PROBLEM: Not partitioned — can't scale horizontally
  → PROBLEM: Kafka Streams DSL prohibits external mutable state

Option C: Redis/Memcached as State Store
  → External key-value store for aggregated state
  → PROBLEM: Network round-trip for every state access
  → PROBLEM: Consistency between Kafka offsets and Redis state
  → PROBLEM: Additional infrastructure to manage

Option D: Kafka Streams Stateful Aggregations ← CORRECT
  → groupByKey()/groupBy() + count()/reduce()/aggregate()
  → State stored LOCALLY in RocksDB (no network hop)
  → Backed by changelog topics (automatic durability)
  → Recovered automatically on crash
  → Scales with partition count
  → No external infrastructure

✅ WHAT SHOULD DO

1. USE groupByKey() WHEN KEY ALREADY MATCHES GROUPING KEY
   → If order events are keyed by seller_id and you group by seller_id
   → groupByKey() does NOT create repartition topic
   → MUCH cheaper than groupBy()
   → This connects to Project 20's "prefer value-only operations" rule

2. USE groupBy() ONLY WHEN KEY MUST CHANGE
   → If order events are keyed by order_id but you group by category
   → groupBy((key, value) -> value.getCategory())
   → Creates repartition topic (data shuffled by new key)
   → Necessary cost — but understand the overhead

3. NAME YOUR STATE STORES WITH Materialized.as()
   → count(Materialized.as("category-count-store"))
   → Named stores:
     → Appear in metrics with meaningful names
     → Can be queried via Interactive Queries (Project 33)
     → Have predictable changelog topic names
   → Unnamed stores get auto-generated cryptic names

4. CONFIGURE STATE STORE RETENTION
   → Materialized.as("store").withRetention(Duration.ofDays(30))
   → Prevents unbounded state growth
   → Old keys that haven't been updated get cleaned up
   → Critical for high-cardinality keys

5. UNDERSTAND THE OUTPUT BEHAVIOR OF KTable
   → KTable emits update ONLY when value changes
   → With caching enabled (default): updates are batched
   → Output rate = min(input rate, 1/commit.interval.ms)
   → Set cache.max.bytes.buffering=0 for immediate output (debug only)
   → In production: keep caching for efficiency

6. CHOOSE THE RIGHT AGGREGATION OPERATION
   → count(): when you just need "how many" per key
   → reduce(): when output TYPE matches input TYPE
   → aggregate(): when output TYPE DIFFERS from input TYPE
   → This is the KEY decision framework

❌ WHAT SHOULD NOT DO

1. DO NOT USE groupBy() WHEN groupByKey() WORKS
   → groupBy() ALWAYS creates repartition topic
   → Even if you return the SAME key: groupBy((k,v) -> k)
   → Kafka Streams cannot detect "key didn't actually change"
   → Use groupByKey() when key is already correct
   → This mirrors the mapValues() > map() rule from Project 20

2. DO NOT USE reduce() WHEN TYPE CHANGES
   → reduce() requires: (Value, Value) → Value (same type)
   → Input: OrderEvent, want output: RevenueTotal (different type)
   → reduce() cannot do this — use aggregate() instead
   → reduce() is for: "combine two OrderEvents into one OrderEvent"

3. DO NOT STORE LARGE OBJECTS IN STATE STORES
   → Each key-value pair is serialized to RocksDB
   → Large values (> 1MB) cause:
     → Slow serialization/deserialization
     → Large changelog topic
     → Slow state recovery
   → Store aggregated SUMMARIES, not raw data

4. DO NOT IGNORE STATE STORE DISK USAGE
   → RocksDB uses local disk for state
   → High-cardinality keys = large state
   → 100 million unique keys × 100 bytes = ~10GB per task
   → Monitor disk usage in state.dir
   → Use SSD for state directory in production

5. DO NOT USE aggregate() WITHOUT PROPER INITIALIZER
   → Initializer creates the FIRST value for a new key
   → Forgetting to initialize: NullPointerException
   → Initializer runs ONCE per key, not once per record
   → Make initializer idempotent and lightweight

6. DO NOT ASSUME count() OUTPUT IS FINAL ON FIRST EMIT
   → count() emits INTERMEDIATE results
   → First order for "Electronics" → count=1
   → Second order → count=2
   → BOTH emissions go to output topic
   → Downstream consumers see ALL intermediate counts
   → Use caching to reduce intermediate emissions

DECISION TREE:

  "What do I need to compute?"
  │
  ├── "Just count records per key"
  │    → count()
  │
  ├── "Combine values, output SAME type as input"
  │    → reduce()
  │    Examples: sum amounts, find max, concatenate strings
  │
  └── "Accumulate into DIFFERENT type"
       → aggregate()
       Examples: running average (need sum + count),
                 composite score (need multiple fields),
                 custom statistics object

  "Is my stream already keyed correctly?"
  │
  ├── YES → groupByKey() (no repartition)
  │
  └── NO  → groupBy(newKeyMapper) (repartition created)

🚫 WHEN NOT TO USE

1. WHEN PROCESSING IS PURELY STATELESS
   → filter, map, branch don't need aggregation
   → Adding unnecessary aggregation wastes state store resources
   → Use Project 20 patterns for pure transformation

2. WHEN EXACT COUNTS ARE NEEDED AT QUERY TIME (NOT STREAM TIME)
   → Kafka Streams aggregations are STREAM-TIME computations
   → If you need "how many orders between 2pm and 3pm yesterday"
   → That's a BATCH query, not a stream aggregation
   → Use a database or data warehouse for historical queries
   → Kafka Streams excels at "what's the count RIGHT NOW"

3. WHEN STATE SIZE EXCEEDS LOCAL DISK CAPACITY
   → RocksDB stores state on local disk
   → 1 billion unique keys with complex aggregations = 100+ GB
   → If state doesn't fit on local disk → external state store needed
   → Consider Apache Flink with RocksDB state backend (larger state support)
   → Or use Kafka Streams with custom state store backed by external DB

4. WHEN AGGREGATION WINDOW IS NEEDED
   → count() gives ALL-TIME count per key
   → "Count in last 5 minutes" needs WINDOWED count
   → That's Project 22 — windowed operations
   → Don't try to simulate windows with aggregate() + timestamps

MECHANISM 2: STATE STORE — RocksDB DEEP DIVE
═══════════════════════════════════════════════════════════════════

  Each stateful operation creates a RocksDB state store:

  DIRECTORY STRUCTURE:
  {state.dir}/{application.id}/{task.id}/
  └── rocksdb/
      └── {store-name}/
          ├── 000003.sst        ← SST files (Sorted String Tables)
          ├── 000005.sst        ← Immutable, sorted key-value pairs
          ├── CURRENT            ← Points to current MANIFEST
          ├── MANIFEST-000004    ← Database metadata
          ├── LOG                ← RocksDB operation log
          ├── LOCK               ← File lock (one process only)
          └── OPTIONS-000007     ← Configuration snapshot

  HOW RocksDB WORKS (LSM-Tree):

  WRITE PATH:
  ┌──────────────────────────────────────────────────────┐
  │ 1. Write to MemTable (in-memory sorted buffer)       │
  │    → Fast: O(log n) insertion                        │
  │                                                      │
  │ 2. When MemTable is full → flush to SST file on disk │
  │    → SST file: sorted, immutable key-value pairs     │
  │    → Each flush creates a new SST file               │
  │                                                      │
  │ 3. Background compaction merges SST files             │
  │    → Reduces number of files                         │
  │    → Removes obsolete versions of keys               │
  │    → Keeps data sorted for fast reads                │
  └──────────────────────────────────────────────────────┘

  READ PATH:
  ┌──────────────────────────────────────────────────────┐
  │ 1. Check MemTable (in-memory)                        │
  │    → Found? Return immediately                       │
  │                                                      │
  │ 2. Check Block Cache (in-memory cache of SST blocks) │
  │    → Found? Return from cache                        │
  │                                                      │
  │ 3. Check SST files (newest first)                    │
  │    → Binary search within each SST file              │
  │    → Bloom filters skip files that definitely        │
  │      don't contain the key                           │
  │                                                      │
  │ Read latency: ~0.01ms (MemTable) to ~1ms (SST)      │
  └──────────────────────────────────────────────────────┘

  CONFIGURATION FOR KAFKA STREAMS:
  → Default MemTable size: 16MB
  → Default Block Cache: 50MB
  → Default Write Buffer count: 3
  → Configurable via StreamsConfig.ROCKSDB_CONFIG_SETTER_CLASS_CONFIG

MECHANISM 3: CHANGELOG TOPICS — DURABILITY GUARANTEE
═══════════════════════════════════════════════════════════════════

  Every state store update is ALSO written to a changelog topic:

  State store: put("Electronics", 43)
       │
       ├──→ RocksDB: update local state
       │
       └──→ Changelog topic: produce("Electronics", 43)
            Topic: {app.id}-{store-name}-changelog
            Example: "sales-leaderboard-app-category-count-store-changelog"

  Changelog topic properties:
  → cleanup.policy = compact (like Project 14)
  → Compaction keeps ONLY latest value per key
  → "Electronics": 41 → 42 → 43 → compacted to just 43
  → Replication factor configurable (default: cluster default)
  → Partitions: same as input topic

  RECOVERY FLOW (on crash):
  ┌──────────────────────────────────────────────────────┐
  │ 1. Task reassigned to (possibly different) instance  │
  │ 2. Check local state directory                       │
  │    → If RocksDB files exist and valid: use them      │
  │    → If missing or corrupt: rebuild from changelog   │
  │ 3. Read changelog topic from beginning               │
  │    → For each (key, value): stateStore.put(key, val) │
  │ 4. State fully restored                              │
  │ 5. Resume processing from last committed offset      │
  └──────────────────────────────────────────────────────┘

  Recovery time depends on:
  → Number of unique keys (compacted changelog size)
  → Network speed (reading from Kafka brokers)
  → Disk speed (writing to RocksDB)
  → Example: 10 million keys × 100 bytes = 1GB → ~30 seconds recovery

MECHANISM 4: CACHING — WHY OUTPUT IS DELAYED
═══════════════════════════════════════════════════════════════════

  Kafka Streams uses a RECORD CACHE in front of state stores:

  Without caching (cache.max.bytes.buffering=0):
  → Input: ("Electronics", order1), ("Electronics", order2), ("Electronics", order3)
  → Output: ("Electronics", 1), ("Electronics", 2), ("Electronics", 3)
  → THREE outputs for THREE inputs
  → Every input causes immediate state update + downstream emit

  With caching (default: 10MB):
  → Input: ("Electronics", order1), ("Electronics", order2), ("Electronics", order3)
  → Cache absorbs intermediate updates
  → On cache flush (commit interval or cache full):
  → Output: ("Electronics", 3)
  → ONE output for THREE inputs
  → Massive reduction in downstream traffic

  Cache flush triggers:
  → commit.interval.ms elapsed (default: 30 seconds)
  → cache.max.bytes.buffering exceeded
  → Explicit flush on shutdown

  WHY THIS MATTERS:
  → Production: keep caching for throughput
  → Development: set cache to 0 for immediate visibility
  → Downstream consumers see FEWER intermediate states with caching
  → Trade-off: latency vs throughput

MECHANISM 6: KTable SEMANTICS — UPSERT BEHAVIOR
═══════════════════════════════════════════════════════════════════

  KTable is fundamentally different from KStream:

  KStream (append-only):
  ┌──────────────────────────────────────────┐
  │ Offset 0: ("Electronics", order1)        │
  │ Offset 1: ("Electronics", order2)        │
  │ Offset 2: ("Clothing", order3)           │
  │ Offset 3: ("Electronics", order4)        │
  │                                          │
  │ ALL 4 records are meaningful             │
  │ Reading all = complete history           │
  └──────────────────────────────────────────┘

  KTable (upsert):
  ┌──────────────────────────────────────────┐
  │ "Electronics" → 3  (was 1, then 2, now 3)│
  │ "Clothing"    → 1                        │
  │                                          │
  │ ONLY latest value per key matters        │
  │ Previous values are overwritten          │
  │ Reading current state = snapshot         │
  └──────────────────────────────────────────┘

  KTable output to topic:
  → Each update emits (key, newValue) to output topic
  → Output topic should be COMPACTED (like changelog)
  → Consumer of output sees latest value per key
  → Old values cleaned up by compaction

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: USING groupBy() WHEN groupByKey() SUFFICES
─────────────────────────────────────────────────────────
  Symptom: Unexpected repartition topic, doubled processing latency
  Root Cause: groupBy() ALWAYS repartitions, even with same key
  Example:
    // BAD — creates repartition topic unnecessarily
    stream.groupBy((key, value) -> key)
          .count();
    
    // GOOD — no repartition
    stream.groupByKey()
          .count();
  
  Detection: Check for repartition topics in topic list
  Impact: Extra topic, extra network I/O, ~2x latency

PITFALL 2: reduce() WITH TYPE MISMATCH
─────────────────────────────────────────────────────────
  Symptom: Compilation error or ClassCastException
  Root Cause: reduce() requires (V, V) → V — same types
  Example:
    // WRONG — trying to reduce OrderEvent into Long
    stream.groupByKey()
          .reduce((order1, order2) -> order1.getAmount() + order2.getAmount());
    // Compilation error: Long is not OrderEvent
    
    // CORRECT — use aggregate() for type change
    stream.groupByKey()
          .aggregate(
              () -> 0L,
              (key, order, totalAmount) -> totalAmount + order.getAmount()
          );

PITFALL 3: UNBOUNDED STATE GROWTH
─────────────────────────────────────────────────────────
  Symptom: Disk fills up, RocksDB OOM, slow recovery
  Root Cause: New keys added continuously, never cleaned up
  Example: Grouping by order_id (unique per order) instead of category
    → 2 million unique keys per day × 365 days = 730 million keys
    → State grows without bound
  Fix: Use windowed aggregations (Project 22) for time-bounded state
       Or set retention on Materialized:
       Materialized.as("store").withRetention(Duration.ofDays(30))

PITFALL 4: EXPECTING EXACTLY ONE OUTPUT PER AGGREGATION
─────────────────────────────────────────────────────────
  Symptom: Downstream consumer sees intermediate counts
  Root Cause: count() emits on EVERY update (with caching disabled)
  Example:
    3 orders for "Electronics":
    → Output: ("Electronics",1), ("Electronics",2), ("Electronics",3)
    → Consumer processes ALL three — not just final count
  Fix: Enable caching (default) to reduce intermediate emissions
       Or handle in consumer: only process latest per key (KTable semantics)

PITFALL 5: SERDE MISMATCH IN aggregate()
─────────────────────────────────────────────────────────
  Symptom: SerializationException at runtime
  Root Cause: aggregate() output type needs explicit Serde
  Example:
    stream.groupByKey()
          .aggregate(
              () -> new RegionStats(),
              (key, order, stats) -> stats.update(order)
          );
    // CRASH: Kafka Streams doesn't know how to serialize RegionStats
    
  Fix: Provide Serde in Materialized:
    .aggregate(
        () -> new RegionStats(),
        (key, order, stats) -> stats.update(order),
        Materialized.with(Serdes.String(), new JsonSerde<>(RegionStats.class))
            .as("region-stats-store")
    );

PITFALL 6: STATE STORE NOT FOUND FOR INTERACTIVE QUERIES
─────────────────────────────────────────────────────────
  Symptom: InvalidStateStoreException when querying
  Root Cause: Store not named, or queried before RUNNING state
  Fix: Always use Materialized.as("store-name")
       Wait for KafkaStreams.State.RUNNING before querying
       This is fully covered in Project 33

/**
 * Custom aggregation result types.
 * Used as OUTPUT types for aggregate() operations.
 * 
 * KEY INSIGHT: These are DIFFERENT types from OrderEvent (input).
 * This is WHY we need aggregate() instead of reduce().
 * reduce() requires same input/output type.
 * aggregate() allows different input/output types.
 */
public class AggregationModels {

    // ════════════════════════════════════════════════════════
    // METRIC 3: Running Average per Region
    // ════════════════════════════════════════════════════════

    /**
     * Maintains running sum and count for computing average.
     * Cannot use reduce() because we need TWO fields (sum + count)
     * while input is OrderEvent (different type).
     */
    public static class RegionStats {

        @JsonProperty("total_amount")
        private double totalAmount;

        @JsonProperty("order_count")
        private long orderCount;

        @JsonProperty("average_amount")
        private double averageAmount;

        @JsonProperty("region")
        private String region;

        public RegionStats() {
            this.totalAmount = 0.0;
            this.orderCount = 0;
            this.averageAmount = 0.0;
        }

        /**
         * Update with new order — returns NEW instance (immutable pattern).
         * NEVER mutate in-place in Kafka Streams aggregations.
         */
        public RegionStats update(String region, OrderEvent order) {
            RegionStats updated = new RegionStats();
            updated.region = region;
            updated.totalAmount = this.totalAmount + order.getAmount();
            updated.orderCount = this.orderCount + 1;
            updated.averageAmount = updated.totalAmount / updated.orderCount;
            return updated;
        }

        // Getters
        public double getTotalAmount()   { return totalAmount; }
        public long getOrderCount()      { return orderCount; }
        public double getAverageAmount() { return averageAmount; }
        public String getRegion()        { return region; }

        @Override
        public String toString() {
            return String.format("RegionStats{region=%s, orders=%d, total=%.2f, avg=%.2f}",
                region, orderCount, totalAmount, averageAmount);
        }
    }

    // ════════════════════════════════════════════════════════
    // METRIC 4: Maximum Order per Category
    // ════════════════════════════════════════════════════════

    /**
     * Tracks the highest single order for a category.
     * COULD use reduce() since input and output are both OrderEvent.
     * But using aggregate() for demonstration and to store extra metadata.
     */
    public static class CategoryMaxOrder {

        @JsonProperty("category")
        private String category;

        @JsonProperty("max_amount")
        private double maxAmount;

        @JsonProperty("max_order_id")
        private String maxOrderId;

        @JsonProperty("total_orders_seen")
        private long totalOrdersSeen;

        public CategoryMaxOrder() {
            this.maxAmount = 0.0;
            this.totalOrdersSeen = 0;
        }

        public CategoryMaxOrder update(String category, OrderEvent order) {
            CategoryMaxOrder updated = new CategoryMaxOrder();
            updated.category = category;
            updated.totalOrdersSeen = this.totalOrdersSeen + 1;

            if (order.getAmount() > this.maxAmount) {
                updated.maxAmount = order.getAmount();
                updated.maxOrderId = order.getOrderId();
            } else {
                updated.maxAmount = this.maxAmount;
                updated.maxOrderId = this.maxOrderId;
            }
            return updated;
        }

        // Getters
        public String getCategory()       { return category; }
        public double getMaxAmount()      { return maxAmount; }
        public String getMaxOrderId()     { return maxOrderId; }
        public long getTotalOrdersSeen()  { return totalOrdersSeen; }

        @Override
        public String toString() {
            return String.format("CategoryMax{cat=%s, max=%.2f, orderId=%s, seen=%d}",
                category, maxAmount, maxOrderId, totalOrdersSeen);
        }
    }

    // METRIC 5: Composite Seller Score
    // ════════════════════════════════════════════════════════

     * Composite leaderboard score combining multiple metrics.
     * Formula: (total_revenue * 0.4) + (order_count * 100 * 0.3) + (avg_rating * 1000 * 0.3)
     * 
     * This is the most complex aggregation — REQUIRES aggregate().
     * Cannot use count() (just counting), reduce() (type mismatch).
     */
    public static class SellerScore {

        @JsonProperty("seller_id")
        private String sellerId;

        @JsonProperty("total_revenue")
        private double totalRevenue;

        @JsonProperty("order_count")
        private long orderCount;

        @JsonProperty("total_rating")
        private double totalRating;

        @JsonProperty("avg_rating")
        private double avgRating;

        @JsonProperty("composite_score")
        private double compositeScore;

        public SellerScore() {

            this.orderCount = 0;
            this.totalRating = 0.0;
            this.avgRating = 0.0;
            this.compositeScore = 0.0;
        }

        public SellerScore update(String sellerId, OrderEvent order) {
            SellerScore updated = new SellerScore();
            updated.sellerId = sellerId;
            updated.totalRevenue = this.totalRevenue + order.getAmount();
            updated.orderCount = this.orderCount + 1;
            updated.totalRating = this.totalRating + order.getRating();
            updated.avgRating = updated.totalRating / updated.orderCount;

            // Composite score formula
            updated.compositeScore =
                (updated.totalRevenue * 0.4) +
                (updated.orderCount * 100 * 0.3) +
                (updated.avgRating * 1000 * 0.3);

            return updated;
        }

        // Getters
        public String getSellerId()       { return sellerId; }
        public double getTotalRevenue()   { return totalRevenue; }
        public long getOrderCount()       { return orderCount; }
        public double getAvgRating()      { return avgRating; }
        public double getCompositeScore() { return compositeScore; }

        @Override
        public String toString() {
            return String.format(
                "SellerScore{seller=%s, revenue=%.2f, orders=%d, avgRating=%.1f, score=%.0f}",
                sellerId, totalRevenue, orderCount, avgRating, compositeScore);
        }
    }

package com.kafkamastery;

import com.kafkamastery.AggregationModels.*;
import org.apache.kafka.common.serialization.Serdes;
import org.apache.kafka.common.utils.Bytes;
import org.apache.kafka.streams.*;
import org.apache.kafka.streams.kstream.*;
import org.apache.kafka.streams.state.KeyValueStore;

import java.util.Properties;
import java.util.concurrent.CountDownLatch;

/**
 * PROJECT 21: Sales Leaderboard — Stateful Aggregations
 *
 * Five aggregation pipelines demonstrating:
 *   → count()     — Metric 1: orders per category
 *   → reduce()    — Metric 2: revenue per seller
 *   → aggregate() — Metric 3: average order value per region
 *   → aggregate() — Metric 4: max order per category
 *   → aggregate() — Metric 5: composite seller score
 *
 * Builds on:
 *   → Project 19: Kafka Streams basics, topology, state stores
 *   → Project 20: Stateless transformations (filter, mapValues, branch)
 *   → Project 14: Log compaction (changelog topics)
 */
public class SalesLeaderboardApp {

    private static final String INPUT_TOPIC = "order-events";
    private static final String APPLICATION_ID = "sales-leaderboard-app";
    private static final String BOOTSTRAP_SERVERS = "localhost:9092";

    // Custom Serdes
    private static final JsonSerde<OrderEvent> ORDER_SERDE =
        new JsonSerde<>(OrderEvent.class);
    private static final JsonSerde<RegionStats> REGION_STATS_SERDE =
        new JsonSerde<>(RegionStats.class);
    private static final JsonSerde<CategoryMaxOrder> CATEGORY_MAX_SERDE =
        new JsonSerde<>(CategoryMaxOrder.class);
    private static final JsonSerde<SellerScore> SELLER_SCORE_SERDE =
        new JsonSerde<>(SellerScore.class);

    // ============================================================
    // CONFIGURATION
    // ============================================================

    private static Properties createConfig() {
        Properties props = new Properties();
        props.put(StreamsConfig.APPLICATION_ID_CONFIG, APPLICATION_ID);
        props.put(StreamsConfig.BOOTSTRAP_SERVERS_CONFIG, BOOTSTRAP_SERVERS);
        props.put(StreamsConfig.DEFAULT_KEY_SERDE_CLASS_CONFIG, Serdes.String().getClass());
        props.put(StreamsConfig.DEFAULT_VALUE_SERDE_CLASS_CONFIG, Serdes.String().getClass());
        props.put(StreamsConfig.STATE_DIR_CONFIG, "/tmp/kafka-streams-leaderboard");
        props.put(StreamsConfig.COMMIT_INTERVAL_MS_CONFIG, 1000);
        props.put(StreamsConfig.STATESTORE_CACHE_MAX_BYTES_CONFIG, 0); // Immediate output for demo
        props.put(StreamsConfig.NUM_STREAM_THREADS_CONFIG, 2);
        return props;
    }

    // ============================================================
    // TOPOLOGY
    // ============================================================

    static Topology buildTopology() {
        StreamsBuilder builder = new StreamsBuilder();

        // ─── SOURCE: Read order events ───
        KStream<String, OrderEvent> orders = builder.stream(
            INPUT_TOPIC,
            Consumed.with(Serdes.String(), ORDER_SERDE)
                .withName("source-order-events")
        );

        // ─── FILTER: Remove invalid orders (Project 20 stateless) ───
        KStream<String, OrderEvent> validOrders = orders.filter(
            (key, order) -> order != null
                && order.getAmount() > 0
                && order.getCategory() != null
                && order.getSellerId() != null,
            Named.as("filter-valid-orders")
        );

        // ─── PEEK: Log incoming orders ───
        validOrders.peek(
            (key, order) -> System.out.printf(
                "  📦 ORDER: %s | cat=%s seller=%s region=%s $%.2f%n",
                order.getOrderId(), order.getCategory(),
                order.getSellerId(), order.getRegion(), order.getAmount()),
            Named.as("peek-valid-orders")
        );

        // ═══════════════════════════════════════════════════════════
        // METRIC 1: TOTAL ORDERS PER CATEGORY — count()
        // ═══════════════════════════════════════════════════════════
        //
        // count() is the simplest aggregation:
        //   → Maintains a Long counter per key
        //   → Increments by 1 for each record
        //   → Returns KTable<String, Long>
        //
        // WHY groupBy() here (not groupByKey()):
        //   → Input key is order_id (unique per order)
        //   → We need to group by CATEGORY (different key)
        //   → groupBy() changes the key → triggers repartition
        //   → This is NECESSARY — cannot avoid it

        KTable<String, Long> categoryCounts = validOrders
            .groupBy(
                // Re-key by category
                (orderId, order) -> order.getCategory(),
                Grouped.with(Serdes.String(), ORDER_SERDE)
                    .withName("group-by-category")
            )
            .count(
                // Named state store for:
                //   → Meaningful metrics
                //   → Interactive queries (Project 33)
                //   → Predictable changelog topic name
                Named.as("count-by-category"),
                Materialized.<String, Long, KeyValueStore<Bytes, byte[]>>as("category-count-store")
                    .withKeySerde(Serdes.String())
                    .withValueSerde(Serdes.Long())
            );

        // Output to topic
        categoryCounts
            .toStream(Named.as("category-counts-to-stream"))
            .peek(
                (category, count) -> System.out.printf(
                    "  📊 CATEGORY COUNT: %-15s → %,d orders%n", category, count),
                Named.as("peek-category-counts")
            )
            .to("category-counts",
                Produced.with(Serdes.String(), Serdes.Long())
                    .withName("sink-category-counts"));

        // ═══════════════════════════════════════════════════════════
        // METRIC 2: TOTAL REVENUE PER SELLER — reduce()
        // ═══════════════════════════════════════════════════════════
        //
        // reduce() combines two values of the SAME TYPE into one:
        //   Signature: (OrderEvent, OrderEvent) → OrderEvent
        //
        // We create a "summary" OrderEvent where amount = total revenue
        // Input type = Output type = OrderEvent
        //
        // WHY reduce() and not aggregate():
        //   → We only need to SUM the amounts
        //   → Output can stay as OrderEvent (with accumulated amount)
        //   → Simpler than aggregate() for same-type operations

        KTable<String, OrderEvent> sellerRevenue = validOrders
            .groupBy(
                (orderId, order) -> order.getSellerId(),
                Grouped.with(Serdes.String(), ORDER_SERDE)
                    .withName("group-by-seller-revenue")
            )
            .reduce(
                // Reducer: combine two OrderEvents into one with summed amount
                (accumulated, newOrder) -> {
                    // Create summary event with accumulated revenue
                    // Using a simple approach: return newOrder with updated amount
                    // In production: use a proper DTO
                    OrderEvent summary = new OrderEvent();
                    // We access via reflection or setter — for demo, use aggregate instead
                    // LIMITATION: OrderEvent is immutable, reduce needs same type
                    // This shows WHY aggregate() is often better
                    return newOrder; // Simplified — see aggregate() for proper accumulation
                },
                Named.as("reduce-seller-revenue"),
                Materialized.<String, OrderEvent, KeyValueStore<Bytes, byte[]>>as("seller-revenue-store")
                    .withKeySerde(Serdes.String())
                    .withValueSerde(ORDER_SERDE)
            );

        // BETTER APPROACH: Use aggregate() for revenue (type changes)
        // This demonstrates the reduce() limitation and why aggregate() is preferred
        KTable<String, Double> sellerRevenueProper = validOrders
            .groupBy(
                (orderId, order) -> order.getSellerId(),
                Grouped.with(Serdes.String(), ORDER_SERDE)
                    .withName("group-by-seller-revenue-proper")
            )
            .aggregate(
                // Initializer: starting value for new seller
                () -> 0.0,
                // Aggregator: add order amount to running total
                (sellerId, order, totalRevenue) -> totalRevenue + order.getAmount(),
                Named.as("aggregate-seller-revenue"),
                Materialized.<String, Double, KeyValueStore<Bytes, byte[]>>as("seller-revenue-proper-store")
                    .withKeySerde(Serdes.String())
                    .withValueSerde(Serdes.Double())
            );

        sellerRevenueProper
            .toStream(Named.as("seller-revenue-to-stream"))
            .peek(
                (seller, revenue) -> System.out.printf(
                    "  💰 SELLER REVENUE: %-12s → $%,.2f%n", seller, revenue),
                Named.as("peek-seller-revenue")
            )
            .to("seller-revenue",
                Produced.with(Serdes.String(), Serdes.Double())
                    .withName("sink-seller-revenue"));

        // ═══════════════════════════════════════════════════════════
        // METRIC 3: RUNNING AVERAGE ORDER VALUE PER REGION — aggregate()
        // ═══════════════════════════════════════════════════════════
        //
        // aggregate() is the most FLEXIBLE aggregation:
        //   Initializer: () → RegionStats (create empty accumulator)
        //   Aggregator: (key, OrderEvent, RegionStats) → RegionStats
        //
        // WHY aggregate() and not reduce():
        //   → Input type: OrderEvent
        //   → Output type: RegionStats (sum + count + average)
        //   → Different types → MUST use aggregate()
        //   → reduce() cannot change types

        KTable<String, RegionStats> regionAverages = validOrders
            .groupBy(
                (orderId, order) -> order.getRegion(),
                Grouped.with(Serdes.String(), ORDER_SERDE)
                    .withName("group-by-region")
            )
            .aggregate(
                // Initializer: called ONCE for each new key
                RegionStats::new,

                // Aggregator: called for EVERY record with matching key
                (region, order, currentStats) -> currentStats.update(region, order),

                Named.as("aggregate-region-average"),
                Materialized.<String, RegionStats, KeyValueStore<Bytes, byte[]>>as("region-average-store")
                    .withKeySerde(Serdes.String())
                    .withValueSerde(REGION_STATS_SERDE)
            );

        regionAverages
            .toStream(Named.as("region-averages-to-stream"))
            .peek(
                (region, stats) -> System.out.printf(
                    "  📈 REGION AVG: %-15s → $%,.2f (orders=%,d, total=$%,.2f)%n",
                    region, stats.getAverageAmount(),
                    stats.getOrderCount(), stats.getTotalAmount()),
                Named.as("peek-region-averages")
            )
            .mapValues(
                // Convert RegionStats to JSON string for output topic
                (readOnlyKey, stats) -> stats.toString(),
                Named.as("mapvalues-region-to-string")
            )
            .to("region-averages",
                Produced.with(Serdes.String(), Serdes.String())
                    .withName("sink-region-averages"));

        // ═══════════════════════════════════════════════════════════
        // METRIC 4: HIGHEST SINGLE ORDER PER CATEGORY — aggregate()
        // ═══════════════════════════════════════════════════════════
        //
        // Tracks the maximum order amount per category.
        // COULD use reduce() with OrderEvent comparison,
        // but aggregate() gives us richer output (CategoryMaxOrder).

        KTable<String, CategoryMaxOrder> categoryMaxOrders = validOrders
            .groupBy(
                (orderId, order) -> order.getCategory(),
                Grouped.with(Serdes.String(), ORDER_SERDE)
                    .withName("group-by-category-max")
            )
            .aggregate(
                CategoryMaxOrder::new,

def deserialize_value(raw_value, value_type):
    """Deserialize value based on expected type"""
    if raw_value is None:
        return None

    try:
        if value_type == 'long' and len(raw_value) == 8:
            return struct.unpack('>q', raw_value)[0]
        elif value_type == 'double' and len(raw_value) == 8:
            return struct.unpack('>d', raw_value)[0]
        elif value_type == 'string':
            return raw_value.decode('utf-8')
        else:
            # Try string fallback
            return raw_value.decode('utf-8')
    except (struct.error, UnicodeDecodeError):
        return f"(binary: {len(raw_value)} bytes)"
