🏷️ Phase 4 (Hard) — Advanced Patterns

🔗 Builds on: Project 18 (Kafka Transactions), Project 34 (Event Sourcing), Project 35 (CQRS)

🔴 HYPOTHETICAL PROBLEM

TravelEase is a booking platform where a single trip reservation involves:

→ Step 1: Reserve a flight (Flight Service)
→ Step 2: Reserve a hotel (Hotel Service)
→ Step 3: Reserve a rental car (Car Service)
→ Step 4: Charge the credit card (Payment Service)

REAL FAILURES HAPPENING:

Scenario A:
  → Flight reserved ✓
  → Hotel reserved ✓
  → Car reservation FAILS ✗
  → Result: Customer has a flight and hotel but no car
  → Flight and hotel charges stuck — money taken, trip incomplete

Scenario B:
  → Flight reserved ✓
  → Hotel reserved ✓
  → Car reserved ✓
  → Payment FAILS ✗
  → Result: All three services reserved inventory
  → No payment collected — company loses revenue, inventory blocked

Scenario C:
  → Flight reserved ✓
  → Hotel service is DOWN for 3 minutes
  → Timeout — what happens to the flight reservation?
  → Nobody knows — orphaned reservation sits forever

The CEO asks: "Why can't we just use a database transaction?"

ANSWER: Because each service has its OWN database.
→ Flight Service → flights_db (PostgreSQL)
→ Hotel Service → hotels_db (MongoDB)
→ Car Service → cars_db (MySQL)
→ Payment Service → payments_db (PostgreSQL)

There is NO single database to wrap BEGIN...COMMIT around.
Traditional ACID transactions CANNOT span multiple services.

We need DISTRIBUTED TRANSACTION COORDINATION without 2PC.

🟡 HOW TO APPROACH IT

CORE INSIGHT:
→ Instead of ONE big transaction, we break it into LOCAL transactions
→ Each service performs its LOCAL transaction independently
→ If any step FAILS, we execute COMPENSATING TRANSACTIONS
   to undo all previously successful steps
→ This pattern is called the SAGA PATTERN

TWO FLAVORS OF SAGA:

┌─────────────────────────────────────────────────────────┐
│          CHOREOGRAPHY SAGA (Event-Driven)               │
│                                                         │
│  → No central coordinator                               │
│  → Each service listens for events and reacts            │
│  → Each service publishes its result as a new event      │
│  → Services form a chain of reactions                    │
│  → Like a relay race — each runner passes the baton      │
│                                                         │
│  Flight ──event──▶ Hotel ──event──▶ Car ──event──▶ Pay  │
│                                                         │
│  If Car fails:                                           │
│  Car ──compensate──▶ Hotel ──compensate──▶ Flight        │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│          ORCHESTRATION SAGA (Central Coordinator)        │
│                                                         │
│  → One ORCHESTRATOR service controls the entire flow     │
│  → It sends COMMANDS to each service                     │
│  → It receives REPLIES from each service                 │
│  → It decides NEXT STEP or ROLLBACK                      │
│  → Like a conductor leading an orchestra                 │
│                                                         │
│         ┌──────────────┐                                │
│         │ ORCHESTRATOR │                                │
│         └──────┬───────┘                                │
│       ┌────────┼────────┬──────────┐                    │
│       ▼        ▼        ▼          ▼                    │
│    Flight    Hotel     Car      Payment                 │
└─────────────────────────────────────────────────────────┘

WHICH ONE FOR OUR PROBLEM?
→ 4 services with complex failure paths = ORCHESTRATION is cleaner
→ We will implement BOTH to understand the difference
→ Kafka is the communication backbone in BOTH cases

🟢 HOW CAN WE SOLVE IT

ARCHITECTURE DECISION:

KAFKA TOPICS NEEDED (Orchestration):

  COMMAND TOPICS (Orchestrator → Services):
    → flight-command     : "reserve flight" or "cancel flight"
    → hotel-command      : "reserve hotel" or "cancel hotel"
    → car-command        : "reserve car" or "cancel car"
    → payment-command    : "charge payment" or "refund payment"

  REPLY TOPICS (Services → Orchestrator):
    → flight-reply       : "flight_reserved" or "flight_failed" or "flight_cancelled"
    → hotel-reply        : "hotel_reserved" or "hotel_failed" or "hotel_cancelled"
    → car-reply          : "car_reserved" or "car_failed" or "car_cancelled"
    → payment-reply      : "payment_charged" or "payment_failed" or "payment_refunded"

  STATE TOPIC (Saga state persistence):
    → saga-state         : Current state of each saga instance (Event Sourcing from Project 34)

SAGA STATE MACHINE:

  HAPPY PATH:
  ┌──────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌───────────┐
  │ STARTED  │───▶│FLIGHT_PENDING│───▶│HOTEL_PENDING │───▶│ CAR_PENDING  │───▶│PAYMENT_PENDING│───▶│ COMPLETED │
  └──────────┘    └──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘    └───────────┘

  COMPENSATION PATH (Car fails):
  ┌──────────────┐    ┌────────────────────┐    ┌─────────────────────┐    ┌──────────┐
  │  CAR_FAILED  │───▶│COMPENSATING_HOTEL  │───▶│COMPENSATING_FLIGHT  │───▶│  FAILED  │
  └──────────────┘    └────────────────────┘    └─────────────────────┘    └──────────┘

COMPENSATING TRANSACTIONS:
  → Flight reserved   → Compensate: CANCEL flight reservation
  → Hotel reserved    → Compensate: CANCEL hotel reservation
  → Car reserved      → Compensate: CANCEL car reservation
  → Payment charged   → Compensate: REFUND payment

CRITICAL RULE: Compensating transactions must be IDEMPOTENT
→ Running "cancel flight" twice must produce same result
→ This connects directly to idempotent producer from Project 17

✅ WHAT SHOULD DO

1. DESIGN STATE MACHINE FIRST
   → Draw every state, transition, and compensation path BEFORE coding
   → Every saga instance must have a unique saga_id (UUID)
   → State transitions must be deterministic — same event always leads to same next state

2. PERSIST SAGA STATE TO KAFKA
   → Use event sourcing (Project 34) to store saga state changes
   → saga-state topic with saga_id as key → compacted topic
   → On orchestrator restart, rebuild saga state from topic

3. MAKE ALL COMPENSATIONS IDEMPOTENT
   → Each service must handle duplicate "cancel" commands gracefully
   → Use command_id to deduplicate
   → "Cancel flight that was never reserved" → return success (not error)

4. SET TIMEOUTS FOR EACH STEP
   → If hotel service doesn't reply in 30 seconds → treat as failure
   → Trigger compensation path
   → Timeout detection via scheduled checks on saga state

5. USE CORRELATION IDS
   → Every command carries saga_id + step_id
   → Every reply carries same saga_id + step_id
   → Orchestrator matches reply to pending saga

6. LOG EVERY STATE TRANSITION
   → Audit trail of exactly what happened
   → "Saga X: STARTED → FLIGHT_PENDING → HOTEL_PENDING → CAR_FAILED → COMPENSATING_HOTEL → COMPENSATING_FLIGHT → FAILED"

❌ WHAT SHOULD NOT DO

1. NEVER USE 2-PHASE COMMIT (2PC) ACROSS MICROSERVICES
   → 2PC requires all participants to be available simultaneously
   → Single coordinator failure blocks ALL participants
   → Does not scale across network boundaries
   → Kafka-based sagas tolerate temporary unavailability

2. NEVER SKIP COMPENSATION STEPS
   → If hotel compensation fails, DO NOT skip it and move to flight compensation
   → RETRY hotel compensation until it succeeds
   → A stuck compensation is better than an inconsistent state

3. NEVER ASSUME ORDERING OF REPLIES
   → Network delays may cause replies to arrive out of order
   → Orchestrator must validate current state before processing reply
   → "I received car_reserved but saga is in COMPENSATING_HOTEL state" → IGNORE

4. NEVER MAKE COMPENSATIONS THAT CANNOT BE RETRIED
   → "Send apology email" is NOT a good compensation
   → Compensations must be idempotent, retriable, and reversible

5. NEVER STORE SAGA STATE ONLY IN MEMORY
   → Orchestrator crash = lost saga state = orphaned reservations
   → Always persist to Kafka (Project 34 event sourcing)

6. NEVER USE CHOREOGRAPHY FOR MORE THAN 4 SERVICES
   → Choreography creates implicit coupling
   → Adding a 5th step requires modifying existing services
   → Debugging becomes nightmare — no single view of saga progress

WHEN TO USE SAGA PATTERN (vs alternatives):

→ USE SAGA WHEN:
  • Multiple services each own their database
  • Business process spans 3+ services
  • Eventual consistency is acceptable
  • Steps can take seconds to minutes

→ USE KAFKA TRANSACTIONS (Project 18) WHEN:
  • All operations are within Kafka (topic-to-topic)
  • No external database writes involved
  • Need atomic produce-and-consume

→ USE EVENT SOURCING + CQRS (Projects 34, 35) WHEN:
  • Single service needs audit trail and temporal queries
  • Saga uses event sourcing INTERNALLY for state persistence

🚫 WHEN NOT TO USE

1. SINGLE DATABASE OPERATIONS
   → If all data lives in one database → use database transactions
   → Saga adds unnecessary complexity

2. SYNCHRONOUS REQUEST-RESPONSE FLOWS
   → If the user MUST get immediate confirmation
   → Saga is inherently asynchronous — user gets "booking in progress"
   → If business requires synchronous → consider API composition pattern

3. OPERATIONS THAT CANNOT BE COMPENSATED
   → "Launch the missile" → cannot be undone
   → "Send physical mail" → cannot be recalled
   → If any step is non-compensable → saga is wrong pattern

4. LOW-VALUE OPERATIONS
   → Saga adds infrastructure complexity
   → For a simple notification system → overkill
   → Reserve saga for business-critical multi-step processes

5. TIGHT LATENCY REQUIREMENTS
   → Each saga step involves Kafka produce → consume → process → produce
   → Minimum ~50ms per step × 4 steps = 200ms minimum
   → If sub-100ms required → saga is too slow

⚙️ HOW IT ONLY WORKS (INTERNAL MECHANICS)

ORCHESTRATION SAGA — STEP-BY-STEP INTERNAL FLOW:

STEP 1: USER TRIGGERS BOOKING
  ┌──────────┐
  │ API Call  │──▶ Orchestrator receives booking request
  └──────────┘
  Orchestrator:
    → Generates saga_id = "saga-7721"
    → Creates initial state: {saga_id: "saga-7721", state: "STARTED", steps_completed: []}
    → Publishes state event to saga-state topic (key: "saga-7721")
    → Transitions to FLIGHT_PENDING
    → Publishes command to flight-command topic:
      {saga_id: "saga-7721", command: "RESERVE", flight: "AA-101", date: "2025-03-15"}

STEP 2: FLIGHT SERVICE PROCESSES
  Flight Service (consumer of flight-command):
    → Reads command
    → Executes LOCAL database transaction: INSERT INTO reservations ...
    → Publishes reply to flight-reply topic:
      {saga_id: "saga-7721", status: "FLIGHT_RESERVED", confirmation: "FL-9982"}

STEP 3: ORCHESTRATOR RECEIVES FLIGHT REPLY
  Orchestrator (consumer of flight-reply):
    → Reads reply for saga-7721
    → Validates current state is FLIGHT_PENDING ← CRITICAL CHECK
    → Updates state: {state: "HOTEL_PENDING", steps_completed: ["FLIGHT"]}
    → Publishes to saga-state topic
    → Publishes command to hotel-command topic:
      {saga_id: "saga-7721", command: "RESERVE", hotel: "Hilton-NYC", dates: [...]}

STEP 4–6: HOTEL AND CAR FOLLOW SAME PATTERN...

STEP 7: CAR SERVICE FAILS
  Car Service (consumer of car-command):
    → Reads command
    → LOCAL database check: no cars available for that date
    → Publishes reply to car-reply topic:
      {saga_id: "saga-7721", status: "CAR_FAILED", reason: "NO_AVAILABILITY"}

STEP 8: ORCHESTRATOR TRIGGERS COMPENSATION
  Orchestrator (consumer of car-reply):
    → Reads failure for saga-7721
    → Current state: CAR_PENDING, steps_completed: ["FLIGHT", "HOTEL"]
    → STATE MACHINE DECISION: Enter compensation path
    → Transitions to COMPENSATING_HOTEL
    → Publishes command to hotel-command topic:
      {saga_id: "saga-7721", command: "CANCEL", hotel: "Hilton-NYC"}

STEP 9: HOTEL COMPENSATION
  Hotel Service:
    → Reads cancel command
    → LOCAL transaction: DELETE FROM reservations WHERE saga_id = 'saga-7721'
    → Publishes reply: {saga_id: "saga-7721", status: "HOTEL_CANCELLED"}

STEP 10: ORCHESTRATOR CONTINUES COMPENSATION
  Orchestrator:
    → Receives HOTEL_CANCELLED
    → Transitions to COMPENSATING_FLIGHT
    → Publishes cancel to flight-command:
      {saga_id: "saga-7721", command: "CANCEL", confirmation: "FL-9982"}

STEP 11: FLIGHT COMPENSATION
  Flight Service:
    → Cancels reservation
    → Publishes: {saga_id: "saga-7721", status: "FLIGHT_CANCELLED"}

STEP 12: SAGA COMPLETES WITH FAILURE
  Orchestrator:
    → Receives FLIGHT_CANCELLED
    → All compensations done
    → Transitions to FAILED
    → Final state: {state: "FAILED", reason: "CAR_NO_AVAILABILITY", compensated: ["HOTEL", "FLIGHT"]}
    → Notifies user: "Booking failed — no cars available. All reservations cancelled."

CHOREOGRAPHY VARIANT — HOW IT DIFFERS:

  No orchestrator exists. Each service listens and reacts:

  Flight Service:
    → Listens to: booking-requested
    → On success: publishes flight-reserved
    → On failure: publishes flight-failed

  Hotel Service:
    → Listens to: flight-reserved
    → On success: publishes hotel-reserved
    → On failure: publishes hotel-failed + triggers flight-cancel

  Car Service:
    → Listens to: hotel-reserved
    → On success: publishes car-reserved
    → On failure: publishes car-failed + triggers hotel-cancel

  PROBLEM: Car Service must know about Hotel Service
  → Implicit coupling grows with each new step
  → No single place to see saga progress
  → This is why orchestration is preferred for 4+ steps

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: ZOMBIE SAGAS
  Problem: Orchestrator crashes between receiving reply and sending next command
  Result: Saga stuck in intermediate state forever
  Solution:
    → On restart, read saga-state topic
    → Find sagas in non-terminal states (not COMPLETED or FAILED)
    → Check timestamp — if stuck > timeout threshold → resume or compensate
    → Kafka transactions (Project 18) minimize this window

PITFALL 2: DUPLICATE COMMANDS
  Problem: Orchestrator retries sending command due to Kafka timeout
  Result: Flight reserved TWICE for same saga
  Solution:
    → Every command has command_id (saga_id + step_name + attempt)
    → Services deduplicate by command_id
    → Idempotent processing — "reserve if not already reserved for this saga"

PITFALL 3: COMPENSATION FAILURE
  Problem: Hotel cancel command fails (hotel service bug)
  Result: Compensation chain stuck — flight remains reserved
  Solution:
    → NEVER skip compensation steps
    → Retry with exponential backoff (Project 26)
    → After N retries → move to COMPENSATION_STUCK state
    → Alert operations team for manual intervention
    → Parking lot topic (Project 26) for stuck compensations

PITFALL 4: OUT-OF-ORDER EVENTS
  Problem: Network delay causes old "flight_reserved" to arrive
           after saga already moved to COMPENSATING state
  Result: Orchestrator might incorrectly advance forward
  Solution:
    → State machine validates: "Is this reply valid for current state?"
    → If saga is in COMPENSATING_HOTEL and receives FLIGHT_RESERVED → IGNORE
    → State machine is the SINGLE SOURCE OF TRUTH

PITFALL 5: SEMANTIC LOCK ANTI-PATTERN
  Problem: Flight is reserved but saga hasn't completed yet
           Another user tries to book same flight → inventory locked
  Result: Inventory blocked for duration of entire saga
  Solution:
    → Use "tentative reservation" status in services
    → Tentative reservations auto-expire after timeout
    → Only mark as "confirmed" when saga reaches COMPLETED

PITFALL 6: SAGA STATE TOPIC GROWS UNBOUNDED
  Problem: Every state transition adds a record to saga-state
  Result: Topic size grows forever
  Solution:
    → Use log compaction (Project 14) on saga-state topic
    → Only latest state per saga_id retained
    → Completed/failed sagas can be archived and tombstoned

📈 SCALABILITY CONSIDERATIONS

HORIZONTAL SCALING OF ORCHESTRATOR:
  → saga-state topic partitioned by saga_id
  → Multiple orchestrator instances in same consumer group
  → Each instance handles subset of sagas
  → On rebalance (Project 10), sagas redistributed
  → State rebuilt from saga-state topic partition

SCALING INDIVIDUAL SERVICES:
  → flight-command topic with N partitions
  → N flight service instances in consumer group
  → Commands for different sagas processed in parallel
  → Commands for SAME saga → same partition (saga_id as key)
    → Guarantees ordering within one saga

THROUGHPUT MATH:
  → 1 saga = 8 Kafka messages minimum (4 commands + 4 replies)
  → At 100K messages/sec Kafka throughput
  → ~12,500 sagas/sec theoretical maximum
  → Real bottleneck: service processing time, not Kafka

from confluent_kafka.admin import AdminClient, NewTopic
import time

def create_saga_topics():
    admin = AdminClient({'bootstrap.servers': 'localhost:9092'})

    topics = [
        # Saga state — compacted (Project 14) for latest state per saga
        NewTopic(
            topic='saga-state',
            num_partitions=12,
            replication_factor=1,
            config={'cleanup.policy': 'compact'}
        ),

        # Command topics — orchestrator sends commands to services
        NewTopic(topic='flight-command', num_partitions=6, replication_factor=1),
        NewTopic(topic='hotel-command', num_partitions=6, replication_factor=1),
        NewTopic(topic='car-command', num_partitions=6, replication_factor=1),
        NewTopic(topic='payment-command', num_partitions=6, replication_factor=1),

        # Reply topics — services send results back to orchestrator
        NewTopic(topic='flight-reply', num_partitions=6, replication_factor=1),
        NewTopic(topic='hotel-reply', num_partitions=6, replication_factor=1),
        NewTopic(topic='car-reply', num_partitions=6, replication_factor=1),
        NewTopic(topic='payment-reply', num_partitions=6, replication_factor=1),

        # Dead letter queue for stuck compensations (Project 25)
        NewTopic(topic='saga-dlq', num_partitions=3, replication_factor=1),

        # Choreography topics (for choreography demo)
        NewTopic(topic='booking-events', num_partitions=6, replication_factor=1),
    ]

    futures = admin.create_topics(topics)
    for topic_name, future in futures.items():
        try:
            future.result()
            print(f"  ✅ Created: {topic_name}")
        except Exception as e:
            if 'already exists' in str(e):
                print(f"  ⏭️  Exists:  {topic_name}")
            else:
                print(f"  ❌ Error:   {topic_name} → {e}")

if __name__ == '__main__':
    print("=" * 60)
    print("CREATING SAGA TOPICS")

from enum import Enum

class SagaState(Enum):

    Every possible state a travel booking saga can be in.

    PAYMENT_PENDING = "PAYMENT_PENDING"

    # Compensation path (rollback)
    COMPENSATING_CAR = "COMPENSATING_CAR"
    COMPENSATING_HOTEL = "COMPENSATING_HOTEL"
    COMPENSATING_FLIGHT = "COMPENSATING_FLIGHT"
    FAILED = "FAILED"


class CommandType(Enum):

    RESERVE_HOTEL = "RESERVE_HOTEL"

@dataclass
class SagaInstance:
    """
    Complete state of one saga instance.
    This gets serialized and stored in saga-state topic.
    Follows Event Sourcing principles from Project 34.
    """
    saga_id: str
    state: SagaState
    booking_request: dict
    steps_completed: List[str] = field(default_factory=list)
    confirmations: Dict[str, str] = field(default_factory=dict)
    failure_reason: Optional[str] = None
    created_at: float = field(default_factory=time.time)
    updated_at: float = field(default_factory=time.time)
    history: List[dict] = field(default_factory=list)

    def to_json(self) -> str:
        data = {
            'saga_id': self.saga_id,
            'state': self.state.value,
            'booking_request': self.booking_request,
            'steps_completed': self.steps_completed,
            'confirmations': self.confirmations,
            'failure_reason': self.failure_reason,
            'created_at': self.created_at,
            'updated_at': self.updated_at,
            'history': self.history
        }
        return json.dumps(data)

    @classmethod
    def from_json(cls, json_str: str) -> 'SagaInstance':
        data = json.loads(json_str)
        data['state'] = SagaState(data['state'])
        return cls(**data)

    def add_history(self, from_state: str, to_state: str, trigger: str):
        self.history.append({
            'from': from_state,
            'to': to_state,
            'trigger': trigger,
            'timestamp': time.time()
        })
        self.updated_at = time.time()

FORWARD_TRANSITIONS = {
    # State                  Reply received           → Next state              Command to send        Target topic
    (SagaState.STARTED, None): (
        SagaState.FLIGHT_PENDING,
        CommandType.RESERVE_FLIGHT,
        'flight-command'
    ),
    (SagaState.FLIGHT_PENDING, ReplyStatus.FLIGHT_RESERVED): (
        SagaState.HOTEL_PENDING,
        CommandType.RESERVE_HOTEL,
        'hotel-command'
    ),
    (SagaState.HOTEL_PENDING, ReplyStatus.HOTEL_RESERVED): (
        SagaState.CAR_PENDING,
        CommandType.RESERVE_CAR,
        'car-command'
    ),
    (SagaState.CAR_PENDING, ReplyStatus.CAR_RESERVED): (
        SagaState.PAYMENT_PENDING,
        CommandType.CHARGE_PAYMENT,
        'payment-command'
    ),
    (SagaState.PAYMENT_PENDING, ReplyStatus.PAYMENT_CHARGED): (
        SagaState.COMPLETED,
        None,
        None
    ),
}

# Compensation triggers: which reply causes compensation to start
# and what is the first compensation step
COMPENSATION_TRIGGERS = {
    (SagaState.FLIGHT_PENDING, ReplyStatus.FLIGHT_FAILED): (
        SagaState.FAILED,  # Nothing to compensate
        None,
        None
    ),
    (SagaState.HOTEL_PENDING, ReplyStatus.HOTEL_FAILED): (
        SagaState.COMPENSATING_FLIGHT,
        CommandType.CANCEL_FLIGHT,
        'flight-command'
    ),
    (SagaState.CAR_PENDING, ReplyStatus.CAR_FAILED): (
        SagaState.COMPENSATING_HOTEL,
        CommandType.CANCEL_HOTEL,
        'hotel-command'
    ),
    (SagaState.PAYMENT_PENDING, ReplyStatus.PAYMENT_FAILED): (
        SagaState.COMPENSATING_CAR,
        CommandType.CANCEL_CAR,
        'car-command'
    ),
}

# Compensation chain: after each successful compensation, what's next
COMPENSATION_CHAIN = {
    (SagaState.COMPENSATING_CAR, ReplyStatus.CAR_CANCELLED): (
        SagaState.COMPENSATING_HOTEL,
        CommandType.CANCEL_HOTEL,
        'hotel-command'
    ),
    (SagaState.COMPENSATING_HOTEL, ReplyStatus.HOTEL_CANCELLED): (
        SagaState.COMPENSATING_FLIGHT,
        CommandType.CANCEL_FLIGHT,
        'flight-command'
    ),
    (SagaState.COMPENSATING_FLIGHT, ReplyStatus.FLIGHT_CANCELLED): (
        SagaState.FAILED,
        None,
        None
    ),
}


def get_next_transition(current_state: SagaState, reply: Optional[ReplyStatus]):
    """
    Given current state and received reply, determine:
    → Next state
    → Command to send (if any)
    → Target topic for that command
    Returns None if transition is invalid (stale/duplicate event)
    """
    # Check forward path
    key = (current_state, reply)
    if key in FORWARD_TRANSITIONS:
        return FORWARD_TRANSITIONS[key]

    # Check compensation triggers
    if key in COMPENSATION_TRIGGERS:
        return COMPENSATION_TRIGGERS[key]

    # Check compensation chain
    if key in COMPENSATION_CHAIN:
        return COMPENSATION_CHAIN[key]

    # Invalid transition — stale event or duplicate
    return None

# PURPOSE: Central saga coordinator that drives the booking flow

from confluent_kafka import Producer, Consumer, KafkaError
import json
import time
import uuid
import threading
from saga_state_machine import (
    SagaState, SagaInstance, CommandType, ReplyStatus,
    get_next_transition

class SagaOrchestrator:
    """
    The conductor of the saga orchestra.
    → Sends commands to service topics
    → Listens for replies from services
    → Manages state transitions using the state machine
    → Persists state to saga-state topic (Event Sourcing)
    """

    def __init__(self, broker='localhost:9092'):
        self.broker = broker

        # Producer for sending commands and state updates
        # Using transactional producer (Project 18)
        self.producer = Producer({
            'bootstrap.servers': broker,
            'acks': 'all',  # Project 11: strongest durability
            'enable.idempotence': True,  # Project 17: no duplicates
            'linger.ms': 5,
            'client.id': 'saga-orchestrator-producer'
        })

        # Consumer for receiving replies from ALL services
        self.consumer = Consumer({
            'bootstrap.servers': broker,
            'group.id': 'saga-orchestrator-group',
            'auto.offset.reset': 'earliest',
            'enable.auto.commit': True,
            'client.id': 'saga-orchestrator-consumer'
        })

        # Subscribe to ALL reply topics
        self.reply_topics = [
            'flight-reply', 'hotel-reply',
            'car-reply', 'payment-reply'
        ]
        self.consumer.subscribe(self.reply_topics)

        # In-memory saga store (rebuilt from saga-state on startup)
        # In production: use RocksDB or Redis
        self.sagas = {}
        self.running = True

        print("=" * 60)
        print("🎼 SAGA ORCHESTRATOR INITIALIZED")
        print(f"   Listening on: {', '.join(self.reply_topics)}")
        print("=" * 60)

    def _publish(self, topic: str, key: str, value: dict):
        """
        Publish a message to a Kafka topic.
        Key = saga_id ensures same saga always goes to same partition.
        """
        self.producer.produce(
            topic=topic,
            key=key.encode('utf-8'),
            value=json.dumps(value).encode('utf-8'),
            callback=self._delivery_callback
        )
        self.producer.flush()

    def _delivery_callback(self, err, msg):
        if err:
            print(f"  ❌ Delivery failed: {err}")
        else:
            topic = msg.topic()
            partition = msg.partition()
            offset = msg.offset()
            print(f"  📤 Delivered to {topic}[{partition}]@{offset}")

    def _persist_saga_state(self, saga: SagaInstance):
        """
        Persist saga state to saga-state topic.
        This is Event Sourcing (Project 34) — every state change is an event.
        Topic is compacted (Project 14) — only latest state retained per saga_id.
        """
        self._publish('saga-state', saga.saga_id, json.loads(saga.to_json()))

    def _build_command_payload(self, saga: SagaInstance, command: CommandType) -> dict:
        """Build the command message payload with all necessary context"""
        booking = saga.booking_request
        base = {
            'saga_id': saga.saga_id,
            'command': command.value,
            'command_id': f"{saga.saga_id}-{command.value}-{int(time.time()*1000)}",
            'timestamp': time.time()
        }

        if command == CommandType.RESERVE_FLIGHT:
            base['flight'] = booking.get('flight', 'AA-101')
            base['date'] = booking.get('date', '2025-03-15')
            base['passenger'] = booking.get('passenger', 'John Doe')

        elif command == CommandType.RESERVE_HOTEL:
            base['hotel'] = booking.get('hotel', 'Hilton-NYC')
            base['check_in'] = booking.get('date', '2025-03-15')
            base['nights'] = booking.get('nights', 3)

        elif command == CommandType.RESERVE_CAR:
            base['car_type'] = booking.get('car_type', 'sedan')
            base['pickup_date'] = booking.get('date', '2025-03-15')
            base['days'] = booking.get('days', 3)

        elif command == CommandType.CHARGE_PAYMENT:
            base['amount'] = booking.get('total_amount', 1500.00)
            base['card_last4'] = booking.get('card_last4', '4242')
            base['confirmations'] = saga.confirmations

        elif command in (CommandType.CANCEL_FLIGHT, CommandType.CANCEL_HOTEL,
                         CommandType.CANCEL_CAR, CommandType.REFUND_PAYMENT):
            base['confirmations'] = saga.confirmations
            base['reason'] = saga.failure_reason or 'Saga compensation'

        return base

    # ============================================================
    # PUBLIC API: Start a new saga
    # ============================================================
    def start_saga(self, booking_request: dict) -> str:
        """
        Entry point: user wants to book a trip.
        Creates a new saga instance and kicks off the first step.
        """
        saga_id = f"saga-{uuid.uuid4().hex[:8]}"

        saga = SagaInstance(
            saga_id=saga_id,
            state=SagaState.STARTED,
            booking_request=booking_request
        )

        print(f"\n{'='*60}")
        print(f"🆕 NEW SAGA: {saga_id}")
        print(f"   Booking: {json.dumps(booking_request, indent=2)}")
        print(f"{'='*60}")

        # Get first transition
        transition = get_next_transition(SagaState.STARTED, None)
        if transition is None:
            print(f"  ❌ No valid initial transition!")
            return saga_id

        next_state, command, target_topic = transition

        # Record state change
        old_state = saga.state.value
        saga.state = next_state
        saga.add_history(old_state, next_state.value, 'SAGA_STARTED')

        # Persist state
        self._persist_saga_state(saga)
        self.sagas[saga_id] = saga

        # Send first command
        if command and target_topic:
            payload = self._build_command_payload(saga, command)
            print(f"  📨 Sending {command.value} → {target_topic}")
            self._publish(target_topic, saga_id, payload)

        return saga_id

    # ============================================================
    # CORE: Process a reply from a service
    # ============================================================
    def _process_reply(self, saga_id: str, reply_status: str,
                       reply_data: dict):
        """
        Core state machine execution.
        Called when a service reply arrives.
        """
        if saga_id not in self.sagas:
            print(f"  ⚠️  Unknown saga: {saga_id} — ignoring")
            return

        saga = self.sagas[saga_id]

        # Parse reply status
        try:
            reply = ReplyStatus(reply_status)
        except ValueError:
            print(f"  ⚠️  Unknown reply status: {reply_status}")
            return

        print(f"\n  📩 Reply received for {saga_id}")
        print(f"     Current state: {saga.state.value}")
        print(f"     Reply: {reply.value}")

        # Check if saga is in terminal state — ignore stale replies
        if saga.state in (SagaState.COMPLETED, SagaState.FAILED):
            print(f"  ⏭️  Saga already in terminal state {saga.state.value} — ignoring")
            return

        # Look up transition in state machine
        transition = get_next_transition(saga.state, reply)

        if transition is None:
            print(f"  ⚠️  No valid transition from {saga.state.value} with {reply.value}")
            print(f"     Possibly stale/duplicate reply — IGNORING")
            return

        next_state, command, target_topic = transition

        # Record confirmation if this was a successful forward step
        confirmation = reply_data.get('confirmation')
        if confirmation:
            step_name = reply.value.split('_')[0].lower()
            saga.confirmations[step_name] = confirmation
            saga.steps_completed.append(step_name)

        # Record failure reason if this was a failure
        failure_reason = reply_data.get('reason')
        if failure_reason and not saga.failure_reason:
            saga.failure_reason = failure_reason

        # Execute transition
        old_state = saga.state.value
        saga.state = next_state
        saga.add_history(old_state, next_state.value, reply.value)

        print(f"  🔄 Transition: {old_state} → {next_state.value}")

        # Persist updated state
        self._persist_saga_state(saga)

        # Handle terminal states
        if next_state == SagaState.COMPLETED:
            print(f"\n  ✅✅✅ SAGA {saga_id} COMPLETED SUCCESSFULLY! ✅✅✅")
            print(f"     Confirmations: {saga.confirmations}")
            duration = time.time() - saga.created_at
            print(f"     Duration: {duration:.2f}s")
            return

        if next_state == SagaState.FAILED:
            print(f"\n  ❌❌❌ SAGA {saga_id} FAILED ❌❌❌")
            print(f"     Reason: {saga.failure_reason}")
            print(f"     Compensated steps: {saga.steps_completed}")
            duration = time.time() - saga.created_at
            print(f"     Duration: {duration:.2f}s")
            return

        # Send next command
        if command and target_topic:
            payload = self._build_command_payload(saga, command)
            is_compensation = 'CANCEL' in command.value or 'REFUND' in command.value
            emoji = "🔙" if is_compensation else "📨"
            label = "COMPENSATING" if is_compensation else "Sending"
            print(f"  {emoji} {label}: {command.value} → {target_topic}")
            self._publish(target_topic, saga_id, payload)

    # ============================================================
    # REPLY LISTENER: Continuously poll for service replies
    # ============================================================
    def listen_for_replies(self):
        """
        Main event loop.
        Consumes from all reply topics and routes to state machine.
        """
        print("\n👂 Listening for service replies...\n")

        while self.running:
            msg = self.consumer.poll(timeout=1.0)
            if msg is None:
                continue

            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    continue
                print(f"  ❌ Consumer error: {msg.error()}")
                continue

            try:
                topic = msg.topic()
                key = msg.key().decode('utf-8') if msg.key() else None
                value = json.loads(msg.value().decode('utf-8'))

                saga_id = value.get('saga_id', key)
                reply_status = value.get('status')

                if saga_id and reply_status:
                    self._process_reply(saga_id, reply_status, value)

            except Exception as e:
                print(f"  ❌ Error processing reply: {e}")
                import traceback
                traceback.print_exc()

    def shutdown(self):
        self.running = False
        self.consumer.close()
        print("\n🛑 Orchestrator shut down")

    def get_saga_status(self, saga_id: str) -> dict:
        """API endpoint: check saga status"""
        if saga_id in self.sagas:
            saga = self.sagas[saga_id]
            return {
                'saga_id': saga_id,
                'state': saga.state.value,
                'steps_completed': saga.steps_completed,
                'confirmations': saga.confirmations,
                'failure_reason': saga.failure_reason,
                'history': saga.history
            }
        return {'error': f'Saga {saga_id} not found'}

from confluent_kafka import Producer, Consumer, KafkaError

class ServiceSimulator:
    """
    Base class for all service simulators.
    Each service:
    → Consumes from its command topic
    → Processes the command (simulated)
    → Publishes reply to its reply topic
    """

    def __init__(self, service_name: str, command_topic: str,
                 reply_topic: str, broker='localhost:9092',
                 failure_rate=0.0, processing_delay_ms=200):

        self.service_name = service_name
        self.command_topic = command_topic
        self.reply_topic = reply_topic
        self.failure_rate = failure_rate
        self.processing_delay = processing_delay_ms / 1000.0

        # In-memory "database" for this service
        # In production: actual PostgreSQL, MongoDB, etc.
        self.reservations = {}
        self.processed_commands = set()  # For idempotency

        self.producer = Producer({
            'bootstrap.servers': broker,
            'acks': 'all',
            'enable.idempotence': True
        })

        self.consumer = Consumer({
            'bootstrap.servers': broker,
            'group.id': f'{service_name}-group',
            'auto.offset.reset': 'earliest',
            'enable.auto.commit': True
        })
        self.consumer.subscribe([command_topic])

        self.running = True
        print(f"  🔧 {service_name} initialized | Listening: {command_topic} | Replying: {reply_topic}")

    def _publish_reply(self, saga_id: str, reply: dict):
        self.producer.produce(
            topic=self.reply_topic,
            key=saga_id.encode('utf-8'),
            value=json.dumps(reply).encode('utf-8')
        )
        self.producer.flush()

    def process_command(self, command_data: dict):
        """Override in each service"""
        raise NotImplementedError

    def run(self):
        while self.running:
            msg = self.consumer.poll(timeout=1.0)
            if msg is None:
                continue
            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    continue
                continue

            try:
                value = json.loads(msg.value().decode('utf-8'))
                command_id = value.get('command_id', '')

                # IDEMPOTENCY CHECK — never process same command twice
                if command_id in self.processed_commands:
                    print(f"  ⏭️  {self.service_name}: Duplicate command {command_id} — skipping")
                    continue

                self.processed_commands.add(command_id)

                # Simulate processing delay
                time.sleep(self.processing_delay)

                # Process and reply
                self.process_command(value)

            except Exception as e:
                print(f"  ❌ {self.service_name} error: {e}")

    def stop(self):
        self.running = False
        self.consumer.close()

# PURPOSE: Demonstrate the CHOREOGRAPHY alternative
#          No central orchestrator — services react to events

from confluent_kafka import Producer, Consumer, KafkaError
import json
import time
import uuid
import random
import threading


BOOKING_EVENTS_TOPIC = 'booking-events'


class ChoreographyService:
    """
    Base class for choreography-based services.
    Each service:
    → Listens for specific event types on booking-events topic
    → Processes locally
    → Publishes result event to SAME booking-events topic
    
    KEY DIFFERENCE FROM ORCHESTRATION:
    → No command/reply separation
    → All events on ONE topic
    → Each service filters for events it cares about
    → No central coordinator
    """

    def __init__(self, service_name: str, listens_for: list,
                 broker='localhost:9092', failure_rate=0.0):
        self.service_name = service_name
        self.listens_for = listens_for
        self.failure_rate = failure_rate

        self.producer = Producer({
            'bootstrap.servers': broker,
            'acks': 'all',
            'enable.idempotence': True
        })

        self.consumer = Consumer({
            'bootstrap.servers': broker,
            'group.id': f'choreo-{service_name}-group',
            'auto.offset.reset': 'earliest',
            'enable.auto.commit': True
        })
        self.consumer.subscribe([BOOKING_EVENTS_TOPIC])

        self.running = True
        self.processed = set()

    def publish_event(self, saga_id: str, event: dict):
        self.producer.produce(
            topic=BOOKING_EVENTS_TOPIC,
            key=saga_id.encode('utf-8'),
            value=json.dumps(event).encode('utf-8')
        )
        self.producer.flush()

    def handle_event(self, event: dict):
        """Override per service"""
        raise NotImplementedError

    def run(self):
        while self.running:
            msg = self.consumer.poll(timeout=1.0)
            if msg is None:
                continue
            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    continue
                continue

            try:
                value = json.loads(msg.value().decode('utf-8'))
                event_type = value.get('event_type', '')

                # FILTER: only process events this service cares about
                if event_type not in self.listens_for:
                    continue

                # Idempotency
                event_id = value.get('event_id', '')
                if event_id in self.processed:
                    continue
                self.processed.add(event_id)

                time.sleep(0.3)  # Simulate processing
                self.handle_event(value)

            except Exception as e:
                print(f"  ❌ {self.service_name}: {e}")

    def stop(self):
        self.running = False
        self.consumer.close()

# CHOREOGRAPHY OBSERVER
# Not a participant — just watches all events for visibility
# This is the PROBLEM with choreography: you NEED this extra
# component just to see what's happening
# ============================================================
class SagaObserver(ChoreographyService):

    def __init__(self):
        super().__init__(
            service_name='choreo-observer',
            listens_for=[
                'BOOKING_REQUESTED', 'FLIGHT_RESERVED', 'FLIGHT_FAILED',
                'HOTEL_RESERVED', 'HOTEL_FAILED', 'CAR_RESERVED',
                'CAR_FAILED', 'PAYMENT_FAILED', 'BOOKING_COMPLETED',
                'FLIGHT_CANCELLED', 'HOTEL_CANCELLED', 'CAR_CANCELLED'
            ]
        )
        self.saga_timelines = {}

    def handle_event(self, event: dict):
        saga_id = event['saga_id']
        event_type = event['event_type']

        if saga_id not in self.saga_timelines:
            self.saga_timelines[saga_id] = []

        self.saga_timelines[saga_id].append({
            'event': event_type,
            'time': time.time()
        })

        # Check for terminal events
        if event_type == 'BOOKING_COMPLETED':
            print(f"\n  📊 [OBSERVER] Saga {saga_id} COMPLETED!")
            self._print_timeline(saga_id)
        elif event_type in ('FLIGHT_FAILED',):
            print(f"\n  📊 [OBSERVER] Saga {saga_id} FAILED at flight!")
            self._print_timeline(saga_id)
        elif event_type == 'FLIGHT_CANCELLED':
            # Last compensation step done
            trigger = event.get('trigger', '')
            print(f"\n  📊 [OBSERVER] Saga {saga_id} fully compensated (trigger: {trigger})")
            self._print_timeline(saga_id)

    def _print_timeline(self, saga_id):
        events = self.saga_timelines.get(saga_id, [])
        if not events:
            return
        start = events[0]['time']
        print(f"     Timeline for {saga_id}:")
        for e in events:
            elapsed = e['time'] - start
            print(f"       [{elapsed:6.3f}s] {e['event']}")

# FILE: saga_state_rebuilder.py
# PURPOSE: On orchestrator restart, rebuild saga state
#          from saga-state topic (Event Sourcing — Project 34)
# THIS IS THE CRASH RECOVERY MECHANISM
# ============================================================

from confluent_kafka import Consumer, KafkaError
import json
import time
from saga_state_machine import SagaState, SagaInstance


class SagaStateRebuilder:
    """
    Reads ALL records from saga-state topic to rebuild
    in-memory saga state.
    
    Because saga-state is compacted (Project 14):
    → Only latest state per saga_id exists
    → Reading entire topic gives current state of ALL sagas
    
    This is Event Sourcing replay (Project 34) applied to saga.
    """

    def __init__(self, broker='localhost:9092'):
        self.consumer = Consumer({
            'bootstrap.servers': broker,
            'group.id': f'saga-rebuilder-{int(time.time())}',
            'auto.offset.reset': 'earliest',
            'enable.auto.commit': False
        })

    def rebuild(self) -> dict:
        """
        Read saga-state topic from beginning.
        Returns dict of saga_id → SagaInstance
        """
        self.consumer.subscribe(['saga-state'])

        sagas = {}
        empty_polls = 0
        max_empty = 10  # Stop after 10 consecutive empty polls

        print("\n" + "=" * 60)
        print("🔄 REBUILDING SAGA STATE FROM TOPIC")
        print("=" * 60)

        while empty_polls < max_empty:
            msg = self.consumer.poll(timeout=1.0)

            if msg is None:
                empty_polls += 1
                continue

            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    empty_polls += 1
                    continue
                print(f"  ❌ Error: {msg.error()}")
                continue

            empty_polls = 0  # Reset on successful read

            try:
                key = msg.key().decode('utf-8') if msg.key() else None
                value = json.loads(msg.value().decode('utf-8'))

                if key:
                    saga = SagaInstance.from_json(json.dumps(value))
                    sagas[key] = saga

            except Exception as e:
                print(f"  ⚠️  Skipping corrupt record: {e}")

        self.consumer.close()

        # Categorize results
        completed = sum(1 for s in sagas.values()
                        if s.state == SagaState.COMPLETED)
        failed = sum(1 for s in sagas.values()
                     if s.state == SagaState.FAILED)
        in_progress = sum(1 for s in sagas.values()
                          if s.state not in
                          (SagaState.COMPLETED, SagaState.FAILED))

        print(f"\n  📊 Rebuilt {len(sagas)} sagas:")
        print(f"     ✅ Completed:   {completed}")
        print(f"     ❌ Failed:      {failed}")
        print(f"     ⏳ In-Progress: {in_progress}")

        # Flag stuck sagas
        if in_progress > 0:
            print(f"\n  ⚠️  STUCK SAGAS DETECTED:")
            for saga_id, saga in sagas.items():
                if saga.state not in (SagaState.COMPLETED, SagaState.FAILED):
                    age = time.time() - saga.updated_at
                    print(f"     {saga_id}: state={saga.state.value}, "
                          f"age={age:.0f}s")
                    if age > 60:
                        print(f"       → CRITICAL: Stuck for >60s — needs intervention")

        print("=" * 60)
        return sagas
