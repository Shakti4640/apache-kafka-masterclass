PROJECT 7: THE MULTILINGUAL POST OFFICE — MESSAGE SERIALIZATION

🎯 HYPOTHETICAL PROBLEM

You manage "BytePost International" — a global postal service

→ Letters arrive from 50 countries in 50 different languages
→ Some letters are in English (plain text)
→ Some are in Chinese (different character encoding)
→ Some contain structured forms (JSON-like documents)
→ Some contain photos (binary data)
→ Some contain encrypted documents (custom binary format)

HERE IS THE PROBLEM:

The central sorting machine ONLY understands ONE format: RAW BYTES
→ It cannot read English, Chinese, or any language directly
→ Every letter must be CONVERTED to bytes before entering the machine
→ Every letter must be CONVERTED BACK from bytes when delivered

SCENARIO A: Sender writes in English, receiver expects English
→ Sender converts: "Hello" → bytes [72, 101, 108, 108, 111]
→ Receiver converts: bytes [72, 101, 108, 108, 111] → "Hello"
→ ✅ Works perfectly

SCENARIO B: Sender writes JSON, receiver expects plain string
→ Sender converts: {"name": "Alice"} → bytes
→ Receiver tries to read as plain string: '{"name": "Alice"}'
→ ⚠️ Technically works but receiver sees raw JSON, not a parsed object

SCENARIO C: Sender writes with Python pickle, receiver uses Java
→ Sender converts: Python object → pickle bytes
→ Receiver (Java) tries to read pickle bytes → 💀 CRASH
→ ❌ Incompatible serialization formats

SCENARIO D: Sender adds a new field, receiver uses old code
→ Sender sends: {"name": "Alice", "age": 30}
→ Old receiver expects only: {"name": "Alice"}
→ What happens? Depends entirely on the serialization format!
→ JSON: silently ignores extra field ✅
→ Fixed binary: reads wrong bytes → CORRUPTION 💀

YOU NEED:
→ A clear strategy for HOW data is converted to/from bytes
→ Producer and consumer MUST agree on the format
→ Format must handle future changes gracefully
→ Format must be efficient for high-throughput systems

  → ALL interpretation happens at the PRODUCER and CONSUMER side
  → Producer and consumer MUST agree on the format

✅ WHAT SHOULD DO

1. ALWAYS ensure producer serializer matches consumer deserializer
   → Producer uses JSON → Consumer must use JSON
   → Producer uses Avro → Consumer must use Avro with same schema
   → Mismatch = crash or silent data corruption

2. ALWAYS serialize BOTH key and value (they are independent)
   → Key might be a simple string (StringSerializer)
   → Value might be complex JSON (JSONSerializer)
   → They do NOT have to use the same serializer

3. ALWAYS handle deserialization errors gracefully
   → A corrupted message should not crash your entire consumer
   → Catch deserialization exceptions
   → Route bad messages to a dead letter queue (Project 25)

4. ALWAYS include a version or schema identifier in your messages
   → Allows consumers to handle different versions
   → {"_version": 1, "name": "Alice"}
   → Or use Schema Registry for formal versioning (Project 16)

5. ALWAYS prefer UTF-8 encoding for string serialization
   → UTF-8 is the universal standard
   → Consistent across Python, Java, Go, Node.js
   → "Hello".encode('utf-8') → same bytes everywhere

6. ALWAYS consider message SIZE in serialization choice
   → JSON: human-readable but verbose (extra quotes, field names)
   → Binary formats: compact but not human-readable
   → At 1 million messages/day, size difference matters for disk and network

❌ WHAT SHOULD NOT DO

1. DO NOT use Python pickle for Kafka messages
   → Pickle is Python-specific — Java/Go consumers cannot read it
   → Pickle is a security risk (arbitrary code execution on deserialize)
   → Pickle output is not deterministic (same object → different bytes)

2. DO NOT assume all consumers use the same programming language
   → Your producer is Python today
   → Tomorrow a Java service might consume the same topic
   → Use language-agnostic formats: JSON, Avro, Protobuf

3. DO NOT send raw Python objects without explicit serialization
   → producer.produce(value=my_dict) → ERROR (expects bytes)
   → producer.produce(value=str(my_dict).encode()) → WORKS but ugly
   → producer.produce(value=json.dumps(my_dict).encode()) → CORRECT ✅

4. DO NOT mix serialization formats within the same topic
   → Message 1: JSON bytes
   → Message 2: Protobuf bytes
   → Consumer tries JSON deserialize on message 2 → CRASH
   → ONE topic = ONE serialization format (enforced by convention or Schema Registry)

5. DO NOT ignore key serialization
   → Keys are also bytes — they need serialization too
   → key="user-123" must become key=b"user-123"
   → Key serialization affects partition routing (hash of KEY BYTES — Project 4)
   → Different encoding of same string → different bytes → different partition!

6. DO NOT embed large binary blobs directly in Kafka messages
   → Kafka default max message size: 1MB
   → Images, videos, large files should go to object storage (S3)
   → Send only the REFERENCE (URL/path) via Kafka

🚫 WHEN NOT TO USE

→ DO NOT use JSON for ultra-high-throughput systems where every byte matters
  (JSON field names repeated in every message — enormous waste at scale)
  (Use Avro or Protobuf instead — field names stored in schema, not message)

→ DO NOT use custom binary formats for cross-team communication
  (Every team needs the encoding/decoding logic — maintenance nightmare)
  (Use standard formats that have libraries in every language)

→ DO NOT use string serialization for structured data
  ("name=Alice,age=30" is fragile — what if value contains a comma?)
  (Use JSON or binary format with proper escaping/structure)

→ DO NOT use Avro/Protobuf for simple prototyping
  (Overhead of schema definition and registry is unnecessary for demos)
  (Start with JSON, migrate to Avro when moving to production)

WHAT KAFKA STORES ON DISK — THE RAW BYTE TRUTH
═══════════════════════════════════════════════════════════════

When you produce a message, Kafka stores a RECORD:

  ┌─────────────────────────────────────────────────────────────┐
  │                    KAFKA RECORD FORMAT                       │
  ├─────────────┬──────────────────────────────────────────────┤
  │ Field       │ Description                                   │
  ├─────────────┼──────────────────────────────────────────────┤
  │ Length      │ Total record size in bytes (varint)           │
  │ Attributes  │ Compression, timestamp type (1 byte)         │
  │ Timestamp Δ │ Relative to batch timestamp (varint)         │
  │ Offset Δ    │ Relative to batch base offset (varint)       │
  │ Key Length  │ Length of key bytes (-1 if null) (varint)     │
  │ Key         │ RAW BYTES of key (your serialized key)       │
  │ Value Length│ Length of value bytes (-1 if null) (varint)   │
  │ Value       │ RAW BYTES of value (your serialized value)   │
  │ Headers     │ Array of (header_key, header_value) pairs     │
  └─────────────┴──────────────────────────────────────────────┘

  CRITICAL INSIGHT:
  → Key and Value are stored as OPAQUE BYTE ARRAYS
  → Kafka does not parse, validate, or interpret them
  → The broker treats them as blob storage
  → ALL meaning comes from the serializer/deserializer


═══════════════════════════════════════════════════════════════
SERIALIZATION: THE BYTE CONVERSION PROCESS
═══════════════════════════════════════════════════════════════

STRING SERIALIZATION:
─────────────────────
  "Hello" → UTF-8 encoding → [72, 101, 108, 108, 111]
  
  Python: "Hello".encode('utf-8') → b'Hello'
  5 characters → 5 bytes (ASCII subset)
  
  "你好" → UTF-8 encoding → [228, 189, 160, 229, 165, 189]
  2 characters → 6 bytes (CJK characters need 3 bytes each)
  
  REVERSE:
  b'Hello' → bytes.decode('utf-8') → "Hello"


JSON SERIALIZATION:
───────────────────
  {"name": "Alice", "age": 30}
      ↓ json.dumps()
  '{"name": "Alice", "age": 30}'     ← string (33 characters)
      ↓ .encode('utf-8')
  b'{"name": "Alice", "age": 30}'    ← bytes (33 bytes)

  STORED ON DISK: 33 bytes

  OVERHEAD ANALYSIS:
  → Actual data: "Alice" (5 bytes) + 30 (2 bytes) = 7 bytes of content
  → JSON overhead: field names, quotes, braces, colon = 26 bytes
  → Efficiency: 7/33 = 21% content, 79% overhead!

  REVERSE:
  b'{"name": "Alice", "age": 30}'
      ↓ .decode('utf-8')
  '{"name": "Alice", "age": 30}'
      ↓ json.loads()
  {"name": "Alice", "age": 30}       ← Python dict


CUSTOM BINARY SERIALIZATION:
─────────────────────────────
  {"name": "Alice", "age": 30}
      ↓ custom encode
  [1]                          ← version byte
  [5] [65, 108, 105, 99, 101] ← name length + name bytes ("Alice")
  [30]                         ← age as single byte
      
  STORED ON DISK: 8 bytes (vs 33 for JSON!)

  → 4x more compact than JSON
  → But requires custom encoder/decoder on both sides
  → No human readability
  → Fragile — any format change breaks everything


═══════════════════════════════════════════════════════════════
KEY SERIALIZATION AND ITS IMPACT ON PARTITIONING
═══════════════════════════════════════════════════════════════

From Project 4: partition = abs(murmur2(key_bytes)) % num_partitions

THE KEY BYTES ARE WHAT GETS HASHED — not the logical key value

  Example: Key is the string "user-123"

  UTF-8 encoding:
    "user-123".encode('utf-8') → b'user-123' → [117, 115, 101, 114, 45, 49, 50, 51]
    murmur2([117, 115, 101, 114, 45, 49, 50, 51]) → some_hash
    some_hash % 4 → Partition X

  UTF-16 encoding:
    "user-123".encode('utf-16') → different bytes!
    murmur2(different_bytes) → different_hash
    different_hash % 4 → Partition Y (DIFFERENT!)

  SAME LOGICAL KEY → DIFFERENT ENCODING → DIFFERENT PARTITION!

  THIS IS WHY: Always use consistent encoding (UTF-8) for keys
  THIS IS WHY: Key serialization must be identical across all producers

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "UnicodeDecodeError when consuming messages"
├── CAUSE: Producer used one encoding, consumer assumes another
├── EXAMPLE: Producer uses UTF-16, consumer tries UTF-8
├── SYMPTOM: UnicodeDecodeError: 'utf-8' codec can't decode byte 0xff
└── FIX: Ensure SAME encoding on both sides (always use UTF-8)

PITFALL 2: "json.loads() fails with JSONDecodeError"
├── CAUSE A: Producer sent plain string, consumer tries JSON parse
│   → Producer: "Hello".encode() → consumer: json.loads("Hello") → ERROR
│   └── FIX: Don't parse non-JSON as JSON
├── CAUSE B: Corrupted message or encoding issue
│   └── FIX: Wrap json.loads in try/except, route to dead letter queue
└── CAUSE C: Different producer sent different format to same topic
    └── FIX: Enforce one format per topic (convention or Schema Registry)

PITFALL 3: "Same key goes to different partition after changing serialization"
├── CAUSE: Different encoding produces different bytes → different hash
├── EXAMPLE: "user-123" in UTF-8 → 8 bytes vs UTF-16 → 18 bytes
│   → Different bytes → different murmur2 hash → different partition
├── IMPACT: Per-entity ordering BROKEN (Project 4)
└── FIX: NEVER change key serialization format for existing topics

PITFALL 4: "Message size exceeds max.message.bytes"
├── CAUSE: Serialized message too large (default limit: 1MB)
├── EXAMPLE: Embedding a large JSON array or binary blob
├── SYMPTOM: MessageSizeTooLargeException
└── FIX: Keep messages small — send references, not content
         OR increase max.message.bytes (not recommended for large data)

PITFALL 5: "Consumer gets garbled data after producer upgrade"
├── CAUSE: Producer changed message format without consumer update
├── EXAMPLE: Producer v1 sends {"name": "Alice"}
│            Producer v2 sends {"full_name": "Alice", "id": 1}
│            Consumer still expects {"name": ...} → KeyError!
├── THIS IS THE SCHEMA EVOLUTION PROBLEM
└── FIX: Version your messages, use backward-compatible changes
         Formal solution: Schema Registry (Project 16)

PITFALL 6: "Null key/value causes TypeError"
├── CAUSE: Calling .decode() on None
├── EXAMPLE: msg.key() returns None → None.decode('utf-8') → TypeError
└── FIX: Always check for None before deserializing
         key = msg.key().decode('utf-8') if msg.key() else None

  → Kafka supports compression: gzip, snappy, lz4, zstd
  → JSON compresses WELL (repetitive field names)

  → Avro + snappy = maximum efficiency

  → At 100K msgs/sec with 5μs per serialize (JSON):
    100,000 × 5μs = 0.5 seconds of CPU per second

# ═══════════════════════════════════════════════════════════
# file: serializers.py
# Reusable serializer/deserializer classes for all formats
# Import this module in all other experiments

import json
import struct
import time

# ─────────────────────────────────────────────────────────
# FORMAT 1: STRING SERIALIZER
# Simplest — just UTF-8 encoding/decoding
# ─────────────────────────────────────────────────────────

class StringSerializer:
    """Converts string ↔ bytes using UTF-8"""
    
    def serialize(self, data):
        """String → bytes"""
        if data is None:
            return None
        if isinstance(data, bytes):
            return data
        return str(data).encode('utf-8')
    
    def deserialize(self, data):
        """Bytes → string"""
        if data is None:
            return None
        return data.decode('utf-8')
    
    @property
    def name(self):
        return "StringSerializer"


# ─────────────────────────────────────────────────────────
# FORMAT 2: JSON SERIALIZER
# Converts Python dicts ↔ JSON bytes
# ─────────────────────────────────────────────────────────

class JSONSerializer:
    """Converts Python dict ↔ JSON UTF-8 bytes"""
    
    def __init__(self, indent=None, sort_keys=False):
        self.indent = indent
        self.sort_keys = sort_keys
    
    def serialize(self, data):
        """Dict → JSON bytes"""
        if data is None:
            return None
        json_str = json.dumps(
            data,
            indent=self.indent,
            sort_keys=self.sort_keys,
            default=str,           # handle non-serializable types
        )
        return json_str.encode('utf-8')
    
    def deserialize(self, data):
        """JSON bytes → dict"""
        if data is None:
            return None
        return json.loads(data.decode('utf-8'))
    
    @property
    def name(self):
        return "JSONSerializer"


# ─────────────────────────────────────────────────────────
# FORMAT 3: MESSAGEPACK SERIALIZER
# Binary JSON-like format — more compact than JSON
# ─────────────────────────────────────────────────────────

class MessagePackSerializer:
    """
    Converts Python dict ↔ MessagePack bytes
    Falls back to JSON if msgpack not installed
    """
    
    def __init__(self):
        try:
            import msgpack
            self._msgpack = msgpack
            self._available = True
        except ImportError:
            self._available = False
            print("⚠️  msgpack not installed — using JSON fallback")
            print("   Install with: pip install msgpack")
    
    def serialize(self, data):
        if data is None:
            return None
        if self._available:
            return self._msgpack.packb(data, use_bin_type=True)
        return json.dumps(data).encode('utf-8')
    
    def deserialize(self, data):
        if data is None:
            return None
        if self._available:
            return self._msgpack.unpackb(data, raw=False)
        return json.loads(data.decode('utf-8'))
    
    @property
    def name(self):
        return "MessagePackSerializer" if self._available else "JSONFallback"


# ─────────────────────────────────────────────────────────
# FORMAT 4: CUSTOM BINARY SERIALIZER
# Hand-crafted binary format — maximum efficiency
# Specifically designed for a "postal message" schema
# ─────────────────────────────────────────────────────────

class PostalMessageBinarySerializer:
    """
    Custom binary format for postal messages:
    
    Wire format:
    [version: 1 byte]
    [sender_len: 2 bytes unsigned short]
    [sender: variable bytes]
    [recipient_len: 2 bytes unsigned short]
    [recipient: variable bytes]
    [country_code: 2 bytes fixed]
    [priority: 1 byte (0=LOW, 1=MEDIUM, 2=HIGH)]
    [content_len: 4 bytes unsigned int]
    [content: variable bytes]
    [timestamp: 8 bytes double]
    """
    
    VERSION = 1
    PRIORITY_MAP = {'LOW': 0, 'MEDIUM': 1, 'HIGH': 2}
    PRIORITY_REVERSE = {0: 'LOW', 1: 'MEDIUM', 2: 'HIGH'}
    
    def serialize(self, data):
        """Dict → custom binary bytes"""
        if data is None:
            return None
        
        sender_bytes = data['sender'].encode('utf-8')
        recipient_bytes = data['recipient'].encode('utf-8')
        country_bytes = data['country_code'].encode('utf-8')[:2].ljust(2, b'\x00')
        priority_byte = self.PRIORITY_MAP.get(data.get('priority', 'LOW'), 0)
        content_bytes = data['content'].encode('utf-8')
        timestamp = data.get('timestamp', time.time())
        
        # Pack using struct
        header = struct.pack(
            '!B',                  # version (1 byte, unsigned)
            self.VERSION
        )
        sender_part = struct.pack(
            f'!H{len(sender_bytes)}s',
            len(sender_bytes),
            sender_bytes
        )
        recipient_part = struct.pack(
            f'!H{len(recipient_bytes)}s',
            len(recipient_bytes),
            recipient_bytes
        )
        country_part = struct.pack('!2s', country_bytes)
        priority_part = struct.pack('!B', priority_byte)
        content_part = struct.pack(
            f'!I{len(content_bytes)}s',
            len(content_bytes),
            content_bytes
        )
        timestamp_part = struct.pack('!d', timestamp)
        
        return (header + sender_part + recipient_part +
                country_part + priority_part + content_part + timestamp_part)
    
    def deserialize(self, data):
        """Custom binary bytes → dict"""
        if data is None:
            return None
        
        offset = 0
        
        # Version
        version = struct.unpack_from('!B', data, offset)[0]
        offset += 1
        
        if version != self.VERSION:
            raise ValueError(f"Unknown version: {version}")
        
        # Sender
        sender_len = struct.unpack_from('!H', data, offset)[0]
        offset += 2
        sender = struct.unpack_from(f'!{sender_len}s', data, offset)[0].decode('utf-8')
        offset += sender_len
        
        # Recipient
        recipient_len = struct.unpack_from('!H', data, offset)[0]
        offset += 2
        recipient = struct.unpack_from(f'!{recipient_len}s', data, offset)[0].decode('utf-8')
        offset += recipient_len
        
        # Country code
        country_code = struct.unpack_from('!2s', data, offset)[0].decode('utf-8').rstrip('\x00')
        offset += 2
        
        # Priority
        priority_byte = struct.unpack_from('!B', data, offset)[0]
        offset += 1
        priority = self.PRIORITY_REVERSE.get(priority_byte, 'LOW')
        
        # Content
        content_len = struct.unpack_from('!I', data, offset)[0]
        offset += 4
        content = struct.unpack_from(f'!{content_len}s', data, offset)[0].decode('utf-8')
        offset += content_len
        
        # Timestamp
        timestamp = struct.unpack_from('!d', data, offset)[0]
        
        return {
            'version': version,
            'sender': sender,
            'recipient': recipient,
            'country_code': country_code,
            'priority': priority,
            'content': content,
            'timestamp': timestamp,
        }
    
    @property
    def name(self):
        return "CustomBinarySerializer"

from confluent_kafka import Producer
import json
import time

config = {
    'bootstrap.servers': 'localhost:9092',
}

producer = Producer(config)

print("🔑 KEY ENCODING IMPACT ON PARTITION ROUTING")
print("   Same logical key, different encodings → different partitions!\n")
print("=" * 70)

# ─────────────────────────────────────────────────────────
# Test: Same key string, different byte encodings
# ─────────────────────────────────────────────────────────

test_key = "user-123"

encodings = {
    'UTF-8': test_key.encode('utf-8'),
    'UTF-16': test_key.encode('utf-16'),
    'UTF-32': test_key.encode('utf-32'),
    'ASCII': test_key.encode('ascii'),
    'Latin-1': test_key.encode('latin-1'),
}

results = {}

def make_callback(encoding_name):
    def callback(err, msg):
        if err:
            return
        results[encoding_name] = {
            'partition': msg.partition(),
            'offset': msg.offset(),
            'key_bytes': len(msg.key()),
        }
    return callback

for enc_name, key_bytes in encodings.items():
    value = json.dumps({
        'encoding': enc_name,
        'key_string': test_key,
        'key_bytes_hex': key_bytes.hex(),
        'key_byte_count': len(key_bytes),
    }).encode('utf-8')
    
    producer.produce(
        topic='postal-messages',
        key=key_bytes,
        value=value,
        callback=make_callback(enc_name),
    )
    producer.poll(0)

producer.flush()

# ─────────────────────────────────────────────────────────
# ANALYSIS
# ─────────────────────────────────────────────────────────

print(f"\n   Logical key: \"{test_key}\"")
print(f"   Topic: postal-messages (2 partitions)\n")

print(f"   {'Encoding':<12} {'Key Bytes':>10} {'Hex (first 20)':>25} {'Partition':>10}")
print(f"   {'─' * 62}")

partitions_seen = set()

for enc_name, key_bytes in encodings.items():
    r = results.get(enc_name, {})
    partition = r.get('partition', '?')
    num_bytes = len(key_bytes)
    hex_preview = key_bytes[:10].hex()
    
    partitions_seen.add(partition)
    
    print(
        f"   {enc_name:<12} {num_bytes:>6} bytes  "
        f"{hex_preview:>25} "
        f"→ P{partition}"
    )

print(f"\n   Unique partitions hit: {len(partitions_seen)}")

if len(partitions_seen) > 1:
    print(f"""
   ⚠️  SAME logical key "{test_key}" routed to DIFFERENT partitions!
   
   ROOT CAUSE:
   → UTF-8  encodes "{test_key}" as {len(encodings['UTF-8']):>3} bytes
   → UTF-16 encodes "{test_key}" as {len(encodings['UTF-16']):>3} bytes (includes BOM)
   → UTF-32 encodes "{test_key}" as {len(encodings['UTF-32']):>3} bytes
   
   → Different bytes → different murmur2 hash → different partition!
   
   RULE: ALL producers for the same topic MUST use the SAME key encoding
   → Convention: ALWAYS use UTF-8 for string keys
   → This is non-negotiable for key-based ordering (Project 4)
""")
else:
    print(f"\n   ✅ All encodings happened to hit same partition (coincidence with 2 partitions)")

# PROOF: Consistent encoding → consistent partition
# ─────────────────────────────────────────────────────────

print(f"   CONSISTENCY PROOF: UTF-8 encoding, multiple keys, 3 rounds each\n")

consistent_results = {}

def track_callback(key_str):
    def callback(err, msg):
        if err:
            return
        if key_str not in consistent_results:
            consistent_results[key_str] = []
        consistent_results[key_str].append(msg.partition())
    return callback

test_keys = ["user-001", "user-002", "user-003", "user-004", "user-005"]

for round_num in range(3):
    for key_str in test_keys:
        key_bytes = key_str.encode('utf-8')     # ALWAYS UTF-8
        value = json.dumps({'round': round_num, 'key': key_str}).encode('utf-8')
        
        producer.produce(
            topic='postal-messages',
            key=key_bytes,
            value=value,
            callback=track_callback(key_str),
        )
        producer.poll(0)

producer.flush()

all_consistent = True
for key_str in test_keys:
    partitions = consistent_results.get(key_str, [])
    is_consistent = len(set(partitions)) == 1
    if not is_consistent:
        all_consistent = False
    status = "✅ CONSISTENT" if is_consistent else "❌ INCONSISTENT"
    print(f"   \"{key_str}\" → partitions across 3 rounds: {partitions} {status}")

if all_consistent:
    print(f"\n   ✅ PROOF: Same key + same encoding → same partition EVERY TIME")
else:
    print(f"\n   ❌ Inconsistency detected — investigate encoding!")
