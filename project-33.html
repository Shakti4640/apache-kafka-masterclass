📋 PHASE 4: HARD — ADVANCED PATTERNS

🔴 HYPOTHETICAL PROBLEM

You run a large retail chain with 500 stores across the country.

EXISTING KAFKA STREAMS PIPELINE (built in Projects 19–22, 32):
  → "inventory-events" topic: Stock movements (received, sold, returned, damaged)
  → Kafka Streams aggregates these into a KTable: product_id → current_stock
  → "order-enrichments" pipeline: Joins orders with customer + product data
  → "sales-aggregations" pipeline: Windowed sales totals per store per hour

THE PIPELINE WORKS PERFECTLY FOR PROCESSING.

THE NEW BUSINESS REQUIREMENTS:
───────────────────────────────
1. POINT QUERY:
   → Store manager asks: "How many units of PROD-0042 do we have RIGHT NOW?"
   → Must return answer in < 50ms
   → Cannot wait for batch processing or database sync

2. RANGE QUERY:
   → Regional manager asks: "Show me stock for ALL products in Electronics category"
   → Must scan a subset of the state store
   → Return sorted results

3. WINDOW QUERY:
   → Dashboard asks: "What were sales totals for Store-15 in the last 6 hours?"
   → Must query windowed state store
   → Return per-hour breakdown

4. MULTI-INSTANCE QUERY:
   → Kafka Streams runs on 6 instances (for 6 partitions)
   → PROD-0042 state lives on Instance 3 (because hash(PROD-0042) % 6 = 3)
   → Query hits Instance 1 → Instance 1 does NOT have PROD-0042's state
   → Must discover which instance has the data and route query there

5. FULL CATALOG QUERY:
   → API client asks: "Give me ALL products with stock below 10"
   → Data is SPREAD across 6 instances
   → Must aggregate results from ALL instances

THE CRISIS WITHOUT INTERACTIVE QUERIES:
───────────────────────────────────────
→ Option A: Sync state to external database (PostgreSQL, Redis)
  → Extra infrastructure, sync lag, consistency issues
  → State changes in Kafka → delayed write to DB → stale reads
  → 500ms–5s staleness for every query

→ Option B: Consume output topic and build custom cache
  → Duplicate state management
  → Must handle rebalancing, compaction, recovery
  → Reinventing what Kafka Streams already maintains

→ Option C: Query Kafka Streams state stores DIRECTLY ← THIS
  → Zero additional infrastructure
  → State is ALWAYS up-to-date (same state used for processing)
  → Latency: microseconds for local, milliseconds for remote
  → Already built, already maintained, already fault-tolerant

  ────────────────────────────────

  3. metadata.queryMetadataForKey("inventory", "PROD-0042")

  7. Instance B returns result to Instance A
  8. Instance A returns result to client

✅ WHAT SHOULD DO

1. NAME ALL STATE STORES EXPLICITLY
   → builder.table("topic", Materialized.as("my-store-name"))
   → Auto-generated names change if topology changes → store lost
   → Named stores survive topology modifications
   → Query by name: streams.store("my-store-name", ...)

2. CONFIGURE application.server FOR DISCOVERY
   → Each instance must advertise its host:port
   → Properties: application.server=host:port
   → This is how other instances discover where to route queries
   → MUST be unique per instance

3. IMPLEMENT BOTH LOCAL AND REMOTE QUERY PATHS
   → Check if key is local → query local store
   → If not local → discover remote instance → forward query
   → Handle remote instance being down (retry, failover)

4. USE ReadOnlyKeyValueStore (NOT KeyValueStore)
   → Interactive queries get READ-ONLY access
   → Cannot modify state through query interface
   → Prevents query side effects from corrupting processing state
   → Type safety enforced at compile time (Java)

5. HANDLE REBALANCING GRACEFULLY
   → During rebalance (Project 10), state stores may be unavailable
   → Queries during rebalance → InvalidStateStoreException
   → Return HTTP 503 (Service Unavailable) with Retry-After header
   → Client retries after rebalance completes

6. SET APPROPRIATE TIMEOUTS FOR REMOTE QUERIES
   → Local query: < 1ms (RocksDB lookup)
   → Remote query: 5-50ms (network + local query)
   → Set HTTP client timeout to 100ms
   → If remote instance slow → timeout → return error

7. CACHE METADATA FOR PERFORMANCE
   → Don't call metadataForKey() for every single query
   → Partition-to-instance mapping changes only during rebalance
   → Cache mapping, invalidate on rebalance notification
   → Reduces metadata lookup overhead by 99%

8. ADD QUERY METRICS
   → Track: queries/sec, latency p50/p95/p99
   → Track: local vs remote query ratio
   → Track: cache hit rate for metadata
   → Track: errors (store unavailable, remote timeout)

❌ WHAT SHOULD NOT DO

1. DO NOT WRITE TO STATE STORES VIA INTERACTIVE QUERIES
   → State stores are managed by Kafka Streams processing
   → Writing directly corrupts state ↔ changelog consistency
   → Changelog topic won't reflect your write → state lost on recovery
   → ALWAYS write via Kafka topics → let processing update state

2. DO NOT EXPOSE MUTABLE STATE STORE REFERENCES
   → Always cast to ReadOnlyKeyValueStore
   → Never expose put(), delete(), flush() to query layer
   → Even in Python: treat query dicts as immutable

3. DO NOT QUERY DURING REBALANCING WITHOUT ERROR HANDLING
   → Rebalance = state stores closing on one instance, opening on another
   → Query hits instance that just lost the partition → exception
   → MUST catch InvalidStateStoreException → return 503
   → Client MUST implement retry with backoff

4. DO NOT FORGET application.server CONFIGURATION
   → Without it, metadata discovery returns empty host/port
   → Remote routing impossible → all remote queries fail silently
   → Every instance MUST set unique application.server

5. DO NOT ASSUME ALL DATA IS LOCAL
   → With N partitions and M instances, each instance has N/M partitions
   → Probability of key being local = 1/M
   → For 6 instances: 83% of queries require remote forwarding
   → Design for remote-first, local as optimization

6. DO NOT BUILD SCATTER-GATHER FOR EVERY QUERY
   → "Get all products with stock < 10" requires ALL instances
   → Scatter-gather is expensive: M network calls + merge
   → For frequent full-scan queries → materialize to output topic
   → Use scatter-gather only for ad-hoc or rare queries

7. DO NOT IGNORE STATE STORE RESTORATION TIME
   → On startup or rebalance, state stores rebuild from changelog
   → 10 million records × 1KB = 10GB to restore
   → Queries return stale/empty results until restoration completes
   → Implement readiness check: only serve queries after restoration
   → Use standby replicas (num.standby.replicas) to speed up failover

🚫 WHEN NOT TO USE

1. WHEN YOU NEED COMPLEX SQL QUERIES
   → Interactive queries support: get, range, all, window fetch
   → No: JOIN, GROUP BY, HAVING, subqueries
   → For complex queries → use ksqlDB or sink to PostgreSQL

2. WHEN QUERY VOLUME EXCEEDS PROCESSING CAPACITY
   → Interactive queries share resources with stream processing
   → Heavy query load → starves processing → consumer lag
   → If query QPS > processing QPS → use external serving store

3. WHEN DATA MUST BE QUERIED ACROSS MULTIPLE STATE STORES
   → "Join inventory store with customer store" → not supported
   → Interactive queries work on ONE store at a time
   → For cross-store queries → build materialized view in processing

4. WHEN YOU NEED STRONG CONSISTENCY GUARANTEES
   → Interactive queries are EVENTUALLY CONSISTENT
   → Record processed at T=0, query at T=1ms might not see it
   → For strong consistency → use transactional database

5. WHEN STATE STORE IS VERY LARGE (>50GB PER INSTANCE)
   → RocksDB handles large state but query latency increases
   → Point queries still fast but range/all scans slow
   → Consider: partitioning strategy, state store optimization
   → Or: external store optimized for queries (Elasticsearch)

          STATE STORE AVAILABILITY DURING REBALANCE
═══════════════════════════════════════════════════════════════

  TIMELINE OF A REBALANCE:
  
  T=0:  Instance C dies
  T=1:  Consumer group coordinator detects failure
  T=2:  Rebalance triggered → all instances stop processing
  T=3:  Partitions 4,5 (from Instance C) reassigned to A and B
  T=4:  Instance A starts restoring partition 4 state store
        → Reads changelog topic from beginning for partition 4
        → State store for partition 4 is UNAVAILABLE
  T=5:  Queries for keys on partition 4 → ERROR
  T=6:  Restoration complete → state store available
  T=7:  Queries work again

  WITH STANDBY REPLICAS (num.standby.replicas=1):
  T=0:  Instance C dies
  T=1:  Instance A already has STANDBY copy of partition 4 state
  T=2:  Rebalance reassigns partition 4 to Instance A
  T=3:  Instance A promotes standby to active → INSTANT availability
  T=4:  Queries work immediately (no restoration delay)

  THIS IS WHY STANDBY REPLICAS MATTER FOR INTERACTIVE QUERIES

🕳️ COMMON PITFALLS AND FAILURE MODES

PITFALL 1: QUERYING BEFORE STATE STORE IS READY
  → Application starts → REST API available immediately
  → State store still restoring from changelog topic
  → Query returns null/empty → client thinks no data exists
  → FIX: Implement readiness endpoint
  → Only return 200 when streams.state() == RUNNING
  → Return 503 during REBALANCING or CREATED states

PITFALL 2: NOT SETTING application.server
  → Metadata discovery returns empty host/port
  → Cannot determine which instance holds a key
  → All remote queries fail with "unknown host"
  → FIX: Set unique application.server for every instance
  → Format: "hostname:port" (must be reachable by other instances)

PITFALL 3: QUERY OVERLOAD STARVING PROCESSING
  → 10,000 queries/sec hitting interactive query endpoint
  → RocksDB reads compete with processing writes
  → Processing falls behind → consumer lag grows
  → FIX: Rate limit query endpoint
  → FIX: Use standby replicas for read-only queries
  → FIX: Separate query instances from processing instances

PITFALL 4: STALE METADATA CACHE
  → Cached partition→instance mapping
  → Rebalance moves partitions → cache stale
  → Queries routed to wrong instance → key not found
  → FIX: Invalidate cache on rebalance listener callback
  → FIX: Handle "not found" by refreshing metadata and retrying

PITFALL 5: RANGE QUERY ON HUGE STATE STORE
  → store.range("A", "Z") on 100 million records
  → Scans entire store → blocks for seconds
  → API timeout → client retries → amplifies load
  → FIX: Paginate range queries (limit + offset)
  → FIX: Use prefix scans instead of full ranges
  → FIX: Set query timeout

PITFALL 6: WINDOW STORE TIME ZONE CONFUSION
  → Window store uses event-time timestamps (UTC)
  → Query specifies local time → wrong window queried
  → Results look empty even though data exists
  → FIX: Always query in UTC
  → FIX: Convert local time → UTC before window fetch

PITFALL 7: SERIALIZATION MISMATCH IN METADATA QUERY
  → metadataForKey uses serializer to compute partition
  → If query uses String serializer but store uses Avro serializer
  → Different bytes → different hash → wrong partition → wrong instance
  → FIX: Use SAME serializer for metadata queries as for processing

📈 SCALABILITY CONSIDERATIONS

DIMENSION 1: QUERY THROUGHPUT
  ┌─────────────────────────────────┬──────────────────────┐
  │ Query Type                      │ Approx Throughput    │
  ├─────────────────────────────────┼──────────────────────┤
  │ Point query (local, cache hit)  │ 500K–1M queries/sec  │
  │ Point query (local, disk)       │ 50K–100K queries/sec │
  │ Point query (remote, HTTP)      │ 5K–20K queries/sec   │
  │ Range query (1000 results)      │ 1K–5K queries/sec    │
  │ All query (full scan)           │ 1–10 queries/sec     │
  │ Scatter-gather (6 instances)    │ 500–2K queries/sec   │
  └─────────────────────────────────┴──────────────────────┘

DIMENSION 2: STATE STORE SIZE vs QUERY LATENCY
  → < 1GB: All data fits in RocksDB block cache → sub-ms queries
  → 1–10GB: Mix of cache hits and disk reads → 1-5ms average
  → 10–50GB: Mostly disk reads → 5-20ms average
  → > 50GB: Consider external optimized store (Elasticsearch, Redis)

DIMENSION 3: NUMBER OF INSTANCES
  → More instances → more partitions per instance (smaller stores)
  → More instances → higher probability of remote queries
  → Sweet spot: instances where store size fits in memory
  → With 6 instances, 60GB total state → 10GB per instance

DIMENSION 4: STANDBY REPLICAS
  → num.standby.replicas=1 → doubles storage
  → But: failover is instant (no restoration delay)
  → Can route read queries to standby replicas (active-standby)
  → Doubles read throughput at cost of storage

DIMENSION 5: CONCURRENT QUERY HANDLING
  → RocksDB supports concurrent reads safely
  → But: too many concurrent reads → disk I/O contention
  → Thread pool for query handling: size = 2 × CPU cores
  → Connection pool for remote queries: size = instances × 2

"""
PROJECT 33: Queryable State Store Engine
==========================================
Builds and maintains state stores from Kafka topics,
then exposes them for interactive queries.

This is the Python equivalent of Kafka Streams' state stores
with ReadOnlyKeyValueStore and ReadOnlyWindowStore interfaces.

Builds on: Project 21 (RocksDB state stores)
           Project 32 (state stores for joins)
"""

import json
import time
import threading
import hashlib
from collections import defaultdict, OrderedDict
from confluent_kafka import Consumer, TopicPartition


# ═══════════════════════════════════════════════════════════════
#  READ-ONLY KEY-VALUE STORE (Equivalent of Kafka Streams)
# ═══════════════════════════════════════════════════════════════

class ReadOnlyKeyValueStore:
    """
    Python equivalent of org.apache.kafka.streams.state.ReadOnlyKeyValueStore

    In Kafka Streams (Java):
      ReadOnlyKeyValueStore<String, Inventory> store = 
          streams.store(
              StoreQueryParameters.fromNameAndType(
                  "inventory-store",
                  QueryableStoreTypes.keyValueStore()
              )
          );
      Inventory item = store.get("PROD-0042");

    This class provides the SAME interface:
      store.get(key)           → single lookup
      store.range(from, to)    → range scan
      store.all()              → full iteration
      store.approximate_num_entries() → count estimate
    """

    def __init__(self, store_name: str):
        self.store_name = store_name
        self._data = OrderedDict()  # Sorted for range queries
        self._lock = threading.RLock()
        self._update_count = 0
        self._last_update_time = 0

    # ── Internal: Used by processing engine (NOT by queries) ──

    def _put(self, key: str, value: dict):
        """Internal: Update store (called by processing, NOT by queries)"""
        with self._lock:
            self._data[key] = value
            self._update_count += 1
            self._last_update_time = time.time()

    def _delete(self, key: str):
        """Internal: Delete from store (tombstone record)"""
        with self._lock:
            self._data.pop(key, None)
            self._update_count += 1

    # ── Public: Read-Only Query Interface ──

    def get(self, key: str) -> dict:
        """
        Point query: retrieve value for a specific key.
        Returns None if key not found.
        Equivalent to Java: store.get(key)
        """
        with self._lock:
            return self._data.get(key)

    def range(self, from_key: str, to_key: str) -> list:
        """
        Range query: retrieve all entries where from_key <= key <= to_key.
        Returns list of (key, value) tuples, sorted by key.
        Equivalent to Java: store.range(from, to)
        """
        with self._lock:
            results = []
            for key, value in self._data.items():
                if from_key <= key <= to_key:
                    results.append((key, value))
            return results

    def prefix_scan(self, prefix: str) -> list:
        """
        Prefix scan: retrieve all entries where key starts with prefix.
        Returns list of (key, value) tuples.
        Equivalent to Java: store.prefixScan(prefix, serializer)
        """
        with self._lock:
            return [
                (key, value) for key, value in self._data.items()
                if key.startswith(prefix)
            ]

    def all(self) -> list:
        """
        Full scan: iterate over all entries.
        Returns list of (key, value) tuples.
        WARNING: Expensive for large stores — use sparingly.
        Equivalent to Java: store.all()
        """
        with self._lock:
            return list(self._data.items())

    def approximate_num_entries(self) -> int:
        """
        Approximate count of entries in the store.
        Equivalent to Java: store.approximateNumEntries()
        """
        with self._lock:
            return len(self._data)

    def filter(self, predicate) -> list:
        """
        Custom filter query: retrieve entries matching predicate.
        predicate: function(key, value) → bool
        NOT part of standard Kafka Streams API — added for convenience.
        """
        with self._lock:
            return [
                (key, value) for key, value in self._data.items()
                if predicate(key, value)
            ]

    def get_metadata(self) -> dict:
        """Store metadata for debugging and monitoring"""
        with self._lock:
            return {
                "store_name": self.store_name,
                "num_entries": len(self._data),
                "update_count": self._update_count,
                "last_update_time": self._last_update_time,
                "last_update_age_sec": round(
                    time.time() - self._last_update_time, 2
                ) if self._last_update_time > 0 else None,
            }

class ReadOnlyWindowStore:
    """
    Python equivalent of org.apache.kafka.streams.state.ReadOnlyWindowStore

    Stores windowed aggregations accessible by key + time range.

    In Kafka Streams (Java):
      ReadOnlyWindowStore<String, SalesTotal> store = 
          streams.store(
              StoreQueryParameters.fromNameAndType(
                  "sales-window-store",
                  QueryableStoreTypes.windowStore()
              )
          );
      WindowStoreIterator<SalesTotal> results = 
          store.fetch("STORE-001", startTime, endTime);
    """

    def __init__(self, store_name: str, window_size_ms: int = 3600000):
        self.store_name = store_name
        self.window_size_ms = window_size_ms
        # Structure: {key: {window_start_ms: aggregated_value}}
        self._data = defaultdict(dict)
        self._lock = threading.RLock()
        self._update_count = 0

    def _compute_window_start(self, timestamp_ms: int) -> int:
        """Compute window start time for a given timestamp"""
        return (timestamp_ms // self.window_size_ms) * self.window_size_ms

    def _put(self, key: str, timestamp_ms: int, value: dict):
        """Internal: Update windowed store"""
        window_start = self._compute_window_start(timestamp_ms)
        with self._lock:
            self._data[key][window_start] = value
            self._update_count += 1

    def _update_aggregate(self, key: str, timestamp_ms: int,
                          update_fn, initial_value: dict):
        """Internal: Update aggregation for a window"""
        window_start = self._compute_window_start(timestamp_ms)
        with self._lock:
            current = self._data[key].get(window_start, initial_value)
            updated = update_fn(current)
            self._data[key][window_start] = updated
            self._update_count += 1

    def fetch(self, key: str, time_from_ms: int, time_to_ms: int) -> list:
        """
        Window query: retrieve all windows for a key within time range.
        Returns list of (window_start, value) tuples.
        Equivalent to Java: store.fetch(key, Instant.from, Instant.to)
        """
        with self._lock:
            windows = self._data.get(key, {})
            results = []
            for window_start, value in sorted(windows.items()):
                window_end = window_start + self.window_size_ms
                if window_start >= time_from_ms and window_start <= time_to_ms:
                    results.append((window_start, window_end, value))
            return results

    def fetch_all(self, time_from_ms: int, time_to_ms: int) -> list:
        """
        Fetch all windows for all keys within time range.
        Returns list of (key, window_start, value) tuples.
        """
        with self._lock:
            results = []
            for key, windows in self._data.items():
                for window_start, value in sorted(windows.items()):
                    if time_from_ms <= window_start <= time_to_ms:
                        results.append((key, window_start,
                                        window_start + self.window_size_ms, value))
            return results

    def all_keys(self) -> list:
        """Get all keys in the window store"""
        with self._lock:
            return list(self._data.keys())

    def get_metadata(self) -> dict:
        with self._lock:
            total_windows = sum(len(w) for w in self._data.values())
            return {
                "store_name": self.store_name,
                "window_size_ms": self.window_size_ms,
                "num_keys": len(self._data),
                "total_windows": total_windows,
                "update_count": self._update_count,
            }

class StateStoreManager:
    """
    Manages multiple state stores, populates them from Kafka topics,
    and provides query access.

    Equivalent to the internal state store management in KafkaStreams.

    Lifecycle:
    1. Register stores
    2. Start background consumers to populate stores
    3. Accept queries against populated stores
    4. Handle rebalancing (store unavailability)

                 instance_id: str = "instance-1",
                 instance_host: str = "localhost",
                 instance_port: int = 7001):

        self.instance_id = instance_id
        self.instance_host = instance_host
        self.instance_port = instance_port

        # ── Registered stores ──
        self.kv_stores: dict[str, ReadOnlyKeyValueStore] = {}
        self.window_stores: dict[str, ReadOnlyWindowStore] = {}

        # ── State ──
        self.state = "CREATED"  # CREATED → RUNNING → REBALANCING → RUNNING
        self.running = False
        self.consumer_threads = []

        # In Kafka Streams: populated by application.server config

            "host": instance_host,
            "port": instance_port,
            "stores": [],
            "partitions": [],
        }

        # In Kafka Streams: managed by StreamsMetadata
        self.partition_metadata = {}  # {partition_num: {host, port}}

    # ═══════════════════════════════════════════════════════════
    #  STORE REGISTRATION
    # ═══════════════════════════════════════════════════════════

    def register_kv_store(self, store_name: str,

        Register a key-value store backed by a Kafka topic.

        Equivalent to:

            "source_topic": source_topic,
            "key_extractor": key_extractor,
            "value_processor": value_processor,

                              initial_value_factory=None):
        """
        Register a window store backed by a Kafka topic.

        Equivalent to:
          builder.stream("sales-events")
                 .groupByKey()
                 .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofHours(1)))
                 .aggregate(initializer, aggregator,
                            Materialized.as("sales-window-store"))

        store = ReadOnlyWindowStore(store_name, window_size_ms)
        self.window_stores[store_name] = {
            "store": store,
            "source_topic": source_topic,
            "aggregator": aggregator,

        }
        self.instance_metadata["stores"].append(store_name)
        print(f"  📋 Registered Window store: '{store_name}' ← topic '{source_topic}' "

    #  STORE POPULATION (Background consumers)

    def start(self):

        print(f"  ✅ State store manager RUNNING")

    def _populate_kv_store(self, store_name: str, config: dict):
        """Background consumer that populates a KV store"""
        store = config["store"]
        topic = config["source_topic"]
        key_extractor = config.get("key_extractor")
        value_processor = config.get("value_processor")

        consumer = Consumer({
            "bootstrap.servers": self.bootstrap,
            "group.id": f"state-store-{store_name}-{self.instance_id}",
            "auto.offset.reset": "earliest",
            "enable.auto.commit": True,
        })
        consumer.subscribe([topic])

        print(f"    [KV:{store_name}] Restoring from topic '{topic}'...")

        while self.running:
            msg = consumer.poll(timeout=1.0)
            if msg is None:
                continue
            if msg.error():
                continue

            try:
                value = json.loads(msg.value().decode("utf-8"))

                # Extract key
                if key_extractor:
                    key = key_extractor(value)
                elif msg.key():
                    key = msg.key().decode("utf-8")
                else:
                    continue

                # Process value
                if value_processor:
                    processed = value_processor(key, store.get(key), value)
                    if processed is not None:
                        store._put(key, processed)
                else:
                    store._put(key, value)

            except (json.JSONDecodeError, UnicodeDecodeError, KeyError):
                continue

        consumer.close()

    def _populate_window_store(self, store_name: str, config: dict):
        """Background consumer that populates a window store"""
        store = config["store"]
        topic = config["source_topic"]
        aggregator = config.get("aggregator")
        initial_factory = config.get("initial_value_factory", lambda: {})

        consumer = Consumer({
            "bootstrap.servers": self.bootstrap,
            "group.id": f"window-store-{store_name}-{self.instance_id}",
            "auto.offset.reset": "earliest",
            "enable.auto.commit": True,
        })
        consumer.subscribe([topic])

        print(f"    [Window:{store_name}] Restoring from topic '{topic}'...")

        while self.running:
            msg = consumer.poll(timeout=1.0)
            if msg is None:
                continue
            if msg.error():
                continue

            try:
                value = json.loads(msg.value().decode("utf-8"))
                key = msg.key().decode("utf-8") if msg.key() else "unknown"
                timestamp = value.get("timestamp", int(time.time() * 1000))

                if aggregator:
                    store._update_aggregate(
                        key=key,
                        timestamp_ms=timestamp,
                        update_fn=lambda current: aggregator(current, value),
                        initial_value=initial_factory(),
                    )
                else:
                    store._put(key, timestamp, value)

            except (json.JSONDecodeError, UnicodeDecodeError, KeyError):
                continue

        consumer.close()

    def _print_store_summary(self):

        return {
            "key": key,

"""
PROJECT 33: Standby Replica Demonstration
============================================
Shows how standby replicas enable fast failover
for interactive queries during rebalancing.

In Kafka Streams:
  props.put(StreamsConfig.NUM_STANDBY_REPLICAS_CONFIG, 1);

This creates a WARM copy of state stores on another instance.
When the active instance dies:
  - Without standby: must rebuild from changelog (minutes)
  - With standby: promote warm copy immediately (seconds)
"""

import time
import threading
import json
from state_store_engine import ReadOnlyKeyValueStore


class StandbyReplicaSimulator:
    """
    Simulates the standby replica mechanism in Kafka Streams.
    
    Architecture:
    ┌──────────────────┐     ┌──────────────────┐
    │  Instance A       │     │  Instance B       │
    │  (ACTIVE)         │     │  (STANDBY)        │
    │                   │     │                   │
    │  Partition 0,1    │     │  Partition 2,3    │
    │  (processing)     │     │  (processing)     │
    │                   │     │                   │
    │  Standby: P2,3    │     │  Standby: P0,1    │
    │  (warm copy)      │     │  (warm copy)      │
    └──────────────────┘     └──────────────────┘
    
    If Instance A dies:
    1. Instance B already has warm copy of P0,P1 state
    2. Promote standby → active (near-instant)
    3. Interactive queries for P0,P1 keys → served by Instance B
    4. No restoration delay from changelog topic
    """

    def __init__(self):
        # Active stores (processing + serving queries)
        self.active_store_a = ReadOnlyKeyValueStore("inventory-active-a")
        self.active_store_b = ReadOnlyKeyValueStore("inventory-active-b")

        # Standby stores (warm copies, read-only)
        self.standby_store_a = ReadOnlyKeyValueStore("inventory-standby-a")
        self.standby_store_b = ReadOnlyKeyValueStore("inventory-standby-b")

    def simulate_failover(self):
        """Demonstrate failover with and without standby replicas"""
        print(f"\n{'═' * 70}")
        print("  STANDBY REPLICA FAILOVER SIMULATION")
        print(f"{'═' * 70}\n")

        # ── Step 1: Populate active stores ──
        print("  STEP 1: Populating active state stores")
        print("  " + "─" * 50)

        num_records = 10000
        for i in range(1, num_records + 1):
            product_id = f"PROD-{i:04d}"
            value = {
                "product_id": product_id,
                "product_name": f"Product {i}",
                "current_stock": 100 + (i % 50),
            }

            # Instance A holds even partitions, B holds odd
            if hash(product_id) % 2 == 0:
                self.active_store_a._put(product_id, value)
                # Standby copy on Instance B
                self.standby_store_b._put(product_id, value)
            else:
                self.active_store_b._put(product_id, value)
                # Standby copy on Instance A
                self.standby_store_a._put(product_id, value)

        print(f"    Instance A active: {self.active_store_a.approximate_num_entries()} entries")
        print(f"    Instance B active: {self.active_store_b.approximate_num_entries()} entries")
        print(f"    Instance A standby: {self.standby_store_a.approximate_num_entries()} entries")
        print(f"    Instance B standby: {self.standby_store_b.approximate_num_entries()} entries")

        # ── Step 2: Simulate Instance A failure ──
        print(f"\n  STEP 2: Instance A FAILS! 💥")
        print("  " + "─" * 50)

        # ── Scenario A: WITHOUT standby replicas ──
        print(f"\n  SCENARIO A: Without Standby Replicas")
        print("  " + "─" * 40)

        restore_start = time.time()

        # Must rebuild from changelog topic (simulate delay)
        restored_store = ReadOnlyKeyValueStore("inventory-restored")
        records_to_restore = self.active_store_a.approximate_num_entries()

        print(f"    Restoring {records_to_restore} records from changelog...")
        print(f"    (Simulating changelog replay...)")

        # Simulate restoration time (0.1ms per record in real Kafka Streams)
        simulated_restore_time = records_to_restore * 0.0001  # seconds
        time.sleep(min(simulated_restore_time, 2))  # Cap at 2s for demo

        # Actually copy the data
        for key, value in self.active_store_a.all():
            restored_store._put(key, value)

        restore_elapsed = time.time() - restore_start
        print(f"    ⏱️  Restoration time: {restore_elapsed:.2f}s")
        print(f"    ❌ Queries UNAVAILABLE during restoration")
        print(f"    ❌ All queries for Instance A keys return 503")

        # ── Scenario B: WITH standby replicas ──
        print(f"\n  SCENARIO B: With Standby Replicas (num.standby.replicas=1)")
        print("  " + "─" * 40)

        promote_start = time.time()

        # Standby already has the data — just promote
        promoted_store = self.standby_store_b  # Already populated!

        promote_elapsed = time.time() - promote_start
        print(f"    ⏱️  Promotion time: {promote_elapsed * 1000:.2f}ms")
        print(f"    ✅ Queries available IMMEDIATELY")
        print(f"    ✅ Standby promoted to active — zero data loss")

        # ── Step 3: Verify query availability ──
        print(f"\n  STEP 3: Query Availability Comparison")
        print("  " + "─" * 50)

        test_key = "PROD-0002"  # Should be on Instance A

        # Without standby: query fails during restoration
        print(f"\n    Query for '{test_key}':")
        print(f"    Without standby: ❌ 503 Service Unavailable "
              f"(restoring for {restore_elapsed:.2f}s)")

        # With standby: query succeeds immediately
        result = promoted_store.get(test_key)
        if result:
            print(f"    With standby:    ✅ 200 OK — "
                  f"Stock: {result.get('current_stock')} "
                  f"(served in {promote_elapsed * 1000:.2f}ms)")
        else:
            print(f"    With standby:    Query returned None (key not in standby)")

        # ── Step 4: Impact summary ──
        print(f"\n  {'─' * 50}")
        print(f"  FAILOVER IMPACT SUMMARY")
        print(f"  {'─' * 50}")
        print(f"""
    ┌────────────────────────┬──────────────┬──────────────┐

    ├────────────────────────┼──────────────┼──────────────┤
    │ Query Downtime         │ {restore_elapsed:.2f}s        │ {promote_elapsed*1000:.2f}ms       │
    │ Data Loss Risk         │ None*        │ None         │
    │ Extra Storage          │ 0%           │ +100%        │
    │ Extra Network          │ 0%           │ +100%        │
    │ Recovery Speed         │ Minutes      │ Milliseconds │
    │ Query SLA Impact       │ HIGH         │ NONE         │
    └────────────────────────┴──────────────┴──────────────┘

    * Data recovered from changelog topic, but queries unavailable during recovery

    → For interactive queries: ALWAYS use num.standby.replicas >= 1
