PROJECT 25: THE POISON PILL HANDLER — DEAD LETTER QUEUES

🏗️ HYPOTHETICAL PROBLEM

SCENARIO: Multi-Source E-Commerce Event Pipeline

→ A large e-commerce platform receives events from 12 different sources:
  - Mobile app (iOS, Android)
  - Web frontend (React)
  - POS terminals (in-store)
  - Partner API integrations
  - IoT shelf sensors
  - Inventory management system

→ All events flow into Kafka topics via:
  - Kafka Connect JDBC Source (Project 23) with SMTs (Project 24)
  - Direct producers from microservices (Project 12)
  - Third-party webhook ingestion service

→ THE DISASTER (Monday 3:47 AM):
  
  INCIDENT TIMELINE:
  03:47 — Partner API sends a batch of 50,000 order events
  03:47 — 847 of those events have MALFORMED JSON
           (truncated payloads due to partner's HTTP timeout)
  03:48 — Kafka Connect Sink connector consuming those events CRASHES
           on the first malformed record
  03:48 — ALL 50,000 events stop flowing — pipeline is BLOCKED
  03:49 — Consumer application reading same topic hits
           deserialization exception → consumer loop dies
  03:50 — Downstream dashboard shows ZERO orders for the last 3 minutes
  03:52 — PagerDuty alert fires
  04:15 — On-call engineer wakes up, investigates
  04:45 — Engineer manually skips the bad offset (risky, loses data)
  05:00 — Pipeline resumes but 847 events are permanently LOST
  05:30 — Business discovers 847 orders were never processed
  
  TOTAL DAMAGE:
  → 1 hour 43 minutes of pipeline downtime
  → 847 lost orders ($127,000 in revenue at risk)
  → 49,153 VALID orders delayed by 73 minutes
  → On-call engineer's sleep destroyed

→ ROOT CAUSE: ONE bad record blocked the ENTIRE pipeline
  This is the "Poison Pill" problem

→ REQUIREMENT:
  1. Bad records must NOT block good records
  2. Bad records must NOT be silently dropped
  3. Bad records must be captured somewhere for investigation
  4. Pipeline must self-heal without human intervention
  5. Alerting must fire when bad records exceed threshold

🔍 HOW TO APPROACH IT

THINKING THROUGH THE PROBLEM:

1. From Project 12 → We know consumers deserialize messages
   → Deserialization failure = consumer crash

2. From Project 15 → We know Kafka Connect has converters
   → Converter failure = task crash

3. From Project 24 → We know SMTs transform records inline
   → SMT exception = task crash

4. EVERY layer of the pipeline can encounter "poison pills"

WHAT IS A POISON PILL?
→ Any message that causes a consumer/connector to FAIL
→ Categories:
  a) Deserialization poison: corrupt bytes, wrong schema
  b) Processing poison: null fields where non-null expected
  c) Schema poison: Avro schema mismatch (Project 16)
  d) Size poison: message exceeds max.message.bytes
  e) Encoding poison: UTF-8 expected, got binary blob

THE FUNDAMENTAL TENSION:
→ FAIL FAST: Stop on error → guarantees no data corruption
   BUT: one bad record blocks millions of good records
→ SKIP SILENTLY: Ignore errors → pipeline keeps flowing
   BUT: data loss with no visibility
→ DEAD LETTER QUEUE: Divert bad records → pipeline flows
   AND: bad records preserved for investigation
   THIS IS THE ANSWER

DLQ PATTERN:
  Main Topic ──→ Consumer/Connector
                      │
                ┌─────┴─────┐
                │           │
            SUCCESS      FAILURE
                │           │
                ▼           ▼
          Process      Dead Letter
          Normally     Queue Topic
                           │
                           ▼
                     Investigation
                     + Retry later
                     + Alert team

🛠️ HOW CAN WE SOLVE IT

SOLUTION: THREE-LAYER DLQ STRATEGY

═══════════════════════════════════════════════════════════
LAYER 1: KAFKA CONNECT — Built-in DLQ Support
═══════════════════════════════════════════════════════════

  Kafka Connect (2.0+) has NATIVE error handling:
  
  "errors.tolerance": "all"
  "errors.deadletterqueue.topic.name": "dlq-connector-name"
  "errors.deadletterqueue.topic.replication.factor": 3
  "errors.deadletterqueue.context.headers.enable": true
  
  FLOW:
  Source Topic → Sink Connector → tries to process
                     │
               ┌─────┴──────┐
               │             │
           Success        Failure
               │             │
               ▼             ▼
          Target DB    DLQ Topic
                    (with error headers)


═══════════════════════════════════════════════════════════
LAYER 2: CONSUMER APPLICATION — Error-Handling Deserializer
═══════════════════════════════════════════════════════════

  Custom Python/Java consumer that catches deserialization
  errors and routes them to DLQ:
  
  Main Topic → Consumer Poll
                    │
              ┌─────┴──────┐
              │             │
          Deserialize   Deserialize
          SUCCESS       FAILURE
              │             │
              ▼             ▼

  Record deserializes fine but PROCESSING fails:

                │             │
            Logic OK     Logic FAILS
                │             │
                ▼             ▼

                       error details +
                       retry count)

✅ WHAT COULD BE DONE

OPTION A: errors.tolerance=none (DEFAULT — fail fast)
  → Connector/consumer stops on first error
  → Guarantees no silent data loss
  → But blocks entire pipeline
  → Suitable for: financial transactions where every record matters

OPTION B: errors.tolerance=all + NO DLQ
  → Connector skips bad records silently
  → Pipeline keeps flowing
  → Bad records vanish without trace
  → NEVER acceptable in production

OPTION C: errors.tolerance=all + DLQ topic ← PRIMARY SOLUTION
  → Connector diverts bad records to DLQ
  → Pipeline keeps flowing
  → Bad records preserved with full context
  → Team can investigate and replay
  → THIS is the production standard

OPTION D: Custom error-handling deserializer ← CONSUMER SOLUTION
  → Application-level DLQ for consumer apps
  → Full control over error handling logic
  → Can implement retry logic before DLQ
  → Required when not using Kafka Connect

OPTION E: Hybrid (Connect DLQ + Application DLQ) ← ENTERPRISE
  → Connect handles converter/SMT errors
  → Application handles business logic errors
  → Two DLQ topics per pipeline stage
  → Maximum visibility and recoverability

✅ WHAT SHOULD DO

1. ALWAYS ENABLE DLQ CONTEXT HEADERS
   → "errors.deadletterqueue.context.headers.enable": true
   → Adds headers to DLQ records:
     __connect.errors.topic        → source topic
     __connect.errors.partition    → source partition
     __connect.errors.offset       → source offset
     __connect.errors.connector.name → which connector
     __connect.errors.task.id      → which task
     __connect.errors.stage        → VALUE_CONVERTER, HEADER_CONVERTER, TRANSFORMATION
     __connect.errors.class.name   → exception class
     __connect.errors.exception.message → error message
   → WITHOUT these: DLQ record is opaque — you know it failed
     but not WHY or WHERE

2. SET DLQ TOPIC REPLICATION FACTOR ≥ 2
   → DLQ data is EVIDENCE — losing it means losing audit trail
   → "errors.deadletterqueue.topic.replication.factor": 3

3. MONITOR DLQ TOPIC LAG / MESSAGE COUNT
   → Set up alerting: if DLQ receives > N messages in M minutes
   → Indicates systemic issue, not just occasional bad record
   → Links to monitoring concepts from Project 28

4. IMPLEMENT DLQ CONSUMER FOR INVESTIGATION
   → Don't just dump to DLQ and forget
   → Build a consumer that reads DLQ, logs to investigation system
   → Human reviews and either fixes + replays or discards

5. PRESERVE ORIGINAL BYTES IN DLQ
   → DLQ should contain the EXACT original message
   → Not a transformed or partial version
   → Enables replay after fixing the issue

6. USE DESCRIPTIVE DLQ TOPIC NAMES
   → Pattern: dlq.<source-connector-or-app>.<source-topic>
   → Example: dlq.jdbc-orders.raw_orders
   → NOT: dlq1, errors, bad-records

7. SET RETENTION ON DLQ TOPICS
   → Bad records need investigation window (7-30 days)
   → Don't keep forever — set retention.ms
   → Don't delete too fast — team needs time to investigate

8. LOG ERRORS EVEN WHEN TOLERATING THEM
   → "errors.log.enable": true
   → "errors.log.include.messages": true
   → Writes error details to Connect worker log
   → Secondary evidence trail alongside DLQ topic

❌ WHAT SHOULD NOT DO

1. DO NOT USE errors.tolerance=all WITHOUT A DLQ
   → This is the WORST configuration
   → Silently swallows bad records
   → You will NEVER know data was lost
   → Equivalent to: catch(Exception e) { /* ignore */ }

2. DO NOT ASSUME DLQ RECORDS ARE AUTOMATICALLY RETRIED
   → Kafka Connect DLQ is write-only from Connect's perspective
   → Connect writes to DLQ and MOVES ON
   → YOU must build the retry mechanism
   → Project 26 covers retry patterns in depth

3. DO NOT PUT DLQ LOGIC INSIDE THE MAIN CONSUMER LOOP WITHOUT TRY-CATCH
   → If your DLQ-producing code ALSO throws an exception
   → You get an infinite error loop
   → Always wrap DLQ production in its own error handling

4. DO NOT USE THE SAME DLQ TOPIC FOR ALL CONNECTORS
   → Different connectors have different error shapes
   → Mixing them makes investigation impossible
   → One DLQ topic per connector or per pipeline stage

5. DO NOT IGNORE DLQ GROWTH RATE
   → 1 bad record per million = normal (partner data quality)
   → 1000 bad records per million = something is BROKEN
   → DLQ growth rate is a critical operational metric

6. DO NOT REPLAY DLQ RECORDS WITHOUT FIXING THE ROOT CAUSE
   → If you replay bad records into the same pipeline
   → They will fail AGAIN and land back in DLQ
   → Fix the parser/schema/SMT first, THEN replay

7. DO NOT STORE SENSITIVE DATA IN DLQ ERROR HEADERS
   → Error messages might contain field values
   → "errors.log.include.messages": true exposes record content in logs
   → In PII-regulated environments, mask before logging
   → Connects to masking concepts from Project 24

🚫 WHEN NOT TO USE

1. WHEN ZERO DATA LOSS IS NON-NEGOTIABLE
   → Banking wire transfers, stock trades
   → Every record MUST be processed or pipeline MUST stop
   → Use errors.tolerance=none + immediate alerting
   → Human decides: fix and resume vs manual skip

2. WHEN DLQ CREATES COMPLIANCE RISK
   → If DLQ topic stores unmasked PII
   → And DLQ doesn't have same access controls as source
   → Bad records containing SSN land in unprotected DLQ
   → MITIGATION: Apply SMTs (Project 24) BEFORE DLQ routing
   → Or restrict DLQ topic ACLs (Project 30)

3. WHEN ERROR RATE INDICATES SYSTEMIC FAILURE
   → DLQ is for OCCASIONAL bad records
   → If 50% of records are failing → the SYSTEM is broken
   → DLQ should NOT be a substitute for fixing root cause
   → Set threshold: if error_rate > 5% → STOP pipeline + alert

4. WHEN RECORD ORDERING IS CRITICAL WITHIN A PARTITION
   → DLQ diverts record out of main flow
   → Remaining records continue in order
   → But the DLQ'd record is now OUT of sequence
   → If downstream requires STRICT ordering including failed records
   → DLQ breaks that guarantee
   → Use errors.tolerance=none instead

ERROR REPORTER INTERNALS:
═══════════════════════════════════════════════════════════

  CLASS: DeadLetterQueueReporter

  When an error occurs:
  
  STEP 1: Capture original ConsumerRecord
          (raw key bytes, raw value bytes, headers, 
           topic, partition, offset, timestamp)

  STEP 2: Create error context headers:
          ┌─────────────────────────────────────────────┐
          │ Header Name                    │ Value       │
          ├────────────────────────────────┼─────────────┤
          │ __connect.errors.topic         │ orders      │
          │ __connect.errors.partition     │ 3           │
          │ __connect.errors.offset        │ 847291      │
          │ __connect.errors.connector.name│ jdbc-sink   │
          │ __connect.errors.task.id       │ 0           │
          │ __connect.errors.stage         │ VALUE_CONV  │
          │ __connect.errors.class.name    │ JsonParseEx │
          │ __connect.errors.exception.msg │ Unexpected  │
          │                                │ char at 127 │
          │ __connect.errors.exception.    │ full stack  │
          │   stacktrace                   │ trace       │
          └────────────────────────────────┴─────────────┘

  STEP 3: Produce to DLQ topic:
          - Key: original key bytes (unchanged)
          - Value: original value bytes (unchanged)
          - Headers: original headers + error context headers
          - The DLQ record is the ORIGINAL record with error metadata

  STEP 4: Log error (if errors.log.enable=true):
          - Writes to Connect worker log at ERROR level
          - Includes record content if errors.log.include.messages=true

  STEP 5: SKIP the record — advance offset
          - Main pipeline continues with next record
          - Failed record's offset is committed (won't be retried)

CONSUMER APPLICATION DLQ — HOW IT WORKS:
═══════════════════════════════════════════════════════════

  Unlike Kafka Connect, consumer apps have NO built-in DLQ
  You must implement it yourself:

  consumer.poll()
      │
      ▼
  ┌─────────────────────────┐
  │ For each message:       │
  │                         │
  │ TRY:                    │
  │   deserialize(msg)      │──→ CATCH DeserError ──┐
  │   process(record)       │──→ CATCH ProcessError ─┤
  │   commit offset         │                        │
  │                         │                        │
  │ DLQ PATH:               │◄───────────────────────┘
  │   produce_to_dlq(       │
  │     original_bytes,     │
  │     error_info,         │
  │     source_metadata     │
  │   )                     │
  │   commit offset         │ ← CRITICAL: commit so we don't
  │                         │    reprocess the same poison pill
  └─────────────────────────┘

  KEY DIFFERENCE FROM CONNECT DLQ:
  → Connect DLQ: automatic, config-only
  → Consumer DLQ: manual, code-required
  → Connect DLQ: limited to Connect pipeline stages
  → Consumer DLQ: can handle ANY error type you define

OFFSET BEHAVIOR WITH DLQ:
═══════════════════════════════════════════════════════════

  BEFORE DLQ (errors.tolerance=none):
  
  Offset:  100  101  102  103  104  105
  Record:  OK   OK   BAD  OK   OK   OK
                      ↑
                      STUCK HERE FOREVER
                      (until manual intervention)
  
  Committed offset: 102 (last successful + 1)
  Consumer lag grows indefinitely


  AFTER DLQ (errors.tolerance=all):
  
  Offset:  100  101  102  103  104  105
  Record:  OK   OK   BAD  OK   OK   OK
           ✓    ✓    →DLQ  ✓    ✓    ✓
  
  Committed offset: 106 (all processed or DLQ'd)
  Consumer lag stays near zero
  DLQ topic has 1 record (offset 102's content)

ERRORS AT EACH STAGE — WHAT GETS CAPTURED:
═══════════════════════════════════════════════════════════

  STAGE: HEADER_CONVERTER
  → Error: Header bytes cannot be converted
  → DLQ header: __connect.errors.stage = HEADER_CONVERTER
  → DLQ value: original raw bytes

  STAGE: KEY_CONVERTER
  → Error: Key bytes cannot be deserialized
  → DLQ header: __connect.errors.stage = KEY_CONVERTER
  → DLQ value: original raw bytes (key + value)

  STAGE: VALUE_CONVERTER
  → Error: Value bytes cannot be deserialized (most common)
  → DLQ header: __connect.errors.stage = VALUE_CONVERTER
  → DLQ value: original raw bytes
  → Common cause: malformed JSON, wrong Avro schema

  STAGE: TRANSFORMATION
  → Error: SMT throws exception (Project 24)
  → DLQ header: __connect.errors.stage = TRANSFORMATION
  → DLQ value: partially converted record
  → Common cause: SMT expects field that doesn't exist

  STAGE: SINK (put/flush)
  → Error: Target system rejects record
  → DLQ header: __connect.errors.stage = PUT
  → DLQ value: fully converted + transformed record
  → Common cause: type mismatch, constraint violation

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: DLQ PRODUCER FAILS
══════════════════════════════
  → Bad record triggers DLQ write
  → But DLQ topic doesn't exist + auto.create.topics.enable=false
  → DLQ production FAILS
  → Connect behavior: depends on implementation
     - Some: skip DLQ write, just log
     - Some: throw exception, task dies
  FIX: Pre-create DLQ topics with appropriate config
       OR ensure auto.create.topics.enable=true


PITFALL 2: DLQ TOPIC FILLS UP / RETENTION EXPIRES
══════════════════════════════════════════════════
  → Bad records land in DLQ
  → Nobody investigates for 30 days
  → DLQ retention.ms = 7 days (default)
  → Records auto-deleted before investigation
  FIX: Set DLQ retention to 30+ days
       Set up alerting on DLQ message count
       Assign ownership for DLQ review


PITFALL 3: INFINITE DLQ LOOP
═════════════════════════════
  → Bad record goes to DLQ
  → DLQ consumer tries to process it
  → Processing fails AGAIN
  → Sends to another DLQ
  → That DLQ consumer tries to process...
  → INFINITE LOOP
  
  FIX: DLQ consumers should NEVER auto-route to another DLQ
       Instead: log, alert, require HUMAN decision
       OR: implement max retry count (Project 26)


PITFALL 4: errors.tolerance=all MASKS SYSTEMIC ISSUES
═════════════════════════════════════════════════════
  → Schema change breaks ALL records
  → errors.tolerance=all diverts 100% to DLQ
  → Pipeline appears "running" but processes NOTHING useful
  → Dashboard shows activity (DLQ writes) but zero real throughput
  
  FIX: Monitor ratio: DLQ_records / total_records
       Alert if ratio > 1% (configurable threshold)
       Alert if DLQ receives > N records in M minutes


PITFALL 5: DLQ RECORDS HAVE NO SCHEMA
══════════════════════════════════════
  → DLQ contains RAW BYTES of failed records
  → These bytes are often MALFORMED (that's why they failed)
  → Cannot deserialize DLQ records with standard deserializer
  → DLQ consumer must handle raw bytes
  
  FIX: Read DLQ records as byte arrays
       Extract error context from headers
       Display raw bytes as hex/string for investigation
       Don't assume DLQ values are valid JSON/Avro


PITFALL 6: COMMITTING OFFSET BEFORE DLQ WRITE COMPLETES
════════════════════════════════════════════════════════
  (Application-level DLQ)
  → Consumer catches error, starts producing to DLQ
  → Commits offset immediately (before DLQ ack)
  → DLQ produce fails
  → Record is LOST: not in main flow, not in DLQ
  
  FIX: Wait for DLQ produce acknowledgment BEFORE committing
       Use synchronous DLQ produce (flush/get)
       Or use transactions (Project 18) spanning both


PITFALL 7: DLQ TOPIC PERMISSIONS
════════════════════════════════
  → Connect worker needs WRITE permission on DLQ topic
  → If ACLs (Project 30) block DLQ writes
  → Error handling itself fails
  → Task may crash or silently skip
  
  FIX: Include DLQ topics in ACL configuration
       Test DLQ write path in staging environment

📈 SCALABILITY CONSIDERATIONS

DLQ TOPIC SIZING:
═════════════════
  → DLQ records = same size as original records + error headers (~500 bytes)
  → If source topic has 1M records/day and 0.1% error rate
  → DLQ receives ~1,000 records/day
  → At 1KB average record: ~1MB/day → negligible
  → BUT: if error rate spikes to 10% → 100K records → 100MB/day
  → PLAN FOR WORST CASE in retention calculations

DLQ PARTITION COUNT:
════════════════════
  → DLQ doesn't need high parallelism (low volume)
  → 1-3 partitions is sufficient for most cases
  → More partitions = more overhead for minimal benefit
  → Exception: if DLQ volume is consistently high → scale up

CONNECT WORKER IMPACT:
══════════════════════
  → Writing to DLQ = additional produce call per bad record
  → If 1 in 10,000 records fails: negligible impact
  → If 1 in 10 records fails: significant overhead
     (each DLQ write = network round trip to broker)
  → At high error rates, DLQ writes become the bottleneck

MULTI-CONNECTOR DLQ STRATEGY:
═════════════════════════════
  → 10 connectors × 1 DLQ topic each = 10 DLQ topics
  → 50 connectors × 1 DLQ topic each = 50 DLQ topics
  → Each DLQ topic has partitions, replicas, metadata
  → OPTION: Shared DLQ with connector name in headers
  → BETTER: Per-connector DLQ for isolation
  → Use naming convention: dlq.<connector-name>

DLQ CONSUMER SCALING:
═════════════════════
  → DLQ consumers are typically LOW throughput
  → Often a single consumer per DLQ is sufficient
  → DLQ processing is MANUAL/SEMI-AUTOMATED
  → Don't over-engineer DLQ consumer parallelism

RETENTION TRADE-OFF:
════════════════════
  → Longer retention = more investigation time
  → Longer retention = more storage
  → Recommendation:
    - DLQ retention: 30 days (investigation window)
    - Archive DLQ to object storage after 30 days
    - Delete from Kafka after 30 days

class ErrorClassification:
    DESERIALIZATION = "DESERIALIZATION_ERROR"
    VALIDATION = "VALIDATION_ERROR"
    BUSINESS_RULE = "BUSINESS_RULE_ERROR"
    PROCESSING = "PROCESSING_ERROR"
    UNKNOWN = "UNKNOWN_ERROR"

# ============================================================
# DLQ Writer
# ============================================================

def send_to_dlq(original_msg, error_class, error_message, error_detail=""):
    """
    Sends failed record to DLQ with full error context
    
    CRITICAL: Preserves original raw bytes
    → DLQ value = original message value (not transformed)
    → Error context goes into HEADERS
    """
    
    # Build error context headers
    error_headers = [
        # Source tracking
        ("dlq.source.topic", 
         original_msg.topic().encode('utf-8') if original_msg.topic() else b"unknown"),
        ("dlq.source.partition", 
         str(original_msg.partition()).encode('utf-8')),
        ("dlq.source.offset", 
         str(original_msg.offset()).encode('utf-8')),
        ("dlq.source.timestamp", 
         str(original_msg.timestamp()[1] or 0).encode('utf-8')),
        
        # Error classification
        ("dlq.error.class", 
         error_class.encode('utf-8')),
        ("dlq.error.message", 
         error_message[:500].encode('utf-8')),  # Truncate long messages
        ("dlq.error.detail", 
         error_detail[:1000].encode('utf-8')),
        
        # Processing metadata
        ("dlq.processor.name", 
         b"order-processor-with-dlq"),
        ("dlq.processor.timestamp", 
         str(int(time.time() * 1000)).encode('utf-8')),
        ("dlq.processor.attempt", 
         b"1"),
    ]
    
    # Preserve original headers if any
    original_headers = original_msg.headers() or []
    for h_key, h_val in original_headers:
        error_headers.append((f"dlq.original.header.{h_key}", h_val))
    
    try:
        dlq_producer.produce(
            topic=DLQ_TOPIC,
            key=original_msg.key(),         # Original key
            value=original_msg.value(),     # Original raw bytes
            headers=error_headers,
            callback=lambda err, msg: (
                print(f"    → DLQ write {'FAILED: ' + str(err) if err else 'OK'}")
            )
        )
        # SYNCHRONOUS FLUSH — must complete before committing offset
        dlq_producer.flush(timeout=10)
        return True
    except Exception as e:
        print(f"    ❌ DLQ WRITE FAILED: {e}")
        print(f"    ⚠️  Record may be LOST (topic={original_msg.topic()}, "
              f"partition={original_msg.partition()}, offset={original_msg.offset()})")
        return False

# Deserialization Layer
# ============================================================

def safe_deserialize(msg):
    """
    Attempts to deserialize message value as JSON
    Returns (success: bool, data: dict or None, error: str or None)
    """
    raw_value = msg.value()
    
    # Handle null/empty values
    if raw_value is None:
        return False, None, "Null value (tombstone or empty)"
    
    if len(raw_value) == 0:
        return False, None, "Empty byte array"
    
    try:
        text = raw_value.decode('utf-8')
    except UnicodeDecodeError as e:
        return False, None, f"UTF-8 decode failed: {e}"
    
    try:
        data = json.loads(text)
    except json.JSONDecodeError as e:
        return False, None, f"JSON parse failed: {e}"
    
    if not isinstance(data, dict):
        return False, None, f"Expected JSON object, got {type(data).__name__}"
    
    return True, data, None

# Validation Layer
# ============================================================

def validate_order(data):
    """
    Business validation rules for order events
    Returns (valid: bool, errors: list of str)
    """
    errors = []
    
    # Required fields
    required = ['order_id', 'customer_name', 'amount', 'currency']
    for field in required:
        if field not in data:
            errors.append(f"Missing required field: {field}")
    
    if errors:
        return False, errors
    
    # Type checks
    if not isinstance(data.get('order_id'), (int, float)):
        errors.append(
            f"order_id must be numeric, got {type(data.get('order_id')).__name__}: "
            f"{repr(data.get('order_id'))}"
        )
    
    if not isinstance(data.get('amount'), (int, float)):
        errors.append(
            f"amount must be numeric, got {type(data.get('amount')).__name__}: "
            f"{repr(data.get('amount'))}"
        )
    elif data['amount'] <= 0:
        errors.append(f"amount must be positive, got {data['amount']}")
    
    if not isinstance(data.get('customer_name'), str):
        errors.append(
            f"customer_name must be string, got "
            f"{type(data.get('customer_name')).__name__}"
        )
    
    valid_currencies = {'USD', 'EUR', 'GBP', 'JPY', 'CAD'}
    if data.get('currency') not in valid_currencies:
        errors.append(
            f"Invalid currency: {data.get('currency')} "
            f"(valid: {valid_currencies})"
        )
    
    return len(errors) == 0, errors

def main():
    consumer.subscribe([SOURCE_TOPIC])
    
    stats = {
        'total': 0,
        'success': 0,
        'deser_error': 0,
        'validation_error': 0,
        'processing_error': 0,
        'dlq_write_fail': 0,
    }
    
    print("=" * 70)
    print("  ORDER PROCESSOR WITH DEAD LETTER QUEUE")
    print("=" * 70)
    print(f"  Source: {SOURCE_TOPIC}")
    print(f"  DLQ:    {DLQ_TOPIC}")
    print("=" * 70)
    
    empty_polls = 0
    
    try:
        while empty_polls < 15:
            msg = consumer.poll(2.0)
            
            if msg is None:
                empty_polls += 1
                continue
            
            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    continue
                print(f"  Consumer error: {msg.error()}")
                continue
            
            empty_polls = 0
            stats['total'] += 1
            record_id = f"P{msg.partition()}:O{msg.offset()}"
            
            # ============================================
            # LAYER 1: DESERIALIZATION
            # ============================================
            success, data, deser_error = safe_deserialize(msg)
            
            if not success:
                stats['deser_error'] += 1
                print(f"\n  [{record_id}] 💀 DESERIALIZATION ERROR")
                print(f"    Error: {deser_error}")
                
                # Peek at raw bytes
                raw = msg.value()
                if raw:
                    preview = raw[:80]
                    print(f"    Raw bytes ({len(raw)} bytes): {preview}")
                
                # Route to DLQ
                dlq_ok = send_to_dlq(
                    msg,
                    ErrorClassification.DESERIALIZATION,
                    deser_error,
                    f"Raw length: {len(raw) if raw else 0}"
                )
                
                if not dlq_ok:
                    stats['dlq_write_fail'] += 1
                
                # Commit offset — move past the poison pill
                consumer.commit(message=msg)
                continue
            
            # ============================================
            # LAYER 2: VALIDATION
            # ============================================
            valid, validation_errors = validate_order(data)
            
            if not valid:
                stats['validation_error'] += 1
                print(f"\n  [{record_id}] ⚠️  VALIDATION ERROR (order_id={data.get('order_id', '?')})")
                for ve in validation_errors:
                    print(f"    → {ve}")
                
                dlq_ok = send_to_dlq(
                    msg,
                    ErrorClassification.VALIDATION,
                    "; ".join(validation_errors),
                    json.dumps(data)[:500]
                )
                
                if not dlq_ok:
                    stats['dlq_write_fail'] += 1
                
                consumer.commit(message=msg)
                continue
            
            # ============================================
            # LAYER 3: PROCESSING
            # ============================================
            try:
                result = process_order(data)
                stats['success'] += 1
                
                if stats['success'] % 10 == 0:
                    print(f"\n  [{record_id}] ✅ Processed: order_id={result['order_id']}, "
                          f"amount=${result['amount']:.2f}")
                
                # Commit offset — successful processing
                consumer.commit(message=msg)
                
            except Exception as proc_error:
                stats['processing_error'] += 1
                print(f"\n  [{record_id}] 🔥 PROCESSING ERROR (order_id={data.get('order_id', '?')})")
                print(f"    Error: {proc_error}")
                
                dlq_ok = send_to_dlq(
                    msg,
                    ErrorClassification.PROCESSING,
                    str(proc_error),
                    traceback.format_exc()
                )
                
                if not dlq_ok:
                    stats['dlq_write_fail'] += 1
                
                consumer.commit(message=msg)
    
    except KeyboardInterrupt:
        print("\n  Shutting down...")
    
    finally:
        consumer.close()
        dlq_producer.flush()
    
    # ============================================
    # Print Summary
    # ============================================
    print(f"\n{'=' * 70}")
    print(f"  PROCESSING SUMMARY")
    print(f"{'=' * 70}")
    print(f"  Total records consumed:     {stats['total']}")
    print(f"  ✅ Successfully processed:  {stats['success']}")
    print(f"  💀 Deserialization errors:  {stats['deser_error']}")
    print(f"  ⚠️  Validation errors:      {stats['validation_error']}")
    print(f"  🔥 Processing errors:       {stats['processing_error']}")
    print(f"  ❌ DLQ write failures:      {stats['dlq_write_fail']}")
    
    total_dlq = stats['deser_error'] + stats['validation_error'] + stats['processing_error']
    error_rate = (total_dlq / stats['total'] * 100) if stats['total'] > 0 else 0
    
    print(f"\n  Total sent to DLQ:          {total_dlq}")
    print(f"  Error rate:                 {error_rate:.1f}%")
    
    if error_rate > 5:
        print(f"\n  🚨 ERROR RATE ABOVE 5% — INVESTIGATE ROOT CAUSE")
    elif error_rate > 1:
        print(f"\n  ⚠️  ERROR RATE ABOVE 1% — MONITOR CLOSELY")
    else:
        print(f"\n  ✅ ERROR RATE ACCEPTABLE")
    
    print(f"{'=' * 70}")

if __name__ == '__main__':

def extract_headers(msg):
    """Extract all headers into a readable dictionary"""
    headers = {}
    raw_headers = msg.headers() or []
    
    for key, value in raw_headers:
        try:
            headers[key] = value.decode('utf-8') if value else ""
        except UnicodeDecodeError:
            headers[key] = f"<binary: {len(value)} bytes>"
    
    return headers

def classify_dlq_source(headers):
    """Determine if DLQ record came from Connect or Application"""
    if any(k.startswith('__connect.errors') for k in headers):
        return "KAFKA_CONNECT"
    elif any(k.startswith('dlq.') for k in headers):
        return "APPLICATION"
    else:
        return "UNKNOWN"

def format_raw_value(raw_bytes, max_display=200):
    """Attempt to display raw value in readable format"""
    if raw_bytes is None:
        return "<NULL>"
    
    if len(raw_bytes) == 0:
        return "<EMPTY>"
    
    # Try UTF-8
    try:
        text = raw_bytes.decode('utf-8')
        if len(text) > max_display:
            return text[:max_display] + f"... ({len(text)} chars total)"
        return text
    except UnicodeDecodeError:
        pass
    
    # Fall back to hex
    hex_preview = raw_bytes[:50].hex()
    return f"<BINARY: {len(raw_bytes)} bytes> {hex_preview}..."

        top_class = max(stats['by_error_class'], key=stats['by_error_class'].get)
        top_count = stats['by_error_class'][top_class]
        top_pct = top_count / total * 100
        
        short_cls = top_class.split('.')[-1] if '.' in top_class else top_class
        
        if 'DESERIALIZATION' in top_class.upper():
            recs.append(
                f"Top error ({top_pct:.0f}%) is DESERIALIZATION → "

            )
        elif 'VALIDATION' in top_class.upper():
            recs.append(
                f"Top error ({top_pct:.0f}%) is VALIDATION → "
                f"Source data doesn't meet schema requirements. "
                f"Coordinate with data producer to fix field types/formats."
            )
        elif 'PROCESSING' in top_class.upper():
            recs.append(
                f"Top error ({top_pct:.0f}%) is PROCESSING → "
                f"Application logic failing. Review business rules "
                f"and exception handling in processor code."
            )
    
    # Check Connect stages
    if stats['by_stage']:
        top_stage = max(stats['by_stage'], key=stats['by_stage'].get)
        if 'VALUE_CONVERTER' in top_stage:
            recs.append(
                "Connect VALUE_CONVERTER errors dominant → "

            recs.append(
                "Connect TRANSFORMATION errors dominant → "
                "SMT chain (Project 24) is failing on some records. "
                "Review SMT field expectations vs actual record structure."
            )
    
    # Volume-based recommendations
    if total > 100:
        recs.append(
            f"HIGH DLQ VOLUME ({total} records) → "
            f"This suggests a systemic issue, not isolated bad records. "
            f"Consider pausing the pipeline until root cause is fixed."

            "DLQ volume is low and error types are mixed → "
            "Likely occasional bad data from external sources. "
            "Normal operational noise. Monitor trends."
        )
    
    recs.append(
        "After fixing root cause, replay DLQ records: "
        "consume from DLQ → produce to source topic → "
        "verify successful processing."
    )
    
    return recs

        self.target_topic_override = target_topic
        self.dry_run = dry_run
        
        self.consumer = Consumer({
            'bootstrap.servers': BOOTSTRAP,

            'enable.auto.commit': False
        })
        
        self.producer = Producer({
            'bootstrap.servers': BOOTSTRAP,
            'acks': 'all'
        })
        
        self.stats = {
            'scanned': 0,
            'replayed': 0,
            'skipped': 0,
            'failed': 0
        }
    
    def extract_source_info(self, msg):
        """Extract original source topic from DLQ headers"""
        headers = dict(msg.headers() or [])
        
        # Try Connect DLQ headers first
        source_topic_bytes = headers.get('__connect.errors.topic')
        if source_topic_bytes:
            return source_topic_bytes.decode('utf-8')
        
        # Try application DLQ headers
        source_topic_bytes = headers.get('dlq.source.topic')
        if source_topic_bytes:
            return source_topic_bytes.decode('utf-8')
        
        return None
    
    def strip_dlq_headers(self, msg):
        """
        Remove DLQ-specific headers before replay
        Keep original headers (if preserved with dlq.original.header. prefix)
        """
        original_headers = []
        
        for key, value in (msg.headers() or []):
            # Restore original headers
            if key.startswith('dlq.original.header.'):
                original_key = key.replace('dlq.original.header.', '')
                original_headers.append((original_key, value))
            # Skip all DLQ metadata headers
            elif key.startswith('dlq.') or key.startswith('__connect.errors'):
                continue
            else:
                # Keep any other headers
                original_headers.append((key, value))
        
        # Add replay tracking header
        original_headers.append((
            'X-DLQ-Replayed', 
            str(int(time.time() * 1000)).encode('utf-8')
        ))
        original_headers.append((
            'X-DLQ-Source-Topic',
            self.dlq_topic.encode('utf-8')
        ))
        
        return original_headers
    
    def replay_filter(self, msg, headers_dict):
        """
        Optional: Filter which DLQ records to replay
        Returns (should_replay: bool, reason: str)
        """
        # Example: Only replay deserialization errors (might be fixed now)

        # error_class = headers_dict.get('dlq.error.class', b'').decode('utf-8')
        # if 'DESERIALIZATION' not in error_class:
        #     return False, f"Skipping non-deser error: {error_class}"
        
        return True, "Included"
    
    def run(self):
        """Execute the replay"""
        
        print("=" * 70)
        print(f"  {'🔍 DRY RUN' if self.dry_run else '🔄 LIVE REPLAY'}: DLQ REPLAY TOOL")
        print("=" * 70)
        print(f"  DLQ Topic:    {self.dlq_topic}")
        print(f"  Target Topic: {self.target_topic_override or '(from DLQ headers)'}")
        print(f"  Mode:         {'DRY RUN (no writes)' if self.dry_run else '⚠️  LIVE (writing to target)'}")
        print("=" * 70)
        
        if not self.dry_run:
            print("\n  ⚠️  LIVE MODE — Records will be produced to target topic")
            print("  Press Ctrl+C within 5 seconds to abort...")
            try:
                time.sleep(5)
            except KeyboardInterrupt:
                print("  Aborted.")
                return
            print("  Proceeding with live replay...\n")
        
        self.consumer.subscribe([self.dlq_topic])
        
        empty_polls = 0
        
        try:
            while empty_polls < 10:
                msg = self.consumer.poll(2.0)
                
                if msg is None:
                    empty_polls += 1
                    continue
                
                if msg.error():
                    if msg.error().code() == KafkaError._PARTITION_EOF:
                        empty_polls += 1
                        continue
                    continue
                
                empty_polls = 0
                self.stats['scanned'] += 1
                
                # Determine target topic
                target = self.target_topic_override or self.extract_source_info(msg)
                
                if not target:
                    print(f"  ⚠️  Record #{self.stats['scanned']}: "
                          f"Cannot determine target topic — SKIPPED")
                    self.stats['skipped'] += 1
                    continue
                
                # Apply replay filter
                headers_dict = dict(msg.headers() or [])
                should_replay, reason = self.replay_filter(msg, headers_dict)

                    continue

                    try:

                    print(f"    Target: {target}")
                    print(f"    Key:    {key_str}")
                    print(f"    Value:  {raw_preview}")
                    print(f"    Headers: {len(clean_headers)} (DLQ headers stripped)")
                    self.stats['replayed'] += 1
                else:
                    try:
                        self.producer.produce(
                            topic=target,
                            key=msg.key(),
                            value=msg.value(),
                            headers=clean_headers,
                            callback=lambda err, m: None
                        )
                        self.producer.flush(timeout=10)
                        self.stats['replayed'] += 1
                        
                        if self.stats['replayed'] % 10 == 0:
                            print(f"  ✅ Replayed {self.stats['replayed']} records...")
                    
                    except Exception as e:
                        print(f"  ❌ Replay failed for record #{self.stats['scanned']}: {e}")
                        self.stats['failed'] += 1
        
        except KeyboardInterrupt:
            print("\n  Interrupted.")
        
        finally:
            self.consumer.close()
            self.producer.flush()
        
        # Summary
        print(f"\n{'=' * 70}")
        print(f"  REPLAY SUMMARY {'(DRY RUN)' if self.dry_run else '(LIVE)'}")
        print(f"{'=' * 70}")
        print(f"  Scanned:  {self.stats['scanned']}")
        print(f"  Replayed: {self.stats['replayed']}")
        print(f"  Skipped:  {self.stats['skipped']}")

        if self.dry_run and self.stats['replayed'] > 0:
            print(f"\n  To execute live replay, run with dry_run=False")
        
        print(f"{'=' * 70}")

    def get_topic_message_count(self, topic):
        """
        Get total message count for a topic using watermark offsets
        (high watermark - low watermark) across all partitions
        """
        consumer = Consumer({
            'bootstrap.servers': BOOTSTRAP,
            'group.id': f'dlq-monitor-probe-{int(time.time())}',
            'auto.offset.reset': 'latest'
        })
        
        try:
            metadata = consumer.list_topics(topic, timeout=5)
            
            if topic not in metadata.topics:
                return 0
            
            partitions = metadata.topics[topic].partitions
            total = 0
            
            for pid in partitions:
                tp = TopicPartition(topic, pid)
                low, high = consumer.get_watermark_offsets(tp, timeout=5)
                total += (high - low)
            
            return total
        except Exception as e:
            print(f"    Error reading {topic}: {e}")
            return 0
        finally:
            consumer.close()
    
    def get_consumer_group_lag(self, topic, group_id):
        """
        Get consumer lag for a group on a topic
        Lag = high watermark - committed offset
        """
        consumer = Consumer({
            'bootstrap.servers': BOOTSTRAP,
            'group.id': group_id,
            'auto.offset.reset': 'latest'
        })
        
        try:
            metadata = consumer.list_topics(topic, timeout=5)
            
            if topic not in metadata.topics:
                return 0
            
            partitions = metadata.topics[topic].partitions
            total_lag = 0
            
            for pid in partitions:
                tp = TopicPartition(topic, pid)
                committed = consumer.committed([tp], timeout=5)
                _, high = consumer.get_watermark_offsets(tp, timeout=5)
                
                if committed[0].offset >= 0:
                    lag = high - committed[0].offset
                else:
                    lag = high  # No committed offset means full lag
                
                total_lag += lag
            
            return total_lag
        except Exception:
            return -1
        finally:
            consumer.close()

            'value': metric_value,
            'threshold': threshold,
        }
        
        self.alerts_fired.append(alert)
        
        icon = '🔴' if severity == 'CRITICAL' else '🟡' if severity == 'WARNING' else '🔵'
        
        print(f"\n  {icon} ALERT [{severity}]: {message}")
        print(f"     Metric: {metric_name} = {metric_value} (threshold: {threshold})")
        print(f"     Time: {alert['timestamp']}")
        
        # In production, this would send to:
        # → Slack webhook
        # → PagerDuty API
        # → Email
        # → Prometheus alertmanager
        # → Custom incident management system
    
    def check_once(self):
        """Run one monitoring check cycle"""
        
        now = time.time()
        timestamp = datetime.now().strftime('%H:%M:%S')
        
        print(f"\n  ─── Check at {timestamp} ───")
        
        for dlq_topic in self.dlq_topics:
            dlq_count = self.get_topic_message_count(dlq_topic)
            
            # Track history for rate calculation
            self.history[dlq_topic].append((now, dlq_count))
            
            # Keep only last 10 minutes of history
            cutoff = now - 600
            self.history[dlq_topic] = [
                (t, c) for t, c in self.history[dlq_topic] if t > cutoff
            ]
            
            print(f"  DLQ: {dlq_topic}")
            print(f"    Total records: {dlq_count}")
            
            # --- Check 1: Total unprocessed DLQ records ---
            if dlq_count > ALERT_THRESHOLDS['dlq_total_unprocessed']:
                self.fire_alert(
                    'WARNING',
                    f"DLQ topic {dlq_topic} has {dlq_count} unprocessed records",
                    'dlq_total_unprocessed',
                    dlq_count,
                    ALERT_THRESHOLDS['dlq_total_unprocessed']
                )
            
            # --- Check 2: DLQ growth rate ---
            history = self.history[dlq_topic]
            if len(history) >= 2:
                oldest_time, oldest_count = history[0]
                newest_time, newest_count = history[-1]
                
                elapsed_minutes = (newest_time - oldest_time) / 60.0
                
                if elapsed_minutes > 0:
                    growth_rate = (newest_count - oldest_count) / elapsed_minutes
                    print(f"    Growth rate: {growth_rate:.1f} records/min")
                    
                    if growth_rate > ALERT_THRESHOLDS['dlq_records_per_minute']:
                        self.fire_alert(
                            'CRITICAL',
                            f"DLQ {dlq_topic} growing at {growth_rate:.1f} records/min",
                            'dlq_records_per_minute',
                            round(growth_rate, 1),
                            ALERT_THRESHOLDS['dlq_records_per_minute']
                        )
        
        # --- Check 3: Error rate (DLQ vs Source) ---
        for dlq_topic, source_topic in zip(self.dlq_topics, self.source_topics):
            dlq_count = self.get_topic_message_count(dlq_topic)
            source_count = self.get_topic_message_count(source_topic)
            
            if source_count > 0:
                error_rate = (dlq_count / source_count) * 100
                print(f"  Error rate ({dlq_topic} / {source_topic}): {error_rate:.2f}%")
                
                if error_rate > ALERT_THRESHOLDS['error_rate_percent']:
                    self.fire_alert(
                        'CRITICAL',
                        f"Error rate {error_rate:.1f}% exceeds threshold "
                        f"({dlq_count} DLQ / {source_count} source)",
                        'error_rate_percent',
                        round(error_rate, 2),
                        ALERT_THRESHOLDS['error_rate_percent']
                    )
    
    def run_continuous(self, duration_seconds=180):
        """Run monitoring loop for specified duration"""
        
        print("=" * 70)
        print("  📡 DLQ CONTINUOUS MONITOR")
        print("=" * 70)
        print(f"  DLQ Topics:    {', '.join(self.dlq_topics)}")
        print(f"  Source Topics: {', '.join(self.source_topics)}")
        print(f"  Check Interval: {ALERT_THRESHOLDS['check_interval_seconds']}s")
        print(f"  Duration: {duration_seconds}s")
        print(f"\n  Thresholds:")
        for key, val in ALERT_THRESHOLDS.items():
            if key != 'check_interval_seconds':
                print(f"    {key}: {val}")
        print("=" * 70)
        
        start = time.time()
        
        try:
            while time.time() - start < duration_seconds:
                self.check_once()
                
                remaining = duration_seconds - (time.time() - start)
                sleep_time = min(
                    ALERT_THRESHOLDS['check_interval_seconds'],
                    max(0, remaining)
                )
                
                if sleep_time > 0:
                    time.sleep(sleep_time)
        
        except KeyboardInterrupt:
            print("\n  Monitor stopped by user.")
        
        # Final summary
        print(f"\n{'=' * 70}")
        print(f"  MONITOR SESSION SUMMARY")
        print(f"{'=' * 70}")
        print(f"  Duration: {time.time() - start:.0f} seconds")
        print(f"  Alerts fired: {len(self.alerts_fired)}")
        
        if self.alerts_fired:
            print(f"\n  Alert History:")
            for alert in self.alerts_fired:
                icon = '🔴' if alert['severity'] == 'CRITICAL' else '🟡'
                print(f"    {icon} [{alert['timestamp']}] {alert['message']}")
        else:
            print(f"  ✅ No alerts — DLQ metrics within normal range")
        
        print(f"{'=' * 70}")
