PROJECT 19: THE REAL-TIME WORD COUNTER — KAFKA STREAMS BASICS

📊 HYPOTHETICAL PROBLEM

A social media analytics company monitors trending topics in real-time

SCENARIO:
→ 50 million social media posts flow through Kafka daily
→ Posts arrive on topic "social-posts" as raw text messages
→ Business needs: "What words are trending RIGHT NOW?"
→ Marketing team needs live word frequency counts
→ Dashboard must update within 2 seconds of a post arriving

THE INITIAL APPROACH (Using Project 18 patterns):

  Developer A writes consume-process-produce manually:
  
  while True:
      msg = consumer.poll()
      words = msg.value().split()
      for word in words:
          word_counts[word] += 1      # In-memory state
          producer.begin_transaction()
          producer.produce("word-counts", key=word, value=count)
          producer.send_offsets_to_transaction(...)
          producer.commit_transaction()

THE CRISIS:
→ Week 1: Works for 1,000 posts/minute
→ Week 2: Traffic spikes to 100,000 posts/minute
→ Problems explode:

  Problem 1: STATE MANAGEMENT
  → word_counts dict is IN-MEMORY → lost on crash
  → 5 million unique words × count = significant memory
  → No way to recover state after restart without replaying ALL history
  → Replay of 50 million messages takes 6 hours

  Problem 2: SCALING
  → Single consumer instance can't handle 100K/min
  → Adding second instance? How to split state?
  → Word "trending" counted on instance A AND instance B
  → Need to MERGE counts → distributed aggregation problem
  → Manual partition assignment → manual state partitioning

  Problem 3: FAULT TOLERANCE
  → Instance crashes → state lost
  → Rebalance assigns partitions to surviving instance
  → That instance doesn't HAVE the state for those partitions
  → Must rebuild from scratch → hours of downtime

  Problem 4: EXACTLY-ONCE
  → Each word count update needs a transaction (Project 18)
  → 100K posts × avg 15 words = 1.5M transactions/minute
  → Transaction coordinator overwhelmed
  → Latency spikes to 30 seconds

  Problem 5: CODE COMPLEXITY
  → 500+ lines for basic word counting
  → State management, recovery, rebalancing, transactions
  → Every new feature (filtering, windowing) adds 200+ lines
  → Bug surface area is enormous

THE QUESTION:
→ Is there a framework that handles ALL of this automatically?
→ Where we write ONLY the business logic?
→ And get state management, scaling, fault tolerance, exactly-once for FREE?

THE ANSWER: KAFKA STREAMS

🔍 HOW TO APPROACH IT

Step 1: Understand WHAT Kafka Streams is
         → A Java/JVM CLIENT LIBRARY (not a separate cluster)
         → Runs inside YOUR application process
         → No separate infrastructure needed (unlike Spark, Flink)
         → Reads from Kafka topics → processes → writes to Kafka topics
         → Handles state, scaling, fault tolerance, EOS automatically

Step 2: Understand WHY it exists
         → Bridge between low-level Consumer/Producer API and full frameworks
         → Consumer/Producer = too much manual work (Projects 1-18 showed this)
         → Spark/Flink = separate cluster, operational overhead
         → Kafka Streams = "just a library" — deploy like any Java app

Step 3: Learn the core abstractions
         → KStream: Unbounded stream of key-value records (like a river)
         → KTable: Changelog stream representing latest value per key (like a database table)
         → Topology: Directed acyclic graph of processing nodes
         → StreamsBuilder: DSL for constructing topologies
         → State Store: Local persistent storage (RocksDB) for aggregations

Step 4: Understand the processing model
         → Input topic partitions = parallelism units
         → Each partition assigned to one stream task
         → Each task has its own state store
         → Tasks can run across multiple application instances
         → Rebalancing redistributes tasks (like consumer group rebalancing — Project 10)

Step 5: Connect to what we already know
         → KStream is a higher-level abstraction over Consumer (Project 5)
         → Output to topic uses Producer internally (Project 12)
         → State stores use changelog topics (like log compaction — Project 14)
         → EOS uses transactions internally (Project 18)
         → Partition assignment uses consumer groups (Project 5)

✅ WHAT SHOULD DO

1. USE StreamsBuilder DSL FOR MOST USE CASES
   → The DSL (Domain Specific Language) covers 90% of stream processing needs
   → filter(), map(), flatMap(), groupBy(), count(), reduce(), join()
   → Much more readable than the low-level Processor API
   → Processor API exists for edge cases (covered later)

2. SET application.id — THIS IS YOUR CONSUMER GROUP
   → application.id serves as the consumer group ID
   → Also used as prefix for internal topics and state store directories
   → Must be unique per Kafka Streams application
   → Must be stable across restarts

3. CONFIGURE STATE DIRECTORY
   → state.dir = where RocksDB stores local state
   → Default: /tmp/kafka-streams
   → In production: use persistent storage (SSD recommended)
   → If state lost: automatically rebuilt from changelog topics
   → Rebuild can take minutes to hours depending on state size

4. START SIMPLE — STATELESS FIRST, STATEFUL LATER
   → This project: basic topology with count() (stateful)
   → Project 20: stateless transformations (filter, map, flatMap)
   → Project 21: stateful aggregations (groupBy, count, reduce, aggregate)
   → Project 22: windowed operations
   → Build understanding layer by layer

5. USE PROPER SERDES (Serializer/Deserializer pairs)
   → Kafka Streams needs Serdes for keys AND values
   → Built-in: Serdes.String(), Serdes.Long(), Serdes.Integer()
   → JSON Serde for complex objects
   → Avro Serde with Schema Registry (connects to Project 16)

6. ADD GRACEFUL SHUTDOWN HOOK
   → Runtime.getRuntime().addShutdownHook(...)
   → Call streams.close() with timeout
   → Ensures state stores are flushed and tasks are cleanly reassigned

❌ WHAT SHOULD NOT DO

1. DO NOT TREAT KAFKA STREAMS AS A CLUSTER
   → It's a LIBRARY, not a service
   → No master node, no cluster manager
   → Each instance is an independent JVM process
   → Scaling = start more instances (like scaling consumer groups — Project 5)
   → Common misconception: "Where do I deploy the Kafka Streams cluster?"
   → Answer: There IS no cluster — deploy like any Java application

2. DO NOT USE KAFKA STREAMS FOR NON-KAFKA SOURCES
   → Input MUST come from Kafka topics
   → Output MUST go to Kafka topics
   → For database → Kafka: use Kafka Connect (Project 15, 23)
   → For HTTP → Kafka: use a producer (Project 12)
   → Kafka Streams is Kafka-to-Kafka ONLY

3. DO NOT IGNORE THE REPARTITIONING COST
   → Operations like groupBy() with a NEW key cause repartitioning
   → Data is reshuffled through an internal repartition topic
   → This adds latency and network I/O
   → Use groupByKey() when the key is already correct
   → Only use groupBy() when key change is necessary

4. DO NOT PUT EXTERNAL I/O IN STREAM PROCESSING
   → No HTTP calls inside map() or filter()
   → No database queries inside flatMap()
   → Blocks the processing thread → causes consumer timeout
   → Use Kafka Streams for pure data transformation
   → External enrichment: use a KTable join (Project 32) or pre-load data

5. DO NOT USE /tmp FOR STATE DIRECTORY IN PRODUCTION
   → Default state.dir = /tmp/kafka-streams
   → /tmp is cleaned on reboot → state lost → full rebuild needed
   → Use persistent SSD storage
   → State rebuild from changelog topics is correct but SLOW

6. DO NOT FORGET: ONE TASK PER INPUT PARTITION
   → If input topic has 6 partitions → max 6 parallel tasks
   → Starting 10 instances → 4 instances sit idle
   → Plan partition count based on desired parallelism
   → This directly connects to partition design (Project 3)

   → Data lives in PostgreSQL, MongoDB, S3, HTTP APIs
   → Kafka Streams cannot read from these directly
   → Use Kafka Connect (Project 15, 23) to bring data INTO Kafka first
   → Then process with Kafka Streams

3. BATCH PROCESSING
   → Kafka Streams is designed for continuous, real-time processing
   → Not suitable for "process yesterday's data" batch jobs
   → Use Spark or Flink for batch processing

4. VERY SIMPLE CONSUME-PRODUCE PATTERNS
   → If you just need: read → filter → write (no state, no aggregation)
   → Manual Consumer/Producer might be simpler
   → Kafka Streams shines when STATE is involved
   → For stateless transforms, either approach works

5. EXTREMELY HIGH THROUGHPUT (> 1M msg/sec per instance)
   → Kafka Streams runs in a single JVM
   → For extreme throughput, dedicated stream processing clusters
     (Flink, Spark) may be better
   → BUT: you can run many Kafka Streams instances horizontally

COMPONENT 1: THE TOPOLOGY — THE PROCESSING GRAPH
═══════════════════════════════════════════════════════════════════

  Every Kafka Streams application defines a TOPOLOGY:
  → A directed acyclic graph (DAG) of processing nodes
  → Data flows from source nodes through processor nodes to sink nodes

  For our word counter:

  ┌──────────────┐
  │ SOURCE NODE  │  ← Reads from "social-posts" topic
  │ (Consumer)   │
  └──────┬───────┘
         │ KStream<String, String>
         ▼
  ┌──────────────┐
  │ FLATMAPVALUES│  ← Splits "hello world" → ["hello", "world"]
  │              │
  └──────┬───────┘
         │ KStream<String, String> (one record per word)
         ▼
  ┌──────────────┐
  │ GROUPBY      │  ← Re-keys by word: (null,"hello") → ("hello","hello")
  │ (selectKey)  │     TRIGGERS REPARTITION (new internal topic)
  └──────┬───────┘
         │ KGroupedStream<String, String>
         ▼
  ┌──────────────┐
  │ COUNT        │  ← Maintains count per key in STATE STORE
  │ (aggregate)  │     State backed by RocksDB + changelog topic
  └──────┬───────┘
         │ KTable<String, Long>
         ▼
  ┌──────────────┐
  │ TOSTREAM     │  ← Converts KTable back to KStream for output
  │              │
  └──────┬───────┘
         │ KStream<String, Long>
         ▼
  ┌──────────────┐
  │ SINK NODE    │  ← Writes to "word-counts" topic
  │ (Producer)   │
  └──────────────┘

COMPONENT 2: STREAM TASKS — THE PARALLELISM UNIT
═══════════════════════════════════════════════════════════════════

  Number of tasks = max(input partitions across all source topics)

  "social-posts" has 6 partitions → 6 stream tasks created

  Each task:
  → Consumes from ONE partition of the input topic
  → Runs the COMPLETE topology for records from that partition
  → Has its OWN state store (isolated from other tasks)
  → Produces to output topic partitions

  ┌─────────────────────────────────────────────────────┐
  │ Task 0                                               │
  │ ┌─────────┐  ┌──────┐  ┌───────┐  ┌──────┐        │

  │ │(part 0) │  │      │  │      │  │State0│        │
  │ └─────────┘  └──────┘  └───────┘  └──────┘        │
  └─────────────────────────────────────────────────────┘
  ┌─────────────────────────────────────────────────────┐
  │ Task 1                                               │
  │ ┌─────────┐  ┌──────┐  ┌───────┐  ┌──────┐        │
  │ │Source   │→│FlatMap│→│GroupBy│→│Count │→ Sink    │

  └─────────────────────────────────────────────────────┘

COMPONENT 3: STATE STORES — PERSISTENT LOCAL STORAGE
═══════════════════════════════════════════════════════════════════

  When count() is called, Kafka Streams creates:

  1. LOCAL STATE STORE (RocksDB)
     → Embedded key-value database
     → Stored on local disk (state.dir)
     → Fast reads: O(1) lookups
     → Each task has its own isolated store
     
     Task 0 State Store:
     ┌─────────┬───────┐
     │ Key     │ Value │
     ├─────────┼───────┤
     │ "hello" │ 42    │
     │ "kafka" │ 15    │
     │ "world" │ 38    │
     └─────────┴───────┘

  2. CHANGELOG TOPIC (for fault tolerance)
     → Internal topic: {application.id}-{store-name}-changelog
     → Example: "word-counter-app-KSTREAM-AGGREGATE-STATE-STORE-0000000003-changelog"
     → Every state store update is written to this topic
     → Topic is COMPACTED (like Project 14) — only latest value per key
     → On crash recovery: state rebuilt by replaying changelog

  RECOVERY FLOW:
  
  Normal: Read → Process → Update RocksDB → Write to changelog
  
  Crash → Restart:
  ┌──────────────────────────────────────────────────────┐
  │ 1. Task assigned to (possibly different) instance    │
  │ 2. Local RocksDB is empty (or stale)                 │
  │ 3. Kafka Streams reads changelog topic from beginning│
  │ 4. Replays all updates into fresh RocksDB            │
  │ 5. State fully restored                              │
  │ 6. Resume processing from last committed offset      │
  └──────────────────────────────────────────────────────┘

COMPONENT 4: REPARTITIONING — WHY groupBy() IS EXPENSIVE
═══════════════════════════════════════════════════════════════════

  Input: "social-posts" partition 0 contains:
    → Key: "user-123", Value: "hello world kafka"
    → Key: "user-456", Value: "hello streams"

  After flatMapValues: 
    → ("user-123", "hello"), ("user-123", "world"), ("user-123", "kafka")
    → ("user-456", "hello"), ("user-456", "streams")

  Problem: We need to count by WORD, but key is still USER
  → "hello" appears in partition 0 (user-123) AND partition 0 (user-456)
  → But on different partitions, "hello" might be on partition 0 AND partition 3
  → count() needs ALL "hello" records in ONE task

  Solution: REPARTITION
  
  groupBy((key, word) -> word) triggers:
  
  ┌──────────────────────────────────────────────────────────┐
  │ 1. Kafka Streams creates INTERNAL repartition topic:     │
  │    "word-counter-app-KSTREAM-AGGREGATE-repartition"      │
  │                                                          │
  │ 2. Re-keys records: key changes from "user-123" to word  │
  │    ("user-123","hello") → ("hello","hello")              │
  │                                                          │
  │ 3. Produces re-keyed records to repartition topic        │
  │    hash("hello") % 6 = partition 2                       │
  │    hash("world") % 6 = partition 4                       │
  │                                                          │
  │ 4. New set of tasks reads from repartition topic          │
  │    Task for partition 2: processes ALL "hello" records    │
  │    Task for partition 4: processes ALL "world" records    │
  │                                                          │
  │ 5. count() now works correctly — all same-key records    │
  │    are in the same task with the same state store        │
  └──────────────────────────────────────────────────────────┘

  COST:
  → Extra topic (repartition topic) created automatically
  → Extra network hop (produce to repartition → consume from repartition)
  → Extra disk I/O
  → ~2x latency for the repartitioning step
  
  OPTIMIZATION:
  → If key is ALREADY the grouping key, use groupByKey()
  → groupByKey() does NOT create repartition topic
  → Only groupBy() with key change triggers repartition

═══════════════════════════════════════════════════════════════════
COMPONENT 5: INTERNAL TOPICS — THE HIDDEN INFRASTRUCTURE
═══════════════════════════════════════════════════════════════════

  Kafka Streams automatically creates several internal topics:

  For our word counter with application.id = "word-counter-app":

  1. REPARTITION TOPIC (if groupBy changes key):
     word-counter-app-KSTREAM-AGGREGATE-STATE-STORE-0000000003-repartition
     → Partitions: same as input topic
     → Cleanup policy: delete
     → Purpose: shuffle data by new key

  2. CHANGELOG TOPIC (for each state store):
     word-counter-app-KSTREAM-AGGREGATE-STATE-STORE-0000000003-changelog
     → Partitions: same as input topic
     → Cleanup policy: compact (like Project 14)
     → Purpose: backup state for fault tolerance

  These topics are managed by Kafka Streams:
  → Created automatically on first run
  → Partition count matches source topic
  → Replication factor configurable via replication.factor config
  → DO NOT delete these topics manually — breaks state recovery

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "WHERE IS MY KAFKA STREAMS CLUSTER?"
─────────────────────────────────────────────────
  Symptom: Trying to install/deploy Kafka Streams as a service
  Root Cause: Misunderstanding — Kafka Streams is a LIBRARY, not a service
  Fix: Add kafka-streams dependency to your Java project
       Run your application like any Java program
       Scale by running more instances of your application

PITFALL 2: STATE STORE RECOVERY TAKING HOURS
─────────────────────────────────────────────────
  Symptom: Application takes hours to start after crash
  Root Cause: Changelog topic has millions of records to replay
  Prevention:
    → Use standby replicas (num.standby.replicas > 0)
    → Standby keeps warm copy of state on another instance
    → Failover reads from warm copy instead of replaying changelog
    → Dramatically reduces recovery time

PITFALL 3: REPARTITION TOPIC EXPLOSION
─────────────────────────────────────────────────
  Symptom: Many internal topics created, disk usage spikes
  Root Cause: Multiple groupBy() operations each create repartition topics
  Fix: Minimize key changes
       Use groupByKey() when possible
       Chain operations efficiently

PITFALL 4: OUTPUT TOPIC NOT RECEIVING DATA
─────────────────────────────────────────────────
  Symptom: Word counts topic is empty
  Root Cause: KTable changes only produce when VALUE changes
    → count() produces a KTable
    → KTable emits update only when count changes
    → If using caching (default), updates are batched
    → May take up to commit.interval.ms (default 30s) to emit
  Fix: Set cache.max.bytes.buffering=0 for immediate output
       Or reduce commit.interval.ms for faster output

PITFALL 5: ClassCastException IN SERDES
─────────────────────────────────────────────────
  Symptom: Runtime ClassCastException when processing
  Root Cause: Default Serde doesn't match actual data type
    → count() produces Long values
    → If default value Serde is String → ClassCastException
  Fix: Set explicit Serdes in operations that change types:
       count(Materialized.with(Serdes.String(), Serdes.Long()))
       Or set default Serdes in configuration

PITFALL 6: FORGETTING TO START THE STREAMS
─────────────────────────────────────────────────
  Symptom: Application starts but nothing happens
  Root Cause: Created topology but didn't call streams.start()
  Fix: Always call kafkaStreams.start() after building topology
       And add shutdown hook for kafkaStreams.close()

        props.put(StreamsConfig.DEFAULT_KEY_SERDE_CLASS_CONFIG,

                  Serdes.String().getClass().getName());

        // Where RocksDB stores local state
        // Use persistent storage in production (NOT /tmp)
        props.put(StreamsConfig.STATE_DIR_CONFIG, "/tmp/kafka-streams-word-counter");

        // Lower = more frequent output but more overhead

        // For this demo: 1 second for faster visibility
        props.put(StreamsConfig.COMMIT_INTERVAL_MS_CONFIG, 1000);

        // Number of processing threads within this JVM

        return props;
    }

    // ============================================================
    // TOPOLOGY DEFINITION — THE HEART OF THE APPLICATION
    // ============================================================

    /**
     * Build the processing topology.
     *
     * TOPOLOGY FLOW:
     *   social-posts → flatMapValues → groupBy → count → toStream → word-counts
     *
     * WHAT EACH STEP DOES:
     *
     * 1. stream("social-posts")
     *    → Creates KStream<String, String> from input topic
     *    → Each record: key=user_id, value="hello world kafka streams"
     *
     * 2. flatMapValues(value -> split into words)
     *    → One input record → multiple output records
     *    → ("user1", "hello world") → ("user1","hello"), ("user1","world")
     *    → STATELESS operation — no state store needed
     *
     * 3. groupBy((key, word) -> word)
     *    → Re-keys the stream by WORD instead of user
     *    → ("user1", "hello") → ("hello", "hello")
     *    → TRIGGERS REPARTITION — internal topic created
     *    → Ensures all records with same word go to same task
     *
     * 4. count()
     *    → STATEFUL operation — creates state store (RocksDB)
     *    → Maintains running count per key (word)
     *    → Returns KTable<String, Long>
     *    → Every new record updates the count and emits change
     *
     * 5. toStream()
     *    → Converts KTable back to KStream
     *    → Emits (word, count) for every count change
     *
     * 6. to("word-counts")
     *    → Writes to output topic
     *    → Key: word (String), Value: count (Long)
     */
    private static Topology buildTopology() {
        StreamsBuilder builder = new StreamsBuilder();

        // ─── STEP 1: SOURCE — Read from input topic ───
        KStream<String, String> socialPosts = builder.stream(INPUT_TOPIC);

        // ─── STEP 2: SPLIT — One post → many words ───
        KStream<String, String> words = socialPosts
            .flatMapValues(
                // Split each post into individual words
                // "Hello World Kafka" → ["hello", "world", "kafka"]
                (readOnlyKey, value) -> {
                    if (value == null || value.trim().isEmpty()) {
                        return Arrays.asList();  // Skip empty messages
                    }
                    return Arrays.asList(
                        WORD_PATTERN.split(value.toLowerCase(Locale.ROOT))
                    );
                },
                Named.as("split-into-words")
            )
            // Filter out empty strings and very short words
            .filter(
                (key, word) -> word != null && word.length() > 2,
                Named.as("filter-short-words")
            );

        // ─── STEP 3: GROUP BY WORD ───
        // This changes the key from user_id to word
        // TRIGGERS REPARTITION: data shuffled by new key
        // All "hello" records go to the same task/partition
        KTable<String, Long> wordCounts = words
            .groupBy(
                // New key = the word itself
                // New value = the word itself (unchanged)
                (key, word) -> word,
                org.apache.kafka.streams.kstream.Grouped.with(
                    Serdes.String(),  // Key serde (word)
                    Serdes.String()   // Value serde (word)
                )
            )
            // ─── STEP 4: COUNT ───
            // Creates state store backed by RocksDB + changelog topic
            .count(
                Materialized.as("word-count-store")
                // Materialized names the state store
                // Store name used for:
                //   → RocksDB directory name
                //   → Changelog topic name
                //   → Interactive queries (Project 33)
            );

        // ─── STEP 5 & 6: OUTPUT — Write counts to output topic ───
        wordCounts
            .toStream(Named.as("counts-to-stream"))
            .peek(
                // Logging for observability
                (word, count) -> System.out.printf(
                    "  📊 Word: %-20s Count: %d%n", word, count
                ),
                Named.as("log-counts")
            )
            .to(
                OUTPUT_TOPIC,
                Produced.with(Serdes.String(), Serdes.Long())
                // MUST specify Serdes because count() produces Long
                // Default value Serde is String — would cause ClassCastException
            );

        return builder.build();
    }

"""
Word Count Output Consumer — Reads results from the Kafka Streams application.

The Kafka Streams app writes word counts to "word-counts" topic.
This consumer reads and displays the running counts.

The output topic has:
  → Key: word (String)
  → Value: count (Long, 8 bytes big-endian)

Builds on: Project 5 (consumer groups), Project 6 (offsets)
"""

import signal
import struct
from collections import defaultdict
from confluent_kafka import Consumer, KafkaError


def run_word_count_consumer():
    print('=' * 70)
    print('WORD COUNT OUTPUT CONSUMER')
    print('=' * 70)
    print('  Reading from: word-counts')
    print('  Key: word (String), Value: count (Long)')
    print('  Press Ctrl+C to stop and see summary')
    print()

    config = {
        'bootstrap.servers': 'localhost:9092',
        'group.id': 'word-count-display-group',
        'auto.offset.reset': 'earliest',
        'enable.auto.commit': True,
    }

    consumer = Consumer(config)
    consumer.subscribe(['word-counts'])

    # Track latest count per word
    word_counts = defaultdict(int)
    total_updates = 0

    running = True

    def signal_handler(sig, frame):
        nonlocal running
        running = False

    signal.signal(signal.SIGINT, signal_handler)

    empty_polls = 0
    max_empty_polls = 40

    try:
        while running:
            msg = consumer.poll(timeout=0.5)

            if msg is None:
                empty_polls += 1
                if empty_polls >= max_empty_polls:
                    print(f'\n  No new messages for {max_empty_polls * 0.5:.0f}s — stopping...')
                    break
                continue

            empty_polls = 0

            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    continue
                print(f'  ❌ Error: {msg.error()}')
                continue

            # ─── DESERIALIZE KEY (String) ───
            try:
                word = msg.key().decode('utf-8') if msg.key() else '(null)'
            except UnicodeDecodeError:
                word = '(binary-key)'

            # ─── DESERIALIZE VALUE (Long — 8 bytes big-endian) ───
            # Kafka Streams Serdes.Long() serializes as 8-byte big-endian
            # Python struct.unpack('>q', ...) reads 8-byte signed big-endian long
            try:
                raw_value = msg.value()
                if raw_value and len(raw_value) == 8:
                    count = struct.unpack('>q', raw_value)[0]
                elif raw_value:
                    # Try as string (in case custom serde used)
                    count = int(raw_value.decode('utf-8'))
                else:
                    count = 0
            except (struct.error, ValueError, UnicodeDecodeError):
                count = -1  # Indicates deserialization failure

            # ─── UPDATE TRACKING ───
            old_count = word_counts.get(word, 0)
            word_counts[word] = count
            total_updates += 1
            delta = count - old_count

            # ─── DISPLAY ───
            partition = msg.partition()
            offset = msg.offset()

            print(f'  [{total_updates:>6}] P{partition}|O{offset:<8} '
                  f'Word: {word:<25} '
                  f'Count: {count:<8} '
                  f'(+{delta})')

    finally:
        # ─── TOP WORDS SUMMARY ───
        print('\n' + '=' * 70)
        print('WORD COUNT SUMMARY — TOP 30 WORDS')
        print('=' * 70)

        sorted_words = sorted(word_counts.items(), key=lambda x: x[1], reverse=True)

        if not sorted_words:
            print('  No word counts received')
        else:
            max_count = sorted_words[0][1] if sorted_words else 1
            for rank, (word, count) in enumerate(sorted_words[:30], 1):
                bar_length = int((count / max_count) * 40) if max_count > 0 else 0
                bar = '█' * bar_length
                print(f'  {rank:>3}. {word:<25} {count:>6}  {bar}')

        print()
        print(f'  Total unique words:    {len(word_counts)}')
        print(f'  Total count updates:   {total_updates}')
        print(f'  Highest count word:    {sorted_words[0][0] if sorted_words else "N/A"} '
              f'({sorted_words[0][1] if sorted_words else 0})')
        print('=' * 70)

        consumer.close()
        print('  ✓ Consumer closed')


# ============================================================
# ENTRY POINT
# ============================================================

if __name__ == '__main__':
    run_word_count_consumer()

def check_consumer_group():
    """Check the Kafka Streams consumer group status"""
    print('📋 CONSUMER GROUP STATUS')
    print('-' * 70)

    admin = AdminClient({'bootstrap.servers': BOOTSTRAP_SERVERS})

    # List consumer groups
    try:
        groups_future = admin.list_consumer_groups()
        groups_result = groups_future.result()

        streams_groups = [
            g for g in groups_result.valid
            if APPLICATION_ID in g.group_id
        ]

        if not streams_groups:
            print(f'  ⚠️  No consumer group found for {APPLICATION_ID}')
            print('     → Kafka Streams application may not be running')
            return

        for group in streams_groups:
            print(f'  Group ID: {group.group_id}')
            print(f'  State:    {group.state}')

            # Describe group for member details
            desc_futures = admin.describe_consumer_groups([group.group_id])
            for gid, future in desc_futures.items():
                try:
                    desc = future.result()
                    print(f'  Members:  {len(desc.members)}')
                    print(f'  Protocol: {desc.partition_assignor}')
                    for member in desc.members:
                        print(f'    → Member: {member.client_id}')
                        if member.assignment:
                            partitions = member.assignment.topic_partitions
                            for tp in partitions:
                                print(f'       Assigned: {tp.topic} P{tp.partition}')
                except Exception as e:
                    print(f'  ⚠️  Could not describe group: {e}')

    except Exception as e:
        print(f'  ❌ Error listing groups: {e}')


def check_consumer_lag():
    """Check processing lag for the Kafka Streams application"""
    print('\n📋 CONSUMER LAG (Processing Backlog)')
    print('-' * 70)

    consumer = Consumer({
        'bootstrap.servers': BOOTSTRAP_SERVERS,
        'group.id': APPLICATION_ID,
        'enable.auto.commit': False,
    })

    try:
        # Get committed offsets
        topics_meta = consumer.list_topics('social-posts', timeout=10)
        if 'social-posts' not in topics_meta.topics:
            print('  ⚠️  Input topic social-posts not found')
            return

        partitions = topics_meta.topics['social-posts'].partitions
        topic_partitions = [TopicPartition('social-posts', p) for p in partitions]

        # Get committed offsets for the streams consumer group
        committed = consumer.committed(topic_partitions, timeout=10)

        # Get high watermarks (latest offsets)
        total_lag = 0
        for tp in committed:
            low, high = consumer.get_watermark_offsets(tp, timeout=10)
            committed_offset = tp.offset if tp.offset >= 0 else 0
            lag = high - committed_offset if committed_offset >= 0 else high

            total_lag += lag
            status = '✅' if lag == 0 else '⚠️' if lag < 100 else '❌'
            print(f'  {status} Partition {tp.partition}: '
                  f'committed={committed_offset:>8} '
                  f'latest={high:>8} '
                  f'lag={lag:>6}')

        print(f'\n  Total lag: {total_lag} messages')
        if total_lag == 0:
            print('  ✅ Fully caught up — no processing backlog')
        elif total_lag < 100:
            print('  ⚠️  Minor lag — likely being processed')
        else:
            print('  ❌ Significant lag — check application health')

    except Exception as e:
        print(f'  ❌ Error checking lag: {e}')
    finally:
        consumer.close()


def check_output_throughput():
    """Sample the output topic to verify data is flowing"""
    print('\n📋 OUTPUT TOPIC HEALTH CHECK')
    print('-' * 70)

    consumer = Consumer({
        'bootstrap.servers': BOOTSTRAP_SERVERS,
        'group.id': f'health-check-{int(time.time())}',
        'auto.offset.reset': 'latest',
        'enable.auto.commit': False,
    })
    consumer.subscribe(['word-counts'])

    message_count = 0
    start_time = time.time()
    check_duration = 5  # seconds

    print(f'  Monitoring word-counts topic for {check_duration} seconds...')

    while time.time() - start_time < check_duration:
        msg = consumer.poll(timeout=0.5)
        if msg is None:
            continue
        if msg.error():
            continue
        message_count += 1

    elapsed = time.time() - start_time
    rate = message_count / elapsed if elapsed > 0 else 0

    consumer.close()

    if message_count > 0:
        print(f'  ✅ Output topic is ACTIVE')
        print(f'     Messages in {elapsed:.1f}s: {message_count}')
        print(f'     Rate: {rate:.1f} messages/second')
    else:
        print(f'  ⚠️  No new messages in {check_duration} seconds')
        print(f'     → Kafka Streams may not be running')
        print(f'     → Or no new input data arriving')
