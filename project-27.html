PROJECT 27: THE GLOBAL NEWS SYNDICATION — MULTI-DATACENTER REPLICATION

🏗️ HYPOTHETICAL PROBLEM

SCENARIO: Global News Wire Service

→ A major news agency (like Reuters/AP) operates globally:
  - PRIMARY CLUSTER: US-East (Virginia) — main newsroom
  - SECONDARY CLUSTER: EU-West (Frankfurt) — European operations
  - TERTIARY CLUSTER: APAC (Singapore) — Asia-Pacific bureau

→ Current architecture:
  - 200+ journalists producing articles to US-East Kafka cluster
  - Topics: breaking_news, politics, finance, sports, technology
  - 50,000 articles/day flowing through the primary cluster
  - Consumers: news websites, mobile apps, TV tickers, partner APIs

→ THE PROBLEMS:

  PROBLEM 1: LATENCY
  ─────────────────
  → European subscribers consume from US-East cluster
  → Network round-trip: Frankfurt → Virginia = 85-120ms per request
  → News consumers poll every 100ms → 85ms latency per poll
  → For real-time breaking news: 85ms is UNACCEPTABLE
  → Competitor delivers breaking news 80ms faster
  → In financial news: 80ms = millions in trading advantage lost

  PROBLEM 2: DISASTER RECOVERY
  ────────────────────────────
  → Tuesday 02:30 UTC: AWS US-East-1 experiences regional outage
  → ALL Kafka brokers in primary cluster go offline
  → ZERO news articles flowing globally
  → Duration: 4 hours 23 minutes
  → 4+ hours of GLOBAL news blackout
  → Competitor (with multi-DC) continues serving
  → Agency loses 3 major media contracts worth $2.1M/year

  PROBLEM 3: DATA SOVEREIGNTY
  ──────────────────────────
  → GDPR requires European user data processed in EU
  → European journalist profiles stored in US-East Kafka
  → Regulatory audit discovers cross-border data flow violation
  → €2.4M fine threatened
  → Need EU-local copy of European-origin data

  PROBLEM 4: REGIONAL AUTONOMY
  ───────────────────────────
  → APAC bureau wants to produce articles locally
  → Currently: Singapore journalist → VPN → US-East → produce
  → Latency: 250ms per produce call
  → Large articles (with images): timeout failures
  → APAC bureau needs LOCAL produce capability

→ REQUIREMENT:
  1. Replicate topics from US-East to EU-West and APAC in near-real-time
  2. If US-East fails, EU-West can serve as primary within minutes
  3. European data stays in EU for GDPR compliance
  4. APAC can produce locally AND receive global content
  5. Consumer offset synchronization across clusters
  6. No custom replication code — use built-in Kafka tooling
  7. Topic-level control over what gets replicated where

🔍 HOW TO APPROACH IT

THINKING THROUGH THE PROBLEM:

1. From Project 11 → We know about replication WITHIN a cluster
   → Replication factor = copies across brokers in SAME cluster
   → But brokers in different datacenters = DIFFERENT clusters
   → Intra-cluster replication ≠ inter-cluster replication

2. From Project 8 → We know Kafka CLI tools
   → kafka-topics.sh works against ONE cluster
   → No built-in "replicate to another cluster" command

3. From Project 25-26 → We have error handling and retry
   → Cross-datacenter replication WILL face network issues
   → Need resilient replication that handles transient failures

4. THE FUNDAMENTAL CHALLENGE:
   → Two Kafka clusters are COMPLETELY INDEPENDENT
   → They share NOTHING: no ZooKeeper, no brokers, no metadata
   → Records in Cluster-A don't automatically appear in Cluster-B
   → We need a BRIDGE between clusters

WHAT IS MIRRORMAKER 2.0?
═════════════════════════
→ Built on top of Kafka Connect framework (Project 15)
→ Consumes from SOURCE cluster → produces to TARGET cluster
→ Handles: topic replication, offset sync, config sync, ACL sync
→ Replaced MirrorMaker 1.0 (deprecated — was just consumer+producer)
→ Part of Apache Kafka since version 2.4

MM2 IS ESSENTIALLY:
  Source Cluster                    Target Cluster
  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ Broker A  │───▶│MirrorMaker│──▶│ Broker X  │
  │ Broker B  │───▶│  2.0      │──▶│ Broker Y  │
  │ Broker C  │    │(Connect)  │    │ Broker Z  │
  └──────────┘    └──────────┘    └──────────┘

THREE REPLICATION ARCHITECTURES:

  A) ACTIVE-PASSIVE (Disaster Recovery)
     US-East (ACTIVE) ──────→ EU-West (PASSIVE/STANDBY)
     All writes go to US-East
     EU-West is read-only replica
     Failover: promote EU-West to active

  B) ACTIVE-ACTIVE (Multi-Region Write)
     US-East ←──────→ EU-West
     Both clusters accept writes
     Each replicates to the other
     COMPLEX: must handle conflicts and avoid infinite loops

  C) HUB-AND-SPOKE (Aggregation)
     US-East ──→ EU-West
     US-East ──→ APAC
     EU-West ──→ US-East (selected topics)
     Central hub receives from all spokes

KEY DESIGN DECISIONS:

  1. TOPIC RENAMING:
     → US-East "breaking_news" → EU-West "us-east.breaking_news"
     → Prefix with source cluster alias
     → Prevents naming collisions in active-active
     → Consumer knows which cluster data originated from

  2. SELECTIVE REPLICATION:
     → APAC only gets finance + technology (not sports/politics)
     → Reduces bandwidth: Singapore ↔ Virginia is expensive
     → Topic filtering via regex patterns

  3. BIDIRECTIONAL (EU-West → US-East):
     → EU-West has "eu_local_news" (European-only content)
     → Replicated to US-East as "eu-west.eu_local_news"
     → US-East aggregates ALL global content

  4. OFFSET SYNCHRONIZATION:
     → Consumer reading from US-East at offset 5000
     → If US-East fails, consumer must resume from EU-West
     → But EU-West offset for same data might be 4987
     → MM2 maintains offset translation mapping

✅ WHAT SHOULD DO

1. USE DEDICATED CLUSTER ALIASES
   → source: "us-east", target: "eu-west"
   → Aliases appear in replicated topic names
   → "us-east.breaking_news" clearly shows origin
   → Avoid generic names like "cluster1" or "primary"

2. START WITH ACTIVE-PASSIVE BEFORE ACTIVE-ACTIVE
   → Active-passive is simpler: one-directional replication
   → Prove it works, measure lag, test failover
   → Then add bidirectional for active-active if needed
   → Active-active has conflict resolution complexity

3. USE TOPIC WHITELISTS (NOT BLACKLISTS)
   → Explicitly list which topics to replicate
   → "topics = breaking_news, politics, finance"
   → NOT "topics.exclude = internal_metrics"
   → New internal topics won't accidentally replicate

4. MONITOR REPLICATION LAG
   → MM2 produces checkpoint records with offset mappings
   → Lag = source high watermark - last replicated offset
   → Alert if lag > threshold (e.g., > 1000 records or > 5 seconds)
   → Lag is THE critical metric for DR readiness

5. TEST FAILOVER REGULARLY
   → Don't wait for actual disaster to test failover
   → Monthly: simulate primary failure, switch to secondary
   → Measure: time to detect failure + time to switch
   → Target: < 5 minutes total failover time

6. SET sync.topic.configs.enabled = true
   → Replicate topic configurations (partitions, retention, etc.)
   → Target topic matches source topic settings
   → Prevents configuration drift between clusters

7. USE HEARTBEATS FOR CONNECTIVITY MONITORING
   → MM2 produces heartbeat records to target cluster
   → Heartbeat topic: "us-east.heartbeats"
   → If heartbeats stop → replication is broken → alert

8. CONFIGURE emit.checkpoints.enabled = true
   → Checkpoints map source offsets to target offsets
   → Required for consumer failover offset translation
   → Without checkpoints: consumers must start from earliest/latest

❌ WHAT SHOULD NOT DO

1. DO NOT REPLICATE INTERNAL KAFKA TOPICS
   → __consumer_offsets, __transaction_state, connect-*
   → These are cluster-internal — replicating causes corruption
   → Use topics.exclude = ".*[\-\.]internal, __.*, connect-.*"

2. DO NOT USE SAME GROUP ID ACROSS CLUSTERS
   → Consumer group "news-reader" on US-East
   → Consumer group "news-reader" on EU-West
   → These are INDEPENDENT groups with INDEPENDENT offsets
   → Don't confuse them — use cluster-prefixed group IDs

3. DO NOT ASSUME OFFSETS ARE THE SAME
   → Source topic offset 5000 ≠ target topic offset 5000
   → Replication may skip, batch, or reorder slightly
   → ALWAYS use MM2 offset translation for failover
   → NEVER hardcode offsets across clusters

4. DO NOT CREATE REPLICATION LOOPS
   → US-East → EU-West (replicate "breaking_news")
   → EU-West → US-East (replicate "us-east.breaking_news")
   → US-East now has "eu-west.us-east.breaking_news"
   → INFINITE PREFIX ACCUMULATION
   → MM2 prevents this with source cluster detection
   → But VERIFY your topic filters exclude remote-origin topics

5. DO NOT REPLICATE WITHOUT BANDWIDTH PLANNING
   → 50,000 articles/day × 10KB average = 500MB/day
   → Cross-region bandwidth: Virginia ↔ Frankfurt = ~$0.02/GB
   → 500MB/day = ~$0.30/day = ~$9/month (acceptable)
   → But: 50GB/day = $30/month, 5TB/day = $3000/month
   → CALCULATE before enabling replication

6. DO NOT FORGET CONSUMER MIGRATION PLANNING
   → Failover isn't just "start consuming from target"
   → Consumers need: new bootstrap servers, translated offsets
   → Plan and document the consumer switchover procedure
   → Automate it — manual failover at 2AM is error-prone

7. DO NOT RUN MM2 ON THE SAME BROKERS
   → MM2 is a Kafka Connect cluster — needs its own resources
   → Running on broker nodes: competes for CPU, memory, network
   → Dedicated MM2 nodes (or containers) for isolation

🚫 WHEN NOT TO USE

1. WHEN CLUSTERS ARE IN THE SAME DATACENTER
   → Use intra-cluster replication factor instead
   → Replication factor = 3 across 3 brokers in same DC
   → MM2 adds unnecessary complexity for same-DC redundancy

2. WHEN DATA VOLUME EXCEEDS BANDWIDTH BUDGET
   → 10TB/day cross-region = significant cost
   → Consider: do ALL topics need replication?
   → Selective replication reduces bandwidth
   → Or: replicate only aggregated/summary topics

3. WHEN SUB-SECOND FAILOVER IS REQUIRED
   → MM2 replication has inherent lag (seconds to minutes)
   → Failover requires: detect failure + switch consumers
   → Total failover: typically 1-10 minutes
   → For sub-second: need synchronous cross-DC writes (not Kafka's strength)

4. WHEN SOURCE AND TARGET HAVE INCOMPATIBLE VERSIONS
   → MM2 uses Kafka Connect — requires compatible APIs
   → Source: Kafka 0.10, Target: Kafka 3.0 → may have issues
   → Upgrade source cluster first, then set up MM2

5. WHEN ONLY SPECIFIC RECORDS (NOT TOPICS) NEED REPLICATION
   → MM2 replicates at TOPIC level, not record level
   → Cannot filter: "only records where region=EU"
   → For record-level filtering: use Kafka Streams (Project 20)
   → to filter into a separate topic, then replicate that topic

MIRRORSOURCECONNECTOR — RECORD REPLICATION FLOW
═══════════════════════════════════════════════════════════

  SOURCE CLUSTER (us-east)         TARGET CLUSTER (eu-west)
  
  Topic: breaking_news             Topic: us-east.breaking_news
  Partition 0: [A B C D E]         Partition 0: [A B C D E]
  Partition 1: [F G H I J]         Partition 1: [F G H I J]
  Partition 2: [K L M N O]         Partition 2: [K L M N O]
  
  INTERNAL STEPS:
  
  STEP 1: MirrorSourceConnector starts as Kafka Connect source connector
          → Connects to SOURCE cluster as a consumer
          → Consumer group: "mm2-MirrorSourceConnector-us-east-eu-west"
  
  STEP 2: Discovers topics matching configured pattern
          → topics = "breaking_news|politics|finance|sports|technology"
          → Regex matching against source cluster topic list
  
  STEP 3: For each matching topic, creates MIRROR topic on target
          → Source: "breaking_news"
          → Target: "us-east.breaking_news"
          → Naming: {source.cluster.alias}.{original.topic.name}
          → SAME partition count as source
  
  STEP 4: Consumes records from source topic
          → Standard Kafka consumer internally
          → Respects partition assignment
          → Tracks offsets per partition
  
  STEP 5: Produces records to target topic
          → Standard Kafka producer internally
          → PRESERVES: key, value, headers, timestamp
          → PRESERVES: partition assignment (same partition number)
          → ADDS: header "source.cluster" = "us-east"
  
  STEP 6: Commits source consumer offsets after target produce ack
          → At-least-once delivery guarantee
          → May produce duplicates during failure recovery
          → (Exactly-once possible with transactional config)


═══════════════════════════════════════════════════════════
TOPIC RENAMING — HOW ALIAS PREFIX WORKS
═══════════════════════════════════════════════════════════

  Configuration:
    source.cluster.alias = us-east
    target.cluster.alias = eu-west
  
  RENAMING RULE:
    Target topic name = {source.cluster.alias}.{source.topic.name}
  
  EXAMPLES:
    us-east:breaking_news    → eu-west:us-east.breaking_news
    us-east:finance          → eu-west:us-east.finance
    us-east:sports           → eu-west:us-east.sports
  
  WHY RENAME?
  → In active-active: both clusters have "breaking_news"
  → US-East's "breaking_news" = local content
  → EU-West's "breaking_news" = local content
  → EU-West's "us-east.breaking_news" = replicated from US-East
  → NO COLLISION — clear origin tracking
  
  LOOP PREVENTION:
  → EU-West has topic "us-east.breaking_news"
  → If EU-West → US-East replication is configured
  → Would it replicate "us-east.breaking_news" back to US-East?
  → NO — MM2 detects the "us-east." prefix
  → Recognizes it as originating FROM us-east
  → SKIPS replication (prevents infinite loop)
  
  DETECTION MECHANISM:
    topic.startsWith(source.cluster.alias + ".") → SKIP
    This is why cluster aliases must be UNIQUE and CONSISTENT

OFFSET TRANSLATION — HOW CHECKPOINT WORKS
═══════════════════════════════════════════════════════════

  PROBLEM:
    Source topic "breaking_news" partition 0:
      Offset 0: Record A
      Offset 1: Record B
      Offset 2: Record C    ← Consumer committed here
      Offset 3: Record D
      Offset 4: Record E
    
    Target topic "us-east.breaking_news" partition 0:
      Offset 0: Record A
      Offset 1: Record B
      Offset 2: Record C    ← Might be offset 2 OR different
      Offset 3: Record D
      Offset 4: Record E
    
    Usually offsets align 1:1 for simple replication
    BUT: if MM2 was restarted, or records were filtered,
    or target had pre-existing data → offsets DIVERGE
  
  SOLUTION: MirrorCheckpointConnector
  
  STEP 1: Periodically (every emit.checkpoints.interval.seconds):
          Read source consumer group offsets
          Read target topic high watermarks
          Calculate mapping
  
  STEP 2: Produce checkpoint record to target cluster:
          Topic: us-east.checkpoints.internal
          Key: {consumer_group}:{topic}:{partition}
          Value: {
            "source.offset": 5000,
            "target.offset": 4987,
            "consumer.group": "news-reader",
            "topic": "breaking_news",
            "partition": 0,
            "timestamp": 1705277800000
          }
  
  STEP 3: During failover, consumer reads checkpoint:
          "I was at offset 5000 on us-east:breaking_news"
          "Checkpoint says: source 5000 = target 4987"
          "I should start at offset 4987 on eu-west:us-east.breaking_news"


═══════════════════════════════════════════════════════════
HEARTBEAT MECHANISM
═══════════════════════════════════════════════════════════

  MirrorHeartbeatConnector produces to target cluster:
  
  Topic: heartbeats
  Frequency: every heartbeat.interval.ms (default 1000ms)
  
  Record:
  {
    "sourceCluster": "us-east",
    "targetCluster": "eu-west",
    "timestamp": 1705277800000
  }
  
  MONITORING:
  → Consumer on target cluster reads "heartbeats" topic
  → If no heartbeat received for > 10 seconds:
    → Replication link is DOWN
    → ALERT: "us-east → eu-west replication broken"
  
  → Heartbeat latency = target_receive_time - source_timestamp
  → Measures end-to-end replication delay
  → Normal: 50-500ms cross-region
  → Concerning: > 5 seconds
  → Critical: > 30 seconds or no heartbeat

REPLICATION MODES — INTERNAL DIFFERENCES
═══════════════════════════════════════════════════════════

  ACTIVE-PASSIVE:
  ┌──────────┐         ┌──────────┐
  │ US-EAST  │────MM2──▶│ EU-WEST  │
  │ (ACTIVE) │         │ (PASSIVE)│
  │ Read+Write│         │ Read-only│
  └──────────┘         └──────────┘
  
  → ONE MM2 instance
  → Unidirectional flow
  → Target is read-only (consumers only)
  → Failover: stop MM2, redirect producers to EU-West
  → Simple, reliable, well-understood

  ACTIVE-ACTIVE:
  ┌──────────┐  MM2-A  ┌──────────┐
  │ US-EAST  │────────▶│ EU-WEST  │
  │          │◀────────│          │
  │ Read+Write│  MM2-B  │ Read+Write│
  └──────────┘         └──────────┘
  
  → TWO MM2 instances (one per direction)
  → Bidirectional flow
  → Both clusters accept writes
  → EACH cluster has:
    - Local topics (e.g., "breaking_news")
    - Remote topics (e.g., "us-east.breaking_news")
  → Consumers must read BOTH local + remote topics
  → Loop prevention via alias prefix detection
  
  CONFLICT SCENARIO:
    US-East: key="article_123" → value="Version A"
    EU-West: key="article_123" → value="Version B"
    (written at same time to different clusters)
    
    After replication:
    US-East has: breaking_news:article_123 = "Version A"
                 eu-west.breaking_news:article_123 = "Version B"
    EU-West has: breaking_news:article_123 = "Version B"
                 us-east.breaking_news:article_123 = "Version A"
    
    → BOTH versions exist in BOTH clusters
    → Application must decide which to use (last-write-wins, merge, etc.)
    → Kafka does NOT resolve conflicts — application responsibility

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: TOPIC EXPLOSION FROM INTERNAL TOPICS
════════════════════════════════════════════════
  → Default topic filter: topics = ".*" (ALL topics)
  → Replicates: __consumer_offsets, connect-configs, etc.
  → Target cluster gets corrupted internal state
  → IMMEDIATE cluster instability
  
  FIX: ALWAYS set explicit topic filter:
       topics = "breaking_news|politics|finance|sports|technology"
       OR: topics.exclude = ".*[\-\.]internal|__.*|connect-.*|mm2-.*"


PITFALL 2: CONSUMER READING WRONG TOPIC AFTER FAILOVER
═══════════════════════════════════════════════════════
  → Consumer on US-East reads: "breaking_news"

════════════════════════════════════

  → Records land in different partitions on target
  → Consumer ordering assumptions violated

       Or: pre-create target topics with matching partition count


PITFALL 4: REPLICATION LAG DURING PEAK TRAFFIC
═══════════════════════════════════════════════
  → Normal: 200ms replication lag
  → During breaking news event: 50x traffic spike
  → MM2 consumer falls behind: lag grows to 30 seconds
  → If failover happens during spike: 30 seconds of data loss
  
  FIX: Scale MM2 tasks: tasks.max = partition_count
       Increase producer batch.size and linger.ms for throughput
       Monitor lag continuously (not just during tests)
       Size MM2 for PEAK, not average


PITFALL 5: NETWORK PARTITION BETWEEN DATACENTERS
═════════════════════════════════════════════════
  → Cross-DC network link goes down for 2 minutes
  → MM2 consumer disconnects from source cluster
  → Reconnects after 2 minutes
  → Must re-consume from last committed offset
  → If offset retention expired: DATA GAP
  
  FIX: Set long offset retention on source cluster
       offsets.retention.minutes = 10080 (7 days)
       MM2 can resume from committed offset after long outage


PITFALL 6: CHECKPOINT LAG IN OFFSET TRANSLATION
════════════════════════════════════════════════

  → Consumer on source at offset 5000
  → Last checkpoint maps: source 4940 → target 4927
  → Failover happens NOW
  → Consumer starts at target offset 4927
  → REPROCESSES 73 records (5000 - 4927)
  → Duplicates in downstream systems
  
  FIX: Reduce emit.checkpoints.interval.seconds (e.g., 10s)
       Accept that some reprocessing will happen during failover
       Ensure downstream is IDEMPOTENT (Project 17)

📈 SCALABILITY CONSIDERATIONS

BANDWIDTH CALCULATION:
══════════════════════
  Formula: bandwidth = records_per_second × average_record_size
  
  Example:
  → 50,000 articles/day = ~0.58 articles/second
  → Average article: 15KB (text + metadata)
  → Bandwidth: 0.58 × 15KB = 8.7KB/s = ~750MB/day
  → Cross-region cost (~$0.02/GB): $0.015/day = $0.45/month
  → VERY AFFORDABLE
  
  High-volume example:
  → 10 million events/day = ~116 events/second
  → Average event: 1KB
  → Bandwidth: 116KB/s = ~10GB/day
  → Cross-region cost: $0.20/day = $6/month
  → STILL AFFORDABLE

MM2 TASK SCALING:
═════════════════
  → tasks.max controls parallelism
  → Each task handles a subset of partitions
  → Optimal: tasks.max = total source partitions across all topics
  → Example: 5 topics × 6 partitions = 30 → tasks.max = 30
  → More tasks than partitions = wasted resources

MULTI-CLUSTER SCALING:
══════════════════════
  → 3 clusters: need 2-6 MM2 instances depending on topology
  → Active-passive (3 clusters): 2 MM2 instances (hub to each spoke)
  → Active-active (3 clusters): 6 MM2 instances (each pair bidirectional)
  → COMPLEXITY grows quadratically with cluster count
  → Beyond 5 clusters: consider hub-and-spoke to limit connections

TARGET CLUSTER SIZING:
══════════════════════
  → Target must handle: local traffic + replicated traffic
  → If source has 10 partitions, target needs ≥ 10 partitions
  → Target broker count: plan for replicated data volume
  → Storage: source retention × replication = target storage needed

def get_topic_stats(bootstrap, topic):
    """Get message count and latest timestamp for a topic"""
    admin = AdminClient({'bootstrap.servers': bootstrap})
    metadata = admin.list_topics(timeout=10)
    
    if topic not in metadata.topics:
        return {'exists': False, 'count': 0, 'partitions': 0}
    
    consumer = Consumer({
        'bootstrap.servers': bootstrap,
        'group.id': f'verifier-{int(time.time()*1000)}',
        'auto.offset.reset': 'latest'
    })
    
    total = 0
    partition_count = len(metadata.topics[topic].partitions)
    
    for pid in metadata.topics[topic].partitions:
        tp = TopicPartition(topic, pid)
        low, high = consumer.get_watermark_offsets(tp, timeout=5)
        total += (high - low)
    
    consumer.close()
    
    return {
        'exists': True,
        'count': total,
        'partitions': partition_count
    }

def sample_records(bootstrap, topic, max_records=3):
    """Read a few records from a topic for comparison"""
    consumer = Consumer({
        'bootstrap.servers': bootstrap,
        'group.id': f'sampler-{int(time.time()*1000)}',
        'auto.offset.reset': 'earliest'
    })
    
    consumer.subscribe([topic])
    records = []
    empty = 0
    
    while len(records) < max_records and empty < 10:
        msg = consumer.poll(1.0)
        if msg is None:
            empty += 1
            continue
        if msg.error():
            if msg.error().code() == KafkaError._PARTITION_EOF:
                empty += 1
                continue
            continue
        
        empty = 0
        try:
            value = json.loads(msg.value().decode('utf-8'))
            records.append({
                'key': msg.key().decode('utf-8') if msg.key() else None,
                'partition': msg.partition(),
                'offset': msg.offset(),
                'article_id': value.get('article_id', 'N/A'),
                'headline': value.get('headline', 'N/A')[:50],

            })

            records.append({
                'key': msg.key().decode('utf-8') if msg.key() else None,
                'partition': msg.partition(),
                'offset': msg.offset(),
                'raw_size': len(msg.value()) if msg.value() else 0
            })
    
    consumer.close()
    return records

from confluent_kafka import Consumer, TopicPartition

def get_watermark(bootstrap, topic, partition):
    """Get low and high watermark for a partition"""
    c = Consumer({
        'bootstrap.servers': bootstrap,
        'group.id': f'wm-probe-{int(time.time()*1000)}',
        'auto.offset.reset': 'latest'
    })
    try:
        tp = TopicPartition(topic, partition)
        low, high = c.get_watermark_offsets(tp, timeout=5)
        return low, high
    except Exception:
        return 0, 0
    finally:
        c.close()

def get_topic_high_watermarks(bootstrap, topic):
    """Get high watermark for all partitions of a topic"""
    admin = AdminClient({'bootstrap.servers': bootstrap})
    metadata = admin.list_topics(topic, timeout=5)
    
    if topic not in metadata.topics:
        return {}
    
    result = {}
    c = Consumer({
        'bootstrap.servers': bootstrap,
        'group.id': f'hw-probe-{int(time.time()*1000)}',
        'auto.offset.reset': 'latest'
    })
    
    try:
        for pid in metadata.topics[topic].partitions:
            tp = TopicPartition(topic, pid)
            _, high = c.get_watermark_offsets(tp, timeout=5)
            result[pid] = high
    except Exception:
        pass
    finally:
        c.close()
    
    return result

class ReplicationMonitor:
    
    def __init__(self):
        self.history = []
        self.alerts = []
    
    def check_replication_lag(self):
        """Calculate per-topic per-partition replication lag"""
        lag_report = {}
        total_source = 0
        total_target = 0
        
        for source_topic in SOURCE_TOPICS:
            target_topic = f"{SOURCE_ALIAS}.{source_topic}"
            
            source_hw = get_topic_high_watermarks(US_EAST, source_topic)
            target_hw = get_topic_high_watermarks(EU_WEST, target_topic)
            
            topic_lag = 0
            partition_lags = {}
            
            for pid, source_high in source_hw.items():
                target_high = target_hw.get(pid, 0)
                lag = max(0, source_high - target_high)
                partition_lags[pid] = lag
                topic_lag += lag
                total_source += source_high
                total_target += target_high
            
            lag_report[source_topic] = {
                'total_lag': topic_lag,
                'partition_lags': partition_lags,
                'source_total': sum(source_hw.values()),
                'target_total': sum(target_hw.values())
            }
        
        overall_lag = total_source - total_target
        
        return lag_report, overall_lag
    
    def check_heartbeat(self):
        """Check latest heartbeat age"""
        admin = AdminClient({'bootstrap.servers': EU_WEST})
        metadata = admin.list_topics(timeout=5)
        
        if 'heartbeats' not in metadata.topics:
            return {'status': 'NO_TOPIC', 'age_seconds': -1}
        
        c = Consumer({
            'bootstrap.servers': EU_WEST,
            'group.id': f'hb-mon-{int(time.time()*1000)}',
            'auto.offset.reset': 'latest'
        })
        
        # Seek to end minus 1
        partitions = []
        for pid in metadata.topics['heartbeats'].partitions:
            tp = TopicPartition('heartbeats', pid)
            _, high = c.get_watermark_offsets(tp, timeout=5)
            if high > 0:
                tp = TopicPartition('heartbeats', pid, max(0, high - 1))
                partitions.append(tp)
        
        latest_ts = None
        
        if partitions:
            c.assign(partitions)
            
            for _ in range(5):
                msg = c.poll(1.0)
                if msg and not msg.error():
                    ts = msg.timestamp()
                    if ts[1] and (latest_ts is None or ts[1] > latest_ts):
                        latest_ts = ts[1]
        
        c.close()
        
        if latest_ts:
            age_seconds = (int(time.time() * 1000) - latest_ts) / 1000.0
            
            if age_seconds < ALERT_THRESHOLDS['heartbeat_stale_seconds']:
                status = 'HEALTHY'
            elif age_seconds < ALERT_THRESHOLDS['heartbeat_dead_seconds']:
                status = 'STALE'
            else:
                status = 'DEAD'
            
            return {'status': status, 'age_seconds': round(age_seconds, 1)}
        
        return {'status': 'NO_DATA', 'age_seconds': -1}
    
    def fire_alert(self, severity, message):
        """Record and display alert"""
        alert = {
            'time': time.strftime('%H:%M:%S'),
            'severity': severity,
            'message': message
        }
        self.alerts.append(alert)
        
        icon = '🔴' if severity == 'CRITICAL' else '🟡'
        print(f"\n  {icon} ALERT [{severity}]: {message}")
    
    def run_check(self):
        """Run one monitoring cycle"""
        timestamp = time.strftime('%H:%M:%S')
        
        # Check lag
        lag_report, overall_lag = self.check_replication_lag()
        
        # Check heartbeat
        hb = self.check_heartbeat()
        
        # Display
        hb_icon = '✅' if hb['status'] == 'HEALTHY' else '⚠️' if hb['status'] == 'STALE' else '❌'
        
        print(f"\n  ─── {timestamp} ─── Heartbeat: {hb_icon} {hb['status']} "
              f"({hb['age_seconds']}s) ── Overall lag: {overall_lag} records ───")
        
        print(f"  {'Topic':<22} {'Lag':<8} {'Visual'}")
        
        for topic, report in lag_report.items():
            lag = report['total_lag']
            
            if lag == 0:
                bar = '✅ synced'
            elif lag < ALERT_THRESHOLDS['lag_records_warning']:
                bar = '▓' * min(lag, 30) + f' ({lag})'
            elif lag < ALERT_THRESHOLDS['lag_records_critical']:
                bar = '█' * min(lag // 10, 30) + f' ⚠️ ({lag})'
            else:
                bar = '█' * 30 + f' 🔴 ({lag})'
            
            print(f"  {topic:<22} {lag:<8} {bar}")
        
        # Alerts
        if overall_lag > ALERT_THRESHOLDS['lag_records_critical']:
            self.fire_alert('CRITICAL', f"Overall replication lag: {overall_lag} records")
        elif overall_lag > ALERT_THRESHOLDS['lag_records_warning']:
            self.fire_alert('WARNING', f"Overall replication lag: {overall_lag} records")
        
        if hb['status'] == 'DEAD':
            self.fire_alert('CRITICAL', f"Heartbeat DEAD — replication link may be broken")
        elif hb['status'] == 'STALE':
            self.fire_alert('WARNING', f"Heartbeat stale ({hb['age_seconds']}s)")
        
        # Record history
        self.history.append({
            'time': timestamp,
            'overall_lag': overall_lag,
            'heartbeat': hb['status'],
            'per_topic': {t: r['total_lag'] for t, r in lag_report.items()}
        })
    
    def run(self, duration_seconds=120, interval_seconds=15):
        """Run continuous monitoring"""
        
        print("=" * 70)
        print("  📡 REPLICATION HEALTH MONITOR")
        print("=" * 70)
        print(f"  Source: US-East ({US_EAST})")
        print(f"  Target: EU-West ({EU_WEST})")
        print(f"  Topics: {', '.join(SOURCE_TOPICS)}")
        print(f"  Interval: {interval_seconds}s | Duration: {duration_seconds}s")
        print(f"  Lag warning: {ALERT_THRESHOLDS['lag_records_warning']} records")
        print(f"  Lag critical: {ALERT_THRESHOLDS['lag_records_critical']} records")
        print("=" * 70)
        
        start = time.time()
        
        try:
            while time.time() - start < duration_seconds:
                self.run_check()
                
                remaining = duration_seconds - (time.time() - start)
                sleep_time = min(interval_seconds, max(0, remaining))
                if sleep_time > 0:
                    time.sleep(sleep_time)
        
        except KeyboardInterrupt:
            print("\n  Monitor stopped.")
        
        # Summary
        print(f"\n{'=' * 70}")
        print(f"  MONITORING SESSION SUMMARY")
        print(f"{'=' * 70}")
        print(f"  Duration: {int(time.time() - start)}s")
        print(f"  Checks: {len(self.history)}")
        print(f"  Alerts: {len(self.alerts)}")
        
        if self.history:
            lags = [h['overall_lag'] for h in self.history]
            print(f"  Lag — min: {min(lags)}, max: {max(lags)}, "
                  f"avg: {sum(lags)/len(lags):.0f}")
        
        if self.alerts:
            print(f"\n  Alert history:")
            for a in self.alerts[-10:]:
                icon = '🔴' if a['severity'] == 'CRITICAL' else '🟡'
                print(f"    {icon} [{a['time']}] {a['message']}")
        
        print(f"{'=' * 70}")


if __name__ == '__main__':
    monitor = ReplicationMonitor()
    monitor.run(duration_seconds=120, interval_seconds=15)
