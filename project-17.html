PROJECT 17: THE EXACTLY-ONCE ATM MACHINE — IDEMPOTENT & TRANSACTIONAL PRODUCER

🏦 HYPOTHETICAL PROBLEM

A national bank operates 50,000 ATM machines across the country

Every withdrawal, deposit, and balance inquiry generates a transaction event
→ These events are published to Kafka for downstream processing
→ Downstream systems include: ledger update, fraud detection, receipt generation

THE CRISIS:
→ Network glitch between ATM and Kafka broker causes producer retry
→ A customer withdraws $500 ONCE
→ The retry causes the message to be produced TWICE
→ The ledger system debits the customer $1,000 instead of $500
→ 200 customers affected in one hour
→ Bank loses $1.2 million in incorrect debits
→ Regulatory audit triggered

ROOT CAUSE ANALYSIS:
→ Producer had acks=all (from Project 11) — durability was guaranteed
→ BUT the broker acknowledged AFTER writing, network dropped BEFORE ack reached producer
→ Producer assumed failure, retried the SAME message
→ Broker wrote it AGAIN — now TWO identical records exist
→ No mechanism existed to detect "I already wrote this exact message"

THE QUESTION:
→ How do we guarantee that even if a producer retries 100 times
→ The broker writes the message EXACTLY ONCE?

🔍 HOW TO APPROACH IT

Step 1: Understand why acks=all from Project 11 is NOT enough
         → acks=all guarantees the broker RECEIVED and REPLICATED
         → It does NOT guarantee the producer KNOWS it was received
         → The gap between "broker wrote" and "producer got ack" is the danger zone

Step 2: Recognize the retry problem
         → retries > 0 means producer will resend on failure
         → Without deduplication, resend = duplicate
         → This is the "at-least-once" problem we identified in Project 11

Step 3: Identify what's needed
         → Broker must recognize "I've seen this exact message before"
         → Broker must silently acknowledge duplicates WITHOUT writing again
         → This requires IDENTITY — each message needs a unique fingerprint

Step 4: Learn that Kafka solved this at the protocol level
         → Idempotent Producer — introduced in Kafka 0.11
         → Producer ID (PID) + Sequence Number = unique message identity
         → Broker tracks sequences per partition per producer
         → Duplicate detected → ack returned → no second write

Step 5: Understand that idempotency covers SINGLE partition
         → What about atomic writes across MULTIPLE partitions?
         → That's where Transactions come in (Project 18)
         → This project focuses on IDEMPOTENT PRODUCER

🛠️ HOW CAN WE SOLVE IT

SOLUTION: Enable Kafka's Idempotent Producer

The idempotent producer adds a protocol-level deduplication layer:

┌──────────────────────────────────────────────────────────┐
│                  WITHOUT IDEMPOTENCE                      │
│                                                          │
│  ATM ──→ [Produce $500 withdrawal] ──→ Broker WRITES    │
│                                         ↓                │
│                              Network drops ACK           │
│                                         ↓                │
│  ATM ──→ [Retry $500 withdrawal] ──→ Broker WRITES AGAIN│
│                                                          │
│  Result: TWO records of $500 withdrawal = $1000 debited  │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│                   WITH IDEMPOTENCE                        │
│                                                          │
│  ATM ──→ [PID=7, SEQ=42, $500 withdrawal] ──→ Broker    │
│           Broker: "PID=7, SEQ=42 → NEW → WRITE"         │
│                                         ↓                │
│                              Network drops ACK           │
│                                         ↓                │
│  ATM ──→ [PID=7, SEQ=42, $500 withdrawal] ──→ Broker    │
│           Broker: "PID=7, SEQ=42 → ALREADY SEEN → ACK"  │
│                                                          │
│  Result: ONE record of $500 withdrawal = $500 debited ✓  │
└──────────────────────────────────────────────────────────┘

✅ WHAT COULD BE DONE

Option A: Application-Level Deduplication (Pre-Kafka 0.11 approach)
  → Assign UUID to each message at application level
  → Consumer checks a database for duplicate UUIDs
  → Discard if already processed
  → PROBLEM: Complex, error-prone, adds latency, requires external state store

Option B: Database Unique Constraints
  → Consumer writes to DB with unique transaction ID
  → DB rejects duplicates via unique constraint
  → PROBLEM: Only works for DB sinks, not for stream processing pipelines

Option C: Kafka Idempotent Producer ← CORRECT SOLUTION
  → Single configuration flag: enable.idempotence=true
  → Zero application code changes
  → Deduplication happens at the BROKER level
  → Works for ALL downstream consumers automatically
  → No external systems needed

Option D: Full Kafka Transactions (Project 18)
  → Overkill for single-topic single-partition deduplication
  → Needed when atomic multi-partition writes are required
  → Builds ON TOP of idempotence

✅ WHAT SHOULD DO

1. ENABLE IDEMPOTENCE EXPLICITLY
   → Set enable.idempotence=true
   → Even though Kafka 3.0+ makes it default, be EXPLICIT in config
   → Makes intent clear to every team member reading the code

2. SET acks=all WHEN USING IDEMPOTENCE
   → Idempotence REQUIRES acks=all
   → If you set acks=0 or acks=1 with idempotence, Kafka will throw ConfigException
   → This was covered in Project 11 — now you understand WHY it's mandatory

3. KEEP retries HIGH (or default to Integer.MAX_VALUE)
   → Idempotence works WITH retries, not against them
   → The whole point: retry safely without duplicates
   → Default retries with idempotence = 2147483647

4. SET max.in.flight.requests.per.connection ≤ 5
   → Idempotence supports up to 5 in-flight requests
   → Beyond 5, broker cannot guarantee ordering with deduplication
   → Kafka enforces this automatically when idempotence is enabled

5. USE STRUCTURED TRANSACTION IDs IN YOUR MESSAGES
   → Even with idempotence, include business-level transaction IDs
   → Helps with end-to-end traceability beyond Kafka
   → ATM example: atm_id + timestamp + sequence = business transaction ID

6. MONITOR PRODUCER METRICS
   → Track "record-error-rate" — should be near zero
   → Track "retry-rate" — expected to be non-zero (retries are normal)
   → Track "produce-throttle-time" — indicates broker pushback

❌ WHAT SHOULD NOT DO

1. DO NOT SET acks=0 OR acks=1 WITH IDEMPOTENCE
   → Kafka will throw: org.apache.kafka.common.config.ConfigException
   → Idempotence fundamentally requires full ISR acknowledgment
   → Revisit Project 11 if unclear why

2. DO NOT CONFUSE IDEMPOTENT PRODUCER WITH EXACTLY-ONCE END-TO-END
   → Idempotent producer = exactly-once PRODUCTION (producer → broker)
   → It does NOT guarantee exactly-once CONSUMPTION
   → Consumer can still read and process the same message twice on rebalance
   → End-to-end exactly-once requires Transactions (Project 18) + Consumer commit coordination

3. DO NOT SET max.in.flight.requests.per.connection > 5 WITH IDEMPOTENCE
   → Breaks the sequence tracking mechanism
   → Kafka 2.0+ will throw error if you try
   → Earlier versions silently lose ordering guarantees

4. DO NOT ASSUME PID SURVIVES PRODUCER RESTART
   → Producer ID (PID) is assigned by broker on producer initialization
   → If producer process crashes and restarts, it gets a NEW PID
   → Broker cannot deduplicate across different PIDs
   → In-flight retries during SAME session are protected, not across sessions
   → Cross-session exactly-once needs Transactional Producer (Project 18)

5. DO NOT RELY ON IDEMPOTENCE FOR MULTI-TOPIC ATOMIC WRITES
   → Idempotence is PER-PARTITION deduplication
   → Writing to topic-A AND topic-B atomically requires Transactions
   → This is the exact bridge to Project 18

6. DO NOT DISABLE IDEMPOTENCE FOR "PERFORMANCE"
   → Overhead is negligible: ~3% latency increase
   → Sequence numbers are lightweight integers
   → The cost of duplicates far exceeds the cost of deduplication

🚫 WHEN NOT TO USE

1. HIGH-THROUGHPUT LOG SHIPPING WHERE DUPLICATES ARE ACCEPTABLE
   → Application logs, click streams, debug events
   → Deduplication overhead (even minimal) is unnecessary
   → Use acks=1 with retries for good-enough durability

2. WHEN PRODUCER RESTARTS ARE FREQUENT AND CROSS-SESSION DEDUP IS NEEDED
   → Idempotence alone won't help — PID changes on restart
   → You need Transactional Producer with a stable transactional.id
   → That's Project 18's territory

3. WHEN USING VERY OLD KAFKA VERSIONS (< 0.11)
   → Idempotent producer protocol didn't exist
   → Must use application-level deduplication
   → Upgrade is strongly recommended

4. WHEN max.in.flight.requests.per.connection MUST BE > 5
   → Some ultra-high-throughput scenarios demand higher parallelism
   → Idempotence caps this at 5
   → Rare scenario — almost always 5 is sufficient

⚙️ HOW IT ONLY WORKS (INTERNAL MECHANICS)

DEEP DIVE: THE IDEMPOTENT PRODUCER PROTOCOL

═══════════════════════════════════════════════════════════════
STEP 1: PRODUCER INITIALIZATION (InitProducerIdRequest)
═══════════════════════════════════════════════════════════════

  Producer starts → Sends InitProducerIdRequest to Broker (Transaction Coordinator)
       ↓
  Broker assigns:
    → PID (Producer ID): Unique 64-bit long integer (e.g., PID=7)
    → Producer Epoch: Starts at 0, increments on each initialization
       ↓
  Producer stores PID + Epoch in memory
  
  NOTE: PID is NOT persisted — lost on crash
        Transactional.id (Project 18) makes it persistent

═══════════════════════════════════════════════════════════════
STEP 2: SEQUENCE NUMBER ASSIGNMENT (Per Partition)
═══════════════════════════════════════════════════════════════

  For EACH TopicPartition, producer maintains:
    → Sequence counter starting at 0
    → Increments by 1 for each message batch

  Message batch structure:
  ┌─────────────────────────────────────────┐
  │ PID = 7                                 │
  │ Epoch = 0                               │
  │ Partition = 3                            │
  │ Base Sequence = 42                       │
  │ Records: [record1, record2, record3]     │
  │ Last Sequence = 44                       │
  └─────────────────────────────────────────┘

═══════════════════════════════════════════════════════════════
STEP 3: BROKER-SIDE SEQUENCE TRACKING
═══════════════════════════════════════════════════════════════

  Broker maintains per <PID, Partition>:
    → Last committed sequence number

  On receiving a produce request:
  
  ┌──────────────────────────────────────────────────────────┐
  │ IF incoming_seq == last_committed_seq + 1                │
  │    → NEW message → WRITE to log → Update last_seq → ACK │
  │                                                          │
  │ IF incoming_seq == last_committed_seq                     │
  │    → DUPLICATE detected → DO NOT write → ACK (success)   │
  │                                                          │
  │ IF incoming_seq > last_committed_seq + 1                  │
  │    → OUT OF ORDER → Return OutOfOrderSequenceException   │
  │    → Producer must handle (usually fatal)                 │
  │                                                          │
  │ IF incoming_seq < last_committed_seq                      │
  │    → STALE DUPLICATE → DO NOT write → ACK (success)      │
  └──────────────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════════
STEP 4: THE 5-FLIGHT WINDOW EXPLAINED
═══════════════════════════════════════════════════════════════

  max.in.flight.requests.per.connection = 5 means:
  
  Producer can have 5 unacknowledged batches simultaneously:
  
  Time →
  Batch SEQ=40 ──→ [in flight]
  Batch SEQ=41 ──→ [in flight]
  Batch SEQ=42 ──→ [in flight]
  Batch SEQ=43 ──→ [in flight]
  Batch SEQ=44 ──→ [in flight]
  Batch SEQ=45 ──→ [BLOCKED - waiting for SEQ=40 ack]
  
  Broker tracks all 5 and can reorder if needed:
  → SEQ=41 arrives before SEQ=40 due to network
  → Broker BUFFERS SEQ=41 until SEQ=40 arrives
  → Then writes both in order
  
  With > 5 in-flight:
  → Buffer becomes too large
  → Ordering guarantees break down
  → Protocol explicitly rejects this configuration

═══════════════════════════════════════════════════════════════
STEP 5: THE RETRY FLOW (WHERE MAGIC HAPPENS)
═══════════════════════════════════════════════════════════════

  SCENARIO: Network drops ack for batch SEQ=42

  Timeline:
  
  T1: Producer sends [PID=7, SEQ=42, "$500 withdrawal"]
  T2: Broker receives → SEQ=42 > last(41) → WRITES → sends ACK
  T3: Network drops ACK before reaching producer
  T4: Producer timeout → marks SEQ=42 as failed
  T5: Producer retries → sends [PID=7, SEQ=42, "$500 withdrawal"] AGAIN
  T6: Broker receives → SEQ=42 == last(42) → DUPLICATE DETECTED
  T7: Broker sends ACK (DuplicateSequenceException internally, success to producer)
  T8: Producer receives ACK → considers SEQ=42 delivered
  
  NET RESULT: One write in log, producer satisfied, customer debited once ✓

═══════════════════════════════════════════════════════════════
STEP 6: PRODUCER EPOCH & FENCING
═══════════════════════════════════════════════════════════════

  If same transactional.id initializes twice (e.g., producer restart):
  
  Old producer: PID=7, Epoch=0
  New producer: PID=7, Epoch=1 (same PID because same transactional.id)
  
  Broker FENCES old producer:
  → Any request from Epoch=0 is rejected with ProducerFencedException
  → Prevents zombie producer from creating duplicates
  → This is fully explored in Project 18 (Transactions)

═══════════════════════════════════════════════════════════════
STEP 7: BROKER-SIDE STATE STORAGE
═══════════════════════════════════════════════════════════════

  Broker stores producer state in:
  → __producer_state snapshot files (per partition)
  → Located in log segment directory
  → Contains: PID → (epoch, last_sequence, last_offset)
  → Survives broker restart
  → Cleaned up when producer state expires (transactional.id.expiration.ms)
  
  Memory footprint per partition:
  → Each active PID entry ≈ 40 bytes
  → 1000 producers × 100 partitions = ~4MB (negligible)

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: "MY IDEMPOTENT PRODUCER STILL CREATES DUPLICATES"
─────────────────────────────────────────────────────────────
  Root Cause: Producer crashed and restarted without transactional.id
  → New PID assigned → Broker cannot correlate with old PID
  → Messages from old PID session already written
  → New PID sends "same" business message → Broker sees it as NEW
  Fix: Use transactional.id for cross-session dedup (Project 18)

PITFALL 2: OutOfOrderSequenceException
─────────────────────────────────────────────────────────────
  Root Cause: Producer state corruption or broker leadership change
  → Sequence gap detected: broker has SEQ=40, receives SEQ=43
  → Missing SEQ=41, 42
  → FATAL for idempotent producer — cannot recover automatically
  Fix: Producer must be restarted (gets new PID, fresh sequence)
  Prevention: Ensure stable network, proper timeout configuration

PITFALL 3: "IDEMPOTENCE ENABLED BUT acks=1"
─────────────────────────────────────────────────────────────
  → ConfigException thrown at producer startup
  → Kafka requires acks=all for idempotence
  → Common when migrating from Project 11 config without updating acks

PITFALL 4: PERFORMANCE DEGRADATION WITH MANY SMALL MESSAGES
─────────────────────────────────────────────────────────────
  → Each message batch gets a sequence number
  → Tiny batches = many sequences = more broker tracking overhead
  → Fix: Increase batch.size and linger.ms (covered in Project 39)
  → Let producer batch multiple ATM events before sending

PITFALL 5: CLUSTER UPGRADE ISSUES
─────────────────────────────────────────────────────────────
  → Idempotence requires inter.broker.protocol.version ≥ 0.11
  → message.format.version must be ≥ 0.11 (v2 record format)
  → Old format doesn't carry PID/sequence in record headers
  → Rolling upgrade: ensure ALL brokers are upgraded before enabling

PITFALL 6: CONFUSING "EXACTLY-ONCE PRODUCTION" WITH "EXACTLY-ONCE PROCESSING"
─────────────────────────────────────────────────────────────
  → Idempotent producer: message written to Kafka exactly once ✓
  → Consumer processing: can still process twice on rebalance ✗
  → True E2E exactly-once needs:
    → Transactional producer (Project 18)
    → Consumer with read_committed isolation (Project 18)
    → OR Kafka Streams with EOS (Project 43)

📈 SCALABILITY CONSIDERATIONS

SEQUENCE STATE MEMORY ON BROKERS
─────────────────────────────────
  → Each active PID costs ~40 bytes per partition
  → 10,000 ATM machines × 10 partitions = 100,000 entries ≈ 4MB
  → Highly scalable — not a concern even at massive scale

SNAPSHOT FILE SIZE
─────────────────────────────────
  → Producer state snapshots written alongside log segments
  → Snapshots grow with active PIDs
  → Old PIDs expire via transactional.id.expiration.ms (default: 7 days)
  → Stale PIDs cleaned up automatically

THROUGHPUT IMPACT
─────────────────────────────────
  Benchmark comparison (single producer, single partition):

  ┌────────────────────────┬──────────────┬──────────────┐
  │ Configuration          │ Throughput   │ Latency (p99)│
  ├────────────────────────┼──────────────┼──────────────┤
  │ acks=1, no idempotence │ 95,000 msg/s │ 5ms          │
  │ acks=all, no idempot.  │ 72,000 msg/s │ 12ms         │
  │ acks=all + idempotent  │ 70,000 msg/s │ 13ms         │
  └────────────────────────┴──────────────┴──────────────┘
  
  → Idempotence adds ~3% overhead on top of acks=all
  → The bottleneck is acks=all (ISR replication), NOT idempotence
  → Since acks=all is REQUIRED anyway, idempotence is essentially "free"

IN-FLIGHT REQUEST LIMIT IMPACT
─────────────────────────────────
  → max.in.flight.requests.per.connection=5 is the cap
  → For most workloads, 5 is more than enough
  → With batching (batch.size=64KB, linger.ms=20), each flight carries thousands of records
  → 5 flights × 64KB = 320KB in-flight per connection — sufficient for most use cases

MULTI-PRODUCER SCALING
─────────────────────────────────
  → Each producer gets unique PID — no interference
  → 50,000 ATMs = 50,000 PIDs
  → Broker handles this via hash map lookups — O(1)
  → Horizontal producer scaling works naturally

HOW TO VERIFY IDEMPOTENCE IS WORKING
─────────────────────────────────────

     ls /var/kafka-logs/atm-transactions-0/
     → Files like: 00000000000000000042.snapshot
   → Use kafka-dump-log tool:
     kafka-dump-log.sh --files 00000000000000000042.snapshot --print-data-log

3. TESTING DUPLICATE REJECTION
   → Intentionally send same PID+SEQ (via test harness)
   → Verify log offset does NOT advance
   → Verify consumer reads only ONE copy

4. CONSUMER-SIDE OBSERVABILITY
   → With idempotent producer, consumer sees CLEAN stream
   → No business-level dedup logic needed in consumer
   → But consumer can still double-process on rebalance (caveat!)

5. READING THE RECORD HEADERS
   → V2 record format includes PID and sequence in record batch header
   → kafka-console-consumer with --property print.value=true won't show these
   → Use kafka-dump-log.sh for full batch inspection:

   baseOffset: 42 lastOffset: 42 count: 1
   producerId: 7 producerEpoch: 0
   baseSequence: 42 lastSequence: 42
   ← THIS confirms idempotent production

"""
ATM Transaction Producer with Idempotent Guarantees
Builds on: Project 11 (acks=all) + Project 12 (Python client) + Project 13 (message design)
"""

import json
import time
import uuid
import random
from datetime import datetime, timezone
from confluent_kafka import Producer, KafkaError


# ============================================================
# CONFIGURATION — THE HEART OF IDEMPOTENT PRODUCTION
# ============================================================

producer_config = {
    'bootstrap.servers': 'localhost:9092',

    # ─── IDEMPOTENCE: THE SINGLE MOST IMPORTANT SETTING ───
    'enable.idempotence': True,

    # ─── REQUIRED BY IDEMPOTENCE (Kafka enforces these) ───
    'acks': 'all',

    # ─── SAFE RETRIES (idempotence makes retries safe) ───
    # With idempotence, retries do NOT cause duplicates
    # Default becomes 2147483647 when idempotence is enabled
    'retries': 2147483647,

    # ─── MAX IN-FLIGHT REQUESTS (must be ≤ 5 for idempotence) ───
    # Broker tracks sequences for up to 5 in-flight batches
    'max.in.flight.requests.per.connection': 5,

    # ─── BATCHING (improves throughput with idempotence) ───
    # Each batch gets ONE sequence number — bigger batches = fewer sequences
    'batch.size': 32768,           # 32KB batch
    'linger.ms': 20,               # Wait 20ms to fill batch

    # ─── COMPRESSION (works perfectly with idempotence) ───
    'compression.type': 'snappy',

    # ─── CLIENT IDENTIFICATION ───
    'client.id': 'atm-producer-01',

    # ─── DELIVERY TIMEOUT ───
    # Total time allowed for a message to be delivered
    # Must be >= retry.backoff.ms * retries (practically)
    'delivery.timeout.ms': 120000,  # 2 minutes

    # ─── REQUEST TIMEOUT ───
    'request.timeout.ms': 30000,    # 30 seconds per attempt
}


# ============================================================
# ATM TRANSACTION GENERATOR
# ============================================================

class ATMTransactionGenerator:
    """
    Simulates 50,000 ATM machines generating financial transactions.
    Each transaction has a globally unique business transaction ID.
    """

    ATM_LOCATIONS = [
        {'atm_id': f'ATM-{str(i).zfill(5)}', 'city': city, 'region': region}
        for i, (city, region) in enumerate([
            ('New York', 'EAST'), ('Los Angeles', 'WEST'),
            ('Chicago', 'CENTRAL'), ('Houston', 'SOUTH'),
            ('Phoenix', 'WEST'), ('Philadelphia', 'EAST'),
            ('San Antonio', 'SOUTH'), ('San Diego', 'WEST'),
            ('Dallas', 'SOUTH'), ('Seattle', 'WEST'),
        ])
    ]

    TRANSACTION_TYPES = ['WITHDRAWAL', 'DEPOSIT', 'BALANCE_INQUIRY', 'TRANSFER']

    def generate_transaction(self):
        atm = random.choice(self.ATM_LOCATIONS)
        txn_type = random.choice(self.TRANSACTION_TYPES)

        # Business-level transaction ID — unique regardless of Kafka dedup
        business_txn_id = f"{atm['atm_id']}-{datetime.now(timezone.utc).strftime('%Y%m%d%H%M%S%f')}-{uuid.uuid4().hex[:8]}"

        amount = 0.0
        if txn_type == 'WITHDRAWAL':
            amount = round(random.choice([20, 40, 50, 60, 80, 100, 200, 500]) * 1.0, 2)
        elif txn_type == 'DEPOSIT':
            amount = round(random.uniform(50, 5000), 2)
        elif txn_type == 'TRANSFER':
            amount = round(random.uniform(10, 2000), 2)

        return {
            'transaction_id': business_txn_id,
            'atm_id': atm['atm_id'],
            'city': atm['city'],
            'region': atm['region'],
            'transaction_type': txn_type,
            'amount': amount,
            'currency': 'USD',
            'customer_id': f'CUST-{random.randint(100000, 999999)}',
            'card_last_four': str(random.randint(1000, 9999)),
            'timestamp_utc': datetime.now(timezone.utc).isoformat(),
            'atm_software_version': '3.2.1',
        }


# ============================================================
# DELIVERY CALLBACK — OBSERVING IDEMPOTENT BEHAVIOR
# ============================================================

class DeliveryTracker:
    """
    Tracks delivery results to observe idempotent producer behavior.
    In a real system, this feeds into monitoring dashboards (Project 28).
    """

    def __init__(self):
        self.success_count = 0
        self.error_count = 0
        self.partition_distribution = {}

    def on_delivery(self, err, msg):
        if err is not None:
            self.error_count += 1
            print(f'  ❌ DELIVERY FAILED: {err}')
            print(f'     Topic: {msg.topic()}, Partition: {msg.partition()}')

            # Check for specific idempotent producer errors
            if err.code() == KafkaError._MSG_TIMED_OUT:
                print('     → Message timed out — will be retried automatically')
            elif err.code() == KafkaError.OUT_OF_ORDER_SEQUENCE_NUMBER:
                print('     → FATAL: Out of order sequence — producer must restart')
            elif err.code() == KafkaError.DUPLICATE_SEQUENCE_NUMBER:
                print('     → Duplicate detected by broker — this is EXPECTED during retries')
        else:
            self.success_count += 1
            partition = msg.partition()
            offset = msg.offset()
            self.partition_distribution[partition] = self.partition_distribution.get(partition, 0) + 1

            # Extract the transaction_id from value for logging
            try:
                value = json.loads(msg.value().decode('utf-8'))
                txn_id = value.get('transaction_id', 'unknown')
            except (json.JSONDecodeError, AttributeError):
                txn_id = 'unknown'

            print(f'  ✅ Delivered: partition={partition}, offset={offset}, txn={txn_id[:30]}...')

    def print_summary(self):
        print('\n' + '=' * 70)
        print('DELIVERY SUMMARY')
        print('=' * 70)
        print(f'  Total Success:  {self.success_count}')
        print(f'  Total Errors:   {self.error_count}')
        print(f'  Partition Distribution:')

            count = self.partition_distribution[p]

class LedgerProcessor:
    """
    Simulates a bank ledger system that processes ATM transactions.
    
    CRITICAL UNDERSTANDING:
    → Without idempotent producer, this processor might see DUPLICATE transactions
    → With idempotent producer, each transaction appears EXACTLY ONCE in Kafka
    → BUT: Consumer can still re-process on rebalance (at-least-once consumption)
    → True end-to-end exactly-once needs Transactions (Project 18)
    """

    def __init__(self):
        self.processed_count = 0
        self.total_withdrawals = 0.0
        self.total_deposits = 0.0
        self.seen_transaction_ids = set()
        self.duplicate_detections = 0
        self.per_atm_counts = {}
        self.per_partition_counts = {}

    def process(self, msg):
        partition = msg.partition()
        offset = msg.offset()

        # ─── DESERIALIZE ───
        try:
            key = msg.key().decode('utf-8') if msg.key() else 'no-key'
            value = json.loads(msg.value().decode('utf-8'))
        except (json.JSONDecodeError, AttributeError, UnicodeDecodeError) as e:
            print(f'  ⚠️  Failed to deserialize message at partition={partition}, offset={offset}: {e}')
            return

        txn_id = value.get('transaction_id', 'unknown')
        txn_type = value.get('transaction_type', 'unknown')
        amount = value.get('amount', 0.0)
        atm_id = value.get('atm_id', 'unknown')

        # ─── APPLICATION-LEVEL DUPLICATE CHECK ───
        # Even with idempotent producer, we add application-level check
        # Because consumer rebalances can cause re-processing
        # This is defense-in-depth until we add Transactions (Project 18)
        if txn_id in self.seen_transaction_ids:
            self.duplicate_detections += 1
            print(f'  🔄 DUPLICATE DETECTED (consumer-side): {txn_id[:40]}...')
            print(f'     → This is a consumer rebalance re-read, NOT a producer duplicate')
            return

        self.seen_transaction_ids.add(txn_id)

        # ─── PROCESS TRANSACTION ───
        if txn_type == 'WITHDRAWAL':
            self.total_withdrawals += amount
        elif txn_type == 'DEPOSIT':
            self.total_deposits += amount

        # ─── TRACK STATISTICS ───
        self.processed_count += 1
        self.per_atm_counts[atm_id] = self.per_atm_counts.get(atm_id, 0) + 1
        self.per_partition_counts[partition] = self.per_partition_counts.get(partition, 0) + 1

        # ─── READ HEADERS ───
        headers = msg.headers() or []
        header_dict = {h[0]: h[1].decode('utf-8') if h[1] else '' for h in headers}

        # ─── LOG PROCESSING ───
        print(f'  📝 [{self.processed_count:>4}] P{partition}|O{offset:<6} '
              f'ATM={atm_id} TYPE={txn_type:<16} '
              f'AMT=${amount:>10.2f} '
              f'REGION={header_dict.get("region", "?")}')

    def print_summary(self):
        print('\n' + '=' * 70)
        print('LEDGER PROCESSING SUMMARY')
        print('=' * 70)
        print(f'  Total Processed:        {self.processed_count}')
        print(f'  Consumer-Side Dupes:    {self.duplicate_detections}')
        print(f'  Unique Transactions:    {len(self.seen_transaction_ids)}')
        print(f'  Total Withdrawals:      ${self.total_withdrawals:,.2f}')
        print(f'  Total Deposits:         ${self.total_deposits:,.2f}')
        print(f'  Net Cash Flow:          ${self.total_deposits - self.total_withdrawals:,.2f}')
        print()
        print('  Per-ATM Transaction Counts:')
        for atm_id in sorted(self.per_atm_counts.keys()):
            count = self.per_atm_counts[atm_id]
            print(f'    {atm_id}: {count}')
        print()
        print('  Per-Partition Distribution:')
        for p in sorted(self.per_partition_counts.keys()):
            count = self.per_partition_counts[p]
            bar = '█' * count
            print(f'    Partition {p}: {count:>4} {bar}')
        print('=' * 70)

def run_atm_consumer():
    print('=' * 70)
    print('ATM LEDGER CONSUMER — STARTING')
    print('=' * 70)
    print('  Consuming from: atm-transactions')
    print('  Consumer group: atm-ledger-processor')
    print('  Press Ctrl+C to stop and see summary')
    print()

    consumer = Consumer(consumer_config)
    consumer.subscribe(['atm-transactions'])
    processor = LedgerProcessor()

    # ─── GRACEFUL SHUTDOWN ───
    running = True

    def signal_handler(sig, frame):
        nonlocal running
        print('\n\n⚡ Shutdown signal received...')
        running = False

    signal.signal(signal.SIGINT, signal_handler)

    # ─── CONSUMPTION LOOP ───
    empty_polls = 0
    max_empty_polls = 20  # Stop after 20 empty polls (10 seconds of no data)

    try:
        while running:
            msg = consumer.poll(timeout=0.5)

            if msg is None:
                empty_polls += 1
                if empty_polls >= max_empty_polls:
                    print(f'\n  No new messages for {max_empty_polls * 0.5:.0f}s — stopping...')
                    break
                continue

            empty_polls = 0  # Reset on successful poll

            if msg.error():
                if msg.error().code() == KafkaError._PARTITION_EOF:
                    # End of partition — not an error
                    print(f'  ℹ️  Reached end of partition {msg.partition()} at offset {msg.offset()}')
                    continue
                else:
                    print(f'  ❌ Consumer error: {msg.error()}')
                    continue

            # ─── PROCESS THE MESSAGE ───
            processor.process(msg)

    finally:
        # ─── CLEANUP ───
        processor.print_summary()
        consumer.close()
        print('\n✓ Consumer closed cleanly')


# ============================================================
# ENTRY POINT
# ============================================================

if __name__ == '__main__':
    run_atm_consumer()

def create_test_topic():

def produce_with_idempotence(transactions):
    """Produce with idempotent producer — duplicates prevented at broker"""
    config = {
        'bootstrap.servers': BOOTSTRAP_SERVERS,
        'enable.idempotence': True,
        'acks': 'all',
        'max.in.flight.requests.per.connection': 5,
        'client.id': 'idempotent-test-producer',
    }

    producer = Producer(config)
    delivered = []

    def on_delivery(err, msg):
        if err:
            print(f'    ❌ Error: {err}')
        else:
            delivered.append({
                'partition': msg.partition(),
                'offset': msg.offset(),
                'key': msg.key().decode('utf-8') if msg.key() else None,
            })

    for txn in transactions:
        producer.produce(
            topic=TEST_TOPIC,
            key=txn['atm_id'].encode('utf-8'),
            value=json.dumps(txn).encode('utf-8'),
            callback=on_delivery,
        )
        producer.poll(0)

    producer.flush(timeout=10)
    return delivered


def consume_all_messages(expected_count, timeout_seconds=15):
    """Consume all messages from test topic"""
    config = {
        'bootstrap.servers': BOOTSTRAP_SERVERS,
        'group.id': f'test-consumer-{uuid.uuid4().hex[:8]}',
        'auto.offset.reset': 'earliest',
        'enable.auto.commit': False,
    }

    consumer = Consumer(config)
    consumer.subscribe([TEST_TOPIC])

    messages = []
    start_time = time.time()

    while time.time() - start_time < timeout_seconds:
        msg = consumer.poll(timeout=1.0)
        if msg is None:
            if len(messages) >= expected_count:
                break
            continue
        if msg.error():
            if msg.error().code() == KafkaError._PARTITION_EOF:
                continue
            continue

        value = json.loads(msg.value().decode('utf-8'))
        messages.append({
            'partition': msg.partition(),
            'offset': msg.offset(),
            'transaction_id': value['transaction_id'],
            'amount': value['amount'],
            'type': value['transaction_type'],
        })

    consumer.close()
    return messages

🔑 KEY TAKEAWAY — CONNECTING PROJECT 11 → 17 → 18

PROJECT 11 TAUGHT:
→ acks=all guarantees the BROKER durably wrote the message
→ BUT the producer may NOT KNOW the broker succeeded (lost ack)
→ Retries cause DUPLICATES — the "at-least-once" problem

PROJECT 17 SOLVES:
→ Idempotent producer adds PID + Sequence Number to every batch
→ Broker tracks sequences and silently deduplicates retries
→ enable.idempotence=true — one config flag, zero code changes
→ Exactly-once PRODUCTION guaranteed within a single producer session
→ Overhead is negligible (~3% on top of acks=all)

WHAT PROJECT 17 DOES NOT SOLVE (→ PROJECT 18):
─────────────────────────────────────────────────

  UNSOLVED PROBLEM 1: CROSS-SESSION DEDUPLICATION
  ─────────────────────────────────────────────────
  → Producer crashes after sending batch SEQ=42
  → Producer restarts → gets NEW PID (e.g., PID=8 instead of PID=7)
  → Producer resends the "same" business message
  → Broker sees PID=8, SEQ=0 → treats as BRAND NEW message
  → DUPLICATE written despite idempotence being enabled
  → FIX: transactional.id gives producer a STABLE identity across restarts
         Broker maps transactional.id → PID, so restarts get SAME PID
         This is Project 18

  UNSOLVED PROBLEM 2: ATOMIC MULTI-PARTITION WRITES
  ─────────────────────────────────────────────────
  → ATM withdrawal must write to BOTH:
      → Topic: "atm-transactions" (partition 3)
      → Topic: "customer-balances" (partition 7)
  → With idempotent producer, each write is independent
  → If first write succeeds but second fails:
      → Transaction recorded but balance NOT updated
      → Data inconsistency across topics
  → FIX: Kafka Transactions wrap MULTIPLE writes in one atomic unit
         Either ALL writes commit or NONE do
         This is Project 18

  UNSOLVED PROBLEM 3: CONSUME-PROCESS-PRODUCE EXACTLY-ONCE
  ─────────────────────────────────────────────────────────
  → A stream processor reads from topic A, transforms, writes to topic B
  → With idempotent producer:
      → Read message from A ✓
      → Process it ✓
      → Write result to B ✓ (idempotent — no duplicate in B)
      → Commit offset for A → CRASH BEFORE COMMIT
      → On restart: re-reads from A → re-processes → writes to B AGAIN
      → B now has DUPLICATE result (different PID on restart)
  → FIX: Kafka Transactions tie offset commit + produce into ONE atomic operation
         Either BOTH happen or NEITHER happens
         This is Project 18 (and later Project 43 for Kafka Streams EOS)

THE MENTAL MODEL:
═══════════════════════════════════════════════════════════════════

  Think of it as LAYERS OF PROTECTION:

  Layer 1 (Project 11): "The message REACHED the vault"
    → acks=all = bank vault confirms receipt
    → But messenger doesn't know vault confirmed (lost receipt)
    → Messenger delivers again → vault stores duplicate

  Layer 2 (Project 17): "The vault recognizes the SAME envelope"
    → Idempotence = envelope has serial number
    → Vault checks: "Serial #42? Already stored. Acknowledged."
    → Single vault, single messenger session

  Layer 3 (Project 18): "The messenger has a PERMANENT badge"
    → transactional.id = messenger's permanent employee badge
    → Even if messenger goes home and comes back next day
    → Vault recognizes badge → resumes from last serial number
    → PLUS: Can deliver to MULTIPLE vaults atomically

  Layer 4 (Project 43): "The ENTIRE delivery chain is tracked"
    → Pick up from vault A → process → deliver to vault B → stamp receipt at A
    → All as ONE atomic operation
    → Kafka Streams handles this automatically

🔗 BRIDGE TO PROJECT 18: THE FUND TRANSFER SYSTEM — KAFKA TRANSACTIONS

NOW THAT YOU UNDERSTAND:
  ✓ Why duplicates happen (ack loss + retry)
  ✓ How PID + Sequence Number enables broker-side dedup
  ✓ That idempotence is per-partition, per-session
  ✓ That cross-session and multi-partition atomicity need MORE

PROJECT 18 WILL INTRODUCE:
  → transactional.id — a STABLE producer identity that survives restarts
  → initTransactions() — registers the producer with the transaction coordinator
  → beginTransaction() — starts an atomic batch of writes
  → commitTransaction() — atomically makes ALL writes visible to consumers
  → abortTransaction() — rolls back ALL writes as if they never happened
  → read_committed isolation — consumers that ONLY see committed transactions
  → Zombie fencing — preventing old producer instances from creating chaos

THE SCENARIO:
  → A customer transfers $1,000 from Account A to Account B
  → This requires TWO writes:
      → Debit event to "account-debits" topic
      → Credit event to "account-credits" topic
  → With idempotent producer (Project 17):
      → Each write is individually safe from duplicates
      → BUT if debit succeeds and credit fails → money DISAPPEARS
  → With transactional producer (Project 18):
      → Both writes wrapped in beginTransaction()...commitTransaction()
      → Either BOTH visible or NEITHER visible
      → Consumers with read_committed never see partial state

WHY YOU MUST UNDERSTAND PROJECT 17 BEFORE PROJECT 18:
  → Transactions BUILD ON idempotence — they don't replace it
  → enable.idempotence=true is AUTOMATICALLY set when using transactions
  → PID + Sequence is still the underlying dedup mechanism
  → Transactions ADD: stable PID mapping, atomic multi-partition commit, consumer isolation
  → Without understanding the PID/SEQ foundation, transactions are a black box
